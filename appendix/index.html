附录​

附录 A：高风险场景降级链参考​

对于医疗预问诊、复杂客服、金融风控等高危场景，默认的四项降级链（L1 结构校验 + DET 值域复验 + logprobs 检查 + 硬编码兜底）是通用底线，不可缩减。在此基础上，可追加以下增强方案：

simple: [L1_retry, DET_value_recheck, logprobs_check, cross_task_extension_check, hardcoded]
escalated: [L1_retry, DET_value_recheck, logprobs_check, cross_task_extension_check, hardcoded, human_takeover]
human_takeover: { target: "人工队列", sla: "5分钟" }

启用条件：仅当场景的"错误代价"判据为"代价大且无客观答案"或"代价大但有客观校验"时，考虑启用 escalated 模式。默认场景仍用 simple（四项降级链 + 硬编码兜底）。

注意：此方案为场景实例化时的参考，不进宪法模板。启用 escalated（升级）模式时，需在场景的边界清单中显式声明人工接管的目标队列和 SLA（服务等级协议）；并联模式下，human_takeover（人工接管）事件由 Case Supervisor（案例监督器）路由到人工审批队列，而非当前会话的 UI（用户界面）。四项降级链不可被 human_takeover（人工接管）替换——human_takeover（人工接管）是五项之后的扩展层，不是替代。

若模型 B 在所有指标上优于模型 A 且达到预设阈值，系统可自动将路由表中的模型选择切换为模型 B，或触发人工确认后切换。两个模型收到完全相同的 contract-in（输入契约），受完全相同的局部宪法约束，输出完全相同的 schema（模式）——对比不是靠人工评估，而是靠 DET 节点对两个结构化输出的质量指标做确定性比较（topic_based（基于主题）的主题相关性评分除外，需 LLM辅助）。

附录 C：工程实施规范​

C.1 执行体调用链追踪（OpenTelemetry 集成）​

问题背景：三类执行体（DET/LLM/HUMAN）的调用缺少全链路追踪，运维监控和故障定位困难。

实施规范： 每次 DET/LLM/HUMAN 调用生成 OpenTelemetry span span 含字段：trace_id / span_id / parent_span_id / executor_type（DET/LLM/HUMAN）/ executor_name / start_time / end_time / status（success/fail/timeout）/ input_summary / output_summary / error_detail 用 OpenTelemetry 标准格式上报（Jaeger/Zipkin 兼容） 运维侧用 Jaeger/Zipkin 可视化调用链 监控指标（校验失败率/偏离率/logprobs 拦截率/embedding 调用次数/topicEvolution event 写入频率/冷启动应急解冻触发次数）直接从 trace 提取 每个会话一个 trace，每个节点一个 span，每个执行体调用一个子 span

集成位置： 第0层全局硬匹配：DET span 第0.5层关键词扫描：DET span 第1层意图识别：LLM span 第2层环节内自治：LLM span（环节内多轮对话一个 span，含 turn_count 子字段） 第3层 DET 复验：DET span（含四项检查各自的子 span） EXECUTING：DET span embedding 调用（topic_based 模式阶段2）：embedding span（含 askingField / similarity_score / protected_count 子字段） topicEvolution event 写入：DET span（含 topicId / eventId / nodeId / stateChange 子字段） 上下文管理器三层（匹配/决策/拼接）：各一个 DET span（含 matched_rule_count / injected_rule_count / token_used 子字段）

收益：运维侧可实时观测执行体调用链，快速定位故障节点；监控指标从 trace 自动提取，无需额外埋点。

C.2 宪法冲突检测引擎​

实施规范：

  • single_param 规则用 json-rules-engine（Node.js）或 Drools（Java）实现确定性 range/enum 校验
  • cross_param 规则由冲突检测引擎在遍历 single_param 完成后执行 compute 表达式求值——用沙箱 eval()（仅允许 + - * / floor ceil min max 运算符 + params 变量 + constants 常量）
  • 参数变更时通过 params 字段做依赖追踪——仅被引用参数的变更触发对应 cross_param 规则重新执行
  • violation_action 默认值：single_param 规则默认为 hard_reject；cross_param 规则默认为 warning
  • violation_action=hard_reject 行为：拒绝保存 + 返回错误详情
  • violation_action=warning 行为：弹窗展示 warning message + suggested_adjustment，用户可选择"仍然保存"或"调整后重试"。warning 不阻止保存
  • compute 表达式求值失败（语法错误、引用不存在参数）→ 视为 hard_reject，阻止保存并返回明确的表达式错误位置
  • conflict_check 规则用 JSON 格式声明，与 tunable 参数一同存在 tunables.json
  • 保存前执行全部规则：single_param 任一 hard_reject → 拒绝保存；cross_param warning → 弹窗但允许继续
  • 冲突检测引擎是确定性代码，不调 LLM

cross_param 规则清单的唯一来源是 §10.1——本附录不再重复规则 JSON，只定义实现规范。single_param 规则示例：

{
  "conflict_rules": [
    {
      "type": "single_param",
      "name": "logprobs_threshold_range_check",
      "condition": {
        "all": [
          { "fact": "parameter", "path": "$.logprobs_threshold", "operator": "lessThan", "value": 0 },
          { "fact": "parameter", "path": "$.logprobs_threshold", "operator": "greaterThan", "value": 1 }
        ]
      },
      "event": { "type": "conflict", "message": "logprobs_threshold 必须在 [0, 1] 范围内" }
    }
  ]
}

C.3 contractStore 物理存储（SQLite + WAL 模式）​

问题背景：contractStore 跨会话持久化，需要支持并发和崩溃恢复。

实施规范：

  • SQLite 单文件数据库，每个用户/项目一个 .db 文件
  • WAL（Write-Ahead Logging）模式支持并发读 + 单写
  • 数据库文件路径：<project_root>/.contractstore/<user_id>.db

contractStore 表结构（SQL DDL 唯一来源——正文各处只描述用途，DDL 统一在此定义）：

-- ANALYZING 节点契约副本（支持意图级纠错逃生舱）
CREATE TABLE IF NOT EXISTS analyzing_contract_in (
    id TEXT PRIMARY KEY,
    userId TEXT NOT NULL,
    rawInput TEXT NOT NULL,
    createdAt TEXT NOT NULL

CREATE TABLE IF NOT EXISTS analyzing_contract_out (
    id TEXT PRIMARY KEY,

-- topic_based 主题演化

-- 房间对话日志（保留期独立于 sessionCheckpoint.ttl，按 importance 分级）
CREATE TABLE IF NOT EXISTS roomConversationLog (

索引：

  • roomConversationLog: (userId, projectId, roomId, segmentType, createdAt)、(userId, roomId, segmentType, summaryImportance) 用于v5.7分段保留拼接
  • conversationArchive: (userId, projectId, roomId, createdAt)、FTS5 on (userMessage, modelOutput) 用于全文搜索（v5.8）
  • roomStateIndex: userId 索引
  • outputRegistry: (roomId, createdAt)、(intent, outputType)、FTS on (outputName, outputSummary, keywords)
  • domainRules: (topicPath)、(stepName)、(status, immutableLevel)

并发控制：

  • WAL 模式支持多读单写
  • 写操作用事务包裹，崩溃时自动回滚
  • 多用户并发时，每个用户独立 .db 文件，无跨用户锁竞争

roomConversationLog 分段保留清理逻辑（v5.7）：

  • 产出物确认保存后，模型后台生成摘要（segmentType='summary'），原始对话轮次删除替换为摘要记录
  • 新摘要生成时立即检查保留数量：critical摘要>10个 / high摘要>5个 / normal摘要>2个 → FIFO删除最旧的
  • SLACK_NODE 不走分段保留，保留最近10轮（FIFO删除最早的），不生成摘要
  • 无产出物的房间（情况1）全量保留，不自动清理
  • 可通过 tunable conversation_segmentation_enabled=false 禁用，退回全量保留

附录 D：align-session 可选扩展规范​

align-session 是可选扩展，非态控架构标准能力。若场景含意图校准需求，可启用 align-session 扩展（在 §3.1 意图收敛机制的收敛链路中插入第 0.1 层）。启用时需：

  • 额外编写 align-session.json 宪法
  • 在路由表中增加 align 相关条目
  • 在 tunable 参数中增加 divergence_threshold
  • 同步更新场景的边界清单
  • 在 §3.1 收敛链路的第0层之后插入第0.1层（align 静默模式触发与处理）

align-session 核心机制（仅在启用时生效）： 静默输入：/align 元指令触发后，挂起语义管道，所有输入走静默缓冲区，直到完成指令或退出词 多模型并行仲裁：主次模型并行 + 规则模型串行 锚点确认：主/次模型标注 core_anchor，用户确认后沉淀 user_confirmed_anchor 迭代裁决：divergence_threshold 判定分歧度 对齐数据包沉淀：rounds[] / divergence_points / user_arbitration

align 相关 tunable 参数（仅在启用时声明）： divergence_threshold：默认 0.7，校准模式分歧度阈值

align-session 宪法 complete 条件：用户最终确认。

align 相关路由条目（仅在启用时配置）： ANALYZING（intent=align）→ IN_SESSION(align) IN_SESSION(align) complete → EXECUTING（DET） IN_SESSION(align) giveup → LISTENING IN_SESSION(align) off-task → ANALYZING（轻量）

与核心架构的边界：align-session 启用时不修改根宪法第 1-4 条及架构机制 M1-M4，仅在路由表和 tunable 参数中扩展，并在收敛链路插入第0.1层。align-session 不启用时，核心架构完全不受影响。

附录 E：编程规则宪法​

定位：约束代码模型的元规则，全局生效于整个 N1→N15 管线。任何涉及代码生成的环节（N13）必须在生成代码前校验此宪法，违反任一条款拒绝生成。

约束对象：代码模型（生成骨架代码的 AI），非运行时模型。

生效范围：母本的一部分，设计者在 P0 认知加载时即需理解，N13 代码模型强制输入。

编号条款内容
1执行体分派铁律每个状态的执行体由 v5.0/N3 决定，代码模型不得自行更改。ANALYZING=LLM，IN_SESSION=LLM，DET 复验=DET，EXECUTING=DET。不得将 ANALYZING/IN_SESSION 降级为正则或 parser。v5.4 修正：IN_SESSION 可调用 DET 工具（function calling）；EXECUTING 仅保留给纯写入/删除操作（record INSERT / delete DELETE），查询/对比类操作在 IN_SESSION 内通过工具调用完成
2DET 职责边界DET 只做确定性计算：元指令 EXACT_MATCH（精确匹配）+ 值域/格式复验 + 关键词扫描标记 + logprobs 阈值裁决 + 路由矩阵裁决（intent × inputNature）+ 规则子图匹配与冲突检测 + 文件操作（产出物复制）+ SQL 查询（作为 IN_SESSION 的工具被调用）。核心是"确定性计算"这个定语——凡是能用确定性代码完成的计算都归 DET，不扩展到意图识别、字段采集、查询理解、语义判断等需要概率推理的任务
3LLM 职责边界LLM 做：意图识别 + 字段采集/追问/偏离判断/查询理解 + 语义边界判断 + 工具调用决策 + 结果格式化（自然语言生成）。不得用正则/parser 替代
4降级规则无 apiKey 时 ANALYZING/IN_SESSION 必须拒绝服务并明确告知原因，不得以任何形式降级为正则/DET 替代。降级模式如存在必须标注"降级非正道"
5契约对齐代码必须严格对齐 N11 契约（contract-in/out schema），不自创接口
6tunable（可调参数）/接口隔离阈值从 getTunable() 读，调度器只调 context-manager（上下文管理器）三接口（匹配/决策/拼接），不内联拼接逻辑。上下文管理器三层职责（匹配→决策→拼接）各自独立，不得合并
7工具分层发现（v5.5 新增）工具分三层：必用工具（环节宪法 @section tools 声明，完整定义自动注入）/ 工具清单（@section tool_catalog，轻量级目录每次注入）/ 选用工具（系统级工具库 toolRegistry，完整定义按需加载）。模型从清单选择→调度器加载完整定义→模型调用。LLM 不直接写 SQL，通过 DET 工具间接操作数据库。search_tools meta-tool 始终可用，用于清单外的工具发现。工具内部支持时间解析（"this_month"→日期范围）

附录 G：领域规则讨论房间（v5.8 新增，建议默认环节）​

定位：所有智能体建议设定的默认环节——集中讨论/确认/管理领域规则。非强制要求，但强烈建议启用。

设计理由：领域规则的确认散落在各个环节里（IN_SESSION complete 时模型推断规则→ruleCandidates→用户确认），没有专门的地方让用户集中管理规则。领域规则讨论房间填补这个缺口。

房间属性：

维度值说明
taskTypetopic_based讨论规则是开放式对话
importancecritical和核心业务环节一样重要
执行体LLM 通用模型规则讨论需要语义理解
topicEvolutionEnabledtruetopic_based 自动启用
元指令切换"切断房间"和其他房间统一

触发条件：

  • 用户主动说"讨论规则"/"修改偏好"/"看看我之前说的规则" → ANALYZING 识别 → 进入 domain-rule-session
  • 其他房间的 model_inferred 规则积累到一定数量（tunable rule_confirmation_threshold，默认5条）→ 系统提示用户进入

房间功能：

功能说明数据操作
查看待确认规则展示 ruleCandidates 中的候选规则读 ruleCandidates
确认候选规则用户确认后升格为 L2，写入 domainRulesruleCandidates → domainRules
修改候选规则用户修改后写入，标记 L2ruleCandidates → domainRules
拒绝候选规则标记 rejectedruleCandidates.status = rejected
主动声明规则用户口述新规则（user_initiated → L2）写入 domainRules
查看规则冲突展示 conflicts_with 关系的规则对读 domainRules.edges
仲裁规则冲突用户选择保留哪条更新 domainRules
修改已有规则用户修改已生效规则更新 domainRules（version+1）
废弃已有规则用户废弃不再适用的规则domainRules.status = archived

注入内容：

  • 待确认的候选规则列表（ruleCandidates，按 validationRate 降序）
  • 已有规则冲突列表（domainRules WHERE edges.conflicts_with 非空）
  • 最近 model_inferred 规则摘要（最近N条，含来源环节信息）
  • 当前 domainRules 概览（按 topicPath/stepName 分组）

产出物：

  • 确认/修改/废弃的规则写入 domainRules → 注册到 outputRegistry
  • 对话过程记录到 roomConversationLog（v5.7 分段保留）
  • 原始对话备份到 conversationArchive（v5.8，供搜索）

环节宪法要点（N9 编写时的指导）：

<!-- @constitution domain-rule-session -->
<!-- @section taskType -->
topic_based
<!-- @section role -->
你是领域规则讨论助手。用户在这里集中管理领域规则——确认候选规则、主动声明新规则、仲裁规则冲突、修改或废弃已有规则。
<!-- @section topic-boundary -->

<!-- @section tools -->

<!-- @importance -->
critical

与其他房间的数据流：

其他房间的 IN_SESSION complete
  → 模型推断规则（model_inferred）→ ruleCandidates
  → 积累到 rule_confirmation_threshold 条
  → 系统提示用户进入 domain-rule-session
  → 用户集中确认/修改/拒绝
  → 确认的规则写入 domainRules
  → 后续其他房间注入已生效规则（按 topicPath/stepName 子图匹配）

N1 声明：场景在 N1 边界清单中声明是否启用领域规则讨论房间。建议默认启用——domainRuleSessionEnabled: true。

风险条目：changeLevel 防偷懒​

风险描述：模型可能偷懒，将所有变更标记为 major（保证不被压缩）或跳过 changeLevel 判定。更隐蔽的偷懒形态是"认真编一个 changeLevelReason 理由但仍然偷懒"——DET 格式校验能拦空值/超长，但挡不住"看起来合理实际仍是偷懒"的理由。

缓解措施：

  • 根宪法第4条已要求 changeLevelReason 理由字段（见 §7.2）
  • DET 格式校验拦截 changeLevelReason 空值/超长
  • 监控指标：changeLevel major 占比阈值从 <1% 调整为 <3%，changeLevelReason 空值率检测
  • 若 major 占比超阈值或空值率异常，触发冷启动应急解冻机制（见 §10）审查参数
  • 抽样人工复核机制：按比例抽检 major 标记的合理性，作为 DET 格式校验之上的第二道防线
    • 默认抽检比例：5%（通过 tunable 参数 changelevel_major_sample_rate 配置，取值范围 [0.01, 0.20]，默认 0.05）
    • 抽检对象：所有被模型标记为 major 的 changeLevel 事件
    • 复核流程：抽样事件推入人工复核队列 → 复核员判断"该变更是否真的属于结构性变更（新增/删除必采字段、修改字段值域、改变 Schema 结构）" → 标记为 confirmed_major / false_positive
    • 反馈闭环：false_positive 比例 > 30% 时，触发冷启动应急解冻机制（见 §10）审查 logprobs_threshold 与 changeLevel 判定 prompt
    • 与冷启动的关系：冷启动期间（cold_start_observation_window 内）抽检比例临时上调至 10%，冷启动结束后恢复默认 5%——冷启动期模型行为未稳定，需要更高密度的反馈信号
    • 复核结果记入变更日志：时间戳、抽检事件 ID、复核员、结论、复核理由

C.4.1 房间比喻四层结构​

房间（环节，N1-N16）—— 环节宪法（房间级，共享）
  └── 项目（智能体实例，如富贵小安/杂碎本）—— 产出物内容（项目级，隔离）
       └── 格子（细则文件，精简总纲+细则）—— 正文内容（文件级，独立）
            └── 变更日志（独立文件）—— 所有变更追溯（全局统一）

项目隔离机制（机制1）见 §三 核心机制。本节规范机制2（分格）和机制3（变更日志独立）。

C.4.2 机制2：分格机制​

解决什么​

单个产出物文件太长（如态控架构总纲 1832 行），每次读取/修改算力消耗大，LLM 修改长文件容易出错（Lost in the Middle + token 消耗）。

分格策略​

产出物：态控架构总纲（1832行，每次全读）
  ↓ 分格
格子1：精简总纲（~200行，每次必读——章节索引+核心定义+速查表）
格子2：§一 架构背景（独立文件）
格子3：§二 设计原则（独立文件）
...
格子12：附录（独立文件）

与认知局部性的关系​

分格机制是认知局部性在文件层的落地——之前认知局部性管的是"prompt 里放什么"，现在扩展到"读取哪些文件"：

层之前分格后
prompt 内只放最小充分条件集不变
文件读取全读总纲只读精简总纲 + 当前需要的细则
上下文窗口被 1832 行总纲占满精简总纲 200 行 + 需要的细则

分格纪律​

  1. 精简总纲只含索引+速查——不含正文详情
  2. 细则文件按章节拆分——每个独立，200-500行
  3. 修改时只改对应细则——不碰其他文件
  4. 需要详情时按需读取——不一次性全读

C.4.3 机制3：变更日志独立机制​

解决什么​

修改文件时 LLM 容易偏离用户意图（误改），且变更难以追溯。变更记录和正文混在一起，难审计。

变更日志独立​

之前：
  v5.5-全量版.md 里混杂着：
    - 正文内容
    - 变更记录（v5.1新增.../v5.2修正.../v5.3...）
    - 版本演进表

问题：
  修改正文时，LLM 可能误改变更记录
  变更记录和正文混在一起，难审计
  变更记录本身也占大量 token

分格后：
  02-12 细则文件 —— 只含正文，不含任何变更过程
  态控架构变更日志.md —— 所有章节的变更统一记录在这里

变更日志格式​

## v5.6（2026-07-18）

### §4.1 节点意图收敛链（05态控架构节点体系-v5.5.md）
- 新增：XXX
- 修改：XXX

### 附录E 编程规则宪法（12态控架构附录-v5.6.md）
- 新增：第7条
- 修改：第1条

变更日志纪律​

  1. 所有变更统一记录在变更日志.md，细则文件中不写任何变更过程
  2. 按版本倒序排列（最新在最前）
  3. 每个版本按章节列出变更（新增/修改/删除）
  4. 标注变更所在的细则文件
  5. 重大变更需说明变更原因

与审计的关系​

维度变更日志独立前变更日志独立后
变更追溯散落在各文件头部/尾部统一在变更日志
误改检测难——要对比整个文件易——对比变更日志即可
LLM 修改风险高——可能误改变更记录低——正文不含变更记录
审计效率低——要翻多个文件高——一个文件看全部变更

C.4.4 文件管理纪律（全局）​

修改基线时的纪律​

  1. 修改基线时另存新版本——不直接覆盖原文件（v5.3 被损坏事件的教训）
  2. 细则文件只含正文——不含变更记录（变更记录在变更日志.md）
  3. 总纲文件是概要索引——每次读取只读总纲，需要详情时读对应细则
  4. 大改动分批进行——每次只改一个细则文件

读取基线时的纪律​

  1. 默认只读概要总纲——了解架构全貌
  2. 需要详情时读对应细则——按章节索引定位
  3. 需要变更历史时读变更日志——不翻细则文件
  4. 需要 SQL DDL 时读附录C.3——DDL 唯一来源
