## 一、机制说明

### 2.1 taskType 双模式机制说明

**区分原点：收敛空间的性质，而非任务内容。** taskType 是态控架构所有设计决策的逻辑起点——上下文拼接方式、校验方式、记忆载体、规则匹配键，全部收敛到这一个二分法上。

**核心判据：是否存在讨论空间**（v5.8 明确）——field_based 无讨论空间（填表式收敛），topic_based 有讨论空间（对话式收敛）。判定口诀：用户说了之后，模型有没有讨论空间？

每个 intent（意图） 在 N1（第一步） 边界划定阶段就确定了 taskType，这个决定直接影响后续所有节点的行为：

| 维度 | field_based（基于字段）——无讨论空间 | topic_based（基于主题）——有讨论空间 |
|------|----------------------------------|---------------------------------------|
| **讨论空间**（v5.8） | **无**——用户说啥就是啥 | **有**——模型主动追问、提供选项 |
| 收敛方式 | 填表式——按预定义字段采集 | 讨论式——在主题边界内自由探索 |
| 典型场景 | 记账、改柜体参数、表单填写、查询、删除 | 角色扮演、方案设计、头脑风暴、**设计引导** |
| askingField 语义 | 字段名（如 "amount"、"time"） | 主题标签（如 "character_motivation"） |
| 上下文管理器 锚点 | 字段级硬门控（DET 确定性匹配） | 主题级硬门控 + 三层注入 |
| 环节宪法写法 | 字段采集规则 + 值域校验 | 主题边界 + 输出格式约束 |
| DET 值域复验 | 值域校验 | 输出格式校验 |
| 模型选择倾向 | 通用模型即可 | 倾向深度模型 |
| 跨节点传递载体 | contract-out JSON + sessionCheckpoint | topicEvolution + 领域规则 |
| 长期记忆载体 | sessionCheckpoints | topicEvolution + topicIndex + domainRules |

**两种模式的统一性：** 尽管 taskType 有两种模式，但 turnType 六值（ask / reply / complete / off-task / giveup / validation_failed）是统一的，态控调度器用同一套逻辑处理所有环节的模型返回。差异仅在 askingField 的语义解释和 上下文管理器 的锚点模式上——这是"框架统一、分支差异"的设计原则。

### 2.2 topicEvolution 机制总览

topicEvolution 是 topic_based 任务的标配跨节点传递载体。

| 层级 | 机制 | 解决的问题 | 核心操作 | 用户确认 | 执行方式 |
|------|------|-----------|---------|---------|---------|
| 第一层：分层留存 | topicEvolution 按 changeLevel（major/minor/invalid）分层留存 | 单个主题内的信息管理——不膨胀、不丢失 | 模型（LLM）在 turnType 中顺带输出 changeLevel 预判字段，态控调度器直接使用；major 永久保留，minor 汇总压缩，invalid 迁入归档表 | 间接（通过 turnType 决定 major） | 同步（模型返回 turnType 时顺带完成，不额外增加 LLM 调用） |
| 第二层：主题关联 | topicIndex 中建立主题间的关联关系（prerequisite/reference/derivative/conflict/cooperative/replacement/variant/evolution） | 多主题间的知识引用——继承优点、避开雷区 | 系统自动发现或用户手动建立，用户确认后生效 | 必须（关联类型和摘要内容均由用户确认） | 异步（不阻塞节点流转） |
| 第三层：规则提炼 | 从多条同类主题的 topicEvolution 和关联关系中提炼候选规则，经交叉验证和用户确认后存入 domainRules 库 | 从多主题实践中抽象可复用知识——越用越老练 | 异步批量触发（基于时间窗口或数量阈值），提炼引擎作为独立后台进程运行；候选规则经交叉验证后提交用户确认 | 必须（每条候选规则均需用户确认、修改或拒绝） | 异步（完全脱离状态机主链路） |

**三层关系：** 分层留存是地基——没有分层，主题演化数据要么膨胀要么丢失；主题关联是桥梁——没有关联，多个主题是孤立的，知识无法传递；异步规则提炼是塔尖——没有提炼，智能体永远是从零开始的菜鸟。地基和桥梁在状态机内完成，塔尖在状态机外异步完成。

**规则提炼从同步触发改为异步批量触发：** 在节点 exit 时同步触发规则提炼的设计存在状态机阻塞风险。v7.0 修正为基于时间窗口或数量阈值的异步批量提炼。态控调度器在节点 exit 时只落盘 topicEvolution event 并放行，不触发任何提炼逻辑。这引入了"知识真空期"——领域规则不保证跨相邻任务的即时传递，仅提供跨中长期任务的累积经验。

### 2.2b sessionCheckpoint 机制总览（v7.2 新增）

sessionCheckpoint 是 field_based 任务的标配跨节点传递载体——与 topicEvolution 在 topic_based 中的地位对等。

```sql
sessionCheckpoints(userId, lastCompletedStep, completedSteps[], stepSnapshots{})
```

| 字段 | 含义 | 谁写 |
|------|------|------|
| lastCompletedStep | 最后一个完成的环节名 | 每 complete → 态控调度器更新 |
| completedSteps | JSON 数组，已完成环节列表 | 同上 |
| stepSnapshots | JSON 对象，每环节收敛时的结构化字段快照 | 同上 |

**写入时机**：环节 complete 时追加进度 + 写入快照；回到已完成环节重做时覆盖快照；ttl 到期仅清理 sessionCheckpoints 进度行，不联动清理 roomConversationLog（roomConversationLog 按 importance 分级独立保留：永久/90天/30天/7天）。

**读取路径**：ANALYZING 恢复时态控调度器检查→决定继续还是从 P0 开始；上下文拼接时读 stepSnapshots[环节名] 的 collectedFields。

**sessionCheckpoint 与 topicEvolution 的对位**：

| 维度 | topicEvolution（topic_based） | sessionCheckpoint（field_based） |
|------|----------------------|----------------------|
| 载体形式 | 事件流：每节点 exit 追加 event | 进度标记：每节点 complete 更新一行 |
| 匹配方式 | topicId embedding 语义检索 | userId DET 精确匹配 |
| 记录粒度 | 事件级 | 环节级 |
| 注入给模型 | 三层注入 | 结构化字段 collectedFields |

### 2.2c contractEvolution 机制总览（v7.2 新增）

contractEvolution 是 field_based 的领域规则演化——与 topic_based 的异步规则提炼对位。两者共享同一套引擎（changeLevel→ruleMiningQueue→异步提炼→domainRules），唯一差异在匹配键：topic_based 用 topicPath 语义检索，field_based 用 stepName DET 精确匹配。

**changeLevel 判定规则（field_based）**：

| 级别 | 判定标准 | 留存 |
|------|---------|------|
| major | 字段的结构性变更——新增/删除必采字段、修改字段值域、改变 Schema 结构 | 永久保留，不可压缩 |
| minor | 字段值微调——默认值修改、取值范围调整、措辞优化 | 30 条压缩为 1 条摘要 |
| invalid | 完全撤销——用户改了又改回原值（与上一版 contract-out 完全相同） | 归档后定期删除（90 天） |

**存储与注入**（复用 contractStore，按 stepName 隔离）：

| 表 | field_based 行为 |
|----|----------------|
| domainRules | ruleId 含 stepName 前缀，按 stepName DET 精确匹配注入 |
| ruleCandidates | 每个 IN_SESSION complete 时模型推断的用户偏好 |
| ruleMiningQueue | 按 stepName 分队列，异步提炼按 stepName 分组处理 |

**与 topic_based 规则提炼的核心对应**：topic_based 的 changeLevel 判定基于用户交互中的明确行为信号（如版本号变化），field_based 基于字段的结构性变更/微调/撤销——判定标准不同，但分层留存策略完全相同。详见总纲 v5.5 §3.7.2.2。


### 2.2d 产出物总索引——Output Registry（v5.3）

outputRegistry 是全系统房间产出物的统一登记簿。与 roomStateIndex（房间状态摘要）互补——roomStateIndex 回答"房间有什么待办"，outputRegistry 回答"房间产出了什么"。详见 v5.5 §3.1.3。

**关键机制**：
- **存储**：contractStore.outputRegistry 表（DDL 见 v5.6 附录 C.3），含 outputId/roomId/outputType/outputPath/outputName/outputSummary/keywords
- **检索**：DET 确定性查询（0 LLM），支持按房间/类型/关键词/时间组合检索
- **注册门槛**：topic_based complete 自动注册；field_based 仅用户命名/导出的产出注册（每轮 complete 裸字段集不注册，避免噪声）
- **跨房间发现**：ANALYZING 语义判断→调度器 DET 精确查询→用户确认→文件复制到目标房间 references/ 目录
- **自由切换第四支柱**：产出物总索引与 roomConversationLog（全量记录）、stepSnapshot（对话快照）、规则提炼按环节隔离并列，构成自由切换的四个支柱


### 2.3 第一层：topicEvolution 分层留存机制

#### 2.3.1 topicEvolution 的定义

topicEvolution（主题演化表）是 topic_based 任务跨节点传递的标配载体，它记录了从主题创建到当前时刻的所有关键状态变更。其核心结构：

```json
{
"topicId": "user123_roleplay_betrayal_mystery_20260708",
"topicName": "背叛与悬疑：失忆侦探",
"keywords": ["背叛", "悬疑", "侦探", "失忆"],
"intent": "roleplay",
"creatorUserId": "user123",
"createdAt": "2026-07-08T10:00:00Z",
"updatedAt": "2026-07-08T11:30:00Z",
"stateSnapshot": {
"status": "active | checkpoint | abandoned",
"summary": "当前主题状态的简要描述",
"keyDecisions": ["已确定角色A的动机是复仇", "已排除角色B是幕后黑手的可能性"],
"pendingExplorations": ["角色C的真实身份尚未揭示"]
},
"relatedTopics": [],
"events": [
{
"eventId": "evt_001",
"changeLevel": "major",
"nodeId": "ANALYZING",
"turnType": "complete",
"timestamp": "2026-07-08T10:00:00Z",
"stateChange": {
"before": "主题初始化",
"after": "识别为角色扮演任务，主题为'背叛与悬疑'，角色A为失忆侦探",
"delta": "创建主题，设定初始角色和基调"
},
"askingField": "theme_selection",
"modelOutputSummary": "用户想要一个关于背叛的悬疑故事，主角是失忆侦探"
}
]
}
```

**字段说明：**

- **topicId：** 全局唯一标识，格式为 `{userId}_{intent}_{topicSlug}_{date}`，由 ANALYZING 阶段 LLM 生成的 topicSlug + userId + intent + date 组合而成，支持同一用户在同一 intent 下有多个并行主题
- **topicName：** 简短标题，由 ANALYZING 阶段 LLM 自动生成，经用户确认后生效
- **keywords：** 3-5 个关键词，由 ANALYZING 阶段 LLM 自动生成，经用户确认后生效
- **stateSnapshot：** 当前主题状态快照，包含状态标记（active/checkpoint/abandoned）、关键决策摘要、待探索方向
- **relatedTopics：** 关联主题列表（详见 §2.4）
- **events：** 事件序列数组，每个节点在 exit 时由态控调度器自动追加一条 event

event 结构：

```json
{
"eventId": "evt_001",
"changeLevel": "major | minor | invalid",
"nodeId": "ANALYZING | IN_SESSION(roleplay) | EXECUTING",
"turnType": "complete | giveup | off-task",
"timestamp": "ISO 8601",
"stateChange": {
"before": "上一状态摘要",
"after": "当前状态摘要",
"delta": "本轮新增/修改/排除的内容"
},
"askingField": "本轮的主题锚点标签",
"modelOutputSummary": "模型产出的关键内容摘要（非全文）"
}
```

#### 2.3.2 changeLevel 分层留存机制

不是按时间新久留，是按信息净价值留。每个 event 在追加时被判定为 major/minor/invalid 三个层级：

| 层级 | 定义 | 示例 | 处理方式 |
|------|------|------|---------|
| major（大版本） | 影响核心走向的变更 | 新增角色、主线转折、世界观设定、topicEvolution 机制引入 | 永久保留，逐条可查，不压缩、不丢弃 |
| minor（小版本） | 对已有内容的改进调整 | 性格微调、情节润色、对话优化、tunable 参数新增 | 汇总压缩成一条摘要，超过 topic_evolution_minor_compaction_threshold（默认 5）条时触发，原事件迁入 topicEvolutionArchive 表 |
| invalid（无效过程） | 试了又放弃的试探 | 先加后删的设定、尝试又撤回的风格、短时间内对同一字段反复修改且最终回退 | 不入主表，迁入 topicEvolutionArchive 归档表，记录"因模型判定为互斥撤销而未入主表"的元数据 |

**判定时机与执行体解耦（防阻塞设计）：**

- 不能在节点 exit 的同步链路中插入一次独立的 LLM 判定调用
- 环节内模型在返回 turnType 和 collectedFields 时，顺带输出 changeLevel 预判字段——模型已经在推理过程中理解了本轮的语义变化，不需要额外的 LLM 调用
- 调度器收到后，直接使用该字段进行 event 追加

**模型返回的 turnType 结构更新：**

```json
{
"turnType": "complete",
"askingField": null,
"changeLevel": "major | minor | invalid",
"message": "...",
"collectedFields": {},
"result": {}
}
```

**兜底机制：**

- 若模型返回了合法的 changeLevel 值 → 态控调度器直接使用
- 若模型未返回该字段或格式错误 → 态控调度器默认标记为 minor（最安全的容错：进入累积压缩流，不丢失信息）

**判定标准的操作化定义（防模型偷懒与歧义）：**

模型不能凭模糊的"感觉"判定，必须在局部宪法的 @section output-schema 中注入明确的判定规则：

- **invalid 的严格定义：** 当且仅当本轮输出完全撤销了上一轮的实质性变更（如上一轮新增实体A，本轮删除实体A且无其他衍生变更），且未产生新的有效信息时，判定为 invalid
- **major 的严格定义：** 涉及核心实体的增删、任务目标的根本性转向、或引入了全新的世界观/规则设定时
- **minor 的兜底：** 凡不属于上述两者的情况，一律判定为 minor

**invalid 丢弃机制的逻辑闭环：** 当模型判定为 invalid 时，态控调度器不将该 event 写入 topicEvolution 主表。但这并不意味着"无痕消失"——为了满足全流程可审计性，态控调度器应将这条被标记为 invalid 的 event 写入 topicEvolutionArchive，并记录"因模型判定为互斥撤销而未入主表"的元数据。这确保了记忆不膨胀，同时审计有迹可循。

**判定边界的重要区分：** 用户反复修改同一属性（如"改林霜的性格，改了三次才满意"），只要方向不是完全互斥（加→删），就算 minor 累积调整，压缩后变成"林霜性格经过多轮调整为外冷内热"，不应该判定为 invalid。判定标准是**本轮操作是否实质上撤销了上一轮的变更且未产生新的有效信息**，而非修改次数。

**minor 压缩机制：** 当 minor 级 event 连续累计达到 topic_evolution_minor_compaction_threshold（默认 5 条，tunable 可配置）时，态控调度器触发一次汇总压缩：

- 调用轻量 LLM 将 N 条 minor event 的 delta 合并为 1 条 compressed_evt
- 原 N 条 minor event 从 topicEvolution 主表移除，迁入 topicEvolutionArchive 表
- compressed_evt 的 changeLevel 仍标记为 minor，modelOutputSummary 为合并后的摘要

压缩示例：

```
minor 汇总前：
evt_011: 林霜性格从冷漠改为外冷内热
evt_012: 林霜出场从第三章调到第五章
evt_013: 林霜的武器从剑改为扇子
evt_014: 林霜的背景故事从孤儿改为世家
evt_015: 林霜的对话风格从古风改为口语

minor 汇总后（1 条 compressed_evt）：
"林霜角色调整：性格外冷内热，出场第五章，武器扇子，背景世家，对话风格口语"
```

**event 的 stateChange 生成机制：** stateChange 摘要压缩在 N10 后台异步进程执行，与 changeLevel 保持一致——均不在态控调度器的同步链路里调 LLM。态控调度器在节点 exit 时只落盘 topicEvolution event 原始数据（collectedFields + message + result），stateChange 摘要由 N10 后台异步消费 ruleMiningQueue 时生成。

#### 2.3.3 topicEvolution 注入时的分层结构

上下文管理器 在 topic_based 模式初始化时，从 contractStore 读取 topicEvolution 注入到上下文：

```
[topicEvolution 注入包]
├── majorEvents（永久保留，全部注入，通常 0-5 条，不压缩不丢弃）
├── compressedMinorSummary（最近一次 minor 压缩摘要，1 段）
├── recentMinorEvents（压缩后新增的 minor 事件，通常 0-4 条）
└── currentStateSnapshot（覆盖写，1 条）
```

这个分层注入结构确保了模型在每个节点开始时的上下文窗口不会因 topicEvolution 的膨胀而无限增长，同时 major 的关键决策永远可查。

#### 2.3.4 topicEvolution 的"完成"与"放弃"语义

topicEvolution 的 stateSnapshot 有三个状态：

- **active：** 讨论中，主题仍在进行
- **checkpoint：** 完成了一个输出（如写完一首诗），但主题仍可继续。标记 checkpoint 时，态控调度器将主题信息推入规则提炼的待处理队列
- **abandoned：** 用户放弃。标记 abandoned 时，态控调度器同样将主题信息推入规则提炼的待处理队列

当模型返回 turnType="complete" 时，态控调度器将 stateSnapshot 标记为 checkpoint。当模型返回 turnType="giveup" 时，标记为 abandoned。

#### 2.3.5 topicEvolution 的自动落地机制

在状态枚举 + taskType 固化阶段（N3），当 taskType 被固化为 topic_based 时，态控调度器自动执行：

1. **初始化：** 在 contractStore 中初始化一条 topicEvolution 记录，topicId 由 ANALYZING 阶段 LLM 生成的 topicSlug + userId + intent 组合而成
2. **写入：** 在 上下文管理器 的 topic_based 模式中，强制开启 topicEvolution 写入（不需要 N9 声明，因为 taskType 本身就是声明）
3. **追加：** 每个涉及该 topic 的节点（ANALYZING、IN_SESSION、EXECUTING）在 exit 时，态控调度器自动追加一条 event 到 topicEvolution——不需要节点开发者手动写代码，态控调度器在收 turnType 时自动完成，并使用模型顺带输出的 changeLevel 预判字段

#### 2.3.6 topicEvolution 的跨会话匹配

topicEvolution 的 topicId 需要一个跨会话的匹配机制：

- 用户说"继续上次那个故事" → ANALYZING 识别出 topicId（从 contractStore 里按用户 + 主题相似度匹配）
- 匹配到已有的 topicEvolution → 注入最新 stateSnapshot + 分层注入包，模型接着走
- 匹配不到 → 新建 topicEvolution，从零开始
- 匹配逻辑放在 topicIndex 的检索中——先用 DET 关键词匹配（基于 topicIndex 表的 topicName 和 keywords），关键词匹配不到时降级为 embedding 语义匹配
- 匹配阈值由 tunable 参数 topic_evolution_match_threshold 控制（默认 0.65，略高于 topic_anchor_similarity_threshold 的默认 0.6，因为跨会话匹配需要更高精度）

#### 2.3.7 topicEvolution 的存储位置

topicEvolution 存入 contractStore，而不是 sessionStore。因为 topicEvolution 是跨会话持久化的长期记忆载体，它的生命周期与 contractStore 一致——Agent 消亡后，主题演化记录仍可被后续会话访问。物理存储用 SQLite + WAL 模式（与现有 contractStore 一致），包含四张表：

- **topicEvolution：** 记录 topicId、stateSnapshot、创建时间、更新时间，只存储 major 级 event + compressedMinor + 压缩后新增的 minor 级 event
- **topicEvolutionEvent：** event 主表，按 topicId 外键关联
- **topicEvolutionArchive：** event 归档表，存储被压缩的 minor 级原始 event + 被判定为 invalid 的 event
- **topicIndex：** 检索索引表（详见 §2.6）

#### 2.3.8 topicEvolution 与认知局部性的关系

topicEvolution 不是对认知局部性的违反，而是对它的正确扩展。

field_based 的 contract-out JSON 同样是跨节点共享的——节点之间不共享的是"推理过程"，但共享"收敛结果"。topicEvolution 的 event 序列就是 topic_based 的"收敛结果"——它记录的不是模型的推理过程，而是"主题在这个节点里完成了什么变化"这一确定性的状态变更。

节点之间的数据传递，不再仅限于"上一个节点的结构化输出"，而是"这个主题从开始到现在的演化历史摘要（按信息净价值分层）"。这仍然是"数据流为中心"——只是数据流的粒度从"节点级"提升到了"主题级"。

该原则同样适用于第二层（主题关联）和第三层（规则提炼）：关联摘要不是"把隔壁主题的数据拿过来"，而是"从隔壁主题的数据里提炼出对当前主题有用的几条结论"；领域规则不是硬约束，而是软参考。关联主题的原始 evt 链不属于当前主题的最小充分条件集，不注入。

#### 2.3.9 分层留存与文档自身演进的对应

这套分层逻辑同时解决了"态控闭环设计流程指南"自身演变过程数据的管理问题：

| 层级 | 文档演进示例 | 处理 |
|------|------------|------|
| major | v3.2→v4.0（引入 taskType、topicEvolution） | 永久保留 |
| minor | v4.0→v4.1（新增 topic_evolution_match_threshold） | 汇总压缩："v4.1 新增了跨会话匹配阈值参数" |
| invalid | "试了用 Redis 存 contractStore，后来改 SQLite" | 直接丢弃，不留 |

### 2.4 第二层：主题关联机制

**核心原则：** 关联在元数据层，隔离在数据层——模型只知道"隔壁主题给了我两条结论"，不知道"隔壁主题的50条聊天记录"。关联不是混同，关联是轻量级元数据引用。

#### 2.4.1 关联类型定义

| 关联类型 | 含义 | 示例 | 注入内容 |
|---------|------|------|---------|
| prerequisite（前提） | A是B的前提基础 | 专利A（网关）→ 专利B（区块链存证） | "A的核心创新点是协议转换效率提升30%" |
| reference（引用） | B引用了A的部分内容 | 专利C引用了专利A的网关架构 | "A的网关架构为YYY" |
| derivative（派生） | B是从A派生出来的 | 专利B是专利A的应用场景扩展 | "B是在A基础上的特殊化应用" |
| conflict（冲突） | A与B的方案互斥 | 专利A和专利D对同一问题给出不同解法 | "A和D的解法不同：A采用X，D采用Y" |
| cooperative（协同） | A与B共同构成完整方案 | 专利A负责硬件，专利E负责软件 | "A和E共同构成完整系统" |
| replacement（替代） | B替代了A，因为A被否定 | 现代风格方案→新中式风格方案 | "A被否定原因；B要继承A的哪些优点；B要避免A的哪些缺点" |
| variant（变体） | B是A的另一种版本，并存可选 | 方案A（预算版）vs 方案B（豪华版） | "A和B的差异点；A和B的共同基础" |
| evolution（演进） | B是A的迭代升级 | v1方案→v2方案（基于同一客户反馈优化） | "v1的反馈要点；v2的优化方向" |

#### 2.4.2 关联的建立方式

- **手动建立：** 用户在对话中说"这个专利是基于之前那个边缘计算网关的"，系统提示"检测到可能与「边缘计算网关」主题相关，是否建立关联？"，用户确认后生效
- **自动发现：** ANALYZING 阶段，LLM 检测到当前输入与已有主题的关键词重叠度高，提示"检测到可能与「XXX」主题相关，是否查看或建立关联？"
- **关联确认：** 所有关联关系（类型、摘要内容）必须经过用户确认后才能生效——用户可修改关联类型、编辑摘要、或拒绝建立关联

#### 2.4.3 关联数据的存储与注入

关联关系存储在 topicIndex 表的 relatedTopics 字段中，是轻量级元数据。注入时，上下文管理器 只拉取关联摘要卡片，不拉取关联主题的原始 evt 链：

```json
{
"currentTopic": {
"topicId": "scheme_B",
"topicName": "新中式风格全屋方案"
},
"relatedTopics": [
{
"topicId": "scheme_A",
"topicName": "现代风格全屋方案",
"relationType": "replacement",
"reason": "客户反馈现代风格色调太冷，希望更温馨",
"inheritedPoints": ["厨房动线设计获得客户认可", "主卧衣帽间尺寸已确认"],
"avoidPoints": ["电视墙石材材质被客户否定", "全屋无主灯设计客户不习惯"]
}
]
}
```

模型拿到这个，写B的时候就知道：要继承厨房动线和主卧尺寸，要避免石材电视墙和无主灯，要回应"太冷"的问题用暖木色系。

#### 2.4.4 关联与认知局部性的关系

关联摘要不是"把隔壁主题的数据拿过来"，而是"从隔壁主题的数据里提炼出对当前主题有用的几条结论"。关联主题的原始 evt 链不属于当前主题的最小充分条件集，不注入。关联摘要 = 隔壁主题的"毕业证书"，不是隔壁主题的"日记本"。

### 2.5 第三层：异步规则提炼机制

**核心原则：** 规则提炼不从原文挖掘，而是从 topicEvolution 的演变流程和主题关联的 inheritedPoints/avoidPoints 中提取"被多次验证的行为模式"。候选规则必须经过用户确认后才生效。规则提炼完全异步，不在状态机流转链路内执行。

#### 2.5.1 提炼的输入源

| 输入源 | 内容 | 示例 |
|-------|------|------|
| 子主题的演变流程（topicEvolution） | 每次 stateChange 的 delta | "厨房动线从'随意'改为'洗切炒'" |
| 主题间的关联机制 | 关联关系中的 inheritedPoints 和 avoidPoints | "现代风格方案的厨房动线被继承到新中式方案" |

系统不做全文挖掘，只从这两个结构化数据源中提取"被多次验证的行为模式"。

#### 2.5.2 提炼触发机制

**触发方式（满足任一条件即触发）：**

- **时间窗口触发：** 每日固定时间（如凌晨2点）统一提炼。由 rule_mining_schedule tunable 配置（cron 表达式，默认 `0 2 * * *`）
- **数量阈值触发：** 每积累 N 个同类主题的 checkpoint 触发一次提炼。由 rule_mining_batch_threshold tunable 配置（默认 5）
- **手动触发：** 用户在系统设置界面的"知识库更新状态"面板中点击"立即提炼"

**态控调度器在节点 exit 时只做：** 落盘 topicEvolution event → 标记主题状态（checkpoint/abandoned）→ 放行。不触发任何提炼逻辑。

**提炼引擎作为独立的后台进程运行：**

- 从 contractStore 中读取待提炼的主题列表
- 每次批量提炼的最大主题数由 rule_mining_batch_size tunable 控制（默认 10）
- 批量处理，写入 ruleCandidates 池
- 等待交叉验证通过后提交用户确认

**知识真空期的用户约定：** 在系统设置界面增加"知识库更新状态"指示器，显示：

- "上次提炼时间：2026-07-08 02:00"
- "待提炼主题数：3"
- "下次计划提炼时间：2026-07-09 02:00"
- 提供"立即提炼"手动触发按钮

这样用户知道"我的经验正在被后台消化"，不会误以为每次任务结束系统就立刻变聪明了。

#### 2.5.3 提炼与确认流程

提炼引擎触发后：

1. **候选规则生成：** 提炼引擎读取待提炼主题的 topicEvolution 和关联关系，提取"稳定的、可复用的行为模式"，生成候选规则，存入 ruleCandidate 池
2. **交叉验证：** 候选规则需要多条同类主题交叉验证后才提交用户确认。默认阈值为 70%（由 domain_rule_validation_threshold tunable 控制，默认 0.7）。例如：5 套方案中 4 套都验证了"厨房应遵循洗切炒动线"→ 验证通过率 80% → 提交用户确认
3. **用户确认：** 系统展示候选规则及证据（"基于8套方案中7套的演变记录"），用户可采纳、修改或拒绝。确认后的规则存入 contractStore.domainRules 库
4. **规则注入：** 同类型新主题初始化时，上下文管理器 自动注入已生效的相关领域规则

规则确认示例：

```
系统："检测到一个可能的通用规则：
厨房布局应遵循洗切炒动线
（基于5套方案中4套的演变记录）
是否采纳为通用规则？"
用户：[采纳] → 存入 domainRule 库，标记为 strong
用户：[修改] → "改成'厨房布局应遵循洗切炒动线，U型布局优先'"
用户：[拒绝] → "这只是巧合，我家厨房小不适合U型"
```

#### 2.5.4 规则的注入

当新的同类型主题初始化时，上下文管理器 除了注入 topicEvolution，还会注入该领域的已生效 domainRule：

```json
{
"domainRules": [
{ "rule": "厨房布局应遵循洗切炒动线，U型布局优先", "source": "已验证于8套方案", "level": "strong", "confirmedBy": "user_001" },
{ "rule": "衣柜应遵循挂叠分区，挂区占比不低于60%", "source": "已验证于5套方案", "level": "medium", "confirmedBy": "user_001" },
{ "rule": "主卧应预留婴儿床位置", "source": "已验证于3套方案", "level": "suggestion", "confirmedBy": "user_001" }
]
}
```

领域规则不是硬约束（不由 DET 强制执行），而是软参考——模型可基于客户个性化需求合理偏离。

#### 2.5.5 规则提炼与主题关联的区别

| 维度 | 主题关联 | 异步规则提炼 |
|------|---------|------------|
| 作用范围 | 两个主题之间 | 一类主题之上 |
| 数据来源 | 单条主题的 topicEvolution | 多条同类主题的交叉验证 |
| 产出 | 关联关系（A是B的前提） | 通用规则（所有厨房都应遵循洗切炒） |
| 注入方式 | 当前主题初始化时按需注入 | 同类型主题初始化时自动注入 |
| 时效性 | 即时（关联建立后立即生效） | 滞后（异步批量提炼，存在知识真空期） |

#### 2.5.6 领域规则与宪法体系的关系

| 层级 | 来源 | 是否可修改 | 确认方式 |
|------|------|-----------|---------|
| 根宪法 | 设计者写死 | 不可修改 | 无（设计时定） |
| 架构机制 | 设计者编写 | 走回归测试修改 | 无（设计时定） |
| 局部宪法 | 设计者编写或动态生成 | 可修改 | 低置信度时用户确认 |
| 领域规则 | 从 topicEvolution + 关联机制提炼 | 用户确认后生效，可随时修改或废弃 | 必须用户确认 |

领域规则是唯一一层由用户在生产过程中逐步建立的约束——它不是设计者拍脑袋定的，是从实际项目中长出来的。
#### 2.5.7 领域规则的来源与权限分层（v7.1 新增）

**规则来源三层：**

| 来源 | 含义 | 灌入时机 | 权限初始值 | 是否走交叉验证 |
|------|------|---------|-----------|-------------|
| **scenario_seed** | 场景固有规则（工艺规范、设计基线、报价模板、安全红线等） | 主题初始化时按 topicPath 加载 | L0 / L1 | ❌ 不走，设计者预设可信 |
| **user_initiated** | 主题确认时用户主动口述的规则 | ANALYZING 确认 topicName 后主动问询 | L2 | ✅ 走，用户可能说错或矛盾 |
| **model_inferred** | IN_SESSION 过程中模型推断的用户偏好 | complete 时顺带输出 → ruleCandidates | L3 | ✅ 需用户确认升格为 L2 |

**规则权限分层（与宪法层级正交）：**

| 层级 | 名称 | 含义 | 可修改 | 冲突时行为 |
|------|------|------|--------|----------|
| **L0** | 钢印规则 | 场景硬性约束，不可触碰 | ❌ 不可修改 | 无条件胜出，不询问 |
| **L1** | 基线规则 | 设计者预设最佳实践 | ⚠️ 需设计者确认 | 与 L2/L3 冲突时触发询问 |
| **L2** | 用户确认规则 | 用户主动说过或确认过的偏好 | ✅ 用户可随时修改 | 优先于 L3 |
| **L3** | 模型推断规则 | 模型 infer 的候选规则 | ✅ 需用户确认升格 | 仅作参考，不主动注入 |

与根宪法的关系：根宪法第 1 条（安全红线）在领域规则层映射为 L0 钢印规则。

#### 2.5.8 规则网络结构（v7.1 新增）

领域规则不再是平铺的列表，而是一个有向图——每条规则是一个节点，节点之间通过关联边建立关系。

**节点定义：**

```json
{
  "ruleId": "kitchen_countertop_height_001",
  "content": "厨房台面高度建议85cm",
  "topicPath": ["全屋定制", "设计", "厨房设计"],
  "source": "scenario_seed",
  "immutableLevel": "L1",
  "status": "active",
  "conditions": { "userHeight": ">=170cm" },
  "edges": {
    "conflicts_with": ["kitchen_countertop_height_002"],
    "depends_on": [],
    "strengthens": ["ergonomic_guideline_003"]
  },
  "version": 1
}
```

**关联边类型：**

| 边类型 | 含义 | 注入时行为 |
|--------|------|----------|
| **depends_on** | A 规则成立的前提是 B 规则已满足 | 匹配 A 时检查 B，若不满足则主动询问 |
| **conflicts_with** | A 和 B 互斥，不能同时应用 | 同时匹配到时触发询问让用户二选一 |
| **strengthens** | A 规则强化 B 规则的可信度 | 匹配到 A 时，B 的权重 +1（累加上限 +3），同层级内优先注入 |

**规则状态：** active（正式生效）/ candidate（待交叉验证）/ archived（已废弃）/ pending_review（L0/L1 修改请求）。

#### 2.5.9 主题层级与规则的作用域边界（v7.1 新增）

topicPath 是一个**层级化数组**，从粗到细逐级展开。每一层都可以有自己独立的规则集。

**层级结构示例：**

```
全屋定制
├── 设计
│   ├── 厨房设计     ← 规则："台面高度85cm""水槽位置""油烟机型号偏好"
│   ├── 卫生间设计   ← 规则："防水等级""瓷砖规格""干湿分离要求"
│   └── 客厅设计
├── 报价             ← 规则："利润率""套餐模板""折扣权限"
├── 客户沟通
├── 拆单
└── 生产
```

**前缀匹配规则**：上级可见于下级，下级不可见于上级/兄弟。层级越深，匹配到的规则集越小——认知局部性原则在规则注入层的自然体现。

| 当前 topicPath | 能匹配到哪些规则 | 不能匹配到 |
|--------------|---------------|----------|
| `["全屋定制"]` | topicPath 前缀为 `["全屋定制"]` 的所有规则——包括"设计""报价""生产"等全部子层规则 | — |
| `["全屋定制", "设计"]` | 前缀为 `["全屋定制", "设计"]` 的规则——"厨房设计""卫生间设计""客厅设计" | `["全屋定制", "报价"]` 的规则（兄弟节点不可见） |
| `["全屋定制", "设计", "厨房设计"]` | 仅 `["全屋定制", "设计", "厨房设计"]` 自身的规则 | "卫生间设计""报价"等的规则（隔层不可见） |
| `["全屋定制", "报价"]` | 仅 `["全屋定制", "报价"]` 自身的规则 | "厨房设计"等的规则 |

**核心纪律**：
- **上级可见于下级**：`["全屋定制", "设计"]` 层的规则对 `["全屋定制", "设计", "厨房设计"]` 可见——前者是后者的前缀。这意味着在"厨房设计"环节，模型能拿到"设计"层的通用规则（如"全屋风格统一为现代简约"）+ "厨房设计"层的专用规则（如"台面高度85cm"）
- **下级不可见于上级/兄弟**：厨房台面高度对报价毫无意义——`["全屋定制", "设计", "厨房设计"]` 的规则不会注入到 `["全屋定制", "设计", "卫生间设计"]` 或 `["全屋定制", "报价"]` 的上下文中
- **层级深度与上下文窗口的关系**：当前 topicPath 越深，匹配到的规则集越小

#### 2.5.10 规则注入匹配逻辑（v7.1 新增，上下文管理器 第0层改造）

**v7.0 原逻辑：** 全量读取 domainRules → 一股脑注入

**v7.1 新逻辑：** 按 topicPath 做子图匹配 → 精准注入

1. 获取当前 topicPath（如 ["全屋定制", "设计", "厨房设计"]）
2. 在 domainRules 中做子图匹配：匹配 topicPath 前缀一致的规则节点 + 沿 depends_on/strengthens 边外扩一层 + 检查 conditions
3. 冲突检测：若匹配到 conflicts_with 关系的节点 → 触发询问，按 immutableLevel 决定询问策略
4. 条件不满足检测：未满足的 conditions → 主动询问补充
5. 注入 prompt：按 L0>L1>L2>L3 → 同层级内按 importance（critical>high>normal>low）→ 同 importance 按 strengthens 权重排序（累加上限+3，v4.7）→ 同权重按 ruleId 字典序

#### 2.5.11 规则提炼队列处理（v7.1 新增）

```
checkpoint / abandoned
  ▼
推入 ruleMiningQueue
  ▼
N10 后台进程消费 → 压缩摘要 → 提取候选规则 → 推断关联边 → 写入 ruleCandidates
  ▼
等待用户确认
  ├── 确认 → 升格为 L2，写入 domainRules
  ├── 修改 → 按用户修改后写入，标记 L2
  └── 否决 → 丢弃或标记 rejected
```

**交叉验证策略按来源区分：** scenario_seed（L0/L1）不走交叉验证；user_initiated（L2）需交叉验证；model_inferred（L3）需交叉验证+用户确认升格。

**inferred_conflict 候选状态：** N10 自动推断的 conflicts_with 关联边标记为 `inferred_conflict`（候选状态），不参与运行时冲突检测。必须经用户确认升格为 `active_conflict` 后才生效。详见总纲 v5.5 §3.7.7。



### 2.6 topicIndex 索引表

为支持高效检索和主题关联，每个 topicEvolution 记录在初始化时同步写入 contractStore.topicIndex 表，作为检索索引。

#### 2.6.1 表结构

```sql
CREATE TABLE topicIndex (
topicId TEXT PRIMARY KEY,
topicName TEXT NOT NULL,           -- 简短标题（如"智能体态控架构专利交底书"）
keywords TEXT NOT NULL,            -- 3-5个关键词，逗号分隔
creatorUserId TEXT NOT NULL,
createdAt TEXT NOT NULL,
lastActiveAt TEXT NOT NULL,
relatedTopics TEXT DEFAULT '[]'    -- JSON数组，存储关联关系
);
```

#### 2.6.2 主题名 + 关键词的用户确认流程

**核心原则：** 主题名和关键词由 ANALYZING 阶段 LLM 自动生成，但必须经用户确认后方可生效。

**匹配到已有主题时：**

```
用户："继续写那个态控的专利"
↓
ANALYZING 匹配到已有 topicEvolution（关键词匹配命中）
↓
系统展示："检测到已有主题「智能体态控架构专利交底书」，是否继续？"
用户：[确认] → 直接续写
用户：[不对，是另一个] → 系统展示相近候选项让用户重选
用户：[不是，我要写一个新的] → 重新初始化
```

**未匹配到已有主题时（新建主题）：**

```
用户："帮我写一个关于边缘计算的专利"
↓
ANALYZING 关键词匹配不到已有记录
↓
LLM 生成候选 topicName + keywords
↓
系统展示：
"已为您新建主题：
主题名：边缘计算网关专利交底书
关键词：边缘计算、网关、协议转换、物联网
是否正确？"
用户：[确认] → 初始化 topicEvolution，进入 IN_SESSION
用户：[修改] → 用户手动编辑主题名或关键词
用户：[不对，我要找的是另一个] → 系统展示相近候选项
```

#### 2.6.3 检索优先级

1. **关键词匹配（DET）：** 用户输入含明确主题名或关键词 → DET 直接匹配 topicName + keywords + userId + 时间范围，0 LLM 成本，毫秒级响应
2. **embedding 语义匹配（LLM）：** 用户表述模糊（"继续上次那个"），关键词匹配不到时降级为 embedding 语义匹配，阈值由 topic_evolution_match_threshold（默认 0.65）控制

**匹配置信度与用户确认的关系：**

- 匹配命中且置信度 ≥ 0.65 → 直接展示匹配到的主题名，用户一键确认
- 匹配命中但置信度 0.5-0.65 → 展示相近候选项（最多3个），用户选择或否定
- 未命中（置信度 < 0.5）→ 展示新生成的候选主题名+关键词，用户确认、修改或手动搜索

### 2.7 上下文管理器 双模式锚点

上下文管理器 的 buildPromptContext() 接口根据当前环节的 taskType 执行不同的锚点策略：

**field_based 模式——字段级硬门控：**

- 保护逻辑：与当前 askingField 同名的历史轮次（即之前追问同一字段时的对话）不被淘汰
- 执行方式：确定性字符串匹配（DET 执行），askingField="amount" 时保护所有 turnHistory 中 askingField="amount" 的轮次
- 保护形态：至少保留为摘要或锚点，不因上下文窗口不足而移除
- 可靠性：高（确定性匹配，不依赖 LLM 判断）

**topic_based 模式——主题级硬门控：**

- 保护逻辑：与当前 askingField 主题标签语义相关的历史轮次不被淘汰
- **三层注入——** 在 DET 关键词粗筛之前，上下文管理器 按优先级注入：
  - 第0层：已生效的领域规则注入——**v7.1：从全量注入改为按 topicPath 子图匹配精准注入**（详见 §2.5.10）。从 contractStore.domainRules 做子图匹配（匹配 topicPath 前缀一致的规则节点 + 沿 depends_on/strengthens 边外扩一层 + 检查 conditions），按 L0>L1>L2>L3 优先级排序后注入。domainRules 中的数据由 N10 规则提炼的后台进程异步写入。如果 domainRules 为空（知识真空期），跳过第0层注入，不影响后续流程
- 第1层：关联主题摘要注入——从 topicIndex.relatedTopics 读取关联主题摘要卡片（inheritedPoints / avoidPoints / reason）
- 第2层：当前主题演化注入——从 contractStore.topicEvolution 读取分层注入包（majorEvents + compressedMinorSummary + recentMinorEvents + currentStateSnapshot）
- 注入优先级：领域规则 > 关联摘要 > topicEvolution 分层包 > turnHistory
- 执行方式（三层注入 + 两阶段检索，0 次 LLM 调用）：
- 阶段 0：领域规则 + 关联摘要 + topicEvolution 分层注入——确保模型在开始推理前，已经获得了当前领域已沉淀的最佳实践、关联主题的关键结论和当前主题的演化历史
- 阶段 1：DET 关键词粗筛（确定性，O(N)）——对 turnHistory 中每轮，DET 提取 askingField 主题标签的关键词（如 "character_motivation" 提取 "character"、"motivation"），与历史轮次的文本做关键词重叠匹配。重叠度 ≥ topic_keyword_overlap_threshold（tunable，默认 0.2）的轮次进入阶段 2
- 阶段 2：embedding 相似度精筛（embedding 模型调用，非 LLM 调用）——对阶段 1 粗筛通过的轮次，用 embedding 模型计算 askingField 主题标签与历史轮次文本的向量相似度（余弦相似度）。相似度 ≥ topic_anchor_similarity_threshold（tunable，默认 0.6）的轮次被保护
- DET 阈值裁决：DET 只做"相似度 ≥ topic_anchor_similarity_threshold"的最终数值比较，确定性裁决
- 保护形态：至少保留为摘要或锚点，不因上下文窗口不足而移除
- 可靠性：中高（embedding 模型可能漏判跨主题依赖，但比纯 LLM 判断更稳定、成本更低；风险等级：中——漏判后果是模型重复追问已讨论过的主题，不影响输出正确性，影响交互效率）
- 工程收益：LLM 调用从 O(turnHistory) 降到 0，embedding 调用从 O(turnHistory) 降到 O(粗筛通过数)；三层注入使模型在新主题启动时就能获得跨主题的知识积累

**两种模式的共同部分：** 综合匹配度评分（其余历史轮次按语义匹配度分配形态）、最低匹配度截断（min_match_score tunable）、偏离处理、（archiveGraph）、校验失败标记（updateGraph）——这些机制两种模式完全相同。

**上下文管理器 三个接口说明：**

| 接口 | 时机 | 职责 | 状态机传入 |
|------|------|------|----------|
| buildPromptContext() | 每轮发消息给模型前（读） | 按三层注入+硬门控+语义匹配+最低截断算出本轮应拼入的上下文片段 | contextGraph, turnHistory, currentAskField, currentTaskType, currentTopicId, currentStepName（field_based 按 stepName 匹配 contractEvolution 规则）, contractStore, tunableParams |
| updateGraph() | 每轮模型返回后（写） | 更新逻辑拼接图，记录本轮对话和校验结果 | userInput, modelTurn, turnType, askingField, changeLevel, validationResult |
| archiveGraph() | 偏离/放弃/完成时（归档） | 归档当前上下文图，新建空图 | — |

**接口职责边界**：调度器的合同是"告诉 上下文管理器 当前在干什么（taskType/topicId）+ 给什么资源（contractStore/tunableParams）"，上下文管理器 的合同是"基于这些信息拼出最优上下文"。调度器不预查询 domainRules/topicEvolution 等数据，不在接口层承担数据组装职责——这确保了"调度器只调接口、不内联拼接逻辑"的纪律在参数层面也成立。

**askingField null 降级规则（两种模式通用）：** 当 askingField 为 null 时（如 reply 轮次、complete 轮次），硬门控保护降级为纯语义匹配。field_based 模式不执行字段名匹配保护，topic_based 模式不执行主题标签匹配保护，均改为 上下文管理器 按综合匹配度评分分配形态。降级不报错不阻断。

### 2.8 taskType 对降级链的影响

四项降级链在两种模式下的执行差异（v8.5：移除跨任务延伸检测）：

| 降级项 | field_based 模式 | topic_based 模式 |
|-------|-----------------|-----------------|
| L1 结构校验 | 相同（JSON.parse + Schema 校验） | 相同 |
| DET 值域复验 | 值域校验（金额上限、时间格式、必填字段…） | 输出格式校验（JSON Schema / 文本格式） |
| logprobs 检查 | 相同（概率判断来自 logprobs + pattern 扫词表） | 相同 |
| 硬编码兜底 | 相同 | 相同 |

> **v8.5 修正：移除跨任务延伸检测**——跨任务是用户显式确认行为，不需要降级链层自动化检测。详见 §6.1 off-task 双重保障 + M1 switch 口令。

### 2.9 五步流水线统一操作模型（v7.2 新增）

两种 taskType 遵循相同的五步环节操作流水线（详见总纲 v5.5 §3.9）：**状态机路由 → 环节上下文加载 → 内容拼接 → 模型计算 → 记录落盘**。接口统一（路由信号→加载→拼接→计算→落盘），内部实现分支（步骤一匹配模式 DET vs embedding、步骤二加载源 sessionCheckpoint vs topicEvolution、步骤三锚点模式字段硬门控 vs 三层注入+两阶段检索、步骤四校验方式值域 vs 格式、步骤五落盘目标 sessionCheckpoint vs topicEvolution event）。核心纪律：态控调度器只管路由和记录，上下文管理器只管拼接，环节宪法只管模型行为规则。

### 2.10 环节内规则与跨主题规则的优先级仲裁（v7.2 新增）

domainRules 表中同时存在两类规则——环节内规则（contractEvolution，来自单环节 changeLog 提炼）和跨主题规则（topicEvolution，来自多主题交叉验证）。冲突仲裁规则（详见总纲 v5.5 §3.2）：

| 维度 | 规则 |
|------|------|
| 优先级 | 环节内规则 L2 > 跨主题规则 L2（同优先级环节内优先，更具体） |
| 冲突 | 不允许自动覆盖，标记冲突进入人工确认队列 |
| 共存 | 不冲突时可共存，注入时按优先级排序 |

### 2.11 实现形态声明（v7.2 新增）

各关键组件的架构固定属性与实现层可变属性（详见总纲 v5.5 §2.7）：

| 组件 | 架构固定 | 实现可变 |
|------|---------|---------|
| 态控调度器 | DET——路由+全量记录+调配 | 路由表结构、状态转移实现 |
| 上下文管理器 | 从调配内容中组装上下文 | DET/LLM/混合，由场景复杂度决定 |
| 环节内执行体 | 局部宪法约束下完成交互 | DET/LLM/人，由 taskType 决定 |
| 规则提炼引擎 | changeLog→候选规则→用户确认 | 触发条件（但必须低频） |
| DET 复验 | 确定性收敛，不调 LLM | 校验规则集、值域定义 |
| contractStore | 存储契约/规则/演化记录，支持子图匹配 | 存储格式、索引结构、GC 策略 |
| taskType 判定 | 判定主体固定——由用户判定 | 判定时机与缓存机制 |

### 2.12 契约演化版本管理（v7.2 新增）

契约 schema 的 semver 版本管理（详见总纲 v5.5 §8.3）：新增可选字段=patch、新增必填=minor、删除/重命名=major。taskType 变更视为 major——导致 askingField 语义、上下文管理器锚点模式、DET 复验方式全部变化。存量会话在下一节点入口执行转换，失败降级至 CLARIFYING。

### 2.13 @importance 重要性分层机制（v7.3 新增）

与 `immutableLevel`（权限轴：谁能改）正交的第二轴——重要程度轴，三层独立计数。详见总纲 v5.5 §3.10。消费方：上下文管理器按 @importance 分级截断 roomConversationLog、规则注入排序、冲突仲裁。跨层冲突时按 immutableLevel 直接裁决，importance 不参与跨层仲裁（详见总纲 §3.6.2）。tunable `critical_room_history_boost`（默认 2.0）控制 critical 房间额外历史轮次。

### 2.14 编程规则宪法（v8.0 新增，v8.1 同步 v5.4/v5.5 扩展为 7 条）

**定位**：编程规则宪法是母本的一部分，在整个 N1→N15 管线全局生效，不是某个节点产出的。任何涉及代码生成的环节（N13）必须在生成代码前校验此宪法，违反任一条款拒绝生成。

**约束对象**：代码模型（生成骨架代码的 AI），非运行时模型。

**七条全局元约束**：

| 编号 | 条款 | 内容 |
|:--:|------|------|
| 1 | **执行体分派铁律** | 每个状态的执行体由 N3 决定，代码模型不得自行更改。ANALYZING=LLM，IN_SESSION=LLM（可调用 DET 工具），DET 复验=DET，EXECUTING=DET（仅纯写入/删除）。不得将 ANALYZING/IN_SESSION 降级为正则或 parser |
| 2 | **DET 职责边界** | DET 只做**确定性计算**：元指令 EXACT_MATCH + 值域/格式复验 + 关键词扫描标记 + logprobs 阈值裁决 + 路由矩阵裁决 + 规则子图匹配与冲突检测 + 文件操作 + SQL 查询（作为 IN_SESSION 的工具被调用）。不扩展到意图识别、字段采集、查询理解、语义判断 |
| 3 | **LLM 职责边界** | LLM 做：意图识别 + 字段采集/追问/偏离判断/查询理解 + 语义边界判断 + 工具调用决策 + 结果格式化（自然语言生成）。不得用正则/parser 替代 |
| 4 | **降级规则** | 无 apiKey 时 ANALYZING/IN_SESSION 必须拒绝服务并明确告知原因，不得以任何形式降级为正则/DET 替代。降级模式如存在必须标注"降级非正道" |
| 5 | **契约对齐** | 代码必须严格对齐 N11 契约（contract-in/out schema），不自创接口 |
| 6 | **tunable/接口隔离** | 阈值从 getTunable() 读，调度器只调 context-manager 三接口（匹配/决策/拼接），不内联拼接逻辑。上下文管理器三层职责各自独立，不得合并 |
| 7 | **工具分层发现（v5.5 新增）** | 工具分三层：必用工具（环节宪法 @section tools 声明，完整定义自动注入）/ 工具清单（@section tool_catalog，轻量级目录每次注入）/ 选用工具（系统级工具库 toolRegistry，完整定义按需加载）。模型从清单选择→调度器加载完整定义→模型调用。LLM 不直接写 SQL，通过 DET 工具间接操作数据库。search_tools meta-tool 始终可用 |

**生效时机**：设计者在 P0 认知加载时即需理解，N13 代码模型强制输入。


### 2.15 IN_SESSION tool calling 机制总览（v8.1 新增，v5.4/v5.5）

v5.4 起，IN_SESSION 环节内的 LLM 可通过 function calling 调用 DET 工具，用于查询/计算等需要数据库或确定性操作的 intent。

**核心纪律**：
- **IN_SESSION 可调用 DET 工具**——查询、对比、计算等操作在 IN_SESSION 内通过工具调用完成
- **EXECUTING 仅保留纯写入/删除**——record INSERT / delete DELETE，不再承载查询逻辑
- **LLM 不直接写 SQL**——通过 DET 工具间接操作数据库

**工具分层发现（v5.5 三层体系）**：

| 层 | 名称 | 注入方式 | 内容 |
|----|------|---------|------|
| 必用工具 | @section tools | 完整定义自动注入每轮 prompt | 环节必须的 DET 工具（如记账环节的 queryLedger/insertRecord） |
| 工具清单 | @section tool_catalog | 轻量级目录每次注入 | 可选工具的摘要列表（名称+一句话描述），供模型选择 |
| 选用工具 | toolRegistry 表 | 完整定义按需加载 | 系统级工具库，模型从清单选择→调度器加载完整定义→模型调用 |

**Two-Phase Hierarchical Selection**：模型先看清单选工具→调度器加载完整定义→模型调用。search_tools meta-tool 始终可用，用于清单外的工具发现。工具内部支持时间解析（如 "this_month"→日期范围）。

**流程中的影响**：
- N3（执行体分派）：IN_SESSION 节点需声明可调用的工具集
- N5（调度器设计）：需实现工具调用路由（工具名→DET 函数映射）
- 第三步（写宪法）：环节宪法需包含 @section tools 和 @section tool_catalog
- N13（骨架生成）：代码模型需遵守编程规则宪法第 7 条（工具分层发现）


### 2.16 项目隔离机制总览（v8.1 新增，v5.6）

v5.6 引入项目隔离机制，解决同一环节下不同智能体项目产出物隔离问题。

**核心数据结构**：
- **currentProjectId**：态控调度器全局字段，一处变更所有房间联动
- **projectRegistry 表**：`{projectId, projectName, projectKeywords, taskType, ...}`
- **userLastProject 表**：`{userId, lastProjectId}`——用户唤醒时自动加载默认项目

**项目切换（S3 物理隔离）**：
```
用户说"切换到杂碎本" → ANALYZING 关键词匹配 projectRegistry.projectKeywords
→ S3 四步：释放当前项目 KV Cache → 归档 → 加载新项目最后状态 → 创建新会话
→ 更新 currentProjectId + userLastProject → 全局联动
```

**项目切换 vs 房间切换**：
| 维度 | 房间切换（switch） | 项目切换 |
|------|-------------------|---------|
| 触发 | "切断房间"口令 | 用户说项目名 |
| roomId | ✅ 变 | ❌ 不变 |
| projectId | ❌ 不变 | ✅ 变 |
| 全局联动 | 不联动 | ✅ 联动 |

**流程中的影响**：
- N1（场景定义）：需声明项目级别的产出物隔离需求
- §2.2.5 schedulerState：补 currentProjectId + projectRegistry + userLastProject
- §十五 contractStore：补 projectRegistry + userLastProject 表


### 2.17 文件管理规范总览（v8.1 新增，v5.6）

v5.6 引入文件管理的"房间比喻四层结构"和三个机制：

```
房间（环节，N1-N16）—— 环节宪法（房间级，共享）
  └── 项目（智能体实例）—— 产出物内容（项目级，隔离）
       └── 格子（细则文件）—— 正文内容（文件级，独立）
            └── 变更日志（独立文件）—— 所有变更追溯（全局统一）
```

**机制1 项目隔离**：见 §2.16
**机制2 分格**：大文件按章节拆成独立细则文件，降低单文件算力成本
**机制3 变更日志独立**：所有变更统一记录在变更日志，细则文件不含变更过程

**设计流程中的纪律**：
- 修改基线时另存新版本，不直接覆盖
- 细则文件只含正文，不含变更记录
- 总纲是概要索引，需要详情时读对应细则
- 大改动分批进行，每次只改一个细则文件

## 二、设计流程

### 第一步：场景理解与边界划定

#### 1.1 目标

搞清楚"这个系统到底要干什么"，以及"它绝对不能干什么"。这一步是所有后续设计的前提，边界没划清楚就设计状态，一定会漏状态。同时，每个 intent 的 taskType（field_based 或 topic_based）需要在此阶段初步判定。边界划定后必须通过边界覆盖完整性测试，验证所有用户输入都有处理路径。

#### 1.2 交付物

##### 1.2.1 一句话场景定义

用一句话说清楚系统是什么。例如：

> "这是一个语音记账工具，用户说出支出，系统记录并支持查询和删除。"

或：

> "这是一个角色扮演助手，用户设定角色和场景，系统在主题边界内进行多轮角色扮演对话。"

或：

> "这是一个全屋定制方案设计助手，用户描述需求，系统通过多轮对话完成方案设计，支持多方案对比、风格替换和方案迭代。"

这句话将成为后续所有设计决策的锚点。任何功能如果无法被这句话覆盖，就不应该被加入。

##### 1.2.2 边界清单

明确系统做什么和不做什么。建议用表格形式，每个 intent 标注 taskType：

| 维度 | 做什么 | 不做什么 | taskType |
|------|--------|---------|----------|
| 核心功能 | 记录支出、查询历史、删除记录 | 不自动分类、不推荐理财产品 | record/query/delete: field_based |
| 核心功能 | 角色扮演对话 | 不生成图片、不执行真实操作 | roleplay: topic_based |
| 核心功能 | 全屋定制方案设计、多方案管理、方案对比 | 不直接对接工厂生产系统 | design: topic_based |
| 交互方式 | 语音输入、文字输入 | 不支持图片识别、不支持手势操作 | — |
| 对话范围 | 业务相关的指令 | 不闲聊、不回答非业务问题 | — |
| 数据处理 | 本地存储用户数据 | 不上传云端分析、不分享给第三方 | — |

**关键规则：** 边界清单中"不做什么"的部分，比"做什么"更重要。它决定了系统在面对超出预期的输入时，应该如何回应。但"不做什么"不意味着"不处理"——功能范围外的输入必须有明确的处理流程（硬拒/软着陆/引导），不能出现"无法处理"的输入。

**taskType 初判标准**（v5.8 明确——核心判据：是否存在讨论空间）：

| 判据 | field_based（无讨论空间） | topic_based（有讨论空间） |
|------|------------------------|------------------------|
| **讨论空间**（v5.8） | **无**——用户说啥就是啥 | **有**——模型主动追问、提供选项 |
| 收敛方式 | 填表式——按预定义字段采集 | 讨论式——在主题边界内自由探索 |
| "完成"的判定标准 | 字段齐全 + 值域通过 | 输出满足主题边界 + 格式约束 |
| 典型场景 | 记账、改柜体参数、表单填写、查询、删除 | 内容生成、角色扮演、头脑风暴、**设计引导** |

**taskType 初判时需考虑的 topicEvolution 因素：** 当判定某个 intent 为 topic_based 时，设计者必须意识到：

- 该 intent 的所有交互将被自动记录到 topicEvolution 表中
- 系统将自动启用三层递进机制：分层留存 + 主题关联 + 异步规则提炼
- 异步规则提炼存在"知识真空期"——领域规则不保证跨相邻任务的即时传递
- 跨会话的"继续上次的XX"将成为系统标配能力
- 多个同类型主题将逐步积累领域规则，智能体越用越老练
- 需要在 N9 环节宪法 @section output-schema 中声明 topicId 字段

**混合型任务处理：** 某些任务可能同时包含两种 taskType 模式。处理方式：在环节宪法中声明主 taskType（topic_based），将 field_based 约束作为 topic_based 的格式约束子条款。DET 复验时先做 field_based 格式校验，再做 topic_based 主题边界校验。

**关于松弛节点：** 如果智能体需要支持"闲聊→任务"的自然过渡（如用户在闲聊中突然说"算了不聊柜子了，帮我写首诗"），边界清单中应显式声明松弛节点的存在及其边界。松弛节点是合法存在的低约束节点，仅受根宪法约束，不区分 taskType。但松弛节点的边界本身也需要经过边界覆盖完整性测试——如果松弛节点的闲聊输入无处理路径或护栏响应不合理，说明松弛节点的边界划定有问题，需要回溯第一步重审边界。

##### 1.2.3 边界紧张度测试

边界定好后，必须用真实用户的自然表达进行压力测试，验证边界覆盖完整性 + 护栏响应质量。

**核心原则：** 智能体必须能够处理所有用户输入的信息。用户输入只有两类：

- **类一：** 落在功能范围内（"做什么"范围内）→ 按意图识别结果由状态机处理
- **类二：** 落在功能范围外（"不做什么"范围内）→ 智能体同样必须有处理流程（硬拒 / 软着陆 / 引导）

**测试目标：**

| 维度 | 测试目标 |
|------|---------|
| 测试对象 | 边界覆盖完整性 + 护栏响应质量 |
| 通过标准 | 所有测试语料都有明确处理路径 + 护栏响应合理 |
| 失败标准 | 存在无处理路径的输入类型 OR 护栏响应不合理 |
| 测试本质 | 验证"覆盖完整 + 响应合理" |

**测试方法（5 步流程）：**

**Step 1：收集测试语料**

收集 20-30 条真实用户的目标场景自然表达。

语料获取路径（按可信度降序）：

| 优先级 | 路径 | 说明 |
|-------|------|------|
| 最佳 | 已有同类产品的真实用户日志（脱敏后） | 有则必用，直接反映真实分布 |
| 次佳 | 找 5-8 个非项目组成员做 10 分钟口语模拟 | 最低要求，不可跳过。非项目组成员不知道边界在哪，其样本不会被边界先验污染 |
| 仅作初稿 | 设计者自写 | 设计者已知道边界在哪，写出的样本会刻意贴着边界写。交付前必须用路径 2 替换至少 50% |

**Step 2：分类标注**

对每条测试语料，标注两个维度：

**维度一：输入类型分类**

- 类一（功能范围内）：落在"做什么"范围内的输入
- 标注：属于哪个 intent（record / query / delete / write_poem / roleplay / ...）
- 预期处理：按意图识别结果由状态机处理
- 类二（功能范围外）：落在"不做什么"范围内的输入
- 标注：属于哪种越界类型（闲聊 / 非业务问题 / 超出能力范围 / ...）
- 预期处理：按边界清单中的护栏策略处理（硬拒 / 软着陆 / 引导）

**维度二：处理路径检查**

- ✅ 有处理路径：该输入类型在边界清单或状态转移图中有明确的处理流程
- ❌ 无处理路径：该输入类型在边界清单和状态转移图中都没有对应的处理流程——这是真正的"缺口"

**Step 3：边界覆盖完整性检查**

检查所有测试语料的处理路径：

| 检查项 | 通过标准 | 失败处理 |
|-------|---------|---------|
| 类一输入处理路径 | 所有类一输入都有对应的 intent + 状态机处理路径 | 回退 N1，补全 intent 清单 |
| 类二输入处理路径 | 所有类二输入都有对应的护栏策略（硬拒 / 软着陆 / 引导） | 回退 N1，补全护栏策略 |
| 处理路径合理性 | 护栏响应语合理（不是"我只记账哦"这种生硬拒绝，而是有引导或软着陆） | 回退 N1，优化护栏响应语 |

**Step 4：护栏响应质量评估**

对类二输入的处理流程，评估响应质量：

| 质量等级 | 表现 | 处理 |
|---------|------|------|
| ✅ 优秀 | 软着陆 + 引导到功能范围内（如"我可以帮你记，你要先记完这条还是先查？"） | 通过 |
| ✅ 合格 | 明确拒绝 + 说明边界 + 提供替代（如"小安只会记账，不会聊天哦。你需要记一笔吗？"） | 通过 |
| ⚠️ 待优化 | 生硬拒绝无引导（如"我只记账哦"） | 回退 N1，优化护栏响应语 |
| ❌ 不合格 | 无处理路径 / 崩溃 / 幻觉回应 | 回退 N1，补全护栏策略 |

**Step 5：生成测试报告**

```
边界覆盖完整性报告：
- 测试语料总数：N 条
- 类一输入（功能范围内）：N1 条
- 有处理路径：N1a 条 ✅
- 无处理路径：N1b 条 ❌
- 类二输入（功能范围外）：N2 条
- 有处理路径：N2a 条 ✅
- 无处理路径：N2b 条 ❌

护栏响应质量报告：
- 优秀：N 条
- 合格：N 条
- 待优化：N 条
- 不合格：N 条

结论：
- 边界覆盖完整（N1b=0 且 N2b=0）+ 护栏响应合理（待优化+不合格=0）→ 进入第二步
- 边界覆盖有缺口（N1b>0 或 N2b>0）OR 护栏响应不合理（待优化+不合格>0）→ 回退 N1
```

**护栏升级路径：** 当边界本身正确但护栏响应质量待优化/不合格时，不应扩张边界，而应将护栏从"硬拒"升级为"软着陆"。例如从"我只记账哦"升级为"我可以帮你记，你要先记完这条还是先查？"——边界位置不变，但边界的交互语义从拒绝变为引导。

**护栏升级的分支目标鉴别规则：** 任何"护栏升级为软着陆"的决策，必须记录软着陆实际调用的分支目标是否已经存在于当前状态转移图中。如果"先查"对应的是状态转移图里已有的 query 子流程和路由，这只是让边界护栏指向已有能力，边界声明未变。如果软着陆引入了状态转移图中不存在的分支路径，这不是护栏升级，而是"收缩不做什么"的伪装形式——必须走正式扩展边界流程，不得以"护栏调整"之名绕过。

##### 1.2.4 用户角色与交互方式

- **谁在用：** 目标用户群体（如个人用户、家庭用户、小微企业主）
- **怎么用：** 语音唤醒还是文字输入？手机 App 还是网页？是否支持多设备同步？
- **交互频率：** 每天记几笔？每次会话持续多久？
- **网络环境：** 在线还是离线？弱网环境下如何处理？

#### 1.3 设计原则

1. **边界覆盖必须完整，所有输入都必须有处理路径。** 智能体必须能够处理所有用户输入，不存在"落出"的输入。功能范围内的输入按意图识别处理，功能范围外的输入按护栏策略处理。边界划定后必须通过边界覆盖完整性测试，验证所有输入类型都有处理路径。
2. **"不做什么"清单必须写入宪法。** 这些规则不是产品文档，而是系统行为的一部分。"不做什么"不意味着"不处理"——功能范围外的输入必须有明确的处理流程。
3. **边界划定需要回答"用户超出边界时系统怎么回应"。** 不能只说"不闲聊"，还要说"用户闲聊时系统回复什么"。护栏响应必须有质量（优秀/合格/待优化/不合格四级评估）。
4. **边界必须经过紧张度测试。** 未经真实语料验证的边界，其覆盖完整性是未知的。测试所用语料必须来自非项目组成员或真实用户日志。
5. **松弛节点的边界同样需要紧张度测试。** 如果智能体支持松弛节点，松弛节点的闲聊边界也要用真实语料验证覆盖完整性。
6. **taskType 初判必须在边界划定阶段完成。** taskType 决定后续所有节点的行为，如果到写宪法阶段才发现 taskType 判错，需要回溯第一步重判。topic_based 的初判同时意味着系统将自动为该 intent 启用三层递进机制（分层留存 + 主题关联 + 异步规则提炼）——设计者需在初判时明确这一点。

---

### 第二步：拆状态（含定执行体 + 路由表 + taskType 固化 + topicEvolution/sessionCheckpoint 初始化）

#### 2.1 目标

将智能体工作流程拆解为离散状态节点，定义转移关系和数据流契约，并为每个节点标注执行体类型。采用环节级自治范式。每个 intent 的 taskType 从初判固化为确定值。当 taskType 被固化为 topic_based 时，态控调度器自动初始化 topicEvolution 三层递进机制。规则提炼完全异步，不在状态机流转链路内执行。

#### 2.2 交付物

##### 2.2.1 架构总览：态控调度器 + 环节内模型自治

系统的核心架构是 "态控调度器（DET）+ 环节内模型实例（LLM）"。

**调度器（状态机）：** 负责意图识别、任务分发、上下文拼接管理、状态调度、以及确定性兜底校验。调度器不再参与环节内的逐轮对话——它将整个任务环节（如"记一笔账"或"写一首诗"）封装为一个自治单元，委托给模型实例完成。topic_based 任务的调度器额外承担 topicIndex 检索、topicEvolution 初始化与分层留存、主题关联发现与确认。规则提炼由独立后台进程异步执行，态控调度器只在节点 exit 时将主题信息推入待处理队列。

**环节内模型实例：** 收到调度器分发的任务后，在局部宪法约束下与用户进行多轮对话，自主追问缺失字段（field_based）或自主探索主题（topic_based）、自主执行第一道值域校验/格式校验、自主判断偏离。模型在每一轮对话结束时，将结果（含状态标记、校验结果、已采集字段/已讨论主题等）以结构化形式返回给调度器，并顺带输出 changeLevel 预判字段（major/minor/invalid）。

**调度器与模型之间的契约：** 调度器向模型注入任务描述 + 局部宪法 + 拼接好的上下文窗口（topic_based 模式额外包含三层注入包：已生效领域规则 + 关联主题摘要 + topicEvolution 分层包）；模型完成后返回结构化 JSON 结果（含 turnType 字段、askingField 字段、changeLevel 预判字段）。

**上下文拼接：** 由态控调度器根据"逻辑拼接机制"管理。模型每次收到的输入是调度器拼接好的上下文（含必要历史摘要和当前任务），模型不自行管理上下文窗口。调度器通过 上下文管理器 模块管理拼接，自身不实现拼接算法——只调不问。

##### 2.2.2 状态枚举

调度器维护以下核心状态：

| 状态 | 类型 | 含义 | 说明 |
|------|------|------|------|
| IDLE | 稳态 | 空闲，等待唤醒 | 无人/无程序 |
| LISTENING | 稳态 | 待命，等待用户输入 | 唤醒后进入 |
| ANALYZING | 瞬态 | 意图识别 | 调 LLM 做意图识别，产出 intent + taskType。field_based 任务先查 sessionCheckpoints 是否存在未过期进度→存在则询问用户继续/切换。topic_based 任务自动检索 topicIndex（关键词优先，embedding 兜底），匹配到已有主题则展示主题名供用户确认续写，未匹配到则生成候选 topicName+keywords 供用户确认新建。topic_based 任务额外执行规则问询（user_initiated 来源），写入 ruleCandidates 供后续注入 |
| IN_SESSION | 稳态 | 环节执行中（可含 PREVIEWING 子环节用于可视化 diff 预览，仅 field_based 启用） | 模型在返回 turnType 时顺带输出 changeLevel 预判字段。态控调度器在节点 exit 时追加 event（使用模型判定的 changeLevel），并将主题信息推入规则提炼的待处理队列。不触发任何提炼逻辑。子状态通过模型返回的 turnType 可观测：ask/reply=追问中，validation_failed=校验失败追问中，off-task=偏离处理中，complete=待 DET 复验 |
| CLARIFYING | 守护态 | 校验失败追问转发 | 调度器复用模型 message 转发用户（不调 LLM）。此状态为监控标记态，便于运维监控校验失败率和触发频率 |
| WAITING_CONFIRM | 守护态 | 等待用户确认 | 不可逆操作前（如删除确认） |
| EXECUTING | 瞬态 | 执行操作 | 写数据库等。追加 event（使用模型判定的 changeLevel），stateSnapshot 标记为 checkpoint，将主题信息推入规则提炼的待处理队列 |
| CLOSING | 瞬态 | 退出 | 清理资源 |

##### 2.2.3 状态转移图

```
IDLE ──[唤醒]──→ LISTENING
↑                  │
│                  ↓
│              ANALYZING (产出 intent + taskType，topic_based 时检索 topicIndex、用户确认 topicName+keywords、初始化 topicEvolution)
│                  │
│     ┌────────────┼──────────┐
│     ↓            ↓          ↓
│  IN_SESSION   LISTENING   IDLE
│  [topic_based: 关联发现提示 + 三层注入 + 模型返回 changeLevel 预判]
│     │
│     ├── turnType="complete" → DET 复验 → EXECUTING
│     │       [topic_based: 使用模型判定的 changeLevel 追加 event + 标记 checkpoint + 推入提炼待处理队列]
│     ├── turnType="validation_failed" → CLARIFYING → IN_SESSION
│     ├── turnType="off-task" → ANALYZING
│     ├── turnType="giveup" → LISTENING
│     │       [topic_based: 使用模型判定的 changeLevel 追加 event + 标记 abandoned + 推入提炼待处理队列]
│     └── turnType="ask"/"reply" → 继续 IN_SESSION
│
│ [意图级纠错逃生舱]
│ 任意状态 ──[用户反馈意图完全错误]──→ contractStore 取 ANALYZING 原始 contract-in
│     └── 重新实例化 ANALYZING → 重新意图识别
│
└─────────────────────────────────────┘

[异步规则提炼]（独立后台进程，不在状态机流转链路内）
触发条件：时间窗口（每日凌晨2点）/ 数量阈值（积累N个同类主题checkpoint）/ 手动触发
提炼引擎：
1. 从 contractStore 读取待提炼主题列表
2. 提取候选规则，存入 ruleCandidate 池
3. 交叉验证（验证通过率 ≥ domain_rule_validation_threshold，默认 0.7）
4. 通过验证的候选规则提交用户确认
5. 用户确认后的规则存入 contractStore.domainRules 库
```

##### 2.2.3.1 节点类型动态转换规则

态控架构将 IN_SESSION 子状态分为收敛节点和松弛节点两类。以下为两者之间的动态转换规则：

| 转换方向 | 触发条件 | 动作 | 守卫规则 |
|---------|---------|------|---------|
| 松弛节点 → 收敛节点 | 用户输入明确任务指令 | 意图识别接管，进入收敛链 | 根宪法元指令集优先匹配；置信度≥0.7 方可转换 |
| 收敛节点 → 松弛节点 | 用户放弃或主动脱离 | contract-out 标记为 abandoned，回退至 IDLE 或进入松弛节点 | 根宪法定义"放弃指令"匹配模式，不接受局部宪法覆盖 |
| 收敛节点 → 另一收敛节点 | 用户发出元指令 | 物理隔离当前上下文，动态实例化新节点 | 元指令集由根宪法定义，不接受动态修改 |

taskType 转换纪律：taskType 在 N1/N3 阶段确定后，运行时不变。如果用户在 field_based 任务中突然切换到 topic_based 任务（如记账中突然说"帮我写首诗"），通过元指令识别机制触发节点切换——物理隔离当前上下文，进入新的 IN_SESSION（topic_based）。taskType 不在环节内动态切换。

##### 2.2.4 态控调度器核心逻辑

用户输入进入调度器：

```
第0层：口令精确匹配（元指令集口令层，EXACT_MATCH→直切，非匹配→放行ANALYZING）
匹配唤醒词 → 进入 LISTENING
匹配退出/取消词 → 进入 CLOSING 或放弃当前流程
匹配器为可配置浅匹配器（子串/编辑距离/同义词组）
注：自然语言疑似切换→ANALYZING 判 meta_suspect→二次确认后才切（语义确认层）

第0.5层：DET 关键词扫描（每轮执行，0 次 LLM）
对用户输入做关键词扫描，检测跨意图词汇：
- 当前环节为 record，输入含"查询/查一下/多少" → offTaskSuspicion="query"
- 当前环节为 record，输入含"删除/删掉" → offTaskSuspicion="delete"
标记附加到模型输入中作为提示，不强制覆盖模型判断。

第1层：意图识别（LISTENING 态收到输入）
→ 调 LLM 做意图识别（注入意图识别宪法）
→ 拿到 {intent, taskType, subType, logprobs（概率分布）, extracted（已提取信息）}
→ field_based 任务：
   a. 先查 sessionCheckpoints 是否存在未过期进度
   b. 若存在 → 生成 resume_chat intent，询问用户"继续 or 去其他环节"
   c. 用户确认继续 → 注入 completedSteps + stepSnapshots → 路由到目标环节
   d. 用户拒绝 → 清空 checkpoint，正常进 P0
→ topic_based 任务：
a. ANALYZING 阶段 LLM 生成候选 topicName + keywords
b. 调度器执行关键词匹配（基于 topicIndex 表），检测是否已有相近主题
c. 若匹配到已有主题 → 系统展示候选主题名，等待用户确认（续写/重选/新建）
d. 若未匹配到 → 系统展示生成的候选主题名+keywords，等待用户确认（确认/修改/搜索已有）
e. 用户确认后，态控调度器生成 topicId，初始化 topicEvolution 记录（写入 contractStore.topicIndex 和 topicEvolution 表，写入第一个 event，changeLevel 由模型预判）
f. 用户否定或修改 → 按用户指示调整后，再初始化
g. 主动问询用户是否有已知的偏好或规则约束（user_initiated 来源）→ 写入 ruleCandidates 供后续注入
h. 系统自动检测当前主题与已有主题的关键词重叠度，若高则提示用户是否建立关联
→ 若 intent="meta_instruction"（语义模板层判定为元指令），触发节点切换

第2层：任务分发
根据 intent 选择局部宪法（含 taskType 声明），注入初始上下文（含 extracted 中已提取的字段）
→ field_based 任务注入：stepSnapshots 结构化字段 + roomConversationLog(roomId) 最近N轮对话 + contractEvolution 已生效规则（按 stepName DET 匹配）
→ topic_based 任务额外注入三层注入包：
第0层：已生效的领域规则（从 contractStore.domainRules 读取，异步提炼的产出）
第1层：关联主题摘要卡片（从 topicIndex.relatedTopics 读取）
第2层：topicEvolution 分层包（majorEvents + compressedMinorSummary + recentMinorEvents + currentStateSnapshot）
→ 创建模型实例 → IN_SESSION
→ 路由表第二层（意图级绑定）覆盖默认模型
注：topic_based intent 倾向绑定深度模型，field_based intent 用默认通用模型

第3层：环节内监督循环（IN_SESSION 态）
循环：
1. 调度器调用 上下文管理器.buildPromptContext() 拼接上下文
2. 调度器将拼接好的上下文（含当前任务、已采集字段/已讨论主题、历史摘要、offTaskSuspicion标记、topic_based 模式额外包含三层注入包）发送给模型实例
3. 模型实例生成下一轮回复，并返回结构化结果（含 turnType + askingField + changeLevel 预判）
- changeLevel 由模型在推理过程中顺带判定（不额外增加 LLM 调用）
- 若模型未返回 changeLevel 或格式错误 → 态控调度器默认标记为 minor
4. 调度器根据 turnType 进行状态调度：
- "ask" / "reply" → 将 message 转发给用户，继续循环
- "validation_failed" → 进入 CLARIFYING 状态。调度器直接复用模型返回的 message 作为追问内容（不调LLM），转发给用户。用户回应后，态控调度器将用户回应作为新一轮输入送回模型实例，同时调用 上下文管理器.updateGraph() 将上一轮的 validationResult 作为上下文标记加入逻辑拼接图，然后继续 IN_SESSION
- "off-task" → 终止环节，调用 上下文管理器.archiveGraph() 归档当前上下文图，将 offTaskInput 送入 ANALYZING（重新意图识别）。topic_based 任务在 off-task 时仍追加 event 到 topicEvolution（记录偏离前的状态，changeLevel 使用模型预判）
- "giveup" → 进入 LISTENING。topic_based 任务在 giveup 时使用模型判定的 changeLevel 追加 event，stateSnapshot 标记为 abandoned，推入提炼待处理队列
- "complete" → 进入第4层 DET值域复验
5. 每轮结束后调用 上下文管理器.updateGraph()，传入 userInput、modelTurn、turnType、askingField、changeLevel、validationResult 更新逻辑拼接图

第4层：DET 值域复验 + L1 结构校验
模型返回 complete 时，对 result 执行第二道 DET 校验：
1. L1 结构校验：JSON.parse + Schema 校验
2. DET 值域复验：
- field_based：值域校验（金额上限、时间格式、必填字段…）
- topic_based：输出格式校验（JSON Schema / 文本格式 / 字数 / 结构…）
3. logprobs 检查：probability（来自模型输出 token 的 logprobs）< threshold + pattern 扫"硬撑式确认"词表
- field_based：DET 字段白名单（默认层即可）
- topic_based：主题边界检查（可能需要 needSemanticExtensionCheck:true 追加轻量 LLM）
复验通过 → EXECUTING

[topic_based: 文件写盘后]
- stateChange 摘要由 N10 后台异步消费 ruleMiningQueue 时生成，态控调度器在节点 exit 时只落盘 topicEvolution event 原始数据
- 使用模型判定的 changeLevel 追加 event：
* major → 永久保留
* minor → 进入累积压缩流，超过阈值时触发汇总压缩
* invalid → 不入主表，迁入 topicEvolutionArchive 归档表
- stateSnapshot 标记为 checkpoint
- 将主题信息推入规则提炼的待处理队列
- 不触发任何提炼逻辑——提炼由独立后台进程异步执行

[意图级纠错逃生舱]（随时可触发）
用户反馈表明当前意图完全错误 → 状态机从 contractStore 取出 ANALYZING 节点的原始 contract-in 副本
→ 重新实例化 ANALYZING 节点（注入原始 contract-in + 宪法）
→ 重新执行意图识别，产生新的 intent（可能伴随 taskType 变化）
```

**上下文管理器 接口隔离原则：** 即使 上下文管理器 尚未实现，状态机也应通过接口调用占位实现，不得将拼接逻辑内联到状态机中。占位实现可直接拼最近 N 轮原文，但必须通过 buildPromptContext() 接口调用，便于后续替换零成本。

##### 2.2.6 路由表设计

路由表是确定性代码，不调 LLM。路由表同时承担两个职责：状态路由（决定下一个状态）和模型选择（决定 LLM 槽位由哪个模型实例填充）。

**两层模型选择机制（设计时预留）：**

- **第一层（状态级默认模型）：** 每个 LLM 节点有预设的默认模型实例，不依赖意图。ANALYZING → 通用轻量模型（意图识别不需要深度能力）；IN_SESSION → 通用模型。此层解决"ANALYZING 节点本身用哪个模型"的循环依赖。
- **第二层（意图级绑定）：** 当上游节点产出 intent 后，下游节点可根据"状态+intent"组合覆盖默认模型。IN_SESSION + intent="write_poem" + taskType="topic_based" → 专用文学模型。此绑定配置在路由表中。

**taskType 对模型选择的影响：** topic_based intent 倾向绑定深度模型（主题理解和创意生成需要深度能力），field_based intent 用默认通用模型即可（字段映射不需要深度能力）。路由表第二层绑定时需考虑 taskType。

即使当前场景只有 field_based 任务，也应在设计时预留 taskType 字段和两层模型选择机制——这是解决 ANALYZING 节点模型选择循环依赖的关键机制，且为未来 topic_based 任务扩展提供零成本路径。

##### 2.2.7 节点执行体分派判据

每个状态在宪法中必须显式声明执行体，判据顺序如下（优先级递减）：

1. **错误代价多大？**
- 代价大且无客观答案 → 人
- 代价大但有客观校验 → LLM + 人确认
- 代价小 → 进入下一条
2. **能否写成硬规则 + 穷举？**
- 能 → 确定性程序（DET）
- 不能 → 进入下一条
3. **是否有客观正确答案？**
- 有（映射类、意图类）→ LLM
- 没有（偏好类、责任类）→ 人

**关键规则：** "代价"为最高优先级判据。代价大的节点无论能否硬规则化，都必须经过人确认或 LLM+人确认。

**与意图收敛机制的对应：** DET 执行确定性收敛；LLM 执行有限展开后收敛，出口被 contract-out schema 硬卡；人执行责任收敛，对不可逆决策负责。

**环节内模型实例的定位：** 环节内模型实例属于 LLM 执行体，但在环节内拥有自治权——态控调度器不参与其逐轮对话，只在环节入口（注入宪法+上下文）和出口（收 turnType + DET 复验）交互。

##### 2.2.8 模型返回的 turnType 统一格式（含 changeLevel 预判字段）

环节内模型实例在每一轮对话结束时，必须返回统一的结构化结果：

```json
{
"turnType": "ask" | "reply" | "complete" | "off-task" | "giveup" | "validation_failed",
"askingField": null | "字段名(field_based)" | "主题标签(topic_based)",
"changeLevel": "null | major | minor | invalid",
"changeLevelReason": "null | 一句话理由（major/minor 时必填）",
"message": "模型对用户说的话（自然语言，30字以内）",
"validationResult": null | {
"field": "amount",
"issue": "amount_too_large",
"userInput": "一百万"
},
"collectedFields": { "category": "猫粮", "amount": null, "time": null },
"offTaskInput": null | "用户偏离输入",
"result": null | { "category": "猫粮", "amount": 35, "time": "昨天" }
}
```

- **turnType="ask"：** 模型在追问用户，message 为追问内容，askingField 为当前追问的字段名（field_based）或主题标签（topic_based），供 上下文管理器 硬门控保护使用
- **turnType="reply"：** 模型在回复用户（非追问），message 为回复内容，askingField 为 null
- **turnType="complete"：** 所有字段齐全（field_based）或输出满足主题边界（topic_based），result 包含完整结构化结果
- **turnType="off-task"：** 检测到偏离，offTaskInput 包含偏离输入
- **turnType="giveup"：** 用户放弃
- **turnType="validation_failed"：** 值域校验失败（field_based）或格式校验失败（topic_based），validationResult 包含失败详情
- **changeLevel：** 由模型在推理过程中顺带判定（不额外增加 LLM 调用）。两种 taskType 模式均适用。若未返回或格式错误 → 态控调度器默认标记为 minor。判定标准见环节宪法 @section output-schema
- **changeLevelReason：** changeLevel 为 major 或 minor 时必须附带一句话理由（如 "新增了核心角色林霜" 或 "优化了对白用词"）。DET 不验证语义正确性，但理由的存在迫使模型为自己的判定负责。reason 不符合一句话格式（空字符串或超过 100 字）→ DET 格式校验拦截，标记 validation_failed

**askingField 双模式语义详解：**

| taskType | askingField 取值 | 示例 | 上下文管理器 锚点 | 执行方式 |
|----------|-----------------|------|---------------------|---------|
| field_based | 预定义字段名，取值来自环节宪法 @section field-rules | "amount"、"time"、"category" | 字段级硬门控——保护与 askingField 同名的历史轮次不被淘汰 | DET 确定性字符串匹配 |
| topic_based | 模型动态生成的主题锚点标签，格式为 snake_case 英文标签 | "character_motivation"、"plot_conflict"、"hypothesis_A" | 主题级硬门控 + 三层注入——保护与 askingField 语义相关的历史轮次不被淘汰，并在每轮初始注入三层注入包 | 已生效领域规则 + 关联摘要 + topicEvolution 分层注入 + embedding 两阶段检索（DET 关键词粗筛 + embedding 精筛 + DET 阈值裁决，0 次 LLM） |

**askingField 主次标注规则：** field_based 任务中，如果几个字段是强关联的（如"金额+币种"），可以一轮问两个，但要在 askingField 里标清楚主次，格式为 "amount+currency"（主字段在前，次字段在后）。topic_based 任务中，askingField 通常为单个主题标签。

调度器用同一套逻辑处理所有环节的模型返回，不因环节类型或 taskType 不同而改变解析逻辑——差异仅在 上下文管理器 的锚点模式上。

#### 2.3 设计原则

1. **状态是有限的、可枚举的。** 标准状态集通常不超过 10 个状态。
2. **状态之间的转移是确定的。** 转移条件必须是状态机可以判断的。
3. **每个状态只做一件事。** IN_SESSION 是"环节执行中"的容器，CLARIFYING 是"校验失败追问转发"的监控标记态。
4. **环节级自治：** 调度器将整个任务环节委托给模型实例，不参与环节内的逐轮对话。
5. **双重校验：** 模型执行第一道值域校验/格式校验和偏离检测（概率性），态控调度器执行第二道 DET 复验和关键词扫描（确定性）。冲突时以调度器 DET 为准。
6. **异常状态必须存在。** 必须有"放弃"、"超时"、"错误"等兜底状态和处理路径。
7. **意图级纠错逃生舱：** contractStore 保存 ANALYZING 原始 contract-in，支持用户反馈意图错误时强制回滚至 ANALYZING 重新分类。
8. **taskType 固化：** taskType 在 N1 初判、N3 固化后运行时不变。元指令触发的节点切换可能伴随 taskType 变化，但这是"新环节新 taskType"，不是"环节内 taskType 动态切换"。
9. **topicEvolution 三层递进初始化：** 当 taskType 被固化为 topic_based 时，态控调度器自动初始化 topicIndex 索引、topicEvolution 分层留存、主题关联发现。topicName 和 keywords 由 LLM 自动生成，经用户确认后生效。关联关系和领域规则同样必须经用户确认。规则提炼由独立后台进程异步执行，不在状态机流转链路内。

#### 2.4 常见错误

| 错误 | 后果 | 示例 |
|------|------|------|
| 状态粒度太粗 | 一个状态里塞了太多逻辑 | 一个"PROCESSING"同时做意图识别、字段采集、数据库操作 |
| 状态粒度太细 | 状态数量爆炸 | 为每个字段的追问都设一个独立状态 |
| 调度器参与环节内追问 | LLM 调用次数翻倍，上下文被切碎 | 每收到用户输入就调一次 LLM 判断 |
| 环节宪法约束不明确 | 模型在环节内自由发散 | 宪法只写"你是一个记账助手"，没写具体追问规则 |
| 拼接逻辑写进状态机 | 上下文管理器 替换零成本承诺失效 | 在 state-machine 里直接拼 turnHistory 而非通过 buildPromptContext() 接口 |
| 未预留两层模型选择机制 | ANALYZING 节点模型选择循环依赖，未来多模型扩展成本高 | 路由表只有状态路由，没有模型选择层 |
| 未实现 contractStore | 意图级纠错逃生舱无法实现 | 用户反馈意图错误时无法回滚至 ANALYZING |
| taskType 未在 N1/N3 阶段确定 | 运行时 taskType 动态切换导致工程复杂度爆炸 | 到写宪法阶段才发现"写诗"应该是 topic_based，但调度器逻辑已按 field_based 写死 |
| askingField 语义未按 taskType 区分 | 上下文管理器 锚点模式错乱 | topic_based 任务用字段级硬门控，导致主题相关历史轮次被淘汰，模型重复追问 |
| topicEvolution 未启用分层留存 | minor 级 event 无限膨胀 | 态控调度器在追加 event 时未执行 changeLevel 判定 |
| topicIndex 未实现 | 跨会话主题检索失败 | 用户说"继续上次那个故事"时无法匹配 |
| 主题名未经用户确认 | 系统自作主张匹配错主题 | 用户想写专利B，系统匹配到了专利A |
| 领域规则未经用户确认 | 错误规则被自动应用，污染新主题 | 规则提炼引擎误提了偶然模式，自动注入到所有同类型新主题 |

### 第三步：写宪法（v8.0 适配 v5.0 三层宪法体系）

#### 3.1 目标

为每个状态编写行为规则。**v5.0 核心变更**：宪法体系从四层（根/公共/环节/领域）简化为三层（根/环节/领域）+ 平行架构机制层。公共规则取消，内容分拆入根宪法和环节宪法。

#### 3.2 宪法体系总览（v5.0 三层结构）

| 层级 | 名称 | 约束对象 | 是否可修改 | 说明 |
|------|------|---------|-----------|------|
| 第0层 | 根宪法 | **所有环节模型**（ANALYZING + IN_SESSION） | 设计者写死，不可运行时修改 | 不可逾越的系统红线 |
| 第1层 | 环节宪法 | 特定环节模型 | 设计者静态编写（默认）；开放型环节可运行时动态生成 | 第一条强制为本环节任务定义 + off-task 声明 |
| 第2层 | 领域规则 | 特定主题类型 | 用户确认后生效，可随时修改或废弃 | 从 topicEvolution + 关联机制异步提炼的经验法则 |
| — | **架构机制** | **调度器/DET** | 设计者定义 | 元指令集、降级链、仲裁权、节点转换守卫——约束调度器和DET行为，非模型约束 |

**v7.3→v8.0 公共规则拆解**：原来三条公共规则的内容已分拆入根宪法和环节宪法——纯JSON输出→根宪法第2条，坚持真实判断→根宪法第3条，一次聚焦→根宪法第1条（环节隔离覆盖），追问模板→环节宪法 @section ask-rules。

#### 3.3 根宪法（v5.0 精简为 4 条）

约束所有环节模型（ANALYZING + IN_SESSION），由设计者写死，不可运行时修改，不可被环节宪法覆盖。

| 编号 | 条款 | 内容 | 可被环节宪法覆盖 |
|:--:|------|------|:---:|
| 1 | **环节隔离** | 只能做本环节任务，非本环节任务返回 off-task。本环节任务由环节宪法第一条定义。ANALYZING 只能做意图识别。IN_SESSION 只能做本环节任务。 | ❌ 不接受覆盖 |
| 2 | **输出格式** | 模型实例返回调度器的结果必须是符合 turnType schema 的结构化 JSON（含 askingField 双模式语义 + changeLevel 预判字段 + changeLevelReason 理由字段，两种 taskType 模式均适用） | ❌ 不接受覆盖 |
| 3 | **收敛义务 + 坚持真实判断** | 所有 LLM 节点必须在当前环节内收敛，输出结构化 JSON。不伪造数据、不假装理解、不迎合用户、不无中生有。确认无误就坚持，确实错了就纠正。 | ❌ 不接受覆盖 |
| 4 | **不编造功能** | 所有 LLM 节点不得生成用户未交代、系统不具备的功能或信息。这是安全红线的真核心——不是场景特定的"禁止恶意内容"，而是全局硬约束"不许无中生有"。 | ❌ 不接受覆盖 |

**ANALYZING（分析态）的环节隔离特化**（v5.0 新增，v8.1 修正）：ANALYZING 的"本环节任务"定义为"意图映射"——将用户输入映射到已知 intent。v8.1 修正：ANALYZING 用强制选择+API logprobs（非 LLM 自由生成 confidence）。如果 logprobs probability 不足以确认任何已知 intent，返回 `intent=other`（低 probability，调度器按 probability 路由）。注意：ANALYZING 只产出 intent（意图），不产出 turnType（轮次类型）——turnType 是 IN_SESSION（会话中）的输出格式。`turnType=off-task`（偏离）是 IN_SESSION 内模型检测到偏离时的输出，不是 ANALYZING 的输出。ANALYZING 不得采集字段、不得追问、不得给出最终回复。

#### 3.3b 架构机制（v5.0 新增，约束调度器/DET）

以下四条从 v7.3 根宪法移出，属于架构规范，约束调度器和 DET 行为，**非模型约束**：

| 编号 | 机制 | 约束对象 | 内容 |
|:--:|------|:---:|------|
| M1 | **元指令集（v8.3：S3 + roomStateIndex）** | 调度器+内存管理+context-manager | 四类通用口令（wake/exit/cancel/switch），口令层 EXACT_MATCH→直切+ S3 会话回收+ roomStateIndex 注入。**v8.3 强化**：`buildRoomIndex()` 生成全窗口房间摘要注入 ANALYZING，使意图识别获得跨房间全局视野，可针对性建议（如"您记账房间还有一条未完成"）。switch="切断房间"触发完整房间切换+S3物理隔离+状态索引 |
| M2 | **降级链四项检查（v8.5 修正）** | 调度器/DET | L1结构校验+DET值域复验+logprobs检查+硬编码兜底。均不可缩减。**v8.5 移除跨任务延伸检测——跨任务是用户显式确认行为** |
| M3 | **仲裁权 DET 优先** | 调度器 | 调度器 DET 校验结果优先于模型内部校验结果。两者冲突时以调度器 DET 为准 |
| M4 | **节点转换守卫** | 调度器 | 定义环节间转换的触发条件和守卫规则。IN_SESSION 期间用户输入偏离时，调度器 DET 关键词扫描可辅助触发偏离标记，但最终判断由模型给出 |

**降级链四项检查详解**（v8.5：移除跨任务延伸检测）：

| 项 | 执行体 | 时机 | 对应约束 | 说明 |
|----|-------|------|------------|------|
| L1 结构校验 | DET | complete 时 | 根宪法第2条（输出格式） | JSON.parse + Schema 校验，失败→重试1次→熔断 |
| DET 值域复验 | DET | complete 时 | — | field_based 做值域校验（金额上限、时间格式、必填字段…），topic_based 做输出格式校验（JSON Schema / 文本格式 / 字数 / 结构…）。规则来源：环节宪法 @section validation 的结构化 JSON |
| logprobs 检查（v8.1修正） | DET | complete 时 | — | 两步均 DET：①ANALYZING 从 API logprobs 转换 probability < logprobs_threshold ②pattern 扫"硬撑式确认"词表。**冷启动期间（cold_start_observation_window 内，默认 50 会话数）采用软拦截——标记 warning 附加到返回中让用户可见，不直接拦截；冷启动结束后恢复硬拦截**。v8.1修正：原 confidence 因 LLM 自由生成不可靠（幻觉式自信）已废弃 |
| 硬编码兜底 | DET | 熔断时 | — | 所有校验失败后的最终兜底，硬编码回复 |

**高风险场景扩展：** 对于医疗预问诊、复杂客服、金融风控等高危场景，在四项检查后可追加 human_takeover 扩展层（详见附录 C）。四项检查是通用底线，human_takeover 是高风险场景的扩展，不可替换四项检查中的任何一项。

#### 3.4 环节宪法编写规范（v8.0 更新）

> **v7.3 的"公共规则"在 v8.0 已取消**。原公共规则内容分拆如下：纯JSON输出→根宪法第2条，坚持真实判断→根宪法第3条，一次聚焦禁止跨任务延伸→根宪法第1条（环节隔离），追问模板→环节宪法 @section ask-rules。

**v8.0 环节宪法变化**：

1. **第一条强制为首要内容**：环节宪法第一条必须定义"本环节任务是什么"并强调"非本环节任务返回 off-task"。这是根宪法第1条（环节隔离）的落地。
2. **追问模板下放到 @section ask-rules**：原来在公共规则里的追问模板，现在由各环节宪法的 @section ask-rules 自行定义。


##### 3.4.1 环节宪法结构模板（含 taskType 声明 + topicId 声明 + changeLevel 判定规则）

```
- topicId 是 topicEvolution 的主键，供后续节点读写主题演化记录
- topic_based 任务自动启用 topicEvolution 三层递进机制（分层留存 + 主题关联 + 异步规则提炼）

- strong：模型应优先遵循
- medium：模型可基于客户需求合理偏离
- suggestion：仅作参考
```

##### 3.5.2 @section validation 结构化 JSON 格式

**field_based 模式的 validation 写法：**

```json
{
"rules": [
{
"field": "amount",
"type": "range",
"min": 0,
"max": 999999,
"error_tag": "amount_too_large"
},
{
"field": "amount",
"type": "range",
"min": 0,
"exclusive_min": true,
"error_tag": "amount_invalid"
},
{
"field": "time",
"type": "format",
"pattern": "^(昨天|今天|明天|\\d+号)$",
"error_tag": "time_format_invalid"
},
{
"field": "time",
"type": "not_future",
"error_tag": "time_future"
},
{
"field": "quantity",
"type": "range",
"min": 0,
"exclusive_min": true,
"error_tag": "quantity_invalid"
}
],
"required_fields": ["category", "amount", "time"]
}
```

**topic_based 模式的 validation 写法：**

```json
{
"rules": [
{
"field": "lines",
"type": "array_length",
"exact": 4,
"error_tag": "line_count_violation"
},
{
"field": "lines",
"type": "char_count_per_element",
"exact": 7,
"error_tag": "char_count_violation"
},
{
"field": "rhyme",
"type": "custom",
"function": "check_rhyme_scheme",
"error_tag": "rhyme_violation"
}
],
"required_fields": ["title", "lines", "style", "theme"]
}
```

**规则类型说明：**

- **range：** 数值范围校验（min/max/exclusive_min/exclusive_max）
- **format：** 正则格式校验（pattern）
- **not_future：** 时间不未来校验
- **enum：** 枚举值校验（values 数组）
- **array_length：** 数组长度校验（exact/min/max）
- **char_count_per_element：** 数组每元素字符数校验（exact/min/max）
- **custom：** 自定义校验（function 字段指定 DET 函数名，需在调度器注册）
- **required：** 必填校验（在 required_fields 数组中声明）

**声明-实现一致性：** @section validation 的结构化 JSON 规则必须与 N5 调度器 DET 值域复验的实现一致。

##### 3.5.5 开放型环节的动态宪法生成（例外路径）

对于开放型环节（如写诗、创意写作等无法在设计时穷举所有约束的环节），保留 CONSTITUTION_GEN 子流程作为例外路径：

1. ANALYZING 识别到开放型 intent（如 write_poem，taskType=topic_based）
2. CONSTITUTION_GEN 节点根据 extracted 字段（如 style: "七言绝句"）动态生成局部宪法
- 可枚举子类型（如"七言/五言"）→ DET 规则引擎生成
- 开放子类型 → LLM + 根宪法校验生成
3. 可选：CONFIRM_CONSTITUTION 节点（低置信度时人确认）
4. 根宪法仲裁（CONFLICT_CHECK，DET）→ 通过后生效
5. 进入 IN_SESSION，使用动态生成的局部宪法

此路径是例外，不是默认——默认情况下局部宪法由设计者静态编写。

#### 3.6 状态间数据传递协议

| 转移路径 | 携带数据 | 清空数据 | 说明 |
|---------|---------|---------|------|
| LISTENING → ANALYZING | 用户原始输入 | 无 | 将输入传递给意图识别 |
| ANALYZING → IN_SESSION | intent, taskType, topicId（topic_based时）, extracted | 无 | 将意图、taskType、topicId 和已提取字段带入环节。ANALYZING 的 contract-in 副本同步写入 contractStore。topic_based 任务同时初始化 topicEvolution 记录 |
| IN_SESSION → EXECUTING | result（complete 时） | 无 | 将完整结果写入数据库/输出文件 |
| IN_SESSION → ANALYZING | offTaskInput | activeSession | 偏离→重新意图识别 |
| IN_SESSION → LISTENING | 无 | activeSession | 放弃。topic_based 任务在 giveup 时使用模型判定的 changeLevel 追加 event，stateSnapshot 标记为 abandoned，推入提炼待处理队列 |
| EXECUTING → LISTENING | _lastRecord, topicEvolution 更新（topic_based时） | 无 | 保留上一条记录用于"记错了"。topic_based 任务使用模型判定的 changeLevel 追加 event，标记 checkpoint，推入提炼待处理队列 |
| 任意状态 → ANALYZING（逃生舱） | contractStore.analyzingContractIn | activeSession | 意图级纠错：强制回滚至 ANALYZING 重新分类（可能伴随 taskType 变化） |

**关键规则：** 状态间的数据传递必须是显式的。不能依赖 LLM 的记忆来传递数据。环节内的数据（collectedFields、turnHistory）由态控调度器的 activeSession 维护，环节完成后清空。

#### 3.7 跨节点契约继承（可选能力）

在物理隔离上下文的同时，允许用户将旧节点的 contract-out 选择性传递给新节点的 contract-in：

- **默认行为：** 不传递（保持纯净初始态）
- **启用条件：** 用户主动声明"保留 XX 信息"，或场景需要跨环节传递上下文
- **传递内容：** 旧节点 contract-out 中对应字段，非全部上下文
- **注入上限：** 不超过新节点 contract-in 总字段数的 contract_inheritance_limit（默认 0.3，即 30%）。此为工程经验值——超过此比例时下游节点的认知负担开始显著增加、幻觉率上升。可通过 tunable 槽位配置
- **注入前校验：** 需经过根宪法校验
- **taskType 跨环节传递：** 如果旧节点是 field_based、新节点是 topic_based（或反之），传递的字段需做语义映射——field_based 的结构化字段注入 topic_based 时，作为主题空间的约束条件注入；topic_based 的创作输出注入 field_based 时，提取可结构化的字段注入。跨 taskType 传递需在环节宪法中声明映射规则

#### 3.8 守卫规则

##### 3.8.1 全局守卫（所有状态通用）

| 输入模式 | 触发动作 | 优先级 |
|---------|---------|-------|
| 唤醒词 | IDLE 态下进入 LISTENING | 最高 |
| 退出词 | 放弃当前流程，进入 CLOSING | 最高 |
| 取消词 | 放弃当前子流程，回到 LISTENING | 最高 |

全局硬匹配采用可配置的浅匹配器（子串匹配/编辑距离/同义词组），词表是产品参数，需维护日志。此为元指令集的口令层。

##### 3.8.2 DET 关键词扫描

每轮用户输入执行 DET 关键词扫描，检测跨意图词汇，附加 offTaskSuspicion 标记到模型输入中。不强制覆盖模型判断——模型做出最终偏离判定。

##### 3.8.3 LISTENING 态守卫

- 合法输入：任何用户输入（都需要走意图识别）
- 异常处理：无

##### 3.8.4 IN_SESSION 态守卫

- 调度器行为：接收模型返回的 turnType（含 changeLevel 预判），进行状态调度
- DET 值域复验：模型返回 complete 时，态控调度器对 result 执行第二道值域校验/格式校验（四项检查 + 硬编码兜底，按 taskType 分支）
- 偏离处理：模型返回 off-task 时，暂存 collectedFields（默认 24h），重新意图识别。topic_based 任务在 off-task 时仍追加 event（含使用模型判定的 changeLevel）
- 意图级纠错逃生舱：用户反馈意图完全错误时，从 contractStore 取出 ANALYZING 原始 contract-in，强制回滚至 ANALYZING

#### 3.9 设计原则

1. **根宪法不可逾越：** 四条最低条款由设计者写死，不接受运行时修改。
2. **环节宪法约束模型自治边界：** 宪法中明确模型在环节内能做什么、不能做什么、校验规则是什么、偏离检测规则是什么。
3. **架构机制铁律不可只靠 LLM 自律：** 态控调度器侧必须有 L1 结构校验 + DET 值域复验 + logprobs 检查作为第二道防线。坚持真实判断由环节宪法在 prompt 中约束，不由质检员扫关键词。
4. **所有环节宪法统一 turnType 格式（含 changeLevel 字段）：** 确保调度器用同一套逻辑处理所有环节的模型返回。
5. **环节宪法 @section ask-rules 采用通用追问模板：** 类型、主题、格式三个维度。具体场景的追问示例放在各环节宪法的 @section ask-rules 中。
6. **元指令集分两层：** 口令层（EXACT_MATCH）+ 语义确认层（ANALYZING 判 meta_suspect→二次确认），自然语言不入口令词表。
7. **taskType 必须在环节宪法中声明：** @section taskType 为必填项，决定 askingField 语义、上下文管理器 锚点模式、DET 复验方式。topic_based 任务自动启用 topicEvolution 三层递进机制（分层留存 + 主题关联 + 异步规则提炼）。
8. **topicId 声明：** topic_based 环节宪法的 @section output-schema 必须声明 topicId 字段。ANALYZING contract-out 必须包含 topicId。
9. **changeLevel 判定规则声明：** topic_based 环节宪法的 @section output-schema 必须声明 changeLevel 的判定规则（invalid/major/minor 的操作化定义）。
10. **领域规则优先级声明：** topic_based 环节宪法可通过 @section domain-rule-priority 声明领域规则的优先级（strong/medium/suggestion）。

### 契约对齐


每个状态的输出必须显式声明格式、值域、可选性、副作用标记。下游状态在接收时，必须做输入契约校验。校验不通过的不进入 LLM，直接退回上游或走异常处理。

调度器与模型实例之间的核心契约是：调度器注入"任务描述 + 局部宪法 + 拼接好的上下文（topic_based 模式额外包含三层注入包：已生效领域规则 + 关联主题摘要 + topicEvolution 分层包）"；模型返回"符合 turnType schema 的结构化 JSON（含 askingField 双模式语义 + changeLevel 预判字段，topic_based 的 ANALYZING 阶段额外返回 topicId）"。

调度器不信任模型返回的任何字段值，只信任 structure 的存在性——structure 本身要先存在（L1 校验），然后关键字段值需要过 DET 复验（第二道值域校验/格式校验，按 taskType 分支）。

### 宪法维护指南


### 7.1 新增一个场景

1. 在意图识别宪法中注册一个新的 intent，标注 taskType（field_based 或 topic_based）
2. 编写对应的环节宪法，包含 taskType 声明、角色、字段规则/主题边界、值域校验/格式校验（第一道，结构化 JSON）、偏离检测、统一输出格式。topic_based 环节宪法需在 @section output-schema 中声明 topicId 字段和 changeLevel 判定规则，可选声明 @section domain-rule-priority
3. 在调度器中注册 intent 到环节宪法的映射。topic_based intent 自动启用 topicEvolution 三层递进机制（分层留存 + 主题关联 + 异步规则提炼）
4. 在路由表中配置意图级模型绑定（第二层，如需专用模型——topic_based intent 更可能需要）
5. 不需要修改其他环节宪法

### 7.2 修改某个场景的行为规则

1. 只修改对应的环节宪法
2. 不需要修改其他环节宪法
3. 不需要修改调度器状态转移逻辑
4. 如果修改 taskType（field_based ↔ topic_based），视为 major 版本变更，需全量回归测试

### 7.3 架构机制修改（v8.0 替代原"公共规则维护"）

v5.0 已取消公共规则。原公共规则内容分拆入根宪法和环节宪法。架构机制（M1-M4）的修改：

1. **先看影响面：** 列出受此机制影响的所有节点 + 受影响节点的转移边
2. **trace 回放验证：** 用受影响节点的历史会话、N14 审骨架的转移边覆盖数据，模拟机制变更后的行为变化
3. **checklist 检查：** 架构机制变更后，必须跑一遍宪法层面检查
4. **灰度上线：** 架构机制变更在受影响的节点上分批生效

### 7.4 根宪法修改

根宪法的四条最低条款不可修改。架构机制 M2（降级链四项检查）可扩展不可缩减。修改根宪法或架构机制需要全量回归测试。

M2 降级链修改特别注意：四项检查均不可缩减。如需扩展（如追加 human_takeover），只能在四项检查之后追加。

### 7.5 宪法版本治理与结构化改造

当环节宪法数量 ≥ 8、或两份宪法的任务边界出现重叠、或架构机制修改后回归测试耗时 > 30 分钟时，应启动宪法结构化改造（YAML/JSON Schema 表述规则约束，text 只保留生成风格指引）。

## 三、用户输入处理与记录

```
用户输入
│
├─ 第0层：口令精确匹配（元指令集口令层，EXACT_MATCH→直切，非匹配→放行ANALYZING）
│   匹配唤醒词 → 进入 LISTENING
│   匹配退出/取消词 → 进入 CLOSING 或放弃当前流程
│   匹配器为可配置浅匹配器
│   注：自然语言疑似切换→ANALYZING判meta_suspect→二次确认后才切（语义确认层）
│
├─ 第0.5层：DET 关键词扫描（0 次 LLM）
│   检测跨意图词汇 → 附加 offTaskSuspicion 标记
│   不强制覆盖模型判断
│
├─ 第1层：意图识别（LISTENING 态，1 次 LLM）
│   拼接【意图识别宪法】→ 调 LLM
│   拿到 {intent, taskType, subType, logprobs（概率分布/来自API logprobs）, extracted（已提取信息）}
│   field_based 任务：
│     a. 先查 sessionCheckpoints 是否存在未过期进度

│   topic_based 任务：
│     a. ANALYZING 阶段 LLM 生成候选 topicName + keywords
│     b. 调度器执行关键词匹配（基于 topicIndex 表），检测是否已有相近主题
│     c. 若匹配到已有主题 → 系统展示候选主题名，等待用户确认（续写/重选/新建）
│     d. 若未匹配到 → 系统展示生成的候选主题名+keywords，等待用户确认（确认/修改/搜索已有）
│     e. 用户确认后，态控调度器生成 topicId，初始化 topicEvolution 记录

│   状态机根据 intent 分发到对应环节，taskType 固化
│   路由表第二层（意图级绑定）覆盖默认模型

│
└─ 第2层：环节内模型自治（IN_SESSION 态）

→ 注入模型实例

→ 模型顺带输出 changeLevel 预判（major/minor/invalid）
→ 返回 turnType（含 askingField 双模式语义 + changeLevel 字段）

turnType="complete" → 四项检查（按 taskType 分支，0 次 LLM）
topic_based 任务在节点 exit 时：
- stateChange 摘要由 N10 后台异步消费 ruleMiningQueue 时生成（不在态控调度器同步链路内调 LLM）
- 使用模型判定的 changeLevel 追加 event（major永久/minor压缩/invalid归档）
- 将主题信息推入规则提炼待处理队列
- 不触发任何提炼逻辑

2. 提取候选规则，存入 ruleCandidate 池

5. 用户确认后的规则存入 contractStore.domainRules 库
```

### 5.1 第0层：口令精确匹配（元指令集口令层）

**核心原则：元指令是口令，不是自然语言。** 口令含系统预设前缀或特定组合（如 `@@exit`、`/quit`、`小安退出`），由 N7 设计者定义为架构机制 M1 词表。口令不诞生于日常用语，自然语言中极低频。

匹配规则：
- **EXACT_MATCH：用户输入与口令词表项完全一致** → 直接触发节点切换，0 次 LLM。用户主动说出约定令牌 = 明确意图
- **非精确匹配（含"退出""算了""不玩了"等自然语言）→ 不切，放行至 §5.3 ANALYZING**

自然语言中的疑似切换意图（如"算了不聊柜子了"）由 ANALYZING LLM 判定为 intent="meta_suspect" → 系统发起二次确认（"你是想退出当前任务吗？"）→ 用户明确回复"是"/"对"/"确认"后才切。此路径不需快照保护——用户确认后才切，不存在误切。

纪律：**口令 = 约定令牌，非口令 = 自然语言 = 不切。** 语言交互中的意图确定性来自口令，不来自对自然语言的猜测。

### 5.2 第0.5层：DET 关键词扫描

每轮用户输入执行 DET 关键词扫描，检测跨意图词汇。标记附加到模型输入中作为提示，不强制覆盖模型判断。理由：关键词匹配能快速发现"用户说了'查一下'"，但无法区分是真正的意图切换还是口语中的随口一提——最终判断留给模型。

**off-task 处理**：第0.5层仅做关键词标记，真正的偏离检测与处理流程详见 §6.1（环节内偏离处理）。

### 5.3 第1层：意图识别

LISTENING 态收到输入 → 调 LLM 做意图识别。LLM 在意图识别宪法约束下做有限展开，将弥散输入收敛到明确的 intent 枚举值 + taskType + 置信度。出口被 contract-out schema 硬卡。

**语义模板层（元指令集第二层）：** 口令层未匹配的自然语言切换意图（如"算了不聊柜子了，帮我写首诗"），由 ANALYZING 的 LLM 在意图识别时顺带判断——若判定为元指令，intent 标记为 meta_instruction，状态机据此触发节点切换。

**意图稳定性防抖动：** 同一会话中同一用户在 60 秒内 intent 在两种类型间来回切换 ≥ 3 次 → 不重判，锁定首次判定的 intent，追问确认用户真实意图。

**extracted 字段合并：** 意图识别阶段可顺带提取已明确的字段，作为环节的初始上下文注入模型实例，减少一轮追问。

**taskType 固化：** 意图识别产出的 taskType 在此阶段从 N1 初判值固化为确定值，写入 activeSession.taskType，后续节点按此执行对应分支。

**topicEvolution 初始化与跨会话匹配：**

- topic_based 任务：ANALYZING 额外生成 topicSlug + topicName + keywords，用户确认后组合成 topicId，态控调度器在 contractStore 中初始化 topicEvolution 记录
- 跨会话匹配：若用户输入含"继续上次的XX"等续写信号，ANALYZING 用关键词匹配（基于 topicIndex 表）检索已有主题，关键词匹配不到时降级为 embedding 语义匹配。匹配阈值由 topic_evolution_match_threshold 控制

### 5.4 第2层：环节内模型自治

调度器通过 上下文管理器.buildPromptContext() 拼接上下文窗口（topic_based 模式额外包含三层注入包：已生效领域规则 + 关联摘要 + topicEvolution 分层包），与局部宪法一起注入模型实例。模型在局部宪法约束下自主完成多轮对话。

**环节内硬解析（field_based 模式）：** 模型在环节内优先尝试硬解析（金额正则匹配、时间词典匹配），硬解析成功→值域校验→通过则直接填入。硬解析失败→追问用户确认。

**环节内主题探索（topic_based 模式）：** 模型在环节内自主决定主题探索路径，每轮生成动态主题锚点标签作为 askingField。

**值域校验/格式校验（第一道，模型内部）：**

- field_based：金额 ≤0 拦截、金额 >999999 拦截、时间未来拦截、数量 ≤0 拦截
- topic_based：字数/结构/风格等格式约束校验
- 校验失败时模型返回 turnType="validation_failed"，态控调度器复用模型 message 转发用户

**偏离检测：**

- **保障1（主力——宪法强制 + 上下文管理器优先注入）：** 根宪法第1条（环节隔离）+ 上下文管理器最高优先级注入环节隔离声明，要求模型每轮首要判断"是否本环节任务"。不属于本环节 → 立即返回 off-task（优先级最高）。大幅降低漏判率。
- **保障2（辅助——ANALYZING 重新识别验证）：** 模型返回 off-task 后，调度器不直接路由。DET 只做非空检查（门卫，不做语义验证），送回 ANALYZING 重新识别 offTaskInput。ANALYZING 的重新识别结果就是验证（v8.1 修正：DET 做不了语义验证，LLM confidence 不可靠）。
- **显式切换口令（M1 switch，v8.1 新增）：** 用户可主动说"切断房间"口令 → DET 硬匹配（EXACT_MATCH，0 次 LLM）→ 直接路由 ANALYZING。显式口令是首选切换机制（100% 确定性），off-task 检测是兜底。
- **设计原则：** 不追求 100% 拦截，而是"降低漏判 + 拦住误判 + 回溯兜底"。用确定性兜底追求 100% 反而会破坏环节自治。
- **DET 不主动介入：** 第0.5层只标记不判断。DET 不在环节内主动触发 ANALYZING——这是对模型自治权的僭越。DET 只在 IN_SESSION 出口后做事后验证。

**changeLevel 预判 + event 追加：**

- 模型在返回 turnType 时顺带输出 changeLevel 预判字段（major/minor/invalid）
- IN_SESSION exit 时：态控调度器使用模型判定的 changeLevel 追加 event；stateChange 摘要由 N10 后台异步生成
- EXECUTING 阶段：同上，stateSnapshot 标记为 checkpoint
- giveup 时：stateSnapshot 标记为 abandoned
- off-task 时：记录偏离前的状态，stateSnapshot 保持 active
- 分层留存：major 级永久保留，minor 级超过阈值时触发汇总压缩，invalid 级迁入归档表

### 5.5 第二道防线：四项检查

模型返回 turnType="complete" 时，态控调度器执行第二道防线。这是确定性代码，零 LLM 调用（跨延伸例外层除外）。复验失败 → 拦截，进入 CLARIFYING。

**检查1：L1 结构校验（对应根宪法第2条输出格式）**

- 执行体：DET
- 执行方式：JSON.parse + Schema 校验
- taskType 影响：无差异
- 失败处理：重试1次→仍失败→熔断→硬编码回复

**检查2：DET 值域复验（field_based）/ 输出格式校验（topic_based）**

- 执行体：DET
- 执行方式（按 taskType 分支）：
- field_based：值域校验（金额上限、时间格式、必填字段…），规则来源：环节宪法 @section validation 的结构化 JSON
- topic_based：输出格式校验（JSON Schema / 文本格式 / 字数 / 结构…），规则来源：环节宪法 @section validation 的结构化 JSON
- 失败处理：进入 CLARIFYING，复用模型 message 转发用户

**检查3：logprobs 检查**

- 执行体：DET（两步均 DET，0 LLM）
- 执行方式：
- 步骤①：数值判断 probability < logprobs_threshold（DET 纯数值比较，来自 logprobs 转换，threshold 从 tunable 读取，默认 0.4）
- 步骤②：轻量 pattern 扫 message 是否含"硬撑式确认"词表（如 `/您说得对|没错|好的给您|已为您/`，词表 tunable 可扩展）
- 两步任一命中即标记 pretend_to_know，拦截
- taskType 影响：无差异（两种模式都执行）
- 失败处理：进入 CLARIFYING，复用模型 message 转发用户

**检查4（v8.5 已移除）：跨任务延伸检测**——跨任务是用户显式确认行为（M1 switch 口令或 off-task→ANALYZING 重新识别→用户确认），不需要降级链层自动化检测。详见 §6.1 off-task 双重保障 + M1 switch 口令。

**检查5：硬编码兜底**

- 执行体：DET
- 时机：上述四项检查均失败/熔断时
- 说明：所有校验失败后的最终兜底，硬编码回复

**仲裁权（架构机制 M3）：** 当模型内部校验（第一道）与调度器 DET 复验（第二道）结果冲突时，以调度器 DET 为准。

### 5.6 L1 校验 + DET 复验

L1 结构校验（完成时）：对 result 执行 JSON.parse + Schema 校验。失败→重试1次→仍失败→熔断→硬编码回复。

DET 值域复验/格式校验（完成时）：对 result 执行第二道 DET 校验（按 taskType 分支，含 logprobs 检查）。复验失败→拦截，进入 CLARIFYING。

### 6.1 环节内偏离处理

**v8.0 off-task 双重保障流程**（v8.1 修正：2026-07-17）：

当模型返回 turnType="off-task" 时：

1. **保障1（主力，已完成）：** 模型基于根宪法第1条 + 上下文管理器优先注入环节隔离声明，首轮即识别偏离
2. **保障2（辅助，ANALYZING 重新识别验证）：** 调度器不直接路由 → DET 只做非空检查（门卫）→ 送回 ANALYZING 重新识别 offTaskInput → ANALYZING 的重新识别结果就是验证
3. **显式切换口令（M1 switch，首选）：** 用户说"切断房间"→ DET 硬匹配 → 直接路由 ANALYZING（0 次 LLM，100% 确定性）。口令是首选，off-task 检测是兜底
4. **设计原则：** DET 不做语义验证（做不了），语义验证由 ANALYZING 重新识别完成。DET 不僭越模型自治权



- 调度器暂存 collectedFields 到 _pendingCollectedFields
- field_based：collectedFields 为已采集的字段
- topic_based：collectedFields 为已讨论的主题摘要
- 暂存时长由系统设置（默认 24 小时），超时自动清理
- 将 offTaskInput 送入意图识别重新分发
- 下次同一用户进入同一环节时，态控调度器询问是否继续上次未完成的记录
- topic_based 任务在 off-task 时，态控调度器自动追加 event 到 topicEvolution（记录偏离前的状态，使用模型判定的 changeLevel），stateSnapshot 保持 active

### 6.2 用户明确离开时

- 直接作废：清空 activeSession
- 不留草稿：不保存未完成记录，不恢复现场
- _lastRecord 不清空：用于"记错了"场景
- topicEvolution 不清空：topicEvolution 跨会话持久化，用户离开后仍可通过"继续上次的XX"恢复（分层注入包确保上下文窗口可控）

### 6.3 用户放弃

模型返回 turnType="giveup" → 调度器清空 activeSession → LISTENING。topic_based 任务在 giveup 时使用模型判定的 changeLevel 追加 event，stateSnapshot 标记为 abandoned，推入提炼待处理队列。topicEvolution 不清空。

### 6.4 意图级纠错（逃生舱）

当用户反馈表明当前意图完全错误时（如"我不是要记账，我是要查询"）：

1. 状态机从 contractStore 取出 ANALYZING 节点的原始 contract-in 副本
2. 重新实例化 ANALYZING 节点（注入原始 contract-in + 宪法）
3. 重新执行意图识别，产生新的 intent（可能伴随 taskType 变化）
4. 清空当前 activeSession，按新 intent 和新 taskType 重新分发

此机制需要 contractStore 持久化保存 ANALYZING 的原始 contract-in。contractStore 独立于执行体实例的生命周期——Agent 消亡后，其产出的契约副本仍可被回滚流程访问。

## 四、受控调参系统

### 8.1 系统设置界面

每个宪法中声明的 @section tunable 参数，在智能体的系统设置界面中自动生成配置表单。界面由宪法中的 tunable 声明自动生成，无需手动开发配置页面。每个参数显示其 description、当前值、允许范围/选项。"检测冲突"按钮在保存前执行一次完整冲突检测。

### 8.2 冲突检测引擎

当用户调整参数并保存时，系统遍历该宪法的 conflict_check 规则。规则分两类：`single_param`（range/enum 硬校验）→ 违规硬拒绝；`cross_param`（跨参数语义校验，v4.7 新增）→ 违规默认 warning，弹建议值，允许用户强制保存。cross_param 初始含 5 条规则（上下文预算 / 阈值排序 / 冷启动窗口 / 规则提炼触发 / 置信度冻结），详见总纲 v5.5 §10.1。冲突检测引擎是确定性代码，不调 LLM（single_param 用 json-rules-engine 或 Drools 实现，cross_param 用沙箱 eval 做 compute 表达式求值）。

### 8.3 变更日志规范

每次参数调整必须记录：时间戳、操作者、宪法名称、变更详情（参数名、旧值、新值、调整原因）、冲突检测结果。日志不可篡改，作为审计追溯的最终依据。

### 8.4 调整生效时机

参数保存后，下一轮新会话开始生效。当前正在进行的会话继续使用旧参数值，不受影响。设计理由：避免运行时突然改变规则导致正在进行的会话行为不一致。

### 8.5 与纯 LLM 运行时动态调整的区别

| 维度 | 纯 LLM 动态调整 | 态控 tunable 槽位 |
|------|---------------|------------------|
| 调整方式 | 在 prompt 里写条件 | 在系统设置里改参数 |
| 调整范围 | 无限制 | range/enum 硬限制 |
| 冲突检测 | 无 | 确定性代码检测 |
| 审计 | 难 | 变更日志，每次调整可追溯 |
| 生效时机 | 下一轮对话立即生效 | 保存后下一轮新会话生效 |

## 五、迭代回路

### 10.1 触发条件

六种情况触发响应：

| 触发类型 | 具体信号 | 处置路径 |
|---------|---------|---------|
| 边界-状态矛盾 | 状态节点处理的输入落在"不做什么"内且无处理路径 | 回溯→重审边界（补全护栏策略） |
| 宪法-状态矛盾 | 规则逻辑不自洽（逻辑bug，不适用累积阈值） | 回溯→修正宪法 |
| 守卫-边界矛盾 | 合法输入被误杀 | 回溯→调整词表或护栏 |
| 参数漂移信号 | 指标连续偏离基线，但边界/宪法/守卫均按预期工作 | 参数调节评估 |
| taskType 误判 | taskType 初判错误导致后续节点行为异常 | 回溯→第一步重判 taskType |
| 边界覆盖缺口 | 功能范围外输入无处理路径或护栏响应质量不合格 | 回溯→第一步补全护栏策略或优化护栏响应语 |

**关键分流逻辑：** 如果系统在"正确执行当前边界和规则"的前提下，监控数据仍然不好看——先去参数调节宪法看是不是参数漂移。不要回溯，不要改边界，去调参数。

### 10.2 回溯时带回的数据

回溯必须携带下游的具体矛盾实例：触发矛盾的状态节点定义、无法自洽的宪法条款原文、误杀统计、词表调整方案记录、taskType 误判的具体案例、边界覆盖缺口的具体输入类型等。

### 10.3 重审边界的合法决策

只有三种合法决策：

1. **收缩"不做什么"：** 将原来不做的事移入"做什么"
2. **重新定义边界护栏策略：** 从"硬拒"升级为"软着陆"。硬约束：软着陆的分支目标必须已在状态转移图中存在
3. **承认该输入类型为合法但降级处理：** 分配显式的降级路径

### 10.4 迭代终止条件与过度拟合止损

- 越界率 < 目标阈值
- 无新增矛盾
- 或达最大回溯轮次（默认 3 轮）

同类矛盾信号须在 N 次（建议 N≥3）真实会话中出现，或单次撞击的误杀率 > X%，才触发回溯。单一样本不构成回溯依据。例外：宪法-状态层面的自洽性矛盾属于逻辑 bug，不适用累积阈值规则。taskType 误判属于结构性错误，不适用累积阈值规则——一旦发现立即回溯。边界覆盖缺口属于结构性错误，不适用累积阈值规则——一旦发现立即回溯。

### 10.5 执行体错配矛盾

| 检测信号 | 现象 | 处理 |
|---------|------|------|
| 信号A（模型校验过松） | 模型第一道校验通过率异常但 DET 复验拦截率异常升高 | 回溯，检查环节宪法校验规则与调度器 DET 规则一致性 |
| 信号B（模型校验过严） | 模型频繁返回 validation_failed 但用户后续修正率低 | 回溯，检查环节宪法校验规则是否过于严格 |
| 信号C（模型假装懂） | logprobs 拦截率异常升高（>10%） | 回溯，检查环节宪法 prompt 是否过于鼓励"给出答案" |
| 信号D（模型跨任务延伸） | 跨任务延伸拦截率异常升高（>5%） | 回溯，检查收敛方向或字段白名单/主题边界 |
| 信号E（意图识别不准） | 意图级纠错触发率异常升高（>5%） | 回溯，检查意图识别宪法 |
| 信号F（topic_based 主题漂移） | 模型频繁重复追问已讨论过的主题（上下文拼接命中率低） | 检查 topic_anchor_similarity_threshold 或 topic_keyword_overlap_threshold |
| 信号G（taskType 误判） | taskType 误判率异常升高（>2%） | 回溯第一步 |
| 信号H（边界覆盖缺口） | 功能范围外输入无处理路径比例>5% 或 护栏响应质量不合格占比>10% | 回溯第一步 |
| 信号I（topicEvolution 机制失效） | topicEvolution event 写入频率过低（<1 event/会话） | 检查调度器是否正确追加 event |
| 信号J（topicEvolution 跨会话匹配失败率高） | 匹配成功率过低（<30%） | 检查 topic_evolution_match_threshold 或 stateSnapshot 描述 |
| 信号K（topicEvolution minor 压缩异常） | 压缩频率过高（>1次/会话）或注入包大小持续增长 | 检查 topic_evolution_minor_compaction_threshold |
| 信号L（invalid 判定过激） | invalid 丢弃率过高（>30%），正常修改被误判 | 检查 changeLevel 判定规则 |
| 信号M（topicIndex 检索失败率高） | 用户明确提到主题名但关键词匹配未命中 | 检查 topicIndex 索引是否正常更新 |
| 信号N（领域规则误导率高） | 领域规则被注入后，用户频繁修正或否定模型输出 | 检查该规则是否需要修改或废弃 |
| 信号O（规则提炼噪声率高） | 候选规则确认率过低（<30%） | 调整 domain_rule_validation_threshold |
| 信号P（changeLevel 判定异常） | invalid 占比 > 30% 或 major 占比 < 3% 或 changeLevelReason 空值率 > 10% | 检查环节宪法的 changeLevel 判定规则；若 changeLevelReason 持续为空，检查模型是否未理解输出格式要求 |
| 信号Q（规则提炼队列积压） | 待处理队列持续增长 | 调整 rule_mining_schedule、rule_mining_batch_threshold 或 rule_mining_batch_size |

### 收束


态控闭环设计方法论的核心是：第一步划边界，第二步拆状态（含 taskType 固化 + topicEvolution 三层递进初始化），第三步写宪法（v5.0 三层体系：根宪法4条 + 环节宪法 + 架构机制 M1-M4 + 领域规则）。编程规则宪法（附录E）作为全局元约束贯穿全管线。——顺序不能乱，每一步都有明确的交付物。三步法的底层逻辑是认知局部性原理——每个节点只接收最小充分条件集，不喂全局。

v7.1：topicEvolution 的三层递进——分层留存让记忆不膨胀（模型判定 changeLevel），主题关联让知识不孤立（关联在元数据层，隔离在数据层），异步规则提炼让智能体从菜鸟变专家（滞后但可靠，不在状态机链路内阻塞用户交互）。**v7.1 将领域规则从平铺列表升级为规则网络图**——规则有来源（scenario_seed/user_initiated/model_inferred）、有权限（L0-L3）、有层级（topicPath 前缀匹配）、有关联边（depends_on/conflicts_with/strengthens），注入方式也从全量注入改为按 topicPath 子图匹配精准注入。三层均遵循"自动生成 + 用户确认"的人机协作原则。领域规则不是设计者拍脑袋定的，是从实际项目中异步长出来的。

topic_based 任务的"契约"定义：field_based 的契约是字段集（contract-out JSON），topic_based 的契约是按信息净价值分层留存的演化记录（topicEvolution）+ 轻量级元数据关联（关联摘要）+ 异步凝结的经验法则（领域规则）。不是按时间新久留，是按信息净价值留——大版本永久保留（结构性的），小版本汇总压缩（渐进的），无效过程归档备查（审计的）。

### 动态验证方法


### 9.1 验证阶段的职责分工

| 阶段 | 验证方法 | 责任人 | 覆盖率目标 |
|------|---------|-------|----------|
| 设计阶段 | 检查清单 | 设计者 | 100%（全量自查） |
| 灰度期（0-2 周） | fuzzing | 开发+QA | ≥60%（必须跑出第一份转移频率热力图） |
| 运行期 | 全自动 | 运维+产品 | ≥90%（真实流量覆盖） |

### 9.2 对话树 fuzzing

用一批半随机化的输入序列跑状态机，收集 (fromState, inputPattern, toState) 的转移覆盖率。未覆盖的边 = 要么是死代码，要么是缺失测试用例。fuzzing 需同时覆盖 field_based 和 topic_based 两种 taskType 的环节，以及功能范围内和功能范围外两种输入类型。

### 9.3 会话日志回放

把真实用户的会话按 (state, input) → nextState 对齐标注，识别高频异常转移和僵尸路径。

### 9.4 环节内监督指标

**执行体调用链追踪（OpenTelemetry 集成）：** 每次 DET/LLM/HUMAN 调用生成 OpenTelemetry span，含字段：trace_id / span_id / parent_span_id / executor_type / executor_name / start_time / end_time / status / input_summary / output_summary / error_detail。运维侧用 Jaeger/Zipkin 可视化调用链。监控指标直接从 trace 提取。

**监督指标清单：**

- **校验失败率：** 模型返回 turnType="validation_failed" 的频率。field_based 和 topic_based 分别统计
- **偏离率：** 模型返回 turnType="off-task" 的频率
- **DET 复验拦截率：** 调度器第二道 DET 复验拦截的 complete 结果比例。field_based 和 topic_based 分别统计
- **CLARIFYING 触发频率：** 调度器进入 CLARIFYING 状态的频率，用于运维监控
- **上下文拼接命中率：** 模型在收到拼接后的上下文后，是否需要重复追问已采集的字段或已讨论的主题。topic_based 的三层注入 + embedding 两阶段检索命中率，field_based 的字段级硬门控命中率。分别统计
- **logprobs 拦截率：** 因 confidence < threshold 且 message 含"硬撑式确认"被拦截的 complete 结果比例。拦截率>10% → 模型频繁假装懂；拦截率<1% → 阈值可能过低
- **跨任务延伸拦截率：** 因 result 含不应产出内容被拦截的比例。field_based 拦截率>5% → 收敛方向不清晰；topic_based 拦截率>5% → 主题边界不清晰
- **意图级纠错触发率：** 用户触发逃生舱的频率。触发率>5% → 意图识别准确率不足；触发率<0.1% → 逃生舱入口不明确
- **taskType 误判率：** 意图识别阶段 taskType 判定错误的频率。误判率>2% → 回溯第一步
- **边界覆盖完整性：** 功能范围外输入的处理路径覆盖率 + 护栏响应质量分布
- **topicIndex 检索命中率：** 用户说"继续上次的XX"时，关键词匹配命中的比例。命中率过低（<50%）→ topicName 或 keywords 生成质量不足
- **主题关联建立率：** 同类型主题间建立关联的比例。过低（<10%）→ 关联自动发现机制不够敏感
- **规则提炼触发频率：** 异步提炼引擎的触发频率
- **领域规则确认率：** 候选规则经用户确认后生效的比例。过低（<30%）→ 规则提炼引擎过于激进
- **领域规则应用覆盖率：** 同类型新主题初始化时注入领域规则的比例。过低（<50%）→ 领域规则分类体系不完善
- **规则提炼队列积压量：** 待处理队列中的主题数量。积压量持续增长 → 提炼引擎处理能力不足或触发频率过低
- **知识真空期平均时长：** 主题从 checkpoint 到其规则被提炼生效的平均时间间隔
- **changeLevel 分布：** 模型判定的 major/minor/invalid 分布。invalid 占比 > 30% → 模型过于激进地判定撤销；**major 占比 < 3% → 模型可能偷懒全判为 minor（注意：阈值从原"1%"放宽至"3%"，降低了误报风险——少量真实 major 事件是正常的，但持续低于 3% 说明模型不愿做出结构性判断）**；changeLevelReason 空值率 > 10% → 模型在规避理由输出
- **embedding 调用次数：** topic_based 模式的 embedding 调用次数，监控成本
- **冷启动应急解冻触发次数：** 监控冷启动期间参数异常

> **多模型 A/B 测试路由配置**：态控架构支持在 IN_SESSION(align) 节点中配置多模型并行对比（如 R1 vs Qwen vs Claude），通过对比节点指标（confidence 分布、校验失败率、用户手动纠错率）选出最优模型。详细路由配置示例见附录 B。

### 附录 C：高风险场景降级链参考（扩展层）

对于医疗预问诊、复杂客服、金融风控等高危场景，默认的四项检查是通用底线，不可缩减。在此基础上，可追加以下增强方案：

```
simple: [L1_retry, DET_value_recheck, logprobs_check, cross_task_extension_check, hardcoded]
escalated: [L1_retry, DET_value_recheck, logprobs_check, cross_task_extension_check, hardcoded, human_takeover]
human_takeover: { target: "人工队列", sla: "5分钟" }
```

**启用条件：** 仅当场景的"错误代价"判据为"代价大且无客观答案"或"代价大但有客观校验"时，考虑启用 escalated 模式。默认场景仍用 simple（四项检查 + 硬编码兜底）。

taskType 在高风险场景中的影响：field_based 高风险场景的 DET 值域复验做严格的值域校验；topic_based 高风险场景的 DET 复验做格式校验 + 可能需要 needSemanticExtensionCheck:true。两种模式的高风险场景都可启用 escalated 模式。

高风险场景的 WAITING_CONFIRM 节点在 escalated 模式下，责任收敛不再止步于"当前用户确认"，而是升级为"人工队列接管"。SLA 超时未响应时，按 preset 降级到 hardcoded 或 abort。

注意：escalated 模式启用时需在场景的边界清单中显式声明人工接管的目标队列和 SLA。四项检查 + 硬编码兜底不可被 human_takeover 替换——human_takeover 是四项检查 + 硬编码兜底之后的扩展层。

### 附录 D：align-session 可选扩展规范

align-session 是可选扩展，非态控架构标准能力。若场景含意图校准需求，可启用 align-session 扩展。启用时需：

1. 额外编写 align-session.json 宪法
2. 在路由表中增加 align 相关条目
3. 在 tunable 参数中增加 divergence_threshold
4. 同步更新场景的边界清单
5. 在 N5 第0层之后插入第0.1层（align 静默模式触发与处理）

align-session 核心机制（仅在启用时生效）：静默输入 / 多模型并行仲裁 / 锚点确认 / 迭代裁决 / 对齐数据包沉淀。

与核心架构的边界：align-session 启用时不修改根宪法第1-4条（核心四条不含 align 内容），仅在路由表和 tunable 参数中扩展，并在 N5 插入第0.1层。align-session 不启用时，核心架构完全不受影响。

### 附录E. 宪法冲突检测引擎实现


**实施规范：**

- `single_param` 规则用 json-rules-engine（Node.js）或 Drools（Java）实现确定性 range/enum 校验
- `cross_param` 规则由冲突检测引擎在遍历 single_param 完成后执行 compute 表达式求值——用沙箱 `eval()`（仅允许 `+ - * / floor ceil min max` 运算符 + params 变量 + constants 常量）
- 参数变更时通过 `params` 字段做依赖追踪——仅被引用参数的变更触发对应 `cross_param` 规则重新执行
- `violation_action` 默认值：`single_param` 规则默认为 `hard_reject`；`cross_param` 规则默认为 `warning`
- `violation_action=hard_reject` 行为：拒绝保存 + 返回错误详情
- `violation_action=warning` 行为：弹窗展示 warning message + suggested_adjustment，用户可选择"仍然保存"或"调整后重试"。warning 不阻止保存
- compute 表达式求值失败（语法错误、引用不存在参数）→ 视为 `hard_reject`，阻止保存并返回明确的表达式错误位置
- conflict_check 规则用 JSON 格式声明，与 tunable 参数一同存在 tunables.json
- 保存前执行全部规则：`single_param` 任一 hard_reject → 拒绝保存；`cross_param` warning → 弹窗但允许继续
- 冲突检测引擎是确定性代码，不调 LLM

**规则示例：**

```json
{
  "conflict_rules": [
    {
      "type": "single_param",
      "name": "logprobs_threshold_range_check",
      "condition": {
        "all": [
          { "fact": "parameter", "path": "$.logprobs_threshold", "operator": "lessThan", "value": 0 },
          { "fact": "parameter", "path": "$.logprobs_threshold", "operator": "greaterThan", "value": 1 }
        ]
      },
      "event": { "type": "conflict", "message": "logprobs_threshold 必须在 [0, 1] 范围内" }
    },
    {
      "type": "cross_param",
      "name": "topic_threshold_ordering_check",
      "violation_action": "hard_reject",
      "params": ["topic_anchor_similarity_threshold", "topic_evolution_match_threshold"],
      "compute": "topic_anchor_similarity_threshold >= topic_evolution_match_threshold",
      "event": {
        "type": "cross_param_conflict",
        "severity": "error",
        "message": "topic_anchor_similarity_threshold 必须严格小于 topic_evolution_match_threshold——跨会话匹配需要比环节内锚点更高的精度。"
      }
    }
  ]
}
```

> 完整 `cross_param` 规则清单（含上下文预算、冷启动窗口、规则提炼触发、置信度冻结一致性等 5 条初始规则）见总纲 v5.5 §10.1，本指南 §8.2 也已同步引用双模式体系。

### 附录F. L2→L3 拆包语义一致性校验


**实施规范：**

- L2 文档中"规则声明"用 `<!-- @rule -->` 标记
- L3 拆包时 DET 自动提取所有 `<!-- @rule -->` 内容
- 对比 L3 的 session-constitution 是否包含对应规则
- 缺失 → 拆包失败

**L2 文档标记示例：**

```
<!-- @rule --> 金额上限为 999999
<!-- @rule --> 时间不能为未来
<!-- @rule --> 必采字段：种类、金额、时间
```

**L3 拆包校验流程：**

1. DET 扫描 L2 文档，提取所有 `<!-- @rule -->` 标记内容
2. DET 扫描 L3 session-constitution 的 @section validation 结构化 JSON
3. 对比每条 L2 规则是否在 L3 validation 中有对应规则
4. 缺失 → 拆包失败，返回缺失规则列表

### 附录G. contractStore 物理存储规范


**实施规范：**

- SQLite 单文件数据库，每个用户/项目一个 .db 文件
- WAL（Write-Ahead Logging）模式支持并发读 + 单写
- 数据库文件路径：`<project_root>/.contractstore/<user_id>.db`

**v7.0 表结构：**

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| analyzing_contract_in | ANALYZING 节点的原始 contract-in 副本（支持意图级纠错逃生舱） | — |
| analyzing_contract_out | ANALYZING 节点的 contract-out 副本 | — |
| topicIndex | 检索索引表，用于 DET 关键词快速检索和主题关联存储 | topicId（主键）、topicName、keywords（逗号分隔）、creatorUserId、createdAt、lastActiveAt、relatedTopics（JSON 数组） |
| topicEvolution | 主题演化主表，只存储 major 级 event + compressedMinor + 压缩后新增的 minor 级 event | topicId（主键）、intent、userId、stateSnapshot（JSON）、createdAt、updatedAt |
| topicEvolutionEvent | 事件主表 | eventId（主键）、topicId（外键）、changeLevel（模型判定）、nodeId、turnType、timestamp、stateChange（JSON）、askingField、modelOutputSummary |
| topicEvolutionArchive | 归档表，存储被压缩的 minor 级原始 event + 被判定为 invalid 的 event | 字段结构与 topicEvolutionEvent 相同。invalid event 附带"因模型判定为互斥撤销而未入主表"的元数据标记 |
| **domainRules**（v7.1 升级） | **领域规则库** | **ruleId（主键）、rule（规则内容）、topicPath（JSON 数组）、edges（JSON 对象，含 depends_on/conflicts_with/strengthens，统一 snake_case + 主动语态，strengthens 表示"当前规则强化目标规则"）、conditions（JSON 对象）、immutableLevel（L0-L3）、source（scenario_seed/user_initiated/model_inferred）、level（strong/medium/suggestion）、confirmedBy、confirmedAt、version、status（active/candidate/archived/pending_review）** |
| ruleCandidates | 候选规则池 | candidateId（主键）、rule、evidenceCount、passCount、validationRate、status（pending/confirmed/rejected） |
| ruleMiningQueue | 规则提炼待处理队列 | queueId（主键）、topicId、intent、checkpointAt、status（pending/processing/done） |
| **sessionCheckpoints** | **field_based 进度检查点** | **userId（主键）、lastCompletedStep、completedSteps（JSON 数组）、stepSnapshots（JSON 对象），ttl 默认 7 天。过期仅清理进度行，不联动清理 roomConversationLog（roomConversationLog 按 importance 分级保留：永久/90天/30天/7天）** |
| **roomConversationLog**（v7.2 新增） | **房间对话日志** | **id（主键自增）、userId、roomId、sessionId、turnIndex、userInput、modelResponse、turnType、askingField、changeLevel、timestamp。按 userId+roomId 索引，支持全文搜索** |

**并发控制：**

- WAL 模式支持多读单写
- 写操作用事务包裹，崩溃时自动回滚
- 多用户并发时，每个用户独立 .db 文件，无跨用户锁竞争

### changeLevel 抽样人工复核机制

**风险：** 模型可能偷懒将所有变更标记为 major 或跳过判定。DET 格式校验能拦空值/超长，但挡不住"编一个合理理由"的隐蔽偷懒。

**缓解：** 按 `changelevel_major_sample_rate`（默认 5%，tunable 可配）抽检 major 标记事件 → 人工判断是否真属于结构性变更 → false_positive 比例>30% 触发冷启动应急解冻。冷启动期间抽检比例临时上调至 10%。详见总纲 v5.5 文末风险条目。
