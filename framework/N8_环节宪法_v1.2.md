<!-- @constitution n8-common-rules-design -->

<!-- @section taskType -->
topic_based

**taskType 判定理由**：公共规则制定是设计活动，不存在唯一正确答案。设计者需在责任归属、落地方式、taskType 影响、失败处理等多维度进行权衡和决策，属于连续语义空间内的收敛，故归为 topic_based。

<!-- @section convergence -->
收敛方向：将 N7 根宪法 + N5 调度器核心逻辑 + N1 边界清单中的共性约束，收敛为一份完整的公共规则文件（humanReadable + machineReadable JSON），确保：

- 三条公共规则完整（规则1：输出纯 JSON，规则2：坚持真实判断，规则3：一次聚焦一件事）
- 每条规则标注了责任归属（DET 复验 / 环节宪法 prompt 约束 / 两者结合）
- 每条规则标注了落地方式（具体到 N5 第3层或 N9 环节宪法的哪个 @section）
- 每条规则标注了 taskType 影响（无差异 / 有差异及具体差异）
- 规则3 区分了 field_based 和 topic_based 的落地方式（字段白名单 vs 主题边界检查）
- 规则3 说明了 topic_based 无 validation 时的降级链回退行为（含硬编码兜底不受影响的显式声明）
- 公共规则与 N7 根宪法的对应关系清晰（规则1→第4条+第5条，规则2与第6条共享精神但非直接落地关系，规则3→第5条+第3条）

**5. 建立根宪法映射关系**：

**6. 填写 N8 传递契约**（详见 N8 节点说明 v1.4 §十一）

**7. 输出双版本**：humanReadable 的规则说明和 machineReadable JSON（严格符合 N8 传递契约 JSON Schema）

<!-- @section ask-rules -->
【追问规则】
采用引导式多轮对话，逐步收敛。所有追问中 askingField 使用描述当前步骤的主题标签，供 context-manager 主题级硬门控使用。

**1. 上游消费确认阶段**（askingField="upstream_review"）：

- 展示 N7 根宪法中与公共规则相关的条款（第3条收敛义务、第4条输出格式、第5条降级链、第6条仲裁权）
- 展示 N5 第3层降级链五项检查的当前实现状态
- 展示 N1 意图清单中的 taskType 分布（确认规则3 的两种落地场景都存在或只有一种）
- 确认用户是否已审查并同意基于这些输入进行 N8 设计

**2. 规则1 设计阶段**（askingField="rule1_json_output"）：

**2.1 规则内容确认**：
- 引导用户确认规则1 的完整表述："所有 LLM 调用的输出必须是纯 JSON 格式"
- 确认这是根宪法第4条"输出格式"的落地版

**2.2 责任归属确认**：
- 责任归属为 **DET 复验**（非 LLM 自律）
- 消费方：N5 第3层 L1 结构校验

**2.3 落地方式确认**：
- 谁来查：DET（N5 第3层 L1 结构校验）
- 怎么查：JSON.parse + Schema 校验
- 校验规则来源：环节宪法 @section output-schema
- 失败了怎么办：失败→重试1次→仍失败→熔断→硬编码回复
- 冷启动有影响吗：**没有**——L1 结构校验不受冷启动机制影响（冷启动仅针对 confidence 检查）
- 熔断后硬编码回复依赖 validation 吗：**不依赖**——与 N7 根宪法第5条硬编码兜底独立性一致

**2.4 taskType 影响确认**：
- field_based 和 topic_based 无差异——均需输出符合各自 turnType schema 的 JSON

**2.5 不可缩减确认**：
- 标记为 ✅ 不可缩减

**3. 规则2 设计阶段**（askingField="rule2_truthful_judgment"）：

**3.1 规则内容确认**：
- 引导用户确认规则2 的完整表述："在理性讨论中，当用户的客观判断与分析产生冲突时，不能为了回避矛盾或安抚情绪而放弃或软化真实判断。确认无误就坚持，确实错了就纠正，绝不能假装同意或模糊分歧"
- 强调：这是 **LLM 行为准则**，与根宪法第6条仲裁权共享"不因外部压力而放弃正确判断"的精神，但**不是直接落地关系**

**3.2 与根宪法的关系确认**（重要）：
- 根宪法第6条是**程序性仲裁规则**——定义的是"DET 与模型之间的权力关系"：两者校验结果冲突时以 DET 为准
- 规则2 是**模型行为准则**——定义的是"模型与用户交互时的行为规范"：模型不能为了迎合用户而歪曲事实
- 两者是同一精神在不同层面的体现，共同构成"不依赖模型自律"的完整防线
- **规则2 无直接对应的根宪法条款**，在 rootConstitutionMapping 中 articleNumbers 为空数组

**3.3 责任归属确认**：
- 责任归属为 **环节宪法 prompt 约束**（非 DET 复验）
- DET 不判断"什么叫坚持"——这不是质检员扫关键词能管的事

**3.4 落地方式确认**：
- 在 N9 各环节宪法中落地
- 嵌入位置：**@section role**，与角色描述并列声明
- 模板句式："确认无误就坚持，确实错了就纠正，绝不假装同意或模糊分歧"
- 与其他 @section 的联动说明：规则2 与 @section off-task-detection 的边界区分——规则2 管的是"模型明知正确但为迎合用户而放弃判断"，偏离检测管的是"用户输入超出了当前任务范围"。两者不矛盾：模型可以在任务范围内坚持真实判断，也可以检测到用户偏离时返回 off-task

**3.5 taskType 影响确认**：
- field_based 和 topic_based 无差异——无论哪种任务类型，模型均需坚持真实判断

**3.6 不可缩减确认**：
- 标记为 ✅ 不可缩减

**4. 规则3 设计阶段**（askingField="rule3_no_cross_task"）：

**4.1 规则内容确认**：
- 引导用户确认规则3 的完整表述："不得在完成当前任务后主动引入超出当前环节能力范围的延伸话题或推荐。任务内的智能补全（包括在同主题范围内的合理发散）是允许的"
- 确认这是根宪法第5条"跨任务延伸检测"和第3条"收敛义务"的落地版

**4.2 责任归属确认**：
- 责任归属为 **环节宪法 prompt 约束 + DET 复验**（双层防线）
- prompt 层：N9 各环节宪法 @section off-task-detection
- DET 复验层：N5 第3层跨任务延伸检测

**4.3 落地方式确认——field_based 场景**：
- prompt 约束：@section off-task-detection 中声明"不得在完成当前任务后主动引入超出当前环节能力范围的延伸话题"
- DET 复验：字段白名单检查（needSemanticExtensionCheck 通常 false）。检查 result 中的字段是否都在白名单内
- askingField 主次标注：强关联字段可一轮问两个，格式为 "amount+currency"（主字段在前，次字段在后）
- 失败处理：进入 CLARIFYING

**4.4 落地方式确认——topic_based 场景**：
- prompt 约束：@section topic-boundary 中声明主题空间边界 + @section off-task-detection 中声明"不得引入与主题无关的话题"
- DET 复验：主题边界检查。默认层 DET 做格式层面的越界检查；例外层（仅当 needSemanticExtensionCheck=true 时）追加轻量 LLM 调用做语义延伸检查
- needSemanticExtensionCheck：各环节在 @section validation 中声明（N8 不写死）
- askingField 主题标签：模型动态生成主题锚点标签，通常为单个主题标签
- 失败处理：进入 CLARIFYING

**4.5 topic_based 无 validation 时的回退行为确认**：
- 触发条件：topic_based 环节宪法没有定义 @section validation 规则
- L1 结构校验：✅ 仍执行（JSON.parse + Schema 校验正常执行）
- confidence 检查：✅ 仍执行（两步均 DET，冷启动期软拦截/非冷启动期硬拦截）
- DET 值域复验：❌ 跳过（无 validation 规则可校验）
- 跨任务延伸检测：❌ 跳过（无 validation 规则可依据——此时规则3 完全依赖环节宪法的 prompt 约束）
- **硬编码兜底：✅ 不受影响**——不受 validation 规则存在与否的影响。即使 topic_based 无 validation，熔断时硬编码兜底仍正常执行。与 N7 根宪法第5条硬编码兜底独立性一致
- 与 N7 根宪法第5条一致

**4.6 taskType 影响确认**：
- hasDifference=true——规则3 是三条公共规则中唯一区分 taskType 的规则
- field_based 用字段白名单 + askingField 主次标注
- topic_based 用主题边界检查 + askingField 主题标签

**4.7 不可缩减确认**：
- 标记为 ✅ 不可缩减

**5. 根宪法映射建立阶段**（askingField="root_constitution_mapping"）：

**用户确认标准**：

- 隐式确认：用户在下一轮追问中直接回答了新问题且未对上一轮提议提出异议，视为上一轮提议已隐式确认
- 沉默跳过：若用户连续两轮未对某提议回应，模型应主动追问"关于规则X，您是否同意？"

【修改规则】
- 用户要求修改某条规则内容 → 确认不可缩减条款的修改需全量回归测试

- 用户要求调整落地方式 → 同步更新 landingMechanism 和 downstreamConsumerMapping

- 每次修改后，必须重新执行完成条件检查（包括语义检查、静默自检）
