## 一、位置与定位

```
N13 L3→骨架生成（已定稿，21 条架构决策全部实现）
  ↓
N14 审骨架 + 跑 20+1 刁钻 case + 查 9+2 机制  ← 你在这里
  ↓
N15 调 tunable + 交付
  ↓
N16 运行期迭代回溯
```

**【人话版定位】**
N13 代码模型把骨架代码生成完了。N14 的任务是三件事：①审查骨架代码的 7 个关键架构点是否真的落地了（不是看代码，是看行为）；②跑 21 个刁钻测试用例，看系统在各种边界和异常情况下表现对不对；③检查 11 项核心机制是否都正常运转。

**N14 是 N13 的"验收测试"**——代码模型说它实现了 21 条架构决策，N14 用实际测试来验证是不是真的实现了。同时 N14 也是 N15 调参的"数据源"——20+1 case 和 9+2 机制的运行结果，会告诉你在 N15 应该调哪些参数。

**N14 的独特定位**：这是整个流程中**人与模型协作最紧密**的节点。模型负责跑测试、记录结果、标记异常；人负责判断测试结果是否可接受、决定是否需要回溯上游节点。

### 4.1 产出一：审骨架 7 项

审骨架不是审代码——代码质量 N13 已经检查过了。审骨架是审**行为**：骨架代码跑起来之后，这 7 个关键架构点是否真的生效了。

| # | 审查项 | 通过标准 | 验证方法 |
|---|--------|---------|---------|
| 1 | 状态转移图与 L2 文档一致 | 所有状态 + 转移路径 + 异常路径覆盖。第 0 层元指令路由（唤醒/退出/取消）属于状态转移的一部分 | 跑所有 21 个 case，从 simulate 日志中提取状态转移序列，与 N4 state-graph.json 对比。每一条转移边至少被触发一次 |
| 2 | DET 双层位置正确 | 第 0.5 层关键词扫描在用户输入后、ANALYZING 前。第 3 层 DET 值域复验在 complete 时。第 0 层全局硬匹配属于状态转移的一部分（由审查项 1 覆盖） | 从 simulate 日志中提取 DET 关键词扫描的时间戳和 DET 值域复验的时间戳，验证双层位置符合预期 |
| 3 | askingField 在 turnType schema 且在 context-manager 调用参数 | currentAskField 传入 buildPromptContext | 从 simulate 日志中提取 buildPromptContext() 的调用参数，验证 currentAskField 字段存在且值与模型返回的 askingField 一致 |
| 4 | tunable 槽位从 constitution.getTunable() 读 | 没有硬编码阈值 | 修改 tunables.json 中某个参数值后重新跑 case，确认行为随之变化 |
| 5 | context-manager 三接口调用占位 | 没有内联拼接逻辑 | 在 context-manager.placeholder.js 中增加日志输出，跑 case 后确认日志存在 |
| 6 | taskType 双模式锚点机制实现 | buildPromptContext() 接收 currentTaskType 参数；field_based 执行字段级硬门控；topic_based 执行三层注入 + embedding 两阶段检索 | 分别跑 field_based case 和 topic_based case，从日志中提取锚点模式，验证两者不同 |

### 4.2 产出二：20+1 刁钻 case

| 2 | 超长输入（远超字段预期） | 截断或提示"太长了" | #1（状态转移——ANALYZING 输入校验）、#1（第 0 层不误匹配为元指令） |

| 4 | 中途偏离到其他意图（记账中突然说"查一下"） | 模型返回 off-task，调度器重新意图识别 | #1（状态转移——IN_SESSION→ANALYZING off-task 边）、#3（DET 第 0.5 层关键词扫描标记 offTaskSuspicion） |

| 10 | 跨轮次引用（"刚才那个也记上"） | 视场景能力，至少不崩溃 | #5（上下文拼接不丢失关键信息）、#6（contractStore 查询逻辑） |

## 五、关键纪律

1. **审骨架审的是行为，不是代码**——代码质量 N13 已经检查过了
2. **模型跑测试、记录结果、标记异常；人判断行为正确性、决定处置路径**
3. **20+1 case 必须全部跑通**——有一个 case 行为不符合预期，就要回溯上游节点
4. **机制检查 9+2 全部触发至少一次**
5. **测试结果直接驱动 N15 调参**——case 11/12/16/18 的结果直接影响参数调整决策
6. **冷启动期 confidence 软拦截必须实际验证**——不能在冷启动窗口外跑 case 12
7. **changeLevelReason 空值率 > 10% 是模型偷懒的强信号**——空值率统计来源为 DET 拦截日志（非 topicEvolutionEvent 表），且两种 taskType 模式均统计
8. **major 占比 < 3% 和 changeLevelReason 空值率 > 10% 是两种不同的模型偷懒信号，需独立统计和告警**——两者根因不同，回溯策略也不同
9. **机制检查的降级策略必须在测试报告中标注**——哪些使用了轻量验证，哪些使用了完整的 OpenTelemetry trace

## 七、常见踩坑

| 坑 | 表现 | 后果 |
|----|------|------|
| 只审了代码，没跑 case | 看了一遍 scheduler.js 觉得没问题就通过了 | 代码逻辑正确但运行时行为异常 |
| case 12 只在非冷启动期跑 | 冷启动窗口已经过了才跑 soft intercept case | 软拦截逻辑从未被实际触发 |
| 机制检查只查了"有没有"，没查"对不对" | 确认了 topicEvolution event 追加逻辑存在，但没检查 changeLevelReason 是否同步提取 | N16 监控失效 |
| 上下文拼接命中率低但未回溯 | 频繁重复追问已采集字段，但觉得"不影响功能"就放过了 | 长对话质量急剧下降 |
| topicEvolution major 占比 < 3% 未告警 | 关键进展被标记为 minor 后压缩 | 跨会话续写时丢失关键上下文 |
| changeLevelReason 空值率 > 10% 未告警 | 觉得"只是没写理由，不影响功能" | 防偷懒机制失效 |
| changeLevelReason 空值率统计来源错误 | 从 topicEvolutionEvent 表统计空值率（所有空值已被 DET 拦截后填充默认值） | 模型偷懒信号永远无法被检测到 |
| changeLevelReason 空值率统计未区分 taskType | 统计时把 field_based 任务的 null 值也计入了 | 空值率虚高，触发误告警 |
| major 占比 < 3% 和 changeLevelReason 空值率 > 10% 混在一起统计 | 两个指标合并为一个告警条件 | 无法区分根因 |
| off-task 场景强制要求 changeLevelReason | 验证时要求 off-task 路径的 topicEvolutionEvent 也必须含 changeLevelReason | 模型偏离时未正常输出，不强求字段存在（缺失时记录信息级日志，不视为错误） |

### 10.2 三步走

**第一步：审骨架 7 项**

跑骨架代码，从日志里抓数据，逐项验证：

- 第 7 项：topicEvolution 的 5 个子项逐个验证——特别注意 7a（changeLevelReason 空值时 DET 拦截日志是否记录了原始空值事件）、7b（off-task 路径不强求 changeLevelReason）

**第二步：跑 20+1 case**

逐个跑 21 个测试用例，记录实际行为和预期行为的对比：
- case 3（超限值）：故意输入金额 9999999，看 DET 第二道有没有拦截

- case 17（跨会话续写）：开新会话说"继续上次那首诗"，看 ANALYZING 有没有匹配到 topicIndex

**第三步：查 9+2 机制**

从日志和 trace 里提取数据，逐项验证 11 项机制：

- 机制 7（上下文拼接命中率）：跑多轮对话，看模型有没有重复追问已经回答过的问题
