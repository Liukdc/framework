## 二、核心概念速查

### 2.1 哲学根

**认知局部性原理（输入侧）**：任何信息处理主体在 t 时刻处理节点 N 时，最优输入集 = N 在当前目标下的最小充分条件集，不是全集 Γ。

- 人：工作记忆约 4 个组块（Cowan, 2001）
- LLM：Lost in the Middle + 近因偏置 + 长上下文幻觉率单调递增
- DET：函数纯度 f(x) 只依赖 x

### 2.2 taskType 双模式

| 维度 | field_based（离散参数空间） | topic_based（连续语义空间） |
|------|---------------------------|---------------------------|
| 收敛目标 | 定位唯一正确值 | 采样满足约束的样本 |
| 验证方式 | DET 值域硬校验 | 格式校验 + 语义判断 |
| 跨节点载体 | sessionCheckpoint | topicEvolution |
| 典型场景 | 记账、改参数、表单 | 写诗、角色扮演、方案设计 |

### 2.3 三类执行体

| 执行体 | 性质 | 职责 |
|--------|------|------|
| DET（确定性程序） | 确定性 | EXACT_MATCH + 值域复验 + logprobs裁决 + SQL查询 + 文件操作 |
| LLM（概率性） | 概率性 | 意图识别 + 字段采集 + 偏离判断 + 工具调用决策 + 结果格式化 |
| HUMAN（人类选择） | 责任性 | 不可逆决策确认 + 规则确认 + 偏好仲裁 |

### 2.4 房间比喻（四层结构）

```
房间（环节，N1-N16）—— 环节宪法（房间级）
  └── 柜子（项目，如富贵小安/杂碎本）—— 产出物内容（柜子级，隔离）
       └── 格子（文件，精简总纲+细则）—— 正文内容（格子级，独立文件）
            └── 变更日志（独立文件）—— 所有变更追溯（全局统一）
```

- **房间隔离**：S3 状态隔离（KV Cache 回收），wake/switch/exit/cancel 四类口令
- **自由切换四支柱**：roomConversationLog（全量记录）+ stepSnapshot（对话快照）+ 规则提炼按环节隔离 + outputRegistry（产出物总索引）

## 三、根宪法4条（约束所有环节模型）

| # | 条款 | 内容 |
|:--:|------|------|
| 1 | 环节隔离 | 只能做本环节任务，非本环节任务返回 off-task |
| 2 | 输出格式 | 符合 turnType schema 的结构化 JSON |
| 3 | 收敛义务+坚持真实判断 | 不伪造、不假装理解、不迎合用户、不无中生有 |
| 4 | 不编造功能 | 不得生成用户未交代、系统不具备的功能或信息 |

**ANALYZING 特化**：只产出意图识别信号（intent/choice/logprobs/inputNature/extracted/topicId），不产出 turnType。

## 四、架构机制M1-M4（约束调度器/DET）

| 编号 | 机制 | 核心内容 |
|:--:|------|---------|
| M1 | 元指令集 | 四类口令（wake/switch/exit/cancel）+ S3会话回收 + roomStateIndex注入 |
| M2 | 降级链四项检查 | L1结构校验 + DET值域复验 + logprobs检查 + 硬编码兜底（全确定性，不可缩减） |
| M3 | 仲裁权DET优先 | DET校验结果优先于模型内部校验 |
| M4 | 节点转换守卫 | 环节间转换的触发条件和守卫规则 |

## 五、编程规则宪法7条（约束代码模型）

| # | 条款 | 核心内容 |
|:--:|------|---------|
| 1 | 执行体分派铁律 | ANALYZING=LLM, IN_SESSION=LLM, DET复验=DET, EXECUTING=DET（仅写入/删除）。IN_SESSION可调用DET工具 |
| 2 | DET职责边界 | 确定性计算：EXACT_MATCH+值域复验+关键词扫描+logprobs裁决+路由矩阵+规则匹配+文件操作+SQL查询 |
| 3 | LLM职责边界 | 意图识别+字段采集+偏离判断+语义边界+工具调用决策+结果格式化 |
| 4 | 降级规则 | 无apiKey时拒绝服务，不降级为正则 |
| 5 | 契约对齐 | 严格对齐N11契约schema |
| 6 | tunable/接口隔离 | 阈值从getTunable()读，调度器只调context-manager三接口（匹配/决策/拼接） |
| 7 | 工具分层发现 | 必用工具+工具清单+选用工具三层 + search_tools meta-tool + LLM不直接写SQL |

## 十二、文件管理机制（v5.5-v5.6）

### 房间比喻四层结构

```
房间（环节，N1-N16）—— 环节宪法（房间级，共享）
  └── 项目（智能体实例，如富贵小安/杂碎本）—— 产出物内容（项目级，隔离）
       └── 格子（细则文件，精简总纲+细则）—— 正文内容（文件级，独立）
            └── 变更日志（独立文件）—— 所有变更追溯（全局统一）
```

### 三个机制

| 机制 | 解决什么 | 落地方式 | 状态 |
|------|---------|---------|------|
| 机制1 项目隔离 | 同环节多项目产出物隔离 | projectId + S3物理隔离 + projectRegistry表 | ✅ v5.6已落地 |
| 机制2 分格 | 大文件拆分，降低算力成本 | 按章节拆成独立细则文件 | ✅ v5.5已落地 |
| 机制3 变更日志独立 | 变更可审计，防止误改 | 所有变更统一记录在变更日志.md | ✅ v5.5已落地 |

### 机制1 项目隔离核心

- **currentProjectId**：全局字段，一处变更所有房间联动
- **默认项目**：userLastProject 表记录用户最后操作的项目
- **项目切换**：ANALYZING 关键词匹配 → S3 物理隔离四步 → 加载新项目最后状态
- **项目切换 vs 房间切换**：两者都需要 S3；房间切换变 roomId，项目切换变 projectId
- **跨项目引用**：走 outputRegistry 跨项目查询 + 文件复制（需用户确认）

### 文件管理纪律

1. **修改基线时另存新版本**——不直接覆盖原文件
2. **细则文件只含正文**——不含变更记录（变更记录在变更日志.md）
3. **总纲文件是概要索引**——每次读取只读总纲，需要详情时读对应细则
4. **大改动分批进行**——每次只改一个细则文件
