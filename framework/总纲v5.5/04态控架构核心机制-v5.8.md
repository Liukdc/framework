# 态控架构 · §三 核心机制

> **本文件是细则文件**——对应总纲概要的 §三。版本 v5.8（含项目隔离 + 对话分段保留 + conversationArchive）。

## 三、核心机制

### 3.1 意图收敛机制

**定位**：从用户模糊的自然语言输入，收敛到系统可执行的结构化指令。

**收敛链路**：
```
用户输入
  → 第0层：口令层（DET EXACT_MATCH，0 次 LLM）
  → 第0.5层：关键词扫描（DET 标记 offTaskSuspicion，0 次 LLM）
  → 第1层：意图识别（LLM 轻量模型，ANALYZING 环节）
  → 第2层：环节内执行（LLM 通用/深度模型，IN_SESSION 环节）
  → 第3层：DET 值域复验（DET 四项检查）
```

#### 3.1.1 off-task 双重保障与显式切换口令

**问题**：环节间自由切换是态控架构的核心目标，但 IN_SESSION（会话中）内用户要切换环节时，当前机制单靠模型概率判断 off-task（偏离），不能保证准确——模型可能漏判（把切换当成当前环节补充）或误判（把补充当成切换）。

```
保障1（主力——宪法强制 + 上下文管理器优先注入）：
  根宪法第1条（环节隔离）→ 模型每轮首要判断"是否本环节任务"
  → 上下文管理器将环节隔离声明（@section role 首位）以最高优先级注入 prompt
  → 模型每轮看到的上下文，第一件事就是"你当前在什么环节，什么是 off-task"
  → 不属于本环节 → 立即返回 off-task（优先级最高）
  → 大幅降低漏判率

保障2（辅助——ANALYZING 重新识别验证）：
  模型返回 off-task 后，调度器不直接路由
  → DET 只做非空检查（门卫，不做语义验证——DET 做不了语义判断）
  → 送回 ANALYZING 重新识别 offTaskInput
  → ANALYZING 的重新识别结果就是验证：
    → 识别出不同 intent → 确认切换，进入新环节
    → 识别出 same intent 或 other → 尊重模型判断，引导重新表达
```

**显式切换口令（M1 switch，触发 S3 物理隔离）**：

用户可主动说"切断房间"口令 → DET 硬匹配（EXACT_MATCH，0 次 LLM）→ 调度器触发 S3 物理机制。S3 物理隔离的完整四步流程与四个元指令（wake/switch/exit/cancel）的 S3 行为详见 §9.4，此处不再重复。

显式口令是首选切换机制（100% 确定性），off-task 检测是兜底（用户忘了口令时的保底）。

**这是态控架构"房间"比喻的物理落地**——传统智能体需要用户新开窗口来切换上下文，态控架构在单一窗口内通过 S3 物理隔离实现房间自由切换。

**全窗口房间状态索引（roomStateIndex）**：

S3 物理隔离解决了"新旧不串"的问题，但没有解决"切换后不知道旧房间有什么没做完"的问题。roomStateIndex 填补这个缺口：

- **实现形态**：物化视图（非实时查询）——roomStateIndex 是一张独立维护的摘要表，存储于 contractStore.roomStateIndex（userId 索引）
- **数据来源**：从 sessionCheckpoint + roomConversationLog + outputRegistry 三个数据源增量同步
- **数据结构**：`[{roomId, roomName, intent, taskType, lastActiveAt, pendingCount, completedCount, lastSummary}]`
- **增量更新触发事件**：房间切换时 / sessionCheckpoint 写入时 / outputRegistry 注册时
- **注入时机**：用户说"切断房间" → 调度器在 S3 物理释放前，先请求 context-manager 的 `buildRoomIndex()` 读取 roomStateIndex 物化视图 → 连同新房间的 checkpoint 一起注入 ANALYZING 的 prompt
- **注入内容**：ANALYZING 收到的 prompt 末尾追加一个 `[全窗口房间状态]` 段落，列出所有房间的状态摘要（如"记账房间：1 条半成品待完成"、"提醒房间：3 条提醒待触发"）
- **意图识别增强**：ANALYZING 有了全局视野后，可以针对性地引导用户——"您记账房间还有一条未完成的记录，要先处理吗？"——而不是简单地返回 intent=other

**roomStateIndex 与 roomConversationLog 的区别**：

| | roomConversationLog | roomStateIndex |
|------|------|------|
| 粒度 | 单房间全量对话原文 | 全窗口房间摘要 |
| 用途 | IN_SESSION 环节内上下文拼接 | ANALYZING 跨房间意图识别 |
| 消费方 | context-manager 拼接层 | context-manager buildRoomIndex() |
| 更新频率 | 每轮追加 | 物化视图增量同步（房间切换/checkpoint 写入/outputRegistry 注册时） |
| 存储 | contractStore.roomConversationLog | contractStore.roomStateIndex（独立表，userId 索引） |

**交互示例**：
```
用户："切断房间"
  → S3 释放记账房间 KV Cache
  → buildRoomIndex() 读取物化视图：记账房间 pending=1，提醒房间 pending=3
  → ANALYZING prompt 末尾注入：
    [全窗口房间状态]
    记账房间：半成品 1 条（7月15日记了半条"午饭25块"）
    提醒房间：待触发 3 条（最近：明天下午3点"开会"）
  → ANALYZING 返回 intent=reminder + suggestion="提醒房间有 3 条待触发提醒"
  → 用户可以选择："先去提醒房间" 或 "继续记账"
```

**设计原则**：不追求 100% 拦截，而是"降低漏判 + 拦住误判 + 回溯兜底"。用确定性兜底追求 100% 反而会破坏环节自治。

#### 3.1.2 ANALYZING 的环节隔离（双维度分类）

ANALYZING（分析态）是"意图识别环节"，受根宪法第 1 条约束：
- **本环节任务**：将用户输入映射到已知 intent（意图），同时分类输入性质：ANALYZING 用结构化 prompt 执行两个并行的强制选择分类：

  **维度1：意图匹配**——从有限选项中选择（如 A/B/C/D/E），开启 API logprobs 获取真实概率分布。返回 choice（选项字母）+ logprobs（概率分布）。DET 读取 logprobs 做阈值裁决（probability ≥ threshold → 放行，< threshold → intent=other）。

  **维度2：输入性质**——判断输入属于哪类：
  - S) 闲聊/松弛：无明确目标，随意表达，社交寒暄
  - T) 任务导向：有明确目标，结构化表达，包含动作意图
  - U) 不确定

  ANALYZING 一次调用返回两个信号：
  ```
  { choice: "C", logprobs: {A:0.05, B:0.03, C:0.78, D:0.08, E:0.06},
    inputNature: "T", inputNatureLogprobs: {S:0.02, T:0.91, U:0.07} }
  ```

  DET 交叉裁决路由：

  | intent match | input nature | 路由 |
  |-------------|-------------|------|
  | A-E（高 probability） | — | 对应正式房间 |
  | other（低 probability） | S（闲聊） | **SLACK_NODE** |
  | other（低 probability） | T（任务导向） | **TEMP_TOPIC** → 创建运行时房间 |
  | other（低 probability） | U（不确定） | 追问澄清 |

  **关键：一次 LLM 调用完成两个分类，不改模型，不追加调用。**只是 prompt 多问一个问题，返回多一个字段。

- **非本环节任务**：不能采集字段（那是 IN_SESSION 的事）、不能追问（那是 IN_SESSION 的事）、不能给出最终回复
- **边界情况**：输入无法映射到任何已知 intent → 返回 `intent=other`（低 probability）。注意：ANALYZING 只产出意图识别相关信号（intent / choice / logprobs / inputNature / extracted / topicId），不产出 turnType——turnType 是 IN_SESSION 的输出格式。

**ANALYZING 的产出物感知**：ANALYZING prompt 末尾追加 `[全系统产出物]` 段落——从 outputRegistry 查询当前用户的最近产出物列表。这使 ANALYZING 在意图识别时就拥有全系统产出物视野，可以在路由建议中主动提示："元智能体房间有 L2 设计文档，需要引用吗？" 详见 §3.1.4。

#### 3.1.3 产出物总索引——Output Registry

**定位**：全系统所有房间产出物的统一登记簿。与 roomStateIndex（房间状态摘要）互补——roomStateIndex 回答"房间有什么待办"，outputRegistry 回答"房间产出了什么"。

**存储**：contractStore 中 `outputRegistry` 表，完整 SQL DDL 见附录 C.3。

**索引**：`(roomId, createdAt)`、`(intent, outputType)`、FTS on `(outputName, outputSummary, keywords)`。

**写入时机与门槛**：

| 时机 | 操作 | 注册门槛 |
|------|------|---------|
| IN_SESSION complete（topic_based） | 用户确认的最终产出自动注册 | 无门槛——topic_based 的 complete 产出本身就是交付物 |
| IN_SESSION complete（field_based） | contract-out JSON 自动注册 | **有门槛**——field_based 的 complete 产出是字段集（如一笔记账），属于操作记录而非交付物。只有以下情况注册：(1) 用户主动命名/导出（如"7月账单汇总"）；(2) 字段集被其他房间引用过；(3) 用户触发"保存为产出物"。每轮 complete 的裸字段集不自动注册，避免 outputRegistry 被零散操作记录淹没 |
| TEMP_TOPIC 用户触发总结 | 总结产出自动注册 | 无门槛 |
| 用户手动导出/保存 | 调度器调用 registry.register() | 无门槛 |

**检索方式**（DET 确定性查询，0 次 LLM）：

```
- 按房间：registry.findByRoom("元智能体")
- 按类型：registry.findByType("document")
- 按关键词：registry.search("L2 设计文档")
- 按时间：registry.findSince("2026-07-01")
- 组合：registry.search({roomName: "元智能体", outputType: "document"})
```

**跨房间产出物发现与复制流程**：

```
1. 用户输入 → ANALYZING（已有 roomStateIndex + outputRegistry 列表注入） → LLM 语义理解："用户想引用元智能体的L2文档" → 返回：{ intent: "patent_write", crossRoomRef: { room: "元智能体", output: "L2设计文档" } } → 调度器 DET 拿 crossRoomRef 去 outputRegistry.search() ↑ ANALYZING 做语义判断，DET 只做精确查询

2. ANALYZING → crossRoomRef: {room:"元智能体", output:"L2设计文档"}
    ↓
DET: registry.search() → 命中1条
    ↓
DET 把匹配结果注入 IN_SESSION(patent_write) 的上下文
    ↓
IN_SESSION 模型：
  "找到元智能体的L2设计文档（7月15日，12KB）。要作为专利背景技术引用吗？"
    ↓
3.用户："对"  →  turnType=complete + confirmedRef=outputId_xxx
    ↓
调度器调用文件操作模块执行复制（DET 确定性）:
   cp {outputPath} → {currentRoomWorkspace}/references/{outputName}
   → 记录操作审计日志

5. 专利撰写房间的上下文管理器下一轮拼接时，自动将 references/ 目录文件列表
   注入房间上下文，模型可以读取引用文件的内容
```

**关键纪律——房间隔离不妥协**：

| 可以跨房间共享 | 不可以跨房间共享 |
|--------------|----------------|
| ✅ 结构化产出物文件（通过文件复制） | ❌ roomConversationLog（房间对话原文） |
| ✅ outputRegistry 元数据（产出物名称、摘要、路径） | ❌ 其他房间的领域规则 |
| ✅ roomStateIndex 状态摘要（每个房间有待办数） | ❌ 其他房间的 sessionCheckpoint 内部字段 |

文件复制是唯一的跨房间数据传递方式——不直接注入原文到上下文（防止上下文膨胀），而是复制文件到目标房间的 workspace/references/ 目录。目标房间的上下文管理器在下一轮拼接时，按需要决定是否将引用文件内容注入当前上下文。**房间之间的对话原文仍然完全隔离。** roomStateIndex 和 outputRegistry 是元数据层索引，不是数据层原文——它们提供"隔壁房间有什么"的摘要，不提供隔壁房间的对话内容，因此不违反房间隔离。

**与 roomStateIndex 的互补**（三表统一对比）：

| 维度 | roomConversationLog | roomStateIndex | outputRegistry |
|------|------|------|------|
| 粒度 | 单房间全量对话原文 | 全窗口房间摘要 | 产出物级明细 |
| 回答什么 | "这个房间聊过什么" | "记账房间还有1条半成品" | "元智能体产出了3份文档、2份代码" |
| 消费方 | context-manager 拼接层 | ANALYZING（跨房间切换建议） | ANALYZING（跨房间产出发现）+ 调度器（文件操作） |
| 更新频率 | 每轮追加 | 物化视图增量同步 | 每次产出物注册时 |
| 保留策略 | 按 importance 分级（永久/90天/30天/7天） | 跟随源表 | 永久（产出物是交付物） |
| 跨房间可见 | ❌ 不可见 | ✅ 摘要可见 | ✅ 元数据可见 |

### 3.2 taskType 双模式

**区分原点：收敛空间的性质，而非任务内容。**

taskType 是态控架构所有设计决策的逻辑起点——上下文拼接方式、校验方式、记忆载体、规则匹配键、追问策略、模型选择策略，全部收敛到这一个二分法上。

**field_based（基于字段）——离散参数空间，定位唯一正确值**

收敛空间是离散的、可枚举的参数集合。目标是从所有可能的候选中定位唯一正确值。存在客观正确答案——找到了，任务完成；没找到，任务未完成。正确与否可被 DET 值域硬校验验证。

典型场景：记账（从所有可能的类别中找到"猫粮"，从所有可能的金额中找到"35"）、改柜体参数（从所有可能的深度值中找到"600mm"）、查询（从所有可能的记录中找到"上周三的订单"）、删除（从所有可能的条目中找到"第5条"）、表单填写。

**topic_based（基于主题）——连续语义空间，采样满足约束的样本**

收敛空间是连续的、不可枚举的语义空间。目标是在受约束的主题空间内采样一个满足条件的样本。不存在唯一正确答案——同一主题下，可以生成无数个不同的合格输出。无法用"对/错"评判，只能用"是否在主题边界内 + 是否满足格式约束"来判断。

典型场景：写诗（在"秋天"主题空间内生成一首满足七言绝句格式的样本）、写专利（在"一种基于 XXX 的方法及系统"空间内生成满足专利撰写规范的样本）、角色扮演（在"林霜是侦探助手"空间内生成符合角色设定的对话）、头脑风暴（在"如何提升用户留存"空间内生成一组可行的想法）。

**全维度差异表**：

| 维度 | field_based（基于字段） | topic_based（基于主题） |
|------|----------------------|----------------------|
| 收敛空间 | 离散参数空间（可枚举） | 连续语义空间（不可枚举） |
| 目标 | 定位唯一正确值 | 采样满足约束的样本 |
| 是否存在客观正确答案 | 是 | 否 |
| 验证方式 | 值域校验（DET 硬规则） | 格式校验 + 主题边界检查（DET + 语义判断） |
| 追问方式 | 按预定义字段顺序追问 | 按主题探索路径追问（模型自主决定） |
| 完成条件 | 所有必采字段齐全 | 输出满足主题边界 + 格式约束 |
| 跨节点传递载体 | contract-out JSON（字段集）+ sessionCheckpoint（进度快照） | topicEvolution（事件序列 + 状态快照） |
| 长期记忆 | sessionCheckpoint（进度快照）+ roomConversationLog | topicEvolution（主题演化历史） |
| 领域规则匹配 | stepName DET 精确匹配 | topicPath 语义匹配 |
| 错误形态 | 值错了（可 DET 校验） | 跑偏了（需语义判断） |

**混合型任务：主 taskType + 子约束**

某些任务同时跨越两种空间。区分标准不是"任务里有没有字段"，而是"收敛空间的主体是连续的还是离散的"：
- 主体是连续语义空间（如创作），但带有离散约束（如格式）→ topic_based 主 + field_based 子。例："写一首关于秋天的七言绝句"——诗歌创作是 topic_based，但"七言绝句"的格式约束（28 字、平仄押韵）是 field_based。DET 复验时先做 field_based 格式校验，再做 topic_based 主题边界校验。
- 主体是填写字段（如录入客户信息），但带有自由文本（如备注）→ field_based 主 + topic_based 子（备注字段不做值域校验，只做主题边界检查）。

主 taskType 决定跨节点载体和长期记忆机制（topic_based 主→topicEvolution；field_based 主→sessionCheckpoint），子约束附加在 DET 复验的末端校验中。长期记忆载体统一为 contractStore——field_based 走 contractStore.sessionCheckpoints，topic_based 走 contractStore.topicEvolution。

**topicEvolution 与 sessionCheckpoint 的对位**：

| 维度 | topicEvolution（topic_based） | sessionCheckpoint（field_based） |
|------|----------------------|----------------------|
| 载体形式 | 事件流：每节点 exit 追加 event | 进度标记：每节点 complete 更新一行 |
| 匹配方式 | topicId embedding 语义检索 | userId DET 精确匹配 |
| 记录粒度 | 事件级——每 turn 的事件序列 | 环节级——每环节一帧快照 |
| 注入给模型 | 三层注入（已生效规则 + 关联主题摘要 + 演化事件包） | 结构化字段 collectedFields |
| 完整定义 | 见 §3.8.1 | 见 §3.8.2 |

topicEvolution 和 sessionCheckpoint 的完整机制定义（存储结构、写入时机、恢复流程、changeLevel 分层留存）统一收敛到 §3.8 跨会话记忆机制，此处只保留对位关系。

**领域规则体系**：规则不再只是平铺的"行为模式列表"，而是一个包含权限分层（L0-L3）、来源分类（scenario_seed / user_initiated / model_inferred）、关联边（depends_on / conflicts_with / strengthens）和条件匹配（conditions）的规则网络图。注入方式"按 topicPath 子图匹配精准注入"。完整定义见 §3.7。

### 3.4 三类执行体分派（DET / LLM / HUMAN）

#### 3.4.1 分派判据

每个状态在宪法中必须显式声明执行体，判据顺序如下（优先级递减）：

A. 错误代价多大？
代价大且无客观答案 → 人
代价大但有客观校验 → LLM + 人确认
代价小 → 进入下一条

B. 能否写成硬规则 + 穷举？
能 → 确定性程序（DET）
不能 → 进入下一条

C. 是否有客观正确答案？
有（映射类、意图类）→ LLM
没有（偏好类、责任类）→ 人

关键规则："代价"为最高优先级判据。代价大的节点无论能否硬规则化，都必须经过人确认或 LLM + 人确认。

#### 3.4.2 与意图收敛机制的对应

DET 执行确定性收敛；LLM 执行有限展开后收敛，出口被 contract-out schema（输出契约模式）硬卡；人执行责任收敛，对不可逆决策负责。

### 3.5 人和 DET 作为终极容错闭环

态控不追求大模型的单次绝对正确，而是构建"犯错-拦截-纠错"的工程闭环。用户不仅是意图源头，更是系统状态的"重置按钮"。系统只需保证"大概率正确"，剩余边界错误由 DET 和用户纠正完成最终收敛。在高风险场景中，这一闭环从单用户现场扩展到组织级人工接管通道。

### 3.6 领域规则体系

**定位**：领域规则是态控架构的第三层宪法（领域规则层）在生产过程中逐步建立的约束体系。与宪法不同——宪法是设计者预设的刚性框架，领域规则是从用户的实际使用中"长出来"的经验沉淀。领域规则从平铺列表升级为可查询、可冲突检测、可按主题路径匹配的**规则网络图**。

**field_based 的等价物——contractEvolution**：topic_based 的规则按 topicPath（行业→品类→场景）分层，field_based 的规则按 stepName（环节名）分层——规则提炼和注入完全按环节隔离。两者机制相同（changeLevel → ruleMiningQueue → 异步提炼 → domainRules），唯一差异是匹配键：topic_based 用 topicPath 语义检索，field_based 用 stepName DET 精确匹配。

#### 3.6.1 规则来源三层

领域规则有三类来源，对应不同的可信度和确认流程：

| 来源 | 含义 | 灌入时机 | 权限初始值 | 是否走交叉验证 |
|------|------|---------|-----------|-------------|
| **scenario_seed** | 场景固有规则（工艺规范、设计基线、报价模板、安全红线等） | 主题初始化时按 topicPath 加载 | L0 / L1 | ❌ 不走，设计者预设可信 |
| **user_initiated** | 主题确认时用户主动口述的规则（如"厨房台面我喜欢高一点"） | ANALYZING 确认 topicName 后主动问询 | L2 | ✅ 走，用户可能说错或矛盾 |
| **model_inferred** | IN_SESSION 过程中模型推断的用户偏好（如"从对话中推测用户偏好 U 型厨房"） | complete 时顺带输出 → ruleCandidates | L3 | ✅ 需用户确认升格为 L2 |

#### 3.6.2 规则权限分层

领域规则内部存在 L0-L3 四级权限梯度。这一维度**与宪法层级正交**——宪法分三层（根/局部/领域），每条领域规则同时具有"宪法层级"（属于第 3 层领域规则）和"权限层级"（L0/L1/L2/L3），分别定义其适用范围和可修改性。

| 层级 | 名称 | 含义 | 可修改 | 默认 importance | 冲突时行为 |
|------|------|------|--------|-----------------|----------|
| **L0** | 钢印规则 | 场景硬性约束，不可触碰 | ❌ 不可修改 | critical（L0 规则默认 critical，受"同层 critical≤1"约束——L0 层内必须用户确认选哪条为 critical，详见 §3.9） | 无条件胜出，不询问 |
| **L1** | 基线规则 | 设计者预设最佳实践（如"厨房台面高度建议 85cm"） | ⚠️ 需设计者确认 | normal | 与 L2/L3 冲突时触发询问 |
| **L2** | 用户确认规则 | 用户主动说过或确认过的偏好 | ✅ 用户可随时修改 | normal | 优先于 L3 |
| **L3** | 模型推断规则 | 模型 infer 的候选规则 | ✅ 需用户确认升格 | normal | 仅作参考，不主动注入 |

**importance 与 immutableLevel 正交**：
- immutableLevel 是权限轴（谁能改），importance 是重要程度轴（哪个更关键）。同 immutableLevel 内，importance 高的胜出
- 每层（L0/L1/L2/L3）内 critical ≤ 1、high ≤ ⌊该层规则总数/3⌋、其余 normal
- L0 规则中必须选出 1 条 critical（安全红线——用户确认），其余可全 normal
- 冲突仲裁：先比 immutableLevel → 同层再比 importance → 同 importance 触发询问

**跨层仲裁的边界声明**：
- 当两条规则 immutableLevel 不同时（如 L0 vs L2），按 immutableLevel 直接裁决（L0 无条件胜出），**importance 不参与跨层仲裁**
- importance 的作用域限定为"**同 immutableLevel 平局时的次级裁决**"——只有在两条规则 immutableLevel 相同时，importance 高的才胜出
- 示例：L0(normal) vs L2(critical) 冲突时，L0 直接胜出，L2 的 critical 不参与仲裁；L2(critical) vs L2(normal) 冲突时，L2(critical) 胜出
- 此声明消除"importance 可以跨层影响仲裁"的误解，避免实现者误以为"高 importance 的低权限规则能压过低 importance 的高权限规则"

**规则状态**：

| 状态 | 含义 | 所在位置 |
|------|------|---------|
| active | 正式生效的规则 | domainRules |
| candidate | 待交叉验证的候选规则 | ruleCandidates |
| archived | 已废弃或被替代的历史版本 | topicEvolutionArchive |
| pending_review | 需人工审核（L0/L1 修改请求） | 单独队列 |

#### 3.6.3 主题层级与规则的作用域边界

topicPath 是一个**层级化数组**，从粗到细逐级展开。每一层都可以有自己独立的规则集——不同层级的规则互不污染。

**层级结构示例**：

```
全屋定制
├── 设计
│   ├── 厨房设计     ← 规则："台面高度85cm""水槽位置""油烟机型号偏好"
│   ├── 卫生间设计   ← 规则："防水等级""瓷砖规格""干湿分离要求"
│   └── 客厅设计
├── 报价             ← 规则："利润率""套餐模板""折扣权限"
├── 客户沟通
├── 拆单
└── 生产
```

**层级匹配规则（前缀匹配）**：

| 当前 topicPath | 能匹配到哪些规则 | 不能匹配到 |
|--------------|---------------|----------|
| `["全屋定制"]` | topicPath 前缀为 `["全屋定制"]` 的所有规则——包括"设计"、"报价"、"生产"等全部子层规则 | — |
| `["全屋定制", "设计"]` | 前缀为 `["全屋定制", "设计"]` 的规则——包括"厨房设计""卫生间设计""客厅设计" | `["全屋定制", "报价"]` 的规则（兄弟节点不可见） |
| `["全屋定制", "设计", "厨房设计"]` | 仅 `["全屋定制", "设计", "厨房设计"]` 自身的规则 | "卫生间设计""报价"等的规则（隔层不可见） |
| `["全屋定制", "报价"]` | 仅 `["全屋定制", "报价"]` 自身的规则 | "厨房设计"等的规则 |

**核心纪律**：
- **上级可见于下级**：`["全屋定制", "设计"]` 层的规则对 `["全屋定制", "设计", "厨房设计"]` 可见——因为前者是后者的前缀
- **下级不可见于上级/兄弟**：`["全屋定制", "设计", "厨房设计"]` 的规则不会注入到 `["全屋定制", "设计", "卫生间设计"]` 或 `["全屋定制", "报价"]` 的上下文中
- **层级深度与上下文窗口的关系**：当前 topicPath 越深，匹配到的规则集越小——这是认知局部性原则在规则注入层的自然体现

**规则提炼时的作用域要求**：

每一轮对话和每一个提炼出来的规则都必须带 topicPath 和 source 标记。模型推断的候选规则（model_inferred）在写入 ruleCandidates 时，必须附带当前对话所处的 topicPath。

#### 3.6.4 规则网络结构

领域规则不再是平铺的列表，而是一个有向图——每条规则是一个节点，节点之间通过关联边建立关系。

**节点定义**：

```json
{
  "ruleId": "kitchen_countertop_height_001",
  "content": "厨房台面高度建议85cm",
  "topicPath": ["全屋定制", "设计", "厨房设计"],
  "source": "scenario_seed",
  "immutableLevel": "L1",
  "importance": "normal",
  "status": "active",
  "conditions": {
    "userHeight": ">=170cm"
  },
  "edges": {
    "conflicts_with": ["kitchen_countertop_height_002"],
    "depends_on": [],
    "strengthens": ["ergonomic_guideline_003"]
  },
  "version": 1,
  "createdAt": "2026-07-09T10:00:00Z",
  "updatedAt": "2026-07-09T10:00:00Z"
}
```

**关联边类型**（统一 snake_case + 主动语态）：

| 边类型 | 含义 | 注入时行为 |
|--------|------|----------|
| **depends_on** | A 规则成立的前提是 B 规则已满足 | 匹配 A 时，检查 B 是否已激活；若否，主动询问 B |
| **conflicts_with** | A 和 B 互斥，不能同时应用 | 同时匹配到 A 和 B 时，触发询问让用户二选一 |
| **strengthens** | A 规则强化 B 规则的可信度 | 匹配到 A 时，B 的权重 +1，同层级内优先注入 |

**命名纪律**：关联边统一使用 snake_case（`depends_on` / `conflicts_with` / `strengthens`），语义统一为主动语态——当前规则是动作的发起者。`strengthens` 表示"当前规则强化了目标规则"，不是"被目标规则强化"。

#### 3.6.5 规则生命周期

```
主题确认（ANALYZING）
  │
  ├── 加载场景种子规则（scenario_seed）→ domainRules
  ├── 主动询问用户规则（user_initiated）→ ruleCandidates
  │
  ▼
IN_SESSION 运行
  │
  ├── 每轮：按 topicPath 做规则网络子图匹配 → 精准注入
  ├── 遇到冲突或不满足条件 → 主动询问用户
  ├── 模型 infer 出新规则（model_inferred）→ ruleCandidates
  │
  ▼
环节结束（complete / giveup / off-task）
  │
  ├── checkpoint → 推入 ruleMiningQueue
  ├── abandoned → 推入 ruleMiningQueue
  │
  ▼
N10 后台异步处理
  │
  ├── 压缩 stateChange 摘要（轻量 LLM）
  ├── 提取候选规则（继承 immutableLevel）
  ├── 自动推断规则间关联边（conflicts_with / strengthens / depends_on）
  │
  ▼
用户确认 → 升格为正式规则 → 写入 domainRules
```

#### 3.6.6 规则注入匹配逻辑（上下文管理器匹配层 + 决策层）

每轮用户输入的处理流程（由上下文管理器的匹配层和决策层执行）：

1. **匹配层**：获取当前 topicPath（如 ["全屋定制", "设计", "厨房设计"]），在 domainRules 中做子图匹配：
   - 匹配 topicPath 前缀一致的规则节点
   - 沿 depends_on 和 strengthens 边向外扩一层
   - 检查 conditions 是否满足
   - 冲突检测：若匹配到的规则中有 conflicts_with 关系的节点 → 标记冲突
   - 条件不满足检测：若匹配到的规则 conditions 字段有未满足的条件 → 标记缺失条件
2. **决策层**：按优先级决定注入哪些内容：
   - 冲突处理：按 immutableLevel 决定询问策略（L0 直接覆盖，L1 推荐 vs L2 用户偏好）
   - 条件缺失处理：生成询问问题
   - 注入排序：L0 > L1 > L2 > L3 → **同层级内按 importance 排序：critical > high > normal > low** → 同 importance 内按 strengthens 边的权重排序 → 同 importance 同 strengthens 权重时，按 ruleId 字典序（确定性 fallback，避免同分乱序）
3. **拼接层**：把决策结果组装成注入包，注入 prompt

**strengthens 权重累加上限**：
- 同一目标规则被多条 strengthens 边指向时，权重累加上限为 **+3**（cap）
- 累加超过 +3 的部分不再计入排序权重——避免一条规则被 N 条 strengthens 加权后垄断排序首位
- cap 值通过 tunable 参数 `strengthens_weight_cap`（默认 3，取值范围 [1, 10]）配置，由上下文管理器决策层消费

**注入包格式**：

```json
{
  "matchedRules": [
    {
      "ruleId": "...",
      "content": "...",
      "immutableLevel": "L1",
      "matchType": "exact_topicPath"
    }
  ],
  "conflictsDetected": [
    {
      "ruleA": "rule_id_1",
      "ruleB": "rule_id_2",
      "resolution": "pending_user_choice"
    }
  ],
  "missingConditions": [
    {
      "ruleId": "...",
      "condition": "userHeight",
      "question": "您的身高大概多少？这样我能更精准地建议台面高度。"
    }
  ]
}
```

**与认知局部性的关系**：模型不再收到"这个领域的所有规则"，而是只收到"当前主题路径上直接相关的那几条规则及其直接关联的规则"。这使上下文窗口占用从 O(领域规则总数) 降为 O(当前主题路径规则的子图规模)，是认知局部性原理在领域规则层的落地。

#### 3.6.7 规则提炼队列处理

```
checkpoint / abandoned
  │
  ▼
推入 ruleMiningQueue
  │
  ▼
N10 后台进程消费
  │
  ├── 压缩 stateChange 摘要（轻量 LLM）
  ├── 提取候选规则（model_inferred，L3）
  ├── 自动推断关联边：
  │   ├── 若两条规则 topicPath 相同且内容矛盾 → conflicts_with（标记为 inferred_conflict 候选状态）
  │   ├── 若一条规则引用了另一条的结果 → depends_on
  │   └── 若一条规则与另一条方向一致 → strengthens
  │
  ▼
写入 ruleCandidates
  │
  ▼
等待用户确认
  │
  ├── 确认 → 升格为 L2，写入 domainRules
  ├── 修改 → 按用户修改后写入，标记 L2
  └── 否决 → 丢弃或标记 rejected
```

**inferred_conflict 候选状态纪律**：

| 状态 | 含义 | 生命周期 | 是否参与运行时冲突检测 |
|------|------|---------|---------------------|
| `inferred_conflict` | 自动推断的冲突关系候选，待用户确认 | N10 自动推断后写入；用户确认前 | ❌ 不参与——避免未确认的 LLM 语义判断直接影响运行时仲裁 |
| `active_conflict` | 经用户确认生效的正式冲突关系 | 用户确认 inferred_conflict 后升格；写入 domainRules 的 edges 字段 | ✅ 参与——§3.6.6 注入匹配逻辑触发询问 |

**纪律依据**：自动推断"两条规则内容矛盾"本身就是 LLM 语义判断，引入了新的概率性环节。强制候选状态 + 用户确认升格，把概率性环节封在异步提炼阶段，运行时仍只消费确定性结果。

**降级处理**：若 inferred_conflict 长期未确认（超过 ruleMiningQueue 的处理周期），不应自动升格为 active_conflict，也不应直接丢弃——应保留为 inferred_conflict 状态，并在下一轮用户确认窗口中提示。

**交叉验证策略按来源区分**：
- scenario_seed（L0/L1）：不走交叉验证，设计者预设可信
- user_initiated（L2）：需交叉验证（用户可能口头说错或与既有规则矛盾）
- model_inferred（L3）：需交叉验证 + 用户确认升格

### 3.7 跨会话记忆机制

态控架构为两种 taskType 分别提供跨会话记忆机制：

| 模式 | 机制 | 匹配方式 | 核心存储 |
|------|------|---------|---------|
| topic_based | topicEvolution | topicId 语义锚点 + embedding 检索 | 四表（topicEvolution / topicEvolutionEvent / topicEvolutionArchive / topicIndex） |
| field_based | sessionCheckpoint | userId 精确匹配 + DET ttl 检查 | 单表（sessionCheckpoints） |
| 通用（两种 taskType 共享） | roomConversationLog | userId + roomId 精确匹配 | 单表（roomConversationLog） |
| field_based 规则演化 | contractEvolution | stepName 精确匹配 | 复用 contractStore 三表 |

**roomConversationLog 的对话分段保留策略**（v5.7 新增，适用于两种 taskType）：

roomConversationLog 是房间级对话原文记录，保留期独立于 sessionCheckpoint.ttl。v5.7 采用**基于产出物确认节点的分段保留机制**——产出物确认前的对话压缩为摘要，确认后全量保留。

**三种情况**：

| 情况 | 保留方式 |
|------|---------|
| 只有对话，无产出物 | 全量保留（没有节点可分界） |
| 产出物确认前的对话 | 压缩为摘要（模型后台生成，用户不可见） |
| 最后产出物到当前 | 全量保留 |

**摘要分级保留**：

| 摘要重要性 | 保留数量 | 来源 |
|-----------|---------|------|
| critical | 保留最近10个 | critical 产出物对应的对话摘要 |
| high | 保留最近5个 | high 产出物对应的对话摘要 |
| normal | 保留最近2个 | normal 产出物对应的对话摘要 |

摘要生成时机：产出物确认保存后（DET复验通过+用户确认），模型后台生成摘要，用户不可见。生成后原始对话轮次删除，替换为摘要记录。清理时机：新摘要生成时立即检查，超过保留数量删除最旧的（FIFO）。

SLACK_NODE（闲聊）不走分段保留——保留最近10轮（FIFO），不生成摘要。

分段保留通过 tunable 参数 `conversation_segmentation_enabled`（默认 true）控制启用，场景可禁用退回全量保留。摘要保留数量通过 `summary_retention_critical/high/normal` 配置。sessionCheckpoint.ttl 到期时只清进度行，**不联动清理 roomConversationLog**。

**conversationArchive 全量文本备份**（v5.8 新增，架构级默认）：

v5.7 分段保留将产出物确认前的原始对话压缩为摘要，细节可能丢失。conversationArchive 作为全量文本备份，保留所有被压缩的原始对话，不参与拼接，仅供用户搜索回溯。

**数据流**：
```
原始对话 → roomConversationLog（segmentType='full'，参与拼接）
  ↓ 产出物确认后
压缩为摘要 → roomConversationLog（segmentType='summary'，参与拼接）
原始对话移到 → conversationArchive（全量文本备份，不参与拼接，仅供搜索）
```

**设计纪律**：
1. conversationArchive **不参与拼接**——buildPromptContext() 不读取此表
2. conversationArchive **全量保留**——不自动清理（纯文本成本极低），用户可手动清理
3. 只有产出物确认后触发压缩时，原始对话才移到 conversationArchive
4. 无产出物的对话（情况1）保留在 roomConversationLog，不移到 conversationArchive
5. 支持全文搜索（FTS5），用户可通过关键词搜索历史对话
6. 通过 tunable `conversation_archive_enabled`（默认 true）控制启用

**搜索功能**：用户说"搜索猫粮" → DET查询 conversationArchive FTS MATCH '猫粮' → 返回匹配的对话轮次（房间、时间、内容）→ 用户可查看原始对话细节。

#### 3.7.1 topicEvolution（topic_based 专属）

主题演化体系——基于 topicId 的语义锚点，支持三层注入（已生效领域规则 + 关联主题摘要 + 分层演化事件包）、异步规则提炼、changeLevel 分层留存（major 永久 / minor 压缩 / invalid 归档）。

**机制本体分三层递进**：

**A 最底层——分层留存**：按信息净价值把每个事件判为 major、minor 或 invalid 三级，major 永久保留，minor 超阈值后汇总压缩，invalid 直接归档不入主表，判定权交还给模型在 turnType 中顺带输出，DET 不参与语义判定。三级判定基于用户在交互中的明确行为信号——例如版本号从 1.0 改为 2.0 是 major，从 2.0 改为 2.1 是 minor，用户明确说"这个内容不用，取消"是 invalid。判定依据是用户交互中的明确行为，不是模型的元认知自省。

**B 中间层——主题关联**：在 topicIndex 中建立八种关联关系，核心纪律是"关联在元数据层，隔离在数据层"，注入时只拉摘要卡片不拉原始事件链，模型知道隔壁主题的结论但不知道隔壁主题的聊天记录。

**C 最顶层——异步规则提炼**：从多条同类主题的演变流程和关联关系中提取被多次验证的行为模式，候选规则经交叉验证和用户确认后存入领域规则库，同类型新主题初始化时自动注入。规则提炼应为低频行为。

**领域规则体系**：规则是一个包含权限分层（L0-L3）、来源分类（scenario_seed / user_initiated / model_inferred）、关联边（depends_on / conflicts_with / strengthens）和条件匹配（conditions）的规则网络图。注入方式"按 topicPath 子图匹配精准注入"。完整定义见 §3.6。

**topicEvolution 与 roomConversationLog 的边界**：

| | topicEvolution | roomConversationLog |
|---|---|---|
| 记录什么 | 结构化变更事件（stateChange before/after/delta + askingField + modelOutputSummary） | 逐字对话原文 |
| 视角 | 演化追溯——记录"主题怎么变的" | 上下文恢复——记录"当时聊了什么" |
| 是否重复 | ❌ 不重复——同一轮对话的两种视角，event 是结构化摘要，log 是原文 | ❌ 不重复 |
| 压缩联动 | minor event 压缩为摘要时，对应 roomConversationLog 原文轮次归档（不删，但不再注入上下文） | 跟随 topicEvolution 的压缩节奏归档 |
| 消费方 | 上下文管理器三层注入 | 上下文管理器拼接层 |

#### 3.7.2 sessionCheckpoint（field_based 专属）

field_based 模式的原生跨会话进度记忆——记录用户在一个多步骤流程中的进度和各步骤已采集字段快照。完整 SQL DDL 见附录 C.3。

**调度器恢复流程**：

```
1. 用户唤醒
2. ANALYZING 阶段 DET 查 sessionCheckpoints
   ├─ 无记录 ∥ 已过期 → 正常进 P0
   └─ 有记录且未过期 →
        生成 resume_chat intent → 询问用户"上次做到 X，继续还是去其他环节？"
        ↓
        确认"继续" → 注入 completedSteps + stepSnapshots
                    N4 找第一个未完成步骤 → 路由到该步骤
        用户说"去 Nj" → 校验 Nj 前置依赖全部已满足
                    → 注入 stepSnapshots["nj"]（Nj 上次收敛快照）
                    → 路由到 IN_SESSION(nj)
        用户拒绝 → 清空 checkpoint，正常进 P0
```

**房间比喻**：每个环节是独立房间——返回 N1 时只注入 N1 的快照，不会混入 N10 的内容。

#### 3.7.3 contractEvolution：field_based 的领域规则演化

contractEvolution 是 §3.6 领域规则体系在 field_based 模式下的具体形态。与 topic_based 的 topicEvolution 不同——topicEvolution 管"主题怎么演化"，contractEvolution 管"环节内部的设计规则怎么变"。contractEvolution 是 topicEvolution 在 field_based 模式下的对位机制——前者按 stepName 精确匹配，后者按 topicId 语义匹配，两者互不替代。

**存储**（复用 contractStore 现有表，按 stepName 隔离）：

| 表 | 用途 | field_based 行为 |
|----|------|----------------|
| domainRules | 已生效领域规则 | ruleId 含 stepName 前缀，按 stepName DET 精确匹配注入 |
| ruleCandidates | 待验证候选规则 | 来源：每个 IN_SESSION complete 时模型推断的用户偏好 |
| ruleMiningQueue | 提炼队列 | 按 stepName 分队列——MiningQueue 的每条记录标注 stepName，异步提炼按 stepName 分组处理 |

**changeLevel 判定规则（field_based）**：

| 级别 | 判定标准 | 留存 |
|------|---------|------|
| major | 字段的结构性变更——新增/删除必采字段、修改字段值域、改变 Schema 结构 | 永久保留，不可压缩 |
| minor | 字段值微调——默认值修改、取值范围调整、措辞优化 | 30 条压缩为 1 条摘要 |
| invalid | 完全撤销——用户改了又改回原值（与上一版 contract-out 完全相同） | 归档后定期删除（90 天） |

**三层注入**（由上下文管理器执行）：

| 层 | 内容 | 匹配方式 |
|----|------|---------|
| 第0层 | 已生效领域规则 | stepName 精确匹配 + immutableLevel 过滤（L1 及以上） |
| 第1层 | 候选规则卡片 | stepName 匹配 + validationRate ≥ 阈值 |
| 第2层 | 变更历史 | 同一 stepName 的近期变更摘要（major 3 条 + minor 压缩摘要 + 当前规则状态快照） |

#### 3.7.4 contractEvolution 与 sessionCheckpoint 的关系

**设计原则**：
- 态控调度器只负责三件事：**全量记录 + 正确路由 + 资源协调**。三个数据源按各自匹配键全量记录，不做注入决策——roomConversationLog 按 userId+roomId，sessionCheckpoints 按 userId，contractStore 按 stepName 或 topicPath
- 每个环节是自治的——它的上下文管理器从态控调度器交给它的内容中按自己的拼接规则组装 prompt。上下文管理器是独立状态机，不执行环节宪法
- ttl 到期自动清除 sessionCheckpoints 进度行——设计假设：7 天内没继续的任务大概率被放弃了。但 roomConversationLog 不联动清理，按 v5.7 对话分段保留机制独立管理

**contractEvolution 与 sessionCheckpoint 的对位**：

| | contractEvolution | sessionCheckpoint |
|---|---|---|
| 记录什么 | 规则变更历史 | 流程进度 |
| 匹配方式 | stepName 精确 | userId 精确 |
| 留存策略 | 分层留存（永久/压缩/归档） | ttl 到期清除进度行（roomConversationLog 不联动） |
| 消费方 | 上下文管理器（三层注入） | 态控调度器（ANALYZING 路由） |

#### 3.7.5 跨会话记忆对比

| 维度 | topicEvolution | sessionCheckpoint | contractEvolution | roomConversationLog |
|------|---------------|-------------------|-------------------|---------------------|
| 适用 taskType | topic_based | field_based | field_based | 通用（两种 taskType） |
| 记录内容 | 主题演化历史 | 流程进度 + 字段快照 | 环节内规则变更历史 | 房间对话原文 |
| 存储规模 | 4 表 + topicIndex 9 张 | 1 表 | 复用 contractStore 3 表 | 1 表 |
| 检索方式 | embedding 两阶段 | DET 精确匹配 | DET stepName 精确匹配 | DET userId+roomId 精确匹配 |
| 留存策略 | 分层留存（永久/压缩/归档） | ttl 到期清除进度行 | 分层留存（永久/压缩/归档） | 按 importance 分级（永久/90天/30天/7天） |
| 注入方式 | 上下文管理器自决 | 态控调度器路由 + 上下文管理器注入字段 | 上下文管理器自决 | 上下文管理器拼接层 |
| changeLevel 判定 | 核心实体增删→major，微调→minor，撤销→invalid | 无（进度不是变更） | 字段结构变更→major，值微调→minor，改回原值→invalid | 无（原文不是变更） |

贯穿三层有两条铁律：一是人作为终极容错闭环，系统自动发现、自动判定、自动提炼，但主题名、关键词、关联关系、领域规则全部必须经用户确认后才生效；二是认知局部性不因主题关联而妥协，关联主题的原始数据不注入当前上下文，只注入经过提炼的结论摘要。整个机制的本质不是存储空间管理，是按信息净价值做筛选——大版本保全过程理解，小版本防膨胀，无效过程减噪声，关联在元数据层不混不孤立，规则异步沉淀越用越老练。

### 3.8 统一环节操作模型——五步流水线

态控架构的两种任务模式（field_based / topic_based）遵循相同的五步操作流水线。差异只在具体实现参数中——不是两套系统，是同一套系统的两种环节配置。

```
┌─────────────────────────────────────────────────────┐
│  步骤一：状态机路由                                    │
│  ┌─────────────────────────────────────────────────┐│
│  │ 用户输入到达 ANALYZING                           ││
│  │   ├─ 元指令匹配（唤醒/退出/取消）→ 直接处理         ││
│  │   ├─ sessionCheckpoint 恢复检查（field_based 专属） ││
│  │   ├─ topicIndex 语义检索（topic_based 专属）        ││
│  │   └─ 意图识别 → 确定目标环节                       ││
│  │ 前置依赖校验 → 通过? → 路由到 IN_SESSION(目标环节)  ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│  步骤二：环节上下文加载                                │
│  ┌─────────────────────────────────────────────────┐│
│  │ 从三个存储来源加载环节上下文:                       ││
│  │   ├─ roomConversationLog(roomId) 最近N轮对话      ││
│  │   ├─ sessionCheckpoints.stepSnapshots[环节]       ││
│  │   └─ contractStore.domainRules(stepName) 已生效规则 ││
│  │ topic_based 额外加载: topicEvolution + topicIndex  ││
│  │ field_based: stepName DET 精确匹配                 ││
│  │ topic_based: topicId embedding 语义匹配            ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│  步骤三：内容拼接（环节自治——上下文管理器三层职责）     │
│  ┌─────────────────────────────────────────────────┐│
│  │ 上下文管理器执行匹配→决策→拼接三层确定性职责:       ││
│  │   ├─ 匹配层：规则子图匹配、冲突检测、条件检查         ││
│  │   ├─ 决策层：按 importance/immutableLevel/strengthens ││
│  │   │           权重决定注入哪些内容                  ││
│  │   └─ 拼接层：组装 prompt，做 token 预算分配与截断    ││
│  │     ├─ 对话窗口: 取最近N轮（默认20轮）              ││
│  │     ├─ 结构化字段: 注入collectedFields（仅field_based）││
│  │     ├─ 规则注入: 按决策层结果注入                   ││
│  │     └─ 搜索片段: 用户引用历史时追加匹配片段          ││
│  │ 上下文管理器是独立状态机——做匹配→决策→拼接，不执行环节宪法││
│  │ 态控调度器不干预拼接                                 ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│  步骤四：模型计算                                     │
│  ┌─────────────────────────────────────────────────┐│
│  │ 模型无状态——它只看到拼接好的上下文 + 用户最新输入    ││
│  │ 在局部宪法约束下自主完成:                          ││
│  │   ├─ 多轮追问                                     ││
│  │   ├─ 字段/值域校验（field_based: DET辅助）         ││
│  │   ├─ 语义边界判断（topic_based: embedding辅助）    ││
│  │   └─ 收敛检测 → turnType=complete               ││
│  │ 出口: 结构化JSON（contract-out schema）           ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│  步骤五：记录落盘                                     │
│  ┌─────────────────────────────────────────────────┐│
│  │ 实时记录: 每turn → roomConversationLog追加一行    ││
│  │ 收敛记录: complete → sessionCheckpoint更新快照    ││
│  │ 变更记录: changeLevel event → contractStore       ││
│  │ 规则提炼: ruleMiningQueue积压 → N10异步提炼        ││
│  │ 上下文清空: 物理清除LLM上下文，仅结构化数据持久化     ││
│  └─────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────┘
```

**五步流水线中的两种环节配置**：

| 步骤 | field_based 配置 | topic_based 配置 |
|------|-----------------|-----------------|
| 路由 | stepName 精确匹配，用户说"去N1"即切 | topicId embedding 检索，"上次那个写诗的项目"需语义匹配 |
| 切换确认 | 进度保护——有未提交字段时弹"保存后切换?" | 意图确认——embedding匹配到多个topicId时弹"是哪一个?" |
| 上下文 | collectedFields + roomConversationLog(roomId) | topicSnapshot + topicEvolution events + roomConversationLog(roomId) |
| 校验 | DET字段白名单 + 值域硬校验 | LLM语义边界 + DET confidence软校验 |
| 出口 | JSON Schema硬卡死，字段齐全=完成 | event类型判断，changeLevel分层留存 |
| 记忆 | sessionCheckpoint + roomConversationLog | topicEvolution + roomConversationLog |

**核心纪律**：
- 态控调度器只管步骤一（路由）和步骤五（记录），不干预步骤二三四
- 上下文管理器管步骤三（匹配→决策→拼接）——独立状态机，不执行环节宪法
- 环节宪法管步骤四（模型行为规则）
- 五步流水线的**接口统一**：两种 taskType 在外部接口上完全相同（路由信号→上下文加载→拼接输出→模型计算→落盘确认）。**内部实现分支**：步骤一匹配模式（DET vs embedding）、步骤二加载源（sessionCheckpoint vs topicEvolution）、步骤三锚点模式（字段硬门控 vs 三层注入+两阶段检索）、步骤四校验方式（值域 vs 格式）、步骤五落盘目标（sessionCheckpoint vs topicEvolution event）各有差异

### 3.9 @importance 重要性分层机制

与 `immutableLevel`（权限轴：谁能改）正交的第二轴——重要程度轴（哪个更关键）。房间多、规则密，需要一套标记告诉上下文管理器"有限的 token 预算优先分给谁"。

**三层结构，独立计数：**

| 层级 | 标记位置 | 说明 | 示例 |
|------|---------|------|------|
| **房间级** | 环节宪法头部 `<!-- @importance -->` | 同一 taskType 下所有环节中，哪个最重要 | field_based 16 个环节中 N1 为 critical；topic_based 全屋定制中设计环节为 critical |
| **产出物级** | 环节宪法 `<!-- @importance:deliverable -->` | 同一环节的 machineReadable 中哪个产出物是核心 | N1 的 doList 为 critical；设计方案的厨房设计为 critical |
| **规则级** | 领域规则节点 `ruleNode.importance` | L0/L1/L2/L3 每层独立，各层内最多 1 条 critical | L0 安全红线为 critical；L1 某条基线规则为 critical |

**枚举值及约束：**

| 值 | 含义 | 每层上限 | 作用 |
|------|------|---------|------|
| critical | 极其重要——此层内最关键 | 1（必须用户确认） | 对话摘要保留10个（v5.7）；规则排prompt头部；UI高亮 |
| high | 高重要 | ≤ ⌊该层总数/3⌋，可为 0 | 对话摘要保留5个（v5.7）；规则优先注入 |
| normal | 普通（默认） | 无限制 | 对话摘要保留2个（v5.7）；正常排序 |
| low | 低重要 | 无限制 | 对话不生成摘要（v5.7，SLACK_NODE保留最近10轮）；可跳过注入 |

**消费方：上下文管理器拼接层策略**

buildPromptContext() 在组装 token 预算时按 importance 分级：

1. **roomConversationLog 拼接**（v5.7 对话分段保留）：拼接时按重要性选择注入——优先注入 segmentType='full' 的全量对话（最后产出物到当前）→ 然后注入 segmentType='summary' AND summaryImportance='critical' 的摘要 → 如果还有token预算注入 high 摘要 → 还有预算注入 normal 摘要。turnHistory_limit 控制全量对话的注入轮数，摘要注入独立于 turnHistory_limit。

2. **规则注入排序**：模型注意力在 prompt 头部衰减最慢。规则注入按 `ruleNode.importance` 排序——critical→high→normal→low。同 importance 内按 strengthens 权重。

3. **冲突仲裁**：两条同 immutableLevel 的规则冲突时，importance 高的直接胜出，不触发 LLM 调用。**跨层冲突时（如 L0 vs L2）按 immutableLevel 直接裁决，importance 不参与跨层仲裁——详见 §3.6.2 跨层仲裁的边界声明**。

约束纪律：high 是稀缺标签——不能随手标。极端合法配置：每层 1 critical + 0 high + 其余全 normal。

**防偷懒规则——每层必须恰好 1 条 critical 或显式声明 allow_no_critical：**

| 层级 | 是否允许 0 critical | 说明 |
|------|-------------------|------|
| 房间级 | ❌ 必须恰好 1 条 | 同一 taskType 下必须有一个核心环节——用户确认 |
| 产出物级 | ❌ 必须恰好 1 条 | 同一环节的 machineReadable 中必须有一个核心产出物——用户确认 |
| 规则级 L0 | ❌ 必须恰好 1 条 | 根宪法 8 条中安全红线即为 critical——设计者确认 |
| 规则级 L1 | ❌ 必须恰好 1 条 | 基线规则中必须有一个命门——设计者确认 |
| 规则级 L2 | ⚠️ 可 `allow_no_critical: true` | 用户确认规则中可能没有"唯一最关键"的偏好——在环节宪法 `@outputs` 旁标注理由 |
| 规则级 L3 | ⚠️ 可 `allow_no_critical: true` | 模型推断规则天然没有"唯一最关键"——在环节宪法 `@outputs` 旁标注理由 |

`allow_no_critical` 由 N2 覆盖率校验时审查。申请时在宪法头部 `<!-- @importance -->` 旁标注：`<!-- @importance:allow_no_critical reason="L2层均为平权用户偏好，无唯一命门" -->`。不允许"0 critical + N high"绕过——要么恰好 1 critical，要么显式声明 0 critical + 理由。



## 3.10 项目隔离机制（v5.6 新增）

### 解决什么

同一环节（如 N1 场景定义）下，不同智能体项目的产出物需要隔离。例如：
- 项目A：富贵小安（field_based 记账）—— N1 产出物是记账场景定义
- 项目B：杂碎本（可能 topic_based 创作）—— N1 产出物是创作场景定义

两个项目共用 N1 环节宪法，但产出物内容必须隔离。

### 数据结构

#### 全局字段

```javascript
const schedulerState = {
  _state: 'IDLE',
  _currentProjectId: 'fugui-xiaoan',  // 全局当前项目
  _activeSession: { ... },
  ...
};
```

**currentProjectId 是全局字段**——一处变更，所有房间的当前项目一起变更。

#### 新增表：projectRegistry

```sql
CREATE TABLE IF NOT EXISTS projectRegistry (
    projectId TEXT PRIMARY KEY,
    projectName TEXT NOT NULL,
    projectKeywords TEXT,             -- JSON数组，用于关键词匹配
    projectDescription TEXT,
    taskType TEXT,
    createdAt TEXT NOT NULL,
    lastActiveAt TEXT NOT NULL,
    status TEXT DEFAULT 'active'
);
```

#### 新增表：userLastProject

```sql
CREATE TABLE IF NOT EXISTS userLastProject (
    userId TEXT PRIMARY KEY,
    lastProjectId TEXT NOT NULL,
    lastActiveAt TEXT NOT NULL,
    FOREIGN KEY (lastProjectId) REFERENCES projectRegistry(projectId)
);
```

#### 现有表加 projectId

| 表 | 现有主键/索引 | 新增字段 |
|------|-------------|---------|
| roomConversationLog | userId + roomId + createdAt | + projectId |
| sessionCheckpoints | userId | + projectId（主键改为 userId + projectId） |
| outputRegistry | roomId + outputId | + projectId |
| roomStateIndex | userId + roomId | + projectId（主键改为 userId + roomId + projectId） |
| topicEvolution | topicId | + projectId（topicId 格式改为 projectId_topicSlug_date） |

### 项目切换流程

#### 默认项目加载

用户唤醒 → 进入任意房间 → 查 userLastProject 表 → 读取 lastProjectId → schedulerState.currentProjectId = lastProjectId → 所有后续操作都带此 projectId

#### 项目切换（S3 物理隔离）

```
用户说"切换到杂碎本"
  ↓
ANALYZING 关键词匹配 projectRegistry.projectKeywords
  ├─ 匹配到唯一项目 → projectId="zasuiben"
  ├─ 匹配到多个项目 → 展示候选列表，用户选择
  └─ 未匹配 → 提示"未找到该项目，是否新建？"
  ↓
S3 物理隔离四步：
  1. 释放当前项目 KV Cache（abort_request）
  2. 归档当前项目状态（sessionCheckpoint + roomConversationLog 最后状态）
  3. 加载新项目的最后状态（从 contractStore 读取新项目的 checkpoint + roomConversationLog）
  4. 创建新推理会话
  ↓
更新 schedulerState.currentProjectId + userLastProject 表
  ↓
全局联动——所有后续房间操作都用新 projectId
  ↓
窗口延续——前端不刷新、不跳转
```

#### 新建项目

用户说"新建项目：XX" → ANALYZING 识别 → 生成 projectId → 写入 projectRegistry → 初始化空的项目状态 → 更新 currentProjectId + userLastProject → 用户从 N1 开始走设计流程

### 项目切换 vs 房间切换

| 维度 | 房间切换（switch口令） | 项目切换（ANALYZING识别） |
|------|---------------------|------------------------|
| 触发 | "切断房间"口令 | 用户说项目名 |
| 识别 | DET EXACT_MATCH（确定性） | ANALYZING 关键词匹配（概率性） |
| S3物理隔离 | ✅ 需要 | ✅ 需要 |
| roomId 变化 | ✅ 变 | ❌ 不变 |
| projectId 变化 | ❌ 不变 | ✅ 变 |
| 加载内容 | 新房间的 checkpoint | 新项目的最后状态 |
| 全局联动 | 不联动（只切房间） | ✅ 联动（所有房间切项目） |
| roomStateIndex 注入 | ✅ 注入 ANALYZING | ❌ 不需要（房间没变） |

**关键区别**：
- 房间切换：切换环节（N1→N5），projectId 不变——"我要去做别的环节"
- 项目切换：切换项目（富贵小安→杂碎本），roomId 不变——"我要做别的智能体"
- 两者都需要 S3：因为 KV Cache 里的上下文都变了，必须物理释放

### 跨项目引用

#### 默认隔离

项目间默认严格隔离——roomConversationLog 不互通，sessionCheckpoint 不互通。

#### 跨项目产出物引用

通过 outputRegistry 跨项目查询实现：

用户在杂碎本项目说"参考一下富贵小安的 N1 设计" → ANALYZING 识别为跨项目引用请求 → 调度器从 outputRegistry 查询 projectId="fugui-xiaoan" AND roomId="N1" 的产出物 → 用户确认后复制产出物到当前项目的 references/ 目录 → 注入当前项目上下文（作为参考，不作为当前项目的产出物）

#### 跨项目引用纪律

- 文件复制是唯一跨项目数据传递方式——不直接注入原文
- 复制需要用户确认——不自动复制
- 复制后作为 references——不污染当前项目的产出物

### 设计纪律

1. 环节宪法不分项目——环节宪法是房间级的，所有项目共用
2. 产出物分项目——roomConversationLog/sessionCheckpoint/outputRegistry 都带 projectId
3. 项目切换需要 S3——和房间切换一样，必须物理释放 KV Cache
4. 项目切换全局联动——currentProjectId 是全局的，切换后所有房间都用新 projectId
5. 跨项目引用走 outputRegistry——文件复制是唯一方式，需要用户确认
6. 新建项目从 N1 开始——没有 checkpoint，从第一步走设计流程
