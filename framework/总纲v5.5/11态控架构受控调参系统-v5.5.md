# 态控架构 · §十 受控调参系统

> **本文件是细则文件**——对应总纲概要的 §十。版本 v5.5。

## 十、受控调参系统（tunable 槽位）

宪法中可声明运行时允许调整的参数，含 type（类型）/ range（范围）/ enum（枚举）/ default（默认值）/ conflict_check（冲突检测）。由系统设置界面自动生成配置表单，冲突检测引擎在保存前校验。参数调整后下一轮新会话生效，变更日志不可篡改。tunable（可调参数）的灵活性是"受控的灵活性"——适应性受限，但可控、可审计、可测试。

与纯 LLM 运行时动态调整的区别：

| 维度 | 纯 LLM 动态调整 | 态控 tunable 槽位 |
|------|--------------|-----------------|
| 调整方式 | 在 prompt 里写条件 | 在系统设置里改参数 |
| 调整范围 | 无限制 | range/enum 硬限制 |
| 冲突检测 | 无 | 确定性代码检测 |
| 审计 | 难 | 变更日志，每次调整可追溯 |
| 生效时机 | 下一轮对话立即生效 | 保存后下一轮新会话生效 |

**关键 tunable（可调参数）参数清单：**

| 参数名 | 默认值 | 取值范围 | 用途 | 消费方 | 适用 taskType |
|--------|--------|---------|------|--------|-------------|

| turnHistory_limit（轮次历史限制） | 20 | [5, 100] | roomConversationLog 拼接时取最近轮数（normal 环节基数；critical 环节 = 基数 × critical_room_history_boost；high 环节 = 基数 × 0.6；low 环节 = 基数 × 0.2）。注意：此参数管"拼接取多少轮"，不管"存多久"——存留期由 room_conversation_log_retention 管 | session-store（会话存储） | 通用 |
| critical_room_history_boost（关键房间历史增强系数） | 2.0 | [1.0, 5.0] | critical 环节的 roomConversationLog 拼接轮次 = turnHistory_limit × 此系数。正常 20 轮，critical 默认 40 轮 | session-store（会话存储） | 通用 |
| **room_conversation_log_retention** | `{"critical":-1,"high":90,"normal":30,"low":7}` | JSON 对象，按 importance 分级；-1=永久；正整数=天数 | **roomConversationLog 保留期，独立于 sessionCheckpoint.ttl。sessionCheckpoint.ttl 到期只清进度行，不联动清理 roomConversationLog** | session-store（会话存储） | 通用（两种 taskType） |

| logprobs_threshold（logprobs阈值） | 0.4 | [0.0, 1.0] | logprobs 检查阈值——ANALYZING 从 API logprobs 获取真实概率分布，DET 将 probability 与此阈值比较，低于此值且 message 含"硬撑式确认"词表→拦截 | DET 复验 | 通用 |
| contract_inheritance_limit（契约继承限制） | 0.3 | [0.1, 0.5] | 跨节点契约继承注入上限（占新节点 contract-in 总字段数的比例，对应 §8.2 的 30% 上限） | 契约继承模块 | 通用 |

| topic_keyword_overlap_threshold（主题关键词重叠阈值） | 0.2 | [0.0, 1.0] | topic_based 模式 DET 关键词粗筛重叠度阈值 | context-manager（topic_based 模式阶段1） | topic_based（基于主题） |

| **changelevel_major_sample_rate** | **0.05** | **[0.01, 0.20]** | **changeLevel major 标记的抽样人工复核比例。冷启动期间临时上调至 0.10** | **抽样人工复核引擎** | **通用** |
| **max_critical_rooms** | **3** | **[1, 10]** | **预计同时活跃的 critical 房间数上限，用于 §10.1 上下文预算约束计算。非硬限制——超出时仅触发 cross_param warning，不阻止运行** | **上下文管理器拼接层（§10.1 约束校验）** | **通用** |

冷启动冻结机制：
冷启动期间参数冻结，但实时观测指标：监测 logprobs 拦截率、topicEvolution event 写入频率、minor 压缩频率、规则提炼触发频率等

应急解冻机制：若指标异常（如 logprobs 拦截率 > cold_start_emergency_threshold），触发"应急解冻"——人工确认后调整参数

解冻后 P 反馈控制自动调节：参数调整 = pid_kp × (目标值 - 实际值)，目标值由 target_logprobs_intercept_rate（默认 0.05）等目标值参数提供。例如 logprobs 拦截率目标 5%，实际 15%，调整 = 0.1 × (0.05 - 0.15) = -0.01，logprobs_threshold 下调 0.01

> 注：当前版本仅启用比例项（Kp），积分项（Ki）和微分项（Kd）预留未启用。如需消除稳态误差可引入 Ki，如需抑制超调可引入 Kd，但当前 P 控制已满足参数微调需求。

所有调整记录变更日志：时间戳、操作者、参数名、旧值、新值、调整原因、触发指标

### 10.1 跨参数联动约束规则

定位：扩展附录 C.2 冲突检测引擎，处理 tunable 参数之间的隐式语义依赖。与 single_param range/enum 校验不同——cross_param 不替用户决策，只做语义校验 + 建议 + 警告。

**规则格式规范：**

| 字段 | 必填 | 说明 |
|------|------|------|
| `type` | 是 | 固定值 `"cross_param"` |
| `name` | 是 | 规则标识符，snake_case |
| `violation_action` | 否 | 默认 `"warning"`——cross_param 违规不阻止保存。可选 `"hard_reject"`（用于确定性越界） |
| `params` | 是 | 被此规则引用的 tunable 参数名数组——仅这些参数变更时重新执行此规则 |
| `constants` | 否 | 计算常量（如模型窗口大小、每轮平均 token 数），声明在此以供引用 |
| `compute` | 是 | 沙箱表达式，仅允许 `+ - * / floor ceil min max` 运算符 + params 变量 + constants 常量。求值结果 true=违规触发 |
| `event` | 是 | 违规事件结构：`{type, severity, message, suggested_adjustment}`。`suggested_adjustment` 可含带 formula 的建议值计算提示 |

**初始规则清单（5 条，唯一来源——附录 C.2 不再重复）：**

**维护纪律：**

- 新增 tunable 参数时同步审查是否需要新增跨参数约束——任何与其他参数存在语义依赖的参数都应至少有一条 cross_param 规则
- 约束不应过度拟合当前模型特性作为常数（如 `model_token_budget=128000`）——此类常数应在 tunable 参数表中暴露为可配置项或至少以 constants 形式集中声明，便于模型升级时统一修改
- 两条参数之间如果存在确定性大小关系（如 topic_anchor_similarity_threshold < topic_evolution_match_threshold），应使用 `hard_reject` 而非 `warning`——这是硬约束，不是警告
- 跨参数约束只覆盖"参数组合可能导致系统行为异常"的情况，不覆盖"这个参数值可能不适合你的业务场景"——后者属于场景设计阶段的人为判断，不是冲突检测引擎的职责
