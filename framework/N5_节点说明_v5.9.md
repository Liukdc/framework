## 一、位置与定位

```
N4 状态转移图 + 路由表（已定稿）
  ↓
N5 调度器核心逻辑 + 上下文管理器 三接口 + taskType 双模式锚点 + topicEvolution/sessionCheckpoint 处理  ← 你在这里
  ↓
N6 数据传递协议 + 守卫规则 + topicEvolution 存储
  ↓
N7-N10 写宪法
```

**【人话版定位】**
N5 把 N4 的"房间+走廊图"翻译成一栋店面的"接待规矩"——客人（用户输入）进门后，经过几道接待台、每道谁管、调不调模型、什么情况回头。

## 四、产出物

N5 产出四样东西：五层接待规则说明书、调度器数据结构完整定义、上下文管理器 三接口规范、N2 双角色串行编排逻辑。

### 4.1 产出一：五层接待规则说明书

#### 第0层：全局硬匹配（DET，0 LLM）

前台保安，站在门口看客人说的第一句话里有没有特殊词——唤醒词、退出词、取消词、align 静默指令。有就立刻处理，不让它进后面的接待台。

| 维度 | 规范 |
|------|------|
| 执行体 | DET（可配置浅匹配器） |
| LLM 调用次数 | 0 |
| 职责 | 匹配唤醒词/退出词/取消词/align 元指令，命中后直接路由，不进入后续各层 |
| 词表来源 | N7 根宪法第2条"元指令集"（唤醒词/退出词/取消词）+ 附录 D（align 元指令，独立声明） |
| 匹配方式 | 可配置：精确匹配 / 子串匹配（默认推荐）/ 编辑距离 / 同义词组 |
| 退出词与取消词优先级 | 退出词优先匹配，命中后不再检查取消词（退出词命中→CLOSING；取消词命中→LISTENING） |
| 适用状态 | 退出词/取消词：所有状态均生效（含 IN_SESSION 中途用户突然说"退出"）。IN_SESSION 中途的退出→走 off-task 路径（N6 v2.7 协议），_pendingCollectedFields 按 source 标记暂存后退出。align 元指令：仅 LISTENING 态生效 |

**纪律**：
- 这一层绝对不调大模型，纯字符串匹配
- 词表直接引用 N7 根宪法，不要自己编

#### 第0.1层：静默模式（align 专用，DET 侧接管前端）

如果客人说了 align 触发词（比如"帮我理一下"），保安不让他进大厅，而是请他进"小黑屋"——一个安静的草稿间。客人在里面可以慢慢打字、反复修改，系统只存着，不做任何分析。等客人说"写完了"，系统把整段文字一次性送去给 align 房间里的专家模型处理。

静默模式下第0.5层关键词扫描暂停。用户输入的跨意图词汇在缓冲区中不做标记，待退出静默模式后由 ANALYZING 统一处理。

| 维度 | 规范 |
|------|------|
| 执行体 | DET（调度器通知前端，前端执行缓冲） |
| LLM 调用次数 | 0 |
| 触发条件 | 第0层命中 /align 元指令 |
| 动作 | 调度器通知前端进入"草稿编辑模式"，挂起语义管道，开启独立缓冲区。此状态下所有用户输入均被静默缓存，不经过第0.5层及之后的任何层 |
| 退出（正常路径） | 收到完成指令后，将缓冲区完整文本送入 IN_SESSION(align)，不经过 ANALYZING（bypassAnalyzing=true） |
| 退出（异常路径） | 退出词仍被第0层监听，命中后清空缓冲区、退出静默模式、回 LISTENING |
| 放弃路径 | 客人说"算了"等取消词 → 回 LISTENING |
| 启用条件 | alignScenarioFlag=true 时启用；false 时本层为 null，跳过 |

#### 第0.5层：DET 关键词扫描（DET，0 LLM）

安检雷达——扫一眼客人说的话里有没有 N1 边界清单里"不做什么"的关键词。只贴标签（offTaskSuspicion），不拦人，不替模型做决定。

"提高怀疑度"的实现方式：第1层 ANALYZING 在 prompt 中看到 offTaskSuspicion 标记时，在 system prompt 中追加一行"注意：用户输入中检测到跨意图关键词 [关键词列表]，请确认用户是否意图切换"；第2层 IN_SESSION 在 prompt 中看到此标记时，在 off-task-detection 的 system prompt 中追加一行"注意：用户输入中检测到跨意图关键词 [关键词列表]，请提高偏离检测敏感度"。

offTaskSuspicion 标记格式：{keywords: ["查询", "删除"], confidence: 0.75}。confidence 表示关键词匹配的强度（匹配到 3 个关键词时 confidence 为 0.8，匹配到 1 个关键词时 confidence 为 0.4）。第1层和第2层可根据 confidence 的不同阈值采取不同行动：confidence > 0.8 时在 prompt 中标注"强烈怀疑意图切换"，confidence 0.5-0.8 时标注"疑似意图切换"，confidence < 0.5 时仅记录日志不标注。

| 维度 | 规范 |
|------|------|
| 执行体 | DET |
| LLM 调用次数 | 0 |
| 职责 | 对每轮用户输入做关键词扫描，检测跨意图词汇（如记账中途说"查一下"） |
| 词表来源 | N1 边界清单"不做什么"列的关键词 |
| 输出 | 标记 offTaskSuspicion 字段（含 keywords 和 confidence），附加到模型输入中作为提示 |
| 纪律 | 只标记，不阻断，不强制覆盖模型判断 |

#### 第1层：意图识别（LLM 快速路由模型，1 LLM）

前台接线员——听一句，判断"这个人想干嘛"，派去对应的房间。必须用快速路由模型（如 Qwen-Turbo），禁止深度推理模型。

**topic_based 任务特殊处理**：
a. ANALYZING 阶段 LLM 生成候选 topicName + keywords
b. 调度器执行关键词匹配（基于 contractStore.topicIndex 表），检测是否已有相近主题
c. 若匹配到已有主题 → 系统展示候选主题名，等待用户确认（续写/重选/新建）
d. 若未匹配到 → 系统展示生成的候选 topicName+keywords，等待用户确认（确认/修改/搜索已有）
e. 用户确认后，调度器生成 topicId（格式为 `{userId}_{intent}_{topicSlug}_{date}`），初始化 topicEvolution 记录（写入 contractStore.topicIndex 和 topicEvolution 表，写入第一个 event，changeLevel 由模型预判）
f. 系统自动检测当前主题与已有主题的关键词重叠度，若高则提示用户是否建立关联
g. **规则问询（v4.2 新增）**：主动问询用户是否有已知的偏好或规则约束（user_initiated 来源）→ 写入 contractStore.ruleCandidates，供后续异步规则提炼升格为正式领域规则
若 intent="meta_instruction"（语义模板层判定为元指令），触发节点切换。

| 维度 | 规范 |
|------|------|
| 执行体 | LLM（快速路由模型，状态级默认） |
| LLM 调用次数 | 1（每轮用户输入触发一次） |
| 模型纪律 | ❌ 禁止深度推理模型 |
| 职责 | 将用户弥散口语收敛为 {intent, subType, taskType, confidence, extracted, topicId(topic_based时)} |
| 宪法文件 | N9 analyzing-session.json |
| 防抖 | 同一会话中同一用户在 60 秒内 intent 在两种类型间来回切换 ≥3 次 → 不重判，锁定首次判定的 intent，追问确认用户真实意图。防抖计数器在会话结束时重置（清空 _antiFlapCounter）。参数从 tunable 读取（antiFlapWindow / antiFlapThreshold），不硬编码 |
| 防抖锁定后的追问规则 | 锁定后，后续的 intent 切换不触发重新判断，调度器直接追问"您是想 [首次判定的 intent] 吗？"用户确认后继续，用户否认后解锁并重新判断。追问轮次不纳入防抖计数 |

#### 第2层：环节内模型自治（LLM 模型实例，环节级自治）

房间里服务员——接线员派客人到了对应的房间，服务员拿着这个房间的服务手册（环节宪法），自己跟客人聊。调度器不参与模型推理过程，但必须消费模型的 turnType 进行状态路由、DET 复验和 topicEvolution/sessionCheckpoint 处理。

**topic_based 任务的 topicEvolution/sessionCheckpoint 处理**：
调度器在 IN_SESSION exit 时，根据 N4 路由表条目的 topicEvolutionEventAppended 和 expectsChangeLevel 标注执行以下动作：

| 触发条件 | 调度器动作 | stateSnapshot 标记 |
|---------|----------|-------------------|
| IN_SESSION exit（complete，expectsChangeLevel=true） | 读取模型返回的 changeLevel 预判（格式规范见下方）；若模型未返回或格式错误（不符合 changeLevel 格式规范），默认 minor | — |
| EXECUTING 完成（topicEvolutionEventAppended=true） | 调用轻量 LLM 生成 stateChange 摘要；使用模型判定的 changeLevel 追加 event | checkpoint |
| giveup | 追加 event；推入提炼待处理队列 | abandoned |
| off-task | 追加 event（记录偏离前的状态） | active（保持） |

**changeLevel 格式规范**：changeLevel 必须为小写字符串，取值必须为 major/minor/invalid 三者之一。大小写不匹配（如 "Major"）、拼写错误（如 "majar"）、中文表述（如 "重要"）均视为格式错误，调度器默认标记为 minor 并记录警告日志。

**分层留存规则**：
- major：永久保留
- minor：累计 topic_evolution_minor_compaction_threshold（默认 5 条）后触发汇总压缩，原 event 迁入 topicEvolutionArchive 表
- invalid：不入主表，迁入 topicEvolutionArchive 归档表

| 维度 | 规范 |
|------|------|
| 执行体 | LLM（模型实例，环节级自治） |
| LLM 调用次数 | 1（整个环节内多轮追问算 1 次模型实例连续会话） |
| 职责 | 模型实例在局部宪法约束下自主追问、自主校验、自主判断偏离。topic_based 任务在返回 turnType 时顺带输出 changeLevel 预判字段 |
| 调度器纪律 | 调度器只调 上下文管理器 接口拼上下文，不内联拼接逻辑。minScoreThreshold 通过 constitution.getTunable('min_match_score') 传入，不硬编码 |
| 宪法文件映射 | 按 intent 加载对应环节宪法，映射关系与 N4 路由表一致 |

**两种任务类型下的 askingField 语义与锚点模式**：

| 任务类型 | 典型场景 | askingField 的语义 | 上下文管理器 锚点模式 | 例子 |
|---------|---------|-------------------|------------------------|------|
| field_based（强收敛） | 记账、改柜体参数、表单填写 | 字段名——当前追问或校验失败的数据库字段名（如 "amount"、"time"） | 字段级硬门控（DET 确定性字符串匹配）——保护与 askingField 同名的历史轮次不被淘汰 | 模型问"多少钱？"→ askingField="amount" |
| topic_based（弱收敛/发散） | 角色扮演、诗歌创作、头脑风暴 | 主题标签——模型动态生成的主题锚点标签（如 "character_motivation"、"plot_conflict"） | 主题级硬门控——三层注入 + embedding 两阶段检索（0 次 LLM 调用） | 模型追问"他为什么背叛？"→ askingField="betrayal_motive" |

**askingField 主次标注规则**：强收敛任务中，如果几个字段是强关联的（如"金额+币种"），可以一轮问两个，但要在 askingField 里标清楚主次，格式为 "amount+currency"。topic_based 任务中，askingField 通常为单个主题标签。

**topic_based 模式的三层注入**（在 DET 关键词粗筛之前，上下文管理器 按优先级注入）：
注入优先级：领域规则 > 关联摘要 > topicEvolution 分层包 > turnHistory

- **第0层：已生效的领域规则注入（按 topicPath 子图匹配精准注入）**——上下文管理器 根据当前 topicPath 在 contractStore.domainRules 中做子图匹配：匹配 topicPath 前缀一致的规则节点 + 沿 depends_on/strengthens 边外扩一层 + 检查 conditions 是否满足。仅注入匹配到的子图规则，而非全量 domainRules。domainRules 中的数据由 N10 规则提炼的后台进程异步写入，调度器在 buildPromptContext() 时执行子图匹配后注入，不触发任何提炼逻辑。如果匹配结果为空（知识真空期），跳过第0层注入，不影响后续流程
- **第1层：关联主题摘要注入**——从 contractStore.topicIndex.relatedTopics 读取关联主题摘要卡片（inheritedPoints / avoidPoints / reason），不拉取关联主题的原始 evt 链
- **第2层：当前主题演化注入**——从 contractStore.topicEvolution 读取分层注入包

**topicEvolution 注入时的分层结构**：

```
[topicEvolution 注入包]
├── majorEvents（永久保留，全部注入，通常 0-5 条，不压缩不丢弃）
├── compressedMinorSummary（最近一次 minor 压缩摘要，1 段）
├── recentMinorEvents（压缩后新增的 minor 事件，通常 0-4 条）
└── currentStateSnapshot（覆盖写，1 条。内容根据 N4 topicEvolutionRouting 中的 stateSnapshotMark 动态选择：
     mark=active → 注入当前活跃状态的快照
     mark=checkpoint → 注入执行完成后的 checkpoint 快照
     mark=abandoned → 注入 abandoned 标记，快照内容为空
     mark=N/A → 不注入快照）
```

**topic_based 模式的两阶段检索**：
- **阶段1**：DET 关键词粗筛（确定性，O(N)）——对 turnHistory 中每轮，DET 提取 askingField 主题标签的关键词，与历史轮次的文本做关键词重叠匹配。重叠度 ≥ topic_keyword_overlap_threshold（tunable，默认 0.2）的轮次进入阶段 2
- **阶段2**：embedding 相似度精筛（embedding 模型调用，非 LLM 调用）——对阶段 1 粗筛通过的轮次，用 embedding 模型计算 askingField 主题标签与历史轮次文本的向量相似度。相似度 ≥ topic_anchor_similarity_threshold（tunable，默认 0.6）的轮次被保护
- **DET 阈值裁决**：DET 只做"相似度 ≥ topic_anchor_similarity_threshold"的最终数值比较，确定性裁决。0 次 LLM 调用

两种模式的共同部分：综合匹配度评分、最低匹配度截断（min_match_score tunable）、偏离处理、archiveGraph、校验失败标记（updateGraph）——这些机制两种模式完全相同。

**askingField null 降级规则**（两种模式通用）：当 askingField 为 null 时（如 reply 轮次、complete 轮次），硬门控保护降级为纯语义匹配。field_based 模式不执行字段名匹配保护，topic_based 模式不执行主题标签匹配保护，均改为 上下文管理器 按综合匹配度评分分配形态。降级不报错不阻断。

**门铃（turnType）六种响声**：

| 门铃声 | 意思 | askingField（field_based） | askingField（topic_based） | 调度器动作 |
|--------|------|--------------------------|--------------------------|----------|
| ask | 服务员在问客人问题 | 当前追问的字段名（如 "amount"） | 当前追问的主题标签（如 "plot_twist"） | 转发消息给用户，继续等。把 askingField 传给 上下文管理器 用于锚点保护 |
| reply | 服务员在回复客人 | null | null | 转发消息给用户，继续等 |
| complete | 服务员说"齐了" | null | null | 送第3层质检（field_based 做值域复验；topic_based 做输出格式校验）；topic_based 任务读取 changeLevel 预判 |
| off-task | 客人跑题了 | null | null | 暂存 collectedFields（默认 24h），把客人跑题内容送回第1层重新判断。topic_based 任务仍追加 event |
| giveup | 客人说"算了" | null | null | 清空 activeSession，回 LISTENING。topic_based 任务追加 event，标记 abandoned，推入提炼待处理队列 |
| validation_failed | 服务员说"金额不对" | 校验失败的字段名（如 "amount"） | 校验失败的主题标签或 "output_format" | 进入 CLARIFYING 状态，复用模型 message 转发用户。调度器不调 LLM 生成追问 |

**两种任务类型的调度器处理对照**：

| 维度 | field_based（强收敛） | topic_based（弱收敛/发散） |
|------|---------------------|-------------------------|
| 收敛目标 | 一组预定义的结构化字段 | 受控的主题空间 |
| complete 条件 | 所有必采字段齐全 | 模型判断当前主题探索已达到满意状态 |
| 第3层 DET 复验 | 值域校验（金额范围、时间格式…） | 输出格式校验（JSON Schema / 文本格式） |
| 半成品暂存内容 | 已采集字段的键值对 | 对话摘要 + 当前主题标签 |
| 半成品摘要来源 | 模型实例在 complete 时顺带输出；若未输出，调度器使用最近一轮的 message 作为摘要 | 同左 |
| 上下文管理器 硬门控 | 字段级：DET 确定性字符串匹配 | 主题级：三层注入 + embedding 两阶段检索（DET + embedding，0 次 LLM） |
| topicEvolution/sessionCheckpoint 处理 | 无 | 模型顺带输出 changeLevel；调度器在 IN_SESSION exit 时追加 event；分层留存 |

**松弛节点（SLACK_NODE）的路由处理**（N1 slackNodeDeclaration.enabled=true 时启用）：
- SLACK_NODE 为稳态闲聊态，不区分 taskType，仅受根宪法约束
- 路由来源：N4 chitchat_path 转移边（ANALYZING→SLACK_NODE，SLACK_NODE↔收敛节点转换，置信度≥0.7 守卫）
- 调度器在 SLACK_NODE 期间的 上下文管理器 行为：field_based 模式（不启用 topicEvolution），不采字段，askingField=null
- SLACK_NODE→收敛节点转换时，ANALYZING 重新执行意图识别，按识别的 intent 分派到对应 IN_SESSION 子类型

#### 第3层：DET 值域复验（五项检查，0 次 LLM，仲裁权归调度器）

质检员——对服务员交上来的结果做一遍确定性校验。

| 检查项 | 执行体 | 时机 | field_based 处理 | topic_based 处理 | 失败路由 | 冷启动特殊处理 |
|--------|--------|------|-----------------|-----------------|---------|--------------|
| L1 结构校验 | DET | complete 时 | JSON.parse + Schema 校验 | 同左 | 重试1次→熔断→硬编码回复 | 无 |
| DET 值域复验 | DET | complete 时 | 值域校验（金额上限、时间格式、必填字段…） | 输出格式校验（JSON Schema / 文本格式 / 字数 / 结构…） | CLARIFYING | 无 |
| confidence 检查 | DET | complete 时 | 两步均 DET：①数值 < threshold（默认 0.4）②pattern 扫"硬撑式确认"词表 | 同左 | CLARIFYING | 冷启动期（cold_start_observation_window 内）：软拦截，标记 confidence_warning 附加到返回中，不进入 CLARIFYING。非冷启动期：硬拦截。应急解冻：若指标异常（如 confidence 拦截率 > cold_start_emergency_threshold），触发人工确认后调整参数 |
| 跨任务延伸检测 | DET（默认）/LLM（例外） | complete 时 | DET 字段白名单（needSemanticExtensionCheck 通常 false） | 主题边界检查（可能 needSemanticExtensionCheck=true 追加轻量 LLM，可复用 ANALYZING 的轻量模型实例。**v4.7 降级路径：辅助 LLM 不可用/超时/置信度低于 confidence_threshold 时，强制退化为 DET 关键词白名单 + 用户主动确认，不允许直接放行**） | CLARIFYING | 无 |
| 硬编码兜底 | DET | 熔断时 | 硬编码回复 | 同左 | LISTENING | 无 |

**校验规则来源**：
- field_based 的值域校验规则：从 N9 环节宪法的 @section validation 结构化 JSON 中读取
- topic_based 的输出格式校验规则：同样从 N9 环节宪法的 @section validation 结构化 JSON 中读取。如果 topic_based 的环节宪法没有定义 validation 规则，则第3层仅跳过 DET 值域复验（依赖 @section validation 规则），不跳过跨任务延伸检测（主题边界检查为语义边界判定，不依赖 @section validation 的输出格式规则）。confidence 检查仍正常执行

仲裁权（根宪法第6条）：当模型内部校验（第一道）与调度器 DET 复验（第二道）结果冲突时，以调度器 DET 为准。

**CLARIFYING 状态**：调度器直接复用模型实例在 validation_failed 轮返回的 message 字段转发用户，不调 LLM 生成追问。用户回应后，将用户回应 + validationResult 作为新一轮输入送回模型实例，继续 IN_SESSION。

**validationResult 格式**：
```json
{
  "field": "amount",
  "issue": "amount_too_large",
  "userInput": "一百万",
  "expectedRange": "0-999999",
  "actualValue": 1000000
}
```
调度器将此对象作为 system prompt 的一部分注入模型实例的下一轮输入，格式为："上一轮校验失败：[field] 的值 [actualValue] 不在允许范围 [expectedRange] 内，请重新询问用户。"

**CLOSING 双出边处理**：调度器在进入 CLOSING 前记录来源标记 _closingSource。_closingSource 为 intent_exit 或 exit_word → 走 IDLE；_closingSource 为 out_of_boundary_hard_reject / out_of_boundary_guided_reject / out_of_boundary_soft_landing → 走 LISTENING。_closingSource 的取值与 N4 转移边的 exitType 映射关系：intent_exit 和 exit_word 对应 exitType=normal；out_of_boundary* 对应 exitType=fallback。

**topic_based 任务文件写盘后**：
- 调度器调用轻量 LLM，将本轮 collectedFields + message + result 压缩成 stateChange 摘要
- 使用模型判定的 changeLevel 追加 event（major 永久保留 / minor 进入累积压缩流 / invalid 迁入归档表）
- stateSnapshot 标记为 checkpoint，将主题信息推入规则提炼的待处理队列
- 不触发任何提炼逻辑——提炼由独立后台进程异步执行

**清理时机**：
- _activeSession：环节结束（complete/off-task/giveup）时清空
- _antiFlapCounter + _antiFlapLocked：防抖触发用户确认后重置（resetReason="user_denied"）；或 60s 时间窗口过期重置（resetReason="window_expired"）。防抖计数器在会话结束时自然清空
- _pendingCollectedFields：**CLOSING→IDLE（exitType=normal）**：保留，按 pending_fields_ttl 过期自动清理。下次进入时根据 source 标记决定行为——source="normal_pause"→主动询问是否继续；source="off_task"→不主动询问。**CLOSING→LISTENING（exitType=fallback）**：清空（越界处理意味着当前会话全部废弃）（N6 v2.7 回写）
- _n2Phase：N2 环节完成时重置为 null
- _closingSource：CLOSING 出边后重置为 null。映射关系：intent_exit/exit_word→normal→_pendingCollectedFields 保留；out_of_boundary*→fallback→_pendingCollectedFields 清空（N6 v2.7 回写）
- _sessionStore：新 develop 会话开始时全部置 null
- _lastRecord：不清空（用于"记错了"场景）

### 4.3 产出三：上下文管理器 三接口规范

核心纪律：调度器只调接口，不内联拼接逻辑。就算 上下文管理器 还没实现，也只能走接口占位（直接拼最近 N 轮原文），严禁自己动手拼。

**taskType 传递链路**：

```
N3 dispatchCriteria.taskType
  → 调度器加载时写入 _activeSession.taskType
    → 每轮调 buildPromptContext() 时作为参数传入
      → 上下文管理器 根据 taskType 选择锚点模式
```

| 接口 | 调用时机 | 输入参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| buildPromptContext() | 每轮发消息给模型前（读） | contextGraph, turnHistory, currentAskField, currentTaskType, currentTopicId, currentStepName, contractStore, tunableParams | PromptFragment[] | 按 taskType 选择锚点模式。所有阈值从 tunableParams 读取。上下文管理器 内部按 topicId/stepName 从 contractStore 查询 domainRules/relatedTopics/topicEvolution/sessionCheckpoints 分层包，调度器不预组装这些数据。**v4.7：roomConversationLog 按环节 @importance 分级截断（critical=10/high=6/normal=3/low=1 轮）；规则注入按 ruleNode.importance 排序（critical→high→normal→low）；同 importance 按 strengthens 权重（累加上限+3）** |
| updateGraph() | 每轮模型返回后（写） | userInput, modelTurn, turnType, askingField, changeLevel, changeLevelReason, validationResult, taskType | void | 更新逻辑拼接图，记录本轮对话和校验结果。调度器从模型返回中提取 changeLevel 和 changeLevelReason，写入 turnHistory 当前轮次的记录。field_based 任务两字段可传入 null |
| archiveGraph() | 偏离/放弃/完成时（归档） | — | void | 归档当前上下文图，新建空图 |

### 4.5 产出五：N2 双角色串行（调度器侧编排逻辑）

N2 边界紧张度测试不走普通 IN_SESSION 的单模型实例自治，调度器需要特殊处理。调度器在 ANALYZING 阶段识别到 intent="n2_boundary_test" 时，将控制权交给 N2 宪法定义的编排逻辑。以下为调度器侧的编排实现：

| 维度 | 规范 |
|------|------|
| 触发条件 | ANALYZING intent="n2_boundary_test" |
| 调度器私有字段 | _n2Phase: null \| "corpus_hunting" \| "boundary_testing"（仅 N2 环节使用） |
| 模式 A（纯人收集） | N2 双角色串行不启用。_n2Phase 始终为 null |
| 模式 B/C（模型参与） | 启用完整双角色串行 |

**流转逻辑（模式 B/C）**：

1. ANALYZING → intent="n2_boundary_test" → _n2Phase = "corpus_hunting"
2. 注入场景定义给角色一。场景定义须满足质量要求（见传递契约 sceneDefinitionQualityRequirements：必含项目名称/目标用户群体/核心功能概述，禁含边界清单/意图清单/N1 交付物全文）。明确不注入：边界清单、意图清单、N1 交付物全文
3. 调度器在注入前做安全检查：如果 N1 边界清单或意图清单出现在待注入上下文中 → 拦截，报错
4. 角色一执行，输出原始语料集
5. 调度器校验角色一输出：采用 N2 v5.10 §4.1 的白名单规则——角色一输出顶层仅允许 corpus/corpus_count/source_distribution，corpus 数组内每条仅允许 id/user_input/source/context。任何非白名单字段 → 判定"语料污染"，记录日志，要求重跑角色一
6. 校验通过 → _n2Phase = "boundary_testing"
7. 注入角色一输出 + N1 边界清单（含处理策略）+ N1 意图清单给角色二
8. 调度器在注入前做完整性检查：如果缺少任一样 → 拦截，报错
9. 角色二执行，输出测试报告
10. 调度器读取角色二结论：通过 → 记录 N2 测试报告，状态机进入 N3；回退 → 记录 N2 测试报告，状态机回退到 N1，携带修正建议
11. _n2Phase = null（重置）

**错误处理**：
- role1_context_contaminated：角色一注入上下文中误含边界清单 → 拦截，报错，不启动角色一
- role1_output_polluted：角色一输出含非白名单字段 → 判定污染，要求重跑角色一
- role2_context_incomplete：角色二注入上下文不完整 → 拦截，报错，不启动角色二
- role2_output_format_error：角色二输出格式不符合 output-schema → 重试 1 次，仍失败则报错
- role2_verdict_rollback：角色二报告结论为"回退" → 路由到 N1，携带修正建议，不清除 N1 产出

## 五、关键纪律

1. 第1层必须用快速路由模型，禁止深度推理——成本关键约束，意图识别不需要深度推理，打断循环依赖
2. 调度器只调接口，不内联拼接上下文——minScoreThreshold 等阈值通过 constitution.getTunable() 传入，不硬编码
3. 第3层仲裁权归调度器 DET，不是模型——根宪法第6条
4. taskType 在 N1/N3 阶段确定，N5 不得自行更改
5. tunable 全部用 getTunable() 读取，严禁硬编码
6. askingField 为 null 时硬门控必须降级
7. CLOSING 出边前必须记录来源——_closingSource 标记与 N4 exitType 保持一致
8. **CLOSING 双出边的 _pendingCollectedFields 处理必须区分——IDLE 保留（按 ttl），LISTENING 清空**（N6 v2.7 回写）
9. topic_based 任务必须处理 topicEvolution——changeLevel 格式必须为小写 major/minor/invalid；分层留存；注入包与 stateSnapshotMark 联动
10. topic_based 任务的 上下文管理器 执行三层注入 + embedding 两阶段检索——0 次 LLM 调用。领域规则注入按 topicPath 子图匹配精准注入
11. confidence 检查区分冷启动期和非冷启动期——冷启动期软拦截，非冷启动期硬拦截

## 七、常见踩坑

| 坑 | 表现 | 后果 |
|----|------|------|
| 第1层模型升级为深度推理 | "意图识别用 R1 更准" | 每轮成本爆炸 + 循环依赖 |
| 第2层调度器内联拼接上下文 | "反正 上下文管理器 还没写好，我先拼一下" | 接口隔离失效 |
| 第3层 CLARIFYING 重新调 LLM 生成追问 | 调度器自己调 LLM 想追问语 | 与模型实例追问重复 |
| 第0层词表和 N7 根宪法不同步 | N7 加了新退出词，N5 第0层没更新 | 新退出词不生效 |
| 防抖参数硬编码 | antiFlapWindow 写死 60 秒 | 想调必须改代码 |
| turnHistory 缺少 changeLevel | 记录里没存 changeLevel | topicEvolution 分层留存无法执行 |
| topic_based 任务中 askingField 始终为 null | 没填主题标签 | 上下文管理器 无法做主题级硬门控保护 |
| field_based 和 topic_based 在同一 IN_SESSION 内动态切换 | 试图让同一个 intent 同时支持填表和闲聊 | 锚点模式混乱 |
| topic_based 任务的第3层校验套用 field_based 的值域规则 | 对诗歌创作结果检查"金额是否在合理范围" | 校验永远失败 |
| topic_based 任务的 上下文管理器 用 LLM 做锚点判断 | 每次上下文拼接都调 LLM | 成本高、延迟大 |
| confidence 检查未区分冷启动/非冷启动 | 冷启动期也硬拦截 | 初期大量误拦 |
| topic_based 任务未初始化 topicEvolution | ANALYZING 时未检索 topicIndex | 跨会话续写失效 |
| N2 角色一注入时误传边界清单 | 调度器没做安全检查 | 语料被污染，N2 测试结果无效 |
| CLOSING 双出边未区分 | 所有 CLOSING 都回 IDLE | 越界处理后用户无法继续对话 |
| changeLevel 格式错误未兜底 | 模型返回"MAJOR"但调度器未处理 | topicEvolution 分层留存逻辑异常 |
| expectsChangeLevel 从转移边读取 | N5 从 transitionEdge 而非 routeTableEntry 读取 | 读取不到字段 |
| 防抖计数器跨会话残留 | 退出再进入后计数器未重置 | 前一会话的防抖状态影响新会话 |
| topicEvolution 注入包未与 stateSnapshotMark 联动 | 注入快照始终相同 | 错误快照内容注入 |
| _closingSource 与 N4 exitType 无映射 | 两者独立定义 | 越界处理路由不一致 |
| CLOSING→LISTENING 未清空 _pendingCollectedFields | 越界处理后半成品残留 | 用户再进来时误读旧草稿（N6 v2.7 回写修复） |
