## 一、位置与定位

```
P0（认知加载，八根支柱，不产文件）
    ↓
N1 场景定义 + 边界清单  ← 你在这里
    ↓
N2 边界紧张度测试（验证边界覆盖完整性 + 护栏响应质量）
    ↓
通过后 N3 拆状态
```

**【人话版定位】**
N1 是整个智能体的"地基"——画清楚用户能做什么、不能做什么、不能做的怎么处理。后面的所有设计（状态、转移、宪法、调度器）都从这个边界清单出发。边界没划清楚就设计状态，一定会漏状态；边界未经 N2 验证就进 N3，一定会返工。

**【哲学根基】**
N1 的设计受认知局部性原理直接驱动：推理阶段每个节点的最优输入 = 当前目标的最小充分条件集。因此边界宁可窄不要宽。

态控架构 v4.7 §1.2 明确"写诗与修改家具参数都是受限空间内的收敛过程，区别在于收敛空间的连续性"。N1 阶段需为每个 intent 标注 taskType（field_based / topic_based），此标注决定后续 askingField 语义、锚点模式、DET 复验方式。

taskType 在 N1 初判、N3 固化，运行时不变。当 N1 将某 intent 标注为 topic_based 时，该 intent 将在 N3 自动启用 topicEvolution 三层递进机制；标注为 field_based 时，将在 N3 自动启用 sessionCheckpoint+contractEvolution+roomConversationLog。N1 阶段设计者需确认已理解此影响。

## 二、前置条件

| 前置条件 | 内容 | 来源 |
|---------|------|------|
| P0 认知加载 | 完成八根支柱认知校准记录，总判定为"通过"。八根支柱：认知局部性 / 意图收敛 / 边界控制优于切碎+环节级自治 / 人作为终极容错闭环+工程上限 / 三层宪法+领域规则体系+tunable / taskType 双模式 / topicEvolution 三层递进机制 / sessionCheckpoint+contractEvolution+roomConversationLog | P0 节点说明 v4.1 |
| 同类产品功能边界对比（可选） | 调研 1-2 个同类产品的功能边界，标注差异及理由 | 设计者调研 |
| 初步意图节点清单草稿 | 基于场景定义构思核心意图节点清单（最小必要集），作为软着陆目标 intent 的参照 | 设计者基于产品需求构思 |
| 每个 intent 的 taskType 标注 | 标注 field_based 或 topic_based，N1 初判、N3 固化 | 设计者基于场景特征判断 |

**taskType 判断标准**（区分原点：收敛空间的性质，非任务内容）：

| taskType | 收敛空间性质 | 判断依据 |
|----------|------------|---------|
| field_based | 离散参数空间，目标定位唯一正确值 | 存在客观正确答案 + 输出可被 DET 值域硬校验验证 + "完成"=所有必采字段齐全。典型场景：记账、改柜体参数、表单填写、参数查询、删除 |
| topic_based | 连续语义空间，目标采样满足约束的样本 | 不存在唯一正确答案 + 用"是否在主题边界内+是否满足格式约束"判断 + "完成"=输出满足主题边界和格式约束。典型场景：内容生成、角色扮演、开放研究、头脑风暴、诗歌创作、专利撰写、方案设计 |
| 模糊场景 | 无法判断收敛空间性质 | 优先追问澄清（根宪法第 8 条），澄清后按标准分派 |

**混合型任务处理**：区分标准不是"任务里有没有字段"，而是"收敛空间的主体是连续的还是离散的"。主体是连续语义空间（如创作）但带有离散约束（如格式）→ topic_based 主 + field_based 子。反过来，主体是填字段但含自由文本→ field_based 主 + topic_based 子。主 taskType 决定跨节点载体和长期记忆机制，子约束附加在 DET 复验中。

**关于松弛节点**：如果智能体需要支持"闲聊→任务"的自然过渡，边界清单中应显式声明 slack-node 的存在及其边界。slack-node 是合法存在的低约束节点，仅受根宪法约束，不���分 taskType。但 slack-node 的边界本身也需要经过边界覆盖完整性测试——如果 slack-node 的闲聊输入无处理路径或护栏响应不合理，说明 slack-node 的边界划定有问题，需要回溯第一步重审边界。

**意图粒度基准（隐性约束）**：一个 intent = 一个完整闭环的用户任务，包含"用户触发 → 系统响应 → 任务完成"的完整生命周期。不要把单个操作拆成多个 intent（如"新增记录"和"确认新增"应为同一个 intent），也不要把多个不相关的任务合并为一个 intent（如"记账+查余额"应拆分）。**topic_based 场景特别提示**：意图粒度还直接影响后续领域规则的 topicPath 层级精度——intent 合并过粗会导致 topicPath 变浅，规则子图匹配精度下降，不同子领域（如"厨房设计"vs"卫生间设计"）的专属规则无法被正确区分注入。

## 四、产出物

N1 产出五样东西：一句话场景定义、边界清单（含处理策略）、护栏策略详解、语料收集要求、N1→下游传递契约（JSON Schema）。

### 4.1 产出一：一句话场景定义（锚点）

用一句话说清"这个系统是什么"。后续所有设计决策的回看锚点。

**合格标准**：陌生开发员看了能判断某功能该不该进。

**示例**：

- field_based："这是一个语音记账工具，用户说出支出，系统记录并支持查询和删除。"

- topic_based："这是一个角色扮演助手，用户设定角色和场景，系统在主题边界内进行多轮角色扮演对话。"

- 混合型："这是一个全屋定制方案设计助手，用户描述需求，系统通过多轮对话完成方案设计，支持多方案对比、风格替换和方案迭代。"

### 4.2 产出二：边界清单（含处理策略）

至少四列：维度、做什么、不做什么（含处理策略）。每条"不做什么"必须带处理策略，禁止裸写"不闲聊"。

| 维度 | 做什么 | 不做什么（含处理策略） |
|------|--------|----------------------|
| 核心功能 | 记支出 record（field_based）、查历史 query（field_based）、删记录 delete（field_based） | 不自动分类（硬拒）；不推荐理财（软着陆→intent: query）；不记收入（硬拒） |
| 交互方式 | 语音唤醒 + 文字输入 | 不支持图片识别（硬拒）；不支持手势操作（硬拒） |
| 对话范围 | 记账相关指令 | 不闲聊（硬拒，可回锚）；不回答非记账问题（硬拒） |
| 数据处理 | 本地存储用户账目 | 不上传云端分析（硬拒）；不分享第三方（硬拒） |

**松弛节点声明**（可选）：若场景需支持"闲聊→任务"的自然过渡，在此声明 slack-node 的边界。示例：`slack-node（topic_based） 闲聊对话 | 受根宪法约束，不区分 taskType | 不涉及敏感话题（硬拒）；不执行任何业务操作（软着陆→ANALYZING）`

每个 intent 后括号标注 taskType，供 N3/N5/N9 继承。

**topicEvolution 启用声明（重要）**：边界清单中标注为 topic_based 的 intent，将在 N3 自动启用 topicEvolution 三层递进机制。边界清单中标注为 field_based 的 intent，将在 N3 自动启用 sessionCheckpoint（进度快照）+ contractEvolution（规则演化）+ roomConversationLog（全量记录）。

### 4.3 产出三：护栏策略详解

#### 4.3.1 三种处理策略

| 策略 | 格式 | 含义 | 示例 |
|------|------|------|------|
| 硬拒 | 硬拒 | 直接拒绝，不加引导 | "小安只会记账哦" |
| 引导性硬拒 | 硬拒（可回锚） | 拒绝后附带回归当前任务的提示 | "小安只会记账哦。（当前在记今天的账）" |
| 软着陆 | 软着陆→intent: <意图名> | 引导到系统已有能力，必须指向已规划的 intent | "我可以帮你记，你要先记完这条还是先查？"（→intent: query） |

#### 4.3.2 引导性硬拒

| 维度 | 规范 |
|------|------|
| 触发条件 | 用户输入触发硬拒规则，且该条标注为"可回锚" |
| 执行动作 | 输出拒绝语后，尝试追加回归当前任务主线的提示文本。锚点文本由调度器从 activeSession 中提取当前 askingField 和已采集字段拼接而成（如"当前在记录 [category] 的金额"） |
| 兜底逻辑 | 当 activeSession 为 null、askingField 为 null、或 collectedFields 为空时，锚点文本不可用——降级为无锚点的普通硬拒，或使用通用回锚文本（如"当前无进行中的任务"），不报错、不阻断 |
| 约束 | 引导语简洁；不改变状态机流转路径 |

#### 4.3.3 护栏策略选择决策树

```
用户偏离输入，系统有能力且应该去处理吗？
  ├── 是 → 软着陆（必须绑已规划的 intent）
  └── 否 → 当前存在进行中的活跃任务？
            ├── 是 → 引导性硬拒（调度器拼接锚点文本；锚点不可用时降级为普通硬拒）
            └── 否 → 硬拒
```

#### 4.3.4 软着陆审计要求（关键纪律）

每条软着陆必须指向已规划的 intent，格式为 `软着陆→intent: <意图名>`。审查者据此核对 N3 草稿中是否存在对应路由。

N3 完成时，调度器自动扫描 softLandingTargets，逐条对比 N3 的最新 intent 枚举。若目标 intent 在 N3 中被合并/拆分/重命名，调度器提示设计者手动更新 softLandingTargets 映射，并在 N3 交付物中附带"软着陆目标重映射报告"。未完成此步骤前，N3 质量门禁不予通过。

### 4.4 产出四：边界紧张度测试的语料收集要求

N1 阶段需为 N2 准备语料收集计划。N2 验证边界覆盖完整性 + 护栏响应质量，通过后方可进 N3。

**语料要求**：收集 20-30 条真实用户的口语指令，必须来自非项目组成员或真实用户日志，设计者自写无效（会被边界先验污染）。

**语料获取路径（按可信度降序）**：

| 优先级 | 路径 | 说明 |
|--------|------|------|
| 最佳 | 已有同类产品的真实用户日志（脱敏后） | 有则必用 |
| 次佳 | 找 5-8 个非项目组成员做 10 分钟口语模拟 | 最低要求，不可跳过 |
| 仅作初稿 | 设计者自写 | 交付前必须用路径 2 替换至少 50% |
| 应急 | 模型生成（推理模型基于场景定义生成） | 必须走 N2 双角色串行+信息隔离 |

**硬纪律**：没有经过路径 1 或路径 2 语料替换的边界紧张度测试，不算完成。

**execModeDecision 决策树**（供设计者填写 corpusCollectionPlan.execModeDecision 时参照）：

- ① 能接触 ≥5 非项目组成员 → 模式 A（纯人收集，最可信）
- ② 不能接触外部但有同类产品日志 → 模式 A（用日志替代）
- ③ 不能接触外部也无日志 → 模式 B（模型生成，走信息隔离）
- ④ 人收集基础语料但不够 20 条 → 模式 C（混合模式）

### 4.5 产出五：N1→下游传递契约（分层 JSON Schema）

## 五、下游关联

### 5.1 N7 根宪法输入

| 边界清单条目 | 对应根宪法条款 | JSON Schema 字段 |
|-------------|--------------|-----------------|
| 安全红线类"不做什么" | 第 1 条（安全红线） | rootConstitutionInputs.safetyRedlines |
| 唤醒词 / 退出词 / 取消词 | 第 2 条（元指令集） | rootConstitutionInputs.metaInstructions |
| 模糊目标处理策略 | 第 8 条（模糊目标追问模板） | rootConstitutionInputs.fuzzyGoalHandling |
| taskType 标注（doItem → askingField 语义） | 第 4 条（输出格式——field_based=字段名 / topic_based=主题标签） | doItem.taskType |

降级链五项均不可缩减——L1 结构校验 + DET 值域复验 + confidence 检查 + 跨任务延伸检测 + 硬编码兜底。第 5 条降级链可扩展不可缩减。冷启动期 confidence 检查采用软拦截。

### 5.2 N8 公共规则输入（场景级定性标记）

母本定义 3 条公共规则。N1 阶段只标注场景级定性标记，各环节的精确参数由 N9 环节宪法独立声明。

| 公共规则 | 内容 | 责任归属 | N1 需标注内容 |
|---------|------|---------|-------------|
| 规则 1 | 输出必须是纯 JSON | DET 复验（L1 结构校验） | 无差异，N1 无需额外标注 |
| 规则 2 | 坚持真实判断 | 环节宪法 prompt 约束 | 无差异，N1 无需额外标注 |
| 规则 3 | 一次聚焦一件事，禁止跨任务延伸 | 环节宪法 prompt + DET 复验（跨任务延伸检测） | rule3Enabled（是否启用，默认 true）；rule3NeedSemanticExtensionCheck（topic_based 场景是否需要 LLM 语义层检查，默认 false） |

- field_based 任务中，如果几个字段是强关联的（如"金额+币种"），可以一轮问两个，但要在 askingField 里标清楚主次——此为 N9 环节宪法中声明，N1 不要求精确值
- topic_based 任务中，模型自己把握节奏，但不要一次抛太多开放性主题
- 不得在完成当前任务后主动引入超出当前环节能力范围的延伸话题或推荐
- 任务内的智能补全（包括在同主题范围内的合理发散）是允许的

### 5.3 taskType 传递链路

```
N1 doList.taskType → N3 taskTypeFinalized → N5 activeSession.taskType → N9 @section taskType
```

决定：askingField 语义（field_based=字段名 / topic_based=主题标签）、锚点模式（field_based=字段级硬门控 DET / topic_based=三层注入 + embedding 两阶段检索）、DET 复验方式（field_based=值域校验 / topic_based=格式校验）、N9 宪法写法。

topic_based intent 在 N3 自动启用 topicEvolution 三层递进机制；field_based intent 在 N3 自动启用 sessionCheckpoint+contractEvolution+roomConversationLog。N1 阶段通过 doItem 的 taskType 字段确认设计者已知晓此影响。

### 5.4 下游一致性校验清单

| 校验项 | 时机 | 不通过处理 |
|--------|------|-----------|
| intent 与 softLandingTargets | N3 完成时 | 回溯 N1 |
| 软着陆目标重映射 | N3 完成时 | 回溯 N1，附带重映射报告 |

**CI/CD 自动校验流水线**：每次提交 N3/N5/N9 的产出物时，自动比对 doList 中的 taskType 是否与 N1 传递契约一致。不一致则阻断合并，并提示"请回溯 N1 或对应节点更新传递契约"。

## 六、N1 与 N2 的衔接

N1 产出五样东西后，进入 N2 边界紧张度测试。

N2 验证边界覆盖完整性 + 护栏响应质量：
- 所有类一输入（功能范围内）都有对应 intent + 处理路径
- 所有类二输入（功能范围外）都有对应护栏策略 + 处理路径
- 护栏响应质量合格（优秀/合格）

**判定**：
- 边界覆盖完整 + 护栏响应合理 → 进入 N3
- 存在无处理路径的输入 OR 护栏响应不合理 → 回退 N1，补全边界清单或优化护栏策略

**N1→N2 数据传递契约**：
- 传递至角色一（n2-corpus-hunter）：仅一句话场景定义。明确不传边界清单、意图清单、N1 交付物全文。调度器在注入前做安全检查——若边界清单或意图清单出现在待注入上下文中 → 拦截，报错
- 传递至角色二（n2-boundary-tester）：a) 角色一的原始语料集 b) N1 边界清单（含处理策略）c) N1 意图清单。调度器在注入前做完整性检查——缺少任一样 → 拦截，报错
- 角色一输出 schema：{ corpus: [{id, user_input, source, context}], corpus_count, source_distribution }
- 角色二输出 schema：{ test_report: { corpus_total, classification: {in_scope, out_of_scope}, coverage_check, guardrail_quality, conclusion, fail_reasons, optimization_suggestions } }

N2 的详细执行模式（A/B/C）、双角色串行设计、schema 校验规则，详见 N2 节点说明。

## 七、关键纪律

1. "不做什么"比"做什么"更重要——LLM 天生倾向"能答就答"，不写"不答什么"它就真去答。每条"不做什么"必须带处理策略
2. 护栏升级 ≠ 边界扩展——软着陆必须指向已规划的 intent。指向不存在的 intent = 以护栏之名扩边界，必须走 N16 正式扩展流程
3. 边界宁可窄（认知局部性原理的直接推论）
4. taskType 在 N1 初判、N3 固化，N5 不得自行更改
5. taskTypeAlignmentStatus 必须声明（aligned / pending_backport），N1/N3 升级后重新校验 N5/N9 一致性
6. N1 只管边界层能确定的参数——第一层模型在 N1 定，第二层绑定在 N4 定，契约继承规则在 N4 回填
7. topic_based intent 在 N1 标注时即确认已知晓 topicEvolution 三层递进机制启用
8. 意图粒度基准：一个 intent = 一个完整闭环的用户任务，不拆散不合并
