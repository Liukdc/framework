## 一、五层接待规则

```
用户输入
│
├─ 第0层：口令精确匹配（M1元指令集，DET EXACT_MATCH，0次LLM）
│   wake口令 → IDLE→LISTENING
│   exit口令 → 任意→CLOSING→IDLE
│   cancel口令 → 任意→LISTENING（中断当前推理不归档）
│   switch口令"切断房间" → 任意→ANALYZING
│     └─ S3物理隔离四步：释放KV Cache→归档→加载新房间状态→创建新会话
│     └─ roomStateIndex注入：buildRoomIndex()生成全窗口房间摘要
│   非口令 → 放行第0.5层
│
├─ 第0.5层：DET关键词扫描（每轮执行，0次LLM）
│   扫描用户输入，检测跨意图词汇：
│   - 当前在record，输入含"查/查询/查一下" → offTaskSuspicion="query"
│   - 当前在record，输入含"删/删除/删掉" → offTaskSuspicion="delete"
│   - 当前在record，输入含"比/对比/比较" → offTaskSuspicion="compare"
│   标记附加到模型输入中作为提示，不强制覆盖模型判断
│
├─ 第1层：意图识别（ANALYZING，1次LLM）
│   拼接【意图识别宪法】→ 调DeepSeek
│   强制选择(A/B/C/D/E) + API logprobs + 第二分类维度(S/T/U)
│   → {choice, logprobs, intent, inputNature, extracted}
│   DET读取logprobs做阈值裁决：
│     probability >= 0.7 → 放行，按intent路由
│     probability < 0.7 → intent=other
│   inputNature路由：
│     intent=已知 + inputNature=T → IN_SESSION(intent)
│     intent=other + inputNature=S → SLACK_NODE
│     intent=other + inputNature=T → TEMP_TOPIC
│     intent=other + inputNature=U → 追问
│   sessionCheckpoint恢复检查（field_based）：
│     查sessionCheckpoints表 → 有未过期进度 → 询问"继续or去其他环节"
│     用户确认继续 → 注入stepSnapshots → 路由到目标环节
│   路由表第二层（意图级绑定）：全null（field_based用默认通用模型）
│
├─ 第2层：环节内模型自治（IN_SESSION，LLM）
│   调度器调用contextManager.buildPromptContext()拼接上下文
│   field_based模式注入：stepSnapshots + roomConversationLog(roomId) + contractEvolution已生效规则
│   工具分层注入（v5.5）：
│     必用工具完整定义：按环节宪法@section tools（如compare→query_total+query_unit_price）
│     工具清单轻量级目录：按环节宪法@section tool_catalog（name+description）
│     meta-tool：search_tools（始终可用）
│   注入模型实例 → 模型在局部宪法约束下自治
│   模型可调用DET工具（v5.4 function calling）：
│     record → 不需要工具（字段采集→EXECUTING做INSERT）
│     query → 调用query_records工具→DET执行SQL→结果返回模型→模型格式化
│     delete → 不需要工具（定位条件采集→WAITING_CONFIRM→EXECUTING做DELETE）
│     compare → 调用query_total/query_unit_price工具→DET执行SQL→结果返回模型→模型格式化
│   模型返回turnType + askingField + changeLevel(全field_based,changeLevel=null) + changeLevelReason
│   调度器按turnType调度：
│     ask/reply → 转发用户，继续循环
│     validation_failed → CLARIFYING（复用message，0次LLM）
│     off-task → off-task双重保障（见§二）
│     complete → 第3层DET复验
│     giveup → LISTENING
│   每轮调用contextManager.updateGraph()更新逻辑拼接图
│
└─ 第3层：DET四项检查（complete时，0次LLM，全确定性）
    检查1：L1结构校验 → JSON.parse + Schema校验 → 失败重试1次→熔断→硬编码回复
    检查2：DET值域复验 → field_based值域校验（金额/时间/必填字段）→ 失败→CLARIFYING
    检查3：logprobs检查 → probability<threshold + pattern扫"硬撑式确认"词表 → 失败→CLARIFYING
      冷启动期（cold_start_observation_window内）：软拦截，标记warning不直接拦截
      非冷启动期：硬拦截
    检查4：硬编码兜底 → 熔断时硬编码回复→LISTENING
    复验通过：
      record → EXECUTING(INSERT) → LISTENING
      delete → WAITING_CONFIRM(人确认) → EXECUTING(DELETE) → LISTENING
      query → 直接LISTENING（工具调用已完成，不经EXECUTING）
      compare → 直接LISTENING（工具调用已完成，不经EXECUTING）
```

## 二、off-task 双重保障

```
保障1（主力——宪法强制 + 上下文管理器优先注入）：
  根宪法第1条（环节隔离）→ 模型每轮首要判断"是否本环节任务"
  上下文管理器将环节隔离声明（@section role首位）以最高优先级注入prompt
  → 不属于本环节 → 立即返回off-task（优先级最高）
  → 大幅降低漏判率

保障2（辅助——ANALYZING重新识别验证）：
  模型返回off-task后，调度器不直接路由
  → DET只做非空检查（门卫，不做语义验证——DET做不了语义判断）
  → 送回ANALYZING重新识别offTaskInput
  → ANALYZING的重新识别结果就是验证：
    → 识别出不同intent → 确认切换，进入新环节
    → 识别出same intent或other → 尊重模型判断，引导重新表达

显式切换口令（M1 switch，首选）：
  用户说"切断房间" → DET硬匹配（EXACT_MATCH，0次LLM）
  → S3物理隔离四步 + roomStateIndex注入
  → 100%确定性，off-task检测是兜底

设计原则：
  不追求100%拦截，而是"降低漏判+拦住误判+回溯兜底"
  DET不做语义验证（做不了），语义验证由ANALYZING重新识别完成
  DET不僭越模型自治权
```

## 三、上下文管理器三接口

| 接口 | 时机 | 职责 | 状态机传入 |
|------|------|------|----------|
| buildPromptContext() | 每轮发消息给模型前（读） | 按硬门控+语义匹配+importance截断算出本轮应拼入的上下文片段。field_based模式执行字段级硬门控（DET确定性匹配，保护与askingField同名的轮次不被淘汰）。工具分层注入（必用工具完整定义+清单轻量目录+search_tools meta-tool） | contextGraph, turnHistory, currentAskField, currentTaskType=field_based, currentStepName, contractStore, tunableParams |
| updateGraph() | 每轮模型返回后（写） | 更新逻辑拼接图，记录本轮对话和校验结果 | userInput, modelTurn, turnType, askingField, changeLevel=null, validationResult |
| archiveGraph() | 偏离/放弃/完成时（归档） | 归档当前上下文图，新建空图 | — |

**接口隔离原则**：调度器只调三接口，不内联拼接逻辑。阈值从getTunable()读。

**askingField null降级规则**：askingField为null时（reply/complete轮次），硬门控保护降级为纯语义匹配，不报错不阻断。

## 四、工具分层注入（v5.5）

调度器进入IN_SESSION时，按环节宪法声明从toolRegistry筛选工具：

| 环节 | 必用工具（完整定义自动注入） | 选用工具清单（轻量级目录） |
|------|--------------------------|------------------------|
| record | insert_record | format_currency |
| query | query_records | get_current_date, format_currency |
| delete | delete_record | query_records（定位记录） |
| compare | query_total, query_unit_price | get_current_date, calculate_percentage, format_currency |
| other | 无 | 无 |

**meta-tool**：search_tools（始终可用，用于清单外的工具发现）

**Two-Phase Selection流程**：
1. 调度器注入必用工具完整定义 + 选用工具清单（name+description）+ search_tools
2. 模型从清单选择需要的工具 → 返回needed_tools
3. 调度器从toolRegistry加载完整定义 → 注入模型
4. 模型调用工具 → 调度器执行SQL → 结果返回模型
5. 模型生成自然语言回复 → turnType=complete

**工具内部时间解析**：所有涉及时间的工具参数支持自然语言时间（"this_month"→日期范围），DET内部解析。

## 六、sessionCheckpoint 处理

**写入时机**：
- record complete → sessionCheckpoints更新（completedSteps追加record, stepSnapshots.record写入collectedFields）
- query/delete/compare complete → 同理更新

**读取路径**：
- ANALYZING恢复：用户唤醒→DET查sessionCheckpoints→有未过期进度→询问"继续or去其他环节"
- 上下文拼接：进入环节时→读stepSnapshots[环节名]的collectedFields

**全环节恢复**：record/query/delete/compare 都有恢复路径

## 八、v5.3 → v5.6 变化说明

| 变化点 | v5.3 | v5.6 |
|--------|------|------|
| 工具分层 | 无 | **v5.5：必用+清单+选用 + search_tools meta-tool** |
| IN_SESSION工具调用 | 无 | **v5.4：function calling，query/compare在IN_SESSION内完成** |
| off-task保障2 | DET事后验证 | **ANALYZING重新识别验证（DET做不了语义验证）** |
| 降级链 | 五项检查 | **四项检查（移除跨任务延伸检测）** |
| logprobs | confidence检查 | **logprobs检查（API返回真实概率，非模型编的）** |
| 第二分类维度 | 无 | **S/T/U（S→SLACK_NODE, T→TEMP_TOPIC, U→追问）** |
| S3物理隔离 | 无 | **switch口令触发S3四步 + roomStateIndex注入** |
| sessionCheckpoint恢复 | 仅record | **全4个业务环节** |
| 上下文管理器 | 简单描述 | **三接口规范 + 工具分层注入** |
