## 二、核心架构决策落地

### 决策1：状态转移严格按 N4 转移图

```
scheduler.js 实现：
- 所有转移边硬编码为路由矩阵（switch/case），不调LLM判断下一状态
- N4 34条转移边全部映射为确定性路由规则
- M1四类口令（wake/exit/cancel/switch）在第0层DET EXACT_MATCH处理
```

### 决策2：五层过滤实现

| 层 | 实现 | 0次LLM |
|----|------|:---:|
| 第0层 | DET EXACT_MATCH口令词表（wake/exit/cancel/switch），switch触发S3四步 | ✅ |
| 第0.5层 | DET关键词扫描（record→查/查询/删/删除/比/对比），标记offTaskSuspicion | ✅ |
| 第1层 | LLM ANALYZING（强制选择A/B/C/D/E+logprobs+S/T/U），DET阈值裁决 | ❌ 1次 |
| 第2层 | LLM IN_SESSION自治（可调用DET工具），模型返回turnType | ❌ 环节内 |
| 第3层 | DET四项检查（L1结构+值域+logprobs+硬编码） | ✅ |

### 决策3：v5.4 IN_SESSION tool calling

```
scheduler.js 实现：
- IN_SESSION(record)→EXECUTING(INSERT)
- IN_SESSION(query)→LISTENING（工具调用完成，不经EXECUTING）
- IN_SESSION(delete)→WAITING_CONFIRM→EXECUTING(DELETE)
- IN_SESSION(compare)→LISTENING（工具调用完成，不经EXECUTING）

tool-registry.js 实现：
- insert_record / query_records / delete_record / query_total / query_unit_price
- 必用工具完整定义 + 选用工具清单 + search_tools meta-tool
- 工具内部时间解析（"this_month"→DET解析为日期范围）
```

### 决策5：off-task 双重保障

```
保障1：上下文管理器优先注入环节隔离声明（@section role首位）
保障2：模型返回off-task→DET非空检查→ANALYZING重新识别
首选：switch口令（DET EXACT_MATCH + S3物理隔离 + roomStateIndex）
```

### 决策6：v5.7 对话分段保留

```
contract-store.js 实现：
- roomConversationLog新增字段：segmentType/relatedOutputId/summaryImportance/originalTurnRange
- 产出物确认后：模型后台生成摘要→原始对话移到conversationArchive（v5.8）→摘要写入roomConversationLog
- 摘要分级保留：critical10/high5/normal2（新摘要生成时FIFO检查）
- buildPromptContext()拼接：优先全量对话→critical摘要→high摘要→normal摘要
```

### 决策7：v5.8 conversationArchive

```
contract-store.js 实现：
- conversationArchive表：不参与拼接，全文搜索（FTS5）
- 产出物确认后原始对话移入（不直接删除）
```

## 六、编程规则宪法校验
