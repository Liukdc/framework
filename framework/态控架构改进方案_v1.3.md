## 一、背景

2026-07-15/16 通过富贵小安的完整实践（N11 契约对齐 → B 任务 stateControl → N13 骨架4硬要求 → N14 验收测试 → UI 脚手架 + LLM 模式），暴露了 4 个态控架构层面的问题。这些问题不是富贵小安特有的，而是架构本身的缺口——不解决，后续基于态控架构的智能体都会踩同样的坑。

**核心矛盾**：架构定义了职责分配（DET/LLM/HUMAN），但流程缺少把职责分配强制传递给代码模型 + 验证代码模型遵守的机制。

## 二、改进1：宪法体系重构

### 问题
v4.7 的宪法体系混了"约束模型的规则"和"架构机制"——根宪法8条里，有的是给模型读的约束（输出格式、收敛义务），有的是给调度器/DET 的机制（降级链、仲裁权、节点转换守卫、元指令集）。两者性质不同，混在一起导致代码模型分不清哪些是给自己的。

### 改前（v4.7 现状）
```
根宪法（8条，约束所有LLM节点）：
  1.安全红线 2.元指令集 3.收敛义务 4.输出格式
  5.降级链 6.仲裁权 7.节点转换守卫 8.模糊目标追问模板

公共规则（3条，所有LLM调用通用）：
  1.输出纯JSON 2.坚持真实判断 3.一次聚焦禁止跨任务延伸

环节宪法（18标签 @section）
领域规则（L0-L3）
```

### 改后（新结构）

**核心原则**：ANALYZING 也是环节（意图识别环节），与 IN_SESSION 统一为"环节"。按约束对象分层，公共规则取消。

```
根宪法（4条，约束所有环节 = ANALYZING + IN_SESSION）：
  1. 环节隔离：只能做本环节任务，非本环节任务返回off-task。
     本环节任务由环节宪法第一条规定。
     （ANALYZING只能做意图识别，IN_SESSION只能做本环节任务）
  2. 输出格式：符合turnType schema
  3. 收敛义务+坚持真实判断（合并）：收敛+不迎合用户
  4. 不编造功能：不无中生有
     （安全红线的真核心——不是场景特定的"禁止恶意内容"，
      是全局硬约束"不许编造"。所有LLM节点都不能无中生有）

公共规则 → 取消
  （追问模板下放到环节宪法 @section ask-rules）

架构机制（从根宪法移出，约束调度器/DET）：
  - 元指令集：口令定义（唤醒/退出/取消），DET硬匹配
  - 降级链五项检查（调度器/DET行为）
  - 仲裁权DET优先（调度器行为）
  - 节点转换守卫（调度器行为）

环节宪法（约束特定环节）：
  第一条：本环节任务定义 + 强调"非本环节任务返回off-task"
  @section ask-rules：追问模板（从公共规则下放）
  其余：field-rules / validation / completion / output-schema / ...

领域规则（L0-L3，不变）
```

### 关键变化
| v4.7 | 改后 | 变化 |
|------|------|------|
| 根宪法8条 | 根宪法4条 | 安全红线重定义为"不编造功能"（全局硬约束，非场景特定）；收敛+真实判断合并；元指令集/降级链/仲裁权/节点转换守卫移到架构机制；模糊目标追问模板下放环节宪法 |
| 公共规则3条 | **取消** | 纯JSON→根宪法第2条；真实判断→根宪法第3条；一次聚焦→根宪法第1条（环节隔离覆盖）；追问模板→环节宪法 @section ask-rules |
| 环节宪法 | 第一条强化 | 明确"非本环节任务返回off-task"为第一条首要内容 |
| ANALYZING | 重新定义 | ANALYZING是"意图识别环节"（非"非环节"），与IN_SESSION统一受环节隔离约束 |

### 三层结构
根宪法（4条）→ 环节宪法（特定环节）→ 领域规则（L0-L3）+ 架构机制（调度器/DET）并行。比 v4.7 四层（根/公共/环节/领域）少一层。

### 影响范围
- **v4.7 全量版**：§7 约束体系重写（根宪法4条 + 取消公共规则 + 加架构机制章节）
- **v5.4 节点说明**：N7（根宪法改4条）、N8（取消公共规则）、N9（环节宪法第一条强化 + 追问模板下放）
- **v7.3 设计流程指南**：第三步写宪法（N7-N10）流程更新

## 三、改进2：编程规则宪法（全局元约束）

### 问题
N13 代码模型读 L3 包 + N5 传递契约生成代码，但缺一份给代码模型读的编程约束宪法。代码模型走捷径用正则替代 LLM，偏离了 v4.7/N3/N5 的执行体分派。

### 改前（v4.7 现状）
N12 产出 L3 包（含环节宪法，给运行时模型读的）。N13 代码模型读 L3 包 + N5 传递契约生成代码。没有给代码模型的编程约束。

### 改后
编程规则宪法是**母本的一部分**，在整个 N1→N15 管线全局生效，不是某个节点产出的。任何涉及代码生成的环节（N13）都必须遵守。

**位置**：
- v4.7 全量版：独立章节（如"附录 E：编程规则宪法"），与根宪法/架构机制并列
- v5.4 节点说明：P0 认知加载时读到（设计者理解后才能进 N1）
- v7.3 设计流程指南：管线全局约束声明

**编程规则宪法内容（6条）**：
1. 执行体分派铁律：每个状态的执行体由 v4.7/N3 决定，代码模型不得自行更改。ANALYZING=LLM，IN_SESSION=LLM，DET复验=DET，EXECUTING=DET
2. DET 职责边界：只做元指令EXACT_MATCH + 值域复验 + 关键词扫描标记。不扩展到意图识别/字段采集/查询理解
3. LLM 职责边界：做意图识别 + 字段采集/追问/偏离判断/查询理解。不得用正则/parser 替代
4. 降级规则：**无 apiKey 时 ANALYZING/IN_SESSION 必须拒绝服务并明确告知原因，不得以任何形式降级为正则/DET 替代**（"拒绝服务"是用户语言，不是技术语言"报错"）
5. 契约对齐：代码必须严格对齐 N11 契约（contract-in/out schema），不自创接口
6. tunable/接口隔离：阈值从 getTunable() 读，调度器只调 context-manager 三接口

### 影响范围
- **v4.7 全量版**：加附录 E（编程规则宪法）
- **v5.4 节点说明**：P0 认知加载增加编程规则宪法为必读母本
- **v7.3 设计流程指南**：管线全局约束声明（非节点产出）

## 四、改进3：N14 审骨架补执行体分派验证

### 问题
N14 验收测试 verdict=pass 但没拦住"正则替代 LLM"——因为审骨架查"行为"（不崩溃、有处理路径），没查"执行体分派是否正确"。

### 改前（v4.7 现状）
N14 审骨架7项：1.状态转移图 2.DET双层位置 3.askingField 4.tunable 5.接口隔离 6.taskType双模式锚点 7.topicEvolution

### 改后
N14 审骨架增加**执行体分派验证**（两层）：

**层1：静态分析（不需要运行）**
- 读代码检查 identifyIntent 函数体内是否有 LLM API 调用（fetch/axios/OpenAI SDK），而非正则/parser
- 读代码检查 createSession 函数体内是否有 LLM API 调用，而非 _local*Session 正则
- 静态分析能直接拦住"代码层面就没调 LLM"的情况

**层2：运行时验证**
- 至少跑 1 次真 LLM 调用，验证 ANALYZING 和 IN_SESSION 确实走了 LLM 路径
- 从 telemetry trace 提取 executor_type，确认有 LLM span
- 运行时验证能确认"确实走了 LLM 路径"（不只是代码里有调用，而是真的调了）

两层结合比纯 telemetry 更稳健——静态分析拦住"代码层面没调 LLM"，运行时验证确认"确实走了 LLM"。

### 影响范围
- **v5.4 节点说明**：N14 审骨架7项→8项（加执行体分派验证，含静态分析+运行时验证）
- **v7.3 设计流程指南**：N14 验证项更新

## 五、改进4：off-task 双重保障

### 问题
架构核心目标是环节间自由切换，但 IN_SESSION 内用户要切换环节时，当前机制单靠模型概率判断 off-task，不能保证准确（漏判/误判）。偏离检测缺少确定性兜底。

### 改前（v4.7 现状）
```
IN_SESSION 内用户输入
  → 第0.5层 DET 关键词扫描（只标记 offTaskSuspicion，不判断）
  → 模型判断（概率性，可能漏判/误判）
  → 返回 turnType=off-task + offTaskInput
  → 调度器直接送回 ANALYZING 重新意图识别
```

### 改后（双重保障）

**取消 v1.0 的保障2（DET 主动触发 ANALYZING）**——与环节内自治根本性冲突，不是"权衡"能解决。DET 主动介入环节内 = 对模型自治权的僭越。

**DET 只在两个位置介入**：(1) IN_SESSION 入口前（第0.5层标记），(2) IN_SESSION 出口后（DET 值域复验 + off-task 验证）。

```
保障1（宪法强制）：
  根宪法第1条（环节隔离）→ 模型每轮首要判断"是否本环节任务"
  → 不属于本环节 → 立即返回 off-task（优先级最高）
  → 大幅降低漏判率

保障2（DET 事后验证）：
  模型返回 off-task 后，调度器不直接路由
  → DET 验证 offTaskInput 是否真的是切换意图
  → 验证通过 → 路由到 ANALYZING
  → 验证不通过（模型误判）→ 拒绝切换，模型继续当前环节
  → 拦住误判

漏判兜底：
  保障1降低漏判率，剩余漏判由 N16 回溯触发器处理
  （用户使用中反馈"切换失败"→ 回溯）
```

**设计原则**：不追求 100% 拦截，而是"降低漏判 + 拦住误判 + 回溯兜底"。用确定性兜底追求 100% 反而会破坏自治。

### 影响范围
- **v4.7 全量版**：§3.1 意图收敛机制（加 off-task 双重保障）+ §6 容错与纠错
- **v5.4 节点说明**：N5（出口后加 off-task 验证步骤）+ N6（off-task 验证数据协议）
- **v7.3 设计流程指南**：N5 调度器逻辑 + off-task 机制说明

## 改进5：ANALYZING意图识别盲区修复——intent=other细分 + TEMP_TOPIC运行时房间

### 问题

当前 ANALYZING（意图识别）只有一个分类维度：将用户输入强制匹配到已知 intent（A/B/C/D/E），当 logprobs 低于阈值时统一返回 `intent=other`。

`intent=other` 掩盖了两种截然不同的输入类型：

| 用户输入 | 实际情况 | 当前路由 | 正确路由 |
|---------|---------|---------|---------|
| "哈哈今天天气真好" | 闲聊，无目标，无方向 | intent=other → 无明确处理 | **SLACK_NODE** |
| "帮我分析最近的新能源趋势" | 有目标，但无匹配房间 | intent=other → 无明确处理 | **TEMP_TOPIC**（运行时创建房间） |

两种 `other` 在系统中混为一谈——闲聊和有目标输入被扔进同一个垃圾桶。而且即使有 roomStateIndex 提供了全局视野，ANALYZING 也只是"看到了"全景，缺少区分"无方向闲聊"和"有方向但无房间"的能力。

### 改后设计

#### 5.1 ANALYZING 增加第二分类维度

当前 ANALYZING 的 prompt 只问一个问题：

> "输入匹配以下哪个已知 intent？A/B/C/D/E"

修改后 prompt 并行问两个问题：

```
【任务1：意图匹配】选择最匹配的已知意图 A/B/C/D/E
【任务2：输入性质】判断此输入属于：
  S) 闲聊/松弛——无明确目标，随意表达，社交寒暄
  T) 任务导向——有明确目标，结构化表达，包含动作意图
  U) 不确定
```

ANALYZING 返回两个信号（一次 LLM 调用，不需追加）：

```
{ choice: "C", logprobs: {A:0.05, B:0.03, C:0.78, D:0.08, E:0.06},
  inputNature: "T", inputNatureLogprobs: {S:0.02, T:0.91, U:0.07} }
```

DET 交叉裁决：

| intent match | input nature | 路由 |
|-------------|-------------|------|
| A-E（高 probability） | — | 对应正式房间 |
| other（低 probability） | S（闲聊） | **SLACK_NODE** |
| other（低 probability） | T（任务导向） | **TEMP_TOPIC** → 创建运行时房间 |
| other（低 probability） | U（不确定） | 追问澄清 |

**一次 ANNLYZING 调用，不改模型，不追加 LLM 调用**——只是 prompt 多问一个问题，返回多一个字段。

#### 5.2 TEMP_TOPIC：运行时创建的房间

TEMP_TOPIC 不是临时草稿。一旦用户命名，就成为**正式房间**，纳入房间体系。

**与正式房间（N1预定义）的对比**：

| | 正式房间 | TEMP_TOPIC |
|------|---------|------------|
| 来源 | N1 边界划定预定义 | ANALYZING other + 用户命名后动态创建 |
| 宪法 | 根宪法 + 环节宪法 + 领域规则 | **仅根宪法 4 条** |
| 环节管道 | ANALYZING→IN_SESSION→EXECUTING→... | **无**（单态，不经过环节管道） |
| 上下文管理 | 上下文管理器 | **上下文管理器**（同样机制） |
| 切换 | "切断房间" → S3 物理释放 | **同样**，"切断房间" → S3 释放 |
| 房间索引 | roomStateIndex | **roomStateIndex**（同样索引） |
| 存储 | contractStore | **contractStore**（同样持久化） |
| 产出 | contract-out JSON | 用户触发总结时生成 |
| 升级路径 | — | N12 规则提炼检测聚集 → 建议升级为正式房间 |

**创建流程**：

```
用户："帮我分析新能源趋势"
  → ANALYZING: intent=other, inputNature=T
  → 调度器确认："这是一个新话题，起个名字方便以后切换。叫什么？"
  
用户："新能源趋势分析"
  → 调度器确认关键词："和'新能源''趋势''分析'这几个词相关，对吗？"
  → 用户确认
  → 调度器创建 room:
     { roomId, roomName: "新能源趋势分析",
       keywords: ["新能源", "趋势", "分析"],   ← 用于 ANALYZING 匹配
       constitution: root_only              ← 仅根宪法4条
       stages: none }                       ← 无环节管道
  → 写入 roomStateIndex（与其他房间并列）
  → 进入 TEMP_TOPIC 单态会话：
     用户输入 → 上下文管理器拼接 conversationLog → 模型回复
```

**退出和切换**：

```
用户："切断房间"
  → 与正式房间完全一致：M1 元指令触发 S3 物理释放 KV Cache
  → buildRoomIndex() 生成全窗口状态快照
  → 用户可从 roomStateIndex 看到 TEMP_TOPIC 房间
  → 用户说"回去新能源趋势分析" → ANALYZING 匹配 keywords → 切换回 TEMP_TOPIC
```

**产出**：

```
用户："帮我总结刚才讨论的要点"
  → 模型输出结构化产出
  → 存入 contractStore.tempTopics
  → topicEvolution 记录
```

#### 5.3 房间三级约束梯度

```
                         ┌──────────────────────────────────┐
 约束最重                │  正式房间（N1 预定义）             │
                         │  根宪法 + 环节宪法 + 领域规则      │
                         │  完整管道 + DET 值域复验           │
                         │  "切断房间" 切换                   │
                         ├──────────────────────────────────┤
                         │  TEMP_TOPIC（运行时创建）          │
                         │  仅根宪法 4 条                     │
                         │  单态（无环节管道）                │
                         │  有 roomId、有关键词、有 conversationLog │
                         │  上下文管理器、roomStateIndex、产出  │
                         │  "切断房间" 切换                   │
                         ├──────────────────────────────────┤
 约束最轻                │  SLACK_NODE（无锚点闲聊）          │
                         │  仅根宪法 4 条                     │
                         │  无 roomId、无存储、无产出          │
                         │  "切断房间" → 回 LISTENING         │
                         └──────────────────────────────────┘
```
三者统一用 M1 元指令"切断房间"退出。约束轻重只决定**房间里有什么**（宪法层级、产出能力、存储），不改变退出方式。

#### 5.4 对现有架构的影响

| 组件 | 需改动 | 影响面 |
|------|--------|--------|
| **ANALYZING prompt** | 新增"输入性质"第二维度分类 | 中（核心改动，但一次调用不改模型） |
| **N3 状态枚举** | 新增 TEMP_TOPIC 系统状态（SLACK_NODE 已是可选状态） | 低 |
| **N4 转移图** | ANALYZING→TEMP_TOPIC 新边 + keywords 匹配路由 | 低 |
| **N5 调度器** | TEMP_TOPIC 路由 + 房间创建/销毁 + "切断房间"复用 | 中 |
| **contractStore** | tempTopics 独立表（或复用 topicIndex 加 source=temp 标记） | 低 |
| **N12 规则提炼** | 远期：检测 temp topic 聚集 → 建议升级 | 低（远期） |

## 改进6：跨房间产出物总索引（Output Registry）

### 问题

当前架构中，每个房间的产出物（contract-out JSON、生成的文档、结构化数据）存储在 contractStore 中，但**没有全局的可发现机制**。roomStateIndex 提供了房间状态摘要（"记账房间有1条半成品"），但没有回答"这个系统的所有产出物有哪些、在哪里、谁生成的"。

实际痛点：用户有一个元智能体（topic_based）生成了 L2 设计文档，后来又用专利撰写智能体写专利。专利撰写智能体需要引用 L2 文档的全文——当前只能靠用户手动找到 L2 文件、手动复制到专利智能体的文件夹。**智能体之间互相不知道彼此产出了什么。**

### 改后设计

#### 6.1 产出物总索引（Output Registry）

在 contractStore 中新增一张 `outputRegistry` 表，作为全系统所有房间产出物的统一登记簿：

```sql
CREATE TABLE IF NOT EXISTS outputRegistry (
    outputId TEXT PRIMARY KEY,          -- 全局唯一产出物ID
    roomId TEXT NOT NULL,               -- 来源房间
    roomName TEXT NOT NULL,             -- 房间名（冗余，便于人类阅读）
    intent TEXT NOT NULL,               -- 房间关联的意图
    taskType TEXT NOT NULL,             -- field_based / topic_based
    outputType TEXT NOT NULL,           -- 产出物类型：contract_json / document / report / code / patent_draft / ...
    outputPath TEXT NOT NULL,           -- 文件系统路径（相对 project_root 或绝对路径）
    outputName TEXT NOT NULL,           -- 产出物名称（人类可读）
    outputSummary TEXT,                 -- 产出物摘要（模型生成，供检索匹配）
    keywords TEXT,                      -- JSON数组，关键标签
    fileSize INTEGER,                   -- 文件大小（字节）
    createdAt TEXT NOT NULL,            -- 创建时间 ISO8601
    createdBy TEXT NOT NULL,            -- 创建者（userId 或 agentId）
    status TEXT DEFAULT 'active',       -- active / archived / deleted
    metadata TEXT                       -- JSON，扩展元数据（字段数、行数、版本等）
);
```

**索引**：`(roomId, createdAt)`、`(intent, outputType)`、`(keywords)`（FTS）。

#### 6.2 写入时机

| 时机 | 操作 |
|------|------|
| IN_SESSION complete（field_based） | contract-out JSON 自动注册到 outputRegistry |
| IN_SESSION complete（topic_based） | 用户确认的最终产出自动注册 |
| TEMP_TOPIC 用户触发"总结" | 总结产出自动注册 |
| 用户手动"导出"/"保存" | 调用 registry.register() |

#### 6.3 检索方式

**DET 确定性查询**（0 次 LLM）：

```
- 按房间：registry.findByRoom("元智能体")
- 按类型：registry.findByType("document")
- 按关键词：registry.search("L2 设计文档")
- 按时间：registry.findSince("2026-07-01")
```

**跨房间发现流程**：

```
1. 用户在专利撰写房间说："调用元智能体之前生成的L2文档，作为专利的背景技术参考"
   → ANALYZING: intent=patent_write（当前房间任务）
   → 第0.5层关键词扫描检测到"调用"+"元智能体"→ 标记 crossRoomRequest
   
2. 调度器查询 outputRegistry:
   registry.search({roomName: "元智能体", keywords: ["L2", "设计"], outputType: "document"})
   → 返回: [{outputId, outputName, outputPath, outputSummary, ...}]

3. 用户确认："对，就是这个文档"

4. 调度器执行 FileCopy:
   cp {outputPath} → {currentRoomWorkspace}/references/
   → 专利撰写房间的上下文管理器在下一轮拼接时注入该文件路径
```

**与 roomStateIndex 的互补**：

| | roomStateIndex | outputRegistry |
|------|------|------|
| 粒度 | 房间级摘要 | 产出物级明细 |
| 回答什么 | "记账房间还有1条半成品" | "元智能体产出了3份文档、2份代码" |
| 消费方 | ANALYZING（跨房间切换建议） | ANALYZING（跨房间产出发现）+ 调度器（文件操作） |

#### 6.4 自动文件复制

当用户确认引用某个产出物时，调度器执行确定性文件操作：

```
调度器.outputRegistry.copyToRoom(outputId, targetRoomId):
  1. 从 outputRegistry 读取 outputPath
  2. 检查文件存在 + 权限
  3. cp {outputPath} → {targetRoomWorkspace}/references/{outputName}
  4. 记录操作日志（谁、什么时候、从哪个房间、复制了什么到哪个房间）
  5. 上下文管理器在下轮拼接时自动将 references/ 目录文件列表注入房间上下文
```

**纪律**：文件复制是唯一的跨房间数据传递方式——不直接注入原文（防止上下文膨胀），而是复制文件到目标房间的 workspace，由目标房间的上下文管理器决定是否引用。这保持了"房间隔离"原则——各房间的 conversationLog 仍然不互通。

#### 6.5 ANALYZING 增强

ANALYZING 在得到 intent 后，可查询 outputRegistry 为模型提供上下文建议：

```
ANALYZING prompt 末尾追加：
[相关产出物]
元智能体房间：L2设计文档_v2.0.md（2026-07-15）、状态机转移图.json（2026-07-15）
专利撰写房间：发明专利交底书_态控架构_v1.0.md（2026-07-14）
```

模型看到后可以主动建议："需要我引用元智能体之前生成的L2设计文档作为专利的技术背景吗？"

#### 6.6 对现有架构的影响

| 组件 | 需改动 | 影响面 |
|------|--------|--------|
| **contractStore** | 新增 outputRegistry 表 + FTS 索引 | 低 |
| **N5 调度器** | 新增 registry 查询和文件复制逻辑 | 低 |
| **ANALYZING** | prompt 末尾追加相关产出物列表 | 低 |
| **上下文管理器** | references/ 目录注入 | 低 |
| **IN_SESSION** | complete 时自动注册产出物 | 低 |
