## 四、产出物
N7 产出两样东西：根宪法文件（root-constitution.json）和架构机制声明，包含以下内容。
### 4.1 根宪法四条（不可逾越的系统红线）
| 编号 | 条款 | 内容 | 可被局部宪法覆盖 | 不可覆盖的原因 |
|------|------|------|----------------|--------------|
| 1 | 环节隔离（Stage Isolation） | 每个环节（ANALYZING/IN_SESSION/CLARIFYING 等）只处理本环节定义的任务范围。非本环节任务必须返回 turnType="off-task"。ANALYZING 也是环节，受环节隔离约束——ANALYZING 只做意图识别，不得越界执行 IN_SESSION 任务 | ❌ 不可覆盖 | 环节隔离是态控架构的基础——每个环节有明确的职责边界，越界处理导致调度逻辑混乱。ANALYZING 越过边界执行 IN_SESSION 任务会导致调度器意图路由失效 |
| 2 | 输出格式（Output Format） | 模型实例返回调度器的结果必须是符合 turnType schema 的结构化 JSON（含 askingField 双模式语义 + changeLevel 预判字段 + changeLevelReason 理由字段，两种 taskType 模式均适用） | ❌ 不可覆盖 | 统一的 turnType schema 是调度器用同一套逻辑处理所有环节的基础。局部宪法不得修改 turnType 的六值定义 |
| 3 | 收敛义务 + 真实判断（Convergence Obligation + Truthful Judgment） | 所有 LLM 节点必须在当前环节内收敛（松弛节点除外），输出结构化 JSON。同时，在理性讨论中，当用户的客观判断与分析产生冲突时，不能为了回避矛盾或安抚情绪而放弃或软化真实判断。确认无误就坚持，确实错了就纠正，绝不能假装同意或模糊分歧 | ❌ 不可覆盖 | 收敛是态控架构的核心原则——每跳往更确定、更结构化的方向收。真实判断是 AI 行为底线——不在安全问题上押注单一概率性判断源 |
| 4 | 不编造功能（Don't Fabricate Features） | 模型不得声称具有其实际不具备的能力（如"我可以帮你下单""我已发送邮件"等未经调度器/DET 确认的操作声明）。模型不得编造系统反馈或虚构操作结果 | ❌ 不可覆盖 | 模型编造功能会误导用户产生错误安全预期。所有功能性操作必须由调度器/DET 实际执行并返回确认，模型只能如实转述确认结果 |
### 4.2 架构机制 M1-M4（约束调度器/DET，非模型约束）
v5.0 新增架构机制 M1-M4。这四项机制是调度器（Scheduler）和确定性执行体（DET，Deterministic Execution Thread）层面的工程约束，不进入模型 prompt——模型不受 M1-M4 约束，M1-M4 是调度器和 DET 在工程实现时必须遵守的机制。
| 编号 | 机制名称 | 约束对象 | 内容 |
|------|---------|---------|------|
| M1 | 元指令集（Meta-instruction Set） | 调度器第0层 | 定义节点切换指令的口令词表——四类通用口令：wake（唤醒，场景特有）/ exit（退出）/ cancel（取消）/ **switch（切断房间，环节切换——v5.1 新增）**。口令层 EXACT_MATCH→直切（0 次 LLM）；自然语言疑似切换→ANALYZING 判 meta_suspect→二次确认后才切（语义确认层）。switch 口令触发 IN_SESSION→ANALYZING，用户主动声明切换环节。口令词表由 N7 设计者定义，N5 第0层直接引用 |
| M2 | 降级链（Degradation Chain） | DET | L1 结构校验 + DET 值域复验 + logprobs 检查（v5.1：替代 confidence，读 API logprobs 做阈值裁决）+ 硬编码兜底，均不可缩减。DET 值域复验按 taskType 分支：field_based 做值域校验，topic_based 做输出格式校验。logprobs 检查区分冷启动期（软拦截）和非冷启动期（硬拦截）。高风险场景可在四项检查后追加 human_takeover 扩展层 |
| M3 | 仲裁权（Arbitration Right） | 调度器 | 调度器 DET 校验结果优先于模型内部校验结果。两者冲突时以调度器 DET 为准。不在安全问题上押注单一概率性判断源 |
| M4 | 节点转换守卫（Node Transition Guard） | DET | 定义两类守卫：(1) 松弛节点↔收敛节点之间转换的触发条件和守卫规则——松弛节点→收敛节点需置信度≥0.7 方可转换，收敛节点→松弛节点通过"放弃指令"匹配模式触发。(2) IN_SESSION 期间用户输入偏离时，调度器 DET 关键词扫描可辅助触发偏离标记，但最终判断由模型给出。具体路由由 N4/N5 实现 |
### 4.3 第4.2条 M2 降级链四项检查详解
| 项 | 执行体 | 时机 | 对应约束 | 说明 |
|----|--------|------|---------|------|
| L1 结构校验 | DET | complete 时 | 根宪法第2条（输出格式） | JSON.parse + Schema 校验，失败→重试1次→熔断→硬编码回复 |
| DET 值域复验 | DET | complete 时 | 环节宪法 @section validation | field_based 做值域校验（金额上限、时间格式、必填字段…），topic_based 做输出格式校验（JSON Schema / 文本格式 / 字数 / 结构…）。规则来源：环节宪法 @section validation 的结构化 JSON。topic_based 无 validation 时仅执行 L1 结构校验 + logprobs 检查，跳过此项 |
| logprobs 检查 | DET | complete 时 | 根宪法第3条（真实判断） | 两步均 DET：①数值 probability < logprobs_threshold（默认 0.4）②pattern 扫"硬撑式确认"词表。冷启动期间（cold_start_observation_window 内）采用软拦截——标记 logprobs_warning 附加到返回中，不进入 CLARIFYING。冷启动结束后恢复硬拦截。冷启动窗口参数由 N10 tunable 清单定义 |
| 硬编码兜底 | DET | 熔断时 | 根宪法第4条（不编造功能） | 所有校验失败后的最终兜底，硬编码回复。不受 validation 规则存在与否的影响 |
### 4.4 off-task 双重保障
v5.0 引入 off-task 双重保障机制：
| 层级 | 保障方式 | 执行体 | 时机 |
|------|---------|--------|------|
| 宪法强制 | 根宪法第1条（环节隔离）强制模型在非本环节任务时返回 turnType="off-task" | 模型（受宪法约束） | 模型输出时 |
双重保障确保：即使模型未自觉遵守环节隔离（宪法强制层失效），ANALYZING重新识别验证也能拦截越界产出。
### 4.5 高风险场景扩展
对于医疗预问诊、复杂客服、金融风控等高危场景，在四项检查后可追加 human_takeover 扩展层。四项检查是通用底线，human_takeover 是高风险场景的扩展，不可替换四项检查中的任何一项。
启用条件：仅当场景的"错误代价"判据为"代价大且无客观答案"或"代价大但有客观校验"时，考虑启用 escalated 模式。默认场景仍用四项检查 + 硬编码兜底。
## 五、关键纪律
1. **根宪法由设计者写死，不接受运行时动态修改**——这是"不可逾越红线"的定义
2. **根宪法四条最低条款必须全覆盖，全部不可覆盖**——各场景不可缩减
3. **M2 降级链四项检查均不可缩减**——高风险场景只能在四项检查之后追加扩展层，不可替换
4. **M2 DET 值域复验按 taskType 分支**——field_based 做值域校验，topic_based 做输出格式校验。这是 taskType 双模式在宪法层面的落地
5. **M2 logprobs 检查区分冷启动期和非冷启动期**——冷启动期软拦截，非冷启动期硬拦截。冷启动窗口参数由 N10 tunable 清单定义
6. **第2条输出格式要求 turnType schema 统一**——含 askingField 双模式语义 + changeLevel 预判字段 + changeLevelReason 理由字段（两种 taskType 模式）
7. **M3 仲裁权绝对优先**——调度器 DET 校验结果与模型内部校验结果冲突时，以调度器 DET 为准
8. **M4 定义两类守卫**——(1) 松弛节点↔收敛节点转换守卫（含置信度≥0.7 和"放弃指令"匹配模式）(2) IN_SESSION 期间偏离守卫。具体路由由 N4/N5 实现，守卫规则由架构机制 M4 定义
9. **M1 元指令集词表由 N7 设计者定义**——N5 第0层直接引用，不得自行修改。节点切换的具体路由规则由 N4/N5 负责
10. **第1条环节隔离适用于所有环节，包括 ANALYZING**——ANALYZING 只做意图识别，不得越界执行 IN_SESSION 任务
11. **第3条合并收敛义务与真实判断**——v5.0 将原 v4.x 根宪法第3条（收敛义务）与部分原公共规则内容合并，统一为 LLM 行为底线
12. **根宪法的修改需要全量回归测试**——影响所有环节宪法和调度器逻辑

## 七、常见踩坑
| 坑 | 表现 | 后果 |
|----|------|------|
| ANALYZING 越界执行 IN_SESSION 任务 | ANALYZING 环节输出了业务处理结果而非意图识别结果 | 调度器意图路由失效，IN_SESSION 环节被跳过 |
| DET 值域复验未按 taskType 分支 | topic_based 任务套用 field_based 的值域校验 | 诗歌创作结果被检查"金额是否在合理范围"，校验永远失败 |
| 仲裁权被弱化 | 设计者声明"当模型 logprobs 高时以模型为准" | 概率性判断源取代确定性程序，安全底线失守 |
| M1 词表与 N5 第0层不同步 | N7 加了新退出词，N5 第0层没更新 | 用户说新退出词不生效 |
| M4 被缩减 | 设计者将松弛节点↔收敛节点转换守卫推出根宪法范围 | 置信度≥0.7 守卫和"放弃指令"匹配模式失去根宪法保护 |
| logprobs 检查未区分冷启动/非冷启动 | 冷启动期也硬拦截 | 初期大量误拦 |
| 硬编码兜底被移除 | 场景设计者认为"不需要兜底" | 所有校验失败时系统无响应 |
| 模型编造功能 | 模型声称"已发送邮件"但实际未执行 | 用户产生错误安全预期 |
| off-task 单层保障 | 仅依赖宪法强制而 ANALYZING重新识别验证未正确配置 | 模型未自觉遵守环节隔离时越界产出无法拦截 |

N7 定下智能体的最高法律——四条不可逾越的红线，全部不可覆盖，由设计者写死。环节隔离（第1条）明确各环节职责边界（ANALYZING 也是环节，只做意图识别），输出格式（第2条）保证概率性展开的出口被硬卡死，收敛义务+真实判断（第3条）合并收敛链和 AI 行为底线，不编造功能（第4条）防止模型误导用户。架构机制 M1-M4——元指令集（M1）、降级链（M2）、仲裁权（M3）、节点转换守卫（M4）——约束调度器和 DET 的工程实现，不进入模型 prompt。off-task 双重保障（宪法强制第1条 + ANALYZING重新识别验证）确保环节隔离万无一失。这份宪法是所有环节宪法和调度器逻辑的根基，修改需全量回归测试。
