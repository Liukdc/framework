N7 根宪法（已定稿）
↓

N9 各环节宪法  ← 你在这里

**【人话版定位】**
N7 定了最高法律（根宪法），N8 核查了架构机制 M1-M4 的工程实现，N9 给每个房间写具体的"服务手册"——记账房间有记账手册，写诗房间有写诗手册。每个手册必须说清楚：这个房间是填表式还是聊天式（taskType）、服务员怎么追问客人（ask-rules）、什么算服务完成（completion）、客人跑题了怎么办（off-task-detection）、交班时给前台什么格式的单子（output-schema）。聊天式的房间还要说明怎么记录关键进展（changeLevel 判定规则）和怎么跨会话续写（topicId 声明）。
**v5.0 核心变化**：
- 环节宪法第一条（环节隔离）为首要内容——明确"非本环节任务返回 turnType='off-task'"为每个环节必须遵守的硬约束
- 追问模板下放——原公共规则中的追问模板内容移至各环节宪法 @section ask-rules
- ANALYZING 也是环节，受环节隔离约束——ANALYZING 只做意图识别，不得越界执行 IN_SESSION 任务
**【与 N7/N8 的关系】**
- 根宪法（N7）是不可逾越红线（v5.0：四条全部不可覆盖），所有环节宪法必须在根宪法的框架内编写
- N8 架构机制核查确认 M1-M4 在调度器/DET 中正确实现后，环节宪法可以安全引用
- 环节宪法（N9）是每个具体任务的行为规则集，受根宪法约束，但可以在允许范围内定制追问策略、完成条件、偏离检测规则等
**【taskType 双模式与 N9 的关系】**
- 每个环节宪法的 taskType 在 N1 初判、N3 固化，N9 直接继承
- field_based 和 topic_based 的宪法写法有本质差异——前者像"填表手册"，后者像"聊天指南"
- N9 必须严格按照 taskType 选择对应的宪法结构模板，两种模板不可混用

### 4.0 产出零：ANALYZING 意图识别宪法（analyzing-session.json）
除了为每个 IN_SESSION 子类型编写环节宪法外，N9 还必须产出一份特殊的 **ANALYZING 意图识别宪法**（`analyzing-session.json`）。
**为什么需要**：N5 第1层（意图识别）需要一份宪法文件来驱动 LLM 从用户输入中识别 intent。N5 v5.7 第1层明确引用 `analyzing-session.json` 作为 ANALYZING 节点的宪法文件。如果不产出这份宪法，N13 骨架生成时 N5 第1层将找不到宪法文件，意图识别无法执行。
**v5.0 重要说明——ANALYZING 也是环节，受环节隔离约束**：根宪法第1条（环节隔离）明确规定 ANALYZING 只做意图识别，不得越界执行 IN_SESSION 任务。ANALYZING 宪法中的 @section off-task-detection 必须落地此约束：非意图识别任务返回 turnType="off-task"。
**ANALYZING 宪法与 IN_SESSION 宪法的关键差异**：
| 维度 | ANALYZING 宪法 | IN_SESSION 宪法 |
|------|---------------|----------------|
| 用途 | 意图识别——将用户输入收敛到 intent 枚举 | 各环节的业务逻辑 |
| taskType | 通常 field_based（收敛到预定义 intent 枚举，可机械校验） | 从 N3 继承 |
| 环节隔离 | ✅ 受约束——只做意图识别，不得越界执行 IN_SESSION 任务（v5.0 新增） | ✅ 受约束——只处理本环节任务 |
| topicId | ❌ 无（ANALYZING 不在 IN_SESSION 内，不涉 topicEvolution） | topic_based 且 topicEvolutionEnabled=true 时必填 |
| changeLevel | ❌ 无（ANALYZING 不在 IN_SESSION 内，不涉 topicEvolution） | topic_based 且 topicEvolutionEnabled=true 时必填 |
| changeLevelReason | ❌ 无 | 两种 taskType 模式均适用，major/minor 时必填 |
| @section output-schema | ✅ 必填——含 intent / taskType / logprobs / extracted / topicId 字段 | ✅ 必填——按 taskType 区分 |
| @section validation | ✅ 必填——intent 枚举校验、logprobs 范围校验、extracted 字段完整性校验 | ✅ 必填 |
| @section ask-rules | ❌ 无（单轮调用，不追问用户） | ✅ 必填——含场景特定追问示例（落地追问模板） |
| @section off-task-detection | ✅ 必填（v5.0 强化——ANALYZING 越界检测） | ✅ 必填 |
**ANALYZING 宪法必须声明的 @section 标签**（最少集）：
- @constitution：`analyzing-session`
- @section taskType：`field_based`
- @section convergence：将用户输入收敛到 N3 定义的 intent 枚举
- @section role：意图识别角色描述（含"非本环节任务返回 off-task"约束）
- @section validation：intent 枚举值校验 + logprobs 范围 [0,1] 校验 + extracted 字段完整性校验
- @section off-task-detection：v5.0 新增——检测非意图识别类输入，返回 off-task
- @section completion：返回 top1 intent + taskType + logprobs + extracted 字段 + topicId
- @section output-schema：含 intent / taskType / logprobs / extracted / topicId 字段的统一输出格式
**output-schema 中的 topicId**：ANALYZING 宪法本身不声明 topicId 字段的判定规则（因为 ANALYZING 不在 IN_SESSION 内），但其 output-schema 必须包含 topicId 字段以支持 topic_based 意图的 topicEvolution 初始化——topicId 由 ANALYZING 阶段 LLM 生成的 topicSlug + userId + intent 组合而成，供后续 IN_SESSION 节点读写主题演化记录。

#### 4.1.2 @section output-schema 详细说明
**统一输出格式**（含 askingField 双模式语义 + changeLevel 字段 + changeLevelReason 字段 + topicId 字段）：
```json
{
"turnType": "ask | reply | complete | off-task | giveup | validation_failed",
"askingField": "null | 字段名(field_based) | 主题标签(topic_based)",
"changeLevel": "major | minor | invalid",
"changeLevelReason": "一句话理由（major/minor 时必填，invalid 时可省略。空值或超100字→DET L1 结构校验拦截，调度器默认填入'未提供'并记录警告，触发重试或进入 CLARIFYING——与 N5 第3层 L1 结构校验失败处理一致）",
"message": "模型对用户说的话（自然语言）",
"validationResult": "null | { field, issue, userInput }",
"collectedFields": {},
"offTaskInput": "null | 用户偏离输入",
"result": "null | 完整结构化结果"
}
```
**askingField 说明**：
- field_based 模式：turnType="ask" 时为当前追问的字段名（供 context-manager 字段级硬门控保护使用）
- topic_based 模式：turnType="ask" 时为模型动态生成的主题锚点标签（供 context-manager 三层注入 + embedding 两阶段检索保护使用）
- turnType="validation_failed" 时为校验失败的字段名或主题维度
- 其他 turnType 时为 null
**askingField 主次标注规则**：强收敛任务中，如果几个字段是强关联的（如"金额+币种"），可以一轮问两个，但要在 askingField 里标清楚主次，格式为 "amount+currency"。topic_based 任务中，askingField 通常为单个主题标签。
**off-task 双重保障说明（v5.0 新增）**：
- 宪法强制层：根宪法第1条（环节隔离）要求模型在 @section off-task-detection 中声明"非本环节任务返回 off-task"
- ANALYZING重新识别验证层：M2 降级链对模型输出进行事后验证
- 即使模型未自觉返回 off-task，ANALYZING重新识别验证也能拦截越界产出
**changeLevel 判定规则（topic_based 模式必填）**：
- invalid：当且仅当本轮输出完全撤销了上一轮的实质性变更（如上一轮新增实体A，本轮删除实体A且无其他衍生变更），且未产生新的有效信息时
- major：涉及核心实体的增删、任务目标的根本性转向、或引入了全新的世界观/规则设定时
- minor：凡不属于上述两者的情况，一律判定为 minor
**changeLevel 判定边界的重要区分**：用户反复修改同一属性（如"改林霜的性格，改了三次才满意"），只要方向不是完全互斥（加→删），就算 minor 累积调整——判定标准是本轮操作是否实质上撤销了上一轮的变更且未产生新的有效信息，而非修改次数。
**changeLevelReason 字段规范（两种 taskType 模式均适用，major/minor 时必填）**：
- 长度：1-100 字。空值或超长→DET 格式校验拦截（L1 结构校验失败），调度器默认填入"未提供"并记录警告，触发重试或进入 CLARIFYING
- 格式：一句话说明判定理由（如"新增了核心角色林霜"或"优化了对白用词"）
- DET 不验证语义正确性，但理由的存在本身迫使模型为自己的判定负责
- 调度器在 turnHistory 中同步记录 changeLevelReason，供 N16 监控 major 占比和 reason 空值率
**兜底机制**：若模型未返回 changeLevel 或格式错误 → 调度器默认标记为 minor（最安全的容错：进入累积压缩流，不丢失信息）。
**topicId 字段声明（topic_based 模式必填）**：
- topic_based 模式的 ANALYZING contract-out 必须包含 topicId 字段
- topicId 由 ANALYZING 阶段 LLM 生成的 topicSlug + userId + intent 组合而成
- topicId 是 topicEvolution 的主键，供后续节点读写主题演化记录
- topic_based 任务自动启用 topicEvolution 三层递进机制（分层留存 + 主题关联 + 异步规则提炼）
#### 4.1.3 @section validation 结构化 JSON 格式
**field_based 模式的 validation 写法**：
```json
{
"rules": [
{"field": "amount", "type": "range", "min": 0, "exclusive_min": true, "error_tag": "amount_invalid"},
{"field": "amount", "type": "range", "min": 0, "max": 999999, "error_tag": "amount_too_large"},
{"field": "time", "type": "format", "pattern": "^(昨天|今天|明天|\\d+号)$", "error_tag": "time_format_invalid"},
{"field": "time", "type": "not_future", "error_tag": "time_future"},
{"field": "quantity", "type": "range", "min": 0, "exclusive_min": true, "error_tag": "quantity_invalid"}
],
"required_fields": ["category", "amount", "time"]
}
```
**topic_based 模式的 validation 写法**：
```json
{
"rules": [
{"field": "lines", "type": "array_length", "exact": 4, "error_tag": "line_count_violation"},
{"field": "lines", "type": "char_count_per_element", "exact": 7, "error_tag": "char_count_violation"},
{"field": "rhyme", "type": "custom", "function": "check_rhyme_scheme", "error_tag": "rhyme_violation"}
],
"required_fields": ["title", "lines", "style", "theme"]
}
```
**topic_based 无格式约束时的 validation 写法**：
```json
{
"rules": [],
"required_fields": []
}
```
注意：即使无格式约束，@section validation 标签本身不可省略——N5 第3层通过检查 rules 数组是否为空来判断回退行为。rules 为空数组时，N5 第3层仅执行 L1 结构校验 + logprobs 检查，跳过 DET 值域复验。**若 rules 为空数组，需在本节下方显式声明 needSemanticExtensionCheck（true/false）**，供 N5 第3层判断是否需要追加 LLM 语义检查。
**规则类型说明**：
| 规则类型 | 含义 | 适用 taskType | 示例 |
|---------|------|-------------|------|
| range | 数值范围校验 | field_based | min/max/exclusive_min/exclusive_max |
| format | 正则格式校验 | field_based | pattern 字段 |
| not_future | 时间不未来校验 | field_based | 无需额外参数 |
| enum | 枚举值校验 | field_based | values 数组 |
| array_length | 数组长度校验 | topic_based | exact/min/max |
| char_count_per_element | 数组每元素字符数校验 | topic_based | exact/min/max |
| custom | 自定义校验 | 两种均可 | function 字段指定 DET 函数名 |
| required | 必填校验 | 两种均可 | 在 required_fields 数组中声明 |
#### 4.1.4 taskType 对宪法写法的影响
| 维度 | field_based 模式 | topic_based 模式 |
|------|----------------|----------------|
| @section taskType | field_based | topic_based |
| @section convergence | "将用户口语收敛到 {字段1, 字段2, …}" | "将用户输入收敛到 [主题空间] 内的 [输出类型]" |

| needSemanticExtensionCheck | 通常 false | 可能 true（各环节在 @section validation 下方声明） |

## 五、关键纪律
1. **taskType 从 N3 继承，N9 不得自行更改**——taskType 在 N1 初判、N3 固化，N9 抄录 N3 固化值。抄录前应检查 N3 交付物中是否有 taskType 相关的 suggestions 或存疑标注
2. **标签必须按 taskType 选择正确的必填/省略规则**——field_based 无 @section topic-boundary、@section domain-rule-priority、@section output-schema 中的 topicId；topic_based 无 @section field-rules、@section parsing。**v5.0：changeLevel + changeLevelReason 两种 taskType 均适用；@importance / @importance:deliverable / @outputs 三种元标签两种 taskType 均必填**
3. **@section validation 标签不可省略，N5 第3层通过检查 rules 数组是否为空来判断回退行为**——rules 为空时仅执行 L1 结构校验 + logprobs 检查，跳过 DET 值域复验。topic_based 无格式约束时保留标签并填入空 rules 数组，同时需在本节下方显式声明 needSemanticExtensionCheck
4. **@section output-schema 必须与 N5 第2层 turnType schema 完全一致**——turnType 六值、askingField 双模式语义、changeLevel 和 changeLevelReason 字段规范不可修改
5. **根宪法必须落地**——第1条（环节隔离）为首要约束——@section role 首位声明 + @section off-task-detection；第2条（output-schema 格式统一）；第3条（真实判断——@section role 中嵌入）
6. **追问模板落地 @section ask-rules**（v5.0 强化：原公共规则追问模板内容）
7. **topic_based 且 topicEvolutionEnabled=true 的环节宪法必须声明 topicId 和 changeLevel 判定规则**——含 changeLevelReason 字段规范（防偷懒机制，含 DET 拦截后兜底行为）
8. **环节宪法的修改只影响该环节**——不需要全量回归测试，但需要验证与其他环节宪法的接口一致性（turnType schema 统一）
9. **N9 作为环节宪法产出节点必须写入 @section 标签要求**——否则 N12 审查时 R1 拆包容易把标签丢掉
10. **topic_based 场景必须声明 needSemanticExtensionCheck**——在 @section validation 下方声明，供 N5 第3层判断是否需要追加 LLM 语义检查
11. **松弛节点（SLACK_NODE）不需要环节宪法**——松弛节点仅受根宪法约束（v5.0），不区分 taskType。N9 只为 IN_SESSION 子类型和 ANALYZING 节点编写环节宪法。设计者不应为松弛节点编写环节宪法
12. **ANALYZING 也是环节，受环节隔离约束**（v5.0 新增）——ANALYZING 只做意图识别，analyzing-session.json 必须包含 @section off-task-detection

## 七、常见踩坑
| 坑 | 表现 | 后果 |
|----|------|------|
| topic_based 的 @section validation 标签被删除 | 无格式约束时设计者直接删了 validation 标签 | N5 第3层无法判断回退行为——标签缺失导致 DET 复验逻辑异常 |
| topic_based 无 validation 时未声明 needSemanticExtensionCheck | rules 为空数组但没有 needSemanticExtensionCheck 声明 | N5 第3层无法判断是否需要 LLM 语义检查 |
| topic_based 的 @section parsing 写了内容 | 设计者给聊天式房间写了硬解析规则 | 语义理解任务不需要硬解析，写了也用不上 |
| changeLevelReason 字段规范遗漏 DET 兜底行为 | 只写"空值拦截"没写"默认'未提供'和重试/CLARIFYING流程" | N13 骨架生成时缺少兜底逻辑实现依据 |
| 忘了在 @section role 首位声明环节隔离 | v5.0 新增要求——@section role 开头没有"非本环节任务返回 off-task" | 环节隔离约束在 prompt 中的首要位置缺失 |
| 忘了在 @section role 里嵌入真实判断 | 角色描述写了但没写坚持真实判断 | 根宪法第3条在环节内无法落地 |
| @section output-schema 的 turnType 六值与 N5 不一致 | 设计者自己加了一个 turnType 值 | N5 第3层校验失败，调度器无法处理 |
| taskType 抄错了 | 从 N3 抄录时看错了行 | 整个宪法的标签选择、追问策略、输出格式全部错误 |
| ANALYZING 宪法遗漏 @section off-task-detection | v5.0 新增要求 | ANALYZING 越界执行 IN_SESSION 任务时无约束 |

## 十、小白操作指南
### 10.0 先搞清楚你在干什么
你现在要干的事：给每个业务房间写一份"服务手册"。N3 已经定好了有哪些房间（IN_SESSION 子类型），你现在要为每个房间写一份详细的服务手册，告诉房间里的大模型服务员：这个房间是填表式还是聊天式、怎么追问客人、什么算服务完成、客人跑题了怎么办、交班时给前台什么格式的单子。
**v5.0 最大的变化**：
- 每条服务手册的第一条纪律就是：**只能干本房间的活，非本房间的活必须说"off-task"**（根宪法第1条环节隔离）
- ANALYZING 前台也一样——ANALYZING 只做意图识别，不能越界跑去帮客人记账或写诗
- 追问模板现在写在每条手册的 @section ask-rules 里
- "坚持真实判断"写在 @section role 里（根宪法第3条）
**一份环节宪法 = 一份服务手册**。如果你的智能体有 3 个业务意图（比如记账、查账、写诗），你就需要写 4 份环节宪法——3 份业务环节宪法 + 1 份 ANALYZING 意图识别宪法（analyzing-session.json）。
**两份模板的区别**：
- **field_based（填表式）**：像填表——有明确的字段要采集，按顺序问，问齐了就完成。记账、改参数、表单填写都是这类
- **topic_based（聊天式）**：像聊天——没有固定字段，在一个主题范围内自由探索，聊到满意了就完成。写诗、角色扮演、方案设计都是这类
**N9 必须遵循的两层约束（v5.0）**：
- **根宪法（N7）**：环节隔离为首要约束（第1条）、turnType 六值格式（第2条）、收敛义务+真实判断（第3条）、不编造功能（第4条）
- **N3/N5 的技术约束**：taskType 从 N3 固化值抄录、output-schema 与 N5 turnType schema 对齐

| 你要用到的东西 | 谁给你的 | 用来干嘛 |
|-------------|---------|---------|
| N3 状态枚举表 | N3 的人 | 看有哪些 IN_SESSION 子类型，每个的 taskType 和 topicEvolutionEnabled 是什么——这是你要写服务手册的房间清单。**特别留意 N3 是否有 taskType 存疑标注** |
| N5 调度器核心逻辑 | N5 的人 | 抄第2层的 turnType schema——你的 @section output-schema 必须和它一模一样 |
| N7 根宪法 | N7 的人 | 确认四条红线：环节隔离（第1条）、输出格式（第2条）、真实判断（第3条）、不编造功能（第4条） |
### 10.2 四步走
**第一步：确定每个房间的 taskType**
去 N3 的交付物里看 inSessionSubtypes 列表。每个子类型都有 intent 和 taskType。直接抄过来，不要自己判断。
**重要**：在抄录前，先检查 N3 交付物中是否有 taskType 存疑的标注。如果 N3 曾标注"判据与 N1 声明不一致但用户坚持原判"或类似内容，先确认是否需要调整宪法写法。
- field_based 的房间用"填表手册模板"
- topic_based 的房间用"聊天指南模板"
如果某个房间的 topicEvolutionEnabled=true，你的手册里必须额外写：topicId 字段声明 + changeLevel 判定规则 + changeLevelReason 字段规范（含 DET 拦截后兜底行为）+ topicEvolution 机制说明。
**第二步：按模板逐节填写标签**
对着 §4.1.1 的标签总览表，逐个填写。不同 taskType 的必填和省略规则不同，注意 topic_based 列"❌ 无"的标签不要写（不是"存在但填空"，是完全不存在）。
**v5.0 关键要求——每个手册开头先声明环节隔离**：
在 @section role 的开头，第一句话必须是"非本环节任务必须返回 turnType='off-task'"。这是 v5.0 的硬性要求。
**通用标签（两种 taskType 都必须写）**：
- `@constitution`：命名规则是 `<intent>-session`
- `@section taskType`：填 `field_based` 或 `topic_based`，从 N3 抄
- `@section convergence`：用一句话说清楚这个房间把用户输入收敛到什么
- `@section role`：写角色描述，**首位声明环节隔离**（v5.0），**必须嵌入真实判断**（根宪法第3条）。topic_based 的还要说明 topicEvolution 机制
- `@section validation`：写结构化 JSON。topic_based 没格式约束时写空 rules 数组——标签不能删。**topic_based 还需要声明 needSemanticExtensionCheck**
- `@section ask-rules`：写追问规则，**必须含追问模板三个维度的场景特定示例**（v5.0）
- `@section completion`：写完成条件
- `@section modification`：写修改规则
- `@section off-task-detection`：写偏离检测规则，**必须落地根宪法第1条——非本环节任务返回 off-task**（v5.0 强化）
- `@section giveup`：写放弃处理
- `@section output-schema`：写输出格式，**必须落地根宪法第2条**
**field_based 专属标签**：@section field-rules、@section parsing
**topic_based 专属标签**：@section topic-boundary（落地根宪法第1条环节隔离）、@section domain-rule-priority（可选）
**第三步：逐房间自检**
- 该有的标签都写了吗？该没有的标签（❌ 无）没多写吗？
- **v5.0**：@section role 首位声明环节隔离了吗？
- 真实判断嵌入了吗？
- turnType 六值完整吗？
- topic_based 且 topicEvolutionEnabled=true 的，topicId 和 changeLevel 判定规则写了吗？
- topic_based 无格式约束的，validation 标签保留且 rules 为空数组了吗？needSemanticExtensionCheck 声明了吗？
**第四步：与 N5 的 turnType schema 对齐**
拿着你写的 @section output-schema，跟 N5 交付物的第2层 turnType schema 逐字段比对。

### 10.4 常见错误
| ❌ 错误 | ✅ 正确 |
|---------|--------|
| topic_based 无格式约束时删了 @section validation 标签 | 标签必须保留，填入空 rules 数组，并在下方声明 needSemanticExtensionCheck |
| topic_based 无 validation 时漏了 needSemanticExtensionCheck 声明 | 在 @section validation 下方声明 needSemanticExtensionCheck |
| topic_based 的房间写了 @section parsing | topic_based 模式此标签不存在（语义理解为主），应删除 |
| changeLevelReason 字段规范只写"空值拦截"没写兜底行为 | 必须完整写 DET 拦截后兜底行为 |
| **v5.0**：忘了在 @section role 首位声明环节隔离 | 第一句话必须是"非本环节任务必须返回 turnType='off-task'" |
| **v5.0**：忘了在 @section role 里嵌入真实判断 | 必须写"确认无误就坚持，确实错了就纠正，绝不假装同意或模糊分歧" |
| **v5.0**：ANALYZING 宪法遗漏 @section off-task-detection | analyzing-session.json 必须包含此标签 |
