## §0 节点定位

P0 是设计者个人认知加载节点，不产出工程文件，但 N1-N16 的质量都取决于这一关。

P0 是最容易被跳过也最贵的跳过：没产出物，"赶进度"时首当其冲。但 P0 没过的人，N3 状态粒度切错、N5 上下文喂多、N9 taskType 声明缺失——根因都可能追溯到 P0。

## §2 八根支柱

### 支柱一：认知局部性（输入侧第一性）

推理阶段每个节点的最优输入 = 当前目标的最小充分条件集，不是全集 Γ。

**装进去的标志**：能说出为什么 ANALYZING 不应看到 LISTENING 全量历史——人受工作记忆 7±2 组块限制，LLM 受 Lost in the Middle 效应和近因偏置影响，DET 要求函数纯度 f(x) 只依赖 x。

### 支柱二：意图收敛（变换侧第二性）

每跳往更确定、更结构化、更接近可执行的方向收。LLM 只在被允许的节点做有限概率性展开，出口被 contract-out schema 硬卡死；DET 执行确定性收敛；人执行责任收敛。整条管道是多轮、分层、分执行体的收敛链。

**装进去的标志**：能区分"收敛"（出口被 schema 钉死）和"展开"（自由生成后面捞）；能举出行业主流是展开、态控改成收敛的例子（如：行业主流是把对话历史全喂给 LLM 让它自己判断意图和执行，态控改成 DET 复验收敛到 valid true/false、EXECUTING 收敛到确定执行结果）。

### 支柱三：边界控制优于切碎 + 环节级自治

拆节点首要动机是切换执行体（引入 DET/HUMAN 兜底），不是"切细了显得专业"。环节级自治 = 态控调度器管路由与全量记录，上下文管理器管上下文拼接，模型管环节内交互——态控调度器只在环节边界执行路由和记录；上下文管理器在环节入口从态控调度器调配的内容中组装上下文；环节内多轮追问、字段校验、偏离检测由模型实例在局部宪法约束下自主完成。

**装进去的标志**：能解释为什么 IN_SESSION 内调度器不逐轮调 LLM（环节内追问封装在模型一次会话中，避免调用翻倍+上下文切碎）；能判断状态粒度是否合理（如为每个字段追问设独立状态就是粒度太细）。

### 支柱四：人作为终极容错闭环 + 工程上限

不追求单次绝对正确，而是"犯错-拦截-纠错"闭环。用户是"重置按钮"。调度器 DET 校验结果优先于模型内部校验结果（架构机制 M3 仲裁权），不在安全问题上押注单一概率性判断源。

**装进去的标志**：能解释为什么仲裁权归 DET；能说出降级链各项检查各管哪类错误；能说出五项降级链的全部五项（L1 结构校验 + DET 值域复验 + confidence 检查 + 跨任务延伸检测 + 硬编码兜底）。

### 支柱五：三层宪法 + 架构机制 + 领域规则体系 + tunable

根宪法 4 条（环节隔离/输出格式/收敛义务+真实判断/不编造功能）+ 架构机制 M1-M4（元指令集/降级链/仲裁权/节点转换守卫——约束调度器/DET，非模型约束）+ 环节宪法（第一条强制 off-task，追问模板下放 @section ask-rules）+ 领域规则（第 2 层，规则来源三层 + 权限分层 L0-L3 + 规则网络结构 + topicPath 子图匹配注入）。v5.0 已取消公共规则——原三条公共规则内容已分拆入根宪法和环节宪法。tunable 通用参数，调整范围受 range/enum 硬限制。

**装进去的标志**：
- **宪法覆盖性**：能说出根宪法 4 条（环节隔离/输出格式/收敛义务+真实判断/不编造功能）各约束什么；架构机制 M1-M4 分别约束谁（调度器/DET）；为什么公共规则被取消（内容分拆入根宪法+环节宪法）
- **领域规则体系**：知道领域规则与宪法体系的关系（是第 2 层，从 topicEvolution + 关联机制异步提炼，用户确认后生效，可随时修改或废弃）；能说出领域规则的三种来源和四级权限，并解释"权限分层 L0-L3 与宪法层级正交"的含义
- **@importance 重要性分层**：同 v5.0

### 支柱六：taskType 双模式

field_based（强收敛，离散参数空间）和 topic_based（弱收敛/发散，连续语义空间）在 N1 初判、N3 固化，运行时不变。

**三个差异点**：

| 维度 | field_based | topic_based |
|------|-----------|------------|
| askingField 语义 | 预定义字段名（如"amount"） | 动态主题标签（如"character_motivation"） |
| 上下文管理器 锚点 | 字段级硬门控，DET 确定性字符串匹配 | 主题级硬门控，三层注入 + embedding 两阶段检索（DET 关键词粗筛 + embedding 精筛 + DET 阈值裁决） |
| DET 复验 | 值域校验 | 格式校验 |

**装进去的标志**：能说出三个差异点；能判断混合型任务的 taskType（如"写七言绝句"是主 topic_based + field_based 格式约束子条款）。

### 支柱七：topicEvolution 三层递进机制

topic_based 任务的标配跨节点传递载体——field_based 的契约是字段集 + sessionCheckpoint（进度快照），topic_based 的契约是事件序列 + 状态快照。

**三层递进结构**：

| 层级 | 机制 | 解决的问题 | 执行方式 |
|------|------|-----------|---------|
| 第一层 | 分层留存（major/minor/invalid） | 单主题信息管理——不膨胀、不丢失 | 同步（模型在 turnType 中顺带输出 changeLevel 预判，调度器直接使用） |
| 第二层 | 主题关联（8 种关联类型） | 多主题知识引用——继承优点、避开雷区 | 异步（不阻塞节点流转），关联摘要注入，原始事件链不注入 |
| 第三层 | 异步规则提炼 | 从多主题实践抽象可复用知识——越用越老练 | 完全异步（独立后台进程，时间窗口/数量阈值/手动触发），经交叉验证和用户确认后生效 |

**装进去的标志**：能说出三层分别解决什么问题；能说出 changeLevel 三级判定的操作化定义（invalid=本轮完全撤销上一轮实质性变更且无新有效信息，major=核心实体增删/目标根本转向/新世界观引入，minor=其余一切）；知道 changeLevelReason 防偷懒机制——模型返回 major 或 minor 时必须附带一句话判定理由，DET 格式校验拦截空值/超长，运行期检测 major 占比 <3% 或 changeLevelReason 空值率 >10% 即告警；能解释为什么规则提炼从同步改为异步（避免状态机阻塞，接受知识真空期换取流转不受阻塞）；能说出 topicEvolution 注入时的分层结构（majorEvents + compressedMinorSummary + recentMinorEvents + currentStateSnapshot）；知道认知局部性与 topicEvolution 的关系（event 序列是 topic_based 的"收敛结果"，关联主题的原始事件链不属于当前主题的最小充分条件集，不注入）。

### 支柱八：sessionCheckpoint + contractEvolution + roomConversationLog（v4.1 新增）

field_based 任务的跨节点传递与记忆体系——sessionCheckpoint 管进度+快照、contractEvolution 管规则演化、roomConversationLog 管对话全量记录。一张表跨 16 个环节：completedSteps 数组追踪已完成环节，stepSnapshots 记录每环节收敛时的结构化字段快照。ttl 过期联动清理 roomConversationLog。上下文管理器在环节入口从三个数据源（roomConversationLog + stepSnapshots + domainRules）中按拼接规则组装上下文。

**装进去的标志**：能说出 sessionCheckpoint 的三个写入时机（complete 追加/覆盖/tll 过期删除）和两种读取路径（ANALYZING 恢复检查 / 上下文拼接）；能说出 contractEvolution 的 changeLevel 判定规则（major=字段结构变更/minor=值微调/invalid=改回原值）；知道 roomConversationLog 的保留期与 sessionCheckpoint.ttl 联动；能解释为什么 sessionCheckpoint 和 topicEvolution 是对位的——前者是 field_based 的跨节点传递载体（进度标记），后者是 topic_based 的（事件流）。

### 3.2 反例识别（3 题）

**基础反例**：给定 LangGraph 客服 Agent（全量历史喂 LLM、自由回复无 schema、两节点间调 LLM、无宪法无降级链无 DET 兜底），指出至少 5 处违背态控之处。

> **参考答案**：(1) 全量历史喂 LLM→违背认知局部性（上下文膨胀触发 Lost in the Middle）(2) 自由回复无 schema→违背收敛义务和根宪法第2条输出格式铁律 (3) 两节点间调 LLM→违背环节级自治（调度器应在入口出口交互，非逐节点调 LLM）(4) 无宪法→违背三层宪法体系（无环节隔离/收敛义务/降级链）(5) 无降级链→违背架构机制 M2 五项检查底线（L1 结构/DET 值域/confidence/跨任务延伸/硬编码兜底全部缺失）(6) 无 DET 兜底→违背架构机制 M3 仲裁权，概率性判断源无确定性程序制衡

**taskType 误判反例**：把"写诗"判为 field_based。指出问题（诗歌无预定义字段集，应为主 topic_based + 格式约束子条款）及误判后果（宪法模板错、锚点模式错、复验方式错、无 topicEvolution 记录导致无法跨会话续写）。

**topicEvolution 机制缺失反例**：给定一个 topic_based 的"角色扮演"场景，但设计者未启用 topicEvolution。指出后果（共 7 项：①无法跨会话续写；②主题演化历史丢失；③无分层留存导致上下文膨胀；④无主题关联导致多主题知识孤立；⑤无异步规则提炼导致智能体永远是菜鸟；⑥上下文管理器 无法执行三层注入，模型每轮从空白开始；⑦turnType schema 缺 changeLevel 字段，分层留存机制无从启动）及修正方案（N1 taskType 初判时明确启用 + N3 初始化 topicIndex 和 topicEvolution + N5 实现 event 追加和分层留存 + N9 环节宪法声明 topicId 和 changeLevel 判定规则 + N6 实现 topicEvolution 存储结构 + N10 声明 topic_based 专用 tunable + N14 审查 topicEvolution 机制实现）。

八根支柱任一不过 → P0 不通过。反例三题任一不过 → P0 不通过。
