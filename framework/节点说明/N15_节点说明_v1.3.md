## 一、位置与定位
```
N14 审骨架 + 跑 20+1 case + 查 9+2 机制（已定稿）
↓
N15 调 tunable + 交付  ← 你在这里
↓ （运行时基线移交，非节点传递）
[N16 运行期迭代回溯]（贯穿全流程，非独立节点）
```
**【人话版定位】**
前面 N1-N14 完成了从设计到实现再到验证的全部流程。N15 是整个态控闭环流程的**收官节点**——做两件事：①根据 N14 的测试结果微调系统参数（tunable）；②确认所有交付物完整且一致，打包交付。N15 的产出是运行时基线——运维侧以此为基准进行运行期监控，当指标偏离基线时触发 N16 迭代回溯。
**N15 的独特定位**：这是整个流程中**最后一次修改机会**。N15 可以调参数、可以修正 L2 文档中的笔误、可以修正骨架代码中 N14 发现的行为偏差。但 N15 **不能**新增设计决策——所有设计决策在 N1-N10 已经完成，N15 只在已有设计框架内优化。

### 10.2 三步走
**第一步：分析 N14 报告，决定调哪些参数**
对照 §4.1.1 的调参决策表，逐条判断。
**特别注意**：
- **PID 自动调节只针对 logprobs_threshold**
- **cold_start_observation_window 是运行时常量**
- **上下文拼接命中率低不一定是参数问题**——field_based 场景优先检查 N9 的 askingField 定义和 N5 的硬门控实现
- **off-task 漏判率异常**优先回溯 N9 设计，不是调参

### 10.4 常见错误
| ❌ 错误 | ✅ 正确 |
|---------|--------|
| 调参后没回归测试就打包 | 至少重跑 case 11/12/16/18 和机制 6/9/+1 |
| PID 自动修改了 cold_start_observation_window | observation_window 是运行时常量——PID 只调 logprobs_threshold |
| logprobs_threshold 从 0.4 直接调到 0.1 | 单次调参幅度不超过默认值的 30% |
| 改了 L2 文档但没同步更新 L3 | L2 和 L3 必须语义一致 |
| 交付物缺少 simulate 报告 | 运维侧需要 simulate 报告作为运行期监控基线 |
| 调参记录只写了"调整参数" | 必须写清楚触发指标、旧值、新值、调整原因 |
| field_based 上下文拼接命中率低就盲目调 topic 参数 | field_based 硬门控基于字段名匹配，先检查 N9 字段名定义和 N5 匹配逻辑 |
