
**【人话版定位】**
前面 N1-N10 产出了设计文档（L2），N11 验证了所有契约一致，N12 把 L2 拆成了纯净的 L3 生成包。N13 的任务是把 L3 生成包"编译"成可运行的骨架代码——scheduler.js（调度器）、context-manager.placeholder.js（上下文管理器）、constitutions/ 目录（各宪法文件反序列化）。
**N13 的独特定位**：这是整个流程中唯一一个**以模型为主角、人只审架构决策**的节点。代码专项模型负责从 L3 生成包到骨架代码的全量生成，开发员负责审查关键架构决策是否正确落地，不审每行代码。

## 五、关键纪律
1. **代码专项模型是 N13 的主力，开发员只审架构决策**——不审每行代码，只审 24 条架构决策是否正确落地
2. **状态转移严格按 state-graph.json**——所有转移边硬编码，不调 LLM 决定下一状态
3. **调度器只调 context-manager 接口，不内联拼接上下文**——这是 N5 v6.0 的铁律
4. **所有 tunable 从 getTunable() 读取，严禁硬编码**
5. **expectsChangeLevel 从路由表条目读取，非转移边 JSON**——与 N4 v5.6 和 N5 v6.0 一致
6. **changeLevel 兜底为 minor + 记录警告日志**
7. **changeLevelReason 兜底为默认"未提供" + 记录警告日志**——空值或超 100 字时的 DET 拦截行为。触发重试或进入 CLARIFYING。与 N5 v1.5 和 N6 v3.0 一致
8. **topicEvolution event 追加时同步写入 changeLevelReason**——changeLevelReason 的提取和兜底处理与 turnHistory 一致
9. **冷启动期间 logprobs_threshold 冻结为默认值 + 软拦截**——解冻后 PID 自动调节 logprobs_threshold。cold_start_observation_window 为运行时常量（仅设计阶段调整 + N15 手动调参）。cold_start_emergency_threshold 解冻后仅通过 N15 手动调参，PID 不自动调节。pid_kp 为 PID 比例系数常量，本身不被 PID 调节
10. **_pendingCollectedFields 保留后需实现恢复行为**——source 标记决定：normal_pause→主动询问，off_task→不主动询问
11. **contractStore 用 SQLite + WAL 模式**——单文件数据库，崩溃时自动回滚
12. **所有宪法文件反序列化后 @section 标签完整**——changeLevelReason 字段规范存在于所有环节宪法中（v5.0：两种 taskType 均适用）

## 七、常见踩坑
| 坑 | 表现 | 后果 |
|----|------|------|
| 调度器中硬编码阈值 | scheduler.js 中直接写 min_match_score=0.3 | 违反 N5 铁律 |
| 调度器内联拼接上下文 | scheduler.js 中直接操作 turnHistory | 接口隔离失效 |
| expectsChangeLevel 从转移边 JSON 读取 | 从 transitionEdge 而非 routeTableEntry 读取 | 与 N4/N5 不一致 |
| changeLevel 兜底缺失 | 模型未返回 changeLevel 时未默认 minor | topicEvolution event 追加时 changeLevel 为空 |
| changeLevelReason 兜底缺失 | 空值或超 100 字时未执行 DET 拦截 | 防偷懒机制失效 |
| changeLevelReason 兜底误加了 validation_failed 标记 | 空值/超长时调度器标记了 validation_failed 并触发 CLARIFYING 流转（注意区分：正确的兜底行为是默认"未提供"+记录警告+触发重试或进入 CLARIFYING，与 N5 v1.5 一致） | 原 v1.0 设计遗漏——v1.1 已修正：空值/超长→DET拦截+默认"未提供"+记录警告+触发重试或进入 CLARIFYING |
| topicEvolution event 追加时遗漏 changeLevelReason | 只追加了 changeLevel，未同步提取和写入 changeLevelReason | N16 无法监控 major 占比和 reason 空值率 |
| 冷启动只冻结了 logprobs_threshold | 其他三个参数未冻结 | 观测基线失真 |
| cold_start_observation_window 被 PID 自动调节 | 解冻后 PID 自动修改了冷启动窗口长度 | 观测窗口长度是设计阶段确定的常量 |
| topicEvolution 注入包未与 stateSnapshotMark 联动 | 注入快照始终相同 | 错误快照内容注入 |
| CLOSING 双出边未区分 | 所有 CLOSING 都回 IDLE | 越界处理后用户无法继续对话 |
| _pendingCollectedFields 只保留未实现恢复行为 | source 标记驱动的恢复逻辑缺失 | 用户体验差 |
| contractStore 未用 WAL 模式 | 并发读被写阻塞 | 高并发场景性能下降 |
| changeLevel 默认值 "minor" 硬编码为魔法字符串 | 代码中多处出现 `changeLevel = 'minor'` | 修改兜底行为需改多处代码 |

### 10.4 常见错误
| ❌ 错误 | ✅ 正确 |
|---------|--------|
| 代码模型生成完后直接进 N14，没审架构决策 | 必须逐条验证 24 条架构决策 |
| scheduler.js 里出现了 `if (score < 0.3)` | 0.3 是硬编码——改成 `getTunable('min_match_score')` |
| context-manager 接口参数签名和 L3 的 interfaces.json 不一致 | 以 L3 的 interfaces.json 为准——修正代码 |
| changeLevel 处理没写兜底 | 加一句：未返回或格式错误→默认 minor + 记录警告日志 |
| changeLevelReason 处理没写兜底 | 加一句：空值/超长→默认"未提供"+ 记录警告日志 |
| changeLevelReason 兜底加了 validation_failed 标记 | 兜底行为：默认"未提供"+记录警告+触发重试或进入 CLARIFYING——与 N5 v1.5 一致 |
| topicEvolution event 追加时只写了 changeLevel，漏了 changeLevelReason | 同步提取模型返回的 changeLevelReason 并写入 topicEvolutionEvent |
| expectsChangeLevel 从 `edge.topicEvolutionEventAppended` 读取 | 改成从 `routeTableEntry.expectsChangeLevel` 读取 |
| 冷启动只冻结了 logprobs_threshold | 四个冷启动参数都不可被 PID 自动修改 |
| cold_start_observation_window 被 PID 自动调节 | observation_window 是设计阶段确定的常量 |
| _pendingCollectedFields 只按 ttl 保留，没写 source 标记的恢复逻辑 | 补全：normal_pause→主动询问，off_task→不主动询问 |
| changeLevel 默认值直接写了 `'minor'` | 定义为常量 `CHANGELEVEL_DEFAULT = 'minor'` |
| changeLevelReason 默认值直接写了 `'未提供'` | 定义为常量 `CHANGELEVEL_REASON_DEFAULT = '未提供'` |
