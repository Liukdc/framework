## 一、位置与定位
```
N3 状态枚举 + 执行体分派 + taskType 固化 + topicEvolution/sessionCheckpoint 初始化配置（已定稿）
↓
N4 状态转移图 + 路由表（含两层模型选择 + taskType 影响 + topicEvolution/sessionCheckpoint 路由）  ← 你在这里
↓
N5 调度器核心逻辑 + 上下文管理器 三接口 + topicEvolution 处理
↓
N6 数据传递协议 + 守卫规则 + topicEvolution 存储
↓
N7-N10 写宪法
```
**【人话版定位】**
N4 把 N3 定好的状态枚举（房间）连成走廊（转移边），并给每条走廊贴上"用哪个模型"的标签（路由表）。每条走廊还要标注"这条走廊是填表式还是聊天式"（taskType），聊天式的走廊还要标注"需要追加 topicEvolution 事件"。下游 N5 按这个走廊图调度状态机流转并初始化 topicEvolution。

## 二、输入
| 输入项 | 来源 | 说明 |
|--------|------|------|
| N3 标准状态集 | N3 交付物 | states 数组，含每个状态的 type/executor/dispatchCriteria |
| N3 IN_SESSION 子类型列表 | N3 交付物 | inSessionSubtypes 数组，含每个子类型的 intent/taskType/topicEvolutionEnabled/triggerCondition |
| N3 topicEvolutionConfig | N3 交付物 | topicEvolution/sessionCheckpoint 初始化配置（enabledIntents/defaultChangeLevel 等） |
| N3 omittedStates | N3 交付物 | N3 省略的状态列表，N4 据此跳过相关转移边 |
| N1 意图清单（含 taskType） | N1 交付物 | 用于路由表第二层 intent 级模型绑定 |
| N1 alignScenarioFlag | N1 交付物 | 判断是否需要 silent_input 路径和 align 路由条目 |
| N1 softLandingTargets | N1 交付物 | 软着陆目标的 intent 映射，用于设计软着陆路由 |

## 三、产出物
N4 产出三样东西：状态转移图（Mermaid 文本 + JSON Schema）、路由表（含两层模型选择）、N4→下游传递契约。
### 3.1 产出一：状态转移图
#### 3.1.1 转移边类型定义
| pathType | 含义 | 是否必覆盖 | topicEvolution 影响 |
|----------|------|----------|-------------------|
| optimal_path | 正常最优路径 | ✅ 必覆盖 | topic_based 任务的 complete 边和 EXECUTING→LISTENING 边需追加 event |
| validation_failed_path | 校验失败路径 | ✅ 必覆盖 | 无直接影响 |
| off_task_path | 偏离路径 | ✅ 必覆盖 | topic_based 任务需追加 event（记录偏离前状态） |
| giveup_path | 放弃路径 | ✅ 必覆盖 | topic_based 任务需追加 event（标记 abandoned） |
| confirm_path | 确认路径（不可逆操作 + topic 确认） | 有不可逆操作或 topic_based 意图时必覆盖 | 无直接影响。v5.4 扩展：同时覆盖 topic_based 主题确认场景 |
| exit_path | 退出路径 | ✅ 必覆盖 | 无直接影响 |
| chitchat_path | 闲聊/越界路径 | 有松弛节点时必覆盖 | 无直接影响 |
| soft_landing_path | 软着陆引导路径 | 当 N1 softLandingTargets 非空时必覆盖 | 无直接影响 |
| silent_input | align 静默输入路径 | 仅 align 场景 | 无直接影响 |
#### 3.1.2 转移边数据结构

```json
"pathType": "optimal_path",

"triggerCondition": "intent='record' AND confidence >= 0.7",
"routeRule": "hard_match",
"modelOverride": null,

"note": ""
}
```
#### 3.1.3 标准状态转移图
**Mermaid 文本**：
```
stateDiagram-v2
[*] --> IDLE
IDLE --> LISTENING : 唤醒
LISTENING --> ANALYZING : 用户输入
ANALYZING --> IN_SESSION : intent=业务意图
ANALYZING --> IN_SESSION : intent IN soft_landing_targets
ANALYZING --> WAITING_CONFIRM : topic_based, topic匹配需确认
ANALYZING --> CLOSING : intent=exit
IN_SESSION --> EXECUTING : complete(经DET复验)
IN_SESSION --> CLARIFYING : validation_failed
IN_SESSION --> ANALYZING : off-task
IN_SESSION --> LISTENING : giveup
CLARIFYING --> IN_SESSION : 用户修正后
IN_SESSION --> WAITING_CONFIRM : 不可逆操作
WAITING_CONFIRM --> EXECUTING : confirm
WAITING_CONFIRM --> IN_SESSION : confirm(topic) | reject(topic→ANALYZING)
EXECUTING --> LISTENING : success
CLOSING --> IDLE : 正常退出
CLOSING --> LISTENING : 越界处理后退出
```
#### 3.1.4 topicEvolution 转移边专项定义
| fromState | toState | pathType | topicEvolutionEventAppended | stateSnapshot 标记 | 说明 |
|-----------|---------|----------|---------------------------|-------------------|------|
| IN_SESSION(topic_based) | EXECUTING | optimal_path | true | N/A | 模型返回 changeLevel，调度器使用 |
| EXECUTING(topic_based) | LISTENING | optimal_path | true | checkpoint | 追加 event，推入提炼待处理队列 |
| IN_SESSION(topic_based) | LISTENING | giveup_path | true | abandoned | 追加 event，推入提炼待处理队列 |
| IN_SESSION(topic_based) | ANALYZING | off_task_path | true | active（保持） | 记录偏离前状态 |
#### 3.1.5 ANALYZING→WAITING_CONFIRM topic 确认路径（v5.4 新增）
以下转移边专门处理 topic_based 任务的跨会话匹配确认：
| transitionId | fromState | toState | pathType | taskType | triggerCondition | 说明 |
|-------------|-----------|---------|----------|----------|-----------------|------|
| N4_T_CONFIRM_TOPIC | ANALYZING | WAITING_CONFIRM | confirm_path | topic_based | topic_based意图 AND topic匹配成功 AND 需要用户确认 | N6 v2.2 协议表第16行源自此边 |
| N4_T_CONFIRM_TOPIC_ACCEPT | WAITING_CONFIRM | IN_SESSION | optimal_path | topic_based | decision='confirm' AND topic_based | 用户确认主题→进入IN_SESSION，携带已确认topicId |
| N4_T_CONFIRM_TOPIC_REJECT | WAITING_CONFIRM | ANALYZING | confirm_path | topic_based | decision='reject' AND topic_based | 用户拒绝/修改→回到ANALYZING重新匹配或新建主题 |
### 3.2 产出二：路由表
#### 3.2.1 两层模型选择机制
**第一层（状态级默认模型）**：ANALYZING → 通用轻量模型；IN_SESSION → 通用模型。
**第二层（意图级绑定）**：N4 阶段只标注 modelHint（field_based→"通用模型即可"，topic_based→"建议深度模型"），具体绑定留到 N5。

- 加粗行为 v5.4 新增（N6 v2.2 回写触发）
- modelHint 值为 null 表示此路由条目不涉及 LLM 节点

## 五、关键纪律
1. 确定性路由：所有转移边必须包含完整的决策信息
2. 路径覆盖：每种 taskType 的适用路径必须全覆盖
3. ANALYZING 用轻量模型
4. topic_based 任务的 topicEvolution 转移边 topicEvolutionEventAppended=true
5. 软着陆路由：N1 softLandingTargets 非空时必须设计
6. topic_based 意图必须设计 ANALYZING→WAITING_CONFIRM 主题确认路径（v5.4 新增）
7. 路由规则全部 hard_match
8. N4 阶段只标注 modelHint，具体绑定留到 N5

## 六、与上下游衔接
### 6.1 与上游衔接检查
| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | N3 状态全覆盖 | stateCoverageCheck.isConsistent=true |
| 2 | N3 IN_SESSION 子类型全覆盖 | 每个子类型在路由表中有对应条目 |
| 3 | taskType 路径覆盖 | 每种 taskType 适用路径已覆盖 |
| 4 | topicEvolutionEnabled 一致性 | N3 topicEvolutionEnabled=true 的 intent 在 N4 边中 topicEvolutionEventAppended=true |
| 5 | topic_based 主题确认路径 | ANALYZING→WAITING_CONFIRM 边存在（v5.4） |
| 6 | alignScenarioFlag 处理 | alignScenarioFlag=true 时有 silent_input 路径 |
### 6.2 与下游衔接
| 下游节点 | 消费内容 |
|---------|---------|
| N5（调度器） | transitions + routeTable + topicEvolutionRouting + modelHint |
| N6（数据协议） | transitions → 数据传递协议（含新增 ANALYZING→WAITING_CONFIRM 第16行，N6 v2.2） |
| N13（骨架生成） | transitions + routeTable → scheduler.js 骨架代码 |

## 七、常见踩坑
| 坑 | 表现 | 后果 |
|----|------|------|
| ANALYZING 不用轻量模型 | 第一层默认模型标了通用模型 | 循环依赖 |
| topic_based 转移边漏标 topicEvolutionEventAppended | 相关边标记为 false | topicEvolution 记录缺失 |
| 遗漏 ANALYZING→WAITING_CONFIRM | topic_based 场景无主题确认路径 | N6 协议表第16行无对应转移边 |
| 路由规则用 LLM | routeRule="llm_match" | 违反路由确定性原则 |
| soft_landing 路由缺失 | 有软着陆目标但无对应转移边 | 软着陆无预定义路径 |
