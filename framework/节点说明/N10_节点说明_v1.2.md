**【人话版定位】**
前面 N7-N9 写了三层宪法（最高法律、通用守则、各房间服务手册），N10 定的是整栋店面的"可调旋钮"——哪些参数可以在系统设置里调整、调整范围多大、谁来消费、改了影响哪段机制。这些旋钮不是让人随便拧的，每个都有明确的取值范围、默认值和冲突检测规则。改完会留日志，可追溯。
**【与 N7/N8/N9 的关系】**
- 根宪法（N7）第5条降级链中的 logprobs_threshold、cold_start_observation_window 等参数在 N10 中声明具体取值
- N8（架构机制核查）的 needSemanticExtensionCheck的 needSemanticExtensionCheck 在各环节宪法中声明，N10 不直接管理此参数
- 各环节宪法（N9）中 @section validation 的阈值参数（如 amount_limit_max）在 N10 中统一管理
- N10 的参数是"受控的灵活性"——适应性受限，但可控、可审计、可测试
**【tunable 参数的定位】**
- tunable 参数是宪法中允许运行时调整的配置项，调整范围受 range/enum 硬限制
- 与纯 LLM 动态调整不同——不是在 prompt 里写条件，而是在系统设置里改参数
- 参数保存后下一轮新会话生效，当前会话不受影响
- 所有调整记录变更日志，不可篡改

#### 4.1.6 cross_param 跨参数联动约束规则（v4.7 新增）
**定位**：扩展 N10 冲突检测引擎，处理 tunable 参数之间的隐式语义依赖。与 single_param range/enum 校验不同——cross_param 不替用户决策，只做语义校验+建议+警告。详见总纲 v4.7 §10.1 和 §C.2。
**初始规则清单**：
| # | 规则名 | 涉及参数 | violation_action | 约束逻辑 |
|---|--------|---------|-----------------|---------|
| 1 | critical_room_context_budget_check | turnHistory_limit, critical_room_history_boost, max_critical_rooms | warning | critical 房间×轮数×每轮 token 不能超过模型窗口 60% |
| 2 | topic_threshold_ordering_check | topic_anchor_similarity_threshold, topic_evolution_match_threshold | **hard_reject** | anchor 必须严格小于 evolution_match（跨会话匹配需更高精度） |
| 3 | cold_start_window_vs_emergency_check | cold_start_observation_window, cold_start_emergency_threshold | warning | window < 30 时统计量不足 |
| 4 | rule_mining_dual_trigger_check | rule_mining_schedule, rule_mining_batch_threshold | warning | batch_threshold > 15 时规则提炼可能长期不触发 |
| 5 | logprobs_cold_start_freeze_check | cold_start_observation_window, logprobs_threshold | warning | window > 150 时 logprobs_threshold 冻结过久 |
**纪律**：cross_param 默认 warning（弹建议，允许强制保存），仅确定性越界约束（如 topic_threshold_ordering）使用 hard_reject。
### 4.2 产出二：冷启动机制说明
#### 4.2.1 冷启动冻结期
| 维度 | 规范 |
|------|------|
| 触发条件 | 系统首次部署或参数重置后，自动进入冷启动观测期 |
| 窗口长度 | cold_start_observation_window 个会话（默认 50） |
| 冻结参数 | 冷启动期间冷启动参数组整体冻结——logprobs_threshold、cold_start_observation_window、cold_start_emergency_threshold、pid_kp。观测窗口长度和应急阈值本身不应在观测期内被修改，确保观测基线稳定 |
| logprobs 检查行为 | **软拦截**——probability < logprobs_threshold 时，不直接拦截 complete 结果。调度器标记 logprobs_warning 附加到返回中让用户可见 |
| 非冻结参数 | 其他参数正常运行，不受冷启动影响 |
| 观测指标 | 监测 logprobs 拦截率、topicEvolution event 写入频率、minor 压缩频率、规则提炼触发频率、冷启动应急解冻触发次数 |
#### 4.2.2 应急解冻
| 维度 | 规范 |
|------|------|
| 触发条件 | 观测指标异常（包括但不限于 logprobs 拦截率 > cold_start_emergency_threshold，默认 0.5） |
| 触发动作 | 系统告警 → 人工确认 → 调整参数 |
| 解冻后行为 | 退出冷启动模式，logprobs 检查恢复硬拦截。PID 反馈控制开始自动调节 |
| 记录要求 | 应急解冻事件记录变更日志：时间戳、触发指标、触发值、操作者、调整内容 |
#### 4.2.3 解冻后 PID 反馈控制
| 维度 | 规范 |
|------|------|
| 触发条件 | 冷启动观测窗口期满或应急解冻触发 |
| 调节公式 | 参数调整 = pid_kp × (目标拦截率 - 实际拦截率) |
| 示例 | logprobs 拦截率目标 5%，实际 15% → 调整 = 0.1 × (0.05 - 0.15) = -0.01 → logprobs_threshold 下调 0.01 |
| 调节频率 | 每 cold_start_observation_window 个会话评估一次 |
| 安全边界 | 调节后的参数值不得超出 N10 声明的取值范围。超出时取边界值并告警 |
#### 4.2.4 变更日志规范
| 字段 | 说明 | 示例 |
|------|------|------|
| 时间戳 | 变更发生的精确时间 | 2026-07-11T14:30:00Z |
| 操作者 | 变更发起人 | 自动调参引擎 / admin_001 |
| 参数名 | 被修改的参数 | logprobs_threshold |
| 旧值 | 变更前的值 | 0.4 |
| 新值 | 变更后的值 | 0.39 |
| 调整原因 | 变更依据 | PID 自动调节：目标拦截率 5%，实际 15% |
| 触发指标 | 触发变更的监控指标 | logprobs 拦截率 15% |

#### 4.3.2 使用说明
1. **填写时机**：N10 设计者完成参数清单和冷启动机制设计后，按照本 Schema 填写 machineReadable JSON
2. **适用性标注**：每个参数的 applicable 字段标注当前场景是否适用
3. **设计阶段参数标注**：boundary_coverage_threshold 和 contract_inheritance_limit 的 isRuntimeParameter=false——这两个参数在设计阶段消费，系统运行期间不生效。运行时系统设置界面中不应暴露
4. **冲突检测**：参数保存前必须通过冲突检测引擎校验——任一规则不满足则拒绝保存
5. **下游消费**：N5 调度器通过 getTunable() 读取参数值；N13 骨架生成时消费 tunable 声明生成配置表单；N14/N15 调参时参考变更日志
6. **变更日志**：每次参数调整必须记录变更日志
## 五、关键纪律
1. **参数默认值不可随意修改**——每个默认值都有设计理由（见 §4.1.4），修改需走变更流程并记录理由
2. **参数调整范围受 range/enum 硬限制**——冲突检测引擎在保存前校验，超出范围则拒绝保存
3. **contract_inheritance_limit 取值范围为 [0.1, 0.5]**——与 N1 传递契约 v6.1 和总纲 v5.3 §8.2 一致，不得放宽
4. **logprobs_threshold 的冷启动软拦截不可跳过**——即使场景不含 topic_based intent，冷启动机制仍需启用
5. **冷启动期间冷启动参数组整体冻结**——logprobs_threshold、cold_start_observation_window、cold_start_emergency_threshold、pid_kp 均不可修改，确保观测基线稳定
6. **应急解冻触发条件为"观测指标异常（包括但不限于 logprobs 拦截率超标）"**——与 N7 根宪法 v1.6 第5条表述一致
7. **topic_based 专用参数在纯 field_based 场景中 applicable=false**——不需要调整的参数应明确标注不适用
8. **所有运行时参数通过 getTunable() 读取，严禁硬编码**——这是 N5 调度器的铁律
9. **变更日志不可篡改**
10. **参数保存后下一轮新会话生效**——当前正在进行的会话继续使用旧参数值
11. **场景特有参数不可滥用**——通用 19 个参数已覆盖绝大多数场景，场景特有参数仅 align 的 divergence_threshold

## 七、常见踩坑
| 坑 | 表现 | 后果 |
|----|------|------|
| 在 N5 调度器代码中硬编码阈值 | 直接写死 min_match_score=0.3 | 想调必须改代码重新部署 |
| contract_inheritance_limit 取值范围用 [0.0, 1.0] | 未与 N1 传递契约对齐 | 可能出现 0 或接近 1 的极端值 |

| cold_start_observation_window 设得太短 | 设为 5 个会话 | 数据不足以建立统计基线 |
| pid_kp 设得太大 | 设为 1.0 | logprobs_threshold 剧烈震荡 |
| 修改默认值不写理由 | 变更日志只写"调整阈值" | 无法追溯调整原因，审计失效 |
| 场景特有参数滥用 | 把环节宪法中所有可配项都声明为 tunable | 系统设置界面参数爆炸 |
| 忘记标注参数的消费方 | 某个参数改了但不知道影响哪段机制 | 调参后果不可预测 |
| 冷启动软拦截机制未启用 | cold_start_observation_window 设为 0 | 冷启动期内 logprobs 低的结果被硬拦截 |
| 将 boundary_coverage_threshold 暴露在运行时系统设置中 | 设计阶段参数出现在运行时配置界面 | 用户可调整一个已不再生效的参数 |

## 十、小白操作指南
### 10.0 先搞清楚你在干什么
你现在要干的事：给智能体装上"可调旋钮"。前面 N7-N9 写了三层宪法，里面有很多参数（比如上下文匹配阈值、logprobs 检查阈值、冷启动窗口大小）。这些参数不能写死在代码里——以后可能需要根据实际运行情况微调。N10 的任务就是把这些参数整理成一份清单，说清楚每个参数的默认值、调整范围、谁来消费、改了影响什么。
**N10 和 N7 的关系**：N7 第5条降级链说了"logprobs 检查区分冷启动期和非冷启动期"，但没写冷启动窗口多大、应急解冻阈值多少——这些具体数字在 N10 里定。
**tunable 参数的"受控灵活性"**：可以调，但不是随便调——每个参数有明确的取值范围。改之前有冲突检测，改之后留日志。改了下一轮新会话才生效。

### 10.2 三步走
**第一步：按场景勾选参数适用性**
- **所有场景都适用**：通用参数
- **场景有 field_based intent**：amount_limit_max 评估是否适用
- **场景有 topic_based intent**：topic_based 专用 8 个全部评估
- **align 场景**：加 divergence_threshold
**特别注意——设计阶段参数**：boundary_coverage_threshold 和 contract_inheritance_limit 是设计阶段参数——在 N2 测试和 N4 设计时消费，系统运行期间不生效。运行时系统设置界面中不应暴露这两个参数（标注 isRuntimeParameter=false）。
**第二步：确认冷启动参数**
冷启动是四个参数协同工作的机制：
- logprobs_threshold（默认 0.4）：冷启动期内冻结为默认值，采用软拦截
- cold_start_observation_window（默认 50）：观测多少个会话后解冻
- cold_start_emergency_threshold（默认 0.5）：观测指标异常时触发应急解冻
- pid_kp（默认 0.1）：解冻后 PID 自动调参的比例系数
四个参数的关系：冷启动开始 → 参数组整体冻结 + 软拦截 → 观测期满或指标异常 → 解冻 → 硬拦截恢复 + PID 自动调参。
**第三步：声明场景特有参数（可选）**——通用参数已覆盖绝大部分需求。不要滥用。
