```
N7 根宪法（已定稿）

  ↓
N9 各环节宪法
  ↓

```

**【人话版定位】**
N7 定了智能体的"最高法律"（根宪法），N8 定的是整栋店面的"通用员工守则"——不管你是前台接线员（ANALYZING）还是房间里服务员（IN_SESSION），只要你是大模型（LLM）干活，就得遵守这三条铁律。这三条规矩不针对某个具体任务，而是所有 LLM 调用都必须遵守的共性约束。

**【与 N7 的关系】**
- 根宪法（N7）是"不可逾越的系统红线"，管的是全局行为底线（安全、收敛、仲裁等）
- 公共规则（N8）是"所有 LLM 调用的输出铁律"，管的是 LLM 的输出格式和行为规范
- 公共规则是根宪法部分条款的落地版——规则1 落地第4条输出格式和第5条 L1 结构校验，规则3 落地第5条跨任务延伸检测和第3条收敛义务
- 规则2（坚持真实判断）是 LLM 行为准则，与根宪法第6条仲裁权共享"不因外部压力而放弃正确判断"的精神，但非直接落地关系——第6条定义的是 DET 与模型之间的仲裁关系，规则2 定义的是模型与用户交互时的行为准则
- 公共规则的修改需要走回归测试，影响所有环节宪法

**公共规则的定位**：
- 公共规则是**所有收敛节点中 LLM 调用的通用约束**，无论哪个环节、哪个任务类型，只要收敛节点调用了 LLM，就必须遵守
- 公共规则共计 **3 条**，不可增减（v4.2 定义）
- **松弛节点除外**——松弛节点仅受根宪法约束（v4.2 §4.2.2），输出为自然语言文本。规则1（输出纯JSON）和规则3（禁止跨任务延伸）不适用于松弛节点。规则2（坚持真实判断）的精神仍适用于松弛节点，但由根宪法约束而非公共规则强制
- 公共规则的修改需要**回归测试**——影响所有环节宪法和调度器逻辑
- 公共规则是根宪法和环节宪法之间的"承重墙"

## 四、产出物

N8 产出一样东西：公共规则文件（common-rules.json），包含 3 条公共规则的完整定义、责任归属、落地方式、taskType 影响。

### 4.1 公共规则三条（所有 LLM 调用的输出铁律）

| 编号 | 规则名称 | 内容 | 责任归属 | 落地方式 | taskType 影响 | 不可缩减 |
|------|---------|------|---------|---------|--------------|---------|
| 规则1 | 输出纯 JSON | 所有 LLM 调用的输出必须是纯 JSON 格式。这是根宪法第4条"输出格式"的落地版 | DET 复验（L1 结构校验，N5 第3层） | N5 调度器第3层执行 JSON.parse + Schema 校验。失败→重试1次→熔断→硬编码回复。校验规则来源：环节宪法 @section output-schema。L1 结构校验不受冷启动机制影响（冷启动仅针对 confidence 检查）。熔断后的硬编码回复不受 validation 规则存在与否的影响——与 N7 根宪法第5条硬编码兜底独立性一致 | 无差异——field_based 和 topic_based 均需输出符合各自 turnType schema 的 JSON。topic_based 输出为 JSON 时同左；纯文本时做格式检查。**松弛节点除外**（输出为自然语言文本，不适用规则1） | ✅ 不可缩减 |
| 规则2 | 坚持真实判断 | 在理性讨论中，当用户的客观判断与分析产生冲突时，不能为了回避矛盾或安抚情绪而放弃或软化真实判断。确认无误就坚持，确实错了就纠正，绝不能假装同意或模糊分歧。这是 LLM 行为准则，与根宪法第6条"仲裁权"共享"不因外部压力而放弃正确判断"的精神——第6条定义的是 DET 与模型之间的仲裁关系，规则2 定义的是模型与用户交互时的行为准则 | 环节宪法（prompt 约束） | 各环节宪法在 @section role 或 prompt 中约束模型行为。DET 不判断"什么叫坚持"——这不是质检员扫关键词能管的事 | 无差异——无论 field_based 还是 topic_based，模型均需坚持真实判断 | ✅ 不可缩减 |
| 规则3 | 一次聚焦一件事，禁止跨任务延伸 | 不得在完成当前任务后主动引入超出当前环节能力范围的延伸话题或推荐。任务内的智能补全（包括在同主题范围内的合理发散）是允许的。这是根宪法第5条"跨任务延伸检测"和第3条"收敛义务"的落地版 | 环节宪法（prompt 约束）+ DET 复验（跨任务延伸检测，N5 第3层） | **prompt 层**：各环节宪法在 @section off-task-detection 或 prompt 中约束模型不得引入跨任务延伸。**DET 复验层**：N5 第3层执行跨任务延伸检测 | **有差异**——field_based：DET 字段白名单（needSemanticExtensionCheck 通常 false），强关联字段可一轮问两个但在 askingField 里标主次。topic_based：主题边界检查（可能需要 needSemanticExtensionCheck=true 追加轻量 LLM），askingField 通常为单个主题标签。**松弛节点除外**（允许发散和跳跃话题，不适用规则3） | ✅ 不可缩减 |

### 4.2 规则3 的详细落地说明

#### 4.2.1 field_based 场景下的规则3 落地

| 维度 | 规范 |
|------|------|
| prompt 约束 | 环节宪法 @section off-task-detection 中声明"不得在完成当前任务后主动引入超出当前环节能力范围的延伸话题" |
| DET 复验 | N5 第3层跨任务延伸检测——DET 字段白名单检查（needSemanticExtensionCheck 通常 false） |
| needSemanticExtensionCheck | 通常 false（字段白名单 DET 即可） |
| askingField 主次标注 | 强关联字段可一轮问两个，格式为 "amount+currency"（主字段在前，次字段在后） |
| 示例 | 记账环节完成后的 modelTurn 不得主动说"需要帮你分析这个月开销吗"——这是跨任务延伸。但如果在同一次记账中，模型主动补全"猫粮是宠物用品类"，这是允许的智能补全 |

#### 4.2.2 topic_based 场景下的规则3 落地

| 维度 | 规范 |
|------|------|
| prompt 约束 | 环节宪法 @section topic-boundary 中声明主题空间边界。@section off-task-detection 中声明"不得引入与主题无关的话题" |
| DET 复验 | N5 第3层跨任务延伸检测——主题边界检查。默认层 DET 做格式层面的越界检查；例外层（仅当 needSemanticExtensionCheck=true 时）追加一次轻量 LLM 调用做 message 文本语义延伸检查 |
| needSemanticExtensionCheck | 可能需要 true（主题越界是语义判断，DET 无法完全覆盖）。各环节在 @section validation 中声明 |
| askingField 主题标签 | 模型动态生成主题锚点标签，通常为单个主题标签 |
| 示例 | 角色扮演环节中，模型不得突然开始讨论"你需要我帮你写首诗吗"——这是跨任务延伸。但在同主题范围内，模型主动探索角色的隐藏动机，这是允许的合理发散 |

#### 4.2.3 topic_based 无 validation 时的降级链回退行为

| 检查项 | 是否执行 | 说明 |
|--------|---------|------|
| L1 结构校验 | ✅ 仍执行 | JSON.parse + Schema 校验正常执行，不受 validation 缺失影响 |
| DET 值域复验 | ❌ 跳过 | 无 validation 规则可校验 |
| confidence 检查 | ✅ 仍执行 | 两步均 DET——数值 < threshold + pattern 扫词表。冷启动期软拦截/非冷启动期硬拦截 |
| 跨任务延伸检测 | ❌ 跳过 | 无 validation 规则可依据——此时规则3 完全依赖环节宪法的 prompt 约束 |
| 硬编码兜底 | ✅ 不受影响 | 不受 validation 规则存在与否的影响 |

**与根宪法的关系**：与 N7 根宪法第5条一致


### 4.3 公共规则与根宪法的对应关系

| 公共规则 | 对应根宪法条款 | 关系说明 |
|---------|--------------|---------|
| 规则1（输出纯 JSON） | 第4条（输出格式）+ 第5条（L1 结构校验） | 规则1 是根宪法第4条和第5条 L1 结构校验的落地版 |
| 规则2（坚持真实判断） | 无直接对应条款（规则2 与第6条仲裁权共享"不因外部压力而放弃正确判断"的精神，但非直接落地关系） | 第6条定义的是 DET 与模型之间的仲裁关系。规则2 定义的是模型与用户交互时的行为准则。两者是同一精神在不同层面的体现 |
| 规则3（一次聚焦，禁止跨任务延伸） | 第5条（跨任务延伸检测）+ 第3条（收敛义务） | 规则3 是根宪法第5条跨任务延伸检测和第3条收敛义务的落地版 |


## 五、关键纪律

1. **公共规则 3 条不可增减**——这是 v4.2 定义的通用底线
2. **规则1 由 DET 复验执行，不由 LLM 自律**——L1 结构校验在 N5 第3层执行，不受冷启动影响，熔断后硬编码回复不受 validation 影响
3. **规则2 由环节宪法在 prompt 中约束，DET 不判断"什么叫坚持"**——规则2 是 LLM 行为准则，与根宪法第6条仲裁权共享精神但非直接落地关系
4. **规则3 分两层落地——prompt 约束（环节宪法）+ DET 复验（N5 第3层）**——两层必须同时存在
5. **规则3 按 taskType 区分落地方式**——field_based 用字段白名单 + askingField 主次标注；topic_based 用主题边界检查
6. **规则3 在 topic_based 无 validation 时仅依赖 prompt 约束**——与 N7 根宪法第5条一致
7. **公共规则的修改需要全量回归测试**——影响所有环节宪法和调度器逻辑
8. **公共规则与根宪法的对应关系必须清晰可追溯**

## 七、常见踩坑

| 坑 | 表现 | 后果 |
|----|------|------|
| 规则2 试图用 DET 关键词扫描落地 | 设计者写了一个词表想自动检测模型是否"不坚持" | DET 无法判断"什么叫坚持" |
| 规则3 在 topic_based 场景下用字段白名单 | 对诗歌创作结果检查"是否含不该有的字段" | 诗歌创作没有预定义字段集 |
| 规则3 needSemanticExtensionCheck 忘记在环节宪法中声明 | topic_based 环节宪法没有声明此字段 | N5 第3层不知道是否需要追加 LLM 语义检查 |
| 规则3 在 topic_based 无 validation 时仍执行 DET 复验 | DET 复验试图校验不存在的 validation 规则 | 校验报错或空转 |
| 公共规则与根宪法脱节 | 规则3 定义了跨任务延伸检测但根宪法第5条没有对应条款 | 公共规则失去根宪法支撑 |

## 九、一句话收 N8

N8 定下所有 LLM 调用的三条通用铁律——规则1（输出纯 JSON）由 N5 第3层 DET 执行 JSON.parse + Schema 校验（不受冷启动影响，熔断后硬编码兜底不受 validation 影响），规则2（坚持真实判断）由 N9 环节宪法在 @section role 中嵌入 prompt 约束，规则3（一次聚焦，禁止跨任务延伸）分两层落地——prompt 约束（N9 环节宪法 @section off-task-detection）+ DET 复验（N5 第3层跨任务延伸检测）。规则3 按 taskType 区分：field_based 用字段白名单 + askingField 主次标注，topic_based 用主题边界检查（可能需 needSemanticExtensionCheck）+ askingField 主题标签。topic_based 无 validation 时仅执行 L1 + confidence 检查，DET 复验层跳过跨任务延伸检测。三条规则均不可缩减，修改需全量回归测试。


## 十、小白操作指南

### 10.0 先搞清楚你在干什么

你现在要干的事：把 N7 的"最高法律"翻译成所有大模型员工都能看懂、能执行的"通用员工守则"。

**N7 和 N8 的关系，用餐饮店比喻**：
- N7 是"食品安全法"——店里任何人不得用过期食材，这是红线，碰了就关门
- N8 是"通用员工守则"——不管你是厨师还是服务员，都得遵守：①上菜必须用盘子（输出纯 JSON）；②不能骗客人（坚持真实判断）；③一次只服务一桌、不要串台（一次聚焦一件事）
- N8 的三条守则中，规则1和规则3能直接追溯到 N7 的某条法律；规则2是与 N7 第6条共享精神的行为准则，不是直接落地

### 10.1 动手前先检查

| 你要用到的东西 | 谁给你的 | 用来干嘛 |
|-------------|---------|---------|
| N7 根宪法 | N7 的人 | 确认规则1和规则3在根宪法中有对应条款；确认规则2与第6条共享精神 |
| N5 调度器核心逻辑 | N5 的人 | 确认规则1和规则3在 N5 中已被实现 |
| N1 边界清单 | N1 的人 | 确认你的场景有哪些 intent，每个的 taskType——这影响规则3 怎么写 |

### 10.2 三步走

**第一步：确认三条守则的"谁负责执行"**

- **规则1（输出纯 JSON）**：执法单位是 **DET 质检员**（N5 第3层 L1 结构校验）。不是让大模型自己"自觉"输出 JSON
- **规则2（坚持真实判断）**：执法单位是 **大模型自己**（靠 N9 环节宪法的 prompt 约束）。质检员没法判断"什么叫坚持"
- **规则3（一次聚焦一件事）**：执法单位是 **两层联合**——①大模型自己（prompt 约束）② DET 质检员（N5 第3层跨任务延伸检测）

**第二步：逐条写出"具体怎么执行"**

每条规则不光要说"是什么"，还要说"怎么查"、"失败了怎么办"。

**规则1**：谁来查=DET（N5第3层），怎么查=JSON.parse+Schema校验，失败→重试1次→熔断→硬编码回复。L1 结构校验不受冷启动影响。熔断后硬编码回复不受 validation 影响。

**规则2**：谁来查=大模型自己（N9 @section role prompt约束），没法用程序查——DET不扫关键词。嵌入位置：各环节宪法 @section role，模板句式："确认无误就坚持，确实错了就纠正，绝不假装同意或模糊分歧"。与 @section off-task-detection 不矛盾。

**规则3**：谁来查=两层（prompt + DET）。field_based=字段白名单（needSemanticExtensionCheck通常false），topic_based=主题边界检查（可能needSemanticExtensionCheck=true）。topic_based无validation时仅执行L1+confidence检查，DET复验跳过跨任务延伸检测。

**第三步：验证与根宪法的对应关系**

| 公共规则 | 对应根宪法哪条 | 对账问题 |
|---------|-------------|---------|
| 规则1 | 第4条（输出格式）+ 第5条（L1 结构校验） | 是否完整落地了这两条？ |
| 规则2 | 无直接对应条款（与第6条共享精神） | 不能写成"落地版"关系 |
| 规则3 | 第5条（跨任务延伸检测）+ 第3条（收敛义务） | 是否区分了 field_based 和 topic_based？ |

### 10.3 自查清单

- [ ] 三条规则都写了"谁来查"（责任归属）
- [ ] 三条规则都写了"怎么查"（落地方式）
- [ ] 三条规则都写了"失败了怎么办"（失败处理）
- [ ] 规则1 注明了冷启动无关性和硬编码兜底独立性
- [ ] 规则2 注明了在 N9 中的嵌入位置（@section role）和与 @section off-task-detection 的联动说明
- [ ] 规则3 区分了 field_based 和 topic_based
- [ ] 规则3 说明了 topic_based 无 validation 时的降级链回退行为
- [ ] 规则2 的根宪法映射正确表述为"与第6条共享精神，非直接落地关系"
- [ ] 没有试图给规则2 写"关键词词表"

### 10.4 常见错误

| ❌ 错误 | ✅ 正确 |
|---------|--------|
| 给规则2 写了一个"妥协词表" | 规则2 不是关键词问题。DET 分不清模型说"好的"然后纠正和模型说"好的"然后妥协——只能靠 prompt 约束 |
| 规则2 写成"根宪法第6条的落地版" | 规则2 是 LLM 行为准则，第6条是 DET 与模型的权力关系。正确表述是"共享精神" |
| 规则3 在 topic_based 场景下写了"字段白名单" | topic_based 没有预定义字段集。改成"主题边界检查" |
| 规则3 只写了 prompt 约束，没写 DET 复验 | 规则3 是双层防线——两道必须都有 |
| 规则3 needSemanticExtensionCheck 写死了 true/false | 这个值不是 N8 定的——是 N9 每个环节宪法根据自己的场景特征分别声明的 |
