# 态控流程指南 · 五、迭代回路

> **细则文件**——05/07。版本 v8.3，基于母本 v5.6。

## 五、迭代回路

### 10.1 触发条件

六种情况触发响应：

| 触发类型 | 具体信号 | 处置路径 |
|---------|---------|---------|
| 边界-状态矛盾 | 状态节点处理的输入落在"不做什么"内且无处理路径 | 回溯→重审边界（补全护栏策略） |
| 宪法-状态矛盾 | 规则逻辑不自洽（逻辑bug，不适用累积阈值） | 回溯→修正宪法 |
| 守卫-边界矛盾 | 合法输入被误杀 | 回溯→调整词表或护栏 |
| 参数漂移信号 | 指标连续偏离基线，但边界/宪法/守卫均按预期工作 | 参数调节评估 |
| taskType 误判 | taskType 初判错误导致后续节点行为异常 | 回溯→第一步重判 taskType |
| 边界覆盖缺口 | 功能范围外输入无处理路径或护栏响应质量不合格 | 回溯→第一步补全护栏策略或优化护栏响应语 |

**关键分流逻辑：** 如果系统在"正确执行当前边界和规则"的前提下，监控数据仍然不好看——先去参数调节宪法看是不是参数漂移。不要回溯，不要改边界，去调参数。

### 10.2 回溯时带回的数据

回溯必须携带下游的具体矛盾实例：触发矛盾的状态节点定义、无法自洽的宪法条款原文、误杀统计、词表调整方案记录、taskType 误判的具体案例、边界覆盖缺口的具体输入类型等。

### 10.3 重审边界的合法决策

只有三种合法决策：

1. **收缩"不做什么"：** 将原来不做的事移入"做什么"
2. **重新定义边界护栏策略：** 从"硬拒"升级为"软着陆"。硬约束：软着陆的分支目标必须已在状态转移图中存在
3. **承认该输入类型为合法但降级处理：** 分配显式的降级路径

### 10.4 迭代终止条件与过度拟合止损

- 越界率 < 目标阈值
- 无新增矛盾
- 或达最大回溯轮次（默认 3 轮）

同类矛盾信号须在 N 次（建议 N≥3）真实会话中出现，或单次撞击的误杀率 > X%，才触发回溯。单一样本不构成回溯依据。例外：宪法-状态层面的自洽性矛盾属于逻辑 bug，不适用累积阈值规则。taskType 误判属于结构性错误，不适用累积阈值规则——一旦发现立即回溯。边界覆盖缺口属于结构性错误，不适用累积阈值规则——一旦发现立即回溯。

### 10.5 执行体错配矛盾

| 检测信号 | 现象 | 处理 |
|---------|------|------|
| 信号A（模型校验过松） | 模型第一道校验通过率异常但 DET 复验拦截率异常升高 | 回溯，检查环节宪法校验规则与调度器 DET 规则一致性 |
| 信号B（模型校验过严） | 模型频繁返回 validation_failed 但用户后续修正率低 | 回溯，检查环节宪法校验规则是否过于严格 |
| 信号C（模型假装懂） | logprobs 拦截率异常升高（>10%） | 回溯，检查环节宪法 prompt 是否过于鼓励"给出答案" |
| 信号D（模型跨任务延伸） | 跨任务延伸拦截率异常升高（>5%） | 回溯，检查收敛方向或字段白名单/主题边界 |
| 信号E（意图识别不准） | 意图级纠错触发率异常升高（>5%） | 回溯，检查意图识别宪法 |
| 信号F（topic_based 主题漂移） | 模型频繁重复追问已讨论过的主题（上下文拼接命中率低） | 检查 topic_anchor_similarity_threshold 或 topic_keyword_overlap_threshold |
| 信号G（taskType 误判） | taskType 误判率异常升高（>2%） | 回溯第一步 |
| 信号H（边界覆盖缺口） | 功能范围外输入无处理路径比例>5% 或 护栏响应质量不合格占比>10% | 回溯第一步 |
| 信号I（topicEvolution 机制失效） | topicEvolution event 写入频率过低（<1 event/会话） | 检查调度器是否正确追加 event |
| 信号J（topicEvolution 跨会话匹配失败率高） | 匹配成功率过低（<30%） | 检查 topic_evolution_match_threshold 或 stateSnapshot 描述 |
| 信号K（topicEvolution minor 压缩异常） | 压缩频率过高（>1次/会话）或注入包大小持续增长 | 检查 topic_evolution_minor_compaction_threshold |
| 信号L（invalid 判定过激） | invalid 丢弃率过高（>30%），正常修改被误判 | 检查 changeLevel 判定规则 |
| 信号M（topicIndex 检索失败率高） | 用户明确提到主题名但关键词匹配未命中 | 检查 topicIndex 索引是否正常更新 |
| 信号N（领域规则误导率高） | 领域规则被注入后，用户频繁修正或否定模型输出 | 检查该规则是否需要修改或废弃 |
| 信号O（规则提炼噪声率高） | 候选规则确认率过低（<30%） | 调整 domain_rule_validation_threshold |
| 信号P（changeLevel 判定异常） | invalid 占比 > 30% 或 major 占比 < 3% 或 changeLevelReason 空值率 > 10% | 检查环节宪法的 changeLevel 判定规则；若 changeLevelReason 持续为空，检查模型是否未理解输出格式要求 |
| 信号Q（规则提炼队列积压） | 待处理队列持续增长 | 调整 rule_mining_schedule、rule_mining_batch_threshold 或 rule_mining_batch_size |

### 收束


态控闭环设计方法论的核心是：第一步划边界，第二步拆状态（含 taskType 固化 + topicEvolution 三层递进初始化），第三步写宪法（v5.0 三层体系：根宪法4条 + 环节宪法 + 架构机制 M1-M4 + 领域规则）。编程规则宪法（附录E）作为全局元约束贯穿全管线。——顺序不能乱，每一步都有明确的交付物。三步法的底层逻辑是认知局部性原理——每个节点只接收最小充分条件集，不喂全局。

v7.1：topicEvolution 的三层递进——分层留存让记忆不膨胀（模型判定 changeLevel），主题关联让知识不孤立（关联在元数据层，隔离在数据层），异步规则提炼让智能体从菜鸟变专家（滞后但可靠，不在状态机链路内阻塞用户交互）。**v7.1 将领域规则从平铺列表升级为规则网络图**——规则有来源（scenario_seed/user_initiated/model_inferred）、有权限（L0-L3）、有层级（topicPath 前缀匹配）、有关联边（depends_on/conflicts_with/strengthens），注入方式也从全量注入改为按 topicPath 子图匹配精准注入。三层均遵循"自动生成 + 用户确认"的人机协作原则。领域规则不是设计者拍脑袋定的，是从实际项目中异步长出来的。

topic_based 任务的"契约"定义：field_based 的契约是字段集（contract-out JSON），topic_based 的契约是按信息净价值分层留存的演化记录（topicEvolution）+ 轻量级元数据关联（关联摘要）+ 异步凝结的经验法则（领域规则）。不是按时间新久留，是按信息净价值留——大版本永久保留（结构性的），小版本汇总压缩（渐进的），无效过程归档备查（审计的）。
