# 态控流程指南 · 一、机制说明

> **细则文件**——01/07。版本 v8.2，基于母本 v5.6。

## 一、机制说明

### 2.1 taskType 双模式机制说明

**区分原点：收敛空间的性质，而非任务内容。** taskType 是态控架构所有设计决策的逻辑起点——上下文拼接方式、校验方式、记忆载体、规则匹配键，全部收敛到这一个二分法上。

每个 intent（意图） 在 N1（第一步） 边界划定阶段就确定了 taskType，这个决定直接影响后续所有节点的行为：

| 维度 | field_based（基于字段）——离散参数空间 | topic_based（基于主题）——连续语义空间 |
|------|----------------------------------|---------------------------------------|
| 收敛空间 | 离散参数空间 | 连续语义空间 |
| 典型场景 | 记账、改柜体参数、表单填写、查询、删除 | 角色扮演、开放研究、诗歌创作、头脑风暴、专利撰写、方案设计 |
| 收敛目标 | 一组预定义的结构化字段 | 受控的主题空间内的输出 |
| askingField 语义 | 字段名——当前追问或校验失败的数据库字段名（如 "amount"、"time"） | 主题标签——模型动态生成的主题核心锚点标签（如 "character_motivation"、"plot_conflict"、"hypothesis_A"） |
| 上下文管理器 锚点 | 字段级硬门控（确定性，DET 执行）——保护与 askingField 同名的历史轮次不被淘汰 | 主题级硬门控 + 三层注入（两阶段检索——DET 关键词粗筛 + embedding 相似度精筛 + DET 阈值裁决，0 次 LLM 调用；注入内容分第0/1/2层：已生效领域规则/关联主题摘要/topicEvolution 分层包）——保护与 askingField 语义相关的历史轮次不被淘汰 |
| 环节宪法写法 | 有明确字段采集规则 + 值域校验（@section field-rules + @section validation） | 只有主题边界 + 输出格式约束（@section topic-boundary + @section output-format） |
| DET 值域复验 | 值域校验（金额上限、时间格式、必填字段…） | 输出格式校验（JSON Schema / 文本格式） |
| 模型选择倾向 | 通用模型即可（字段映射不需要深度能力） | 倾向使用深度模型（主题理解和创意生成需要深度能力） |
| 跨节点传递的载体 | contract-out JSON（字段集）+ sessionCheckpoint（进度标记 + 快照） | topicEvolution（事件序列 + 状态快照）+ 主题关联 + 领域规则（异步批量提炼） |
| 新开对话时注入什么 | 上下文管理器拼接的内容（roomConversationLog + stepSnapshot + 已生效规则） | topicEvolution 的分层注入包 + 关联主题摘要卡片 + 已生效的领域规则 |
| 长期记忆载体 | contractStore + sessionCheckpoints | contractStore.topicEvolution + contractStore.topicIndex + contractStore.domainRules |
| 是否标配 | 是（没有 contract-out 就不叫 field_based） | 是（没有 topicEvolution 就不叫 topic_based） |

**两种模式的统一性：** 尽管 taskType 有两种模式，但 turnType 六值（ask / reply / complete / off-task / giveup / validation_failed）是统一的，态控调度器用同一套逻辑处理所有环节的模型返回。差异仅在 askingField 的语义解释和 上下文管理器 的锚点模式上——这是"框架统一、分支差异"的设计原则。

### 2.2 topicEvolution 机制总览

topicEvolution 是 topic_based 任务的标配跨节点传递载体。

| 层级 | 机制 | 解决的问题 | 核心操作 | 用户确认 | 执行方式 |
|------|------|-----------|---------|---------|---------|
| 第一层：分层留存 | topicEvolution 按 changeLevel（major/minor/invalid）分层留存 | 单个主题内的信息管理——不膨胀、不丢失 | 模型（LLM）在 turnType 中顺带输出 changeLevel 预判字段，态控调度器直接使用；major 永久保留，minor 汇总压缩，invalid 迁入归档表 | 间接（通过 turnType 决定 major） | 同步（模型返回 turnType 时顺带完成，不额外增加 LLM 调用） |
| 第二层：主题关联 | topicIndex 中建立主题间的关联关系（prerequisite/reference/derivative/conflict/cooperative/replacement/variant/evolution） | 多主题间的知识引用——继承优点、避开雷区 | 系统自动发现或用户手动建立，用户确认后生效 | 必须（关联类型和摘要内容均由用户确认） | 异步（不阻塞节点流转） |
| 第三层：规则提炼 | 从多条同类主题的 topicEvolution 和关联关系中提炼候选规则，经交叉验证和用户确认后存入 domainRules 库 | 从多主题实践中抽象可复用知识——越用越老练 | 异步批量触发（基于时间窗口或数量阈值），提炼引擎作为独立后台进程运行；候选规则经交叉验证后提交用户确认 | 必须（每条候选规则均需用户确认、修改或拒绝） | 异步（完全脱离状态机主链路） |

**三层关系：** 分层留存是地基——没有分层，主题演化数据要么膨胀要么丢失；主题关联是桥梁——没有关联，多个主题是孤立的，知识无法传递；异步规则提炼是塔尖——没有提炼，智能体永远是从零开始的菜鸟。地基和桥梁在状态机内完成，塔尖在状态机外异步完成。

**规则提炼从同步触发改为异步批量触发：** 在节点 exit 时同步触发规则提炼的设计存在状态机阻塞风险。v7.0 修正为基于时间窗口或数量阈值的异步批量提炼。态控调度器在节点 exit 时只落盘 topicEvolution event 并放行，不触发任何提炼逻辑。这引入了"知识真空期"——领域规则不保证跨相邻任务的即时传递，仅提供跨中长期任务的累积经验。

### 2.2b sessionCheckpoint 机制总览（v7.2 新增）

sessionCheckpoint 是 field_based 任务的标配跨节点传递载体——与 topicEvolution 在 topic_based 中的地位对等。

```sql
sessionCheckpoints(userId, lastCompletedStep, completedSteps[], stepSnapshots{})
```

| 字段 | 含义 | 谁写 |
|------|------|------|
| lastCompletedStep | 最后一个完成的环节名 | 每 complete → 态控调度器更新 |
| completedSteps | JSON 数组，已完成环节列表 | 同上 |
| stepSnapshots | JSON 对象，每环节收敛时的结构化字段快照 | 同上 |

**写入时机**：环节 complete 时追加进度 + 写入快照；回到已完成环节重做时覆盖快照；ttl 到期仅清理 sessionCheckpoints 进度行，不联动清理 roomConversationLog（roomConversationLog 按 importance 分级独立保留：永久/90天/30天/7天）。

**读取路径**：ANALYZING 恢复时态控调度器检查→决定继续还是从 P0 开始；上下文拼接时读 stepSnapshots[环节名] 的 collectedFields。

**sessionCheckpoint 与 topicEvolution 的对位**：

| 维度 | topicEvolution（topic_based） | sessionCheckpoint（field_based） |
|------|----------------------|----------------------|
| 载体形式 | 事件流：每节点 exit 追加 event | 进度标记：每节点 complete 更新一行 |
| 匹配方式 | topicId embedding 语义检索 | userId DET 精确匹配 |
| 记录粒度 | 事件级 | 环节级 |
| 注入给模型 | 三层注入 | 结构化字段 collectedFields |

### 2.2c contractEvolution 机制总览（v7.2 新增）

contractEvolution 是 field_based 的领域规则演化——与 topic_based 的异步规则提炼对位。两者共享同一套引擎（changeLevel→ruleMiningQueue→异步提炼→domainRules），唯一差异在匹配键：topic_based 用 topicPath 语义检索，field_based 用 stepName DET 精确匹配。

**changeLevel 判定规则（field_based）**：

| 级别 | 判定标准 | 留存 |
|------|---------|------|
| major | 字段的结构性变更——新增/删除必采字段、修改字段值域、改变 Schema 结构 | 永久保留，不可压缩 |
| minor | 字段值微调——默认值修改、取值范围调整、措辞优化 | 30 条压缩为 1 条摘要 |
| invalid | 完全撤销——用户改了又改回原值（与上一版 contract-out 完全相同） | 归档后定期删除（90 天） |

**存储与注入**（复用 contractStore，按 stepName 隔离）：

| 表 | field_based 行为 |
|----|----------------|
| domainRules | ruleId 含 stepName 前缀，按 stepName DET 精确匹配注入 |
| ruleCandidates | 每个 IN_SESSION complete 时模型推断的用户偏好 |
| ruleMiningQueue | 按 stepName 分队列，异步提炼按 stepName 分组处理 |

**与 topic_based 规则提炼的核心对应**：topic_based 的 changeLevel 判定基于用户交互中的明确行为信号（如版本号变化），field_based 基于字段的结构性变更/微调/撤销——判定标准不同，但分层留存策略完全相同。详见总纲 v5.5 §3.7.2.2。


### 2.2d 产出物总索引——Output Registry（v5.3）

outputRegistry 是全系统房间产出物的统一登记簿。与 roomStateIndex（房间状态摘要）互补——roomStateIndex 回答"房间有什么待办"，outputRegistry 回答"房间产出了什么"。详见 v5.5 §3.1.3。

**关键机制**：
- **存储**：contractStore.outputRegistry 表（DDL 见 v5.6 附录 C.3），含 outputId/roomId/outputType/outputPath/outputName/outputSummary/keywords
- **检索**：DET 确定性查询（0 LLM），支持按房间/类型/关键词/时间组合检索
- **注册门槛**：topic_based complete 自动注册；field_based 仅用户命名/导出的产出注册（每轮 complete 裸字段集不注册，避免噪声）
- **跨房间发现**：ANALYZING 语义判断→调度器 DET 精确查询→用户确认→文件复制到目标房间 references/ 目录
- **自由切换第四支柱**：产出物总索引与 roomConversationLog（全量记录）、stepSnapshot（对话快照）、规则提炼按环节隔离并列，构成自由切换的四个支柱


### 2.3 第一层：topicEvolution 分层留存机制

#### 2.3.1 topicEvolution 的定义

topicEvolution（主题演化表）是 topic_based 任务跨节点传递的标配载体，它记录了从主题创建到当前时刻的所有关键状态变更。其核心结构：

```json
{
"topicId": "user123_roleplay_betrayal_mystery_20260708",
"topicName": "背叛与悬疑：失忆侦探",
"keywords": ["背叛", "悬疑", "侦探", "失忆"],
"intent": "roleplay",
"creatorUserId": "user123",
"createdAt": "2026-07-08T10:00:00Z",
"updatedAt": "2026-07-08T11:30:00Z",
"stateSnapshot": {
"status": "active | checkpoint | abandoned",
"summary": "当前主题状态的简要描述",
"keyDecisions": ["已确定角色A的动机是复仇", "已排除角色B是幕后黑手的可能性"],
"pendingExplorations": ["角色C的真实身份尚未揭示"]
},
"relatedTopics": [],
"events": [
{
"eventId": "evt_001",
"changeLevel": "major",
"nodeId": "ANALYZING",
"turnType": "complete",
"timestamp": "2026-07-08T10:00:00Z",
"stateChange": {
"before": "主题初始化",
"after": "识别为角色扮演任务，主题为'背叛与悬疑'，角色A为失忆侦探",
"delta": "创建主题，设定初始角色和基调"
},
"askingField": "theme_selection",
"modelOutputSummary": "用户想要一个关于背叛的悬疑故事，主角是失忆侦探"
}
]
}
```

**字段说明：**

- **topicId：** 全局唯一标识，格式为 `{userId}_{intent}_{topicSlug}_{date}`，由 ANALYZING 阶段 LLM 生成的 topicSlug + userId + intent + date 组合而成，支持同一用户在同一 intent 下有多个并行主题
- **topicName：** 简短标题，由 ANALYZING 阶段 LLM 自动生成，经用户确认后生效
- **keywords：** 3-5 个关键词，由 ANALYZING 阶段 LLM 自动生成，经用户确认后生效
- **stateSnapshot：** 当前主题状态快照，包含状态标记（active/checkpoint/abandoned）、关键决策摘要、待探索方向
- **relatedTopics：** 关联主题列表（详见 §2.4）
- **events：** 事件序列数组，每个节点在 exit 时由态控调度器自动追加一条 event

event 结构：

```json
{
"eventId": "evt_001",
"changeLevel": "major | minor | invalid",
"nodeId": "ANALYZING | IN_SESSION(roleplay) | EXECUTING",
"turnType": "complete | giveup | off-task",
"timestamp": "ISO 8601",
"stateChange": {
"before": "上一状态摘要",
"after": "当前状态摘要",
"delta": "本轮新增/修改/排除的内容"
},
"askingField": "本轮的主题锚点标签",
"modelOutputSummary": "模型产出的关键内容摘要（非全文）"
}
```

#### 2.3.2 changeLevel 分层留存机制

不是按时间新久留，是按信息净价值留。每个 event 在追加时被判定为 major/minor/invalid 三个层级：

| 层级 | 定义 | 示例 | 处理方式 |
|------|------|------|---------|
| major（大版本） | 影响核心走向的变更 | 新增角色、主线转折、世界观设定、topicEvolution 机制引入 | 永久保留，逐条可查，不压缩、不丢弃 |
| minor（小版本） | 对已有内容的改进调整 | 性格微调、情节润色、对话优化、tunable 参数新增 | 汇总压缩成一条摘要，超过 topic_evolution_minor_compaction_threshold（默认 5）条时触发，原事件迁入 topicEvolutionArchive 表 |
| invalid（无效过程） | 试了又放弃的试探 | 先加后删的设定、尝试又撤回的风格、短时间内对同一字段反复修改且最终回退 | 不入主表，迁入 topicEvolutionArchive 归档表，记录"因模型判定为互斥撤销而未入主表"的元数据 |

**判定时机与执行体解耦（防阻塞设计）：**

- 不能在节点 exit 的同步链路中插入一次独立的 LLM 判定调用
- 环节内模型在返回 turnType 和 collectedFields 时，顺带输出 changeLevel 预判字段——模型已经在推理过程中理解了本轮的语义变化，不需要额外的 LLM 调用
- 调度器收到后，直接使用该字段进行 event 追加

**模型返回的 turnType 结构更新：**

```json
{
"turnType": "complete",
"askingField": null,
"changeLevel": "major | minor | invalid",
"message": "...",
"collectedFields": {},
"result": {}
}
```

**兜底机制：**

- 若模型返回了合法的 changeLevel 值 → 态控调度器直接使用
- 若模型未返回该字段或格式错误 → 态控调度器默认标记为 minor（最安全的容错：进入累积压缩流，不丢失信息）

**判定标准的操作化定义（防模型偷懒与歧义）：**

模型不能凭模糊的"感觉"判定，必须在局部宪法的 @section output-schema 中注入明确的判定规则：

- **invalid 的严格定义：** 当且仅当本轮输出完全撤销了上一轮的实质性变更（如上一轮新增实体A，本轮删除实体A且无其他衍生变更），且未产生新的有效信息时，判定为 invalid
- **major 的严格定义：** 涉及核心实体的增删、任务目标的根本性转向、或引入了全新的世界观/规则设定时
- **minor 的兜底：** 凡不属于上述两者的情况，一律判定为 minor

**invalid 丢弃机制的逻辑闭环：** 当模型判定为 invalid 时，态控调度器不将该 event 写入 topicEvolution 主表。但这并不意味着"无痕消失"——为了满足全流程可审计性，态控调度器应将这条被标记为 invalid 的 event 写入 topicEvolutionArchive，并记录"因模型判定为互斥撤销而未入主表"的元数据。这确保了记忆不膨胀，同时审计有迹可循。

**判定边界的重要区分：** 用户反复修改同一属性（如"改林霜的性格，改了三次才满意"），只要方向不是完全互斥（加→删），就算 minor 累积调整，压缩后变成"林霜性格经过多轮调整为外冷内热"，不应该判定为 invalid。判定标准是**本轮操作是否实质上撤销了上一轮的变更且未产生新的有效信息**，而非修改次数。

**minor 压缩机制：** 当 minor 级 event 连续累计达到 topic_evolution_minor_compaction_threshold（默认 5 条，tunable 可配置）时，态控调度器触发一次汇总压缩：

- 调用轻量 LLM 将 N 条 minor event 的 delta 合并为 1 条 compressed_evt
- 原 N 条 minor event 从 topicEvolution 主表移除，迁入 topicEvolutionArchive 表
- compressed_evt 的 changeLevel 仍标记为 minor，modelOutputSummary 为合并后的摘要

压缩示例：

```
minor 汇总前：
evt_011: 林霜性格从冷漠改为外冷内热
evt_012: 林霜出场从第三章调到第五章
evt_013: 林霜的武器从剑改为扇子
evt_014: 林霜的背景故事从孤儿改为世家
evt_015: 林霜的对话风格从古风改为口语

minor 汇总后（1 条 compressed_evt）：
"林霜角色调整：性格外冷内热，出场第五章，武器扇子，背景世家，对话风格口语"
```

**event 的 stateChange 生成机制：** stateChange 摘要压缩在 N10 后台异步进程执行，与 changeLevel 保持一致——均不在态控调度器的同步链路里调 LLM。态控调度器在节点 exit 时只落盘 topicEvolution event 原始数据（collectedFields + message + result），stateChange 摘要由 N10 后台异步消费 ruleMiningQueue 时生成。

#### 2.3.3 topicEvolution 注入时的分层结构

上下文管理器 在 topic_based 模式初始化时，从 contractStore 读取 topicEvolution 注入到上下文：

```
[topicEvolution 注入包]
├── majorEvents（永久保留，全部注入，通常 0-5 条，不压缩不丢弃）
├── compressedMinorSummary（最近一次 minor 压缩摘要，1 段）
├── recentMinorEvents（压缩后新增的 minor 事件，通常 0-4 条）
└── currentStateSnapshot（覆盖写，1 条）
```

这个分层注入结构确保了模型在每个节点开始时的上下文窗口不会因 topicEvolution 的膨胀而无限增长，同时 major 的关键决策永远可查。

#### 2.3.4 topicEvolution 的"完成"与"放弃"语义

topicEvolution 的 stateSnapshot 有三个状态：

- **active：** 讨论中，主题仍在进行
- **checkpoint：** 完成了一个输出（如写完一首诗），但主题仍可继续。标记 checkpoint 时，态控调度器将主题信息推入规则提炼的待处理队列
- **abandoned：** 用户放弃。标记 abandoned 时，态控调度器同样将主题信息推入规则提炼的待处理队列

当模型返回 turnType="complete" 时，态控调度器将 stateSnapshot 标记为 checkpoint。当模型返回 turnType="giveup" 时，标记为 abandoned。

#### 2.3.5 topicEvolution 的自动落地机制

在状态枚举 + taskType 固化阶段（N3），当 taskType 被固化为 topic_based 时，态控调度器自动执行：

1. **初始化：** 在 contractStore 中初始化一条 topicEvolution 记录，topicId 由 ANALYZING 阶段 LLM 生成的 topicSlug + userId + intent 组合而成
2. **写入：** 在 上下文管理器 的 topic_based 模式中，强制开启 topicEvolution 写入（不需要 N9 声明，因为 taskType 本身就是声明）
3. **追加：** 每个涉及该 topic 的节点（ANALYZING、IN_SESSION、EXECUTING）在 exit 时，态控调度器自动追加一条 event 到 topicEvolution——不需要节点开发者手动写代码，态控调度器在收 turnType 时自动完成，并使用模型顺带输出的 changeLevel 预判字段

#### 2.3.6 topicEvolution 的跨会话匹配

topicEvolution 的 topicId 需要一个跨会话的匹配机制：

- 用户说"继续上次那个故事" → ANALYZING 识别出 topicId（从 contractStore 里按用户 + 主题相似度匹配）
- 匹配到已有的 topicEvolution → 注入最新 stateSnapshot + 分层注入包，模型接着走
- 匹配不到 → 新建 topicEvolution，从零开始
- 匹配逻辑放在 topicIndex 的检索中——先用 DET 关键词匹配（基于 topicIndex 表的 topicName 和 keywords），关键词匹配不到时降级为 embedding 语义匹配
- 匹配阈值由 tunable 参数 topic_evolution_match_threshold 控制（默认 0.65，略高于 topic_anchor_similarity_threshold 的默认 0.6，因为跨会话匹配需要更高精度）

#### 2.3.7 topicEvolution 的存储位置

topicEvolution 存入 contractStore，而不是 sessionStore。因为 topicEvolution 是跨会话持久化的长期记忆载体，它的生命周期与 contractStore 一致——Agent 消亡后，主题演化记录仍可被后续会话访问。物理存储用 SQLite + WAL 模式（与现有 contractStore 一致），包含四张表：

- **topicEvolution：** 记录 topicId、stateSnapshot、创建时间、更新时间，只存储 major 级 event + compressedMinor + 压缩后新增的 minor 级 event
- **topicEvolutionEvent：** event 主表，按 topicId 外键关联
- **topicEvolutionArchive：** event 归档表，存储被压缩的 minor 级原始 event + 被判定为 invalid 的 event
- **topicIndex：** 检索索引表（详见 §2.6）

#### 2.3.8 topicEvolution 与认知局部性的关系

topicEvolution 不是对认知局部性的违反，而是对它的正确扩展。

field_based 的 contract-out JSON 同样是跨节点共享的——节点之间不共享的是"推理过程"，但共享"收敛结果"。topicEvolution 的 event 序列就是 topic_based 的"收敛结果"——它记录的不是模型的推理过程，而是"主题在这个节点里完成了什么变化"这一确定性的状态变更。

节点之间的数据传递，不再仅限于"上一个节点的结构化输出"，而是"这个主题从开始到现在的演化历史摘要（按信息净价值分层）"。这仍然是"数据流为中心"——只是数据流的粒度从"节点级"提升到了"主题级"。

该原则同样适用于第二层（主题关联）和第三层（规则提炼）：关联摘要不是"把隔壁主题的数据拿过来"，而是"从隔壁主题的数据里提炼出对当前主题有用的几条结论"；领域规则不是硬约束，而是软参考。关联主题的原始 evt 链不属于当前主题的最小充分条件集，不注入。

#### 2.3.9 分层留存与文档自身演进的对应

这套分层逻辑同时解决了"态控闭环设计流程指南"自身演变过程数据的管理问题：

| 层级 | 文档演进示例 | 处理 |
|------|------------|------|
| major | v3.2→v4.0（引入 taskType、topicEvolution） | 永久保留 |
| minor | v4.0→v4.1（新增 topic_evolution_match_threshold） | 汇总压缩："v4.1 新增了跨会话匹配阈值参数" |
| invalid | "试了用 Redis 存 contractStore，后来改 SQLite" | 直接丢弃，不留 |

### 2.4 第二层：主题关联机制

**核心原则：** 关联在元数据层，隔离在数据层——模型只知道"隔壁主题给了我两条结论"，不知道"隔壁主题的50条聊天记录"。关联不是混同，关联是轻量级元数据引用。

#### 2.4.1 关联类型定义

| 关联类型 | 含义 | 示例 | 注入内容 |
|---------|------|------|---------|
| prerequisite（前提） | A是B的前提基础 | 专利A（网关）→ 专利B（区块链存证） | "A的核心创新点是协议转换效率提升30%" |
| reference（引用） | B引用了A的部分内容 | 专利C引用了专利A的网关架构 | "A的网关架构为YYY" |
| derivative（派生） | B是从A派生出来的 | 专利B是专利A的应用场景扩展 | "B是在A基础上的特殊化应用" |
| conflict（冲突） | A与B的方案互斥 | 专利A和专利D对同一问题给出不同解法 | "A和D的解法不同：A采用X，D采用Y" |
| cooperative（协同） | A与B共同构成完整方案 | 专利A负责硬件，专利E负责软件 | "A和E共同构成完整系统" |
| replacement（替代） | B替代了A，因为A被否定 | 现代风格方案→新中式风格方案 | "A被否定原因；B要继承A的哪些优点；B要避免A的哪些缺点" |
| variant（变体） | B是A的另一种版本，并存可选 | 方案A（预算版）vs 方案B（豪华版） | "A和B的差异点；A和B的共同基础" |
| evolution（演进） | B是A的迭代升级 | v1方案→v2方案（基于同一客户反馈优化） | "v1的反馈要点；v2的优化方向" |

#### 2.4.2 关联的建立方式

- **手动建立：** 用户在对话中说"这个专利是基于之前那个边缘计算网关的"，系统提示"检测到可能与「边缘计算网关」主题相关，是否建立关联？"，用户确认后生效
- **自动发现：** ANALYZING 阶段，LLM 检测到当前输入与已有主题的关键词重叠度高，提示"检测到可能与「XXX」主题相关，是否查看或建立关联？"
- **关联确认：** 所有关联关系（类型、摘要内容）必须经过用户确认后才能生效——用户可修改关联类型、编辑摘要、或拒绝建立关联

#### 2.4.3 关联数据的存储与注入

关联关系存储在 topicIndex 表的 relatedTopics 字段中，是轻量级元数据。注入时，上下文管理器 只拉取关联摘要卡片，不拉取关联主题的原始 evt 链：

```json
{
"currentTopic": {
"topicId": "scheme_B",
"topicName": "新中式风格全屋方案"
},
"relatedTopics": [
{
"topicId": "scheme_A",
"topicName": "现代风格全屋方案",
"relationType": "replacement",
"reason": "客户反馈现代风格色调太冷，希望更温馨",
"inheritedPoints": ["厨房动线设计获得客户认可", "主卧衣帽间尺寸已确认"],
"avoidPoints": ["电视墙石材材质被客户否定", "全屋无主灯设计客户不习惯"]
}
]
}
```

模型拿到这个，写B的时候就知道：要继承厨房动线和主卧尺寸，要避免石材电视墙和无主灯，要回应"太冷"的问题用暖木色系。

#### 2.4.4 关联与认知局部性的关系

关联摘要不是"把隔壁主题的数据拿过来"，而是"从隔壁主题的数据里提炼出对当前主题有用的几条结论"。关联主题的原始 evt 链不属于当前主题的最小充分条件集，不注入。关联摘要 = 隔壁主题的"毕业证书"，不是隔壁主题的"日记本"。

### 2.5 第三层：异步规则提炼机制

**核心原则：** 规则提炼不从原文挖掘，而是从 topicEvolution 的演变流程和主题关联的 inheritedPoints/avoidPoints 中提取"被多次验证的行为模式"。候选规则必须经过用户确认后才生效。规则提炼完全异步，不在状态机流转链路内执行。

#### 2.5.1 提炼的输入源

| 输入源 | 内容 | 示例 |
|-------|------|------|
| 子主题的演变流程（topicEvolution） | 每次 stateChange 的 delta | "厨房动线从'随意'改为'洗切炒'" |
| 主题间的关联机制 | 关联关系中的 inheritedPoints 和 avoidPoints | "现代风格方案的厨房动线被继承到新中式方案" |

系统不做全文挖掘，只从这两个结构化数据源中提取"被多次验证的行为模式"。

#### 2.5.2 提炼触发机制

**触发方式（满足任一条件即触发）：**

- **时间窗口触发：** 每日固定时间（如凌晨2点）统一提炼。由 rule_mining_schedule tunable 配置（cron 表达式，默认 `0 2 * * *`）
- **数量阈值触发：** 每积累 N 个同类主题的 checkpoint 触发一次提炼。由 rule_mining_batch_threshold tunable 配置（默认 5）
- **手动触发：** 用户在系统设置界面的"知识库更新状态"面板中点击"立即提炼"

**态控调度器在节点 exit 时只做：** 落盘 topicEvolution event → 标记主题状态（checkpoint/abandoned）→ 放行。不触发任何提炼逻辑。

**提炼引擎作为独立的后台进程运行：**

- 从 contractStore 中读取待提炼的主题列表
- 每次批量提炼的最大主题数由 rule_mining_batch_size tunable 控制（默认 10）
- 批量处理，写入 ruleCandidates 池
- 等待交叉验证通过后提交用户确认

**知识真空期的用户约定：** 在系统设置界面增加"知识库更新状态"指示器，显示：

- "上次提炼时间：2026-07-08 02:00"
- "待提炼主题数：3"
- "下次计划提炼时间：2026-07-09 02:00"
- 提供"立即提炼"手动触发按钮

这样用户知道"我的经验正在被后台消化"，不会误以为每次任务结束系统就立刻变聪明了。

#### 2.5.3 提炼与确认流程

提炼引擎触发后：

1. **候选规则生成：** 提炼引擎读取待提炼主题的 topicEvolution 和关联关系，提取"稳定的、可复用的行为模式"，生成候选规则，存入 ruleCandidate 池
2. **交叉验证：** 候选规则需要多条同类主题交叉验证后才提交用户确认。默认阈值为 70%（由 domain_rule_validation_threshold tunable 控制，默认 0.7）。例如：5 套方案中 4 套都验证了"厨房应遵循洗切炒动线"→ 验证通过率 80% → 提交用户确认
3. **用户确认：** 系统展示候选规则及证据（"基于8套方案中7套的演变记录"），用户可采纳、修改或拒绝。确认后的规则存入 contractStore.domainRules 库
4. **规则注入：** 同类型新主题初始化时，上下文管理器 自动注入已生效的相关领域规则

规则确认示例：

```
系统："检测到一个可能的通用规则：
厨房布局应遵循洗切炒动线
（基于5套方案中4套的演变记录）
是否采纳为通用规则？"
用户：[采纳] → 存入 domainRule 库，标记为 strong
用户：[修改] → "改成'厨房布局应遵循洗切炒动线，U型布局优先'"
用户：[拒绝] → "这只是巧合，我家厨房小不适合U型"
```

#### 2.5.4 规则的注入

当新的同类型主题初始化时，上下文管理器 除了注入 topicEvolution，还会注入该领域的已生效 domainRule：

```json
{
"domainRules": [
{ "rule": "厨房布局应遵循洗切炒动线，U型布局优先", "source": "已验证于8套方案", "level": "strong", "confirmedBy": "user_001" },
{ "rule": "衣柜应遵循挂叠分区，挂区占比不低于60%", "source": "已验证于5套方案", "level": "medium", "confirmedBy": "user_001" },
{ "rule": "主卧应预留婴儿床位置", "source": "已验证于3套方案", "level": "suggestion", "confirmedBy": "user_001" }
]
}
```

领域规则不是硬约束（不由 DET 强制执行），而是软参考——模型可基于客户个性化需求合理偏离。

#### 2.5.5 规则提炼与主题关联的区别

| 维度 | 主题关联 | 异步规则提炼 |
|------|---------|------------|
| 作用范围 | 两个主题之间 | 一类主题之上 |
| 数据来源 | 单条主题的 topicEvolution | 多条同类主题的交叉验证 |
| 产出 | 关联关系（A是B的前提） | 通用规则（所有厨房都应遵循洗切炒） |
| 注入方式 | 当前主题初始化时按需注入 | 同类型主题初始化时自动注入 |
| 时效性 | 即时（关联建立后立即生效） | 滞后（异步批量提炼，存在知识真空期） |

#### 2.5.6 领域规则与宪法体系的关系

| 层级 | 来源 | 是否可修改 | 确认方式 |
|------|------|-----------|---------|
| 根宪法 | 设计者写死 | 不可修改 | 无（设计时定） |
| 架构机制 | 设计者编写 | 走回归测试修改 | 无（设计时定） |
| 局部宪法 | 设计者编写或动态生成 | 可修改 | 低置信度时用户确认 |
| 领域规则 | 从 topicEvolution + 关联机制提炼 | 用户确认后生效，可随时修改或废弃 | 必须用户确认 |

领域规则是唯一一层由用户在生产过程中逐步建立的约束——它不是设计者拍脑袋定的，是从实际项目中长出来的。
#### 2.5.7 领域规则的来源与权限分层（v7.1 新增）

**规则来源三层：**

| 来源 | 含义 | 灌入时机 | 权限初始值 | 是否走交叉验证 |
|------|------|---------|-----------|-------------|
| **scenario_seed** | 场景固有规则（工艺规范、设计基线、报价模板、安全红线等） | 主题初始化时按 topicPath 加载 | L0 / L1 | ❌ 不走，设计者预设可信 |
| **user_initiated** | 主题确认时用户主动口述的规则 | ANALYZING 确认 topicName 后主动问询 | L2 | ✅ 走，用户可能说错或矛盾 |
| **model_inferred** | IN_SESSION 过程中模型推断的用户偏好 | complete 时顺带输出 → ruleCandidates | L3 | ✅ 需用户确认升格为 L2 |

**规则权限分层（与宪法层级正交）：**

| 层级 | 名称 | 含义 | 可修改 | 冲突时行为 |
|------|------|------|--------|----------|
| **L0** | 钢印规则 | 场景硬性约束，不可触碰 | ❌ 不可修改 | 无条件胜出，不询问 |
| **L1** | 基线规则 | 设计者预设最佳实践 | ⚠️ 需设计者确认 | 与 L2/L3 冲突时触发询问 |
| **L2** | 用户确认规则 | 用户主动说过或确认过的偏好 | ✅ 用户可随时修改 | 优先于 L3 |
| **L3** | 模型推断规则 | 模型 infer 的候选规则 | ✅ 需用户确认升格 | 仅作参考，不主动注入 |

与根宪法的关系：根宪法第 1 条（安全红线）在领域规则层映射为 L0 钢印规则。

#### 2.5.8 规则网络结构（v7.1 新增）

领域规则不再是平铺的列表，而是一个有向图——每条规则是一个节点，节点之间通过关联边建立关系。

**节点定义：**

```json
{
  "ruleId": "kitchen_countertop_height_001",
  "content": "厨房台面高度建议85cm",
  "topicPath": ["全屋定制", "设计", "厨房设计"],
  "source": "scenario_seed",
  "immutableLevel": "L1",
  "status": "active",
  "conditions": { "userHeight": ">=170cm" },
  "edges": {
    "conflicts_with": ["kitchen_countertop_height_002"],
    "depends_on": [],
    "strengthens": ["ergonomic_guideline_003"]
  },
  "version": 1
}
```

**关联边类型：**

| 边类型 | 含义 | 注入时行为 |
|--------|------|----------|
| **depends_on** | A 规则成立的前提是 B 规则已满足 | 匹配 A 时检查 B，若不满足则主动询问 |
| **conflicts_with** | A 和 B 互斥，不能同时应用 | 同时匹配到时触发询问让用户二选一 |
| **strengthens** | A 规则强化 B 规则的可信度 | 匹配到 A 时，B 的权重 +1（累加上限 +3），同层级内优先注入 |

**规则状态：** active（正式生效）/ candidate（待交叉验证）/ archived（已废弃）/ pending_review（L0/L1 修改请求）。

#### 2.5.9 主题层级与规则的作用域边界（v7.1 新增）

topicPath 是一个**层级化数组**，从粗到细逐级展开。每一层都可以有自己独立的规则集。

**层级结构示例：**

```
全屋定制
├── 设计
│   ├── 厨房设计     ← 规则："台面高度85cm""水槽位置""油烟机型号偏好"
│   ├── 卫生间设计   ← 规则："防水等级""瓷砖规格""干湿分离要求"
│   └── 客厅设计
├── 报价             ← 规则："利润率""套餐模板""折扣权限"
├── 客户沟通
├── 拆单
└── 生产
```

**前缀匹配规则**：上级可见于下级，下级不可见于上级/兄弟。层级越深，匹配到的规则集越小——认知局部性原则在规则注入层的自然体现。

| 当前 topicPath | 能匹配到哪些规则 | 不能匹配到 |
|--------------|---------------|----------|
| `["全屋定制"]` | topicPath 前缀为 `["全屋定制"]` 的所有规则——包括"设计""报价""生产"等全部子层规则 | — |
| `["全屋定制", "设计"]` | 前缀为 `["全屋定制", "设计"]` 的规则——"厨房设计""卫生间设计""客厅设计" | `["全屋定制", "报价"]` 的规则（兄弟节点不可见） |
| `["全屋定制", "设计", "厨房设计"]` | 仅 `["全屋定制", "设计", "厨房设计"]` 自身的规则 | "卫生间设计""报价"等的规则（隔层不可见） |
| `["全屋定制", "报价"]` | 仅 `["全屋定制", "报价"]` 自身的规则 | "厨房设计"等的规则 |

**核心纪律**：
- **上级可见于下级**：`["全屋定制", "设计"]` 层的规则对 `["全屋定制", "设计", "厨房设计"]` 可见——前者是后者的前缀。这意味着在"厨房设计"环节，模型能拿到"设计"层的通用规则（如"全屋风格统一为现代简约"）+ "厨房设计"层的专用规则（如"台面高度85cm"）
- **下级不可见于上级/兄弟**：厨房台面高度对报价毫无意义——`["全屋定制", "设计", "厨房设计"]` 的规则不会注入到 `["全屋定制", "设计", "卫生间设计"]` 或 `["全屋定制", "报价"]` 的上下文中
- **层级深度与上下文窗口的关系**：当前 topicPath 越深，匹配到的规则集越小

#### 2.5.10 规则注入匹配逻辑（v7.1 新增，上下文管理器 第0层改造）

**v7.0 原逻辑：** 全量读取 domainRules → 一股脑注入

**v7.1 新逻辑：** 按 topicPath 做子图匹配 → 精准注入

1. 获取当前 topicPath（如 ["全屋定制", "设计", "厨房设计"]）
2. 在 domainRules 中做子图匹配：匹配 topicPath 前缀一致的规则节点 + 沿 depends_on/strengthens 边外扩一层 + 检查 conditions
3. 冲突检测：若匹配到 conflicts_with 关系的节点 → 触发询问，按 immutableLevel 决定询问策略
4. 条件不满足检测：未满足的 conditions → 主动询问补充
5. 注入 prompt：按 L0>L1>L2>L3 → 同层级内按 importance（critical>high>normal>low）→ 同 importance 按 strengthens 权重排序（累加上限+3，v4.7）→ 同权重按 ruleId 字典序

#### 2.5.11 规则提炼队列处理（v7.1 新增）

```
checkpoint / abandoned
  ▼
推入 ruleMiningQueue
  ▼
N10 后台进程消费 → 压缩摘要 → 提取候选规则 → 推断关联边 → 写入 ruleCandidates
  ▼
等待用户确认
  ├── 确认 → 升格为 L2，写入 domainRules
  ├── 修改 → 按用户修改后写入，标记 L2
  └── 否决 → 丢弃或标记 rejected
```

**交叉验证策略按来源区分：** scenario_seed（L0/L1）不走交叉验证；user_initiated（L2）需交叉验证；model_inferred（L3）需交叉验证+用户确认升格。

**inferred_conflict 候选状态：** N10 自动推断的 conflicts_with 关联边标记为 `inferred_conflict`（候选状态），不参与运行时冲突检测。必须经用户确认升格为 `active_conflict` 后才生效。详见总纲 v5.5 §3.7.7。



### 2.6 topicIndex 索引表

为支持高效检索和主题关联，每个 topicEvolution 记录在初始化时同步写入 contractStore.topicIndex 表，作为检索索引。

#### 2.6.1 表结构

```sql
CREATE TABLE topicIndex (
topicId TEXT PRIMARY KEY,
topicName TEXT NOT NULL,           -- 简短标题（如"智能体态控架构专利交底书"）
keywords TEXT NOT NULL,            -- 3-5个关键词，逗号分隔
creatorUserId TEXT NOT NULL,
createdAt TEXT NOT NULL,
lastActiveAt TEXT NOT NULL,
relatedTopics TEXT DEFAULT '[]'    -- JSON数组，存储关联关系
);
```

#### 2.6.2 主题名 + 关键词的用户确认流程

**核心原则：** 主题名和关键词由 ANALYZING 阶段 LLM 自动生成，但必须经用户确认后方可生效。

**匹配到已有主题时：**

```
用户："继续写那个态控的专利"
↓
ANALYZING 匹配到已有 topicEvolution（关键词匹配命中）
↓
系统展示："检测到已有主题「智能体态控架构专利交底书」，是否继续？"
用户：[确认] → 直接续写
用户：[不对，是另一个] → 系统展示相近候选项让用户重选
用户：[不是，我要写一个新的] → 重新初始化
```

**未匹配到已有主题时（新建主题）：**

```
用户："帮我写一个关于边缘计算的专利"
↓
ANALYZING 关键词匹配不到已有记录
↓
LLM 生成候选 topicName + keywords
↓
系统展示：
"已为您新建主题：
主题名：边缘计算网关专利交底书
关键词：边缘计算、网关、协议转换、物联网
是否正确？"
用户：[确认] → 初始化 topicEvolution，进入 IN_SESSION
用户：[修改] → 用户手动编辑主题名或关键词
用户：[不对，我要找的是另一个] → 系统展示相近候选项
```

#### 2.6.3 检索优先级

1. **关键词匹配（DET）：** 用户输入含明确主题名或关键词 → DET 直接匹配 topicName + keywords + userId + 时间范围，0 LLM 成本，毫秒级响应
2. **embedding 语义匹配（LLM）：** 用户表述模糊（"继续上次那个"），关键词匹配不到时降级为 embedding 语义匹配，阈值由 topic_evolution_match_threshold（默认 0.65）控制

**匹配置信度与用户确认的关系：**

- 匹配命中且置信度 ≥ 0.65 → 直接展示匹配到的主题名，用户一键确认
- 匹配命中但置信度 0.5-0.65 → 展示相近候选项（最多3个），用户选择或否定
- 未命中（置信度 < 0.5）→ 展示新生成的候选主题名+关键词，用户确认、修改或手动搜索

### 2.7 上下文管理器 双模式锚点

上下文管理器 的 buildPromptContext() 接口根据当前环节的 taskType 执行不同的锚点策略：

**field_based 模式——字段级硬门控：**

- 保护逻辑：与当前 askingField 同名的历史轮次（即之前追问同一字段时的对话）不被淘汰
- 执行方式：确定性字符串匹配（DET 执行），askingField="amount" 时保护所有 turnHistory 中 askingField="amount" 的轮次
- 保护形态：至少保留为摘要或锚点，不因上下文窗口不足而移除
- 可靠性：高（确定性匹配，不依赖 LLM 判断）

**topic_based 模式——主题级硬门控：**

- 保护逻辑：与当前 askingField 主题标签语义相关的历史轮次不被淘汰
- **三层注入——** 在 DET 关键词粗筛之前，上下文管理器 按优先级注入：
  - 第0层：已生效的领域规则注入——**v7.1：从全量注入改为按 topicPath 子图匹配精准注入**（详见 §2.5.10）。从 contractStore.domainRules 做子图匹配（匹配 topicPath 前缀一致的规则节点 + 沿 depends_on/strengthens 边外扩一层 + 检查 conditions），按 L0>L1>L2>L3 优先级排序后注入。domainRules 中的数据由 N10 规则提炼的后台进程异步写入。如果 domainRules 为空（知识真空期），跳过第0层注入，不影响后续流程
- 第1层：关联主题摘要注入——从 topicIndex.relatedTopics 读取关联主题摘要卡片（inheritedPoints / avoidPoints / reason）
- 第2层：当前主题演化注入——从 contractStore.topicEvolution 读取分层注入包（majorEvents + compressedMinorSummary + recentMinorEvents + currentStateSnapshot）
- 注入优先级：领域规则 > 关联摘要 > topicEvolution 分层包 > turnHistory
- 执行方式（三层注入 + 两阶段检索，0 次 LLM 调用）：
- 阶段 0：领域规则 + 关联摘要 + topicEvolution 分层注入——确保模型在开始推理前，已经获得了当前领域已沉淀的最佳实践、关联主题的关键结论和当前主题的演化历史
- 阶段 1：DET 关键词粗筛（确定性，O(N)）——对 turnHistory 中每轮，DET 提取 askingField 主题标签的关键词（如 "character_motivation" 提取 "character"、"motivation"），与历史轮次的文本做关键词重叠匹配。重叠度 ≥ topic_keyword_overlap_threshold（tunable，默认 0.2）的轮次进入阶段 2
- 阶段 2：embedding 相似度精筛（embedding 模型调用，非 LLM 调用）——对阶段 1 粗筛通过的轮次，用 embedding 模型计算 askingField 主题标签与历史轮次文本的向量相似度（余弦相似度）。相似度 ≥ topic_anchor_similarity_threshold（tunable，默认 0.6）的轮次被保护
- DET 阈值裁决：DET 只做"相似度 ≥ topic_anchor_similarity_threshold"的最终数值比较，确定性裁决
- 保护形态：至少保留为摘要或锚点，不因上下文窗口不足而移除
- 可靠性：中高（embedding 模型可能漏判跨主题依赖，但比纯 LLM 判断更稳定、成本更低；风险等级：中——漏判后果是模型重复追问已讨论过的主题，不影响输出正确性，影响交互效率）
- 工程收益：LLM 调用从 O(turnHistory) 降到 0，embedding 调用从 O(turnHistory) 降到 O(粗筛通过数)；三层注入使模型在新主题启动时就能获得跨主题的知识积累

**两种模式的共同部分：** 综合匹配度评分（其余历史轮次按语义匹配度分配形态）、最低匹配度截断（min_match_score tunable）、偏离处理、（archiveGraph）、校验失败标记（updateGraph）——这些机制两种模式完全相同。

**上下文管理器 三个接口说明：**

| 接口 | 时机 | 职责 | 状态机传入 |
|------|------|------|----------|
| buildPromptContext() | 每轮发消息给模型前（读） | 按三层注入+硬门控+语义匹配+最低截断算出本轮应拼入的上下文片段 | contextGraph, turnHistory, currentAskField, currentTaskType, currentTopicId, currentStepName（field_based 按 stepName 匹配 contractEvolution 规则）, contractStore, tunableParams |
| updateGraph() | 每轮模型返回后（写） | 更新逻辑拼接图，记录本轮对话和校验结果 | userInput, modelTurn, turnType, askingField, changeLevel, validationResult |
| archiveGraph() | 偏离/放弃/完成时（归档） | 归档当前上下文图，新建空图 | — |

**接口职责边界**：调度器的合同是"告诉 上下文管理器 当前在干什么（taskType/topicId）+ 给什么资源（contractStore/tunableParams）"，上下文管理器 的合同是"基于这些信息拼出最优上下文"。调度器不预查询 domainRules/topicEvolution 等数据，不在接口层承担数据组装职责——这确保了"调度器只调接口、不内联拼接逻辑"的纪律在参数层面也成立。

**askingField null 降级规则（两种模式通用）：** 当 askingField 为 null 时（如 reply 轮次、complete 轮次），硬门控保护降级为纯语义匹配。field_based 模式不执行字段名匹配保护，topic_based 模式不执行主题标签匹配保护，均改为 上下文管理器 按综合匹配度评分分配形态。降级不报错不阻断。

### 2.8 taskType 对降级链的影响

四项降级链在两种模式下的执行差异（v8.5：移除跨任务延伸检测）：

| 降级项 | field_based 模式 | topic_based 模式 |
|-------|-----------------|-----------------|
| L1 结构校验 | 相同（JSON.parse + Schema 校验） | 相同 |
| DET 值域复验 | 值域校验（金额上限、时间格式、必填字段…） | 输出格式校验（JSON Schema / 文本格式） |
| logprobs 检查 | 相同（概率判断来自 logprobs + pattern 扫词表） | 相同 |
| 硬编码兜底 | 相同 | 相同 |

> **v8.5 修正：移除跨任务延伸检测**——跨任务是用户显式确认行为，不需要降级链层自动化检测。详见 §6.1 off-task 双重保障 + M1 switch 口令。

### 2.9 五步流水线统一操作模型（v7.2 新增）

两种 taskType 遵循相同的五步环节操作流水线（详见总纲 v5.5 §3.9）：**状态机路由 → 环节上下文加载 → 内容拼接 → 模型计算 → 记录落盘**。接口统一（路由信号→加载→拼接→计算→落盘），内部实现分支（步骤一匹配模式 DET vs embedding、步骤二加载源 sessionCheckpoint vs topicEvolution、步骤三锚点模式字段硬门控 vs 三层注入+两阶段检索、步骤四校验方式值域 vs 格式、步骤五落盘目标 sessionCheckpoint vs topicEvolution event）。核心纪律：态控调度器只管路由和记录，上下文管理器只管拼接，环节宪法只管模型行为规则。

### 2.10 环节内规则与跨主题规则的优先级仲裁（v7.2 新增）

domainRules 表中同时存在两类规则——环节内规则（contractEvolution，来自单环节 changeLog 提炼）和跨主题规则（topicEvolution，来自多主题交叉验证）。冲突仲裁规则（详见总纲 v5.5 §3.2）：

| 维度 | 规则 |
|------|------|
| 优先级 | 环节内规则 L2 > 跨主题规则 L2（同优先级环节内优先，更具体） |
| 冲突 | 不允许自动覆盖，标记冲突进入人工确认队列 |
| 共存 | 不冲突时可共存，注入时按优先级排序 |

### 2.11 实现形态声明（v7.2 新增）

各关键组件的架构固定属性与实现层可变属性（详见总纲 v5.5 §2.7）：

| 组件 | 架构固定 | 实现可变 |
|------|---------|---------|
| 态控调度器 | DET——路由+全量记录+调配 | 路由表结构、状态转移实现 |

### 2.14 编程规则宪法（v8.0 新增，v8.1 同步 v5.4/v5.5 扩展为 7 条）

**定位**：编程规则宪法是母本的一部分，在整个 N1→N15 管线全局生效，不是某个节点产出的。任何涉及代码生成的环节（N13）必须在生成代码前校验此宪法，违反任一条款拒绝生成。

**约束对象**：代码模型（生成骨架代码的 AI），非运行时模型。

**七条全局元约束**：

| 编号 | 条款 | 内容 |
|:--:|------|------|
| 1 | **执行体分派铁律** | 每个状态的执行体由 N3 决定，代码模型不得自行更改。ANALYZING=LLM，IN_SESSION=LLM（可调用 DET 工具），DET 复验=DET，EXECUTING=DET（仅纯写入/删除）。不得将 ANALYZING/IN_SESSION 降级为正则或 parser |
| 2 | **DET 职责边界** | DET 只做**确定性计算**：元指令 EXACT_MATCH + 值域/格式复验 + 关键词扫描标记 + logprobs 阈值裁决 + 路由矩阵裁决 + 规则子图匹配与冲突检测 + 文件操作 + SQL 查询（作为 IN_SESSION 的工具被调用）。不扩展到意图识别、字段采集、查询理解、语义判断 |
| 3 | **LLM 职责边界** | LLM 做：意图识别 + 字段采集/追问/偏离判断/查询理解 + 语义边界判断 + 工具调用决策 + 结果格式化（自然语言生成）。不得用正则/parser 替代 |
| 4 | **降级规则** | 无 apiKey 时 ANALYZING/IN_SESSION 必须拒绝服务并明确告知原因，不得以任何形式降级为正则/DET 替代。降级模式如存在必须标注"降级非正道" |
| 5 | **契约对齐** | 代码必须严格对齐 N11 契约（contract-in/out schema），不自创接口 |
| 6 | **tunable/接口隔离** | 阈值从 getTunable() 读，调度器只调 context-manager 三接口（匹配/决策/拼接），不内联拼接逻辑。上下文管理器三层职责各自独立，不得合并 |
| 7 | **工具分层发现（v5.5 新增）** | 工具分三层：必用工具（环节宪法 @section tools 声明，完整定义自动注入）/ 工具清单（@section tool_catalog，轻量级目录每次注入）/ 选用工具（系统级工具库 toolRegistry，完整定义按需加载）。模型从清单选择→调度器加载完整定义→模型调用。LLM 不直接写 SQL，通过 DET 工具间接操作数据库。search_tools meta-tool 始终可用 |

**生效时机**：设计者在 P0 认知加载时即需理解，N13 代码模型强制输入。


### 2.15 IN_SESSION tool calling 机制总览（v8.1 新增，v5.4/v5.5）

v5.4 起，IN_SESSION 环节内的 LLM 可通过 function calling 调用 DET 工具，用于查询/计算等需要数据库或确定性操作的 intent。

**核心纪律**：
- **IN_SESSION 可调用 DET 工具**——查询、对比、计算等操作在 IN_SESSION 内通过工具调用完成
- **EXECUTING 仅保留纯写入/删除**——record INSERT / delete DELETE，不再承载查询逻辑
- **LLM 不直接写 SQL**——通过 DET 工具间接操作数据库

**工具分层发现（v5.5 三层体系）**：

| 层 | 名称 | 注入方式 | 内容 |
|----|------|---------|------|
| 必用工具 | @section tools | 完整定义自动注入每轮 prompt | 环节必须的 DET 工具（如记账环节的 queryLedger/insertRecord） |
| 工具清单 | @section tool_catalog | 轻量级目录每次注入 | 可选工具的摘要列表（名称+一句话描述），供模型选择 |
| 选用工具 | toolRegistry 表 | 完整定义按需加载 | 系统级工具库，模型从清单选择→调度器加载完整定义→模型调用 |

### 2.16 项目隔离机制总览（v8.1 新增，v5.6）

v5.6 引入项目隔离机制，解决同一环节下不同智能体项目产出物隔离问题。

**核心数据结构**：
- **currentProjectId**：态控调度器全局字段，一处变更所有房间联动
- **projectRegistry 表**：`{projectId, projectName, projectKeywords, taskType, ...}`
- **userLastProject 表**：`{userId, lastProjectId}`——用户唤醒时自动加载默认项目

**项目切换（S3 物理隔离）**：
```
用户说"切换到杂碎本" → ANALYZING 关键词匹配 projectRegistry.projectKeywords
→ S3 四步：释放当前项目 KV Cache → 归档 → 加载新项目最后状态 → 创建新会话
→ 更新 currentProjectId + userLastProject → 全局联动
```

**项目切换 vs 房间切换**：
| 维度 | 房间切换（switch） | 项目切换 |
|------|-------------------|---------|
| 触发 | "切断房间"口令 | 用户说项目名 |
| roomId | ✅ 变 | ❌ 不变 |
| projectId | ❌ 不变 | ✅ 变 |
| 全局联动 | 不联动 | ✅ 联动 |

**流程中的影响**：
- N1（场景定义）：需声明项目级别的产出物隔离需求
- §2.2.5 schedulerState：补 currentProjectId + projectRegistry + userLastProject
- §十五 contractStore：补 projectRegistry + userLastProject 表


### 2.17 文件管理规范总览（v8.1 新增，v5.6）

v5.6 引入文件管理的"房间比喻四层结构"和三个机制：

```
房间（环节，N1-N16）—— 环节宪法（房间级，共享）
  └── 项目（智能体实例）—— 产出物内容（项目级，隔离）
       └── 格子（细则文件）—— 正文内容（文件级，独立）
            └── 变更日志（独立文件）—— 所有变更追溯（全局统一）
```

**机制1 项目隔离**：见 §2.16
**机制2 分格**：大文件按章节拆成独立细则文件，降低单文件算力成本
**机制3 变更日志独立**：所有变更统一记录在变更日志，细则文件不含变更过程

**设计流程中的纪律**：
- 修改基线时另存新版本，不直接覆盖
- 细则文件只含正文，不含变更记录
- 总纲是概要索引，需要详情时读对应细则
- 大改动分批进行，每次只改一个细则文件
