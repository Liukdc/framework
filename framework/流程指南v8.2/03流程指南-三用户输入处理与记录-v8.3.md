# 态控流程指南 · 三、用户输入处理与记录

> **细则文件**——03/07。版本 v8.3，基于母本 v5.6。

## 三、用户输入处理与记录

```
用户输入
│
├─ 第0层：口令精确匹配（元指令集口令层，EXACT_MATCH→直切，非匹配→放行ANALYZING）
│   匹配唤醒词 → 进入 LISTENING
│   匹配退出/取消词 → 进入 CLOSING 或放弃当前流程
│   匹配器为可配置浅匹配器
│   注：自然语言疑似切换→ANALYZING判meta_suspect→二次确认后才切（语义确认层）
│
├─ 第0.5层：DET 关键词扫描（0 次 LLM）
│   检测跨意图词汇 → 附加 offTaskSuspicion 标记
│   不强制覆盖模型判断
│
├─ 第1层：意图识别（LISTENING 态，1 次 LLM）
│   拼接【意图识别宪法】→ 调 LLM
│   拿到 {intent, taskType, subType, logprobs（概率分布/来自API logprobs）, extracted（已提取信息）}
│   field_based 任务：
│     a. 先查 sessionCheckpoints 是否存在未过期进度
│     b. 若存在→生成 resume_chat intent，询问用户"继续 or 去其他环节"
│     c. 用户确认→注入 completedSteps+stepSnapshots→路由到目标环节
│   topic_based 任务：
│     a. ANALYZING 阶段 LLM 生成候选 topicName + keywords
│     b. 调度器执行关键词匹配（基于 topicIndex 表），检测是否已有相近主题
│     c. 若匹配到已有主题 → 系统展示候选主题名，等待用户确认（续写/重选/新建）
│     d. 若未匹配到 → 系统展示生成的候选主题名+keywords，等待用户确认（确认/修改/搜索已有）
│     e. 用户确认后，态控调度器生成 topicId，初始化 topicEvolution 记录
│     f. 系统自动检测当前主题与已有主题的关键词重叠度，若高则提示用户是否建立关联
│   若 intent="meta_instruction"（语义模板层判定为元指令），触发节点切换
│   状态机根据 intent 分发到对应环节，taskType 固化
│   路由表第二层（意图级绑定）覆盖默认模型
│   注：topic_based intent 倾向绑定深度模型，同时自动启用 topicEvolution 三层递进机制
│
└─ 第2层：环节内模型自治（IN_SESSION 态）
调度器通过 上下文管理器.buildPromptContext() 拼接上下文
→ topic_based 模式三层注入：已生效领域规则 + 关联摘要 + topicEvolution 分层包
→ 注入模型实例
模型自主追问/校验/偏离检测
→ 模型顺带输出 changeLevel 预判（major/minor/invalid）
→ 返回 turnType（含 askingField 双模式语义 + changeLevel 字段）
调度器根据 turnType 进行状态调度
turnType="complete" → 四项检查（按 taskType 分支，0 次 LLM）
topic_based 任务在节点 exit 时：
- stateChange 摘要由 N10 后台异步消费 ruleMiningQueue 时生成（不在态控调度器同步链路内调 LLM）
- 使用模型判定的 changeLevel 追加 event（major永久/minor压缩/invalid归档）
- 将主题信息推入规则提炼待处理队列
- 不触发任何提炼逻辑

[异步规则提炼]（独立后台进程，不在状态机流转链路内）
触发条件：时间窗口（每日凌晨2点）/ 数量阈值（积累N个同类主题checkpoint）/ 手动触发
提炼引擎：
1. 从 contractStore 读取待提炼主题列表
2. 提取候选规则，存入 ruleCandidate 池
3. 交叉验证（验证通过率 ≥ domain_rule_validation_threshold，默认 0.7）
4. 通过验证的候选规则提交用户确认
5. 用户确认后的规则存入 contractStore.domainRules 库
```

### 5.1 第0层：口令精确匹配（元指令集口令层）

**核心原则：元指令是口令，不是自然语言。** 口令含系统预设前缀或特定组合（如 `@@exit`、`/quit`、`小安退出`），由 N7 设计者定义为架构机制 M1 词表。口令不诞生于日常用语，自然语言中极低频。

匹配规则：
- **EXACT_MATCH：用户输入与口令词表项完全一致** → 直接触发节点切换，0 次 LLM。用户主动说出约定令牌 = 明确意图
- **非精确匹配（含"退出""算了""不玩了"等自然语言）→ 不切，放行至 §5.3 ANALYZING**

自然语言中的疑似切换意图（如"算了不聊柜子了"）由 ANALYZING LLM 判定为 intent="meta_suspect" → 系统发起二次确认（"你是想退出当前任务吗？"）→ 用户明确回复"是"/"对"/"确认"后才切。此路径不需快照保护——用户确认后才切，不存在误切。

纪律：**口令 = 约定令牌，非口令 = 自然语言 = 不切。** 语言交互中的意图确定性来自口令，不来自对自然语言的猜测。

### 5.2 第0.5层：DET 关键词扫描

每轮用户输入执行 DET 关键词扫描，检测跨意图词汇。标记附加到模型输入中作为提示，不强制覆盖模型判断。理由：关键词匹配能快速发现"用户说了'查一下'"，但无法区分是真正的意图切换还是口语中的随口一提——最终判断留给模型。

**off-task 处理**：第0.5层仅做关键词标记，真正的偏离检测与处理流程详见 §6.1（环节内偏离处理）。

### 5.3 第1层：意图识别

LISTENING 态收到输入 → 调 LLM 做意图识别。LLM 在意图识别宪法约束下做有限展开，将弥散输入收敛到明确的 intent 枚举值 + taskType + 置信度。出口被 contract-out schema 硬卡。

**语义模板层（元指令集第二层）：** 口令层未匹配的自然语言切换意图（如"算了不聊柜子了，帮我写首诗"），由 ANALYZING 的 LLM 在意图识别时顺带判断——若判定为元指令，intent 标记为 meta_instruction，状态机据此触发节点切换。

**意图稳定性防抖动：** 同一会话中同一用户在 60 秒内 intent 在两种类型间来回切换 ≥ 3 次 → 不重判，锁定首次判定的 intent，追问确认用户真实意图。

**extracted 字段合并：** 意图识别阶段可顺带提取已明确的字段，作为环节的初始上下文注入模型实例，减少一轮追问。

**taskType 固化：** 意图识别产出的 taskType 在此阶段从 N1 初判值固化为确定值，写入 activeSession.taskType，后续节点按此执行对应分支。

**topicEvolution 初始化与跨会话匹配：**

- topic_based 任务：ANALYZING 额外生成 topicSlug + topicName + keywords，用户确认后组合成 topicId，态控调度器在 contractStore 中初始化 topicEvolution 记录
- 跨会话匹配：若用户输入含"继续上次的XX"等续写信号，ANALYZING 用关键词匹配（基于 topicIndex 表）检索已有主题，关键词匹配不到时降级为 embedding 语义匹配。匹配阈值由 topic_evolution_match_threshold 控制

### 5.4 第2层：环节内模型自治

调度器通过 上下文管理器.buildPromptContext() 拼接上下文窗口（topic_based 模式额外包含三层注入包：已生效领域规则 + 关联摘要 + topicEvolution 分层包），与局部宪法一起注入模型实例。模型在局部宪法约束下自主完成多轮对话。

**环节内硬解析（field_based 模式）：** 模型在环节内优先尝试硬解析（金额正则匹配、时间词典匹配），硬解析成功→值域校验→通过则直接填入。硬解析失败→追问用户确认。

**环节内主题探索（topic_based 模式）：** 模型在环节内自主决定主题探索路径，每轮生成动态主题锚点标签作为 askingField。

**值域校验/格式校验（第一道，模型内部）：**

- field_based：金额 ≤0 拦截、金额 >999999 拦截、时间未来拦截、数量 ≤0 拦截
- topic_based：字数/结构/风格等格式约束校验
- 校验失败时模型返回 turnType="validation_failed"，态控调度器复用模型 message 转发用户

**偏离检测：**

- **保障1（主力——宪法强制 + 上下文管理器优先注入）：** 根宪法第1条（环节隔离）+ 上下文管理器最高优先级注入环节隔离声明，要求模型每轮首要判断"是否本环节任务"。不属于本环节 → 立即返回 off-task（优先级最高）。大幅降低漏判率。
- **保障2（辅助——ANALYZING 重新识别验证）：** 模型返回 off-task 后，调度器不直接路由。DET 只做非空检查（门卫，不做语义验证），送回 ANALYZING 重新识别 offTaskInput。ANALYZING 的重新识别结果就是验证（v8.1 修正：DET 做不了语义验证，LLM confidence 不可靠）。
- **显式切换口令（M1 switch，v8.1 新增）：** 用户可主动说"切断房间"口令 → DET 硬匹配（EXACT_MATCH，0 次 LLM）→ 直接路由 ANALYZING。显式口令是首选切换机制（100% 确定性），off-task 检测是兜底。
- **设计原则：** 不追求 100% 拦截，而是"降低漏判 + 拦住误判 + 回溯兜底"。用确定性兜底追求 100% 反而会破坏环节自治。
- **DET 不主动介入：** 第0.5层只标记不判断。DET 不在环节内主动触发 ANALYZING——这是对模型自治权的僭越。DET 只在 IN_SESSION 出口后做事后验证。

**changeLevel 预判 + event 追加：**

- 模型在返回 turnType 时顺带输出 changeLevel 预判字段（major/minor/invalid）
- IN_SESSION exit 时：态控调度器使用模型判定的 changeLevel 追加 event；stateChange 摘要由 N10 后台异步生成
- EXECUTING 阶段：同上，stateSnapshot 标记为 checkpoint
- giveup 时：stateSnapshot 标记为 abandoned
- off-task 时：记录偏离前的状态，stateSnapshot 保持 active
- 分层留存：major 级永久保留，minor 级超过阈值时触发汇总压缩，invalid 级迁入归档表

### 5.5 第二道防线：四项检查

模型返回 turnType="complete" 时，态控调度器执行第二道防线。这是确定性代码，零 LLM 调用（跨延伸例外层除外）。复验失败 → 拦截，进入 CLARIFYING。

**检查1：L1 结构校验（对应根宪法第2条输出格式）**

- 执行体：DET
- 执行方式：JSON.parse + Schema 校验
- taskType 影响：无差异
- 失败处理：重试1次→仍失败→熔断→硬编码回复

**检查2：DET 值域复验（field_based）/ 输出格式校验（topic_based）**

- 执行体：DET
- 执行方式（按 taskType 分支）：
- field_based：值域校验（金额上限、时间格式、必填字段…），规则来源：环节宪法 @section validation 的结构化 JSON
- topic_based：输出格式校验（JSON Schema / 文本格式 / 字数 / 结构…），规则来源：环节宪法 @section validation 的结构化 JSON
- 失败处理：进入 CLARIFYING，复用模型 message 转发用户

**检查3：logprobs 检查**

- 执行体：DET（两步均 DET，0 LLM）
- 执行方式：
- 步骤①：数值判断 probability < logprobs_threshold（DET 纯数值比较，来自 logprobs 转换，threshold 从 tunable 读取，默认 0.4）
- 步骤②：轻量 pattern 扫 message 是否含"硬撑式确认"词表（如 `/您说得对|没错|好的给您|已为您/`，词表 tunable 可扩展）
- 两步任一命中即标记 pretend_to_know，拦截
- taskType 影响：无差异（两种模式都执行）
- 失败处理：进入 CLARIFYING，复用模型 message 转发用户

**检查4（v8.5 已移除）：跨任务延伸检测**——跨任务是用户显式确认行为（M1 switch 口令或 off-task→ANALYZING 重新识别→用户确认），不需要降级链层自动化检测。详见 §6.1 off-task 双重保障 + M1 switch 口令。

**检查5：硬编码兜底**

- 执行体：DET
- 时机：上述四项检查均失败/熔断时
- 说明：所有校验失败后的最终兜底，硬编码回复

**仲裁权（架构机制 M3）：** 当模型内部校验（第一道）与调度器 DET 复验（第二道）结果冲突时，以调度器 DET 为准。

### 5.6 L1 校验 + DET 复验

L1 结构校验（完成时）：对 result 执行 JSON.parse + Schema 校验。失败→重试1次→仍失败→熔断→硬编码回复。

DET 值域复验/格式校验（完成时）：对 result 执行第二道 DET 校验（按 taskType 分支，含 logprobs 检查）。复验失败→拦截，进入 CLARIFYING。

### 6.1 环节内偏离处理

**v8.0 off-task 双重保障流程**（v8.1 修正：2026-07-17）：

当模型返回 turnType="off-task" 时：

1. **保障1（主力，已完成）：** 模型基于根宪法第1条 + 上下文管理器优先注入环节隔离声明，首轮即识别偏离
2. **保障2（辅助，ANALYZING 重新识别验证）：** 调度器不直接路由 → DET 只做非空检查（门卫）→ 送回 ANALYZING 重新识别 offTaskInput → ANALYZING 的重新识别结果就是验证
3. **显式切换口令（M1 switch，首选）：** 用户说"切断房间"→ DET 硬匹配 → 直接路由 ANALYZING（0 次 LLM，100% 确定性）。口令是首选，off-task 检测是兜底
4. **设计原则：** DET 不做语义验证（做不了），语义验证由 ANALYZING 重新识别完成。DET 不僭越模型自治权



- 调度器暂存 collectedFields 到 _pendingCollectedFields
- field_based：collectedFields 为已采集的字段
- topic_based：collectedFields 为已讨论的主题摘要
- 暂存时长由系统设置（默认 24 小时），超时自动清理
- 将 offTaskInput 送入意图识别重新分发
- 下次同一用户进入同一环节时，态控调度器询问是否继续上次未完成的记录
- topic_based 任务在 off-task 时，态控调度器自动追加 event 到 topicEvolution（记录偏离前的状态，使用模型判定的 changeLevel），stateSnapshot 保持 active

### 6.2 用户明确离开时

- 直接作废：清空 activeSession
- 不留草稿：不保存未完成记录，不恢复现场
- _lastRecord 不清空：用于"记错了"场景
- topicEvolution 不清空：topicEvolution 跨会话持久化，用户离开后仍可通过"继续上次的XX"恢复（分层注入包确保上下文窗口可控）

### 6.3 用户放弃

模型返回 turnType="giveup" → 调度器清空 activeSession → LISTENING。topic_based 任务在 giveup 时使用模型判定的 changeLevel 追加 event，stateSnapshot 标记为 abandoned，推入提炼待处理队列。topicEvolution 不清空。

### 6.4 意图级纠错（逃生舱）

当用户反馈表明当前意图完全错误时（如"我不是要记账，我是要查询"）：

1. 状态机从 contractStore 取出 ANALYZING 节点的原始 contract-in 副本
2. 重新实例化 ANALYZING 节点（注入原始 contract-in + 宪法）
3. 重新执行意图识别，产生新的 intent（可能伴随 taskType 变化）
4. 清空当前 activeSession，按新 intent 和新 taskType 重新分发

此机制需要 contractStore 持久化保存 ANALYZING 的原始 contract-in。contractStore 独立于执行体实例的生命周期——Agent 消亡后，其产出的契约副本仍可被回滚流程访问。
