### 动态验证方法


### 9.1 验证阶段的职责分工

| 阶段 | 验证方法 | 责任人 | 覆盖率目标 |
|------|---------|-------|----------|
| 设计阶段 | 检查清单 | 设计者 | 100%（全量自查） |
| 灰度期（0-2 周） | fuzzing | 开发+QA | ≥60%（必须跑出第一份转移频率热力图） |
| 运行期 | 全自动 | 运维+产品 | ≥90%（真实流量覆盖） |

### 9.2 对话树 fuzzing

用一批半随机化的输入序列跑状态机，收集 (fromState, inputPattern, toState) 的转移覆盖率。未覆盖的边 = 要么是死代码，要么是缺失测试用例。fuzzing 需同时覆盖 field_based 和 topic_based 两种 taskType 的环节，以及功能范围内和功能范围外两种输入类型。

### 9.3 会话日志回放

把真实用户的会话按 (state, input) → nextState 对齐标注，识别高频异常转移和僵尸路径。

### 9.4 环节内监督指标

**执行体调用链追踪（OpenTelemetry 集成）：** 每次 DET/LLM/HUMAN 调用生成 OpenTelemetry span，含字段：trace_id / span_id / parent_span_id / executor_type / executor_name / start_time / end_time / status / input_summary / output_summary / error_detail。运维侧用 Jaeger/Zipkin 可视化调用链。监控指标直接从 trace 提取。

**监督指标清单：**

- **校验失败率：** 模型返回 turnType="validation_failed" 的频率。field_based 和 topic_based 分别统计
- **偏离率：** 模型返回 turnType="off-task" 的频率
- **DET 复验拦截率：** 调度器第二道 DET 复验拦截的 complete 结果比例。field_based 和 topic_based 分别统计
- **CLARIFYING 触发频率：** 调度器进入 CLARIFYING 状态的频率，用于运维监控
- **上下文拼接命中率：** 模型在收到拼接后的上下文后，是否需要重复追问已采集的字段或已讨论的主题。topic_based 的三层注入 + embedding 两阶段检索命中率，field_based 的字段级硬门控命中率。分别统计
- **logprobs 拦截率：** 因 confidence < threshold 且 message 含"硬撑式确认"被拦截的 complete 结果比例。拦截率>10% → 模型频繁假装懂；拦截率<1% → 阈值可能过低
- **跨任务延伸拦截率：** 因 result 含不应产出内容被拦截的比例。field_based 拦截率>5% → 收敛方向不清晰；topic_based 拦截率>5% → 主题边界不清晰
- **意图级纠错触发率：** 用户触发逃生舱的频率。触发率>5% → 意图识别准确率不足；触发率<0.1% → 逃生舱入口不明确
- **taskType 误判率：** 意图识别阶段 taskType 判定错误的频率。误判率>2% → 回溯第一步
- **边界覆盖完整性：** 功能范围外输入的处理路径覆盖率 + 护栏响应质量分布
- **topicIndex 检索命中率：** 用户说"继续上次的XX"时，关键词匹配命中的比例。命中率过低（<50%）→ topicName 或 keywords 生成质量不足
- **主题关联建立率：** 同类型主题间建立关联的比例。过低（<10%）→ 关联自动发现机制不够敏感
- **规则提炼触发频率：** 异步提炼引擎的触发频率
- **领域规则确认率：** 候选规则经用户确认后生效的比例。过低（<30%）→ 规则提炼引擎过于激进
- **领域规则应用覆盖率：** 同类型新主题初始化时注入领域规则的比例。过低（<50%）→ 领域规则分类体系不完善
- **规则提炼队列积压量：** 待处理队列中的主题数量。积压量持续增长 → 提炼引擎处理能力不足或触发频率过低
- **知识真空期平均时长：** 主题从 checkpoint 到其规则被提炼生效的平均时间间隔
- **changeLevel 分布：** 模型判定的 major/minor/invalid 分布。invalid 占比 > 30% → 模型过于激进地判定撤销；**major 占比 < 3% → 模型可能偷懒全判为 minor（注意：阈值从原"1%"放宽至"3%"，降低了误报风险——少量真实 major 事件是正常的，但持续低于 3% 说明模型不愿做出结构性判断）**；changeLevelReason 空值率 > 10% → 模型在规避理由输出
- **embedding 调用次数：** topic_based 模式的 embedding 调用次数，监控成本
- **冷启动应急解冻触发次数：** 监控冷启动期间参数异常

> **多模型 A/B 测试路由配置**：态控架构支持在 IN_SESSION(align) 节点中配置多模型并行对比（如 R1 vs Qwen vs Claude），通过对比节点指标（confidence 分布、校验失败率、用户手动纠错率）选出最优模型。详细路由配置示例见附录 B。
