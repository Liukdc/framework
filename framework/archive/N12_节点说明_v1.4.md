## 一、位置与定位
```
N10 tunable 槽位声明（已定稿）
↓
N11 契约对齐（已定稿，verdict="pass"）
↓
N12 L2→L3 拆包  ← 你在这里
↓
N13 骨架生成
↓
N14 审骨架 + 跑 case + 查机制
↓
N15 调 tunable + 交付
```
**【人话版定位】**
前面 N1-N10 产出了 L2 文档（人读母本），N11 确认了所有契约之间没有矛盾。N12 的任务是把 L2 文档"翻译"成 L3 生成包——把设计者读的"为什么这么设计""常见错误有哪些""这一步在流程中的位置"等解释性内容全部剥离，只保留机器和下游节点能直接消费的纯净规则集。
**打个比方**：L2 是一本带注释、有例题、有常见错误提醒的教科书。L3 是从这本教科书里抽出来的公式表和定理清单——没有推导过程，没有例题，只有公式本身。
**N12 不修改 L2 的内容**。N12 只是从 L2 中提取、过滤、重组，生成一份"只含规则、不含理由"的 L3 包。如果拆包过程中发现 L2 内容缺失或矛盾，N12 标注出来，回退 L2 修正。

## 四、产出物
N12 产出两样东西：L3 生成包（`<project>-l3/` 目录）、N12→下游传递契约。
### 4.1 产出一：L3 生成包
L3 生成包是从 L2 文档中提取、过滤、重组后的纯净规则集，只保留机器和下游节点能直接消费的内容。
#### 4.1.1 过滤纪律（最高优先级）
L2 文档中**以下内容不进 L3**：
| 过滤类别 | 具体内容 | 过滤原因 |
|---------|---------|---------|
| 设计理由 | "为什么这么设计""设计者选择 X 而非 Y 的原因"等解释性文本 | L3 消费者（N13 代码模型）不需要知道"为什么"，只需要知道"是什么" |
| 设计阶段对比性内容 | L2 中用于对比两种方案/模式的对比表 | 对比分析属于设计决策过程，非可执行规则 |
| 设计阶段专属元数据 | 标注为"设计阶段参数，系统运行期间不生效"的参数的运行时约束声明（如 isRuntimeParameter=false） | L3 消费者只需要参数的 name/defaultValue/valueRange/purpose/consumer/applicable |
| 落地关系标注 | N9 环节宪法和其他 L2 文档中的落地关系标注——包括但不限于"（根宪法第3条 落地）""（落地环节宪法 @section ask-rules）""（与 N5 第 3 层 DET 复验对齐）"等用于说明规则来源或用途的元注释 | 属于设计阶段的落地关系说明，不是规则本身。L3 只保留规则文本，不含元注释 |
| 常见错误 | "常见踩坑"表格、"常见错误"章节 | 常见错误是设计阶段的人工提醒，不是可执行规则 |
| 实现状态标记 | "已实现""待实现""已定稿""草案"等状态标注 | L3 生成包假设所有内容均为已定稿状态 |
| LLM 调用次数对比 | "这个方案调 1 次 LLM，那个方案调 3 次"等对比分析 | 属于设计决策过程的分析，非可执行规则 |
| 导航提示 | "详见 §X""如 §Y 所述""此处参考 N5 节点说明"等交叉引用 | L3 包是自包含的规则集，消费者不需要导航回 L2 |
| 章节说明 | "本节介绍...""以下将讨论..."等章节引导语 | 纯元数据，不包含规则内容 |
| 版本变更摘要和版本对比内容 | 文档尾部及章节内嵌的版本变更摘要和版本对比内容 | 版本演变历史属于 L2 文档的元信息，不是可执行规则 |
| 示例性内容（路径示例） | L2 中的"路径示例" | 路径示例是 L2 的说明性内容，容易与通用规则混淆。不要把路径示例当成通用规则写进 session-constitution |
| 小白操作指南 | "§十 小白操作指南"中的比喻、解释、分步说明 | 面向人类设计者的引导性内容，不是可执行规则 |

以下内容**必须保留**并纳入 L3 生成包：
| 保留类别 | 具体内容 | 提取目标文件 |

## 五、关键纪律
1. **过滤纪律是最高优先级**——L2 中的设计理由、设计阶段对比性内容、设计阶段专属元数据、落地关系标注、常见错误、实现状态标记、LLM 调用次数对比、导航提示、章节说明、版本变更摘要和版本对比内容、路径示例、小白操作指南一律不进 L3
2. **宪法文本中的落地关系标注需全部剥离**——L3 只保留规则文本本身
3. **`<!-- @rule -->` 标记是拆包语义一致性的唯一校验锚点**——标记缺失的规则可能在拆包时被遗漏
4. **声明-实现一致性检查是拆包的最后一道防线**——N9 的 @section validation 规则必须能在 N5 的 DET 复验中找到对应实现。changeLevelReason 字段规范的兜底行为一致
5. **N12 不修改 L2 的内容**——只提取、过滤、重组
6. **N11 对齐报告 verdict="pass" 是 N12 启动的前置条件**
7. **15 个 @section + 3 个元标签是 session-constitution 完整性的硬指标**
8. **topic_based 和 field_based 的宪法模板不可混用**
9. **松弛节点（SLACK_NODE）不需要环节宪法**——仅受根宪法约束（v5.3 §4.2.2），不区分 taskType，不在环节宪法约束范围内。session/ 目录不为松弛节点生成宪法文件

## 七、常见踩坑
| 坑 | 表现 | 后果 |
|----|------|------|
| 路径示例混入 session-constitution | 把 L2 中"记账环节的字段追问顺序示例"当成了通用字段规则写进宪法 | 模型在环节内被过度约束 |

## 十、小白操作指南
### 10.0 先搞清楚你在干什么
你现在要干的事：当"纯净规则提取器"。L2 是一本写给人类设计师看的教科书——里面有设计理由、常见错误提醒、操作指南、章节导航、落地关系标注。N13 的代码模型不需要这些，它只需要"公式本身"——状态怎么定义、校验规则是什么、turnType 有哪些值、changeLevelReason 的兜底行为是什么。
**你的任务就是把教科书里的公式表抽出来，扔掉所有的例题、注释、导读和落地关系标注**。
**特别注意容易遗漏的东西**：
- 路径示例——"记账环节先问种类再问金额"是 L2 用来举例的，不是通用规则
- 落地关系标注——"（根宪法第3条 落地）""（落地环节宪法 @section ask-rules）"等元注释是给设计者看的，不是规则本身
- changeLevelReason——它不在"15 个 @section + 3 个元标签"的显式列表中，但它是 @section output-schema 的必备内容，且三处都要有
- tunable 参数的 name 和 consumer——容易在过滤设计阶段元数据时被误杀，但它们对 N13 骨架生成至关重要
- tunable 参数的数量——通用 10 个 + topic_based 专用 8 个 + field_based 专用 1 个，少了任何一个都可能影响 N13 生成

### 10.4 常见错误
