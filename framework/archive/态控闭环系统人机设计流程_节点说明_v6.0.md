## 🔵 第一步：划边界

### N1（节点1）场景定义 + 边界清单 + taskType 初判

- **输入**：产品需求 / 同类产品日志（如有）
- **输出**：
  - 一句话场景定义（锚点）
  - 边界清单表格（做什么 / 不做什么 / 交互方式 / 数据处理）
  - "不做什么"的护栏策略（v5.3：自由切换四个支柱含产出物总索引，见 §2.6）（硬拒 vs 软着陆）
  - **intentTaskTypeMap**：为每个 intent 标注 taskType
- **责任人**：智能体开发员
- **关键纪律**：「不做什么」比「做什么」更重要；护栏升级 ≠ 边界扩展，软着陆的分支目标必须已在转移图中

**taskType 初判要求**（区分原点：收敛空间的性质，非任务内容。详见 v5.3 §3.2）：
- field_based 判据：离散参数空间，存在客观正确答案——输出有预定义字段集 + 字段值有明确值域 + "完成"的判定标准是字段齐全 + 错误可 DET 值域硬校验。典型场景：记账、改柜体参数、表单填写、参数查询
- topic_based 判据：连续语义空间，不存在唯一正确答案——输出在主题空间内采样满足约束的样本 + "完成"的判定标准是输出满足主题边界和格式约束。典型场景：各种内容生成、角色扮演、开放研究、头脑风暴、诗歌创作、专利撰写、方案设计
- 混合型任务：主 taskType + field_based 约束作为子条款
- taskType 初判写入 intentTaskTypeMap 字段，传递给 N3 固化
- **架构决策原则**：当用户不确定某个 intent 的 taskType 时，提供 2-3 个选项 + 每个选项的利弊分析，由用户选择。不直接给答案

**taskType 初判时需考虑的 topicEvolution 因素**：
- 当判定某个 intent 为 topic_based 时，设计者必须意识到：
  - 该 intent 的所有交互将被自动记录到 topicEvolution 表中
  - 系统将自动启用三层递进机制：分层留存 + 主题关联 + 异步规则提炼
  - 异步规则提炼存在"知识真空期"——领域规则不保证跨相邻任务的即时传递
  - 跨会话的"继续上次的XX"将成为系统标配能力
  - 多个同类型主题将逐步积累领域规则，智能体越用越老练
  - 需要在 N9 环节宪法 @section output-schema 中声明 topicId 字段和 changeLevel 判定规则

**taskType 初判时需考虑的 sessionCheckpoint 因素**：
- 当判定某个 intent 为 field_based 时，设计者必须意识到：
  - 该 intent 的多步骤流程将由 sessionCheckpoint 自动记录进度和每环节字段快照
  - 系统将自动支持跨会话断点续做——用户唤醒后可选择"继续"或跳到已满足前置的任意环节
  - contractEvolution 将为每个环节按 stepName 隔离提炼设计规则，越用越老练
  - 需要在 N6 的 schedulerState 和 contractStore 中包含 sessionCheckpoints 和 roomConversationLog**
  - 需要在 N9 环节宪法 @section output-schema 中声明 changeLevel 和 changeLevelReason 字段（两种模式均适用）

**taskType 初判时需考虑的 @importance 因素**：
- 设计者在 N1 阶段就应预判哪些环节/产出物/规则为 critical：
  - 房间级：同一 taskType 下哪个环节是核心（如 field_based 中 N1 为 critical）
  - 产出物级：各环节的 machineReadable 中哪个字段是核心（如 N1 的 doList 为 critical）
  - 规则级：L0/L1 必须各有 1 条 critical（安全红线 + 基线命门——在后续 N9 写宪法时确定具体规则）
- @importance 的 critical 房间获得 session-store 保留轮次 ×2 的增强（tunable `critical_room_history_boost`，默认 2.0）
- 每层 critical≤1、high≤⌊该层总数/3⌋——初判时不要太慷慨，预留后续环节扩展空间

**元指令口令词表设计**：
- N1 边界划定阶段就应定义场景的元指令口令词表（如 @@exit / @@switch / @@cancel）。口令≠自然语言——"退出""算了"等自然语言走语义确认层二次确认，不直接触切
- 口令词表在 N7 根宪法中形式化，N5 调度器第0层 DET EXACT_MATCH 执行

**混合型任务处理**：某些任务可能同时包含两种 taskType 模式。处理方式：在环节宪法中声明主 taskType（topic_based），将 field_based 约束作为 topic_based 的格式约束子条款。DET 复验时先做 field_based 格式校验，再做 topic_based 主题边界校验。

**关于松弛节点**：如果智能体需要支持"闲聊→任务"的自然过渡，边界清单中应显式声明 slack-node 的存在及其边界。slack-node 是合法存在的低约束节点，仅受根宪法约束，不区分 taskType。但 slack-node 的边界本身也需要经过边界覆盖完整性测试——如果 slack-node 的闲聊输入无处理路径或护栏响应不合理，说明 slack-node 的边界划定有问题，需要回溯第一步重审边界。

**一句话场景定义示例**：
- field_based："这是一个语音记账工具，用户说出支出，系统记录并支持查询和删除。"
- topic_based："这是一个角色扮演助手，用户设定角色和场景，系统在主题边界内进行多轮角色扮演对话。"
- 混合型："这是一个全屋定制方案设计助手，用户描述需求，系统通过多轮对话完成方案设计，支持多方案对比、风格替换和方案迭代。"

**边界清单示例**：

| 维度 | 做什么 | 不做什么 | taskType |
|------|--------|---------|----------|
| 核心功能 | 记录支出、查询历史、删除记录 | 不自动分类、不推荐理财产品 | record/query/delete: field_based |
| 核心功能 | 角色扮演对话 | 不生成图片、不执行真实操作 | roleplay: topic_based |
| 核心功能 | 全屋定制方案设计、多方案管理、方案对比 | 不直接对接工厂生产系统 | design: topic_based |
| 交互方式 | 语音输入、文字输入 | 不支持图片识别、不支持手势操作 | — |
| 对话范围 | 业务相关的指令 | 不闲聊、不回答非业务问题 | — |
| 数据处理 | 本地存储用户数据 | 不上传云端分析、不分享给第三方 | — |

关键规则：边界清单中"不做什么"的部分，比"做什么"更重要。但"不做什么"不意味着"不处理"——功能范围外的输入必须有明确的处理流程（硬拒/软着陆/引导），不能出现"无法处理"的输入。

### N2（节点2）边界紧张度测试

- **输入**：边界清单 + 20-30 条真实用户表达
- **输出**：边界覆盖完整性报告 + 护栏响应质量报告
- 边界覆盖完整 + 护栏响应质量合格 → 进入 N3（拆状态）
- 边界覆盖有缺口或护栏响应质量不合格 → 回退 N1（补全边界清单 + 护栏策略）
- **责任人**：智能体开发员

#### 核心原则

智能体必须能够处理所有用户输入的信息。用户输入只有两类：
- **类一：落在功能范围内**（"做什么"范围内）→ 按意图识别结果由状态机处理
- **类二：落在功能范围外**（"不做什么"范围内）→ 智能体同样必须有处理流程（硬拒 / 软着陆 / 引导）

任何输入都必须有对应的处理路径。不存在"落出"的输入。

#### 测试目标

| 维度 | 测试目标 |
|------|---------|
| 测试对象 | 边界覆盖完整性 + 护栏响应质量 |
| 通过标准 | 所有测试语料都有明确处理路径 + 护栏响应质量合格 |
| 失败标准 | 存在无处理路径的输入类型 OR 护栏响应不合理 |
| 测试本质 | 验证"覆盖完整 + 响应合理" |

#### 测试方法（5 步流程）

**Step 1：收集测试语料**（20-30 条真实用户自然表达）
- 语料获取路径（按可信度降序）：已有真实用户日志（脱敏后） > 非项目组成员口语模拟（5-8 人，各 10 分钟） > 设计者自写
- **硬纪律**：没有经过路径 1 或路径 2 语料替换的边界紧张度测试，不算完成。设计者自写语料不可超过 50%。

**Step 2：分类标注**
- 维度一：输入类型分类（类一功能范围内 / 类二功能范围外）
- 维度二：处理路径检查（✅有处理路径 / ❌无处理路径）

**Step 3：边界覆盖完整性检查**

| 检查项 | 通过标准 | 失败处理 |
|--------|---------|---------|
| 类一输入处理路径 | 所有类一输入都有对应的 intent + 状态机处理路径 | 回退 N1，补全 intent 清单 |
| 类二输入处理路径 | 所有类二输入都有对应的护栏策略 | 回退 N1，补全护栏策略 |
| 处理路径合理性 | 护栏响应语合理 | 回退 N1，优化护栏响应语 |

**Step 4：护栏响应质量评估**

| 质量等级 | 表现 | 处理 |
|---------|------|------|
| ✅ 优秀 | 软着陆 + 引导到功能范围内（如"我可以帮你记，你要先记完这条还是先查？"） | 通过 |
| ✅ 合格 | 明确拒绝 + 说明边界 + 提供替代（如"小安只会记账，不会聊天哦。你需要记一笔吗？"） | 通过 |
| ⚠️ 待优化 | 生硬拒绝无引导（如"我只记账哦"） | 回退 N1，优化护栏响应语 |
| ❌ 不合格 | 无处理路径 / 崩溃 / 幻觉回应 | 回退 N1，补全护栏策略 |

**护栏升级的分支目标鉴别规则**：任何"护栏升级为软着陆"的决策，必须记录软着陆实际调用的分支目标是否已经存在于当前状态转移图中。如果"先查"对应的是状态转移图里已有的 query 子流程，这只是让边界护栏指向已有能力，边界声明未变。如果软着陆引入了状态转移图中不存在的分支路径，这不是护栏升级，而是"收缩不做什么"的伪装形式——必须走正式扩展边界流程，不得以"护栏调整"之名绕过。

**Step 5：生成测试报告**

```
边界覆盖完整性报告：
- 测试语料总数：N 条
- 类一输入（功能范围内）：N1 条
  - 有处理路径：N1a 条 ✅
  - 无处理路径：N1b 条 ❌
- 类二输入（功能范围外）：N2 条
  - 有处理路径：N2a 条 ✅
  - 无处理路径：N2b 条 ❌

护栏响应质量报告：
- 优秀：N 条
- 合格：N 条
- 待优化：N 条
- 不合格：N 条

结论：
- 边界覆盖完整（N1b=0 且 N2b=0）+ 护栏响应质量合格（待优化+不合格=0）→ 进入 N3
- 边界覆盖有缺口 OR 护栏响应质量不合格 → 回退 N1
```

#### N2 双角色串行机制

N2 是需要双角色串行且信息隔离的环节，不走普通 IN_SESSION 的单模型实例自治。

**Step 1**：调度器判断当前环节为 N2 → 进入 N2 "双角色串行"流程

**Step 2**：启动子环节一（角色一：真实语料搜索员）
- 注入的上下文：仅场景定义（一句话）
- 明确不注入：边界清单、意图清单、N1 交付物全文
- 态控调度器在注入前做一次安全检查：如果 N1 边界清单或意图清单出现在待注入上下文中 → 拦截，报错
- 角色一执行，输出原始语料集

**n2-corpus-hunter output-schema**：

```json
{
  "corpus": [
    {
      "id": "corpus_001",
      "user_input": "今天天气真好",
      "source": "real_user_log | non_project_member_simulation | designer_draft",
      "context": "（可选）语料产生的上下文描述"
    }
  ],
  "corpus_count": 20,
  "source_distribution": {
    "real_user_log": 10,
    "non_project_member_simulation": 8,
    "designer_draft": 2
  }
}
```

**Step 3**：调度器校验角色一的输出
- 采用 **schema 校验 + 日志告警** 策略：
  - 核心字段校验：corpus / corpus_count / source_distribution 三个字段必须存在——任一缺失 → 拦截，报错要求重跑角色一
  - 额外字段处理：允许额外字段存在，但记录告警日志（如 `"角色一输出含额外字段：note, suggestion_strategy, expected_intent"`），供后续审查
  - 不因额外字段而判定"语料污染"或拒绝通过——额外字段不阻断流程，只记录告警
- 核心字段存在 → 进入 Step 4

**Step 4**：启动子环节二（角色二：边界测试执行器）
- 注入的上下文包含三样：a) 角色一输出的原始语料集 b) N1 边界清单（含处理策略）c) N1 意图清单
- 态控调度器在注入前做一次完整性检查：如果缺少任一样 → 拦截，报错
- 角色二执行，输出测试报告

- 结论为"pass"：记录 N2 测试报告，状态机进入 N3
- 结论为"fail"：记录 N2 测试报告，状态机回退到 N1，携带修正建议，提示设计者"修改 N1 边界清单后，必须重新跑完整 N2（角色一 + 角色二）"

#### 调度器状态机中的对应状态

N2 双角色串行在状态机中表现为两个连续的 IN_SESSION 子状态。调度器内部维护一个 `_n2Phase` 字段（仅 N2 环节使用，存入 schedulerState._sceneExtensions）：

```javascript
// 调度器私有字段，仅 N2 环节使用，存入 _sceneExtensions
_sceneExtensions: {
  _n2Phase: null,  // "corpus_hunting" | "boundary_testing" | null
}
```

**流转逻辑**：

```
ANALYZING → intent="n2_boundary_test"
→ _sceneExtensions._n2Phase = "corpus_hunting"
→ 注入场景定义 → 角色一执行
→ 角色一返回，校验输出（schema 校验 + 日志告警）
→ 核心字段存在 → _sceneExtensions._n2Phase = "boundary_testing"
→ 注入角色一输出 + 边界清单 + 意图清单 → 角色二执行
→ 角色二返回，读取结论
→ _sceneExtensions._n2Phase = null（重置）
→ 根据结论路由（pass → N3 / fail → N1）
```

#### 错误处理

- 角色一注入上下文中误含边界清单 → 拦截，报错，不启动角色一
- 角色一输出核心字段缺失（corpus / corpus_count / source_distribution 任一缺失）→ 拦截，要求重跑角色一
- 角色一输出含额外字段 → 记录告警日志，不阻断，继续流程
- 角色二注入上下文不完整 → 拦截，报错，不启动角色二
- 角色二输出格式不符合 output-schema → 重试 1 次，仍失败则报错
- 角色二报告结论为"fail" → 路由到 N1，携带修正建议，不清除 N1 产出

#### 设计原则

- 智能体必须处理所有输入：不存在"落出"的输入
- 类二输入不是错误：用户输入功能范围外的内容是正常行为
- 测试本质是验证覆盖完整性：不是测量"窄得合理"
- 护栏响应质量是核心指标：不是"拦住了多少"，而是"拦住后响应得怎么样"
- 边界重划的触发条件：存在无处理路径的输入类型——不是边界太窄，而是边界有缺口
- 松弛节点的边界同样需要紧张度测试

## 🔵 第二步：拆状态

### N3（节点3）状态枚举 + 执行体分派 + taskType 固化 + topicEvolution/sessionCheckpoint 初始化

- **输入**：边界清单（含 intentTaskTypeMap）
- **输出**：标准状态集（示例为 11 状态：IDLE / LISTENING / ANALYZING / IN_SESSION / CLARIFYING / WAITING_CONFIRM / EXECUTING / SLACK_NODE / TEMP_TOPIC / CLOSING，各项目可裁剪）+ 每个状态标执行体（DET/LLM/HUMAN）+ taskTypeFinalized + topicEvolution 初始化（topic_based 任务时）。SLACK_NODE（松弛闲聊）和 TEMP_TOPIC（运行时创建的仅根宪法房间）为 v5.3 系统级状态，N3 阶段确认是否启用，不随场景变化
- **判据优先级**：代价 > 能否硬规则化 > 有无客观答案
- **责任人**：智能体开发员

**taskType 固化要求**：将 N1 初判的 taskType 固化为确定值。固化后运行时不变。固化值写入 taskTypeFinalized 字段，传递给 N5/N9 等后续节点。

**topicEvolution 初始化要求**（topic_based 任务时）：
- 当 taskType 被固化为 topic_based 时，调度器自动初始化 topicEvolution 三层递进机制：
  - 在 contractStore 中初始化一条 topicEvolution 记录（topicId 由 ANALYZING 阶段 LLM 生成的 topicSlug + userId + intent 组合而成，格式为 `{userId}_{intent}_{topicSlug}_{date}`）
  - 同步写入 topicIndex 索引表（topicId 主键、topicName、keywords、creatorUserId、createdAt、lastActiveAt）
  - topicName 和 keywords 由 ANALYZING 阶段 LLM 自动生成，经用户确认后生效
  - 写入第一个 event（changeLevel 由模型在 ANALYZING 阶段顺带预判）
- 规则提炼完全异步，不在状态机流转链路内执行——调度器只在节点 exit 时将主题信息推入规则提炼的待处理队列

**taskType 固化后的影响**：
- N5 调度器按 taskType 执行对应的调度逻辑（field_based 字段级硬门控 DET 确定性匹配 / topic_based 主题级硬门控 embedding 两阶段检索 + 三层注入）
- N5 调度器按 taskType 执行 topicEvolution event 追加逻辑（topic_based 任务）
- N9 环节宪法按 taskType 选择对应的宪法结构模板（field_based 用 @section field-rules / topic_based 用 @section topic-boundary）
- N9 环节宪法 topic_based 模板含 topicId 字段声明和 changeLevel 判定规则
- N5 DET 复验按 taskType 分支（field_based 值域校验 / topic_based 格式校验）
- N10 tunable 参数按 taskType 启用（topic_based 启用 topic_anchor_similarity_threshold + topic_keyword_overlap_threshold + topic_evolution_match_threshold + topic_evolution_minor_compaction_threshold + domain_rule_validation_threshold 等）

**环节内模型实例的定位补充**：环节内模型实例属 LLM 执行体，但在环节内拥有自治权——态控调度器不参与其逐轮对话，只在环节入口（注入宪法+上下文，topic_based 模式额外注入三层注入包）和出口（收 turnType + changeLevel 预判 + DET 复验）交互。

### N4（节点4）状态转移图 + 路由表（含两层模型选择 + taskType 影响）

- **输入**：状态枚举 + 意图清单 + taskTypeFinalized
- **输出**：
  - DOT 格式转移图（覆盖最优路径 / 校验失败 / 偏离 / 放弃 / 删除确认 / 退出 / 闲聊）
  - 路由表：第一层状态级默认模型 + 第二层 intent 级模型绑定
- **关键纪律**：ANALYZING 用轻量模型（打断循环依赖）；topic_based intent 可绑专用深度模型
- **责任人**：智能体开发员

**taskType 对模型选择的影响**：
- field_based intent：用默认通用模型即可（字段映射不需要深度能力）
- topic_based intent：倾向绑定深度模型（主题理解和创意生成需要深度能力）
- 此影响在路由表第二层（intent 级绑定）中配置

**路由表示例**：

| 当前状态 | contract-out 关键字段 | 下一状态 | 路由规则 | 模型覆盖（第二层） |
|---------|----------------------|---------|---------|------------------|
| ANALYZING | intent="resize" (field_based) | IN_SESSION | 硬匹配 | —（用默认通用模型） |
| ANALYZING | intent="write_poem" (topic_based) | IN_SESSION | 硬匹配 | 专用文学模型 |
| ANALYZING | intent="roleplay" (topic_based) | IN_SESSION | 硬匹配 | 专用角色扮演模型 |
| ANALYZING | intent="exit" | CLOSING | 硬匹配 | — |
| **ANALYZING** | **intent=other + inputNature=S（闲聊）** | **SLACK_NODE** | **硬匹配** | **通用轻量模型** |
| **ANALYZING** | **intent=other + inputNature=T（任务导向）** | **TEMP_TOPIC** | **硬匹配（先经用户确认命名）** | **通用模型** |
| IN_SESSION | turnType="complete" + intent="resize" | EXECUTING（经 DET 值域复验） | 硬匹配 | — |

ANALYZING 节点本身使用其状态级默认模型（通用轻量模型），不依赖第二层绑定，循环依赖打断。

**两层模型选择机制**：
- 第一层（状态级默认模型）：每个 LLM 节点有预设的默认模型实例，不依赖意图。ANALYZING → 通用轻量模型；IN_SESSION → 通用模型。
- 第二层（意图级绑定）：当上游节点产出 intent 后，下游节点可根据"状态+intent"组合覆盖默认模型。此绑定配置在路由表中。
- 即使当前场景只有 field_based 任务，也应在设计时预留 taskType 字段和两层模型选择机制——为未来 topic_based 任务扩展提供零成本路径。

### N5（节点5）调度器核心逻辑 + 上下文管理器三接口 + taskType 双模式锚点 + topicEvolution/sessionCheckpoint 处理

- **输入**：转移图 + tunable 清单 + taskTypeFinalized
- **输出**：第0-3层过滤 + DET 复验 + 三接口调用占位 + taskType 双模式锚点机制（field_based：stepSnapshots+roomConversationLog；topic_based：三层注入）+ topicEvolution event 追加 + sessionCheckpoint 更新
- **责任人**：智能体开发员

#### 用户输入处理的三层过滤 + 第二道防线（含 topicEvolution）

```
用户输入
│
├─ 第0层：口令精确匹配（元指令集口令层，EXACT_MATCH→直切，非匹配→放行ANALYZING）
│   匹配唤醒词 → 进入 LISTENING
│   匹配退出/取消词 → 进入 CLOSING 或放弃当前流程
│   匹配器为可配置浅匹配器（子串/编辑距离/同义词组）
│   注：自然语言疑似切换→ANALYZING判meta_suspect→二次确认后才切（语义确认层）
│
├─ 第0.5层：DET 关键词扫描（每轮执行，0 次 LLM）
│   检测跨意图词汇 → 附加 offTaskSuspicion 标记
│   不强制覆盖模型判断
│
├─ 第1层：意图识别（LISTENING 态，1 次 LLM）
│   拼接【意图识别宪法】→ 调 LLM
│   拿到 {intent, taskType, subType, logprobs（概率分布/来自API logprobs）, extracted（已提取信息）}
│   field_based 任务：
│     a. 先查 sessionCheckpoints 是否存在未过期进度
│     b. 若存在 → 生成 resume_chat intent，询问用户"继续 or 去其他环节"
│     c. 用户确认 → 注入 completedSteps + stepSnapshots → 路由到目标环节 IN_SESSION
│   topic_based 任务：
│     a. ANALYZING 阶段 LLM 生成候选 topicName + keywords
│     b. 调度器执行关键词匹配（基于 topicIndex 表），检测是否已有相近主题
│     c. 若匹配到已有主题 → 系统展示候选主题名，等待用户确认（续写/重选/新建）
│     d. 若未匹配到 → 系统展示生成的候选主题名+keywords，等待用户确认（确认/修改/搜索已有）
│     e. 用户确认后，调度器生成 topicId，初始化 topicEvolution 记录
│     f. 系统自动检测当前主题与已有主题的关键词重叠度，若高则提示用户是否建立关联
│     g. 主动问询用户是否有已知的偏好或规则约束（user_initiated 来源）→ 写入 ruleCandidates 供后续注入
│   若 intent="meta_instruction"（语义确认层判定为元指令），触发节点切换
│   意图稳定性防抖：同一会话中同一用户在 60 秒内 intent 在两种类型间来回切换 ≥ 3 次 → 不重判，锁定首次判定的 intent，追问确认
│   状态机根据 intent 分发到对应环节，taskType 固化
│   路由表第二层（意图级绑定）覆盖默认模型
│
└─ 第2层：环节内模型自治（IN_SESSION 态）
      态控调度器通过 上下文管理器.buildPromptContext()（三层：匹配→决策→拼接）拼接上下文
      → field_based 模式：stepSnapshots 结构化字段 + roomConversationLog(roomId) + contractEvolution 已生效规则
      → topic_based 模式三层注入：已生效领域规则（按 topicPath 子图匹配精准注入）+ 关联主题摘要 + topicEvolution 分层包

      模型在局部宪法约束下自主追问/校验/偏离检测
```

#### 第0层：全局硬匹配（元指令集口令层）

匹配唤醒词/退出词/取消词，0 次 LLM。匹配器为可配置浅匹配器，词表需维护日志。此为元指令集的口令层，处理高频明确的元指令。口令层无法覆盖的元指令变体由 ANALYZING 的 LLM 在意图识别时顺带判断（语义确认层）。

#### 第0.5层：DET 关键词扫描

每轮用户输入执行 DET 关键词扫描，检测跨意图词汇。标记附加到模型输入中作为提示，不强制覆盖模型判断。理由：关键词匹配能快速发现"用户说了'查一下'"，但无法区分是真正的意图切换还是口语中的随口一提——最终判断留给模型。

#### 第1层：意图识别

LISTENING 态收到输入 → 调 LLM 做意图识别。LLM 在意图识别宪法约束下做有限展开，将弥散输入收敛到明确的 intent 枚举值 + taskType + 置信度。出口被 contract-out schema 硬卡。field_based 任务：先查 sessionCheckpoints 是否存在未过期进度，若存在则询问用户"继续 or 去其他环节"；topic_based 任务：检索 topicIndex（关键词优先，embedding 兜底），匹配到已有主题则展示主题名供用户确认续写，未匹配到则生成候选 topicName+keywords 供用户确认新建。ANALYZING 的 contract-in 副本同步写入 contractStore。

#### 第2层：环节内模型自治

态控调度器通过 上下文管理器.buildPromptContext() 拼接上下文窗口，与局部宪法一起注入模型实例。模型在局部宪法约束下自主完成多轮对话。

**buildPromptContext() 接口接收参数**：
- contextGraph
- turnHistory
- currentAskField
- currentTaskType
- currentTopicId（topic_based 时存在）
- currentStepName（field_based 时按 stepName 匹配 contractEvolution 规则）
- contractStore（上下文管理器内部按 topicId/stepName 查询 domainRules/relatedTopics/topicEvolution/sessionCheckpoints，态控调度器不预组装）
- tunableParams

**上下文管理器 按 taskType 执行对应的锚点模式**：

**field_based 模式（字段级硬门控，DET 确定性匹配）**：
- 与当前 askingField 同名的历史轮次不被淘汰——确定性规则，不依赖 LLM 判断
- 当前追问字段从模型返回的 askingField 获取（field_based 模式为字段名，如 "amount"）
- 典型场景：记账、改柜体参数、表单填写

**topic_based 模式（三层注入 + embedding 两阶段检索）**：

**三层注入（在 DET 关键词粗筛之前，上下文管理器 按优先级注入）**：
- **第0层：已生效的领域规则注入**——从 contractStore.domainRules 读取当前主题类型已生效的领域规则（异步提炼的产出，可能存在知识真空期）
- **第1层：关联主题摘要注入**——从 topicIndex.relatedTopics 读取关联主题摘要卡片（inheritedPoints / avoidPoints / reason），不拉取关联主题的原始 evt 链
- **第2层：当前主题演化注入**——从 contractStore.topicEvolution 读取分层注入包（majorEvents + compressedMinorSummary + recentMinorEvents + currentStateSnapshot）

**两阶段检索**：
- **阶段1：DET 关键词粗筛（确定性，O(N)）**——对 turnHistory 中每轮，DET 提取 askingField 主题标签的关键词，与历史轮次的文本做关键词重叠匹配。重叠度 ≥ topic_keyword_overlap_threshold（tunable，默认 0.2）的轮次进入阶段 2
- **阶段2：embedding 相似度精筛（embedding 模型调用，非 LLM 调用）**——对阶段 1 粗筛通过的轮次，用 embedding 模型计算 askingField 主题标签与历史轮次文本的向量相似度。相似度 ≥ topic_anchor_similarity_threshold（tunable，默认 0.6）的轮次被保护
- **DET 阈值裁决**：DET 只做"相似度 ≥ topic_anchor_similarity_threshold"的最终数值比较，确定性裁决。0 次 LLM 调用

**注入优先级**：领域规则 > 关联摘要 > topicEvolution 分层包 > turnHistory

**两种模式的共同部分**：综合匹配度评分、最低匹配度截断（min_match_score tunable）、偏离处理、archiveGraph、校验失败标记（updateGraph）——这些机制两种模式完全相同。

**askingField null 降级规则（两种模式通用）**：当 askingField 为 null 时（如 reply 轮次、complete 轮次），硬门控保护降级为纯语义匹配。field_based 模式不执行字段名匹配保护，topic_based 模式不执行主题标签匹配保护，均改为 上下文管理器 按综合匹配度评分分配形态。降级不报错不阻断。

**changeLevel 预判 + event 追加**：
- 模型在返回 turnType 时顺带输出 changeLevel 预判字段（null/major/minor/invalid）+ changeLevelReason
- 判定标准由环节宪法 @section output-schema 的操作化定义约束：
  - invalid：当且仅当本轮输出完全撤销了上一轮的实质性变更，且未产生新的有效信息时
  - major：涉及核心实体的增删、任务目标的根本性转向、或引入了全新的世界观/规则设定时
  - minor：凡不属于上述两者的情况，一律判定为 minor
- 调度器收到 turnType 后直接使用模型判定的 changeLevel（若模型未返回或格式错误 → 默认标记为 minor）
- IN_SESSION exit 时：stateChange 摘要由 N10 后台异步消费 ruleMiningQueue 时生成，态控调度器只落盘 topicEvolution event 原始数据（collectedFields + message + result），并使用模型判定的 changeLevel 追加 event
- EXECUTING 阶段：同上，stateSnapshot 标记为 checkpoint
- giveup 时：stateSnapshot 标记为 abandoned，推入提炼待处理队列
- off-task 时：记录偏离前的状态，stateSnapshot 保持 active

**分层留存**：major 级永久保留；minor 级超过 topic_evolution_minor_compaction_threshold（默认 5 条）时触发汇总压缩，原 event 迁入 topicEvolutionArchive 表；invalid 级不入主表，迁入 topicEvolutionArchive 归档表

**topicEvolution 注入时的分层结构**：
```
[topicEvolution 注入包]
├── majorEvents（永久保留，全部注入，通常 0-5 条，不压缩不丢弃）
├── compressedMinorSummary（最近一次 minor 压缩摘要，1 段）
├── recentMinorEvents（压缩后新增的 minor 事件，通常 0-4 条）
└── currentStateSnapshot（覆盖写，1 条）
```

**topicEvolution 跨会话匹配**：
- 用户说"继续上次那个故事" → ANALYZING 识别出 topicId（从 contractStore 按用户 + 主题相似度匹配）
- 匹配逻辑：先用 DET 关键词匹配（基于 topicIndex 表的 topicName 和 keywords），关键词匹配不到时降级为 embedding 语义匹配
- 匹配阈值由 topic_evolution_match_threshold 控制（默认 0.65）
- 匹配到已有的 topicEvolution → 注入最新 stateSnapshot + 分层注入包，模型接着走
- 匹配不到 → 新建 topicEvolution，从零开始

**topicEvolution 关联发现**：
- 建立方式：手动建立（用户说"这个专利是基于之前那个XX的"）或自动发现（ANALYZING 检测到当前输入与已有主题的关键词重叠度高）
- 关联类型（8种）：prerequisite / reference / derivative / conflict / cooperative / replacement / variant / evolution
- 所有关联关系必须经过用户确认后才能生效——用户可修改关联类型、编辑摘要、或拒绝建立关联
- 注入时，只拉取关联摘要卡片（inheritedPoints / avoidPoints / reason），不拉取关联主题的原始 evt 链

**显式纪律**：调度器只调接口、不内联拼接逻辑，minScoreThreshold 通过 constitution.getTunable('min_match_score') 传入，currentTaskType 从 activeSession.taskType 读取。

模型返回 turnType + askingField（按 taskType 语义）+ changeLevel 预判 → 调度器按 turnType 调度。turnType="complete" → 第3层。

#### 第3层：DET 值域复验（四项检查，0 次 LLM，仲裁权归调度器，按 taskType 分支）

- → 通过：EXECUTING → LISTENING
- → 失败：CLARIFYING（复用模型 message 转发用户）

**四项降级链**（通用底线，不可缩减）：

| 项 | 执行体 | 时机 | 说明 |
|----|--------|------|------|
| 检查1：L1 结构校验 | DET | complete 时 | JSON.parse + Schema 校验，失败→重试1次→熔断→硬编码回复。taskType 无差异 |
| 检查2：DET 值域复验 | DET | complete 时 | field_based 做值域校验（金额上限、时间格式、必填字段…），topic_based 做输出格式校验（JSON Schema / 文本格式 / 字数 / 结构…）。规则来源：环节宪法 @section validation 的结构化 JSON |
| 检查3：logprobs 检查（v6.1 修正：替代 logprobs 检查） | DET | complete 时 | v6.1：ANALYZING 用强制选择+logprobs，DET 读取 logprobs 转换为 probability 做阈值裁决（probability < logprob_threshold → intent=other）。IN_SESSION 的"硬撑式确认"检测保留 pattern 扫描词表。冷启动期间采用软拦截——标记 warning 附加到返回中，不直接拦截；冷启动结束后恢复硬拦截。原 logprobs 检查因 LLM 自由生成 confidence 不可靠已废弃 |
| 检查5：硬编码兜底 | DET | 熔断时 | 所有校验失败后的最终兜底，硬编码回复 |

**logprobs 检查冷启动软拦截机制**：
- 冷启动观测窗口（默认 50 会话）内：probability < logprobs_threshold 时，不直接拦截 complete 结果，而是标记 `logprob_warning` 并附加到模型返回中，如"[提示：模型对此结果概率较低（{probability}），请审查]"
- 冷启动期间实时观测指标：监测 logprobs 拦截率等
- 应急解冻：若指标异常（如 logprobs 拦截率 > cold_start_emergency_threshold，默认 0.5），触发人工确认后调整参数
- 冷启动结束后恢复硬拦截，解冻后 P 反馈控制自动调节

**仲裁权（架构机制 M3）**：当模型内部校验（第一道）与态控调度器 DET 复验（第二道）结果冲突时，以态控调度器 DET 为准。

#### 上下文管理器 三接口

| 接口 | 时机 | 职责 | 状态机传入 |
|------|------|------|-----------|
| buildPromptContext() | 每轮发消息给模型前（读） | 按三层注入+硬门控+语义匹配+importance截断算出本轮应拼入的上下文片段。roomConversationLog 按环节 @importance 分级保留（critical=10轮, high=6轮, normal=3轮, low=1轮）；规则注入按 ruleNode.importance 排序（critical→high→normal→low）；同 importance 按 strengthens 权重 | contextGraph, turnHistory, currentAskField, currentTaskType, currentTopicId, currentStepName（field_based 按 stepName 匹配 contractEvolution 规则）, contractStore（内部按 topicId/stepName 查询 domainRules/relatedTopics/topicEvolution/sessionCheckpoints，态控调度器不预组装）, tunableParams |
| updateGraph() | 每轮模型返回后（写） | 更新逻辑拼接图，记录本轮对话和校验结果 | userInput, modelTurn, turnType, askingField, changeLevel, validationResult |
| archiveGraph() | 偏离/放弃/完成时（归档） | 归档当前上下文图，新建空图 | — |

**拼接算法归 上下文管理器 原则**：即使 上下文管理器 未实现，调度器也只能走接口占位（直接拼最近 N 轮原文），严禁把拼接逻辑写回 scheduler。

### N6（节点6）数据传递协议 + 守卫规则 + topicEvolution/sessionCheckpoint 存储

- **输入**：状态转移图 + taskTypeFinalized
- **输出**：
  - 每跳的携带数据 / 清空数据表
  - 全局守卫（唤醒/退出/取消，可配置浅匹配器）
  - DET 关键词扫描（每轮，offTaskSuspicion 标记）
  - IN_SESSION 内守卫（DET 复验 + 偏离暂存）
  - topicEvolution 存储结构（contractStore 扩展）
- **责任人**：智能体开发员

#### 状态转移数据协议

| 转移路径 | 携带数据 | 清空数据 | 说明 |
|---------|---------|---------|------|
| LISTENING → ANALYZING | 用户原始输入 | 无 | 将输入传递给意图识别 |
| ANALYZING → IN_SESSION | intent, taskType, topicId（topic_based时）, extracted | 无 | 将意图、taskType、topicId 和已提取字段带入环节。ANALYZING 的 contract-in 副本同步写入 contractStore。topic_based 任务同时初始化 topicEvolution 记录 |
| IN_SESSION → EXECUTING | result（complete 时） | 无 | 将完整结果写入数据库/输出文件 |
| IN_SESSION → ANALYZING | offTaskInput | activeSession | 偏离→重新意图识别。topic_based 任务在 off-task 时仍追加 event 到 topicEvolution（记录偏离前的状态） |
| IN_SESSION → LISTENING | 无 | activeSession | 放弃。topic_based 任务在 giveup 时使用模型判定的 changeLevel 追加 event，stateSnapshot 标记为 abandoned，推入提炼待处理队列 |
| EXECUTING → LISTENING | _lastRecord, topicEvolution 更新（topic_based时） | 无 | 保留上一条记录用于"记错了"。topic_based 任务使用模型判定的 changeLevel 追加 event，标记 checkpoint，推入提炼待处理队列 |
| 任意状态 → ANALYZING（逃生舱） | contractStore.analyzingContractIn | activeSession | 意图级纠错：强制回滚至 ANALYZING 重新分类（可能伴随 taskType 变化） |

关键规则：状态间的数据传递必须是显式的。不能依赖 LLM 的记忆来传递数据。环节内的数据（collectedFields、turnHistory）由态控态控调度器的 activeSession 维护，环节完成后清空。topicEvolution 跨会话持久化，不清空。

**turnHistory 要求**：
- 上限 20 轮（受 turnHistory_limit tunable 管）
- 每条结构 `{userInput, modelTurn, turnType, askingField, changeLevel}`
- askingField 字段必须落地，供 上下文管理器 硬门控保护使用
- changeLevel 字段必须落地（两种 taskType 模式均适用），供 topicEvolution/sessionCheckpoint 使用
- changeLevelReason 字段必须落地（major/minor 时必填，供 DET 格式校验防偷懒）
- contextGraph 为有向关联图（环节内逻辑拼接图）
- activeSession 含 taskType 字段和 topicId 字段

#### 守卫规则

**全局守卫（所有状态通用）**：

| 输入模式 | 触发动作 | 优先级 |
|---------|---------|--------|
| 唤醒词 | IDLE 态下进入 LISTENING | 最高 |
| 退出词 | 放弃当前流程，进入 CLOSING | 最高 |
| 取消词 | 放弃当前子流程，回到 LISTENING | 最高 |

全局硬匹配采用可配置的浅匹配器（子串匹配/编辑距离/同义词组），词表是产品参数，需维护日志。此为元指令集的口令层。

**DET 关键词扫描**：每轮用户输入执行 DET 关键词扫描，检测跨意图词汇，附加 offTaskSuspicion 标记到模型输入中。不强制覆盖模型判断。

**IN_SESSION 态守卫**：
- 态控调度器行为：接收模型返回的 turnType（含 changeLevel 预判），进行状态调度
- DET 值域复验：模型返回 complete 时，调度器对 result 执行第二道值域校验/格式校验（四项检查 + 硬编码兜底，按 taskType 分支）。冷启动期间 logprobs 检查为软拦截
- 偏离处理：模型返回 off-task 时，暂存 collectedFields（默认 24h），重新意图识别。topic_based 任务在 off-task 时仍追加 event

#### 意图级纠错（逃生舱）

当用户反馈表明当前意图完全错误时：
- 状态机从 contractStore 取出 ANALYZING 节点的原始 contract-in 副本
- 重新实例化 ANALYZING 节点（注入原始 contract-in + 宪法）
- 重新执行意图识别，产生新的 intent（可能伴随 taskType 变化）
- 清空当前 activeSession，按新 intent 和新 taskType 重新分发

#### contractStore 物理存储

- SQLite 单文件数据库，每个用户/项目一个 .db 文件
- WAL 模式支持并发读 + 单写
- 数据库文件路径：`<project_root>/.contractstore/<user_id>.db`
- 核心表结构：

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| analyzing_contract_in | ANALYZING 节点的原始 contract-in 副本 | — |
| analyzing_contract_out | ANALYZING 节点的 contract-out 副本 | — |
| topicIndex | 检索索引表 | topicId（主键）、topicName、keywords、creatorUserId、createdAt、lastActiveAt、relatedTopics（JSON） |
| topicEvolution | 主题演化主表 | topicId（主键）、intent、userId、stateSnapshot（JSON）、createdAt、updatedAt |
| topicEvolutionEvent | 事件主表 | eventId（主键）、topicId（外键）、changeLevel、nodeId、turnType、timestamp、stateChange（JSON）、askingField、modelOutputSummary |

#### 异步规则提炼机制

**触发的三个条件**（满足任一即触发，由独立后台进程执行，不在状态机流转链路内）：
- 时间窗口触发：每日固定时间（如凌晨2点）。cron 表达式由 rule_mining_schedule tunable 配置（默认 `0 2 * * *`）
- 数量阈值触发：每积累 N 个同类主题的 checkpoint 触发一次。由 rule_mining_batch_threshold tunable 配置（默认 5）
- 手动触发：用户在系统设置界面的"知识库更新状态"面板中点击"立即提炼"

**提炼流程**：
1. 候选规则生成：提炼引擎读取待提炼主题的 topicEvolution 和关联关系，提取"稳定的、可复用的行为模式"，生成候选规则，存入 ruleCandidate 池
2. 交叉验证：候选规则需要多条同类主题交叉验证后才提交用户确认。验证通过率 ≥ domain_rule_validation_threshold（默认 0.7）
3. 用户确认：系统展示候选规则及证据，用户可采纳、修改或拒绝。确认后的规则存入 contractStore.domainRules 库
4. 规则注入：同类型新主题初始化时，上下文管理器自动注入已生效的相关领域规则。field_based 的 contractEvolution 规则按 stepName 精确匹配注入——机制相同，唯一差异在匹配键

**知识真空期**：规则提炼异步执行意味着领域规则不保证跨相邻任务的即时传递。系统设置界面提供"知识库更新状态"指示器，显示上次提炼时间、待提炼主题数、下次计划提炼时间，并提供"立即提炼"手动触发按钮。

**topicEvolution 与认知局部性的关系**：topicEvolution 不是对认知局部性的违反。field_based 的 contract-out JSON 同样是跨节点共享的——节点之间不共享的是"推理过程"，但共享"收敛结果"。topicEvolution 的 event 序列就是 topic_based 的"收敛结果"——它记录的不是模型的推理过程，而是"主题在这个节点里完成了什么变化"这一确定性的状态变更。关联摘要不是"把隔壁主题的数据拿过来"，而是"从隔壁主题的数据里提炼出对当前主题有用的几条结论"。关联主题的原始 evt 链不属于当前主题的最小充分条件集，不注入。

**领域规则体系（v5.1 新增）**：领域规则由 v5.0 的平铺列表升级为规则网络图。规则按来源分为三层（scenario_seed 场景种子 / user_initiated 用户口述 / model_inferred 模型推断），按权限分为 L0-L3 四级（钢印/基线/用户确认/模型推断），节点间通过 depends_on / conflicts_with / strengthens 三种关联边建立关系。注入方式从全量注入改为按 topicPath 前缀子图匹配——上级可见于下级，下级不可见于上级/兄弟。规则提炼由独立的 N10 后台进程异步执行，自动推断关联边后经交叉验证和用户确认存入 domainRules。

**off-task 双重保障**（v6.0 新增，v6.1 修正：2026-07-17）：

N5 在 IN_SESSION 出口增加 off-task 验证步骤：

1. **保障1（主力——宪法强制 + 上下文管理器优先注入，IN_SESSION 入口）**：根宪法第 1 条（环节隔离）要求模型每轮首要判断"是否本环节任务"，非本环节→立即返回 off-task。上下文管理器将环节隔离声明以最高优先级注入 prompt
2. **保障2（辅助——ANALYZING 重新识别验证，IN_SESSION 出口）**：模型返回 off-task 后，调度器不直接路由 → DET 只做非空检查（门卫，不做语义验证）→ 送回 ANALYZING 重新识别 offTaskInput → ANALYZING 的重新识别结果就是验证。v6.1 修正原因：DET 做不了语义验证，LLM 自由生成的 confidence 不可靠
3. **显式切换口令（M1 switch，v6.3 强化：S3 + roomStateIndex）**：用户说"切断房间" → DET 硬匹配 → 调度器触发 S3 释放当前房间 KV Cache + `buildRoomIndex()` 生成全窗口房间状态摘要 → 注入 ANALYZING → 创建新推理会话。ANALYZING 获得跨房间全局视野后可针对性建议（如"您记账房间还有一条未完成"）。显式口令是首选切换机制（100% 确定性），off-task 检测是兜底。
4. **设计原则**：不追求 100% 拦截，而是"降低漏判 + 拦住误判 + 回溯兜底"。DET 不在环节内主动介入——这是对模型自治权的僭越。DET 只在出口做门卫检查（非空），语义验证由 ANALYZING 重新识别完成

### N7（节点7）根宪法（v6.0 适配 v5.0）

- **输入**：场景安全/收敛/仲裁需求 + taskTypeFinalized
- **输出**：root-constitution.json
- **责任人**：设计者

**v5.0 核心变更**：根宪法从 8 条精简为 4 条，约束对象统一为"所有环节模型（ANALYZING + IN_SESSION）"。原第2条（元指令集）、第5条（降级链）、第6条（仲裁权）、第7条（节点转换守卫）移入架构机制（M1-M4）。原第8条（追问模板）下放到环节宪法 @section ask-rules。

**根宪法 4 条**（不可逾越的系统红线，由设计者写死）：

| 编号 | 条款 | 内容 | 可被环节宪法覆盖 |
|:--:|------|------|:---:|
| 1 | **环节隔离** | 只能做本环节任务，非本环节任务返回 off-task。本环节任务由环节宪法第一条定义。ANALYZING 只能做意图识别。IN_SESSION 只能做本环节任务。 | ❌ 不接受覆盖 |
| 2 | **输出格式** | 模型实例返回调度器的结果必须是符合 turnType schema 的结构化 JSON | ❌ 不接受覆盖 |
| 3 | **收敛义务 + 坚持真实判断** | 所有 LLM 节点必须在当前环节内收敛，不伪造数据、不假装理解、不迎合用户、不无中生有 | ❌ 不接受覆盖 |
| 4 | **不编造功能** | 所有 LLM 节点不得生成用户未交代、系统不具备的功能或信息 | ❌ 不接受覆盖 |

**ANALYZING 的环节隔离特化**：ANALYZING 的"本环节任务"定义为"意图映射"。如果输入无法映射到任何已知 intent，返回 `intent=other`（低 probability（来自 logprobs））。注意：ANALYZING 只产出 intent，不产出 turnType——turnType 是 IN_SESSION 的输出格式。`turnType=off-task` 是 IN_SESSION 内模型检测到偏离时的输出。

#### @section output-schema 说明

**统一输出格式**（含 askingField 双模式语义 + changeLevel 字段 + topicId 字段，topic_based 模式）：

```json
{
  "turnType": "ask | reply | complete | off-task | giveup | validation_failed",
  "askingField": "null | 字段名(field_based) | 主题标签(topic_based)",
  "changeLevel": "null | major | minor | invalid",
  "changeLevelReason": "null | 一句话理由（major/minor 时必填。DET 不验证语义正确性）",
  "message": "模型对用户说的话（自然语言）",
  "validationResult": "null | { field, issue, userInput }",
  "collectedFields": {},
  "offTaskInput": "null | 用户偏离输入",
  "result": "null | 完整结构化结果"
}
```

**askingField 说明**：
- field_based 模式：turnType="ask" 时为当前追问的字段名（供 上下文管理器 字段级硬门控保护使用）
- topic_based 模式：turnType="ask" 时为模型动态生成的主题锚点标签（供 上下文管理器 三层注入 + embedding 两阶段检索保护使用）
- turnType="validation_failed" 时为校验失败的字段名或主题维度
- 其他 turnType 时为 null

**askingField 主次标注规则**：field_based 任务中，如果几个字段是强关联的（如"金额+币种"），可以一轮问两个，但要在 askingField 里标清楚主次，格式为 "amount+currency"。topic_based 任务中，askingField 通常为单个主题标签。

**changeLevel 判定规则（topic_based 模式必填）**：
- invalid：当且仅当本轮输出完全撤销了上一轮的实质性变更（如上一轮新增实体A，本轮删除实体A且无其他衍生变更），且未产生新的有效信息时
- major：涉及核心实体的增删、任务目标的根本性转向、或引入了全新的世界观/规则设定时
- minor：凡不属于上述两者的情况，一律判定为 minor

判定边界的重要区分：用户反复修改同一属性（如"改林霜的性格，改了三次才满意"），只要方向不是完全互斥（加→删），就算 minor 累积调整——判定标准是本轮操作是否实质上撤销了上一轮的变更且未产生新的有效信息，而非修改次数。

**兜底机制**：若模型未返回 changeLevel 或格式错误 → 调度器默认标记为 minor（最安全的容错：进入累积压缩流，不丢失信息）。

**topicId 字段声明（topic_based 模式必填）**：
- topic_based 模式的 ANALYZING contract-out 必须包含 topicId 字段
- topicId 由 ANALYZING 阶段 LLM 生成的 topicSlug + userId + intent 组合而成
- topicId 是 topicEvolution 的主键，供后续节点读写主题演化记录
- topic_based 任务自动启用 topicEvolution 三层递进机制（分层留存 + 主题关联 + 异步规则提炼）

#### @section validation 结构化 JSON 格式

**field_based 模式的 validation 写法**：

```json
{
  "rules": [
    {"field": "amount", "type": "range", "min": 0, "exclusive_min": true, "error_tag": "amount_invalid"},
    {"field": "amount", "type": "range", "min": 0, "max": 999999, "error_tag": "amount_too_large"},
    {"field": "time", "type": "format", "pattern": "^(昨天|今天|明天|\\d+号)$", "error_tag": "time_format_invalid"},
    {"field": "time", "type": "not_future", "error_tag": "time_future"},
    {"field": "quantity", "type": "range", "min": 0, "exclusive_min": true, "error_tag": "quantity_invalid"}
  ],
  "required_fields": ["category", "amount", "time"]
}
```

**topic_based 模式的 validation 写法**：

```json
{
  "rules": [
    {"field": "lines", "type": "array_length", "exact": 4, "error_tag": "line_count_violation"},
    {"field": "lines", "type": "char_count_per_element", "exact": 7, "error_tag": "char_count_violation"},
    {"field": "rhyme", "type": "custom", "function": "check_rhyme_scheme", "error_tag": "rhyme_violation"}
  ],
  "required_fields": ["title", "lines", "style", "theme"]
}
```

**规则类型说明**：range / format / not_future / enum / array_length / char_count_per_element / custom / required

| 维度 | field_based 模式 | topic_based 模式 |
|------|----------------|----------------|
| @section taskType | field_based | topic_based |
| @section convergence | "将用户口语收敛到 {字段1, 字段2, …}" | "将用户输入收敛到 [主题空间] 内的 [输出类型]" |
| @section field-rules | ✅ 必填 | ❌ 省略 |
| @section topic-boundary | ❌ 省略 | ✅ 必填 |
| @section parsing | ✅ 有（硬解析优先） | ❌ 无（语义理解为主） |
| @section validation | 结构化 JSON，值域校验规则 | 结构化 JSON，格式校验规则 |
| @section ask-rules | 按字段顺序追问 | 按主题探索路径追问 |

#### 环节宪法示例：记账宪法（field_based）

```
<!-- @constitution record-session -->
<!-- @section taskType -->
field_based

<!-- @section convergence -->
收敛方向：将用户口语记账指令收敛到 {category, amount, time, quantity, unit}

<!-- @section role -->
【角色】
你现在是用户的记账助手。用户正在记账，你的任务是引导用户完成一笔完整的记账。你需要主动询问缺失的字段，直到所有必采字段齐全。
你的上下文窗口由态控调度器管理，你只消费调度器提供的上下文，不自行拼接。

<!-- @section field-rules -->
【字段采集规则】
简单模式必采字段（按顺序追问）：种类 → 金额 → 时间。数量默认1。
细致模式必采字段（按顺序追问）：种类 → 金额 → 时间 → 数量(含单位)。

<!-- @section parsing -->
【字段解析规则】
- 金额、数量字段：优先尝试数字正则匹配。匹配成功→进入值域校验，通过则直接填入。
- 时间字段：优先尝试时间词词典匹配（昨天/今天/明天/几号）。
- 种类字段：直接记录用户原话，不做分类转换。
- 硬解析失败时，追问用户确认。

<!-- @section validation -->
{
  "rules": [
    {"field": "amount", "type": "range", "min": 0, "exclusive_min": true, "error_tag": "amount_invalid"},
    {"field": "amount", "type": "range", "min": 0, "max": 999999, "error_tag": "amount_too_large"},
    {"field": "time", "type": "not_future", "error_tag": "time_future"},
    {"field": "quantity", "type": "range", "min": 0, "exclusive_min": true, "error_tag": "quantity_invalid"}
  ],
  "required_fields": ["category", "amount", "time"]
}

<!-- @section ask-rules -->
【追问规则】
- 一次只问一项，按顺序。
- 用户回答 → 填入该字段，继续检查下一项。
- 用户不说具体时间 → 默认"今天"。
- 一直追问到所有必采字段齐全。

<!-- @section completion -->
【记账完成条件】
所有必采字段齐全后，回复一句确认，然后将结果以结构化 JSON 返回给调度器（turnType="complete"）。

<!-- @section modification -->
【修改】
用户说"记错了"/"不对"/"改一下" → 追问"哪个需要改？种类、金额还是时间？"
用户指定字段（如"改金额"）→ 回到该字段重问。

<!-- @section off-task-detection -->
【偏离检测】
如果用户输入明显不属于当前记账任务（如查询、删除、闲聊、退出），返回 turnType="off-task"。

<!-- @section giveup -->
【用户放弃】
用户说"算了"、"不记了" → 返回 turnType="giveup"。

<!-- @section output-schema -->
【统一输出格式】（field_based 模式）
{"turnType":"...", "askingField":null|"category"|"amount"|"time"|"quantity", "changeLevel":"null|major|minor|invalid", "changeLevelReason":"null|一句话理由（major/minor时必填）", "message":"...", "validationResult":null|{...}, "collectedFields":{...}, "offTaskInput":null|"...", "result":null|{...}}
```

```
- turnType="ask" 时，askingField 为模型动态生成的主题锚点标签

- 其他 turnType 时，askingField 为 null

**changeLevel 判定规则（两种 taskType 模式均适用）：**
- invalid：当且仅当本轮输出完全撤销了上一轮的实质性变更（如上一轮加入"落叶"意象，本轮删除"落叶"且无其他衍生变更），且未产生新的有效信息时
```

**参数默认值说明**：
- logprobs_threshold 默认 0.4——0.6 太严，正常模型 confidence 普遍在 0.7-0.9，0.6 几乎拦不到；0.4 更合理，只拦真正低置信度的"硬撑"。冷启动观测窗口内采用软拦截
- topic_anchor_similarity_threshold 默认 0.6——过低（如 0.3）会导致过多历史轮次被保护，上下文窗口膨胀；过高（如 0.8）会导致保护力度不足
- topic_keyword_overlap_threshold 默认 0.2——过低（如 0.1）粗筛通过太多失去意义，过高（如 0.5）粗筛漏报多
- topic_evolution_match_threshold 默认 0.65——略高于 topic_anchor_similarity_threshold（0.6），因为跨会话匹配需要更高精度

**冷启动冻结机制**：
- 冷启动期间参数冻结，但实时观测指标：监测 logprobs 拦截率、topicEvolution event 写入频率、minor 压缩频率、sessionCheckpoint 恢复率等
- **冷启动期 logprobs 检查采用软拦截**：probability < logprobs_threshold 时，不直接拦截 complete 结果，而是标记 logprob_warning 并附加到模型返回中让用户可见（如"[提示：模型对此结果概率较低（0.35），请审查]"）。冷启动结束后（观测窗口期满或应急解冻后）恢复硬拦截
- 应急解冻机制：若指标异常（如 logprobs 拦截率 > cold_start_emergency_threshold），触发"应急解冻"——人工确认后调整参数
- 解冻后 P 反馈控制自动调节：参数调整 = pid_kp × (目标拦截率 - 实际拦截率)（当前仅启用比例项 Kp）
- 所有调整记录变更日志：时间戳、操作者、参数名、旧值、新值、调整原因、触发指标

**系统设置界面**：每个宪法中声明的 tunable 参数，在智能体的系统设置界面中自动生成配置表单。topic_based 场景的"知识库更新状态"面板显示上次提炼时间、待提炼主题数、下次计划提炼时间，并提供"立即提炼"手动触发按钮。

**跨参数联动约束（v4.7，详见总纲 §10.1）**：N10 的 tunables.json 中可选声明 `cross_param` 规则——对 tunable 参数间语义依赖做跨参数校验。规则分为 `single_param`（range/enum，违规硬拒绝）和 `cross_param`（语义校验，违规默认 warning，允许用户强制保存）。cross_param 规则清单含 5 条：critical_room_context_budget_check、topic_match_precision_order（hard_reject）、cold_start_observation_confidence、rule_mining_dual_trigger、turnHistory_pending_fields_ttl。

### N16（节点16）回溯触发器

**八类信号 → 处置**：

| 信号类型 | 处置路径 |
|---------|---------|
| 边界-状态矛盾 | 回溯 N1（重审边界） |
| 宪法-状态矛盾 | 回溯 N7-N10（修 bug，无累积阈值） |
| 守卫-边界矛盾 | 回溯 N6（调词表/护栏） |
| 执行体错配 | 回溯 N3（见下方检测信号 A-Q） |
| 参数漂移 | 调 tunable，不回溯 |
| taskType 误判 | 回溯 N1（重判 taskType）。误判率>0% → 立即回溯 |
| 边界覆盖缺口 | 功能范围外输入无处理路径或护栏响应质量不合格 → 回溯 N1 |
| topicEvolution 机制失效 | 回溯 N3/N5/N9（见下方检测信号 J-Q） |

**关键分流逻辑**：如果系统在"正确执行当前边界和规则"的前提下，监控数据仍然不好看——先去参数调节宪法看是不是参数漂移。不要回溯，不要改边界，去调参数。

**执行体错配矛盾的检测信号（A-M 共 13 项）**：

| 信号 | 说明 | 处理方式 |
|------|------|---------|
| A | DET 复验拦截率高 | 回溯第二步或第三步，检查环节宪法的值域校验/格式校验规则是否与态控调度器 DET 规则一致 |
| B | validation_failed 多但用户修正率低 | 回溯第三步，检查环节宪法的值域校验/格式校验规则是否过于严格 |
| C | logprobs 拦截率高（>10%） | 回溯第三步，检查环节宪法 prompt 或调整 logprobs_threshold |
| E | topic_based 主题漂移 | 回溯第三步，调整 topic_anchor_similarity_threshold 或 topic_keyword_overlap_threshold，或检查主题边界声明 |
| F | taskType 误判率高（>2%） | 回溯第一步，检查 taskType 初判标准 |
| G | 意图识别不准（意图级纠错触发率>5%） | 回溯第三步，检查意图识别宪法 |
| H | 模型校验过松（第一道通过率高但 DET 复验拦截率高） | 回溯第三步，检查环节宪法校验规则与态控调度器 DET 规则一致性 |
| I | 模型校验过严（频繁返回 validation_failed 但用户修正率低） | 回溯第三步，检查环节宪法校验规则是否过于严格 |
| J | topicEvolution event 写入频率过低（<1 event/会话） | 检查调度器是否正确追加 event |
| K | topicEvolution 跨会话匹配失败率高（>70%） | 检查 topic_evolution_match_threshold 或 stateSnapshot 描述 |
| L | topicEvolution minor 压缩异常（压缩频率过高或注入包大小持续增长） | 检查 topic_evolution_minor_compaction_threshold |
| M | invalid 丢弃率过高（>30%） | 检查环节宪法的 changeLevel 判定规则 |
| N | topicIndex 检索失败率高（用户明确提到主题名但关键词匹配未命中） | 检查 topicIndex 索引是否正常更新 |
| O | 规则提炼噪声率高（候选规则确认率<30%） | 调整 domain_rule_validation_threshold |
| P | changeLevel 判定异常（invalid 占比 > 30% 或 major 占比 < 3% 或 changeLevelReason 空值率 > 10%） | 检查环节宪法的 changeLevel 判定规则；若 changeLevelReason 持续为空，检查模型是否未理解输出格式要求 |
| Q | 规则提炼队列积压（待处理队列持续增长） | 调整 rule_mining_schedule、rule_mining_batch_threshold 或 rule_mining_batch_size |

**回溯时带回的数据**：回溯必须携带下游的具体矛盾实例——触发矛盾的状态节点定义、无法自洽的宪法条款原文、误杀统计、词表调整方案记录、taskType 误判的具体案例、边界覆盖缺口的具体输入类型、topicEvolution 机制失效的具体证据等。

**重审边界的合法决策**（只有三种）：
1. 收缩"不做什么"：将原来不做的事移入"做什么"
2. 重新定义边界护栏策略：从"硬拒"升级为"软着陆"。硬约束：软着陆的分支目标必须已在状态转移图中存在
3. 承认该输入类型为合法但降级处理：分配显式的降级路径

**迭代终止条件**：越界率 < 阈值 / 无新增矛盾 / 达最大回溯轮次（默认 3 轮）。同类矛盾信号须在 N≥3 次真实会话中出现，或单次撞击的误杀率 > X%，才触发回溯。单一样本不构成回溯依据。例外：宪法-状态层面的自洽性矛盾属于逻辑 bug、taskType 误判属于结构性错误、边界覆盖缺口属于结构性错误——不适用累积阈值规则，一旦发现立即回溯。

## 附录 D：align-session 可选扩展规范

align-session 是可选扩展，非态控架构标准能力。若场景含意图校准需求，可启用 align-session 扩展。启用时需：
1. 额外编写 align-session.json 宪法
2. 在路由表中增加 align 相关条目
3. 在 tunable 参数中增加 divergence_threshold（默认 0.7）
4. 同步更新场景的边界清单
5. 在 N5 第0层之后插入第0.1层（align 静默模式触发与处理）

**align-session 核心机制**（仅在启用时生效）：
- 静默输入：/align 元指令触发后，挂起语义管道，所有输入走静默缓冲区，直到完成指令或退出词
- 多模型并行仲裁：主次模型并行 + 规则模型串行
- 锚点确认：主/次模型标注 core_anchor，用户确认后沉淀 user_confirmed_anchor
- 迭代裁决：divergence_threshold 判定分歧度
- 对齐数据包沉淀：rounds[] / divergence_points / user_arbitration

**与核心架构的边界**：align-session 启用时不修改根宪法第1-4条及架构机制 M1-M4（核心四条 + M1-M4 不含 align 内容），仅在路由表和 tunable 参数中扩展，并在 N5 插入第0.1层。align-session 不启用时，核心架构完全不受影响。
