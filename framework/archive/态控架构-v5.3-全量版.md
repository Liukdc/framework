# 态控架构 v5.3（全量版）

**定位**：智能体设计总纲——回答"如何系统化地构建一个计算输出可控和成本最优的智能体"。

**态控架构的目的**：在状态机的调度约束下让所有场景大模型（智能体）和用户适配适合的输入和输出。

**文档版本**：v5.3

## 一、架构背景

### 1.1 模型是一个无状态的有自己权重分布概率计算单元

模型的计算能力是有限的，而且受到训练结果约束，权重分布是不均衡的。无论是模型本身的计算资源，还是模型外部的计算载体，还有消耗的能源都是有限的，但是要计算的内容是趋向膨胀的。所有 AI 应用本质是消耗直接能源和间接资源的计算行为，每次单位计算资源的消耗是基本固定的，所以当次计算内容对任务的匹配度越高越好。

AI 应用追求计算性价比的路径：首先找到当次输入任务最匹配的内容，然后去合理分配这些内容，把跟任务最匹配的内容放在最适合的位置，然后按这个标准去分配计算内容。

### 1.2 任何任务皆为受限计算

在实际应用中，不存在"纯无限制的自由计算"。写诗、写小说等创意任务同样具有方向、主题与格式限制。在态控视角下，写诗与修改家具参数都是受限空间内的收敛过程，区别在于收敛空间的连续性——前者是在连续语义空间中采样满足约束的样本，后者是在离散参数空间中定位唯一正确路径。

### 1.3 认知局部性原理（输入侧）

任何信息处理主体（人、LLM、确定性程序），在 t 时刻处理节点 N 时，最优输入集 = N 在当前目标下的最小充分条件集（contract-in（输入契约）），不是全集 Γ。这一条是认知必然：

A 人：工作记忆约 4 个组块（Cowan, 2001），逼人看全局反而信息过载、分析瘫痪。

B LLM（推理阶段）：Lost in the Middle（中间迷失）效应使中部信息召回率显著低于头尾；近因偏置使尾部信息被不成比例地加权；长上下文幻觉率随长度单调递增。喂全局是跟注意力机制对着干。

C 确定性程序（DET）：函数纯度要求 f(x) 只依赖 x，不依赖全局变量。

## 二、架构哲学与设计原则

态控架构是一套以"认知局部性"为哲学根、面向 LLM 智能体收敛空间设计的工程框架。态控架构认为智能体 = 数据流网络 + 三类执行体（确定性程序 / 概率性 LLM / 人类选择）。态控架构根本立场："以数据流为中心，LLM 是被调度的资源"——它把上下文剪裁权从概率模型内部彻底剥离，移交给外部确定性状态机。这个状态机不关心每个节点跑的是什么模型、处理什么模态、由谁执行，只定义节点之间的数据格式和路由规则。每个节点被封装为自治单元。模型只需要精准匹配当前智能体场景的需求。同类执行环节可以定向后训练专用深度模型，理论上每个 intent 都可以配备专用小模型，成本从"全量大模型"降为"按需分配"。

### 2.1 数据流定义

"数据从哪里来、到哪里去、经过什么变换"；执行体定义"每个变换节点由谁执行"。每个节点是一个独立的变换函数，接收系统拼接的结构化输出（contract-in（输入契约）），根据当前节点的场景描述进行处理，产生系统需要的结构化数据（contract-out（输出契约））。节点之间不共享内部状态，不依赖全局上下文。每个节点只拥有它收到的输入数据和它产生的输出数据。

### 2.2 长上下文管理原则



**前缀缓存纪律（成本效率关键）**：上下文拼接必须遵循"稳定前缀在前、追加式日志在后"的拼接铁律——宪法条款 + 环节宪法 @section role + 领域规则中不变的部分作为稳定前缀（prefix），roomConversationLog 最近 N 轮作为追加式尾部（append-only）。稳定前缀在 KV Cache 中可跨轮复用（自动前缀缓存命中），仅在切换房间或规则变更时重建。未遵循此拼接顺序的实现会每轮触发全量 prefill，计算成本不可接受。详见 §2.2-A。

本架构通过数据流切段、契约传递、每轮按规则拼接主动管理上下文长度。每个环节的上下文窗口由上下文管理器控制在高效区间——从本环节 roomConversationLog（全量记录）/ 对话快照 / 已确定内容 / 领域规则 + 其他环节与本环节关联内容（例如基于字段的场景中前置环节的产出物和基于主题场景中的规则网络图）这些内容中按上下文管理器的拼接机制进行拼接。

规模增强（scale up 才需要）：当单一环节的计算需求超出单实例能力时，启用并联实例化（多实例并行处理 + 结果合并）与长上下文显式管理（分段加载 + 滚动窗口）。并联实例化的触发条件、实例间通信协议、结果合并策略由场景设计阶段在环节宪法 `@section scale` 中声明，非运行时动态决定。

### 2.3 受控灵活性原则

宪法主体在设计时写死，但允许在运行时通过预设的 tunable（可调参数）槽位调整指定参数。调整范围受 range（范围）/ enum（枚举）限定，变更日志可追溯。

### 2.4 环节级自治原则与上下文管理器三层职责

态控调度器管路由与全量记录，上下文管理器管上下文的匹配→决策→拼接，模型管环节内交互。三个实体各司其职：

- **态控调度器**：只在环节边界执行路由和记录（详见 §2.4-A 调度器职责声明）
- **上下文管理器**：在环节入口执行**三层确定性职责**——
  1. **匹配层**：按 topicPath / stepName 做规则子图匹配、冲突检测、条件检查（确定性计算）
  2. **决策层**：按 importance / immutableLevel / strengthens 权重决定注入哪些内容（确定性计算）
  3. **拼接层**：把决策结果组装成 prompt，做 token 预算分配与截断（确定性计算）
- **环节内执行体（模型）**：环节内的多轮追问、字段校验、偏离检测由模型实例在局部宪法约束下自主完成
- **DET 兜底**：态控调度器在关键节点执行确定性兜底校验

上下文管理器是确定性状态机，做匹配→决策→拼接三步，**但不执行环节宪法**——不决定模型怎么追问、怎么校验、怎么收敛。上下文管理器的实现形态由场景复杂度决定（确定性程序 / LLM / 混合），属于实现层选择，架构层不固定其形态。

**§2.4-A 态控调度器职责声明**：态控调度器承担三类职责——
1. **路由**：状态转移、回滚、偏离处理、意图级纠错
2. **全量记录**：roomConversationLog / sessionCheckpoint / contractEvolution / outputRegistry 的写入触发与协调
3. **资源协调**：S3 会话回收（KV Cache 释放）（调内存管理模块）、跨房间文件复制（调文件操作模块）、context-manager 调配、CLARIFYING 消息转发

调度器亲自做"路由决策"和"记录触发"，把"内存管理""文件操作"委托给专用子模块（内存管理模块 / 文件操作模块），自己只做协调调用——避免调度器变成大杂烩。

### 2.5 最小可识别单元

态控的最小可识别单元不是"三分+契约+并联+tunable（可调参数）"全套，而是——一个态控调度器 + 一个环节的每状态标执行体 + 每状态 contract-in（输入契约）/ out（输出契约） + 一部宪法。这四件套齐了，"态控"就成立。

### 2.6 房间比喻——态控的核心灵魂

态控架构的灵魂是"每个环节就是一个独立子任务房间，态控调度器会判断用户每次想去哪个房间"。

**房间里有什么**：每个环节是一个独立房间，房间里有这个环节上次讨论的全量记录（roomConversationLog——整个环节对话原文）和对话快照（stepSnapshot——collectedFields）+ 已确认的产出物 + 相关的领域规则。门一关，其他房间的内容不进入当前房间的上下文拼接——隔离对象是"其他环节无关内容"，隔离动作是"不进入上下文拼接"，不是"环节之间逻辑上不可见"。门一开，态控调度器把全量记录和对话快照 + 已确认的产出物 + 相关的领域规则和用户最新输入一起交给房间内部的上下文管理器，由上下文管理器进行内容拼接，然后输入给模型计算。上下文管理器是独立于环节宪法的专用拼接机制——它只管"怎么从态控调度器交给它的内容中组装出任务计算需要的上下文"。

**模型在这套系统里的真实位置**：模型本身是无状态的。它不知道自己上一轮输出了什么，不知道用户上次改了什么参数。房间里保存的全量记录和对话快照就是模型的上下文——"记性"不来自模型，来自系统的内容保存和分类机制。所以用户可以在房间之间自由穿梭：做完 N10 的 tunable，说一声"改一下 N1 的场景定义"，状态机立刻匹配 intent=N1，取出 N1 房间的相关内容交给 N1 的内容管理器，内容管理器拼接后递给模型。模型看到的只有 N1 房间里的内容，不掺杂 N10 的参数细节。

**roomConversationLog 的独立保留策略**：roomConversationLog 的保留期独立于 sessionCheckpoint.ttl——roomConversationLog 有自己的分级保留策略，按房间 importance 分级：

| 房间 importance | roomConversationLog 保留期 | 说明 |
|----------------|---------------------------|------|
| critical | 永久保留 | 核心房间对话不清理，支持任意时间回溯 |
| high | 90 天 | 高重要房间保留一个季度 |
| normal | 30 天 | 普通房间保留一个月 |
| low | 7 天 | 低重要房间保留一周 |

此保留策略同时适用于 field_based 和 topic_based 两种 taskType——roomConversationLog 是房间级记录，不区分 taskType。sessionCheckpoint.ttl 到期时，只清理 sessionCheckpoints 表的进度行，**不联动清理 roomConversationLog**——roomConversationLog 按自己的保留期独立过期。这保证了"自由穿梭"承诺：即使 field_based 任务的进度检查点过期了，用户返回房间时仍能从 roomConversationLog 恢复对话上下文。

**自由切换的四个支柱**：

| 支柱 | 做什么 |
|------|--------|
| 全量记录（roomConversationLog） | 每个环节的每轮对话实时落盘——切换环节时内容不丢弃，返回时完整可用（按 importance 分级保留，见上表） |
| 对话快照（stepSnapshot） | 每个环节收敛时的结构化字段快照——快速定位已确认的决策 |
| 规则提炼按环节隔离 | 每个环节的 changeLog 独立推送 ruleMiningQueue——N12 的规则提炼只消费 N12 的变更历史 |
| **产出物总索引** | **每个房间产出物的统一登记簿——跨房间可发现、可检索、可自动复制。DET 确定性查询（0 LLM）。如"元智能体的 L2 设计文档"可被专利撰写房间复制引用** |

**自由切换的适配目标**：自由切换是用户在真实使用中的实际行为。架构的价值命题是"让切换正确"（正确匹配 intent、取出正确房间内容、不串台），不是"鼓励切换"。

**这就是为什么 field_based 和 topic_based 都需要跨会话记忆**：topic_based 用 topicEvolution 管记录，field_based 用 sessionCheckpoint 管记录——两种机制互不替代，但共享同一个灵魂：模型无状态，状态机管全量记录，内容管理器管拼接，用户自由穿梭。

### 2.7 实现形态声明

本节集中声明各关键组件的实现形态属性，划清架构层固定属性与实现层可变属性的边界，避免下游实现者按默认假设做偏差。

| 组件 | 架构层固定属性 | 实现层可变属性 |
|------|--------------|--------------|
| 态控调度器 | 职责固定——路由 + 全量记录 + 资源协调（含内存管理协调、文件操作协调、context-manager 调配）；必须是确定性程序 | 路由表结构、记录存储格式、状态转移具体实现、子模块（内存管理/文件操作）的拆分方式 |
| 上下文管理器 | 职责固定——匹配→决策→拼接三层确定性职责；不执行环节宪法 | 实现形态可变——确定性程序 / LLM / 混合，由场景复杂度决定；匹配算法、决策权重、拼接策略、token 上限、降级方案 |
| 环节内执行体 | 职责固定——在局部宪法约束下完成环节内交互 | 实现形态可变——确定性程序 / LLM / 人类选择，由 taskType 和节点状态决定 |
| DET 复验 | 职责固定——确定性收敛，不调 LLM | 校验规则集、值域定义、格式 schema |
| 规则提炼引擎 | 职责固定——从 changeLog 和 topicEvolution 提取候选规则，经用户确认后入库 | 触发条件（时间窗口 / 数量阈值）——但必须保证低频性；提炼算法、交叉验证策略 |
| contractStore | 职责固定——存储契约、规则、演化记录，支持按 topicPath 子图匹配 | 存储格式、索引结构、分层存储策略、GC 策略 |
| taskType 判定 | 判定主体固定——由用户判定 | 判定时机与缓存机制——由实现层根据后续章节定义 |

## 三、核心机制

### 3.1 意图收敛机制

**定位**：从用户模糊的自然语言输入，收敛到系统可执行的结构化指令。

**收敛链路**：
```
用户输入
  → 第0层：口令层（DET EXACT_MATCH，0 次 LLM）
  → 第0.5层：关键词扫描（DET 标记 offTaskSuspicion，0 次 LLM）
  → 第1层：意图识别（LLM 轻量模型，ANALYZING 环节）
  → 第2层：环节内执行（LLM 通用/深度模型，IN_SESSION 环节）
  → 第3层：DET 值域复验（DET 四项检查）
```

#### 3.1.1 off-task 双重保障与显式切换口令

**问题**：环节间自由切换是态控架构的核心目标，但 IN_SESSION（会话中）内用户要切换环节时，当前机制单靠模型概率判断 off-task（偏离），不能保证准确——模型可能漏判（把切换当成当前环节补充）或误判（把补充当成切换）。

```
保障1（主力——宪法强制 + 上下文管理器优先注入）：
  根宪法第1条（环节隔离）→ 模型每轮首要判断"是否本环节任务"
  → 上下文管理器将环节隔离声明（@section role 首位）以最高优先级注入 prompt
  → 模型每轮看到的上下文，第一件事就是"你当前在什么环节，什么是 off-task"
  → 不属于本环节 → 立即返回 off-task（优先级最高）
  → 大幅降低漏判率

保障2（辅助——ANALYZING 重新识别验证）：
  模型返回 off-task 后，调度器不直接路由
  → DET 只做非空检查（门卫，不做语义验证——DET 做不了语义判断）
  → 送回 ANALYZING 重新识别 offTaskInput
  → ANALYZING 的重新识别结果就是验证：
    → 识别出不同 intent → 确认切换，进入新环节
    → 识别出 same intent 或 other → 尊重模型判断，引导重新表达
```

**显式切换口令（M1 switch，触发 房间状态隔离（S3））**：

用户可主动说"切断房间"口令 → DET 硬匹配（EXACT_MATCH，0 次 LLM）→ 调度器触发 房间状态隔离机制（S3）。房间状态隔离（S3）的完整四步流程与三个元指令（wake/switch/exit/cancel）的 S3 行为详见 §9.4，此处不再重复。

显式口令是首选切换机制（100% 确定性），off-task 检测是兜底（用户忘了口令时的保底）。

**这是态控架构"房间"比喻的物理落地**——传统智能体需要用户新开窗口来切换上下文，态控架构在单一窗口内通过 房间状态隔离（S3）实现房间自由切换。

**全窗口房间状态索引（roomStateIndex）**：

房间状态隔离（S3）解决了"新旧不串"的问题，但没有解决"切换后不知道旧房间有什么没做完"的问题。roomStateIndex 填补这个缺口：

- **实现形态**：物化视图（非实时查询）——roomStateIndex 是一张独立维护的摘要表，存储于 contractStore.roomStateIndex（userId 索引）
- **数据来源**：从 sessionCheckpoint + roomConversationLog + outputRegistry 三个数据源增量同步
- **数据结构**：`[{roomId, roomName, intent, taskType, lastActiveAt, pendingCount, completedCount, lastSummary}]`
- **增量更新触发事件**：房间切换时 / sessionCheckpoint 写入时 / outputRegistry 注册时
- **注入时机**：用户说"切断房间" → 调度器在 S3 会话回收前，先请求 context-manager 的 `buildRoomIndex()` 读取 roomStateIndex 物化视图 → 连同新房间的 checkpoint 一起注入 ANALYZING 的 prompt
- **注入内容**：ANALYZING 收到的 prompt 末尾追加一个 `[全窗口房间状态]` 段落，列出所有房间的状态摘要（如"记账房间：1 条半成品待完成"、"提醒房间：3 条提醒待触发"）
- **意图识别增强**：ANALYZING 有了全局视野后，可以针对性地引导用户——"您记账房间还有一条未完成的记录，要先处理吗？"——而不是简单地返回 intent=other

**roomStateIndex 与 roomConversationLog 的区别**：

| | roomConversationLog | roomStateIndex |
|------|------|------|
| 粒度 | 单房间全量对话原文 | 全窗口房间摘要 |
| 用途 | IN_SESSION 环节内上下文拼接 | ANALYZING 跨房间意图识别 |
| 消费方 | context-manager 拼接层 | context-manager buildRoomIndex() |
| 更新频率 | 每轮追加 | 物化视图增量同步（房间切换/checkpoint 写入/outputRegistry 注册时） |
| 存储 | contractStore.roomConversationLog | contractStore.roomStateIndex（独立表，userId 索引） |

**交互示例**：
```
用户："切断房间"
  → S3 释放记账房间 KV Cache
  → buildRoomIndex() 读取物化视图：记账房间 pending=1，提醒房间 pending=3
  → ANALYZING prompt 末尾注入：
    [全窗口房间状态]
    记账房间：半成品 1 条（7月15日记了半条"午饭25块"）
    提醒房间：待触发 3 条（最近：明天下午3点"开会"）
  → ANALYZING 返回 intent=reminder + suggestion="提醒房间有 3 条待触发提醒"
  → 用户可以选择："先去提醒房间" 或 "继续记账"
```

**设计原则**：不追求 100% 拦截，而是"降低漏判 + 拦住误判 + 回溯兜底"。用确定性兜底追求 100% 反而会破坏环节自治。

#### 3.1.2 ANALYZING 的环节隔离（双维度分类）

ANALYZING（分析态）是"意图识别环节"，受根宪法第 1 条约束：
- **本环节任务**：将用户输入映射到已知 intent（意图），同时分类输入性质：ANALYZING 用结构化 prompt 执行两个并行的强制选择分类：

  **维度1：意图匹配**——从有限选项中选择（如 A/B/C/D/E），开启 API logprobs 获取真实概率分布。返回 choice（选项字母）+ logprobs（概率分布）。DET 读取 logprobs 做阈值裁决（probability ≥ threshold → 放行，< threshold → intent=other）。

  **维度2：输入性质**——判断输入属于哪类：
  - S) 闲聊/松弛：无明确目标，随意表达，社交寒暄
  - T) 任务导向：有明确目标，结构化表达，包含动作意图
  - U) 不确定

  ANALYZING 一次调用返回两个信号：
  ```
  { choice: "C", logprobs: {A:0.05, B:0.03, C:0.78, D:0.08, E:0.06},
    inputNature: "T", inputNatureLogprobs: {S:0.02, T:0.91, U:0.07} }
  ```

  DET 交叉裁决路由：

> **模型约束**：ANALYZING 强制使用支持 API logprobs 的轻量模型（如 Qwen-Turbo、GPT-4o-mini）。推理模型（如 DeepSeek-R1、o3）不可用于 ANALYZING——它们不支持 logprobs 返回。若目标部署环境无合适的轻量模型，需使用 LLM 自由生成 confidence 作为降级方案，并在部署文档中声明"logprobs 降级模式已启用"。

  | intent match | input nature | 路由 |
  |-------------|-------------|------|
  | A-E（高 probability） | — | 对应正式房间 |
  | other（低 probability） | S（闲聊） | **SLACK_NODE** |
  | other（低 probability） | T（任务导向） | **TEMP_TOPIC** → 创建运行时房间 |
  | other（低 probability） | U（不确定） | 追问澄清 |

  **关键：一次 LLM 调用完成两个分类，不改模型，不追加调用。**只是 prompt 多问一个问题，返回多一个字段。

- **非本环节任务**：不能采集字段（那是 IN_SESSION 的事）、不能追问（那是 IN_SESSION 的事）、不能给出最终回复
- **边界情况**：输入无法映射到任何已知 intent → 返回 `intent=other`（低 probability）。注意：ANALYZING 只产出意图识别相关信号（intent / choice / logprobs / inputNature / extracted / topicId），不产出 turnType——turnType 是 IN_SESSION 的输出格式。

**ANALYZING 的产出物感知**：ANALYZING prompt 末尾追加 `[全系统产出物]` 段落——从 outputRegistry 查询当前用户的最近产出物列表。这使 ANALYZING 在意图识别时就拥有全系统产出物视野，可以在路由建议中主动提示："元智能体房间有 L2 设计文档，需要引用吗？" 详见 §3.1.3。

#### 3.1.3 产出物总索引——Output Registry

**定位**：全系统所有房间产出物的统一登记簿。与 roomStateIndex（房间状态摘要）互补——roomStateIndex 回答"房间有什么待办"，outputRegistry 回答"房间产出了什么"。

**存储**：contractStore 中 `outputRegistry` 表，完整 SQL DDL 见附录 C.3。

**索引**：`(roomId, createdAt)`、`(intent, outputType)`、FTS on `(outputName, outputSummary, keywords)`。

**写入时机与门槛**：

| 时机 | 操作 | 注册门槛 |
|------|------|---------|
| IN_SESSION complete（topic_based） | 用户确认的最终产出自动注册 | 无门槛——topic_based 的 complete 产出本身就是交付物 |
| IN_SESSION complete（field_based） | contract-out JSON 自动注册 | **有门槛**——field_based 的 complete 产出是字段集（如一笔记账），属于操作记录而非交付物。只有以下情况注册：(1) 用户主动命名/导出（如"7月账单汇总"）；(2) 字段集被其他房间引用过；(3) 用户触发"保存为产出物"。每轮 complete 的裸字段集不自动注册，避免 outputRegistry 被零散操作记录淹没 |
| TEMP_TOPIC 用户触发总结 | 总结产出自动注册 | 无门槛 |
| 用户手动导出/保存 | 调度器调用 registry.register() | 无门槛 |

**检索方式**（DET 确定性查询，0 次 LLM）：

```
- 按房间：registry.findByRoom("元智能体")
- 按类型：registry.findByType("document")
- 按关键词：registry.search("L2 设计文档")
- 按时间：registry.findSince("2026-07-01")
- 组合：registry.search({roomName: "元智能体", outputType: "document"})
```

**跨房间产出物发现与复制流程**：

```
1. 用户输入 → ANALYZING（已有 roomStateIndex + outputRegistry 列表注入） → LLM 语义理解："用户想引用元智能体的L2文档" → 返回：{ intent: "patent_write", crossRoomRef: { room: "元智能体", output: "L2设计文档" } } → 调度器 DET 拿 crossRoomRef 去 outputRegistry.search() ↑ ANALYZING 做语义判断，DET 只做精确查询

2. ANALYZING → crossRoomRef: {room:"元智能体", output:"L2设计文档"}
    ↓
DET: registry.search() → 命中1条
    ↓
DET 把匹配结果注入 IN_SESSION(patent_write) 的上下文
    ↓
IN_SESSION 模型：
  "找到元智能体的L2设计文档（7月15日，12KB）。要作为专利背景技术引用吗？"
    ↓
3.用户："对"  →  turnType=complete + confirmedRef=outputId_xxx
    ↓
调度器调用文件操作模块执行复制（DET 确定性）:
   cp {outputPath} → {currentRoomWorkspace}/references/{outputName}
   → 记录操作审计日志

5. 专利撰写房间的上下文管理器下一轮拼接时，自动将 references/ 目录文件列表
   注入房间上下文，模型可以读取引用文件的内容
```

**关键纪律——房间隔离不妥协**：

| 可以跨房间共享 | 不可以跨房间共享 |
|--------------|----------------|
| ✅ 结构化产出物文件（通过文件复制） | ❌ roomConversationLog（房间对话原文） |
| ✅ outputRegistry 元数据（产出物名称、摘要、路径） | ❌ 其他房间的领域规则 |
| ✅ roomStateIndex 状态摘要（每个房间有待办数） | ❌ 其他房间的 sessionCheckpoint 内部字段 |

文件复制是唯一的跨房间数据传递方式——不直接注入原文到上下文（防止上下文膨胀），而是复制文件到目标房间的 workspace/references/ 目录。目标房间的上下文管理器在下一轮拼接时，按需要决定是否将引用文件内容注入当前上下文。**房间之间的对话原文仍然完全隔离。** roomStateIndex 和 outputRegistry 是元数据层索引，不是数据层原文——它们提供"隔壁房间有什么"的摘要，不提供隔壁房间的对话内容，因此不违反房间隔离。

**与 roomStateIndex 的互补**（三表统一对比）：

| 维度 | roomConversationLog | roomStateIndex | outputRegistry |
|------|------|------|------|
| 粒度 | 单房间全量对话原文 | 全窗口房间摘要 | 产出物级明细 |
| 回答什么 | "这个房间聊过什么" | "记账房间还有1条半成品" | "元智能体产出了3份文档、2份代码" |
| 消费方 | context-manager 拼接层 | ANALYZING（跨房间切换建议） | ANALYZING（跨房间产出发现）+ 调度器（文件操作） |
| 更新频率 | 每轮追加 | 物化视图增量同步 | 每次产出物注册时 |
| 保留策略 | 按 importance 分级（永久/90天/30天/7天） | 跟随源表 | 永久（产出物是交付物） |
| 跨房间可见 | ❌ 不可见 | ✅ 摘要可见 | ✅ 元数据可见 |

### 3.2 taskType 双模式

**区分原点：收敛空间的性质，而非任务内容。**

taskType 是态控架构所有设计决策的逻辑起点——上下文拼接方式、校验方式、记忆载体、规则匹配键、追问策略、模型选择策略，全部收敛到这一个二分法上。

**field_based（基于字段）——离散参数空间，定位唯一正确值**

收敛空间是离散的、可枚举的参数集合。目标是从所有可能的候选中定位唯一正确值。存在客观正确答案——找到了，任务完成；没找到，任务未完成。正确与否可被 DET 值域硬校验验证。

典型场景：记账（从所有可能的类别中找到"猫粮"，从所有可能的金额中找到"35"）、改柜体参数（从所有可能的深度值中找到"600mm"）、查询（从所有可能的记录中找到"上周三的订单"）、删除（从所有可能的条目中找到"第5条"）、表单填写。

**topic_based（基于主题）——连续语义空间，采样满足约束的样本**

收敛空间是连续的、不可枚举的语义空间。目标是在受约束的主题空间内采样一个满足条件的样本。不存在唯一正确答案——同一主题下，可以生成无数个不同的合格输出。无法用"对/错"评判，只能用"是否在主题边界内 + 是否满足格式约束"来判断。

典型场景：写诗（在"秋天"主题空间内生成一首满足七言绝句格式的样本）、写专利（在"一种基于 XXX 的方法及系统"空间内生成满足专利撰写规范的样本）、角色扮演（在"林霜是侦探助手"空间内生成符合角色设定的对话）、头脑风暴（在"如何提升用户留存"空间内生成一组可行的想法）。

**全维度差异表**：

| 维度 | field_based（基于字段） | topic_based（基于主题） |
|------|----------------------|----------------------|
| 收敛空间 | 离散参数空间（可枚举） | 连续语义空间（不可枚举） |
| 目标 | 定位唯一正确值 | 采样满足约束的样本 |
| 是否存在客观正确答案 | 是 | 否 |
| 验证方式 | 值域校验（DET 硬规则） | 格式校验 + 主题边界检查（DET + 语义判断） |
| 追问方式 | 按预定义字段顺序追问 | 按主题探索路径追问（模型自主决定） |
| 完成条件 | 所有必采字段齐全 | 输出满足主题边界 + 格式约束 |
| 跨节点传递载体 | contract-out JSON（字段集）+ sessionCheckpoint（进度快照） | topicEvolution（事件序列 + 状态快照） |
| 长期记忆 | sessionCheckpoint（进度快照）+ roomConversationLog | topicEvolution（主题演化历史） |
| 领域规则匹配 | stepName DET 精确匹配 | topicPath 语义匹配 |
| 错误形态 | 值错了（可 DET 校验） | 跑偏了（需语义判断） |

**混合型任务：主 taskType + 子约束**

某些任务同时跨越两种空间。区分标准不是"任务里有没有字段"，而是"收敛空间的主体是连续的还是离散的"：
- 主体是连续语义空间（如创作），但带有离散约束（如格式）→ topic_based 主 + field_based 子。例："写一首关于秋天的七言绝句"——诗歌创作是 topic_based，但"七言绝句"的格式约束（28 字、平仄押韵）是 field_based。DET 复验时先做 field_based 格式校验，再做 topic_based 主题边界校验。
- 主体是填写字段（如录入客户信息），但带有自由文本（如备注）→ field_based 主 + topic_based 子（备注字段不做值域校验，只做主题边界检查）。

主 taskType 决定跨节点载体和长期记忆机制（topic_based 主→topicEvolution；field_based 主→sessionCheckpoint），子约束附加在 DET 复验的末端校验中。长期记忆载体统一为 contractStore——field_based 走 contractStore.sessionCheckpoints，topic_based 走 contractStore.topicEvolution。

**topicEvolution 与 sessionCheckpoint 的对位**：

| 维度 | topicEvolution（topic_based） | sessionCheckpoint（field_based） |
|------|----------------------|----------------------|
| 载体形式 | 事件流：每节点 exit 追加 event | 进度标记：每节点 complete 更新一行 |
| 匹配方式 | topicId embedding 语义检索 | userId DET 精确匹配 |
| 记录粒度 | 事件级——每 turn 的事件序列 | 环节级——每环节一帧快照 |
| 注入给模型 | 三层注入（已生效规则 + 关联主题摘要 + 演化事件包） | 结构化字段 collectedFields |
| 完整定义 | 见 §3.7.1 | 见 §3.7.2 |

topicEvolution 和 sessionCheckpoint 的完整机制定义（存储结构、写入时机、恢复流程、changeLevel 分层留存）统一收敛到 §3.7 跨会话记忆机制，此处只保留对位关系。

**领域规则体系**：规则不再只是平铺的"行为模式列表"，而是一个包含权限分层（L0-L3）、来源分类（scenario_seed / user_initiated / model_inferred）、关联边（depends_on / conflicts_with / strengthens）和条件匹配（conditions）的规则网络图。注入方式"按 topicPath 子图匹配精准注入"。完整定义见 §3.6。

### 3.4 三类执行体分派（DET / LLM / HUMAN）

#### 3.4.1 分派判据

每个状态在宪法中必须显式声明执行体，判据顺序如下（优先级递减）：

A. 错误代价多大？
代价大且无客观答案 → 人
代价大但有客观校验 → LLM + 人确认
代价小 → 进入下一条

B. 能否写成硬规则 + 穷举？
能 → 确定性程序（DET）
不能 → 进入下一条

C. 是否有客观正确答案？
有（映射类、意图类）→ LLM
没有（偏好类、责任类）→ 人

关键规则："代价"为最高优先级判据。代价大的节点无论能否硬规则化，都必须经过人确认或 LLM + 人确认。

#### 3.4.2 与意图收敛机制的对应

DET 执行确定性收敛；LLM 执行有限展开后收敛，出口被 contract-out schema（输出契约模式）硬卡；人执行责任收敛，对不可逆决策负责。

### 3.5 人和 DET 作为终极容错闭环

态控不追求大模型的单次绝对正确，而是构建"犯错-拦截-纠错"的工程闭环。用户不仅是意图源头，更是系统状态的"重置按钮"。系统只需保证"大概率正确"，剩余边界错误由 DET 和用户纠正完成最终收敛。在高风险场景中，这一闭环从单用户现场扩展到组织级人工接管通道。

### 3.6 领域规则体系

**定位**：领域规则是态控架构的第三层宪法（领域规则层）在生产过程中逐步建立的约束体系。与宪法不同——宪法是设计者预设的刚性框架，领域规则是从用户的实际使用中"长出来"的经验沉淀。领域规则从平铺列表升级为可查询、可冲突检测、可按主题路径匹配的**规则网络图**。

**field_based 的等价物——contractEvolution**：topic_based 的规则按 topicPath（行业→品类→场景）分层，field_based 的规则按 stepName（环节名）分层——规则提炼和注入完全按环节隔离。两者机制相同（changeLevel → ruleMiningQueue → 异步提炼 → domainRules），唯一差异是匹配键：topic_based 用 topicPath 语义检索，field_based 用 stepName DET 精确匹配。

#### 3.6.1 规则来源三层

领域规则有三类来源，对应不同的可信度和确认流程：

| 来源 | 含义 | 灌入时机 | 权限初始值 | 是否走交叉验证 |
|------|------|---------|-----------|-------------|
| **scenario_seed** | 场景固有规则（工艺规范、设计基线、报价模板、安全红线等） | 主题初始化时按 topicPath 加载 | L0 / L1 | ❌ 不走，设计者预设可信 |
| **user_initiated** | 主题确认时用户主动口述的规则（如"厨房台面我喜欢高一点"） | ANALYZING 确认 topicName 后主动问询 | L2 | ✅ 走，用户可能说错或矛盾 |
| **model_inferred** | IN_SESSION 过程中模型推断的用户偏好（如"从对话中推测用户偏好 U 型厨房"） | complete 时顺带输出 → ruleCandidates | L3 | ✅ 需用户确认升格为 L2 |

#### 3.6.2 规则权限分层

领域规则内部存在 L0-L3 四级权限梯度。这一维度**与宪法层级正交**——宪法分三层（根/局部/领域），每条领域规则同时具有"宪法层级"（属于第 3 层领域规则）和"权限层级"（L0/L1/L2/L3），分别定义其适用范围和可修改性。

| 层级 | 名称 | 含义 | 可修改 | 默认 importance | 冲突时行为 |
|------|------|------|--------|-----------------|----------|
| **L0** | 钢印规则 | 场景硬性约束，不可触碰 | ❌ 不可修改 | critical（L0 规则默认 critical，受"同层 critical≤1"约束——L0 层内必须用户确认选哪条为 critical，详见 §3.9） | 无条件胜出，不询问 |
| **L1** | 基线规则 | 设计者预设最佳实践（如"厨房台面高度建议 85cm"） | ⚠️ 需设计者确认 | normal | 与 L2/L3 冲突时触发询问 |
| **L2** | 用户确认规则 | 用户主动说过或确认过的偏好 | ✅ 用户可随时修改 | normal | 优先于 L3 |
| **L3** | 模型推断规则 | 模型 infer 的候选规则 | ✅ 需用户确认升格 | normal | 仅作参考，不主动注入 |

**importance 与 immutableLevel 正交**：
- immutableLevel 是权限轴（谁能改），importance 是重要程度轴（哪个更关键）。同 immutableLevel 内，importance 高的胜出
- 每层（L0/L1/L2/L3）内 critical ≤ 1、high ≤ ⌊该层规则总数/3⌋、其余 normal
- L0 规则中必须选出 1 条 critical（安全红线——用户确认），其余可全 normal
- 冲突仲裁：先比 immutableLevel → 同层再比 importance → 同 importance 触发询问

**跨层仲裁的边界声明**：
- 当两条规则 immutableLevel 不同时（如 L0 vs L2），按 immutableLevel 直接裁决（L0 无条件胜出），**importance 不参与跨层仲裁**
- importance 的作用域限定为"**同 immutableLevel 平局时的次级裁决**"——只有在两条规则 immutableLevel 相同时，importance 高的才胜出
- 示例：L0(normal) vs L2(critical) 冲突时，L0 直接胜出，L2 的 critical 不参与仲裁；L2(critical) vs L2(normal) 冲突时，L2(critical) 胜出

**规则状态**：

| 状态 | 含义 | 所在位置 |
|------|------|---------|
| active | 正式生效的规则 | domainRules |
| candidate | 待交叉验证的候选规则 | ruleCandidates |
| archived | 已废弃或被替代的历史版本 | topicEvolutionArchive |
| pending_review | 需人工审核（L0/L1 修改请求） | 单独队列 |

#### 3.6.3 主题层级与规则的作用域边界

topicPath 是一个**层级化数组**，从粗到细逐级展开。每一层都可以有自己独立的规则集——不同层级的规则互不污染。层级匹配规则为前缀匹配，核心纪律：上级可见于下级，下级不可见于上级/兄弟。

**规则提炼时的作用域要求**：每一轮对话和每一个提炼出来的规则都必须带 topicPath 和 source 标记。

#### 3.6.4 规则网络结构

领域规则是一个有向图——每条规则是一个节点，节点之间通过 `depends_on` / `conflicts_with` / `strengthens`（snake_case + 主动语态）三种关联边建立关系。

#### 3.6.5 规则生命周期

```
主题确认 → 加载种子规则 → IN_SESSION 运行 → 环节结束 → N10 异步提炼 → 用户确认升格
```

#### 3.6.6 规则注入匹配逻辑（上下文管理器匹配层 + 决策层）

每轮由上下文管理器的匹配层和决策层执行：
1. **匹配层**：topicPath 子图匹配 + 冲突检测 + 条件检查
2. **决策层**：按 immutableLevel → importance → strengthens 权重排序注入，strengthens 权重累加上限 +3（通过 tunable `strengthens_weight_cap` 配置）
3. **拼接层**：组装注入包

#### 3.6.7 规则提炼队列处理

异步批量提炼，交叉验证按来源区分。inferred_conflict 强制候选状态 + 用户确认升格，不直接参与运行时冲突检测。


> **实现提示**：每轮全量落盘虽覆盖最坏情况，但写路径成本可能占执行时间的显著比例。实现层应优先采用异步批量写入（每 N 轮 flush 一次，或 turnType=complete 时强制 flush），而非逐轮同步写入。规则提炼队列（ruleMiningQueue）的写入触发为环节 exit 时，非每轮。详见 §2.2-A 成本监控。


### 3.7 跨会话记忆机制

态控架构为两种 taskType 分别提供跨会话记忆机制：

| 模式 | 机制 | 匹配方式 | 核心存储 |
|------|------|---------|---------|
| topic_based | topicEvolution | topicId 语义锚点 + embedding 检索 | 四表（topicEvolution / topicEvolutionEvent / topicEvolutionArchive / topicIndex） |
| field_based | sessionCheckpoint | userId 精确匹配 + DET ttl 检查 | 单表（sessionCheckpoints） |
| 通用（两种 taskType 共享） | roomConversationLog | userId + roomId 精确匹配 | 单表（roomConversationLog） |
| field_based 规则演化 | contractEvolution | stepName 精确匹配 | 复用 contractStore 三表 |

**roomConversationLog 的独立保留策略**（适用于两种 taskType）：

| 房间 importance | 保留期 | 过期处理 |
|----------------|--------|---------|
| critical | 永久 | 不清理 |
| high | 90 天 | 过期后归档到 topicEvolutionArchive（如为 topic_based）或 contractEvolution archive（如为 field_based） |
| normal | 30 天 | 过期后归档 |
| low | 7 天 | 过期后直接删除（低价值，不归档） |

保留期通过 tunable 参数 `room_conversation_log_retention` 配置。sessionCheckpoint.ttl 到期时，只清理 sessionCheckpoints 表的进度行，**不联动清理 roomConversationLog**。

#### 3.7.1 topicEvolution（topic_based 专属）

主题演化体系——三层递进：分层留存（major 永久/minor 压缩/invalid 归档）、主题关联（八种关联关系，元数据层可查但数据层隔离）、异步规则提炼（低频，用户确认后生效）。

**topicEvolution 与 roomConversationLog 的边界**：

| | topicEvolution | roomConversationLog |
|---|---|---|
| 记录什么 | 结构化变更事件（stateChange + modelOutputSummary） | 逐字对话原文 |
| 视角 | 演化追溯——记录"主题怎么变的" | 上下文恢复——记录"当时聊了什么" |
| 压缩联动 | minor event 压缩时，对应原文轮次归档 | 跟随 topicEvolution 压缩节奏 |

#### 3.7.2 sessionCheckpoint（field_based 专属）

field_based 模式的原生跨会话进度记忆。存储结构见附录 C.3。

**调度器恢复流程**：ANALYZING 阶段 DET 查 sessionCheckpoints，有记录且未过期→询问继续或跳转；无记录或已过期→正常进 P0。

#### 3.7.3 contractEvolution：field_based 的领域规则演化

contractEvolution 是 topicEvolution 在 field_based 模式下的对位机制——前者按 stepName 精确匹配，后者按 topicId 语义匹配，两者互不替代。存储复用 contractStore 三表，按 stepName 隔离。

#### 3.7.4 contractEvolution 与 sessionCheckpoint 的关系

| | contractEvolution | sessionCheckpoint |
|---|---|---|
| 记录什么 | 规则变更历史 | 流程进度 |
| 匹配方式 | stepName 精确 | userId 精确 |
| 留存策略 | 分层留存 | ttl 到期清除进度行（roomConversationLog 不联动） |

#### 3.7.5 跨会话记忆对比

| 维度 | topicEvolution | sessionCheckpoint | contractEvolution | roomConversationLog |
|------|---------------|-------------------|-------------------|---------------------|
| 适用 taskType | topic_based | field_based | field_based | 通用（两种 taskType） |
| 记录内容 | 主题演化历史 | 流程进度 + 字段快照 | 环节内规则变更历史 | 房间对话原文 |
| 检索方式 | embedding 两阶段 | DET 精确匹配 | DET stepName 精确匹配 | DET userId+roomId 精确匹配 |
| 留存策略 | 分层留存（永久/压缩/归档） | ttl 到期清除进度行 | 分层留存（永久/压缩/归档） | 按 importance 分级（永久/90天/30天/7天） |



**用户确认预算**：为防止运行时打断过多导致"确认疲劳"，确认操作按频次分为两类：
- **设计期一次性确认**：L0 层 critical 选择、每层 importance 标定——在场景设计阶段（N1/N2）由设计者一次性确认，运行时不再询问
- **运行时低频确认**：候选规则升格（ruleCandidates → domainRules，积累到阈值后批量确认）、跨房间引用（用户首次提及该产出物时确认一次）、TEMP_TOPIC 命名（仅创建时一次）
- **量化约束**：运行时每 10 轮对话至多触发 1 次用户确认操作（设计期确认不在此预算内）

贯穿三层有两条铁律：人作为终极容错闭环，认知局部性不因主题关联而妥协。

### 3.8 统一环节操作模型——五步流水线

态控架构的两种任务模式遵循相同的五步操作流水线：
1. **状态机路由**：ANALYZING → intent + inputNature → 路由到目标房间
2. **环节上下文加载**：从 roomConversationLog + sessionCheckpoint + domainRules 加载
3. **内容拼接**：上下文管理器执行匹配→决策→拼接三层职责
4. **模型计算**：模型在局部宪法约束下自主完成环节内交互
5. **记录落盘**：roomConversationLog 实时追加 + sessionCheckpoint 更新 + topicEvolution event 追加

- 态控调度器只管步骤一和步骤五
- 上下文管理器管步骤三（匹配→决策→拼接）
- 环节宪法管步骤四

### 3.9 @importance 重要性分层机制

与 `immutableLevel` 正交的第二轴——三层结构（房间级/产出物级/规则级），枚举值 critical/high/normal/low，每层 critical≤1、high≤⌊该层总数/3⌋。

**消费方**：roomConversationLog 按 importance 分级保留期 + 拼接轮次；规则注入按 importance 排序；同 immutableLevel 冲突时 importance 高者胜出。

**防偷懒规则**：每层必须恰好 1 critical 或显式声明 allow_no_critical。

## 四、节点体系

### 4.1 节点意图收敛链

| 节点 | 输入 | 收敛目标 | 执行体 |
|------|------|---------|--------|
| ANALYZING | 用户口语 | `{choice, logprobs, intent, inputNature, extracted, topicId}` | LLM |
| IN_SESSION(field_based) | intent + 参数树 | `{turnType, askingField, collectedFields, result, changeLevel}` | LLM（**可调用 DET 工具**） |
| IN_SESSION(topic_based) | intent + 主题边界 + topicId | `{turnType, askingField, collectedFields, result, changeLevel}` | LLM（**可调用 DET 工具**） |
| DET 复验 | result + constraints | `{valid:true/false, reason}` | DET |
| WAITING_CONFIRM | previewUrl + diffSummary | `{decision: "confirm"/"reject"}` | 人 |
| EXECUTING | fieldPath + newVal | `{status:"success"}` | DET（**仅写入/删除操作**） |

**IN_SESSION 工具调用能力（v5.4 新增）**：IN_SESSION 模型可在环节内调用 DET 工具（function calling），用于查询/计算等需要数据库操作的 intent（如 query/compare）。LLM 做语义理解+结果格式化，DET 工具做 SQL 查询。工具调用结果直接由 LLM 消费并生成自然语言回复，不经过独立 EXECUTING 状态。

**EXECUTING 职责调整（v5.4 新增）**：EXECUTING 仅保留给纯写入/删除操作（如 record INSERT、delete DELETE）。查询/对比类 intent 的数据库操作在 IN_SESSION 内通过工具调用完成，不经过 EXECUTING。

**IN_SESSION 工具分层发现机制（v5.5 新增）**：工具按三层体系组织，解决工具数量多时上下文超长+准确率下降问题（行业验证：100+工具时准确率降至~15%，两阶段选择可恢复至80%+）：

1. **必用工具**（环节宪法 @section tools 声明）：环节核心工具的**完整定义**（含参数 schema），调度器进入 IN_SESSION 时自动注入。每个环节 1-3 个，常驻上下文。
2. **工具清单**（环节宪法 @section tool_catalog 声明）：选用工具的**轻量级目录**（仅 name + 一句话描述），每次注入。占用极少 token（每个工具约 20 token）。模型从清单中识别需要的工具。
3. **选用工具**（系统级工具库注册）：完整定义按需加载——模型从清单中选择后返回工具名，调度器从工具库加载完整定义注入。不用时上下文占用为 0。

**工具发现流程**（Two-Phase Hierarchical Selection，行业最佳实践）：
```
阶段1 Search：模型看工具清单（轻量级）→ 识别需要的工具 → 返回 needed_tools 列表
阶段2 Load：调度器从工具库加载完整定义 → 注入模型上下文
阶段3 Execute：模型调用工具 → DET 执行 SQL → 结果返回模型 → 模型生成回复
```

**meta-tool search_tools**（始终可用）：当工具清单中没有所需能力时，模型可调用 search_tools(query) 从系统级工具库检索，扩展工具清单。这是 MCP-Zero 主动发现机制的落地。

**工具内部时间解析**：涉及时间的工具参数支持自然语言时间表达（如 "this_month" / "yesterday"），DET 工具内部解析为具体日期。减少模型对辅助工具（如 get_current_date）的依赖。

**与认知局部性的关系**：工具分层是认知局部性在工具层的落地——每个环节只收到必用工具完整定义 + 选用工具轻量级清单，不是全部工具的完整定义。

PREVIEWING 是 IN_SESSION 的可选子环节，不是独立节点。ANALYZING 只产出意图识别相关信号，不产出 turnType。
