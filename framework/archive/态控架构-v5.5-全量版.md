## 一、架构背景

### 1.1 模型是一个无状态的有自己权重分布概率计算单元

模型的计算能力是有限的，而且受到训练结果约束，权重分布是不均衡的。无论是模型本身的计算资源，还是模型外部的计算载体，还有消耗的能源都是有限的，但是要计算的内容是趋向膨胀的。所有 AI 应用本质是消耗直接能源和间接资源的计算行为，每次单位计算资源的消耗是基本固定的，所以当次计算内容对任务的匹配度越高越好。

AI 应用追求计算性价比的路径：首先找到当次输入任务最匹配的内容，然后去合理分配这些内容，把跟任务最匹配的内容放在最适合的位置，然后按这个标准去分配计算内容。

### 1.2 任何任务皆为受限计算

在实际应用中，不存在"纯无限制的自由计算"。写诗、写小说等创意任务同样具有方向、主题与格式限制。在态控视角下，写诗与修改家具参数都是受限空间内的收敛过程，区别在于收敛空间的连续性——前者是在连续语义空间中采样满足约束的样本，后者是在离散参数空间中定位唯一正确路径。

### 1.3 认知局部性原理（输入侧）

任何信息处理主体（人、LLM、确定性程序），在 t 时刻处理节点 N 时，最优输入集 = N 在当前目标下的最小充分条件集（contract-in（输入契约）），不是全集 Γ。这一条是认知必然：

A 人：工作记忆 7±2 组块，逼人看全局反而信息过载、分析瘫痪。

B LLM（推理阶段）：Lost in the Middle（中间迷失）效应使中部信息召回率显著低于头尾；近因偏置使尾部权重远高于前部；长上下文幻觉率随长度单调递增。喂全局是跟注意力机制对着干。

C 确定性程序（DET）：函数纯度要求 f(x) 只依赖 x，不依赖全局变量。

## 二、架构哲学与设计原则

态控架构是一套以"认知局部性"为哲学根、面向 LLM 智能体收敛空间设计的工程框架。态控架构认为智能体 = 数据流网络 + 三类执行体（确定性程序 / 概率性 LLM / 人类选择）。态控架构根本立场："以数据流为中心，LLM 是被调度的资源"——它把上下文剪裁权从概率模型内部彻底剥离，移交给外部确定性状态机。这个状态机不关心每个节点跑的是什么模型、处理什么模态、由谁执行，只定义节点之间的数据格式和路由规则。每个节点被封装为自治单元。模型只需要精准匹配当前智能体场景的需求。同类执行环节可以定向后训练专用深度模型，理论上每个 intent 都可以配备专用小模型，成本从"全量大模型"降为"按需分配"。

### 2.1 数据流定义

"数据从哪里来、到哪里去、经过什么变换"；执行体定义"每个变换节点由谁执行"。每个节点是一个独立的变换函数，接收系统拼接的结构化输出（contract-in（输入契约）），根据当前节点的场景描述进行处理，产生系统需要的结构化数据（contract-out（输出契约））。节点之间不共享内部状态，不依赖全局上下文。每个节点只拥有它收到的输入数据和它产生的输出数据。

### 2.2 长上下文管理原则

本架构通过数据流切段、契约传递、每轮按规则拼接主动管理上下文长度。每个环节的上下文窗口由上下文管理器控制在高效区间——从本环节 roomConversationLog（全量记录）/ 对话快照 / 已确定内容 / 领域规则 + 其他环节与本环节关联内容（例如基于字段的场景中前置环节的产出物和基于主题场景中的规则网络图）这些内容中按上下文管理器的拼接机制进行拼接。

规模增强（scale up 才需要）：当单一环节的计算需求超出单实例能力时，启用并联实例化（多实例并行处理 + 结果合并）与长上下文显式管理（分段加载 + 滚动窗口）。并联实例化的触发条件、实例间通信协议、结果合并策略由场景设计阶段在环节宪法 `@section scale` 中声明，非运行时动态决定。

### 2.3 受控灵活性原则

宪法主体在设计时写死，但允许在运行时通过预设的 tunable（可调参数）槽位调整指定参数。调整范围受 range（范围）/ enum（枚举）限定，变更日志可追溯。

### 2.4 环节级自治原则与上下文管理器三层职责

态控调度器管路由与全量记录，上下文管理器管上下文的匹配→决策→拼接，模型管环节内交互。三个实体各司其职：

- **态控调度器**：只在环节边界执行路由和记录（详见 §2.4-A 调度器职责声明）
- **上下文管理器**：在环节入口执行**三层确定性职责**——
  1. **匹配层**：按 topicPath / stepName 做规则子图匹配、冲突检测、条件检查（确定性计算）
  2. **决策层**：按 importance / immutableLevel / strengthens 权重决定注入哪些内容（确定性计算）
  3. **拼接层**：把决策结果组装成 prompt，做 token 预算分配与截断（确定性计算）
- **环节内执行体（模型）**：环节内的多轮追问、字段校验、偏离检测由模型实例在局部宪法约束下自主完成
- **DET 兜底**：态控调度器在关键节点执行确定性兜底校验

上下文管理器是确定性状态机，做匹配→决策→拼接三步，**但不执行环节宪法**——不决定模型怎么追问、怎么校验、怎么收敛。上下文管理器的实现形态由场景复杂度决定（确定性程序 / LLM / 混合），属于实现层选择，架构层不固定其形态。

**§2.4-A 态控调度器职责声明**：态控调度器承担三类职责——
1. **路由**：状态转移、回滚、偏离处理、意图级纠错
2. **全量记录**：roomConversationLog / sessionCheckpoint / contractEvolution / outputRegistry 的写入触发与协调
3. **资源协调**：S3 物理 KV Cache 释放（调内存管理模块）、跨房间文件复制（调文件操作模块）、context-manager 调配、CLARIFYING 消息转发

调度器亲自做"路由决策"和"记录触发"，把"内存管理""文件操作"委托给专用子模块（内存管理模块 / 文件操作模块），自己只做协调调用——避免调度器变成大杂烩。

### 2.5 最小可识别单元

态控的最小可识别单元不是"三分+契约+并联+tunable（可调参数）"全套，而是——一个态控调度器 + 一个环节的每状态标执行体 + 每状态 contract-in（输入契约）/ out（输出契约） + 一部宪法。这四件套齐了，"态控"就成立。

### 2.6 房间比喻——态控的核心灵魂

态控架构的灵魂是"每个环节就是一个独立子任务房间，态控调度器会判断用户每次想去哪个房间"。

**房间里有什么**：每个环节是一个独立房间，房间里有这个环节上次讨论的全量记录（roomConversationLog——整个环节对话原文）和对话快照（stepSnapshot——collectedFields）+ 已确认的产出物 + 相关的领域规则。门一关，其他房间的内容不进入当前房间的上下文拼接——隔离对象是"其他环节无关内容"，隔离动作是"不进入上下文拼接"，不是"环节之间逻辑上不可见"。门一开，态控调度器把全量记录和对话快照 + 已确认的产出物 + 相关的领域规则和用户最新输入一起交给房间内部的上下文管理器，由上下文管理器进行内容拼接，然后输入给模型计算。上下文管理器是独立于环节宪法的专用拼接机制——它只管"怎么从态控调度器交给它的内容中组装出任务计算需要的上下文"。

**模型在这套系统里的真实位置**：模型本身是无状态的。它不知道自己上一轮输出了什么，不知道用户上次改了什么参数。房间里保存的全量记录和对话快照就是模型的上下文——"记性"不来自模型，来自系统的内容保存和分类机制。所以用户可以在房间之间自由穿梭：做完 N10 的 tunable，说一声"改一下 N1 的场景定义"，状态机立刻匹配 intent=N1，取出 N1 房间的相关内容交给 N1 的内容管理器，内容管理器拼接后递给模型。模型看到的只有 N1 房间里的内容，不掺杂 N10 的参数细节。

**roomConversationLog 的独立保留策略**：roomConversationLog 的保留期独立于 sessionCheckpoint.ttl——roomConversationLog 有自己的分级保留策略，按房间 importance 分级：

| 房间 importance | roomConversationLog 保留期 | 说明 |
|----------------|---------------------------|------|
| critical | 永久保留 | 核心房间对话不清理，支持任意时间回溯 |
| high | 90 天 | 高重要房间保留一个季度 |
| normal | 30 天 | 普通房间保留一个月 |
| low | 7 天 | 低重要房间保留一周 |

此保留策略同时适用于 field_based 和 topic_based 两种 taskType——roomConversationLog 是房间级记录，不区分 taskType。sessionCheckpoint.ttl 到期时，只清理 sessionCheckpoints 表的进度行，**不联动清理 roomConversationLog**——roomConversationLog 按自己的保留期独立过期。这保证了"自由穿梭"承诺：即使 field_based 任务的进度检查点过期了，用户返回房间时仍能从 roomConversationLog 恢复对话上下文。

**自由切换的四个支柱**：

| 支柱 | 做什么 |
|------|--------|
| 全量记录（roomConversationLog） | 每个环节的每轮对话实时落盘——切换环节时内容不丢弃，返回时完整可用（按 importance 分级保留，见上表） |
| 对话快照（stepSnapshot） | 每个环节收敛时的结构化字段快照——快速定位已确认的决策 |
| 规则提炼按环节隔离 | 每个环节的 changeLog 独立推送 ruleMiningQueue——N12 的规则提炼只消费 N12 的变更历史 |
| **产出物总索引** | **每个房间产出物的统一登记簿——跨房间可发现、可检索、可自动复制。DET 确定性查询（0 LLM）。如"元智能体的 L2 设计文档"可被专利撰写房间复制引用** |

**自由切换的适配目标**：自由切换是用户在真实使用中的实际行为。架构的价值命题是"让切换正确"（正确匹配 intent、取出正确房间内容、不串台），不是"鼓励切换"。

**这就是为什么 field_based 和 topic_based 都需要跨会话记忆**：topic_based 用 topicEvolution 管记录，field_based 用 sessionCheckpoint 管记录——两种机制互不替代，但共享同一个灵魂：模型无状态，状态机管全量记录，内容管理器管拼接，用户自由穿梭。

### 2.7 实现形态声明

本节集中声明各关键组件的实现形态属性，划清架构层固定属性与实现层可变属性的边界，避免下游实现者按默认假设做偏差。

| 组件 | 架构层固定属性 | 实现层可变属性 |
|------|--------------|--------------|
| 态控调度器 | 职责固定——路由 + 全量记录 + 资源协调（含内存管理协调、文件操作协调、context-manager 调配）；必须是确定性程序 | 路由表结构、记录存储格式、状态转移具体实现、子模块（内存管理/文件操作）的拆分方式 |
| 上下文管理器 | 职责固定——匹配→决策→拼接三层确定性职责；不执行环节宪法 | 实现形态可变——确定性程序 / LLM / 混合，由场景复杂度决定；匹配算法、决策权重、拼接策略、token 上限、降级方案 |
| 环节内执行体 | 职责固定——在局部宪法约束下完成环节内交互 | 实现形态可变——确定性程序 / LLM / 人类选择，由 taskType 和节点状态决定 |
| DET 复验 | 职责固定——确定性收敛，不调 LLM | 校验规则集、值域定义、格式 schema |
| 规则提炼引擎 | 职责固定——从 changeLog 和 topicEvolution 提取候选规则，经用户确认后入库 | 触发条件（时间窗口 / 数量阈值）——但必须保证低频性；提炼算法、交叉验证策略 |
| contractStore | 职责固定——存储契约、规则、演化记录，支持按 topicPath 子图匹配 | 存储格式、索引结构、分层存储策略、GC 策略 |
| taskType 判定 | 判定主体固定——由用户判定 | 判定时机与缓存机制——由实现层根据后续章节定义 |

## 三、核心机制

### 3.1 意图收敛机制

**定位**：从用户模糊的自然语言输入，收敛到系统可执行的结构化指令。

**收敛链路**：
```
用户输入
  → 第0层：口令层（DET EXACT_MATCH，0 次 LLM）
  → 第0.5层：关键词扫描（DET 标记 offTaskSuspicion，0 次 LLM）
  → 第1层：意图识别（LLM 轻量模型，ANALYZING 环节）
  → 第2层：环节内执行（LLM 通用/深度模型，IN_SESSION 环节）
  → 第3层：DET 值域复验（DET 四项检查）
```

#### 3.1.1 off-task 双重保障与显式切换口令

**问题**：环节间自由切换是态控架构的核心目标，但 IN_SESSION（会话中）内用户要切换环节时，当前机制单靠模型概率判断 off-task（偏离），不能保证准确——模型可能漏判（把切换当成当前环节补充）或误判（把补充当成切换）。

```
保障1（主力——宪法强制 + 上下文管理器优先注入）：
  根宪法第1条（环节隔离）→ 模型每轮首要判断"是否本环节任务"
  → 上下文管理器将环节隔离声明（@section role 首位）以最高优先级注入 prompt
  → 模型每轮看到的上下文，第一件事就是"你当前在什么环节，什么是 off-task"
  → 不属于本环节 → 立即返回 off-task（优先级最高）
  → 大幅降低漏判率

保障2（辅助——ANALYZING 重新识别验证）：
  模型返回 off-task 后，调度器不直接路由
  → DET 只做非空检查（门卫，不做语义验证——DET 做不了语义判断）
  → 送回 ANALYZING 重新识别 offTaskInput
  → ANALYZING 的重新识别结果就是验证：
    → 识别出不同 intent → 确认切换，进入新环节
    → 识别出 same intent 或 other → 尊重模型判断，引导重新表达
```

**显式切换口令（M1 switch，触发 S3 物理隔离）**：

用户可主动说"切断房间"口令 → DET 硬匹配（EXACT_MATCH，0 次 LLM）→ 调度器触发 S3 物理机制。S3 物理隔离的完整四步流程与三个元指令（wake/switch/exit/cancel）的 S3 行为详见 §9.4，此处不再重复。

显式口令是首选切换机制（100% 确定性），off-task 检测是兜底（用户忘了口令时的保底）。

**这是态控架构"房间"比喻的物理落地**——传统智能体需要用户新开窗口来切换上下文，态控架构在单一窗口内通过 S3 物理隔离实现房间自由切换。

**全窗口房间状态索引（roomStateIndex）**：

S3 物理隔离解决了"新旧不串"的问题，但没有解决"切换后不知道旧房间有什么没做完"的问题。roomStateIndex 填补这个缺口：

- **实现形态**：物化视图（非实时查询）——roomStateIndex 是一张独立维护的摘要表，存储于 contractStore.roomStateIndex（userId 索引）
- **数据来源**：从 sessionCheckpoint + roomConversationLog + outputRegistry 三个数据源增量同步
- **数据结构**：`[{roomId, roomName, intent, taskType, lastActiveAt, pendingCount, completedCount, lastSummary}]`
- **增量更新触发事件**：房间切换时 / sessionCheckpoint 写入时 / outputRegistry 注册时
- **注入时机**：用户说"切断房间" → 调度器在 S3 物理释放前，先请求 context-manager 的 `buildRoomIndex()` 读取 roomStateIndex 物化视图 → 连同新房间的 checkpoint 一起注入 ANALYZING 的 prompt
- **注入内容**：ANALYZING 收到的 prompt 末尾追加一个 `[全窗口房间状态]` 段落，列出所有房间的状态摘要（如"记账房间：1 条半成品待完成"、"提醒房间：3 条提醒待触发"）
- **意图识别增强**：ANALYZING 有了全局视野后，可以针对性地引导用户——"您记账房间还有一条未完成的记录，要先处理吗？"——而不是简单地返回 intent=other

**roomStateIndex 与 roomConversationLog 的区别**：

| | roomConversationLog | roomStateIndex |
|------|------|------|
| 粒度 | 单房间全量对话原文 | 全窗口房间摘要 |
| 用途 | IN_SESSION 环节内上下文拼接 | ANALYZING 跨房间意图识别 |
| 消费方 | context-manager 拼接层 | context-manager buildRoomIndex() |
| 更新频率 | 每轮追加 | 物化视图增量同步（房间切换/checkpoint 写入/outputRegistry 注册时） |
| 存储 | contractStore.roomConversationLog | contractStore.roomStateIndex（独立表，userId 索引） |

**交互示例**：
```
用户："切断房间"
  → S3 释放记账房间 KV Cache
  → buildRoomIndex() 读取物化视图：记账房间 pending=1，提醒房间 pending=3
  → ANALYZING prompt 末尾注入：
    [全窗口房间状态]
    记账房间：半成品 1 条（7月15日记了半条"午饭25块"）
    提醒房间：待触发 3 条（最近：明天下午3点"开会"）
  → ANALYZING 返回 intent=reminder + suggestion="提醒房间有 3 条待触发提醒"
  → 用户可以选择："先去提醒房间" 或 "继续记账"
```

**设计原则**：不追求 100% 拦截，而是"降低漏判 + 拦住误判 + 回溯兜底"。用确定性兜底追求 100% 反而会破坏环节自治。

#### 3.1.2 ANALYZING 的环节隔离（双维度分类）

ANALYZING（分析态）是"意图识别环节"，受根宪法第 1 条约束：
- **本环节任务**：将用户输入映射到已知 intent（意图），同时分类输入性质：ANALYZING 用结构化 prompt 执行两个并行的强制选择分类：

  **维度1：意图匹配**——从有限选项中选择（如 A/B/C/D/E），开启 API logprobs 获取真实概率分布。返回 choice（选项字母）+ logprobs（概率分布）。DET 读取 logprobs 做阈值裁决（probability ≥ threshold → 放行，< threshold → intent=other）。

  **维度2：输入性质**——判断输入属于哪类：
  - S) 闲聊/松弛：无明确目标，随意表达，社交寒暄
  - T) 任务导向：有明确目标，结构化表达，包含动作意图
  - U) 不确定

  ANALYZING 一次调用返回两个信号：
  ```
  { choice: "C", logprobs: {A:0.05, B:0.03, C:0.78, D:0.08, E:0.06},
    inputNature: "T", inputNatureLogprobs: {S:0.02, T:0.91, U:0.07} }
  ```

  DET 交叉裁决路由：

  | intent match | input nature | 路由 |
  |-------------|-------------|------|
  | A-E（高 probability） | — | 对应正式房间 |
  | other（低 probability） | S（闲聊） | **SLACK_NODE** |
  | other（低 probability） | T（任务导向） | **TEMP_TOPIC** → 创建运行时房间 |
  | other（低 probability） | U（不确定） | 追问澄清 |

  **关键：一次 LLM 调用完成两个分类，不改模型，不追加调用。**只是 prompt 多问一个问题，返回多一个字段。

- **非本环节任务**：不能采集字段（那是 IN_SESSION 的事）、不能追问（那是 IN_SESSION 的事）、不能给出最终回复
- **边界情况**：输入无法映射到任何已知 intent → 返回 `intent=other`（低 probability）。注意：ANALYZING 只产出意图识别相关信号（intent / choice / logprobs / inputNature / extracted / topicId），不产出 turnType——turnType 是 IN_SESSION 的输出格式。

**ANALYZING 的产出物感知**：ANALYZING prompt 末尾追加 `[全系统产出物]` 段落——从 outputRegistry 查询当前用户的最近产出物列表。这使 ANALYZING 在意图识别时就拥有全系统产出物视野，可以在路由建议中主动提示："元智能体房间有 L2 设计文档，需要引用吗？" 详见 §3.1.4。

#### 3.1.3 产出物总索引——Output Registry

**定位**：全系统所有房间产出物的统一登记簿。与 roomStateIndex（房间状态摘要）互补——roomStateIndex 回答"房间有什么待办"，outputRegistry 回答"房间产出了什么"。

**存储**：contractStore 中 `outputRegistry` 表，完整 SQL DDL 见附录 C.3。

**索引**：`(roomId, createdAt)`、`(intent, outputType)`、FTS on `(outputName, outputSummary, keywords)`。

**写入时机与门槛**：

| 时机 | 操作 | 注册门槛 |
|------|------|---------|
| IN_SESSION complete（topic_based） | 用户确认的最终产出自动注册 | 无门槛——topic_based 的 complete 产出本身就是交付物 |
| IN_SESSION complete（field_based） | contract-out JSON 自动注册 | **有门槛**——field_based 的 complete 产出是字段集（如一笔记账），属于操作记录而非交付物。只有以下情况注册：(1) 用户主动命名/导出（如"7月账单汇总"）；(2) 字段集被其他房间引用过；(3) 用户触发"保存为产出物"。每轮 complete 的裸字段集不自动注册，避免 outputRegistry 被零散操作记录淹没 |
| TEMP_TOPIC 用户触发总结 | 总结产出自动注册 | 无门槛 |
| 用户手动导出/保存 | 调度器调用 registry.register() | 无门槛 |

**检索方式**（DET 确定性查询，0 次 LLM）：

```
- 按房间：registry.findByRoom("元智能体")
- 按类型：registry.findByType("document")
- 按关键词：registry.search("L2 设计文档")
- 按时间：registry.findSince("2026-07-01")
- 组合：registry.search({roomName: "元智能体", outputType: "document"})
```

**跨房间产出物发现与复制流程**：

```
1. 用户输入 → ANALYZING（已有 roomStateIndex + outputRegistry 列表注入） → LLM 语义理解："用户想引用元智能体的L2文档" → 返回：{ intent: "patent_write", crossRoomRef: { room: "元智能体", output: "L2设计文档" } } → 调度器 DET 拿 crossRoomRef 去 outputRegistry.search() ↑ ANALYZING 做语义判断，DET 只做精确查询

2. ANALYZING → crossRoomRef: {room:"元智能体", output:"L2设计文档"}
    ↓
DET: registry.search() → 命中1条
    ↓
DET 把匹配结果注入 IN_SESSION(patent_write) 的上下文
    ↓
IN_SESSION 模型：
  "找到元智能体的L2设计文档（7月15日，12KB）。要作为专利背景技术引用吗？"
    ↓
3.用户："对"  →  turnType=complete + confirmedRef=outputId_xxx
    ↓
调度器调用文件操作模块执行复制（DET 确定性）:
   cp {outputPath} → {currentRoomWorkspace}/references/{outputName}
   → 记录操作审计日志

5. 专利撰写房间的上下文管理器下一轮拼接时，自动将 references/ 目录文件列表
   注入房间上下文，模型可以读取引用文件的内容
```

**关键纪律——房间隔离不妥协**：

| 可以跨房间共享 | 不可以跨房间共享 |
|--------------|----------------|
| ✅ 结构化产出物文件（通过文件复制） | ❌ roomConversationLog（房间对话原文） |
| ✅ outputRegistry 元数据（产出物名称、摘要、路径） | ❌ 其他房间的领域规则 |
| ✅ roomStateIndex 状态摘要（每个房间有待办数） | ❌ 其他房间的 sessionCheckpoint 内部字段 |

文件复制是唯一的跨房间数据传递方式——不直接注入原文到上下文（防止上下文膨胀），而是复制文件到目标房间的 workspace/references/ 目录。目标房间的上下文管理器在下一轮拼接时，按需要决定是否将引用文件内容注入当前上下文。**房间之间的对话原文仍然完全隔离。** roomStateIndex 和 outputRegistry 是元数据层索引，不是数据层原文——它们提供"隔壁房间有什么"的摘要，不提供隔壁房间的对话内容，因此不违反房间隔离。

**与 roomStateIndex 的互补**（三表统一对比）：

| 维度 | roomConversationLog | roomStateIndex | outputRegistry |
|------|------|------|------|
| 粒度 | 单房间全量对话原文 | 全窗口房间摘要 | 产出物级明细 |
| 回答什么 | "这个房间聊过什么" | "记账房间还有1条半成品" | "元智能体产出了3份文档、2份代码" |
| 消费方 | context-manager 拼接层 | ANALYZING（跨房间切换建议） | ANALYZING（跨房间产出发现）+ 调度器（文件操作） |
| 更新频率 | 每轮追加 | 物化视图增量同步 | 每次产出物注册时 |
| 保留策略 | 按 importance 分级（永久/90天/30天/7天） | 跟随源表 | 永久（产出物是交付物） |
| 跨房间可见 | ❌ 不可见 | ✅ 摘要可见 | ✅ 元数据可见 |

### 3.2 taskType 双模式

**区分原点：收敛空间的性质，而非任务内容。**

taskType 是态控架构所有设计决策的逻辑起点——上下文拼接方式、校验方式、记忆载体、规则匹配键、追问策略、模型选择策略，全部收敛到这一个二分法上。

**field_based（基于字段）——离散参数空间，定位唯一正确值**

收敛空间是离散的、可枚举的参数集合。目标是从所有可能的候选中定位唯一正确值。存在客观正确答案——找到了，任务完成；没找到，任务未完成。正确与否可被 DET 值域硬校验验证。

典型场景：记账（从所有可能的类别中找到"猫粮"，从所有可能的金额中找到"35"）、改柜体参数（从所有可能的深度值中找到"600mm"）、查询（从所有可能的记录中找到"上周三的订单"）、删除（从所有可能的条目中找到"第5条"）、表单填写。

**topic_based（基于主题）——连续语义空间，采样满足约束的样本**

收敛空间是连续的、不可枚举的语义空间。目标是在受约束的主题空间内采样一个满足条件的样本。不存在唯一正确答案——同一主题下，可以生成无数个不同的合格输出。无法用"对/错"评判，只能用"是否在主题边界内 + 是否满足格式约束"来判断。

典型场景：写诗（在"秋天"主题空间内生成一首满足七言绝句格式的样本）、写专利（在"一种基于 XXX 的方法及系统"空间内生成满足专利撰写规范的样本）、角色扮演（在"林霜是侦探助手"空间内生成符合角色设定的对话）、头脑风暴（在"如何提升用户留存"空间内生成一组可行的想法）。

**全维度差异表**：

| 维度 | field_based（基于字段） | topic_based（基于主题） |
|------|----------------------|----------------------|
| 收敛空间 | 离散参数空间（可枚举） | 连续语义空间（不可枚举） |
| 目标 | 定位唯一正确值 | 采样满足约束的样本 |
| 是否存在客观正确答案 | 是 | 否 |
| 验证方式 | 值域校验（DET 硬规则） | 格式校验 + 主题边界检查（DET + 语义判断） |
| 追问方式 | 按预定义字段顺序追问 | 按主题探索路径追问（模型自主决定） |
| 完成条件 | 所有必采字段齐全 | 输出满足主题边界 + 格式约束 |
| 跨节点传递载体 | contract-out JSON（字段集）+ sessionCheckpoint（进度快照） | topicEvolution（事件序列 + 状态快照） |
| 长期记忆 | sessionCheckpoint（进度快照）+ roomConversationLog | topicEvolution（主题演化历史） |
| 领域规则匹配 | stepName DET 精确匹配 | topicPath 语义匹配 |
| 错误形态 | 值错了（可 DET 校验） | 跑偏了（需语义判断） |

**混合型任务：主 taskType + 子约束**

某些任务同时跨越两种空间。区分标准不是"任务里有没有字段"，而是"收敛空间的主体是连续的还是离散的"：
- 主体是连续语义空间（如创作），但带有离散约束（如格式）→ topic_based 主 + field_based 子。例："写一首关于秋天的七言绝句"——诗歌创作是 topic_based，但"七言绝句"的格式约束（28 字、平仄押韵）是 field_based。DET 复验时先做 field_based 格式校验，再做 topic_based 主题边界校验。
- 主体是填写字段（如录入客户信息），但带有自由文本（如备注）→ field_based 主 + topic_based 子（备注字段不做值域校验，只做主题边界检查）。

主 taskType 决定跨节点载体和长期记忆机制（topic_based 主→topicEvolution；field_based 主→sessionCheckpoint），子约束附加在 DET 复验的末端校验中。长期记忆载体统一为 contractStore——field_based 走 contractStore.sessionCheckpoints，topic_based 走 contractStore.topicEvolution。

**topicEvolution 与 sessionCheckpoint 的对位**：

| 维度 | topicEvolution（topic_based） | sessionCheckpoint（field_based） |
|------|----------------------|----------------------|
| 载体形式 | 事件流：每节点 exit 追加 event | 进度标记：每节点 complete 更新一行 |
| 匹配方式 | topicId embedding 语义检索 | userId DET 精确匹配 |
| 记录粒度 | 事件级——每 turn 的事件序列 | 环节级——每环节一帧快照 |
| 注入给模型 | 三层注入（已生效规则 + 关联主题摘要 + 演化事件包） | 结构化字段 collectedFields |
| 完整定义 | 见 §3.8.1 | 见 §3.8.2 |

topicEvolution 和 sessionCheckpoint 的完整机制定义（存储结构、写入时机、恢复流程、changeLevel 分层留存）统一收敛到 §3.8 跨会话记忆机制，此处只保留对位关系。

**领域规则体系**：规则不再只是平铺的"行为模式列表"，而是一个包含权限分层（L0-L3）、来源分类（scenario_seed / user_initiated / model_inferred）、关联边（depends_on / conflicts_with / strengthens）和条件匹配（conditions）的规则网络图。注入方式"按 topicPath 子图匹配精准注入"。完整定义见 §3.7。

### 3.4 三类执行体分派（DET / LLM / HUMAN）

#### 3.4.1 分派判据

每个状态在宪法中必须显式声明执行体，判据顺序如下（优先级递减）：

A. 错误代价多大？
代价大且无客观答案 → 人
代价大但有客观校验 → LLM + 人确认
代价小 → 进入下一条

B. 能否写成硬规则 + 穷举？
能 → 确定性程序（DET）
不能 → 进入下一条

C. 是否有客观正确答案？
有（映射类、意图类）→ LLM
没有（偏好类、责任类）→ 人

关键规则："代价"为最高优先级判据。代价大的节点无论能否硬规则化，都必须经过人确认或 LLM + 人确认。

#### 3.4.2 与意图收敛机制的对应

DET 执行确定性收敛；LLM 执行有限展开后收敛，出口被 contract-out schema（输出契约模式）硬卡；人执行责任收敛，对不可逆决策负责。

### 3.5 人和 DET 作为终极容错闭环

态控不追求大模型的单次绝对正确，而是构建"犯错-拦截-纠错"的工程闭环。用户不仅是意图源头，更是系统状态的"重置按钮"。系统只需保证"大概率正确"，剩余边界错误由 DET 和用户纠正完成最终收敛。在高风险场景中，这一闭环从单用户现场扩展到组织级人工接管通道。

### 3.6 领域规则体系

**定位**：领域规则是态控架构的第三层宪法（领域规则层）在生产过程中逐步建立的约束体系。与宪法不同——宪法是设计者预设的刚性框架，领域规则是从用户的实际使用中"长出来"的经验沉淀。领域规则从平铺列表升级为可查询、可冲突检测、可按主题路径匹配的**规则网络图**。

**field_based 的等价物——contractEvolution**：topic_based 的规则按 topicPath（行业→品类→场景）分层，field_based 的规则按 stepName（环节名）分层——规则提炼和注入完全按环节隔离。两者机制相同（changeLevel → ruleMiningQueue → 异步提炼 → domainRules），唯一差异是匹配键：topic_based 用 topicPath 语义检索，field_based 用 stepName DET 精确匹配。

#### 3.6.1 规则来源三层

领域规则有三类来源，对应不同的可信度和确认流程：

| 来源 | 含义 | 灌入时机 | 权限初始值 | 是否走交叉验证 |
|------|------|---------|-----------|-------------|
| **scenario_seed** | 场景固有规则（工艺规范、设计基线、报价模板、安全红线等） | 主题初始化时按 topicPath 加载 | L0 / L1 | ❌ 不走，设计者预设可信 |
| **user_initiated** | 主题确认时用户主动口述的规则（如"厨房台面我喜欢高一点"） | ANALYZING 确认 topicName 后主动问询 | L2 | ✅ 走，用户可能说错或矛盾 |
| **model_inferred** | IN_SESSION 过程中模型推断的用户偏好（如"从对话中推测用户偏好 U 型厨房"） | complete 时顺带输出 → ruleCandidates | L3 | ✅ 需用户确认升格为 L2 |

#### 3.6.2 规则权限分层

领域规则内部存在 L0-L3 四级权限梯度。这一维度**与宪法层级正交**——宪法分三层（根/局部/领域），每条领域规则同时具有"宪法层级"（属于第 3 层领域规则）和"权限层级"（L0/L1/L2/L3），分别定义其适用范围和可修改性。

| 层级 | 名称 | 含义 | 可修改 | 默认 importance | 冲突时行为 |
|------|------|------|--------|-----------------|----------|
| **L0** | 钢印规则 | 场景硬性约束，不可触碰 | ❌ 不可修改 | critical（L0 规则默认 critical，受"同层 critical≤1"约束——L0 层内必须用户确认选哪条为 critical，详见 §3.9） | 无条件胜出，不询问 |
| **L1** | 基线规则 | 设计者预设最佳实践（如"厨房台面高度建议 85cm"） | ⚠️ 需设计者确认 | normal | 与 L2/L3 冲突时触发询问 |
| **L2** | 用户确认规则 | 用户主动说过或确认过的偏好 | ✅ 用户可随时修改 | normal | 优先于 L3 |
| **L3** | 模型推断规则 | 模型 infer 的候选规则 | ✅ 需用户确认升格 | normal | 仅作参考，不主动注入 |

**importance 与 immutableLevel 正交**：
- immutableLevel 是权限轴（谁能改），importance 是重要程度轴（哪个更关键）。同 immutableLevel 内，importance 高的胜出
- 每层（L0/L1/L2/L3）内 critical ≤ 1、high ≤ ⌊该层规则总数/3⌋、其余 normal
- L0 规则中必须选出 1 条 critical（安全红线——用户确认），其余可全 normal
- 冲突仲裁：先比 immutableLevel → 同层再比 importance → 同 importance 触发询问

**跨层仲裁的边界声明**：
- 当两条规则 immutableLevel 不同时（如 L0 vs L2），按 immutableLevel 直接裁决（L0 无条件胜出），**importance 不参与跨层仲裁**
- importance 的作用域限定为"**同 immutableLevel 平局时的次级裁决**"——只有在两条规则 immutableLevel 相同时，importance 高的才胜出
- 示例：L0(normal) vs L2(critical) 冲突时，L0 直接胜出，L2 的 critical 不参与仲裁；L2(critical) vs L2(normal) 冲突时，L2(critical) 胜出
- 此声明消除"importance 可以跨层影响仲裁"的误解，避免实现者误以为"高 importance 的低权限规则能压过低 importance 的高权限规则"

**规则状态**：

| 状态 | 含义 | 所在位置 |
|------|------|---------|
| active | 正式生效的规则 | domainRules |
| candidate | 待交叉验证的候选规则 | ruleCandidates |
| archived | 已废弃或被替代的历史版本 | topicEvolutionArchive |
| pending_review | 需人工审核（L0/L1 修改请求） | 单独队列 |

#### 3.6.3 主题层级与规则的作用域边界

topicPath 是一个**层级化数组**，从粗到细逐级展开。每一层都可以有自己独立的规则集——不同层级的规则互不污染。

**层级结构示例**：

```
全屋定制
├── 设计
│   ├── 厨房设计     ← 规则："台面高度85cm""水槽位置""油烟机型号偏好"
│   ├── 卫生间设计   ← 规则："防水等级""瓷砖规格""干湿分离要求"
│   └── 客厅设计
├── 报价             ← 规则："利润率""套餐模板""折扣权限"
├── 客户沟通
├── 拆单
└── 生产
```

**层级匹配规则（前缀匹配）**：

| 当前 topicPath | 能匹配到哪些规则 | 不能匹配到 |
|--------------|---------------|----------|
| `["全屋定制"]` | topicPath 前缀为 `["全屋定制"]` 的所有规则——包括"设计"、"报价"、"生产"等全部子层规则 | — |
| `["全屋定制", "设计"]` | 前缀为 `["全屋定制", "设计"]` 的规则——包括"厨房设计""卫生间设计""客厅设计" | `["全屋定制", "报价"]` 的规则（兄弟节点不可见） |
| `["全屋定制", "设计", "厨房设计"]` | 仅 `["全屋定制", "设计", "厨房设计"]` 自身的规则 | "卫生间设计""报价"等的规则（隔层不可见） |
| `["全屋定制", "报价"]` | 仅 `["全屋定制", "报价"]` 自身的规则 | "厨房设计"等的规则 |

**核心纪律**：
- **上级可见于下级**：`["全屋定制", "设计"]` 层的规则对 `["全屋定制", "设计", "厨房设计"]` 可见——因为前者是后者的前缀
- **下级不可见于上级/兄弟**：`["全屋定制", "设计", "厨房设计"]` 的规则不会注入到 `["全屋定制", "设计", "卫生间设计"]` 或 `["全屋定制", "报价"]` 的上下文中
- **层级深度与上下文窗口的关系**：当前 topicPath 越深，匹配到的规则集越小——这是认知局部性原则在规则注入层的自然体现

**规则提炼时的作用域要求**：

每一轮对话和每一个提炼出来的规则都必须带 topicPath 和 source 标记。模型推断的候选规则（model_inferred）在写入 ruleCandidates 时，必须附带当前对话所处的 topicPath。

#### 3.6.4 规则网络结构

领域规则不再是平铺的列表，而是一个有向图——每条规则是一个节点，节点之间通过关联边建立关系。

**节点定义**：

```json
{
  "ruleId": "kitchen_countertop_height_001",
  "content": "厨房台面高度建议85cm",
  "topicPath": ["全屋定制", "设计", "厨房设计"],
  "source": "scenario_seed",
  "immutableLevel": "L1",
  "importance": "normal",
  "status": "active",
  "conditions": {
    "userHeight": ">=170cm"
  },
  "edges": {
    "conflicts_with": ["kitchen_countertop_height_002"],
    "depends_on": [],
    "strengthens": ["ergonomic_guideline_003"]
  },
  "version": 1,
  "createdAt": "2026-07-09T10:00:00Z",
  "updatedAt": "2026-07-09T10:00:00Z"
}
```

**关联边类型**（统一 snake_case + 主动语态）：

| 边类型 | 含义 | 注入时行为 |
|--------|------|----------|
| **depends_on** | A 规则成立的前提是 B 规则已满足 | 匹配 A 时，检查 B 是否已激活；若否，主动询问 B |
| **conflicts_with** | A 和 B 互斥，不能同时应用 | 同时匹配到 A 和 B 时，触发询问让用户二选一 |
| **strengthens** | A 规则强化 B 规则的可信度 | 匹配到 A 时，B 的权重 +1，同层级内优先注入 |

**命名纪律**：关联边统一使用 snake_case（`depends_on` / `conflicts_with` / `strengthens`），语义统一为主动语态——当前规则是动作的发起者。`strengthens` 表示"当前规则强化了目标规则"，不是"被目标规则强化"。

#### 3.6.5 规则生命周期

```
主题确认（ANALYZING）
  │
  ├── 加载场景种子规则（scenario_seed）→ domainRules
  ├── 主动询问用户规则（user_initiated）→ ruleCandidates
  │
  ▼
IN_SESSION 运行
  │
  ├── 每轮：按 topicPath 做规则网络子图匹配 → 精准注入
  ├── 遇到冲突或不满足条件 → 主动询问用户
  ├── 模型 infer 出新规则（model_inferred）→ ruleCandidates
  │
  ▼
环节结束（complete / giveup / off-task）
  │
  ├── checkpoint → 推入 ruleMiningQueue
  ├── abandoned → 推入 ruleMiningQueue
  │
  ▼
N10 后台异步处理
  │
  ├── 压缩 stateChange 摘要（轻量 LLM）
  ├── 提取候选规则（继承 immutableLevel）
  ├── 自动推断规则间关联边（conflicts_with / strengthens / depends_on）
  │
  ▼
用户确认 → 升格为正式规则 → 写入 domainRules
```

#### 3.6.6 规则注入匹配逻辑（上下文管理器匹配层 + 决策层）

每轮用户输入的处理流程（由上下文管理器的匹配层和决策层执行）：

1. **匹配层**：获取当前 topicPath（如 ["全屋定制", "设计", "厨房设计"]），在 domainRules 中做子图匹配：
   - 匹配 topicPath 前缀一致的规则节点
   - 沿 depends_on 和 strengthens 边向外扩一层
   - 检查 conditions 是否满足
   - 冲突检测：若匹配到的规则中有 conflicts_with 关系的节点 → 标记冲突
   - 条件不满足检测：若匹配到的规则 conditions 字段有未满足的条件 → 标记缺失条件
2. **决策层**：按优先级决定注入哪些内容：
   - 冲突处理：按 immutableLevel 决定询问策略（L0 直接覆盖，L1 推荐 vs L2 用户偏好）
   - 条件缺失处理：生成询问问题
   - 注入排序：L0 > L1 > L2 > L3 → **同层级内按 importance 排序：critical > high > normal > low** → 同 importance 内按 strengthens 边的权重排序 → 同 importance 同 strengthens 权重时，按 ruleId 字典序（确定性 fallback，避免同分乱序）
3. **拼接层**：把决策结果组装成注入包，注入 prompt

**strengthens 权重累加上限**：
- 同一目标规则被多条 strengthens 边指向时，权重累加上限为 **+3**（cap）
- 累加超过 +3 的部分不再计入排序权重——避免一条规则被 N 条 strengthens 加权后垄断排序首位
- cap 值通过 tunable 参数 `strengthens_weight_cap`（默认 3，取值范围 [1, 10]）配置，由上下文管理器决策层消费

**注入包格式**：

```json
{
  "matchedRules": [
    {
      "ruleId": "...",
      "content": "...",
      "immutableLevel": "L1",
      "matchType": "exact_topicPath"
    }
  ],
  "conflictsDetected": [
    {
      "ruleA": "rule_id_1",
      "ruleB": "rule_id_2",
      "resolution": "pending_user_choice"
    }
  ],
  "missingConditions": [
    {
      "ruleId": "...",
      "condition": "userHeight",
      "question": "您的身高大概多少？这样我能更精准地建议台面高度。"
    }
  ]
}
```

**与认知局部性的关系**：模型不再收到"这个领域的所有规则"，而是只收到"当前主题路径上直接相关的那几条规则及其直接关联的规则"。这使上下文窗口占用从 O(领域规则总数) 降为 O(当前主题路径规则的子图规模)，是认知局部性原理在领域规则层的落地。

#### 3.6.7 规则提炼队列处理

```
checkpoint / abandoned
  │
  ▼
推入 ruleMiningQueue
  │
  ▼
N10 后台进程消费
  │
  ├── 压缩 stateChange 摘要（轻量 LLM）
  ├── 提取候选规则（model_inferred，L3）
  ├── 自动推断关联边：
  │   ├── 若两条规则 topicPath 相同且内容矛盾 → conflicts_with（标记为 inferred_conflict 候选状态）
  │   ├── 若一条规则引用了另一条的结果 → depends_on
  │   └── 若一条规则与另一条方向一致 → strengthens
  │
  ▼
写入 ruleCandidates
  │
  ▼
等待用户确认
  │
  ├── 确认 → 升格为 L2，写入 domainRules
  ├── 修改 → 按用户修改后写入，标记 L2
  └── 否决 → 丢弃或标记 rejected
```

**inferred_conflict 候选状态纪律**：

| 状态 | 含义 | 生命周期 | 是否参与运行时冲突检测 |
|------|------|---------|---------------------|
| `inferred_conflict` | 自动推断的冲突关系候选，待用户确认 | N10 自动推断后写入；用户确认前 | ❌ 不参与——避免未确认的 LLM 语义判断直接影响运行时仲裁 |
| `active_conflict` | 经用户确认生效的正式冲突关系 | 用户确认 inferred_conflict 后升格；写入 domainRules 的 edges 字段 | ✅ 参与——§3.6.6 注入匹配逻辑触发询问 |

**纪律依据**：自动推断"两条规则内容矛盾"本身就是 LLM 语义判断，引入了新的概率性环节。强制候选状态 + 用户确认升格，把概率性环节封在异步提炼阶段，运行时仍只消费确定性结果。

**降级处理**：若 inferred_conflict 长期未确认（超过 ruleMiningQueue 的处理周期），不应自动升格为 active_conflict，也不应直接丢弃——应保留为 inferred_conflict 状态，并在下一轮用户确认窗口中提示。

**交叉验证策略按来源区分**：
- scenario_seed（L0/L1）：不走交叉验证，设计者预设可信
- user_initiated（L2）：需交叉验证（用户可能口头说错或与既有规则矛盾）
- model_inferred（L3）：需交叉验证 + 用户确认升格

### 3.7 跨会话记忆机制

态控架构为两种 taskType 分别提供跨会话记忆机制：

| 模式 | 机制 | 匹配方式 | 核心存储 |
|------|------|---------|---------|
| topic_based | topicEvolution | topicId 语义锚点 + embedding 检索 | 四表（topicEvolution / topicEvolutionEvent / topicEvolutionArchive / topicIndex） |
| field_based | sessionCheckpoint | userId 精确匹配 + DET ttl 检查 | 单表（sessionCheckpoints） |
| 通用（两种 taskType 共享） | roomConversationLog | userId + roomId 精确匹配 | 单表（roomConversationLog） |
| field_based 规则演化 | contractEvolution | stepName 精确匹配 | 复用 contractStore 三表 |

**roomConversationLog 的独立保留策略**（适用于两种 taskType）：

roomConversationLog 是房间级对话原文记录，保留期独立于 sessionCheckpoint.ttl，按房间 importance 分级保留：

| 房间 importance | 保留期 | 过期处理 |
|----------------|--------|---------|
| critical | 永久 | 不清理 |
| high | 90 天 | 过期后归档到 topicEvolutionArchive（如为 topic_based）或 contractEvolution archive（如为 field_based） |
| normal | 30 天 | 过期后归档 |
| low | 7 天 | 过期后直接删除（低价值，不归档） |

保留期通过 tunable 参数 `room_conversation_log_retention` 配置（JSON 对象，按 importance 分级设置天数）。sessionCheckpoint.ttl 到期时，只清理 sessionCheckpoints 表的进度行，**不联动清理 roomConversationLog**——这保证了"自由穿梭"承诺：即使进度检查点过期，用户返回房间时仍能从 roomConversationLog 恢复对话上下文。

#### 3.7.1 topicEvolution（topic_based 专属）

主题演化体系——基于 topicId 的语义锚点，支持三层注入（已生效领域规则 + 关联主题摘要 + 分层演化事件包）、异步规则提炼、changeLevel 分层留存（major 永久 / minor 压缩 / invalid 归档）。

**机制本体分三层递进**：

**A 最底层——分层留存**：按信息净价值把每个事件判为 major、minor 或 invalid 三级，major 永久保留，minor 超阈值后汇总压缩，invalid 直接归档不入主表，判定权交还给模型在 turnType 中顺带输出，DET 不参与语义判定。三级判定基于用户在交互中的明确行为信号——例如版本号从 1.0 改为 2.0 是 major，从 2.0 改为 2.1 是 minor，用户明确说"这个内容不用，取消"是 invalid。判定依据是用户交互中的明确行为，不是模型的元认知自省。

**B 中间层——主题关联**：在 topicIndex 中建立八种关联关系，核心纪律是"关联在元数据层，隔离在数据层"，注入时只拉摘要卡片不拉原始事件链，模型知道隔壁主题的结论但不知道隔壁主题的聊天记录。

**C 最顶层——异步规则提炼**：从多条同类主题的演变流程和关联关系中提取被多次验证的行为模式，候选规则经交叉验证和用户确认后存入领域规则库，同类型新主题初始化时自动注入。规则提炼应为低频行为。

**领域规则体系**：规则是一个包含权限分层（L0-L3）、来源分类（scenario_seed / user_initiated / model_inferred）、关联边（depends_on / conflicts_with / strengthens）和条件匹配（conditions）的规则网络图。注入方式"按 topicPath 子图匹配精准注入"。完整定义见 §3.6。

**topicEvolution 与 roomConversationLog 的边界**：

| | topicEvolution | roomConversationLog |
|---|---|---|
| 记录什么 | 结构化变更事件（stateChange before/after/delta + askingField + modelOutputSummary） | 逐字对话原文 |
| 视角 | 演化追溯——记录"主题怎么变的" | 上下文恢复——记录"当时聊了什么" |
| 是否重复 | ❌ 不重复——同一轮对话的两种视角，event 是结构化摘要，log 是原文 | ❌ 不重复 |
| 压缩联动 | minor event 压缩为摘要时，对应 roomConversationLog 原文轮次归档（不删，但不再注入上下文） | 跟随 topicEvolution 的压缩节奏归档 |
| 消费方 | 上下文管理器三层注入 | 上下文管理器拼接层 |

#### 3.7.2 sessionCheckpoint（field_based 专属）

field_based 模式的原生跨会话进度记忆——记录用户在一个多步骤流程中的进度和各步骤已采集字段快照。完整 SQL DDL 见附录 C.3。

**调度器恢复流程**：

```
1. 用户唤醒
2. ANALYZING 阶段 DET 查 sessionCheckpoints
   ├─ 无记录 ∥ 已过期 → 正常进 P0
   └─ 有记录且未过期 →
        生成 resume_chat intent → 询问用户"上次做到 X，继续还是去其他环节？"
        ↓
        确认"继续" → 注入 completedSteps + stepSnapshots
                    N4 找第一个未完成步骤 → 路由到该步骤
        用户说"去 Nj" → 校验 Nj 前置依赖全部已满足
                    → 注入 stepSnapshots["nj"]（Nj 上次收敛快照）
                    → 路由到 IN_SESSION(nj)
        用户拒绝 → 清空 checkpoint，正常进 P0
```

**房间比喻**：每个环节是独立房间——返回 N1 时只注入 N1 的快照，不会混入 N10 的内容。

#### 3.7.3 contractEvolution：field_based 的领域规则演化

contractEvolution 是 §3.6 领域规则体系在 field_based 模式下的具体形态。与 topic_based 的 topicEvolution 不同——topicEvolution 管"主题怎么演化"，contractEvolution 管"环节内部的设计规则怎么变"。contractEvolution 是 topicEvolution 在 field_based 模式下的对位机制——前者按 stepName 精确匹配，后者按 topicId 语义匹配，两者互不替代。

**存储**（复用 contractStore 现有表，按 stepName 隔离）：

| 表 | 用途 | field_based 行为 |
|----|------|----------------|
| domainRules | 已生效领域规则 | ruleId 含 stepName 前缀，按 stepName DET 精确匹配注入 |
| ruleCandidates | 待验证候选规则 | 来源：每个 IN_SESSION complete 时模型推断的用户偏好 |
| ruleMiningQueue | 提炼队列 | 按 stepName 分队列——MiningQueue 的每条记录标注 stepName，异步提炼按 stepName 分组处理 |

**changeLevel 判定规则（field_based）**：

| 级别 | 判定标准 | 留存 |
|------|---------|------|
| major | 字段的结构性变更——新增/删除必采字段、修改字段值域、改变 Schema 结构 | 永久保留，不可压缩 |
| minor | 字段值微调——默认值修改、取值范围调整、措辞优化 | 30 条压缩为 1 条摘要 |
| invalid | 完全撤销——用户改了又改回原值（与上一版 contract-out 完全相同） | 归档后定期删除（90 天） |

**三层注入**（由上下文管理器执行）：

| 层 | 内容 | 匹配方式 |
|----|------|---------|
| 第0层 | 已生效领域规则 | stepName 精确匹配 + immutableLevel 过滤（L1 及以上） |
| 第1层 | 候选规则卡片 | stepName 匹配 + validationRate ≥ 阈值 |
| 第2层 | 变更历史 | 同一 stepName 的近期变更摘要（major 3 条 + minor 压缩摘要 + 当前规则状态快照） |

#### 3.7.4 contractEvolution 与 sessionCheckpoint 的关系

**设计原则**：
- 态控调度器只负责三件事：**全量记录 + 正确路由 + 资源协调**。三个数据源按各自匹配键全量记录，不做注入决策——roomConversationLog 按 userId+roomId，sessionCheckpoints 按 userId，contractStore 按 stepName 或 topicPath
- 每个环节是自治的——它的上下文管理器从态控调度器交给它的内容中按自己的拼接规则组装 prompt。上下文管理器是独立状态机，不执行环节宪法
- ttl 到期自动清除 sessionCheckpoints 进度行——设计假设：7 天内没继续的任务大概率被放弃了。但 roomConversationLog 不联动清理，按自己的 importance 分级保留期独立过期

**contractEvolution 与 sessionCheckpoint 的对位**：

| | contractEvolution | sessionCheckpoint |
|---|---|---|
| 记录什么 | 规则变更历史 | 流程进度 |
| 匹配方式 | stepName 精确 | userId 精确 |
| 留存策略 | 分层留存（永久/压缩/归档） | ttl 到期清除进度行（roomConversationLog 不联动） |
| 消费方 | 上下文管理器（三层注入） | 态控调度器（ANALYZING 路由） |

#### 3.7.5 跨会话记忆对比

| 维度 | topicEvolution | sessionCheckpoint | contractEvolution | roomConversationLog |
|------|---------------|-------------------|-------------------|---------------------|
| 适用 taskType | topic_based | field_based | field_based | 通用（两种 taskType） |
| 记录内容 | 主题演化历史 | 流程进度 + 字段快照 | 环节内规则变更历史 | 房间对话原文 |
| 存储规模 | 4 表 + topicIndex 9 张 | 1 表 | 复用 contractStore 3 表 | 1 表 |
| 检索方式 | embedding 两阶段 | DET 精确匹配 | DET stepName 精确匹配 | DET userId+roomId 精确匹配 |
| 留存策略 | 分层留存（永久/压缩/归档） | ttl 到期清除进度行 | 分层留存（永久/压缩/归档） | 按 importance 分级（永久/90天/30天/7天） |
| 注入方式 | 上下文管理器自决 | 态控调度器路由 + 上下文管理器注入字段 | 上下文管理器自决 | 上下文管理器拼接层 |
| changeLevel 判定 | 核心实体增删→major，微调→minor，撤销→invalid | 无（进度不是变更） | 字段结构变更→major，值微调→minor，改回原值→invalid | 无（原文不是变更） |

贯穿三层有两条铁律：一是人作为终极容错闭环，系统自动发现、自动判定、自动提炼，但主题名、关键词、关联关系、领域规则全部必须经用户确认后才生效；二是认知局部性不因主题关联而妥协，关联主题的原始数据不注入当前上下文，只注入经过提炼的结论摘要。整个机制的本质不是存储空间管理，是按信息净价值做筛选——大版本保全过程理解，小版本防膨胀，无效过程减噪声，关联在元数据层不混不孤立，规则异步沉淀越用越老练。

### 3.8 统一环节操作模型——五步流水线

态控架构的两种任务模式（field_based / topic_based）遵循相同的五步操作流水线。差异只在具体实现参数中——不是两套系统，是同一套系统的两种环节配置。

```
┌─────────────────────────────────────────────────────┐
│  步骤一：状态机路由                                    │
│  ┌─────────────────────────────────────────────────┐│
│  │ 用户输入到达 ANALYZING                           ││
│  │   ├─ 元指令匹配（唤醒/退出/取消）→ 直接处理         ││
│  │   ├─ sessionCheckpoint 恢复检查（field_based 专属） ││
│  │   ├─ topicIndex 语义检索（topic_based 专属）        ││
│  │   └─ 意图识别 → 确定目标环节                       ││
│  │ 前置依赖校验 → 通过? → 路由到 IN_SESSION(目标环节)  ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│  步骤二：环节上下文加载                                │
│  ┌─────────────────────────────────────────────────┐│
│  │ 从三个存储来源加载环节上下文:                       ││
│  │   ├─ roomConversationLog(roomId) 最近N轮对话      ││
│  │   ├─ sessionCheckpoints.stepSnapshots[环节]       ││
│  │   └─ contractStore.domainRules(stepName) 已生效规则 ││
│  │ topic_based 额外加载: topicEvolution + topicIndex  ││
│  │ field_based: stepName DET 精确匹配                 ││
│  │ topic_based: topicId embedding 语义匹配            ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│  步骤三：内容拼接（环节自治——上下文管理器三层职责）     │
│  ┌─────────────────────────────────────────────────┐│
│  │ 上下文管理器执行匹配→决策→拼接三层确定性职责:       ││
│  │   ├─ 匹配层：规则子图匹配、冲突检测、条件检查         ││
│  │   ├─ 决策层：按 importance/immutableLevel/strengthens ││
│  │   │           权重决定注入哪些内容                  ││
│  │   └─ 拼接层：组装 prompt，做 token 预算分配与截断    ││
│  │     ├─ 对话窗口: 取最近N轮（默认20轮）              ││
│  │     ├─ 结构化字段: 注入collectedFields（仅field_based）││
│  │     ├─ 规则注入: 按决策层结果注入                   ││
│  │     └─ 搜索片段: 用户引用历史时追加匹配片段          ││
│  │ 上下文管理器是独立状态机——做匹配→决策→拼接，不执行环节宪法││
│  │ 态控调度器不干预拼接                                 ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│  步骤四：模型计算                                     │
│  ┌─────────────────────────────────────────────────┐│
│  │ 模型无状态——它只看到拼接好的上下文 + 用户最新输入    ││
│  │ 在局部宪法约束下自主完成:                          ││
│  │   ├─ 多轮追问                                     ││
│  │   ├─ 字段/值域校验（field_based: DET辅助）         ││
│  │   ├─ 语义边界判断（topic_based: embedding辅助）    ││
│  │   └─ 收敛检测 → turnType=complete               ││
│  │ 出口: 结构化JSON（contract-out schema）           ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│  步骤五：记录落盘                                     │
│  ┌─────────────────────────────────────────────────┐│
│  │ 实时记录: 每turn → roomConversationLog追加一行    ││
│  │ 收敛记录: complete → sessionCheckpoint更新快照    ││
│  │ 变更记录: changeLevel event → contractStore       ││
│  │ 规则提炼: ruleMiningQueue积压 → N10异步提炼        ││
│  │ 上下文清空: 物理清除LLM上下文，仅结构化数据持久化     ││
│  └─────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────┘
```

**五步流水线中的两种环节配置**：

| 步骤 | field_based 配置 | topic_based 配置 |
|------|-----------------|-----------------|
| 路由 | stepName 精确匹配，用户说"去N1"即切 | topicId embedding 检索，"上次那个写诗的项目"需语义匹配 |
| 切换确认 | 进度保护——有未提交字段时弹"保存后切换?" | 意图确认——embedding匹配到多个topicId时弹"是哪一个?" |
| 上下文 | collectedFields + roomConversationLog(roomId) | topicSnapshot + topicEvolution events + roomConversationLog(roomId) |
| 校验 | DET字段白名单 + 值域硬校验 | LLM语义边界 + DET confidence软校验 |
| 出口 | JSON Schema硬卡死，字段齐全=完成 | event类型判断，changeLevel分层留存 |
| 记忆 | sessionCheckpoint + roomConversationLog | topicEvolution + roomConversationLog |

**核心纪律**：
- 态控调度器只管步骤一（路由）和步骤五（记录），不干预步骤二三四
- 上下文管理器管步骤三（匹配→决策→拼接）——独立状态机，不执行环节宪法
- 环节宪法管步骤四（模型行为规则）
- 五步流水线的**接口统一**：两种 taskType 在外部接口上完全相同（路由信号→上下文加载→拼接输出→模型计算→落盘确认）。**内部实现分支**：步骤一匹配模式（DET vs embedding）、步骤二加载源（sessionCheckpoint vs topicEvolution）、步骤三锚点模式（字段硬门控 vs 三层注入+两阶段检索）、步骤四校验方式（值域 vs 格式）、步骤五落盘目标（sessionCheckpoint vs topicEvolution event）各有差异

### 3.9 @importance 重要性分层机制

与 `immutableLevel`（权限轴：谁能改）正交的第二轴——重要程度轴（哪个更关键）。房间多、规则密，需要一套标记告诉上下文管理器"有限的 token 预算优先分给谁"。

**三层结构，独立计数：**

| 层级 | 标记位置 | 说明 | 示例 |
|------|---------|------|------|
| **房间级** | 环节宪法头部 `<!-- @importance -->` | 同一 taskType 下所有环节中，哪个最重要 | field_based 16 个环节中 N1 为 critical；topic_based 全屋定制中设计环节为 critical |
| **产出物级** | 环节宪法 `<!-- @importance:deliverable -->` | 同一环节的 machineReadable 中哪个产出物是核心 | N1 的 doList 为 critical；设计方案的厨房设计为 critical |
| **规则级** | 领域规则节点 `ruleNode.importance` | L0/L1/L2/L3 每层独立，各层内最多 1 条 critical | L0 安全红线为 critical；L1 某条基线规则为 critical |

**枚举值及约束：**

| 值 | 含义 | 每层上限 | 作用 |
|------|------|---------|------|
| critical | 极其重要——此层内最关键 | 1（必须用户确认） | roomConversationLog 保留永久；规则排 prompt 头部；UI 高亮 |
| high | 高重要 | ≤ ⌊该层总数/3⌋，可为 0 | roomConversationLog 保留 90 天；规则优先注入 |
| normal | 普通（默认） | 无限制 | roomConversationLog 保留 30 天；正常排序 |
| low | 低重要 | 无限制 | roomConversationLog 保留 7 天；可跳过注入 |

**消费方：上下文管理器拼接层策略**

buildPromptContext() 在组装 token 预算时按 importance 分级：

1. **roomConversationLog 截断**：汇总全部环节的历史记录时，按环节 `@importance` 分配保留轮次——critical=永久保留但拼接取最近10轮 / high=90天保留但拼接取最近6轮 / normal=30天保留但拼接取最近3轮 / low=7天保留但拼接取最近1轮。保留期管"存多久"，拼接轮次管"这次用多少"，两者独立。

2. **规则注入排序**：模型注意力在 prompt 头部衰减最慢。规则注入按 `ruleNode.importance` 排序——critical→high→normal→low。同 importance 内按 strengthens 权重。

3. **冲突仲裁**：两条同 immutableLevel 的规则冲突时，importance 高的直接胜出，不触发 LLM 调用。**跨层冲突时（如 L0 vs L2）按 immutableLevel 直接裁决，importance 不参与跨层仲裁——详见 §3.6.2 跨层仲裁的边界声明**。

约束纪律：high 是稀缺标签——不能随手标。极端合法配置：每层 1 critical + 0 high + 其余全 normal。

**防偷懒规则——每层必须恰好 1 条 critical 或显式声明 allow_no_critical：**

| 层级 | 是否允许 0 critical | 说明 |
|------|-------------------|------|
| 房间级 | ❌ 必须恰好 1 条 | 同一 taskType 下必须有一个核心环节——用户确认 |
| 产出物级 | ❌ 必须恰好 1 条 | 同一环节的 machineReadable 中必须有一个核心产出物——用户确认 |
| 规则级 L0 | ❌ 必须恰好 1 条 | 根宪法 8 条中安全红线即为 critical——设计者确认 |
| 规则级 L1 | ❌ 必须恰好 1 条 | 基线规则中必须有一个命门——设计者确认 |
| 规则级 L2 | ⚠️ 可 `allow_no_critical: true` | 用户确认规则中可能没有"唯一最关键"的偏好——在环节宪法 `@outputs` 旁标注理由 |
| 规则级 L3 | ⚠️ 可 `allow_no_critical: true` | 模型推断规则天然没有"唯一最关键"——在环节宪法 `@outputs` 旁标注理由 |

`allow_no_critical` 由 N2 覆盖率校验时审查。申请时在宪法头部 `<!-- @importance -->` 旁标注：`<!-- @importance:allow_no_critical reason="L2层均为平权用户偏好，无唯一命门" -->`。不允许"0 critical + N high"绕过——要么恰好 1 critical，要么显式声明 0 critical + 理由。

## 四、节点体系

### 4.1 节点意图收敛链

每个节点将弥散输入收敛到当前目标所要求的结构化输出。在环节级自治（环节级自治）范式下，收敛链（收敛链）的"环节内"部分由模型实例自主完成：

| 节点 | 输入（弥散态） | 收敛目标（contract-out（输出契约）） | 执行体 | taskType 影响 |
|------|--------------|----------------------------------|--------|-------------|
| **ANALYZING（分析态）** | 用户口语"柜体深度改一下" | `{choice, logprobs, intent, inputNature, extracted, topicId}`——双维度分类，强制选择 + API logprobs 替代 LLM 自由生成 confidence。intent 由 DET 从 choice 映射（如 choice=A→intent="resize"）。ANALYZING 产出意图识别相关信号（intent/choice/logprobs/inputNature/extracted/topicId），**不产出 turnType**——turnType 是 IN_SESSION 的输出格式 | LLM | choice 由模型从有限选项中选择；logprobs 由 API 返回；inputNature 并行分类；DET 读取 logprobs 交叉裁决路由 |
| IN_SESSION(field_based) | intent+口语+参数树（环节内多轮） | `{turnType, askingField=字段名, message, collectedFields, result, changeLevel, changeLevelReason}` | LLM（环节内自治） | askingField 为预定义字段名；每 turn 实时写入 roomConversationLog；complete 时更新 sessionCheckpoint 和 contractEvolution |
| IN_SESSION(topic_based) | intent+口语+主题边界+topicId+topicEvolution 最新 stateSnapshot（环节内多轮） | `{turnType, askingField=主题标签, message, collectedFields, result, changeLevel, changeLevelReason}` | LLM（环节内自治） | askingField 为动态主题标签；每 turn 实时写入 roomConversationLog；exit 时追加 event 到 topicEvolution；模型可 infer 新规则（model_inferred）→ ruleCandidates |
| DET 复验 (field_based) | result+constraints（complete 时） | `{valid:true}/{valid:false, reason}` | DET | 值域校验 + contractEvolution 检测（changeLevel 事件分层留存） |
| DET 复验 (topic_based) | result+format-constraints（complete 时） | `{valid:true, reason}` | DET | 格式校验 + topicEvolution event 追加 |
| WAITING_CONFIRM（等待确认） | previewUrl（预览地址）+diffSummary（差异摘要） | `{decision（决策）:"confirm（确认）"/"reject（拒绝）"}` | 人 | — |
| EXECUTING（执行态） | fieldPath（字段路径）+newVal（新值）+modelId（模型标识） / result+outputTarget（输出目标） | `{status（状态）:"success"}` | DET | 确定性收敛；topic_based 任务在 EXECUTING 阶段将输出写入文件后，追加 event 到 topicEvolution。**v5.4 修正：EXECUTING 仅保留给纯写入/删除操作（record INSERT / delete DELETE）；查询/对比类 intent 的数据库操作在 IN_SESSION 内通过工具调用完成，不经过 EXECUTING** |

**IN_SESSION 工具调用能力（v5.4 新增）**：IN_SESSION 模型可在环节内调用 DET 工具（function calling），用于查询/计算等需要数据库操作的 intent（如 query/compare）。LLM 做语义理解+结果格式化，DET 工具做 SQL 查询。工具调用结果直接由 LLM 消费并生成自然语言回复，不经过独立 EXECUTING 状态。

**IN_SESSION 工具分层发现机制（v5.5 新增）**：工具按三层体系组织，解决工具数量多时上下文超长+准确率下降问题（行业验证：100+工具时准确率降至~15%，两阶段选择可恢复至80%+）：

1. **必用工具**（环节宪法 @section tools 声明）：环节核心工具的**完整定义**（含参数 schema），调度器进入 IN_SESSION 时自动注入。每个环节 1-3 个，常驻上下文。
2. **工具清单**（环节宪法 @section tool_catalog 声明）：选用工具的**轻量级目录**（仅 name + 一句话描述），每次注入。占用极少 token（每个工具约 20 token）。模型从清单中识别需要的工具。
3. **选用工具**（系统级工具库 toolRegistry 注册）：完整定义按需加载——模型从清单中选择后返回工具名，调度器从工具库加载完整定义注入。不用时上下文占用为 0。

**工具发现流程**（Two-Phase Hierarchical Selection，行业最佳实践）：
```
阶段1 Search：模型看工具清单（轻量级）→ 识别需要的工具 → 返回 needed_tools 列表
阶段2 Load：调度器从工具库加载完整定义 → 注入模型上下文
阶段3 Execute：模型调用工具 → DET 执行 SQL → 结果返回模型 → 模型生成回复
```

**meta-tool search_tools**（始终可用）：当工具清单中没有所需能力时，模型可调用 search_tools(query) 从系统级工具库检索，扩展工具清单。这是 MCP-Zero 主动发现机制的落地。

**工具内部时间解析**：涉及时间的工具参数支持自然语言时间表达（如 "this_month" / "yesterday"），DET 工具内部解析为具体日期。减少模型对辅助工具（如 get_current_date）的依赖。

**与认知局部性的关系**：工具分层是认知局部性在工具层的落地——每个环节只收到必用工具完整定义 + 选用工具轻量级清单，不是全部工具的完整定义。

**PREVIEWING（预览态）**：PREVIEWING 是 IN_SESSION 的可选子环节，不是独立节点。在需要可视化 diff（差异）的不可逆操作场景下启用，通常仅 field_based（基于字段）任务需要（如改柜体参数前预览 diff），topic_based（基于主题）任务通常不需要。PREVIEWING 的 contract-in/out 为 `{fieldPath, old, new, modelId} → {previewUrl, diffSummary}`，作为 IN_SESSION 内部的一次子调用，不独立出现在节点意图收敛链中。详见 §4.2.1。

**turnType 与 changeLevel 的区别**：
- `turnType`：轮次类型（ask/reply/complete/validation_failed/off-task/giveup），描述当前轮的交互状态，每 turn 都输出
- `changeLevel`：变更级别（major/minor/invalid），描述本轮变更相对上一版的信息净价值，仅在内容发生变更时输出
- 两者独立判定：turnType=complete 时 changeLevel 可能为 major（结构性变更后完成），也可能为 minor（微调后完成）

### 4.2 节点状态

#### 4.2.1 标准状态集（环节级自治方式）

| 状态 | 类型 | 含义 | 默认执行体 | 默认模型（LLM节点） | 备注 |
|------|------|------|-----------|-------------------|------|
| IDLE（空闲态） | 稳态 | 空闲，等待唤醒 | — | — | 无人/无程序 |
| LISTENING（监听态） | 稳态 | 待命，等待用户输入 | 人 | — | 唤醒后进入 |
| ANALYZING（分析态） | 瞬态 | 意图识别环节 | LLM | 通用轻量模型 | 有限展开后收敛到 intent（意图）。ANALYZING 是环节，受根宪法第1条（环节隔离）约束——只能做意图识别，不能采字段、不能追问、不能给最终回复。topic_based 时自动生成 topicId 并初始化 topicEvolution。topic_based 任务额外执行规则问询（user_initiated 来源） |
| IN_SESSION（会话中） | 稳态 | 环节执行中 | LLM | 通用模型（状态级默认）/ 专用模型（意图级绑定） | 模型实例在环节内自主对话，调度器等待返回。子状态通过模型返回的 turnType（轮次类型）可观测：ask（追问）/reply（回复）=追问中，validation_failed（校验失败）=校验失败追问中，off-task（偏离任务）=偏离处理中，complete=待 DET 复验。taskType（field_based / topic_based）在意图识别阶段确定，影响 askingField 语义和 context-manager 锚点模式。topic_based 任务在节点 exit 时调度器自动追加 event 到 topicEvolution |
| └─ PREVIEWING（预览态，可选子环节） | 子环节 | 可视化 diff 预览 | DET（可选） | — | IN_SESSION 的可选子环节，非独立节点。在涉及不可逆操作的场景启用（如改柜体参数前预览）。contract-in：`{fieldPath, old, new, modelId}`；contract-out：`{previewUrl, diffSummary}`。完成后回到 IN_SESSION 等待用户确认，进入 WAITING_CONFIRM。通常仅 field_based 启用 |
| CLARIFYING（澄清态） | 守护态 | 校验失败追问转发 | DET（转发） | — | 调度器复用模型 message 转发用户（不调 LLM）。此状态为监控标记态，便于运维监控校验失败率和触发频率 |
| WAITING_CONFIRM（等待确认） | 守护态 | 等待用户确认 | 人 | — | 责任节点，不可逆决策 |
| EXECUTING（执行态） | 瞬态 | 执行操作 | DET | — | 确定性收敛。执行前先过 DET 值域复验。topic_based 任务在 EXECUTING 阶段将输出写入文件后，调度器自动追加 event 到 topicEvolution，stateSnapshot 标记为 checkpoint（如 turnType 为 complete）或 active（如为中间步骤） |
| **SLACK_NODE（松弛节点）** | **稳态** | **仅根宪法约束的闲聊** | **LLM** | **通用轻量模型** | 用户闲聊、发散、跳跃话题。输出自然语言文本，不进入下游确定性节点。受根宪法4条约束（含环节隔离——但 SLACK_NODE 的"本环节任务"定义为"自然对话，无需收敛"）。"切断房间"→ 回 LISTENING。详见 §4.2.2 |
| **TEMP_TOPIC（临时主题）** | **稳态** | **运行时创建的仅根宪法房间** | **LLM** | **通用模型** | ANALYZING 返回 intent=other + inputNature=T 时，用户命名后动态创建。仅受根宪法4条约束，无环节管道。一旦创建即纳入房间体系：roomId、keywords、roomConversationLog、上下文管理器、roomStateIndex。"切断房间"切换。用户可触发总结产出。N12 远期可检测聚集建议升级为正式房间。详见 §4.2.2 |
| CLOSING（关闭态） | 瞬态 | 退出 | DET | — | 清理资源 |

#### 4.2.2 节点类型体系（三级约束梯度）

态控有三类节点，分别对应不同的约束强度和收敛义务：

| 节点类型 | 约束强度 | 宪法约束 | contract-out 要求 | 是否参与收敛链 | 切换 |
|---------|---------|---------|-------------------|-------------|------|
| 收敛节点 | 高 | 环节宪法 + 根宪法 | 结构化 JSON（field_based）或 结构化 JSON + topicEvolution event 追加（topic_based） | 是 | "切断房间" |
| **TEMP_TOPIC（临时主题）** | **中** | **仅根宪法 4 条** | **用户触发总结时生成结构化产出** | **否（单态，无环节管道）** | **"切断房间"** |
| 松弛节点（SLACK_NODE） | 低 | 仅根宪法 4 条 | 自然语言文本 | 否（暂停态） | "切断房间" → LISTENING |

三者统一用 M1 元指令"切断房间"退出。约束轻重只决定**房间里有什么**（宪法层级、产出能力、存储），不改变退出方式。

**TEMP_TOPIC**：运行时动态创建的房间——不是 N1 预定义的正式房间，也不是无锚点的 SLACK_NODE。一旦用户命名（如"新能源趋势分析"）并确认关键词，即纳入 roomStateIndex，与其他房间并列。特性：

- **宪法**：仅根宪法4条，无环节宪法、无领域规则
- **环节管道**：无（单态，不经过 ANALYZING→IN_SESSION→EXECUTING 管道）
- **上下文管理**：上下文管理器拼接 conversationLog（与其他房间同一机制）
- **切换**："切断房间" + S3 物理释放，切换后可从 roomStateIndex 返回
- **产出**：用户触发"帮我总结"时生成结构化产出，存入 contractStore
- **升级路径**：N12 规则提炼远期可检测 temp topic 聚集模式 → 建议升级为正式房间

**TEMP_TOPIC 创建流程**：
```
ANALYZING: intent=other + inputNature=T
  → 调度器："这是一个新话题，起个名字方便以后切换。叫什么？"
  → 用户起名 + 确认关键词
  → 调度器创建 room:
     { roomId, roomName, keywords: [...], constitution: root_only, stages: none }
  → 写入 roomStateIndex
  → 进入 TEMP_TOPIC 单态会话
```

**松弛节点（SLACK_NODE 纳入统一体系）**：合法存在的低约束节点，仅受根宪法约束。用户在松弛节点中可以闲聊、发散、跳跃话题。松弛节点的输出不进入下游确定性节点——它是一段"暂停"，不是管道的一部分。**"切断房间" → 回 LISTENING，与其他房间统一用 M1 元指令退出。**

松弛节点的存在使智能体能够支持"闲聊→任务"的自然过渡（如用户在闲聊中突然说"算了不聊柜子了，帮我写首诗"）。但松弛节点的边界本身需要经过边界紧张度测试——如果闲聊落出率高，说明松弛节点的边界划定有问题，需要回溯第一步重审边界。

松弛节点不区分 taskType——松弛节点本身就是"暂停"，不参与收敛。TEMP_TOPIC 同样不区分 taskType——TEMP_TOPIC 本身就是没有环节管道的单态房间。

#### 4.2.3 节点类型动态转换规则

| 转换方向 | 触发条件 | 动作 | 守卫规则 |
|---------|---------|------|---------|
| 松弛节点 → 收敛节点 | 用户输入明确任务指令 | 意图识别接管，进入收敛链 | 架构机制 M1 优先匹配；logprobs probability≥0.7 方可转换 |
| 收敛节点 → 松弛节点 | 用户放弃或主动脱离 | contract-out 标记为 abandoned，回退至 IDLE 或进入松弛节点 | 架构机制 M1 定义放弃指令匹配模式，不接受环节宪法覆盖 |
| 收敛节点 → 另一收敛节点 | 用户发出元指令 | 物理隔离当前上下文 Γ，动态实例化新节点 | 元指令集由架构机制 M1 定义，不接受动态修改 |
| **TEMP_TOPIC → 收敛节点** | **用户输入匹配已知 intent** | **ANALYZING 重新识别，进入对应正式房间** | **logprobs probability≥0.7** |
| **收敛节点 → TEMP_TOPIC** | **ANALYZING: intent=other + inputNature=T** | **用户命名后创建 TEMP_TOPIC 房间** | **用户确认房间名和关键词** |
| **SLACK_NODE → LISTENING** | **"切断房间"** | **M1 口令精确匹配 → S3 物理释放 → 回 LISTENING** | **DET EXACT_MATCH** |
| **TEMP_TOPIC → LISTENING** | **"切断房间"** | **M1 口令精确匹配 → S3 物理释放 → 归档 → 回 LISTENING** | **DET EXACT_MATCH** |

taskType（任务类型）转换：taskType 在意图识别阶段确定后，运行时不变。如果用户在 field_based（基于字段）任务中突然切换到 topic_based（基于主题）任务（如记账中突然说"帮我写首诗"），这通过元指令识别机制触发节点切换——物理隔离当前上下文，进入新的 IN_SESSION（会话中）（topic_based（基于主题））。taskType 不在环节内动态切换。

#### 4.2.4 节点模型选择

第一层（状态级默认模型）：每个 LLM 节点有预设的默认模型实例，不依赖意图。ANALYZING（分析态）→ 通用轻量模型（意图识别不需要深度能力）；IN_SESSION（会话中）→ 通用模型。此层解决"ANALYZING（分析态）节点本身用哪个模型"的循环依赖。

第二层（意图级绑定）：当节点产出意图后，下游节点根据路由表中的"意图-模型绑定"覆盖默认模型。IN_SESSION（会话中）+ intent（意图）="write_poem（写诗）" → 专用文学模型。详见 §5 路由表设计。路由表中"模型覆盖"列为"—"时使用第一层默认模型；为具体模型名时使用第二层覆盖。

taskType 对模型选择的影响：topic_based（基于主题）任务倾向使用深度模型（主题理解和创意生成需要深度能力），field_based（基于字段）任务通用模型即可（字段映射不需要深度能力）。路由表第二层绑定时，topic_based（基于主题）intent 可绑定专用深度模型，field_based（基于字段）intent 用默认通用模型。

## 五、路由表

路由表是确定性代码，不调 LLM。路由规则必须在设计时显式定义。路由表同时承担两个职责：状态路由（决定下一个状态）和模型选择（决定 LLM 槽位由哪个模型实例填充）。

两层模型选择机制：
第一层（状态级默认模型）：每个 LLM 节点有预设的默认模型实例，不依赖意图。此默认值定义在节点定义中（见 §4.2.1 标准状态集的"默认模型"列）。
第二层（意图级绑定）：当上游节点产出 intent（意图）后，下游节点可根据"状态+intent（意图）"组合覆盖默认模型。此绑定配置在路由表中。

**路由表示例：**

| 当前状态 | contract-out（输出契约）关键字段 | 下一状态 | 路由规则 | 模型覆盖（第二层） |
|---------|--------------------------------|---------|---------|-----------------|
| ANALYZING（分析态） | intent="resize（调整尺寸）" (field_based) | IN_SESSION | 硬匹配 | —（IN_SESSION 用默认通用模型） |
| ANALYZING（分析态） | intent="write_poem（写诗）" (topic_based) | IN_SESSION（含 CONSTITUTION_GEN 例外路径） | 硬匹配 | 专用文学模型（topic_based 需深度能力） |
| ANALYZING（分析态） | intent="roleplay（角色扮演）" (topic_based) | IN_SESSION | 硬匹配 | 专用角色扮演模型 |
| ANALYZING（分析态） | intent="exit（退出）" | CLOSING（关闭态） | 硬匹配 | — |
| **ANALYZING** | **intent=other + inputNature=S（闲聊）** | **SLACK_NODE** | **硬匹配** | **通用轻量模型** |
| **ANALYZING** | **intent=other + inputNature=T（任务导向）** | **TEMP_TOPIC** | **硬匹配（先经用户确认命名）** | **通用模型** |
| **SLACK_NODE** | **"切断房间"（M1 口令）** | **LISTENING** | **DET EXACT_MATCH** | **—** |
| **TEMP_TOPIC** | **"切断房间"（M1 口令）** | **LISTENING** | **DET EXACT_MATCH** | **—** |
| **TEMP_TOPIC** | **用户输入匹配已知 intent** | **ANALYZING** | **硬匹配** | **—** |
| IN_SESSION | turnType（轮次类型）="complete" + intent="resize" | EXECUTING（执行态）（经 DET 值域复验） | 硬匹配 | — |
| IN_SESSION | turnType="complete" + intent="write_poem" | EXECUTING（执行态）（经 DET 格式校验） | 硬匹配 | 专用文学模型 |
| IN_SESSION | turnType="validation_failed" | CLARIFYING（澄清态） | 硬匹配 | — |
| IN_SESSION | turnType（轮次类型）="off-task" | ANALYZING（分析态） | 硬匹配 | — |
| IN_SESSION | turnType="giveup" | LISTENING（监听态） | 硬匹配 | — |
| IN_SESSION | turnType（轮次类型）="ask"/"reply" | IN_SESSION（继续） | 硬匹配 | — |
| CLARIFYING（澄清态） | — | IN_SESSION | 硬匹配 | — |
| EXECUTING（执行态） | status="success" | LISTENING（监听态） | 硬匹配 | — |

ANALYZING（分析态）节点本身使用其状态级默认模型（通用轻量模型），不依赖第二层绑定，循环依赖打断。

## 六、容错与纠错机制

### 6.1 软约束与硬降级链

#### 6.1.1 硬卡机制

contract-out schema（输出契约模式）是硬边界，格式错误或违反宪法直接退回上游重试。

#### 6.1.2 降级机制

重试上限触发后，强制降级路由至 CLARIFYING（澄清态）或 WAITING_CONFIRM（等待确认）。

#### 6.1.3 四项降级链（通用底线，不可缩减）

L1 结构校验 + DET 值域复验 + logprobs 检查 + 硬编码兜底。field_based 和 topic_based 的差异仅在 DET 值域复验（值域校验 vs 格式校验）。

#### 6.1.4 增强降级链（高风险场景扩展）

对于医疗预问诊、复杂客服、金融风控等高危场景，在四项降级链之后可追加 human_takeover（人工接管）扩展层（simple（简单）→ escalated（升级）→ human_takeover（人工接管），含 SLA（服务等级协议））。SLA 超时未响应时按 preset（预设）降级到 hardcoded（硬编码）或 abort（中止）。详见附录 A。四项降级链是通用底线，human_takeover（人工接管）是高风险场景的扩展，不可替换四项中的任何一项。

#### 6.1.5 降级链检查 + 硬编码兜底

| 项 | 执行体 | 时机 | 对应约束 | 说明 |
|----|--------|------|-------------|------|
| L1（一级）结构校验 | DET | complete（完成）时 | 根宪法第2条（输出格式） | JSON.parse（JSON解析）+ Schema（模式）校验，失败→重试1次→熔断 |
| DET 值域复验 | DET | complete 时 | — | field_based（基于字段）做值域校验（金额上限、时间格式、必填字段…），topic_based（基于主题）做输出格式校验（JSON Schema / 文本格式 / 字数 / 结构…）。规则来源：环节宪法 @section validation 的结构化 JSON） |
| logprobs 检查 | DET | complete 时 | — | ANALYZING 用强制选择+logprobs，DET 读取 logprobs 转换为 probability 做阈值裁决（probability < threshold → intent=other）。IN_SESSION 的"硬撑式确认"检测保留 pattern 扫描词表。冷启动期间采用软拦截——标记 warning 附加到返回中，不直接拦截；冷启动结束后恢复硬拦截 |
| 硬编码兜底 | DET | 熔断时 | — | 所有校验失败后的最终兜底，硬编码回复 |

**规则冲突时的询问逻辑作为扩展守卫**——当匹配到的领域规则中存在 conflicts_with 关系的节点时，触发询问让用户决策。此守卫不属于四项检查，是运行时决策辅助，按 immutableLevel 决定询问策略（L0 直接覆盖不询问，L1 推荐询问，L2 偏好询问）。

### 6.2 用户纠错的状态路由

状态机维护一个持久化的 contractStore（契约存储），保存每个节点完成时的 contract-in（输入契约）和 contract-out（输出契约）副本。contractStore（契约存储）独立于执行体实例的生命周期——Agent（智能体）亡后，其产出的契约副本仍可被后续节点或回滚流程访问。

参数级纠错：路由至 CLARIFYING（澄清态），提取新参数，重新进入执行管道。CLARIFYING（澄清态）复用模型 message 转发用户（不调 LLM——调度器的消息转发职责，详见 §2.4-A）。field_based（基于字段）任务的参数级纠错是字段值修改，topic_based（基于主题）任务的参数级纠错是主题方向微调。

意图级纠错（逃生舱）：若用户反馈表明当前意图完全错误，状态机支持强制回滚至 ANALYZING（分析态）节点重新分类（调度器的路由职责，详见 §2.4-A）。回滚数据流如下：

- 状态机从 contractStore（契约存储）取出 ANALYZING（分析态）节点的原始输入副本（ANALYZING 的输入是用户口语，非结构化 contract-in，存储的是原始用户输入文本）
- 重新实例化 ANALYZING（分析态）节点（注入原始输入 + 宪法）
- 重新执行意图识别，产生新的 intent（可能伴随 taskType 变化）
- 清空当前 activeSession，按新 intent 和新 taskType 重新分发

环节内偏离处理：当模型返回 turnType（轮次类型）="off-task（偏离任务）"时，调度器暂存 collectedFields（已采集字段）到 _pendingCollectedFields（待定已采集字段）（默认 24h，可配置，存储位置后续章节定义）——调度器的资源协调职责（状态暂存）。将 offTaskInput（偏离任务输入）送入 ANALYZING（分析态）重新意图识别。下次同一用户进入同一环节时，调度器询问是否继续上次未完成的记录。field_based（基于字段）任务的 collectedFields（已采集字段）是已采集的字段，topic_based（基于主题）任务的 collectedFields（已采集字段）是已讨论的主题摘要。

用户明确离开：直接作废，清空 activeSession（活跃会话）。不留草稿，不保存未完成记录，不恢复现场。_lastRecord（最后记录）不清空（用于"记错了"场景，后续章节定义）。

用户放弃：模型返回 turnType（轮次类型）="giveup（放弃）"→ 调度器清空 activeSession（活跃会话）→ LISTENING（监听态）。

### 6.3 系统崩溃恢复快照

非正常退出（系统崩溃、断电等）时，依赖 sessionCheckpoint 的自动快照恢复——activeSession 的 collectedFields 和当前环节名自动落盘，下次启动恢复现场。此为独立的容错机制，与元指令切换无关——元指令是用户主动行为，不需要快照保护。

## 七、约束体系

### 7.1 三层宪法体系总览

按约束对象分层，每一层的约束对象唯一。

| 层级 | 名称 | 约束对象 | 是否可修改 | 说明 |
|------|------|---------|-----------|------|
| 第0层 | 根宪法 | **所有环节模型**（ANALYZING + IN_SESSION） | 设计者写死，不可运行时修改 | 不可逾越的系统红线 |
| 第1层 | 环节宪法 | **特定环节模型** | 设计者静态编写（默认）；开放型环节可运行时动态生成 | 环节内模型的行为规则 |
| 第2层 | 领域规则 | 特定主题类型 | 用户确认后生效，可随时修改或废弃 | L0-L3 四级权限梯度 |
| — | **架构机制** | **调度器/DET** | 设计者定义 | 元指令集、降级链、仲裁权、节点转换守卫——约束调度器和DET行为，非模型约束 |

### 7.2 根宪法（4 条，约束所有环节模型）

**定位**：所有环节模型（ANALYZING + IN_SESSION）都必须遵守的底线条款。由设计者写死，不可运行时修改，不可被环节宪法覆盖。

| 编号 | 条款 | 内容 | 可被环节宪法覆盖 |
|:--:|------|------|:---:|
| 1 | **环节隔离** | 只能做本环节任务，非本环节任务返回 off-task（偏离）。本环节任务由环节宪法第一条定义。ANALYZING（分析态）只能做意图识别。IN_SESSION（会话中）只能做本环节任务。 | ❌ 不接受覆盖 |
| 2 | **输出格式** | 模型实例返回调度器的结果必须是符合 turnType schema 的结构化 JSON（含 askingField 双模式语义 + changeLevel 预判字段 + changeLevelReason 理由字段，两种 taskType 模式均适用） | ❌ 不接受覆盖 |
| 3 | **收敛义务 + 坚持真实判断** | 所有 LLM 节点必须在当前环节内收敛，输出结构化 JSON。不伪造数据、不假装理解、不迎合用户、不无中生有。确认无误就坚持，确实错了就纠正。 | ❌ 不接受覆盖 |
| 4 | **不编造功能** | 所有 LLM 节点不得生成用户未交代、系统不具备的功能或信息。这是安全红线的真核心——不是场景特定的"禁止恶意内容"，而是全局硬约束"不许无中生有"。 | ❌ 不接受覆盖 |

**ANALYZING（分析态）的环节隔离特化**：ANALYZING 用强制选择+API logprobs（非 LLM 自由生成 confidence）。如果 logprobs probability 不足以确认任何已知 intent，返回 `intent=other`（低 probability，调度器按 probability 路由）。**ANALYZING 只产出意图识别相关信号（intent / choice / logprobs / inputNature / extracted / topicId），不产出 turnType——turnType 是 IN_SESSION 的输出格式。** `turnType=off-task` 是 IN_SESSION 内模型检测到偏离时的输出，不是 ANALYZING 的输出。ANALYZING 不得采集字段、不得追问、不得给出最终回复。

### 7.3 架构机制（约束调度器/DET，非模型约束）

以下四条属于架构规范，约束调度器和 DET 行为：

| 编号 | 机制 | 约束对象 | 内容 |
|:--:|------|:---:|------|
| M1 | **元指令集（四类口令 + S3 + roomStateIndex）** | 调度器（第0层）+ 内存管理模块 + context-manager | 四类通用口令：**wake / switch / exit / cancel**。口令层 EXACT_MATCH→直切（0 次 LLM）。wake/switch/exit/cancel 触发 S3 物理 KV Cache 释放 + roomStateIndex 注入。cancel = 中断当前推理请求但不归档房间（区别于 exit 的"归档回 IDLE"），用于"用户想重来这一轮"的场景——释放当前轮 KV Cache，房间状态不变，重新等待用户输入。wake=创建新推理会话 + 从 contractStore 恢复 checkpoint + `buildRoomIndex()` 注入 ANALYZING；switch=释放当前房间 KV Cache + 归档 + 创建新会话 + 恢复目标房间 checkpoint + `buildRoomIndex()` 注入；exit=释放 KV Cache + 归档回 IDLE。roomStateIndex 提供全窗口房间摘要（各房间待完成/已完成数），使 ANALYZING 可针对性建议。设计者定义，DET+内存管理+context-manager 执行。完整 S3 行为详见 §9.4 |
| M2 | **降级链四项检查** | 调度器/DET | L1 结构校验（DET）+ DET 值域复验（DET，field_based值域/topic_based格式）+ logprobs 检查（DET）+ 硬编码兜底（DET）。均不可缩减 |
| M3 | **仲裁权 DET 优先** | 调度器 | 调度器 DET 校验结果优先于模型内部校验结果。两者冲突时以调度器 DET 为准 |
| M4 | **节点转换守卫** | 调度器 | 定义环节间转换的触发条件和守卫规则。IN_SESSION 期间用户输入偏离时，调度器 DET 关键词扫描可辅助触发偏离标记，但最终判断由模型给出 |

### 7.4 环节宪法（约束特定环节模型）

1. **第一条强制为首要内容**：环节宪法第一条必须定义"本环节任务是什么"并强调"非本环节任务返回 off-task"。
2. **追问模板下放到 @section ask-rules**：追问模板（追问类型/主题/格式三维度）由各环节宪法的 @section ask-rules 自行定义。
3. **环节宪法结构模板**（18 标签 @section）：
   ```
   <!-- @constitution <session_name> -->
   <!-- @section taskType -->
   <!-- @section role -->（第一条：本环节任务定义 + off-task 声明）
   <!-- @section convergence -->
   <!-- @section field-rules | topic-boundary -->
   <!-- @section parsing -->
   <!-- @section validation -->
   <!-- @section ask-rules -->（含追问模板）
   <!-- @section completion -->
   <!-- @section modification -->
   <!-- @section off-task-detection -->
   <!-- @section giveup -->
   <!-- @section output-schema -->
   <!-- @section tunable -->
   <!-- @section domain-rule-priority -->
   <!-- @section scale -->（并联实例化配置，可选）
   <!-- @importance -->
   <!-- @outputs -->
   ```

## 八、契约体系

### 8.1 双层契约

Contract-in（输入契约）：为当前节点提供最小充分条件集，屏蔽无关历史。

Contract-out（输出契约）：硬性 Schema（模式）约束，强制模型输出结构化数据，阻断发散。

契约的模态无关性：Contract（契约）定义的是数据格式、值域、可选性和副作用标记，不定义产生该数据的执行体类型或模态。多模态模型和单模态模型可以在同一个状态机中共存，状态机只需验证 contract-out（输出契约）是否符合 schema（模式），无需知道上游的模态。

契约的 taskType 敏感性：Contract-out（输出契约）的 schema（模式）随 taskType 变化——field_based（基于字段）任务的 contract-out（输出契约）包含明确的字段集（如 {category（类别）, amount（金额）, time（时间）}），topic_based（基于主题）任务的 contract-out（输出契约）包含主题边界内的自由文本或结构化创作输出（如 {title, lines}）。两种模式的 contract-out（输出契约）都必须是结构化 JSON，但字段定义方式不同。

topic_based 契约的双载体定义：topic_based 的"契约"不仅包括 contract-out JSON（格式约束），还包括 topicEvolution（事件序列 + 状态快照）。field_based 的契约是字段集（contract-out JSON），topic_based 的契约是事件序列 + 状态快照（topicEvolution）。两者都是跨节点传递的标配载体，不是可选附件。

**turnType（轮次类型）统一格式：**

```json
{
  "turnType": "ask | reply | complete | off-task | giveup | validation_failed",
  "askingField": "null | category | amount | time | quantity | character_motivation | plot_conflict | ...",
  "message": "模型对用户说的话（自然语言）",
  "validationResult": "null | { \"field\": \"amount\", \"issue\": \"amount_too_large\", \"userInput\": \"一百万\" }",
  "collectedFields": {
    "category": "猫粮",
    "amount": null,
    "time": null
  },
  "offTaskInput": "null | 用户偏离输入",
  "result": "null | { \"category\": \"猫粮\", \"amount\": 35, \"time\": \"昨天\" }",
  "changeLevel": "null | major | minor | invalid",
  "changeLevelReason": "null | 变更原因说明"
}
```

turnType="ask（追问）"：模型在追问用户，askingField（追问字段）为当前追问的字段名（field_based）或主题标签（topic_based），供 context-manager（上下文管理器）硬门控保护使用

turnType="reply（回复）"：模型在回复用户（非追问），askingField（追问字段）为 null（空）

turnType="complete（完成）"：所有字段齐全（field_based）或输出满足主题边界（topic_based），result（结果）包含完整结构化结果

turnType="off-task（偏离任务）"：检测到偏离，offTaskInput（偏离任务输入）包含偏离输入

turnType="giveup（放弃）"：用户放弃

turnType="validation_failed（校验失败）"：值域校验失败（field_based）或格式校验失败（topic_based），askingField（追问字段）为校验失败的字段名或主题维度

注：`changeLevel` 和 `changeLevelReason` 描述本轮变更的信息净价值（major/minor/invalid），仅在内容发生变更时输出，null 表示无变更。与 `turnType` 独立判定——turnType=complete 时 changeLevel 可能为 major（结构性变更后完成）或 minor（微调后完成）。

**askingField（追问字段）双模式语义：**

| taskType（任务类型） | askingField（追问字段）取值 | 示例 | context-manager（上下文管理器）锚点 | 执行方式 |
|---------------------|-------------------------|------|----------------------------------|---------|
| field_based（基于字段） | 预定义字段名，取值来自环节宪法 @section field-rules | "amount"、"time"、"category" | 字段级硬门控——保护与 askingField 同名的历史轮次不被淘汰 | DET 确定性字符串匹配 |
| topic_based（基于主题） | 模型动态生成的主题锚点标签，格式为 snake_case 英文标签 | "character_motivation"、"plot_conflict"、"hypothesis_A" | 主题级硬门控——保护与 askingField 语义相关的历史轮次不被淘汰 | embedding 两阶段检索 |

askingField（追问字段）主次标注规则：强收敛任务中，如果几个字段是强关联的（如"金额+币种"），可以一轮问两个，但要在 askingField 里标清楚主次，格式为 "amount+currency"（主字段在前，次字段在后）。topic_based（基于主题）任务中，askingField 通常为单个主题标签，但也可以标注主次（如 "character_motivation+backstory"）。

askingField（追问字段）null（空）降级规则（两种模式通用）：当 askingField（追问字段）为 null（空）时（如 reply（回复）轮次、complete（完成）轮次），硬门控保护降级为纯语义匹配。field_based（基于字段）模式不执行字段名匹配保护，topic_based（基于主题）模式不执行主题标签匹配保护，均改为 context-manager（上下文管理器）按综合匹配度评分分配形态。降级不报错不阻断——reply（回复）/complete（完成）轮次本来就不需要硬门控保护。

### 8.2 跨节点契约继承（可选能力）

在物理隔离上下文 Γ 的同时，允许用户将旧节点的 contract-out（输出契约）选择性传递给新节点的 contract-in（输入契约）：

默认行为：不传递（保持纯净初始态）

启用条件：用户主动声明"保留 XX 信息"，或场景需要跨环节传递上下文（如"记完账后顺便查一下余额"）

传递内容：旧节点 contract-out（输出契约）中对应字段，非全部上下文

注入上限：不超过新节点 contract-in（输入契约）总字段数的 30%。此为工程经验值——超过此比例时下游节点的认知负担开始显著增加、幻觉率上升。具体阈值可根据节点复杂度和模型上下文窗口调整，30% 为默认建议上限，可通过 tunable（可调参数）槽位配置（见 §10 contract_inheritance_limit）

注入前校验：需经过根宪法校验

taskType 跨环节传递：如果旧节点是 field_based（基于字段）、新节点是 topic_based（基于主题）（或反之），传递的字段需做语义映射——field_based（基于字段）的结构化字段注入 topic_based（基于主题）时，作为主题空间的约束条件注入；topic_based（基于主题）的创作输出注入 field_based（基于字段）时，提取可结构化的字段注入。跨 taskType（任务类型）传递需在环节宪法中声明映射规则。

### 8.3 契约演化版本管理

| 维度 | 规范 |
|------|------|
| 版本号格式（契约 schema） | semver（语义版本）（major.minor.patch）——用于契约 schema 版本 |
| 向后兼容定义 | 新增可选字段=patch（补丁版本）；新增必填字段=minor（次版本）；删除/重命名字段=major（主版本） |
| 转换时机 | 新版本部署后，存量会话在进入下一个节点时执行转换 |
| 转换失败处理 | 降级至 CLARIFYING（澄清态），通知用户 |
| EXECUTING（执行态）节点例外 | EXECUTING 节点的契约版本转换在该节点操作完成后执行，不中断当前执行。若操作完成后转换失败，降级至 CLARIFYING |
| taskType 变更 | taskType 变更视为 major（主版本）版本变更——因为 taskType 变更导致 askingField（追问字段）语义、context-manager（上下文管理器）锚点模式、DET 复验方式全部变化，不向后兼容 |

## 九、元指令与动态节点切换

### 9.1 元指令识别机制

元指令集采用"第0层全局硬匹配"定位，在 LISTENING（监听态）→ ANALYZING（分析态）之间插入 DET 预处理步骤。机制分两层：

**口令层（DET 预处理 + 精确匹配）：** 对用户原始文本执行口令精确匹配（基于架构机制 M1 的口令词表）。匹配方式：

- **精确匹配（EXACT_MATCH）：用户输入与口令词表项完全一致**（如用户输入 "@@exit" 匹配口令 "@@exit"）→ 直接触发节点切换，0 次 LLM。此为口令设计的目标——用户主动说出系统约定令牌 = 明确切换意图
- **非精确匹配 → 不切：** 任何非精确匹配的输入（含"退出""算了""不玩了"等自然语言）**一律不做口令匹配**，放行至 ANALYZING

**语义确认层（ANALYZING 顺带判断 + 二次确认）：** 自然语言中可能表达切换意图的输入（如"算了不聊柜子了，帮我写首诗""我不想继续了"），由 ANALYZING 的 LLM 在意图识别时顺带判断。若 LLM 判定用户想退出/切换：

- **不直接切**，返回 intent="meta_suspect"，message 含二次确认语（如"你是想退出当前的任务吗？退出后之前的进度不会保存。"）
- 用户明确回复"是"/"对"/"确认" → 触发节点切换
- 用户回复其他内容 → 继续当前任务

**口令精确匹配的纪律：** 口令是用户主动发出的切换信号，不需要 undo（用户说口令 = 用户主动想切，不存在误切）。自然语言二次确认路径也不需 undo（用户确认后才切）。快照机制仅适用于系统崩溃恢复，不适用于元指令路径。

### 9.2 元指令触发行为

元指令识别后（口令精确匹配 or 语义确认层用户确认切），系统不将其作为任务内容处理，直接触发节点切换。元指令集以口令词表+语义模板形式定义在架构机制（M1）中，不接受动态覆盖。

### 9.3 元指令切换时的 taskType 变化

元指令触发的节点切换可能伴随 taskType（任务类型）变化（如从 field_based（基于字段）的记账切换到 topic_based（基于主题）的写诗）。切换时物理隔离当前上下文 Γ，新节点按自己的 taskType 重新初始化 askingField（追问字段）语义和 context-manager（上下文管理器）锚点模式。

### 9.4 房间切换与 S3 物理隔离

态控架构的"房间"概念不仅是逻辑分组——M1 的四个核心元指令（wake / switch / exit / cancel）通过 S3 机制在物理层实现房间间的上下文隔离。这是态控架构区别于所有传统 Agent 框架的核心能力：**单一窗口内，多个房间之间通过 KV Cache 物理释放实现零泄漏切换**。

**四个元指令的 S3 行为**：

| 元指令 | 语义 | KV Cache 行为 | contractStore 行为 | 新推理会话 | 房间状态 |
|------|------|:---:|------|:---:|------|
| **wake**（唤醒，如"大猩猩慢慢醒"） | 从 IDLE 唤醒系统 | 从无到有，创建新会话 | 查询 contractStore → 恢复上次该房间的 sessionCheckpoint（如有）+ roomConversationLog | ✅ 创建 | 房间恢复现场 |
| **switch**（切断房间） | 切换到另一个房间 | **释放当前房间 KV Cache**（`abort_request`）→ 创建新会话 | 归档当前房间（sessionCheckpoint + roomConversationLog 当前轮）→ 从 contractStore 读取目标房间 checkpoint | ✅ 创建（新房间独立会话） | 当前房间归档，目标房间恢复 |
| **exit**（退出，如"大猩猩，飞走吧"） | 退出整个系统 | 释放当前 KV Cache | 归档当前房间 | ❌ 不创建，回 IDLE | 房间归档，系统回 IDLE |
| **cancel**（取消当前轮，如"重来"） | 取消当前这一轮推理，重新等待输入 | **释放当前轮 KV Cache**（`abort_request` 当前轮），房间状态不变 | 不归档（房间未结束），不写 checkpoint | ✅ 创建新轮次会话（同房间） | 房间状态保持，当前轮作废重来 |

**cancel 与 exit 的区别**：
- **cancel**：用户想"重来这一轮"——房间不结束，只作废当前轮的推理，重新等待用户输入。适用于"刚才说错了，重新来"的场景
- **exit**：用户想"结束整个房间"——房间归档，系统回 IDLE。适用于"这个任务做完了/不想做了"的场景

**实现路径**：

- **自建推理引擎（vLLM/SGLang）**：调度器调用 `vLLM.abort_request(roomA_session_id)` 实现完整 S3 物理释放（调度器的资源协调职责，调内存管理模块）
- **云 API（DeepSeek/OpenAI）**：不传旧 messages[]，功能等价——LLM 物理上看不到旧房间上下文
- 开销：`abort_request` + 创建新会话 ≈ 几十毫秒，远小于一次 LLM 调用的延迟

**S3 物理隔离四步流程（以 switch 为例）**：

1. **释放当前房间 KV Cache**：内存管理模块向底层推理引擎发送显式内存回收指令（`abort_request(roomId)`），当前房间的 KV Cache 内存块引用计数清零，交还给 GPU 显存分配器
2. **归档当前房间**：contractStore 记录当前房间的 sessionCheckpoint + roomConversationLog（roomConversationLog 按自己的保留期独立管理，不因 switch 而清理）
3. **创建新推理会话**：新房间的 LLM 推理会话独立创建，从 contractStore 读取对应房间的 checkpoint（如存在），不继承任何旧房间的 KV 缓存
4. **窗口延续**：前端不刷新、不跳转——用户在同一个界面，只是背后推理引擎换了一个会话

## 十、受控调参系统（tunable 槽位）

宪法中可声明运行时允许调整的参数，含 type（类型）/ range（范围）/ enum（枚举）/ default（默认值）/ conflict_check（冲突检测）。由系统设置界面自动生成配置表单，冲突检测引擎在保存前校验。参数调整后下一轮新会话生效，变更日志不可篡改。tunable（可调参数）的灵活性是"受控的灵活性"——适应性受限，但可控、可审计、可测试。

与纯 LLM 运行时动态调整的区别：

| 维度 | 纯 LLM 动态调整 | 态控 tunable 槽位 |
|------|--------------|-----------------|
| 调整方式 | 在 prompt 里写条件 | 在系统设置里改参数 |
| 调整范围 | 无限制 | range/enum 硬限制 |
| 冲突检测 | 无 | 确定性代码检测 |
| 审计 | 难 | 变更日志，每次调整可追溯 |
| 生效时机 | 下一轮对话立即生效 | 保存后下一轮新会话生效 |

**关键 tunable（可调参数）参数清单：**

| 参数名 | 默认值 | 取值范围 | 用途 | 消费方 | 适用 taskType |
|--------|--------|---------|------|--------|-------------|

| pid_kp（P 控制比例系数） | 0.1 | [0.01, 1.0] | P 反馈控制比例系数（参数调整 = Kp × (目标值 - 实际值)，目标值由 target_*_rate 系列参数提供） | 自动调参引擎 | 通用 |

冷启动冻结机制：
冷启动期间参数冻结，但实时观测指标：监测 logprobs 拦截率、topicEvolution event 写入频率、minor 压缩频率、规则提炼触发频率等

应急解冻机制：若指标异常（如 logprobs 拦截率 > cold_start_emergency_threshold），触发"应急解冻"——人工确认后调整参数

解冻后 P 反馈控制自动调节：参数调整 = pid_kp × (目标值 - 实际值)，目标值由 target_logprobs_intercept_rate（默认 0.05）等目标值参数提供。例如 logprobs 拦截率目标 5%，实际 15%，调整 = 0.1 × (0.05 - 0.15) = -0.01，logprobs_threshold 下调 0.01

> 注：当前版本仅启用比例项（Kp），积分项（Ki）和微分项（Kd）预留未启用。如需消除稳态误差可引入 Ki，如需抑制超调可引入 Kd，但当前 P 控制已满足参数微调需求。

所有调整记录变更日志：时间戳、操作者、参数名、旧值、新值、调整原因、触发指标

### 10.1 跨参数联动约束规则

定位：扩展附录 C.2 冲突检测引擎，处理 tunable 参数之间的隐式语义依赖。与 single_param range/enum 校验不同——cross_param 不替用户决策，只做语义校验 + 建议 + 警告。

**规则格式规范：**

| 字段 | 必填 | 说明 |
|------|------|------|
| `type` | 是 | 固定值 `"cross_param"` |
| `name` | 是 | 规则标识符，snake_case |
| `violation_action` | 否 | 默认 `"warning"`——cross_param 违规不阻止保存。可选 `"hard_reject"`（用于确定性越界） |
| `params` | 是 | 被此规则引用的 tunable 参数名数组——仅这些参数变更时重新执行此规则 |
| `constants` | 否 | 计算常量（如模型窗口大小、每轮平均 token 数），声明在此以供引用 |
| `compute` | 是 | 沙箱表达式，仅允许 `+ - * / floor ceil min max` 运算符 + params 变量 + constants 常量。求值结果 true=违规触发 |
| `event` | 是 | 违规事件结构：`{type, severity, message, suggested_adjustment}`。`suggested_adjustment` 可含带 formula 的建议值计算提示 |

**初始规则清单（5 条，唯一来源——附录 C.2 不再重复）：**

- 两条参数之间如果存在确定性大小关系（如 topic_anchor_similarity_threshold < topic_evolution_match_threshold），应使用 `hard_reject` 而非 `warning`——这是硬约束，不是警告
- 跨参数约束只覆盖"参数组合可能导致系统行为异常"的情况，不覆盖"这个参数值可能不适合你的业务场景"——后者属于场景设计阶段的人为判断，不是冲突检测引擎的职责

### 附录 C：工程实施规范

#### C.1 执行体调用链追踪（OpenTelemetry 集成）

问题背景：三类执行体（DET/LLM/HUMAN）的调用缺少全链路追踪，运维监控和故障定位困难。

实施规范：
每次 DET/LLM/HUMAN 调用生成 OpenTelemetry span
span 含字段：trace_id / span_id / parent_span_id / executor_type（DET/LLM/HUMAN）/ executor_name / start_time / end_time / status（success/fail/timeout）/ input_summary / output_summary / error_detail
用 OpenTelemetry 标准格式上报（Jaeger/Zipkin 兼容）
运维侧用 Jaeger/Zipkin 可视化调用链
监控指标（校验失败率/偏离率/logprobs 拦截率/embedding 调用次数/topicEvolution event 写入频率/冷启动应急解冻触发次数）直接从 trace 提取
每个会话一个 trace，每个节点一个 span，每个执行体调用一个子 span

集成位置：
第0层全局硬匹配：DET span
第0.5层关键词扫描：DET span
第1层意图识别：LLM span
第2层环节内自治：LLM span（环节内多轮对话一个 span，含 turn_count 子字段）
第3层 DET 复验：DET span（含四项检查各自的子 span）
EXECUTING：DET span
embedding 调用（topic_based 模式阶段2）：embedding span（含 askingField / similarity_score / protected_count 子字段）
topicEvolution event 写入：DET span（含 topicId / eventId / nodeId / stateChange 子字段）
上下文管理器三层（匹配/决策/拼接）：各一个 DET span（含 matched_rule_count / injected_rule_count / token_used 子字段）

收益：运维侧可实时观测执行体调用链，快速定位故障节点；监控指标从 trace 自动提取，无需额外埋点。

#### C.2 宪法冲突检测引擎

实施规范：
- `single_param` 规则用 json-rules-engine（Node.js）或 Drools（Java）实现确定性 range/enum 校验
- `cross_param` 规则由冲突检测引擎在遍历 single_param 完成后执行 compute 表达式求值——用沙箱 `eval()`（仅允许 `+ - * / floor ceil min max` 运算符 + params 变量 + constants 常量）
- 参数变更时通过 `params` 字段做依赖追踪——仅被引用参数的变更触发对应 `cross_param` 规则重新执行
- `violation_action` 默认值：`single_param` 规则默认为 `hard_reject`；`cross_param` 规则默认为 `warning`
- `violation_action=hard_reject` 行为：拒绝保存 + 返回错误详情
- `violation_action=warning` 行为：弹窗展示 warning message + suggested_adjustment，用户可选择"仍然保存"或"调整后重试"。warning 不阻止保存
- compute 表达式求值失败（语法错误、引用不存在参数）→ 视为 `hard_reject`，阻止保存并返回明确的表达式错误位置
- conflict_check 规则用 JSON 格式声明，与 tunable 参数一同存在 tunables.json
- 保存前执行全部规则：`single_param` 任一 hard_reject → 拒绝保存；`cross_param` warning → 弹窗但允许继续
- 冲突检测引擎是确定性代码，不调 LLM

**cross_param 规则清单的唯一来源是 §10.1**——本附录不再重复规则 JSON，只定义实现规范。`single_param` 规则示例：

```json
{
  "conflict_rules": [
    {
      "type": "single_param",
      "name": "logprobs_threshold_range_check",
      "condition": {
        "all": [
          { "fact": "parameter", "path": "$.logprobs_threshold", "operator": "lessThan", "value": 0 },
          { "fact": "parameter", "path": "$.logprobs_threshold", "operator": "greaterThan", "value": 1 }
        ]
      },
      "event": { "type": "conflict", "message": "logprobs_threshold 必须在 [0, 1] 范围内" }
    }
  ]
}
```

#### C.3 contractStore 物理存储（SQLite + WAL 模式）

问题背景：contractStore 跨会话持久化，需要支持并发和崩溃恢复。

实施规范：
- SQLite 单文件数据库，每个用户/项目一个 .db 文件
- WAL（Write-Ahead Logging）模式支持并发读 + 单写
- 数据库文件路径：`<project_root>/.contractstore/<user_id>.db`

**contractStore 表结构（SQL DDL 唯一来源——正文各处只描述用途，DDL 统一在此定义）：**

```sql
-- ANALYZING 节点契约副本（支持意图级纠错逃生舱）
CREATE TABLE IF NOT EXISTS analyzing_contract_in (
    id TEXT PRIMARY KEY,
    userId TEXT NOT NULL,
    rawInput TEXT NOT NULL,
    createdAt TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS analyzing_contract_out (
    id TEXT PRIMARY KEY,
    userId TEXT NOT NULL,
    choice TEXT, logprobs TEXT, intent TEXT,
    inputNature TEXT, inputNatureLogprobs TEXT,
    extracted TEXT, topicId TEXT,
    createdAt TEXT NOT NULL
);

-- topic_based 主题演化
CREATE TABLE IF NOT EXISTS topicEvolution (
    topicId TEXT PRIMARY KEY,
    userId TEXT NOT NULL,
    topicName TEXT NOT NULL,
    stateSnapshot TEXT NOT NULL,  -- active/checkpoint/abandoned
    createdAt TEXT NOT NULL,
    updatedAt TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS topicEvolutionEvent (
    eventId TEXT PRIMARY KEY,
    topicId TEXT NOT NULL,
    nodeId TEXT NOT NULL,
    turnType TEXT NOT NULL,
    timestamp TEXT NOT NULL,
    stateChange TEXT,  -- JSON: {before, after, delta}
    askingField TEXT,
    modelOutputSummary TEXT,
    FOREIGN KEY (topicId) REFERENCES topicEvolution(topicId)
);

CREATE TABLE IF NOT EXISTS topicEvolutionArchive (
    archiveId TEXT PRIMARY KEY,
    topicId TEXT NOT NULL,
    eventId TEXT NOT NULL,
    archiveType TEXT NOT NULL,  -- minor_compacted / invalid
    archivedAt TEXT NOT NULL
);

-- 领域规则网络图
CREATE TABLE IF NOT EXISTS domainRules (
    ruleId TEXT PRIMARY KEY,
    content TEXT NOT NULL,
    topicPath TEXT NOT NULL,     -- JSON 数组
    stepName TEXT,               -- field_based 专属
    source TEXT NOT NULL,        -- scenario_seed / user_initiated / model_inferred
    immutableLevel TEXT NOT NULL,-- L0/L1/L2/L3
    importance TEXT DEFAULT 'normal',
    status TEXT DEFAULT 'active',
    conditions TEXT,             -- JSON 对象
    edges TEXT,                  -- JSON: {depends_on:[], conflicts_with:[], strengthens:[]}
    version INTEGER DEFAULT 1,
    createdAt TEXT NOT NULL,
    updatedAt TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS ruleCandidates (
    candidateId TEXT PRIMARY KEY,
    rule TEXT NOT NULL,
    topicPath TEXT,
    stepName TEXT,
    evidenceCount INTEGER DEFAULT 0,
    passCount INTEGER DEFAULT 0,
    validationRate REAL DEFAULT 0,
    status TEXT DEFAULT 'pending', -- pending/confirmed/rejected
    createdAt TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS ruleMiningQueue (
    queueId TEXT PRIMARY KEY,
    topicId TEXT,
    stepName TEXT,
    intent TEXT,
    checkpointAt TEXT NOT NULL,
    status TEXT DEFAULT 'pending'  -- pending/processing/done
);

-- field_based 进度检查点（ttl 到期只清此表，不联动清理 roomConversationLog）
CREATE TABLE IF NOT EXISTS sessionCheckpoints (
    userId TEXT PRIMARY KEY,
    lastCompletedStep TEXT NOT NULL,
    completedSteps TEXT NOT NULL,    -- JSON: ["p0","n1","n3",...]
    stepSnapshots TEXT NOT NULL,     -- JSON: {"n1":{"collectedFields":{...},"timestamp":"..."}}
    resumedAt TEXT,
    expiredAt TEXT,
    ttl INTEGER DEFAULT 604800      -- 秒，默认 7 天
);

-- 房间对话日志（保留期独立于 sessionCheckpoint.ttl，按 importance 分级）
CREATE TABLE IF NOT EXISTS roomConversationLog (
    logId TEXT PRIMARY KEY,
    userId TEXT NOT NULL,
    roomId TEXT NOT NULL,
    roomName TEXT,
    importance TEXT DEFAULT 'normal', -- critical/high/normal/low，决定保留期
    turnNumber INTEGER NOT NULL,
    userMessage TEXT,
    modelOutput TEXT,
    turnType TEXT,
    askingField TEXT,
    createdAt TEXT NOT NULL,
    expiresAt TEXT,                  -- 按 importance 分级计算：critical=NULL(永久)/high=+90d/normal=+30d/low=+7d
    archived INTEGER DEFAULT 0       -- 0=活跃, 1=已归档（不再注入上下文但保留）
);

-- 全窗口房间状态索引（物化视图）
CREATE TABLE IF NOT EXISTS roomStateIndex (
    userId TEXT NOT NULL,
    roomId TEXT NOT NULL,
    roomName TEXT NOT NULL,
    intent TEXT,
    taskType TEXT,
    lastActiveAt TEXT NOT NULL,
    pendingCount INTEGER DEFAULT 0,
    completedCount INTEGER DEFAULT 0,
    lastSummary TEXT,
    PRIMARY KEY (userId, roomId)
);

-- 产出物总索引
CREATE TABLE IF NOT EXISTS outputRegistry (
    outputId TEXT PRIMARY KEY,
    roomId TEXT NOT NULL,
    roomName TEXT NOT NULL,
    intent TEXT NOT NULL,
    taskType TEXT NOT NULL,
    outputType TEXT NOT NULL,       -- contract_json / document / report / code / patent_draft / ...
    outputPath TEXT NOT NULL,       -- 文件系统路径
    outputName TEXT NOT NULL,       -- 人类可读名称
    outputSummary TEXT,             -- 模型生成的摘要，供检索匹配
    keywords TEXT,                  -- JSON数组，关键标签
    fileSize INTEGER,
    createdAt TEXT NOT NULL,
    createdBy TEXT NOT NULL,
    status TEXT DEFAULT 'active',
    metadata TEXT                   -- JSON，扩展元数据
);
```

**索引**：
- `roomConversationLog`: `(userId, roomId, createdAt)`、`(userId, expiresAt)` 用于过期清理
- `roomStateIndex`: `userId` 索引
- `outputRegistry`: `(roomId, createdAt)`、`(intent, outputType)`、FTS on `(outputName, outputSummary, keywords)`
- `domainRules`: `(topicPath)`、`(stepName)`、`(status, immutableLevel)`

**并发控制**：
- WAL 模式支持多读单写
- 写操作用事务包裹，崩溃时自动回滚
- 多用户并发时，每个用户独立 .db 文件，无跨用户锁竞争

**roomConversationLog 过期清理任务**：
- 定时任务（cron，默认每日凌晨 3 点）扫描 `expiresAt < now() AND archived = 0`
- critical 房间（expiresAt IS NULL）永不清理
- low 房间过期后直接删除（低价值，不归档）
- normal/high 房间过期后标记 `archived = 1`（归档，不再注入上下文但保留记录）

### 附录 D：align-session 可选扩展规范

align-session 是可选扩展，非态控架构标准能力。若场景含意图校准需求，可启用 align-session 扩展（在 §3.1 意图收敛机制的收敛链路中插入第 0.1 层）。启用时需：
- 额外编写 align-session.json 宪法
- 在路由表中增加 align 相关条目
- 在 tunable 参数中增加 divergence_threshold
- 同步更新场景的边界清单
- 在 §3.1 收敛链路的第0层之后插入第0.1层（align 静默模式触发与处理）

align-session 核心机制（仅在启用时生效）：
静默输入：/align 元指令触发后，挂起语义管道，所有输入走静默缓冲区，直到完成指令或退出词
多模型并行仲裁：主次模型并行 + 规则模型串行
锚点确认：主/次模型标注 core_anchor，用户确认后沉淀 user_confirmed_anchor
迭代裁决：divergence_threshold 判定分歧度
对齐数据包沉淀：rounds[] / divergence_points / user_arbitration

align 相关 tunable 参数（仅在启用时声明）：
divergence_threshold：默认 0.7，校准模式分歧度阈值

align-session 宪法 complete 条件：用户最终确认。

align 相关路由条目（仅在启用时配置）：
ANALYZING（intent=align）→ IN_SESSION(align)
IN_SESSION(align) complete → EXECUTING（DET）
IN_SESSION(align) giveup → LISTENING
IN_SESSION(align) off-task → ANALYZING（轻量）

与核心架构的边界：align-session 启用时不修改根宪法第 1-4 条及架构机制 M1-M4，仅在路由表和 tunable 参数中扩展，并在收敛链路插入第0.1层。align-session 不启用时，核心架构完全不受影响。

### 附录 E：编程规则宪法

**定位**：约束代码模型的元规则，全局生效于整个 N1→N15 管线。任何涉及代码生成的环节（N13）必须在生成代码前校验此宪法，违反任一条款拒绝生成。

**约束对象**：代码模型（生成骨架代码的 AI），非运行时模型。

**生效范围**：母本的一部分，设计者在 P0 认知加载时即需理解，N13 代码模型强制输入。

| 编号 | 条款 | 内容 |
|:--:|------|------|
| 1 | **执行体分派铁律** | 每个状态的执行体由 v5.0/N3 决定，代码模型不得自行更改。ANALYZING=LLM，IN_SESSION=LLM，DET 复验=DET，EXECUTING=DET。不得将 ANALYZING/IN_SESSION 降级为正则或 parser。**v5.4 修正：IN_SESSION 可调用 DET 工具（function calling）；EXECUTING 仅保留给纯写入/删除操作（record INSERT / delete DELETE），查询/对比类操作在 IN_SESSION 内通过工具调用完成** |
| 2 | **DET 职责边界** | DET 只做**确定性计算**：元指令 EXACT_MATCH（精确匹配）+ 值域/格式复验 + 关键词扫描标记 + logprobs 阈值裁决 + 路由矩阵裁决（intent × inputNature）+ 规则子图匹配与冲突检测 + 文件操作（产出物复制）+ **SQL 查询（作为 IN_SESSION 的工具被调用）**。核心是"确定性计算"这个定语——凡是能用确定性代码完成的计算都归 DET，不扩展到意图识别、字段采集、查询理解、语义判断等需要概率推理的任务 |
| 3 | **LLM 职责边界** | LLM 做：意图识别 + 字段采集/追问/偏离判断/查询理解 + 语义边界判断 + **工具调用决策 + 结果格式化（自然语言生成）**。不得用正则/parser 替代 |
| 4 | **降级规则** | 无 apiKey 时 ANALYZING/IN_SESSION 必须**拒绝服务并明确告知原因**，不得以任何形式降级为正则/DET 替代。降级模式如存在必须标注"降级非正道" |
| 5 | **契约对齐** | 代码必须严格对齐 N11 契约（contract-in/out schema），不自创接口 |
| 6 | **tunable（可调参数）/接口隔离** | 阈值从 getTunable() 读，调度器只调 context-manager（上下文管理器）三接口（匹配/决策/拼接），不内联拼接逻辑。上下文管理器三层职责（匹配→决策→拼接）各自独立，不得合并 |
| **7** | **工具分层发现（v5.5 新增）** | **工具分三层：必用工具（环节宪法 @section tools 声明，完整定义自动注入）/ 工具清单（@section tool_catalog，轻量级目录每次注入）/ 选用工具（系统级工具库 toolRegistry，完整定义按需加载）。模型从清单选择→调度器加载完整定义→模型调用。LLM 不直接写 SQL，通过 DET 工具间接操作数据库。search_tools meta-tool 始终可用，用于清单外的工具发现。工具内部支持时间解析（"this_month"→日期范围）** |

### 风险条目：changeLevel 防偷懒

**风险描述**：模型可能偷懒，将所有变更标记为 major（保证不被压缩）或跳过 changeLevel 判定。更隐蔽的偷懒形态是"认真编一个 changeLevelReason 理由但仍然偷懒"——DET 格式校验能拦空值/超长，但挡不住"看起来合理实际仍是偷懒"的理由。

**缓解措施**：
- 根宪法第4条已要求 changeLevelReason 理由字段（见 §7.2）
- DET 格式校验拦截 changeLevelReason 空值/超长
- 监控指标：changeLevel major 占比阈值从 <1% 调整为 <3%，changeLevelReason 空值率检测
- 若 major 占比超阈值或空值率异常，触发冷启动应急解冻机制（见 §10）审查参数
- **抽样人工复核机制**：按比例抽检 major 标记的合理性，作为 DET 格式校验之上的第二道防线
  - 默认抽检比例：5%（通过 tunable 参数 `changelevel_major_sample_rate` 配置，取值范围 [0.01, 0.20]，默认 0.05）
  - 抽检对象：所有被模型标记为 major 的 changeLevel 事件
  - 复核流程：抽样事件推入人工复核队列 → 复核员判断"该变更是否真的属于结构性变更（新增/删除必采字段、修改字段值域、改变 Schema 结构）" → 标记为 confirmed_major / false_positive
  - 反馈闭环：false_positive 比例 > 30% 时，触发冷启动应急解冻机制（见 §10）审查 logprobs_threshold 与 changeLevel 判定 prompt
  - 与冷启动的关系：冷启动期间（cold_start_observation_window 内）抽检比例临时上调至 10%，冷启动结束后恢复默认 5%——冷启动期模型行为未稳定，需要更高密度的反馈信号
  - 复核结果记入变更日志：时间戳、抽检事件 ID、复核员、结论、复核理由
