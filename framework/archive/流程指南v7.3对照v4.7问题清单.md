## 问题 1（P0）：母本文件名/标题与内容不符

### 现象

工作区 `态控架构/` 目录下有一份文件：
- **文件名**：`态控架构-v4.6-全量版.md`
- **文档标题**（line 1）：`# 态控架构 v4.6（全量版）`
- **版本号字段**（line 7）：`**文档版本**：v4.7`
- **实际内容**：完整的 v4.7（含 v4.7 changelog、cross_param §10.1、C.2 双模式、max_critical_rooms、@importance 移至 §3.10）

### 问题

文件名和标题声称是 v4.6，但内容是 v4.7。这会导致：
- 任何人看到文件名/标题以为是 v4.6，但实际读到的是 v4.7
- 流程指南 v7.3 引用"基于母本 v4.7"，但工作区找不到文件名为 v4.7 的文件——初次对照时会误判为"v4.7 不存在"（本次对照就发生了这个误判）
- 版本演进历史混乱——v4.6 的 changelog 和 v4.7 的 changelog 都在同一份文件里，但文件名只反映 v4.6

### 建议修复

1. 重命名文件：`态控架构-v4.6-全量版.md` → `态控架构-v4.7-全量版.md`
2. 更新标题：`# 态控架构 v4.6（全量版）` → `# 态控架构 v4.7（全量版）`
3. 保留 v4.5 全量版作为历史快照（如果还需要单独的 v4.6 快照，应在 v4.7 升级前先 cp 一份 v4.6 出来）

## 问题 4（P1）：§5.5 检查4 缺失 v4.6 精修 4 的降级路径描述

### 现象

v4.6 精修 4（topic_based 跨任务延伸检测 LLM 不可用时降级路径）在流程指南中**有两处描述，详略不一致**：

**§2.8 taskType 对降级链的影响（line 730）—— 有降级路径**：
> `v4.7 降级路径：topic_based 模式下当辅助 LLM 不可用/超时/置信度低于 confidence_threshold 时，强制退化为 DET 关键词白名单 + 用户主动确认，不允许直接放行。`

**§5.5 第二道防线五项检查的检查4（line 1919-1925）—— 缺失降级路径**：
> 检查4 只描述了"默认层 DET + 例外层 LLM"两层结构，失败处理只说"进入 CLARIFYING"，**没有描述 LLM 不可用/超时/低置信度时的降级路径**。

### 问题

§2.8 和 §5.5 对同一机制（topic_based 跨任务延伸检测）的描述详略不一致。§5.5 是"第二道防线五项检查"的具体执行步骤，是落地实现者最直接参考的章节——这里缺失降级路径，会导致实现者只看到"两层结构 + 失败进 CLARIFYING"，不知道 LLM 不可用时该怎么办。

### 建议修复

在 §5.5 检查4 的 topic_based 分支中，补充降级路径描述（与 §2.8 对齐）：
> `降级路径：当辅助 LLM 不可用 / 超时 / 置信度低于 confidence_threshold 时，强制退化为 DET 关键词白名单 + 用户主动确认，不允许直接放行。`
