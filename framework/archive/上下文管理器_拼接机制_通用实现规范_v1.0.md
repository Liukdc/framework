# 上下文管理器（context-manager）拼接机制 · 通用实现规范

**版本**：v1.0
**日期**：2026-07-14
**基于**：态控架构 v4.7 §2.4 / §3.9 步骤三 / §3.10 @importance
**适用场景**：field_based（基于字段）和 topic_based（基于主题）通用

---

## 一、定位

上下文管理器（context-manager）是态控架构中的独立状态机，不是环节宪法的一部分。

**它只管一件事**：从态控调度器交给它的内容中，按拼接规则组装出本轮 LLM（大语言模型）模型实例计算所需的上下文窗口。

态控调度器负责路由和全量记录。上下文管理器负责拼接。环节宪法负责模型行为规则。三者各司其职。

---

## 二、输入

态控调度器在每轮调用模型实例前，将以下内容交给上下文管理器：

| 输入 | 来源 | 说明 |
|------|------|------|
| taskType（任务类型） | activeSession.taskType（活跃会话.任务类型） | field_based（基于字段）或 topic_based（基于主题） |
| roomConversationLog[] | contractStore（契约存储）按 userId（用户标识）+roomId（房间标识）查询 | 当前房间的全量历史对话原文 |
| contractOut | sessionCheckpoint（会话检查点）或 topicEvolution（主题演化） | 已确认的结构化字段/状态快照 |
| domainRules[] | contractStore（契约存储）按 stepName（环节名）或 topicPath（主题路径）匹配 | 已生效领域规则（经 importance（重要性标记）排序） |
| relatedTopics[] | topicIndex（主题索引）（仅 topic_based（基于主题）） | 关联主题摘要卡片 |
| topicEvolution 分层包 | topicEvolution（主题演化）表（仅 topic_based（基于主题）） | major（主版本）事件 + minor（次版本）压缩摘要 + stateSnapshot（状态快照） |
| currentAskField | 上一轮 turnType（轮次类型）返回的 askingField（追问字段） | 本轮硬门控保护锚点 |
| tunableParams | tunables.json（可调参数） | turnHistory_limit（轮次历史限制）/ critical_room_history_boost（关键房间历史增强系数）/ 各阈值 |
| offTaskSuspicion | 态控调度器 DET（确定性程序）关键词扫描 | 偏离嫌疑标记 |

## 四、拼接流程（四步）

### 第 0 步：@importance（重要性标记）截断基数计算

根据当前房间的 @importance（重要性标记）等级，计算 roomConversationLog（房间对话日志）保留轮次上限：

| importance（重要性标记） | 保留轮次 |
|:--:|------|
| critical（极其重要） | turnHistory_limit（轮次历史限制） × critical_room_history_boost（关键房间历史增强系数） |
| high（高） | turnHistory_limit（轮次历史限制） × 0.6 |
| normal（普通） | turnHistory_limit（轮次历史限制） × 0.3 |
| low（低） | turnHistory_limit（轮次历史限制） × 0.2 |

取 roomConversationLog（房间对话日志）的最近 N 轮（N = 上述计算值），作为候选集。

**纪律**：这个截断在硬门控保护和综合评分之前执行。超出上限的最久远轮次直接丢弃，不进入任何通道。

### 第 1 步：硬门控保护（DET（确定性程序），0 次 LLM（大语言模型）调用）

对候选集中的每一轮，执行以下检查。命中任意一项 → 标记为保护态 → 从评分通道候选集移除 → 放入保护态队列。

#### 1.1 field_based（基于字段）模式：字段级硬门控

```
规则：若 currentAskField（当前追问字段）≠ null（空），
     遍历候选轮次，若某轮的 askingField（追问字段）=== currentAskField（当前追问字段），
     则该轮被保护。

保护上限：不超过 turnHistory_limit（轮次历史限制）的 50%。

示例：
  currentAskField = "amount"（金额）
  候选轮次 #3: askingField = "category"（品类）→ 不保护，留在评分通道
  候选轮次 #5: askingField = "amount"（金额）→ 保护，移入保护态队列
  候选轮次 #8: askingField = "amount"（金额）→ 保护，移入保护态队列

实现：
  function fieldLevelHardGate(candidates, currentAskField, limit) {
    const protected = candidates.filter(c => c.askingField === currentAskField);
    const maxProtected = Math.floor(limit * 0.5);
    return {
      protected: protected.slice(0, maxProtected),  // 最多一半窗口
      remaining: candidates.filter(c => !protected.slice(0, maxProtected).includes(c)),
    };
  }
```

#### 1.2 topic_based（基于主题）模式：主题级硬门控（两阶段检索）

```
阶段一：DET（确定性程序）关键词粗筛
  对每轮候选的 userInput（用户输入）+ modelOutput（模型输出）提取关键词
  计算与 currentAskField（当前追问字段）（主题标签）的关键词重叠度
  重叠度 ≥ topic_keyword_overlap_threshold（主题关键词重叠阈值）（默认 0.2）→ 进入阶段二

阶段二：embedding（向量嵌入）精筛
  对阶段一通过的候选，计算其与 currentAskField（当前追问字段）的余弦相似度
  相似度 ≥ topic_anchor_similarity_threshold（主题锚点相似度阈值）（默认 0.6）→ 保护

保护上限：同 field_based（基于字段），不超过 turnHistory_limit（轮次历史限制）的 50%。
```

#### 1.3 通用增强：逻辑依赖保护（可选，场内有向关联图时启用）

```
若场内有向关联图 G_rel 已构建：
  对每个候选轮次，查询从当前轮次到该轮次的可达性（≤3 跳）
  可达 = true → 保护（与 1.1/1.2 的保护合并，不重复计数）

此层独立于字段/主题级门控——即使 currentAskField（当前追问字段）= null（空）（reply（回复）/complete（完成）轮次），
逻辑依赖保护仍生效。
```

**纪律**：被保护的内容以完整原文形态注入，不经过评分通道的形态降级处理。无论语义匹配度多低，至少保留为原文形态。

### 第 2 步：综合匹配度评分（对未被保护的剩余候选）

对留在评分通道的候选，依次计算：

```
综合匹配度得分 = 语义匹配度 × 位置衰减系数
```

#### 2.1 语义匹配度（_scoreMatch）

```
function _scoreMatch(candidate, currentInput, taskType) {
  if (taskType === 'field_based（基于字段）') {
    // 中文：bigram 分词后 Jaccard 相似度
    // 数字/英文：精确匹配贡献度
    return jaccardBigram(candidate.text, currentInput);
  }
  if (taskType === 'topic_based（基于主题）') {
    // embedding 余弦相似度 + bigram Jaccard 加权
    return cosineSimilarity(candidate.embedding, currentInput.embedding) * 0.7
           + jaccardBigram(candidate.text, currentInput) * 0.3;
  }
}

// 中文 bigram 分词
function toBigrams(text) {
  const chars = text.replace(/[^\u4e00-\u9fff]/g, '');
  const bigrams = [];
  for (let i = 0; i < chars.length - 1; i++) {
    bigrams.push(chars[i] + chars[i + 1]);
  }
  return bigrams;
}

// Jaccard 相似度
function jaccardBigram(textA, textB) {
  const setA = new Set(toBigrams(textA));
  const setB = new Set(toBigrams(textB));
  const intersection = [...setA].filter(b => setB.has(b));
  const union = new Set([...setA, ...setB]);
  return union.size === 0 ? 0 : intersection.length / union.size;
}
```

#### 2.2 位置衰减系数

```
positionDecay = 1.0 - (轮次距离 / turnHistory_limit（轮次历史限制）)
轮次距离 = |当前轮次序号 - 候选轮次序号|
衰减范围：最近轮次 ≈ 1.0，最远轮次 ≈ 0.0
```

#### 2.3 形态分配

按综合匹配度得分降序排列，分配注入形态：

| 得分区间 | 形态 | token 占比 |
|---------|------|-----------|
| ≥ 0.7 | 完整原文 | 100% |
| 0.4 ~ 0.7 | 摘要（取首句 + 结尾句） | ~30% |
| 0.2 ~ 0.4 | 一句话锚点（提取关键实体） | ~10% |
| < 0.2 | 移除 | 0% |

### 第 3 步：两通道联合编排

```
最终上下文窗口拼接顺序：

[系统提示层]
  ├─ 环节宪法（@section role + @section field-rules/topic-boundary）
  ├─ 已采集字段 collectedFields（仅 field_based（基于字段））
  └─ 领域规则 domainRules（按 importance（重要性标记）排序：critical（极其重要）→ high（高）→ normal（普通）→ low（低））

[保护态队列]（按时间倒序，最近轮次在前）
  ├─ 硬门控保护的轮次（完整原文）
  └─ 逻辑依赖保护的轮次（完整原文）

[评分通道队列]（按综合匹配度得分降序）
  ├─ 得分 ≥ 0.7：完整原文
  ├─ 得分 0.4~0.7：摘要
  ├─ 得分 0.2~0.4：锚点
  └─ 得分 < 0.2：移除

[偏离标记]（如有 offTaskSuspicion（偏离嫌疑））
  └─ "态控调度器提示：本轮可能偏离到 [query/delete/exit]（查询/删除/退出）"

[当前用户输入]
```

### 第 4 步：最低匹配度截断 + token 预算校验

```
截断规则：
  评分通道队列中，从第一个得分 < minimumMatchThreshold（最低匹配度阈值）的候选开始，
  该候选及其后续所有更低分候选全部截断，不注入。
  保护态队列中的内容不受此截断影响。

  最低匹配度阈值默认值：field_based（基于字段）= 0.2，topic_based（基于主题）= 0.3

token 预算校验：
  拼接完成后估算总 token 消耗。若超出模型窗口的 70%，从评分通道队列末尾开始，
  逐条降级（完整→摘要→锚点→移除），每次降级后重新估算，直至在 70% 以内。
  保护态队列在 token 预算校验中不受影响——保护即豁免。
  若仅保护态内容已超出 70%，告警日志但不截断保护态——此时上下文窗口本身就是
  合理的"大上下文"而非膨胀失控。
```

## 五、特别的：currentAskField（当前追问字段）为 null（空）时的降级

当 turnType（轮次类型）="reply（回复）"/"complete（完成）"时，currentAskField（当前追问字段）= null（空）。

此时：
- field_based（基于字段）模式：字段级硬门控不执行（没有追问字段可匹配），降级为纯语义匹配度评分
- topic_based（基于主题）模式：主题级硬门控不执行，降级为纯语义匹配度评分
- 逻辑依赖保护（如有 G_rel）仍生效
- 降级不报错不阻断——reply（回复）/complete（完成）轮次本来就不需要硬门控保护

## 六、拼接后的结构化标记规范

所有以摘要、锚点、占位符形态进入上下文窗口的内容，均以以下标记包裹：

```
[摘要-第N轮] 首句摘要... 结尾句摘要 [/摘要]
[锚点-第N轮] 关键实体词1, 关键实体词2 [/锚点]
```

模型在 prompt 中被告知：带标记的内容是"供给性上下文"而非"待回复的对话内容"。

## 八、taskType（任务类型）差异总览

| 维度 | field_based（基于字段） | topic_based（基于主题） |
|------|----------------------|----------------------|
| 硬门控锚点 | 字段名 DET（确定性程序）精确匹配 | 主题标签 embedding（向量嵌入）两阶段检索 |
| 语义匹配度 | bigram Jaccard | cosine（余弦相似度）×0.7 + bigram Jaccard ×0.3 |
| 结构化字段注入 | collectedFields（已采集字段）（JSON（JavaScript对象表示法）） | topicSnapshot（主题快照）+ topicEvolution（主题演化）分层包 |
| 领域规则注入 | stepName（环节名）精确匹配，按 importance（重要性标记）排序 | topicPath（主题路径）前缀匹配 + 子图扩散一层 |
| 最低匹配度阈值 | 0.2 | 0.3 |
| askingField（追问字段）= null（空）降级 | 字段门控停用，纯语义评分 | 主题门控停用，纯语义评分 |
