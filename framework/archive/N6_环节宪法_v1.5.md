<!-- @section ask-rules -->
【追问规则】
采用引导式多轮对话，逐步收敛。所有追问中 askingField 使用描述当前步骤的主题标签，供 context-manager 主题级硬门控使用。
**1. 上游消费确认阶段**（askingField="upstream_review"）：
- 展示 N4 转移图摘要（v5.4，含 ANALYZING→WAITING_CONFIRM 新增边）
- 展示 N5 v5.8 关键变更摘要（CLOSING 双出边 _pendingCollectedFields 区分处理已回写）
- 展示 N3 状态枚举摘要（IN_SESSION 子类型列表、topicEvolutionEnabled 标注）
- 确认用户是否已审查并同意基于这些输入进行 N6 设计
**2. 数据传递协议设计阶段**（askingField="protocol_design"）：
**2.1 逐条转移边确认**：
- 逐条展示 N4 转移图的所有转移边（含 ANALYZING→WAITING_CONFIRM）
- 若 N1 slackNodeDeclaration.enabled=true，协议表新增第17行 ANALYZING→SLACK_NODE（chitchat_path），携带数据为空，清空数据为空
- 对每条边，逐一确认：
- field_based 任务下的携带数据（字段名、类型、是否必传、说明）
- topic_based 任务下的携带数据（字段名、类型、是否必传、说明）
- 清空数据列表
- 补充说明
**2.2 关键差异确认**：
- 确认 ANALYZING→IN_SESSION 的 extracted 差异（field_based=键值对，topic_based=主题标签或空对象）
- 确认 IN_SESSION→EXECUTING 的 topic_based 附加数据（changeLevel 预判。expectsChangeLevel 从 N4 路由表条目读取）
- 确认 EXECUTING→LISTENING 的 _lastRecord 差异（field_based=结构化键值对，topic_based=最终输出文本，不是对话摘要）
- 确认 IN_SESSION→ANALYZING 的暂存内容差异（field_based=键值对，topic_based=对话摘要+主题标签+关键上下文片段。摘要生成方式：模型实例在 complete 时顺带输出；若未输出，调度器使用最近一轮的 message 作为摘要）
**2.3 topicEvolution 相关转移边确认**：
- 确认 topic_based 任务的 ANALYZING→IN_SESSION 携带 topicId 并初始化 topicEvolution 记录
- 确认 topic_based 任务的 ANALYZING→WAITING_CONFIRM 携带候选主题信息（topicId + candidateTopicName + candidateKeywords + matchResult）
- 确认 topic_based 任务的 IN_SESSION→EXECUTING 携带 changeLevel 预判（格式规范见第 4 阶段）
- 确认 topic_based 任务的 EXECUTING→LISTENING 追加 event 到 topicEvolution（调用轻量 LLM 生成 stateChange 摘要，stateSnapshot 标记为 checkpoint，推入提炼待处理队列）
- 确认 topic_based 任务的 giveup 追加 event 到 topicEvolution（stateSnapshot 标记为 abandoned，推入提炼待处理队列）
- 确认 topic_based 任务的 off-task 追加 event 到 topicEvolution（stateSnapshot 保持 active）
**2.4 CLOSING 双出边和半成品 source 标记确认**：
- 确认 CLOSING→IDLE 保留 _pendingCollectedFields（按 ttl 过期）。此规则已在 N5 v5.8 回写实现
- 确认 CLOSING→LISTENING 清空 _pendingCollectedFields。此规则已在 N5 v5.8 回写实现
- 确认 _closingSource 与 N4 exitType 映射关系：intent_exit/exit_word→normal→保留半成品；out_of_boundary*→fallback→清空半成品
- 确认 _pendingCollectedFields 含 source 标记（off_task / normal_pause）
- 确认 source="off_task" 时下次进入不主动询问；source="normal_pause" 时下次主动询问
**3. 守卫规则设计阶段**（askingField="guard_design"）：
**3.1 全局守卫确认**：
- 确认优先级层级：最高（退出词）> 高（唤醒词/取消词/align）> 普通（关键词扫描）
- 确认退出词优先于取消词
- 确认 align 元指令仅 LISTENING 态有效，仅当 alignScenarioFlag=true 时启用。IN_SESSION 中途的 align 需求通过 off-task 路径处理（与 N5 v5.8 第0层一致）
- 确认 DET 关键词扫描合并执行机制（全局守卫和 IN_SESSION 内守卫复用同一套代码）
**3.2 IN_SESSION 内守卫确认**：
- 逐项确认降级链纯四项检查：L1 结构校验、DET 值域复验、logprobs 检查、硬编码兜底
- 确认硬编码兜底不受 validation 规则存在与否的影响
- 确认 logprobs 检查区分冷启动期和非冷启动期
- 确认的两种模式：field_based（字段白名单）、topic_based（主题边界检查，可能追加轻量 LLM）
**3.3 无 validation 时的回退行为确认**：
- 确认 topic_based 无 validation 时仅执行 L1 结构校验 + logprobs 检查
- 确认跳过 DET 值域复验
- 确认硬编码兜底不受影响
