## 一、N2 在流程里的位置
### 1.1 触发条件（最高优先级）
**N2 仅当 N1 意图清单中至少存在 1 个 topic_based intent 时触发。** 若 N1 产出的所有 intent 均为 field_based，则跳过 N2，N1 边界清单直接冻结，流程直接进入 N3。
| 条件 | N2 行为 | N1→N3 衔接 |
|------|--------|------------|
| 所有 intent 均为 field_based | **跳过 N2** | N1 产出后直接冻结，进 N3。N3 需感知"无 N2 报告"状态。 |
| 至少 1 个 topic_based intent | **正常执行 N2** | 按原有流程，N2 通过后方可进 N3 |
**跳过理由**：field_based 场景的边界由字段集机械定义——字段映射成功即类一、映射失败即类二，不存在语义模糊性。N2 的逐条判类一/类二、护栏响应质量评估、topicEvolution 跨会话测试、双 confidence 机制等项目对纯 field_based 场景均无意义。跳过 N2 不损失边界可靠性。
跳过 N2 时，调度器需在 N3 消费的上下文（原 N2 测试报告位置）中标记为 `n2Skipped: true, n2SkipReason: "all_intents_field_based"`，供 N16 审计追溯。
### 1.2 完整流程（含 topic_based intent 时）
```
N1 产出边界清单 + 语料收集计划
↓
[触发条件判断：是否有 topic_based intent？]
├── 否（纯 field_based）→ 跳过 N2，直接进 N3
└── 是 → N2 边界紧张度测试
↓
N2 边界紧张度测试（独立节点，拿真实口语验证边界覆盖完整性 + 护栏响应质量）
↓
├── 覆盖完整 + 护栏响应合理 → 进 N3 拆状态
└── 覆盖有缺口 OR 护栏响应不合理 → 回 N1 修正
```
**关键定位**：N2 是独立节点。N2 吃 N1 的产出，产测试报告（humanReadable + machineReadable），归档到 L2 附录 N2 节，供 N16 回溯审计。N2 不修改 N1 的边界清单——除非回退或护栏升级建议被 N1 采纳后 N1 自己改。
N2 无设计者独立产出动作——角色一/二由模型执行、人审查，N3 起恢复设计者决策节点。
**N2 环节自身属性**：
- 调度侧 taskType：topic_based（N2 的核心计算行为——角色一生成口语语料、角色二做意图归类和护栏质量判断——是语义推理任务，非字段采集任务。测试报告的 machineReadable 结构化校验作为 topic_based 的格式校验子条款，由调度器 DET 复验执行）
- 子角色 taskType：角色一/二均为 topic_based（子角色宪法 @section taskType 声明 topic_based；子角色不采字段、askingField=null、在主题空间内生成语料/判决报告）
- N2 角色一/二的子角色调用模式：调度器直调角色一/二，不经过 IN_SESSION 状态。角色一/二是直调直出的单轮调用（注入上下文 → 模型输出 → complete），不涉及多轮追问。askingField 恒为 null（无追问目标），turnType 为四值（complete/reply/off-task/giveup）

## 二、执行体（三种模式可选）
| 模式 | 语料收集 | 逐条判内/外 | 审查 |
|------|---------|------------|------|
| A（纯人收集） | 人（非项目组成员/真实日志） | 模型（角色二） | 人 |
| B（模型参与） | 模型（角色一：语料搜索员，走信息隔离） | 模型（角色二） | 人 |
| C（混合模式） | 人收集基础 + 模型扩充（角色一，走信息隔离） | 模型（角色二） | 人 |
**模式选择纪律**：用户在 N2 开始时选择模式，选择后不可中途切换。模式选择理由记录在 N2 测试报告中。

| 优先级 | 路径 | 说明 |
|--------|------|------|

| 仅作初稿 | 设计者自写 | 交付前必须用路径 2 替换 ≥50% |
| 应急 | 模型生成（推理模型基于场景定义生成） | 必须走双角色串行+信息隔离 |

## 四、双角色串行 + 信息隔离机制（模式 B/C 启用）
### 4.1 角色一：真实语料搜索员（n2-corpus-hunter）
**启用条件**：仅模式 B/C 启用。模式 A 不启用。
**信息隔离原则（最高优先级）**：
- 角色一的输入只包含场景定义
- 不注入：边界清单、意图清单、N1 交付物全文、根宪法第 2 条核心元指令集（唤醒/退出/取消词表）
- 调度器在注入前做安全检查：只检查注入上下文中是否含有 N1 传递契约的 JSON 字段名（如 doList、notDoList、guardrailStrategies、intent、metaInstructions 等），不检查自然语言文本中的词汇。如果上述字段名出现在注入上下文中 → 拦截，报错
**语料多样性最低要求**：
- 硬性下限：边界模糊表达 ≥ 3 条、明显越界表达 ≥ 2 条
- **跨会话续写表达 ≥ 2 条，且必须作为语料集的第 1-2 条（优先位置）**——确保角色二无需跳过其他语料即可优先测试 topicEvolution 跨会话匹配能力
- 正向指令自然填充剩余（通常 15-25 条）
- 比例分布（正向 55-65%、明显越界 10-20%、边界模糊 20-30%）仅作参考，不强制精确控制
- 硬性下限不满足时，提示设计者补充，不强制重跑
- 模式 A（纯人收集）时，设计者收集的语料也应满足相同的硬性下限要求，且跨会话续写表达置于语料集前 2 位
- 角色一宪法中明确："你的任务是模拟真实用户的自然表达，包括偏离、边界模糊的表达、以及跨会话续写表达（如'继续上次那个角色扮演'、'继续上次那首诗'）。真实用户不会只说场景定义内的指令。跨会话续写表达必须放在你输出语料集的前 2 条位置。"
**语料多样性自动评分**：调度器在角色一 complete 后自动检查：
- 边界模糊表达数量 ≥ 3？
- 明显越界表达数量 ≥ 2？
- 跨会话续写表达数量 ≥ 2？
- 跨会话续写表达是否为前 2 条？
- 语料总数在 20-30 之间？
- 如未满足硬性下限，自动提示"语料多样性不足，请补充"并阻止进入角色二
**污染检测**：采用严格白名单校验——角色一输出顶层只允许 `corpus` / `corpus_count` / `source_distribution` 三个顶层字段。corpus 数组内每个语料对象只允许 `id`、`user_input`、`source`、`context` 四个字段。任何额外字段 → 判定"语料污染"，记录日志，要求重跑角色一。`source_distribution` 供 humanReadable 审查使用，角色二不消费。
### 4.2 角色二：边界测试执行器（n2-boundary-tester）
**启用条件**：所有模式都启用。
**注入上下文**：
- 角色一输出的原始语料集（已校验无污染，第 1-2 条为跨会话续写表达）（模式 B/C）或人收集的语料集（模式 A）
- N1 边界清单
- N1 意图清单（含 taskType 和 topicEvolutionEnabled）
- alignScenarioFlag
- taskTypeAlignmentStatus
- N1 topicEvolutionEnabled 标注
- N1 softLandingTargets（软着陆目标 intent 映射表）
**边界速查表**：调度器自动生成，按硬拒组/软着陆组/引导性硬拒组分组。生成规则：从 N1 边界清单的 notDoList 按 guardrailStrategy 分组——hard_reject → hardRejectGroup（topic 值），soft_landing → softLandingGroup（topic + targetIntent），guided_reject → guidedRejectGroup（topic 值）。若 N1 边界清单中无对应项，对应数组为空。结构为：
```json
{
"hardRejectGroup": ["不闲聊", "不推荐理财", "不记收入", "不支持图片识别", "不支持手势操作"],
"softLandingGroup": [{"topic": "理财咨询", "targetIntent": "query"}],
"guidedRejectGroup": ["不闲聊（可回锚）"]
}
```
调度器在注入前做完整性检查。
角色二的产出：测试报告（含逐条判决 + 边界覆盖完整性判定 + 护栏响应质量评级 + 缺口分析 + 修正建议）。
**【taskType 归类逻辑指引】**（角色二宪法中实现）：
- field_based：用字段对应判断——用户表达能否映射到某个 intent 的必采字段
- topic_based：用主题边界判断——用户表达是否在主题边界内
- 不要混用两种归类方式
- 归类逻辑：先根据语料内容选择最匹配的 intent，再根据该 intent 的 taskType 执行对应的归类方式
- Fallback 规则：如果无法确定最匹配的 intent，则使用场景的主 taskType 进行归类。主 taskType 按以下优先级确定：
- 如果 N1 边界清单中 **topic_based intent 数量 ≥ field_based intent 数量** → 主 taskType = topic_based
- **否则** → 主 taskType = field_based
- 理由：topic_based 的主题边界判断比 field_based 的字段对应判断更宽松（包容性更强）。在模糊情况下优先采用更宽松的归类方式，避免将潜在的功能范围内输入误判为类二
- 在 judgment.note 中标注"intent 匹配模糊，使用场景主 taskType（topic_based / field_based）"
- 如果某条语料的 taskType 归类结果与 N1 中该 intent 的 taskType 不一致（例如 N1 标为 field_based 但角色二用 topic_based 方式归类），自动在 note 中标注"taskType 归类方式与 N1 声明不一致，请审查"
- 如果某条语料的 taskType 归类让你感到明显不合逻辑，在 judgment.note 中标注"疑似 taskType 误判"，供 N3 审查
**【confidence 填写要求】**：
- confidence 的角色二是对自己"判内/外"这一判断的置信度评价
- 正常填写：按角色二认为正确的归类方式，如实填写置信度
- **双重 confidence（taskType 归类方式与 N1 声明不一致时）**：如果角色二认为 N1 的 taskType 标注有误，在 note 中填写两个 confidence 值——"按 N1 声明（field_based）归类时 confidence=0.3；按当前方式（topic_based）归类时 confidence=0.8"
- DET 复验以 judgment.confidence 主值为准执行拦截（即角色二按自己认为正确的方式归类时的 confidence）
- **logprobs 检查统一规则**（集中声明）：对所有 judgment 执行，使用通用 logprobs_threshold=0.4
- 冷启动期（cold_start_observation_window 内）：软拦截——标记 logprobs_warning 附加到返回中让用户可见，不直接丢弃
- 非冷启动期：硬拦截——confidence < 0.4 时 DET 拦截。拦截结果标记为"需人工审查"，不直接丢弃。人工审查后可决定采纳或驳回该 judgment（这是 N2 测试环节的特有处理——N2 不是生产环节，低置信度 judgment 不应打断测试流程，由人工最终裁决）
- 角色二可在 note 中自愿标注"低置信度"，由人工审查在 DET 拦截之外额外关注
**【跨会话续写表达归类指南】**（角色二宪法中实现）：
- 任何表达形式为"继续上次的XX"、"接着之前的XX"、"回到那个XX"等，如果在 N1 意图清单中存在对应的 topic_based intent（其 topicEvolutionEnabled=true），一律归类为类一
- 如果 N1 意图清单中不存在对应的 topic_based intent，在 note 中标注"跨会话续写表达找不到对应 topic_based intent"
- 语料集第 1-2 条为此类表达，需优先测试
**【publicRuleCompliance 自检要求】**（角色二宪法必须包含）：
- 规则 2（坚持真实判断）：按上述 logprobs 检查统一规则执行自检——列出 confidence < 0.4 的语料 ID 和 confidence 值
- 规则 3（一次聚焦，禁止跨任务延伸）：测试报告是否越界产出？声明"测试报告未越界产出"
- topicEvolution 启用检查：topic_based intent 的 topicEvolutionEnabled 是否均为 true？是否有 topic_based intent 但 topicEvolutionEnabled=false 的异常情况？如有则标注
### 4.3 流转逻辑
```
N2 环节启动
→ 调度器展示 execModeDecision 选择提示（A/B/C）
→ 用户选择模式，调度器记录选择理由
→ 调度器置 _n2Phase = "corpus_hunting"（模式 B/C）或 skip（模式 A 直进 boundary_testing）
→ 模式 B/C：
• 调度器直调角色一，注入场景定义（信息隔离）
• 角色一 complete → 白名单校验 + 多样性自动评分 + 硬性下限检查（含跨会话续写 ≥ 2 条且为前 2 条）
• 调度器置 _n2Phase = "boundary_testing"
• 调度器直调角色二，注入角色一输出+边界清单+意图清单+边界速查表+softLandingTargets+topicEvolutionEnabled标注+coveragePassThreshold
→ 模式 A：
• 调度器置 _n2Phase = "boundary_testing"
• 调度器直调角色二，注入人收集语料+边界清单+意图清单+边界速查表+softLandingTargets+topicEvolutionEnabled标注+coveragePassThreshold
→ 角色二 complete → 覆盖完整性判决
→ _n2Phase = null（调度器清掉）
→ 进入 §六归档流程（DET 复验 + WAITING_CONFIRM + 用户确认），verdict 路由在用户确认后执行
```
`_n2Phase` 由调度器在 N2 环节私有临时变量中维护（存入 schedulerState._sceneExtensions._n2Phase），不对外暴露，N2 complete 后清掉。N2 角色二虽然是直调直出，但重试时调度器可将 retryHint 作为注入上下文的额外字段传给角色二（首次调用为 null，重试时填写）。
### 4.4 错误处理
| 错误场景 | 处理 |
|---------|------|
| 角色一注入上下文中含 N1 传递契约的 JSON 字段名 | 拦截，报错 |
| 角色一输出含非白名单字段（顶层或 corpus 数组内） | 判定污染，要求重跑角色一 |
| 角色一语料多样性不满足硬性下限（边界模糊 < 3 条 或 明显越界 < 2 条 或 跨会话续写 < 2 条 或 跨会话续写非前 2 条） | 自动提示补充，阻止进入角色二 |
| 角色二注入上下文不完整 | 拦截，报错 |
| 角色二输出格式不符合 output-schema | 重试 1 次，仍失败则报错 |
| 角色二报告结论为"回退" | 路由到 N1，携带修正建议 |

## 五、执行步骤
### Step 0：优先测试 topicEvolution 跨会话续写类表达（语料集第 1-2 条）
语料集的第 1-2 条为跨会话续写表达（角色一宪法已要求置于优先位置），角色二优先测试并验证 N1 边界清单中 topic_based intent 的 topicEvolution 跨会话匹配能力是否被覆盖。若语料集中无意涵盖跨会话续写表达，需在测试报告的 `topicEvolutionCoverageCheck` 中标注为"缺样本"。
N2 判断与 N1 意图清单的一致性——若跨会话续写表达在 N1 意图清单中找不到对应的 topic_based intent，需在测试报告中特别标注。
### Step 1：逐条判"内 / 外" + 护栏响应质量评估
对每条语料：
- 判断落在"做什么"内（类一）还是"不做什么"外（类二）
- 类一：标注对应 intent + 是否有处理路径
- 类二：标注对应护栏策略 + 是否有处理路径 + 护栏响应质量（优秀/合格/待优化/不合格）
**归类规则表**：
| 语料特征 | 归类 | 理由 |
|---------|------|------|
| 落在"做什么"内（field_based：字段对应；topic_based：主题边界内） | 类一 | 系统功能边界内 |
| 落在"不做什么"外，处理策略是软着陆→intent | 类一 | 软着陆引导到系统已有能力 |
| 用户表达内含意图切换，以最终意图为准（前提：最终意图在 doList 中） | 类一 | 最终意图在系统功能边界内 |
| align 触发词表达（alignScenarioFlag=true 时） | 类一 | align 是系统功能 |
| topic_based intent 的跨会话续写表达（如"继续上次那首诗"） | 类一 | topicEvolution 跨会话匹配是 topic_based 标配能力 |
| 松弛节点边界内的闲聊输入（slackNodeDeclaration.enabled=true 时） | 类一 | 松弛节点是合法存在的低约束节点，仅受根宪法约束 |
| 落在"不做什么"外，处理策略是硬拒 | 类二 | 系统无处理路径 |
| 落在"不做什么"外，处理策略是引导性硬拒（可回锚） | 类二 | 系统拒绝但引导回当前任务，仍算越界 |
n**松弛节点边界测试**（slackNodeDeclaration.enabled=true 时执行）：
- 松弛节点的闲聊输入按类一处理——松弛节点是合法存在的低约束节点，闲聊输入本身就在其边界内
- 松弛节点护栏策略从 slackNodeDeclaration.guardrailStrategy 读取（硬拒/引导性硬拒/软着陆）
- 若松弛节点闲聊输入无处理路径或护栏响应不合理→在测试报告中标注 slackNodeDefect
**taskType 对归类规则的影响**：
| taskType | 归类方式 | 示例 |
|----------|---------|------|
| field_based | 字段对应——用户表达能映射到某个 intent 的字段 | "猫粮35"→record 的 category+amount，类一 |
| topic_based | 主题边界——用户表达在主题边界内 | "他为什么背叛"→角色扮演主题内，类一 |
**护栏响应质量评估**（仅对类二语料）：
| 质量等级 | 表现 | 示例 |
|---------|------|------|
| ✅ 优秀 | 软着陆 + 引导到功能范围内 | "我可以帮你记，你要先记完这条还是先查？" |
| ✅ 合格 | 明确拒绝 + 说明边界 + 提供替代 | "小安只会记账，不会聊天哦。你需要记一笔吗？" |
| ⚠️ 待优化 | 硬拒 + 无替代引导 | "我不做这个" |
| ❌ 不合格 | 无处理路径 / 崩溃 / 幻觉回应 | — |
**软着陆归类的 N3 风险标注**：软着陆目标 intent 的存在性由 N3 完成时校验。N2 阶段按 N1 声明的软着陆目标归类，但需在测试报告中标注"以下软着陆归类假设 N1 声明的目标 intent 在 N3 阶段保持不变。如 N3 发生 intent 合并/拆分/改名，需重新评估。"
### Step 2：边界覆盖完整性校验
覆盖率 = 有处理路径的语料数 / 总语料数（类一 intent+路径 ✅ + 类二护栏+路径 ✅）÷ 总语料数。
N2 使用 `coveragePassThreshold`——该参数从 **N1 tunableDeclarations.generalParameters 中的 boundary_coverage_threshold** 中读取（默认 0.85，范围 [0.5, 1.0]，消费方 = N2 覆盖率校验）。N2 将其记录在 testEnvironment 中供审计。若场景要求严格零缺口，设计者在 N1 阶段将此值调至 1.0。
- 覆盖率 ≥ coveragePassThreshold 且 无"不合格"护栏响应 → 通过，进 N3
- 覆盖率 < coveragePassThreshold OR 存在"不合格"护栏响应 → 回 N1
### Step 3：缺口分析与处置
当覆盖率 ≥ coveragePassThreshold 但仍有缺口时，分析缺口类型：
| 缺口类型 | 处置 |
|---------|------|
| 缺口全在"软着陆目标 intent 缺失"类 | 先补 N3 草稿 intent，再决定是否回 N1 |
| 缺口在"硬拒类但频繁触发" | 护栏升级候选（硬拒→软着陆或引导性硬拒） |
| 缺口在"类一 intent 漏标" | 重划边界（走正式扩展流程） |
**护栏升级的硬纪律**：软着陆指向的 intent 必须已在 N3 草稿中存在。护栏升级建议 N1 必须采纳（mandatory: true），修订后无需重测（升级只会改善覆盖，不会新增缺口）。

## 八、与 N1 / N3 的衔接
### 8.0 N2 跳过时的 N1→N3 衔接（纯 field_based 场景）
当触发条件判定为跳过 N2（所有 intent 均为 field_based）时：
- N1 边界清单直接冻结，N1 传递契约中的 taskTypeAlignmentStatus 设为 `aligned`
- N3 消费的 N2 报告位置由调度器注入一个最小化占位对象：`{ n2Skipped: true, n2SkipReason: "all_intents_field_based", n1VersionAtTestTime: "<N1 版本>" }`
- N2 的所有字段（coverageRate / verdict / gapAnalysis / topicEvolutionCoverageCheck / ambiguousIntentMatches / guardrailQuality 等）在 field_based 场景下不适用，N3 不应依赖这些字段
- N16 审计时，N2 占位对象标记为 `审计状态: N2已跳过（纯field_based场景）`，不视为流程缺口
### 8.1 N2 ← N1（正常执行时）
N2 吃 N1 的边界清单 + 意图清单 + 语料收集计划 + taskTypeAlignmentStatus + topicEvolutionEnabled 标注 + softLandingTargets + tunableDeclarations.generalParameters 中的 boundary_coverage_threshold（N2 内部称 coveragePassThreshold）。publicRuleInputs、modelSelectionMechanism、contractInheritance、tunableDeclarations 其余参数不消费但完整传递至下游节点。
### 8.2 N2 → N3（正常执行时）
N2 通过，N1 边界清单才算"冻结"，N3 才能按这份清单拆状态、写路由、绑 intent。
N3 消费 N2 测试报告的：coverageRate / verdict / gapAnalysis / taskTypeAlignmentStatus / topicEvolutionCoverageCheck / n3SoftLandingRiskNote / n1IntentListSnapshot / ambiguousIntentMatches / n1VersionAtTestTime / n2ReportVersion。
**ambiguousIntentMatches 的 N3 审查指引**：ambiguousIntentMatches 中的语料 ID 对应的 intent 在 N3 阶段应被重点审查——如果多条语料指向同一个 intent 但匹配模糊，说明该 intent 的粒度可能过粗，N3 应考虑拆分。
**n1VersionAtTestTime 的 N3 校验规则**：N3 消费快照时，先校验版本号是否与当前 N1 一致。如果不一致，按以下规则处理：
- N1 的 doList 发生了 intent 增删或 taskType 变更（不仅仅是措辞调整）→ N2 测试报告被视为过期，**必须重新执行 N2**
- 仅有措辞调整（如场景定义修改、描述文字修改）→ 允许 N3 人工审查差异后继续，但需在 N3 交付物中标注"N2 基于旧版 N1 执行，已人工审查差异"
### 8.3 N2 → N1（回退）
覆盖率 < coveragePassThreshold 或存在"不合格"护栏响应时回 N1，修正后重新跑 N2。回退时携带缺口分析和修正建议。重新执行 N2 时自动生成新的 n2ReportVersion。
### 8.4 N2 → N1（护栏升级，不回退）
覆盖率 ≥ coveragePassThreshold 时，护栏升级建议 N1 必须采纳（mandatory: true），但不需要重新跑 N2（升级只会改善覆盖）。软着陆升级的 target intent 须已在 N3 草稿中存在。N1 修改边界清单后，调度器自动将对应 suggestions 的 adoptionStatus 更新为 adopted 或 rejected_with_reason。如果 N1 拒绝了 mandatory 建议，调度器报错并阻止进入 N3。
### 8.5 N2 与降级链四项检查+硬编码兜底的关联
- L1 结构校验：N2 测试报告的 JSON 格式校验
- DET 值域复验：覆盖率范围、verdict 枚举等
- logprobs 检查：对所有 judgment 执行，使用通用 logprobs_threshold=0.4。冷启动期软拦截；非冷启动期硬拦截 + 需人工审查
- ：测试报告是否越界产出
- 硬编码兜底：上述均失败时的最终兜底

## 九、一句话收 N2
纯 field_based 场景跳过 N2，N1 直接冻结进 N3。含 topic_based intent 时，N2 验证 N1 边界清单的覆盖完整性和护栏响应质量。拿 20-30 条非项目组成员的真实口语（或模型生成走信息隔离）逐条判类一/类二，类一全有 intent+处理路径、类二全有护栏策略+处理路径且响应质量合格 → 覆盖率 ≥ coveragePassThreshold（从 N1 tunableDeclarations.generalParameters 中的 boundary_coverage_threshold 读取，默认 0.85）→ 通过进 N3。覆盖率不达标或有"不合格"响应 → 缺口分析 → 回 N1 修正。角色一校验用白名单+多样性自动评分（跨会话续写 ≥ 2 条且置于第 1-2 条），角色二归类遵循先匹配 intent + Fallback 含平局处理（topic_based 优先），confidence < 0.4 时冷启动期软拦截/非冷启动期硬拦截+人工审查，taskType 不一致时填双 confidence。护栏升级建议 N1 必须采纳，adoptionStatus 全程跟踪。N1 版本变更时 intent 增删/taskType 变更强制重跑 N2。
