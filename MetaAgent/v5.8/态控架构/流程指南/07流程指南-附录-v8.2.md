### 附录 C：高风险场景降级链参考（扩展层）

对于医疗预问诊、复杂客服、金融风控等高危场景，默认的四项检查是通用底线，不可缩减。在此基础上，可追加以下增强方案：

```
simple: [L1_retry, DET_value_recheck, logprobs_check, cross_task_extension_check, hardcoded]
escalated: [L1_retry, DET_value_recheck, logprobs_check, cross_task_extension_check, hardcoded, human_takeover]
human_takeover: { target: "人工队列", sla: "5分钟" }
```

**启用条件：** 仅当场景的"错误代价"判据为"代价大且无客观答案"或"代价大但有客观校验"时，考虑启用 escalated 模式。默认场景仍用 simple（四项检查 + 硬编码兜底）。

taskType 在高风险场景中的影响：field_based 高风险场景的 DET 值域复验做严格的值域校验；topic_based 高风险场景的 DET 复验做格式校验 + 可能需要 needSemanticExtensionCheck:true。两种模式的高风险场景都可启用 escalated 模式。

高风险场景的 WAITING_CONFIRM 节点在 escalated 模式下，责任收敛不再止步于"当前用户确认"，而是升级为"人工队列接管"。SLA 超时未响应时，按 preset 降级到 hardcoded 或 abort。

注意：escalated 模式启用时需在场景的边界清单中显式声明人工接管的目标队列和 SLA。四项检查 + 硬编码兜底不可被 human_takeover 替换——human_takeover 是四项检查 + 硬编码兜底之后的扩展层。

### 附录 D：align-session 可选扩展规范

align-session 是可选扩展，非态控架构标准能力。若场景含意图校准需求，可启用 align-session 扩展。启用时需：

1. 额外编写 align-session.json 宪法
2. 在路由表中增加 align 相关条目
3. 在 tunable 参数中增加 divergence_threshold
4. 同步更新场景的边界清单
5. 在 N5 第0层之后插入第0.1层（align 静默模式触发与处理）

align-session 核心机制（仅在启用时生效）：静默输入 / 多模型并行仲裁 / 锚点确认 / 迭代裁决 / 对齐数据包沉淀。

与核心架构的边界：align-session 启用时不修改根宪法第1-4条（核心四条不含 align 内容），仅在路由表和 tunable 参数中扩展，并在 N5 插入第0.1层。align-session 不启用时，核心架构完全不受影响。

### 附录E. 宪法冲突检测引擎实现


**实施规范：**

- `single_param` 规则用 json-rules-engine（Node.js）或 Drools（Java）实现确定性 range/enum 校验
- `cross_param` 规则由冲突检测引擎在遍历 single_param 完成后执行 compute 表达式求值——用沙箱 `eval()`（仅允许 `+ - * / floor ceil min max` 运算符 + params 变量 + constants 常量）
- 参数变更时通过 `params` 字段做依赖追踪——仅被引用参数的变更触发对应 `cross_param` 规则重新执行
- `violation_action` 默认值：`single_param` 规则默认为 `hard_reject`；`cross_param` 规则默认为 `warning`
- `violation_action=hard_reject` 行为：拒绝保存 + 返回错误详情
- `violation_action=warning` 行为：弹窗展示 warning message + suggested_adjustment，用户可选择"仍然保存"或"调整后重试"。warning 不阻止保存
- compute 表达式求值失败（语法错误、引用不存在参数）→ 视为 `hard_reject`，阻止保存并返回明确的表达式错误位置
- conflict_check 规则用 JSON 格式声明，与 tunable 参数一同存在 tunables.json
- 保存前执行全部规则：`single_param` 任一 hard_reject → 拒绝保存；`cross_param` warning → 弹窗但允许继续
- 冲突检测引擎是确定性代码，不调 LLM

**规则示例：**

```json
{
  "conflict_rules": [
    {
      "type": "single_param",
      "name": "logprobs_threshold_range_check",
      "condition": {
        "all": [
          { "fact": "parameter", "path": "$.logprobs_threshold", "operator": "lessThan", "value": 0 },
          { "fact": "parameter", "path": "$.logprobs_threshold", "operator": "greaterThan", "value": 1 }
        ]
      },
      "event": { "type": "conflict", "message": "logprobs_threshold 必须在 [0, 1] 范围内" }
    },
    {
      "type": "cross_param",
      "name": "topic_threshold_ordering_check",
      "violation_action": "hard_reject",
      "params": ["topic_anchor_similarity_threshold", "topic_evolution_match_threshold"],
      "compute": "topic_anchor_similarity_threshold >= topic_evolution_match_threshold",
      "event": {
        "type": "cross_param_conflict",
        "severity": "error",
        "message": "topic_anchor_similarity_threshold 必须严格小于 topic_evolution_match_threshold——跨会话匹配需要比环节内锚点更高的精度。"
      }
    }
  ]
}
```

> 完整 `cross_param` 规则清单（含上下文预算、冷启动窗口、规则提炼触发、置信度冻结一致性等 5 条初始规则）见总纲 v5.5 §10.1，本指南 §8.2 也已同步引用双模式体系。

### 附录F. L2→L3 拆包语义一致性校验


**实施规范：**

- L2 文档中"规则声明"用 `<!-- @rule -->` 标记
- L3 拆包时 DET 自动提取所有 `<!-- @rule -->` 内容
- 对比 L3 的 session-constitution 是否包含对应规则
- 缺失 → 拆包失败

**L2 文档标记示例：**

```
<!-- @rule --> 金额上限为 999999
<!-- @rule --> 时间不能为未来
<!-- @rule --> 必采字段：种类、金额、时间
```

**L3 拆包校验流程：**

1. DET 扫描 L2 文档，提取所有 `<!-- @rule -->` 标记内容
2. DET 扫描 L3 session-constitution 的 @section validation 结构化 JSON
3. 对比每条 L2 规则是否在 L3 validation 中有对应规则
4. 缺失 → 拆包失败，返回缺失规则列表

### 附录G. contractStore 物理存储规范


**实施规范：**

- SQLite 单文件数据库，每个用户/项目一个 .db 文件
- WAL（Write-Ahead Logging）模式支持并发读 + 单写
- 数据库文件路径：`<project_root>/.contractstore/<user_id>.db`

**v7.0 表结构：**

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| analyzing_contract_in | ANALYZING 节点的原始 contract-in 副本（支持意图级纠错逃生舱） | — |
| analyzing_contract_out | ANALYZING 节点的 contract-out 副本 | — |
| topicIndex | 检索索引表，用于 DET 关键词快速检索和主题关联存储 | topicId（主键）、topicName、keywords（逗号分隔）、creatorUserId、createdAt、lastActiveAt、relatedTopics（JSON 数组） |
| topicEvolution | 主题演化主表，只存储 major 级 event + compressedMinor + 压缩后新增的 minor 级 event | topicId（主键）、intent、userId、stateSnapshot（JSON）、createdAt、updatedAt |
| topicEvolutionEvent | 事件主表 | eventId（主键）、topicId（外键）、changeLevel（模型判定）、nodeId、turnType、timestamp、stateChange（JSON）、askingField、modelOutputSummary |
| topicEvolutionArchive | 归档表，存储被压缩的 minor 级原始 event + 被判定为 invalid 的 event | 字段结构与 topicEvolutionEvent 相同。invalid event 附带"因模型判定为互斥撤销而未入主表"的元数据标记 |
| **domainRules**（v7.1 升级） | **领域规则库** | **ruleId（主键）、rule（规则内容）、topicPath（JSON 数组）、edges（JSON 对象，含 depends_on/conflicts_with/strengthens，统一 snake_case + 主动语态，strengthens 表示"当前规则强化目标规则"）、conditions（JSON 对象）、immutableLevel（L0-L3）、source（scenario_seed/user_initiated/model_inferred）、level（strong/medium/suggestion）、confirmedBy、confirmedAt、version、status（active/candidate/archived/pending_review）** |
| ruleCandidates | 候选规则池 | candidateId（主键）、rule、evidenceCount、passCount、validationRate、status（pending/confirmed/rejected） |
| ruleMiningQueue | 规则提炼待处理队列 | queueId（主键）、topicId、intent、checkpointAt、status（pending/processing/done） |
| **sessionCheckpoints** | **field_based 进度检查点** | **userId（主键）、lastCompletedStep、completedSteps（JSON 数组）、stepSnapshots（JSON 对象），ttl 默认 7 天。过期仅清理进度行，不联动清理 roomConversationLog（roomConversationLog 按 importance 分级保留：永久/90天/30天/7天）** |
| **roomConversationLog**（v7.2 新增） | **房间对话日志** | **id（主键自增）、userId、roomId、sessionId、turnIndex、userInput、modelResponse、turnType、askingField、changeLevel、timestamp。按 userId+roomId 索引，支持全文搜索** |

**并发控制：**

- WAL 模式支持多读单写
- 写操作用事务包裹，崩溃时自动回滚
- 多用户并发时，每个用户独立 .db 文件，无跨用户锁竞争

### changeLevel 抽样人工复核机制

**风险：** 模型可能偷懒将所有变更标记为 major 或跳过判定。DET 格式校验能拦空值/超长，但挡不住"编一个合理理由"的隐蔽偷懒。

**缓解：** 按 `changelevel_major_sample_rate`（默认 5%，tunable 可配）抽检 major 标记事件 → 人工判断是否真属于结构性变更 → false_positive 比例>30% 触发冷启动应急解冻。冷启动期间抽检比例临时上调至 10%。详见总纲 v5.5 文末风险条目。
