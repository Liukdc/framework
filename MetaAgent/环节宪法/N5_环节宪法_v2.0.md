# N5 环节宪法：调度器核心逻辑与上下文管理设计（元智能体 v2.0·草案）

**版本**：v2.0 | **日期**：2026-07-12 | **状态**：已定稿 | **基于母本**：N5 环节宪法 v1.5

<!-- @section constitution -->
field_based
taskType 判定理由：元智能体的调度器核心逻辑全部基于确定性规则——16 步意图路由为 hard_match、五层接待规则每层均为 DET 或固定配置、context-manager 三接口为契约式调用。不存在语义模糊空间或需要模型权衡的设计决策，故归为 field_based。

收敛方向：将 N4 转移图+路由表+N3 状态枚举+N1 边界清单转化为一份完整的 N5 调度器设计方案，确保：
- 五层接待规则完整（第0层元指令词表匹配 + 第0.1层静默模式 + 第0.5层关键词扫描 + 第1层意图识别 + 第2层环节宪法注入 + 第3层降级链）
- 第2层宪法映射与 N4 路由表一致
- 第2层 contractEvolution 三层注入（领域规则+候选预览+变更历史包）+ stepName 精确匹配 + DET 字段检索
- 降级链六项检查完整（field_based 值域校验 + contractEvolution 检测）
- context-manager 三接口完整（含 changeLevel/changeLevelReason——用于 contractEvolution 分层留存）
- 调度器数据结构完整（含 _contractStore 规则提炼相关表）

<!-- @section ask-rules -->
采用引导式多轮对话，逐步收敛。所有追问中 askingField 使用描述当前步骤的主题标签。

**1. 上游消费确认阶段**（askingField="upstream_review"）：
- 展示 N4 转移图摘要（状态数量、路径类型覆盖、降级链五步转移边）
- 展示 N3 状态枚举摘要（8 框架 + 16 IN_SESSION + 省略清单）
- 展示 N1 边界清单摘要（16 intent 全部 field_based、元指令词表、alignScenarioFlag=false、slackNodeDeclaration.enabled=false）
- 展示 N7 根宪法摘要（第2条元指令集词表、第5条降级链、第6条仲裁权）
- 展示 N9 环节宪法摘要（16 份宪法文件名映射 + @section validation 校验规则概览）
- 展示 N10 tunable 清单摘要（通用参数 10 个 + 规则提炼参数 4 个）
- 确认用户是否已审查并同意基于这些输入进行 N5 设计

**2. 第0层全局硬匹配设计**（askingField="layer0_design"）：
- 确认唤醒词/退出词/取消词词表来源为 N1 元指令词表
- 确认匹配方式：子串匹配
- 确认优先级规则：退出词优先匹配，命中后不再检查取消词
- 确认适用状态：所有状态均生效
- 确认纪律：纯字符串匹配，0 次 LLM 调用

**3. 第0.1层静默模式设计**（askingField="layer0_1_design"）：
- alignScenarioFlag=false → layer0_1=null。跳过本阶段。

**4. 第0.5层关键词扫描设计**（askingField="layer0_5_design"）：
- 确认关键词词表来源为 N1 notDoList
- 确认输出格式：offTaskSuspicion = { keywords: [...], confidence: 0.x }
- 确认 confidence 分级阈值标注为 tunable 引用（不硬编码）：
  - 默认高阈值 0.8 / 低阈值 0.5
- 确认纪律：只标记不阻断，不强制覆盖模型判断

**5. 第1层意图识别设计**（askingField="layer1_design"）：
- 确认 modelName=DeepSeek V4 快速版，禁止深度推理模型
- 确认宪法文件为 N9 产出的 analyzing-session.json
- 确认防抖配置：antiFlapWindow 和 antiFlapThreshold 标注为 tunable 引用（不硬编码）
- 确认防抖锁定后的追问规则：锁定后不重判，追问"您是想 [首次判定 intent] 吗？"
- 确认防抖计数器在会话结束时自然清空

**5.5 前置依赖校验设计**（askingField="prerequisite_guard"）：
- 确认 16 步前置依赖链：P0→N1→N3→N4→N5→N6→N7→N8→N9→N10→N11→N12→N13→N14→N15
- 确认校验逻辑：ANALYZING 匹配到 intent=Ni 后，查 N4 转移边的 prerequisiteDependencies 字段（N4 §3.1 前置依赖矩阵）
- 确认校验失败行为：定位"已完成最高环节=Nk"，引导至 Nk+1（而非目标 Ni）
- 确认执行体：DET 确定性校验，0 次 LLM
- 确认引导文案模板："Ni 需要 [前置环节列表] 先完成。当前 Nk+1 还没走——我们先继续？"

**6. 第2层环节内模型自治设计**（askingField="layer2_design"）：
- 确认 constitutionMapping 与 N4 路由表一致（16 个子类型全部 hard_match）
- 确认 turnTypes 六值完整（ask/reply/complete/off-task/giveup/validation_failed）
- 确认 field_based 锚点模式：字段级硬门控，DET 确定性字符串匹配
- 确认 changeLevel/changeLevelReason 用于修改日志记录（不作为 topicEvolution）
- 确认 WAITING_CONFIRM 触发条件：已完成的步骤被重跑时
- 确认 EXECUTING 写入时同步记录变更日志

**7. 第3层 DET 值域复验设计**（askingField="layer3_design"）：
- 确认五项检查完整：L1 结构校验 + DET 值域复验 + confidence 检查 + 跨任务延伸检测 + 硬编码兜底
- 确认 DET 值域复验：field_based=值域校验（按 N9 @section validation 规则）
- 确认 confidence 检查区分冷启动/非冷启动：
  - 冷启动期：软拦截，标记 warning 附加到返回中
  - 非冷启动期：硬拦截，进入 CLARIFYING
- 确认仲裁权：arbitrator=scheduler_det，arbitrationRule=det_overrides_model
- 确认 CLARIFYING 状态：复用模型 message 转发用户，不调 LLM 生成追问
- 确认 CLOSING 双出边：normal→IDLE / fallback→LISTENING

**8. 调度器数据结构定义**（askingField="scheduler_state"）：
- 确认 schedulerState 完整字段：_state/_activeSession（含 intent/taskType/turnHistory）/_antiFlapCounter/_pendingCollectedFields/_closingSource/_contractStore（含 ruleCandidates/ruleMiningQueue/domainRules 规则提炼相关表）
- 确认 turnHistory 每条含 changeLevel 和 changeLevelReason（用于修改日志记录）

**9. context-manager 三接口规范定义**（askingField="context_manager"）：
- 确认 buildPromptContext() 7 参数
- 确认 updateGraph() 8 参数（含 changeLevel/changeLevelReason——写入 turnHistory 供修改日志和规则提炼使用）
- 确认 archiveGraph() 接口
- 确认纪律：调度器只调接口，不内联拼接逻辑

**用户确认标准**：
- **显式确认**：用户明确说"确认""可以""没问题""行"等肯定词
- **隐式确认**：用户在下一轮追问中直接回答了新问题且未对上一轮提议提出异议
- **沉默跳过**：若用户连续两轮未对某提议回应，模型应主动追问"关于 XX，您是否同意？"
