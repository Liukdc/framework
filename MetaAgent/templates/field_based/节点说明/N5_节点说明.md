**版本**：v2.0 | **日期**：2026-07-12 | **状态**：已定稿 | **基于母本**：N5 节点说明 v5.8

## 一、位置与定位

```
N4 状态转移图 + 路由表（已定稿）
  ↓
N5 调度器核心逻辑 + context-manager 三接口  ← 你在这里
  ↓
N6 数据传递协议 + 守卫规则
```

**定位**：基于 N4 转移图和路由表，设计五层接待规则（第0层~第3层）、context-manager 三接口、调度器数据结构。全部 field_based，无 topicEvolution，无 N2 双角色。

## 四、产出物

N5 产出四样东西：五层接待规则、context-manager 三接口规范、调度器数据结构、N5→下游传递契约。

### 3.1 产出一：五层接待规则

#### 第0层：全局硬匹配

纯字符串匹配，0 次 LLM 调用。词表来源：N7 根宪法第2条。优先级：退出词 > 取消词。

| 词表类型 | 元智能体词表 |
|---------|-----------|
| 唤醒词 | `小智` |
| 退出词 | `小智拜拜` / `小智退出` |
| 取消词 | `小智取消这个节点` |
| 匹配方式 | 子串匹配 |
| 适用状态 | 所有状态均生效 |

#### 第0.1层：静默模式

alignScenarioFlag=false → layer0_1=null，不启用。

#### 第0.5层：关键词扫描

词表来源：N1 notDoList。只标记不阻断，不强制覆盖模型判断。

| 配置项 | 值 |
|--------|-----|
| keywordSource | N1_boundary_list_notDo_column |
| blockModel | false |
| outputFlagName | offTaskSuspicion |
| 分级阈值 | off_task_suspicion_high_threshold（默认 0.8）/ off_task_suspicion_low_threshold（默认 0.5）——均标注为 tunable 引用 |

#### 第1层：意图识别

| 配置项 | 值 |
|--------|-----|
| modelName | DeepSeek V4 快速版 |
| forbiddenModels | 深度推理模型 |
| constitutionFile | analyzing-session.json |
| antiFlapWindow | tunable 引用 |
| antiFlapThreshold | tunable 引用 |
| 防抖锁定追问 | "您是想 [首次判定 intent] 吗？" |

#### 前置依赖校验（ANALYZING→IN_SESSION 路由守卫）

元智能体 16 步有严格的前置依赖链（P0→N1→N3→...→N15）。ANALYZING 匹配到 intent 后，不直接路由到 IN_SESSION——先校验该 intent 的前置环节是否全部完成。

**校验逻辑**（DET 执行，0 次 LLM）：

```
ANALYZING 匹配到 intent=Ni
  ↓
查 N4 转移边中 Ni 的前置依赖条件
  ↓
├─ 前置全部完成 → IN_SESSION(Ni)
└─ 前置未完成 → 定位"已完成最高环节=Nk"
     → 不进入 IN_SESSION(Ni)
     → 引导至 Nk+1
     例：用户说"帮我做 N5"，但 N3 还没做
     → 回复"N5 需要 N3 和 N4 先完成。当前 N3 还没走——我们先定义状态枚举？"
     → 进入 IN_SESSION(n3)
```

| 配置项 | 值 |
|--------|-----|
| 执行体 | DET（确定性校验） |
| 依赖数据 | N4 转移边 prerequisiteDependencies 字段（N4 §3.1 前置依赖矩阵） |
| 校验失败行为 | 引导至下一个未完成环节，不进入目标 IN_SESSION |
| 0 次 LLM | 纯字段校验，不调模型 |

#### 第2层：环节内模型自治

| 配置项 | 值 |
|--------|-----|
| constitutionMapping | 与 N4 路由表一致，16 个子类型各映射对应宪法文件 |
| turnTypes | ask / reply / complete / off-task / giveup / validation_failed |
| anchorMode | **fieldBased**：字段级硬门控，DET 确定性字符串匹配。**topicBased**：不适用（无 topic_based intent） |

**上下文注入说明**：全部 field_based，topic_based 的三层注入包不适用。异步规则提炼独立启用——同一 stepName 再次执行时，context-manager 注入该 stepName 对应的已生效领域规则。

#### 第3层：DET 值域复验

五步降级链：

| 步骤 | 执行体 | field_based 行为 |
|------|--------|----------------|
| L1 结构校验 | DET | JSON.parse + Schema，失败→重试1次→熔断 |
| DET 值域复验 | DET | 按各环节宪法 @section validation 规则校验 |
| confidence 检查 | DET | 冷启动软拦截 / 非冷启动硬拦截→CLARIFYING |
| 跨任务延伸检测 | DET | 字段白名单——16 步内字段不触发越界 |
| 硬编码兜底 | DET | 熔断后兜底回复落盘 |

| 配置项 | 值 |
|--------|-----|
| arbitrator | scheduler_det |
| arbitrationRule | det_overrides_model |
| clarifyingMessageSource | model_returned_message |

## 五、关键纪律

1. 第0层纯字符串匹配，0 次 LLM 调用
2. 第1层使用快速路由模型，禁止深度推理模型
3. 调度器只调 context-manager 接口，不内联拼接
4. 全部 field_based，anchorMode 仅启用 fieldBased
5. 所有阈值通过 getTunable() 读取，严禁硬编码
6. 降级链五步完整，arbitrator=scheduler_det

## 七、常见踩坑

| 坑 | 表现 | 后果 |
|----|------|------|
| 第0层用了 LLM 匹配 | 唤醒词匹配调了模型 | 每次唤醒多一次 LLM 调用 |
| 第1层用了深度推理模型 | 意图识别用了 deepseek-r1 | 简单意图识别耗时过长 |
| 防抖参数硬编码 | antiFlapWindow=5 写死在代码里 | 无法根据实际运行情况调优 |
| context-manager 接口被绕过 | 调度器直接操作 turnHistory | 上下文拼接逻辑散落各处 |

## 九、小白操作指南

### 7.0 先搞清楚你在干什么

N4 画了走廊和路牌。你的任务是写"接待员"——用户走进来怎么接待（五层规则），接待员用什么工具（context-manager），接待员记什么账（调度器状态）。

### 7.1 三步走

**第一步：设五层规则**

第0层：词表匹配（纯字符串，不调模型）。第0.5层：关键词扫描（只标记不阻断）。**前置依赖守卫**：ANALYZING 匹配 intent 后先查 N4 前置条件——没完成的步骤不能跳过。第1层：意图识别（快速模型）。第2层：宪法映射（16 步各对应一份宪法）。第3层：降级链（五步防线）。

**第二步：定接口**

buildPromptContext 7 参数，updateGraph 8 参数（含 changeLevel/changeLevelReason——用于修改日志记录），archiveGraph 无参数。

**第三步：填状态**

_state 8 值枚举，_activeSession 含 intent/taskType/turnHistory，_contractStore 含规则提炼相关表。

## 十、一句话收 N5

元智能体 N5 五层规则全部 field_based——第0层字符串匹配词表（唤醒/退出/取消），前置依赖守卫（DET 校验 N4 转移边前置条件，0 LLM），第1层快速模型意图识别，第2层 16 宪法映射+fieldBased 硬门控，第3层五步降级链（arbitrator=scheduler_det）。context-manager 三接口七参数，调度器只调不拼接。无 topicEvolution，无 N2 双角色。
