# N3 节点说明（元智能体 v2.0）

**版本**：v2.0 | **日期**：2026-07-12 | **状态**：已定稿 | **基于母本**：N3 节点说明 v5.5

---

## 一、位置与定位

```
P0（认知加载，不产文件）
  ↓
N1 场景定义 + 边界清单（已定稿）
  ↓
[纯 field_based → N2 跳过，直进 N3]
  ↓
N3 状态枚举 + 执行体分派 + taskType 固化  ← 你在这里
  ↓
N4 状态转移图 + 路由表
```

**定位**：元智能体 16 个 intent 全部 field_based，N2 跳过。N3 的任务是：把 16 步设计流程翻译为标准状态集 + 执行体分派表。全部 intent 的 taskType 固化为 field_based，topicEvolutionEnabled=false。

**规则提炼说明**：元智能体为复杂 field_based 任务——每个环节内模型与用户交互自由度大，可启用规则提炼。规则提炼独立于 topicEvolution，不与 changeLevel 分层。每个规则直接映射到 stepName，经交叉验证和用户确认后存入领域规则库，同一环节再次执行时注入。规则提炼参数在 N10 独立声明。

## 三、产出物

### 3.1 标准状态集

以下为标准状态集全量菜单。元智能体 16 步设计流程：P0→N1→[N2 跳过]→N3→N4→N5→N6→N7→N8→N9→N10→N11→N12→N13→N14→N15。

| 状态 | 类型 | 执行体 | 必选 | 说明 |
|------|------|--------|------|------|
| IDLE | 稳态 | — | ✅ | 空闲，等待唤醒词"小智" |
| LISTENING | 稳态 | 人 | ✅ | 等待用户输入 |
| ANALYZING | 瞬态 | LLM 轻量 | ✅ | 意图识别，匹配 16 个步骤 |
| IN_SESSION(p0~n15) | 稳态容器 | LLM 实例 | ✅ | 16 个设计环节，环节级自治 |
| EXECUTING | 瞬态 | DET | ✅ | N11-N15 产出写入项目目录 |
| CLOSING | 瞬态 | DET | ✅ | 会话清理 |
| CLARIFYING | 守护态 | DET | ✅ | 校验失败追问转发 |
| WAITING_CONFIRM | 守护态 | 人 | ✅ | 覆盖已有产出需确认；修改时记录变更日志（timeStamp/field/changeLevel/reason），原版本保留 |

### 3.2 省略状态

| 省略状态 | 理由 |
|---------|------|
| SLACK_NODE | N1 slackNodeDeclaration.enabled=false |
| PARAM_PARSING | 合并入 ANALYZING——16 步意图识别简单，无需独立参数提取 |
| VALIDATING | 合并入 IN_SESSION + DET 复验——各环节宪法已有 @section validation |
| PREVIEWING | 无需可视化 diff |
| CONSTITUTION_GEN | 静态宪法，不动态生成 |
| CONFIRM_CONSTITUTION | 不适用 |

### 3.3 执行体分派表

16 个 IN_SESSION 子类型：全部 field_based，全部 topicEvolutionEnabled=false。

| 子类型 | 执行体 | dispatchCriteria | 分派理由 |
|--------|--------|-----------------|---------|
| IN_SESSION(p0~n10) | llm_instance | costLevel=low, canHardcode=false, hasObjectiveAnswer=true, taskType=field_based | 设计引导——用户表达多样化，无法用正则穷举。如"帮我设计记账智能体"vs"做个财务助手" |
| IN_SESSION(n11~n12) | llm_instance | costLevel=low, canHardcode=false, hasObjectiveAnswer=true, taskType=field_based | 模型为主的契约对齐和拆包——逻辑复杂，需语义理解 |
| IN_SESSION(n13) | llm_instance | costLevel=low, canHardcode=false, hasObjectiveAnswer=true, taskType=field_based | 代码模型主导生成，人审架构决策 |
| IN_SESSION(n14~n15) | llm_instance | costLevel=low, canHardcode=false, hasObjectiveAnswer=true, taskType=field_based | 测试和调参——需分析测试结果、判断调参方向 |
| ANALYZING | llm_light | costLevel=low, canHardcode=false, hasObjectiveAnswer=true, taskType=field_based | 用户可能说"开始设计""帮我做N5""继续上一步"——需语义理解 |
| EXECUTING | det | costLevel=low, canHardcode=true, hasObjectiveAnswer=true, taskType=field_based | 确定性程序——文件写入、目录创建 |
| CLOSING | det | costLevel=low, canHardcode=true, hasObjectiveAnswer=true, taskType=field_based | 清理态 |
| CLARIFYING | det | costLevel=low, canHardcode=true, hasObjectiveAnswer=true, taskType=field_based | message 复用，确定性转发 |
| WAITING_CONFIRM | human | costLevel=high, canHardcode=false, hasObjectiveAnswer=false, taskType=N/A | 重跑某步覆盖已有产出——不可逆，需人确认 |

### 3.4 环节内模型实例定位说明

IN_SESSION 内的模型实例属于 LLM 执行体，拥有环节内自治权——调度器不参与其逐轮对话，只在环节入口（注入宪法+上下文）和出口（收 turnType + DET 复验）交互。

元智能体全部 field_based，不启用 topicEvolution，三层注入包不适用。但异步规则提炼独立启用——同一 stepName 再次执行时，context-manager 注入该 stepName 对应的已生效领域规则。

## 四、关键纪律

1. 必选状态全覆盖——IDLE/LISTENING/ANALYZING/IN_SESSION×16/EXECUTING/CLOSING/CLARIFYING/WAITING_CONFIRM
2. 16 个 intent 全部 field_based，全部 topicEvolutionEnabled=false
3. N2 跳过是预期行为——纯 field_based 场景字段本身就是边界，无语义模糊空间。无 N2 报告不构成审计缺口
4. 执行体分派按三步判据推导，rationale 含表达模式举例
5. taskType 从 N1 直接继承，N3 固化后运行时不变
6. 异步规则提炼独立启用，按 stepName 映射，field_based 场景同样适用

## 六、常见踩坑

| 坑 | 表现 | 后果 |
|----|------|------|
| 给 N2 创建 IN_SESSION(n2) | 16 步中 N2 被跳过但照样建了子类型 | 多个永不触发的空房间 |
| 忘记标记 n2Skipped | intentCoverageCheck 没有 N2 跳过标记 | N16 审计时无法判断是"预期跳过"还是"遗漏" |
| EXECUTING 被省略 | 以为"纯对话场景不需要 EXECUTING" | N11-N15 无法写入文件——EXECUTING 不可省略 |
| rationale 仅复述字段值 | "costLevel=low, canHardcode=false，故用 LLM" | 失去推导价值 |
| topicEvolutionEnabled 误标为 true | 照搬 N3 v5.5 基底模板未修改 | N5 试图初始化不存在的 topicEvolution |

## 七、小白操作指南

### 7.0 先搞清楚你在干什么

元智能体 16 个步骤全部是 field_based，N2 跳过。你的任务是把 16 步翻译成状态房间——每步一个 IN_SESSION 子类型，再加上 8 个框架房间（IDLE/LISTENING/ANALYZING/EXECUTING/CLOSING/CLARIFYING/WAITING_CONFIRM）。

### 7.1 动手前先检查

| 你要用到的东西 | 谁给你的 | 用来干嘛 |
|-------------|---------|---------|
| N1 意图清单 | N1 的人 | 16 个 intent 全部 field_based，taskType 直接抄过来 |
| N1 元指令词表 | N1 的人 | 确认 ANALYZING 的意图识别规则覆盖唤醒/取消/退出 |

> N2 跳过——不需要 N2 测试报告。纯 field_based 场景字段本身就是边界。

### 7.2 两步走

**第一步：勾选框架房间**

IDLE / LISTENING / ANALYZING / EXECUTING / CLOSING / CLARIFYING / WAITING_CONFIRM——全部必选。

**第二步：给每个步骤开一个房间**

命名规则：IN_SESSION(p0) ~ IN_SESSION(n15)。全部 field_based，全部 topicEvolutionEnabled=false。每个房间的执行体为 llm_instance，autonomyLevel=full。

### 7.3 自查清单

- [ ] 8 个框架状态 + 16 个 IN_SESSION 子类型齐全（共 24 个）
- [ ] 6 个省略状态在 omittedStates 中声明了理由
- [ ] 全部 topicEvolutionEnabled=false
- [ ] 每个状态的 rationale 含至少 1 个表达模式举例
- [ ] intentCoverageCheck.n2Skipped=true

### 7.4 常见错误

| ❌ 错误 | ✅ 正确 |
|---------|--------|
| 给 N2 建了 IN_SESSION(n2) | N2 跳过——不建子类型。护栏规则二的前置依赖链中 N2 也不存在 |
| EXECUTING 被省略 | N11-N15 产出写入项目目录需要 EXECUTING——不可省略 |
| rationale 写"costLevel=low, canHardcode=false" | 写具体原因："用户可能说'帮我设计记账智能体'也可能说'做个财务助手'——表达多样化，无法用正则穷举" |
| 忘了标记 n2Skipped | intentCoverageCheck 中标注 n2Skipped=true |

## 八、一句话收 N3

元智能体 N3 产出 8 个框架状态 + 16 个 IN_SESSION 子类型（p0~n15）的完整状态集——全部 field_based，全部 topicEvolutionEnabled=false。N2 跳过（纯 field_based 场景，字段即边界）。6 个不适用状态已声明省略理由。异步规则提炼独立启用，按 stepName 映射。
