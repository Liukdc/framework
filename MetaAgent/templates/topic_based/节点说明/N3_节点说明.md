## 一、位置与定位

```
N1 边界清单 + 意图清单（含 taskType + alignScenarioFlag）（已冻结）
  ↓
N2 边界紧张度测试（已通过，含 taskType 分布验证 + topicEvolution 覆盖检查）
  ↓
N3 状态枚举 + 执行体分派 + taskType 固化 + topicEvolution 初始化  ← 你在这里
  ↓
N4 状态转移图 + 路由表
  ↓
N5 调度器核心逻辑 + context-manager 三接口 + topicEvolution 处理
  ↓
N6 数据传递协议 + 守卫规则 + topicEvolution 存储
  ↓
N7-N10 写宪法
```

**【人话版定位】**
N3 把 N1 划好的边界翻译成"有哪些房间"（状态）和"每个房间谁管事"（执行体）。每个房间还要标注"这房间是填表式还是聊天式"（taskType），聊天式的房间自动启用 topicEvolution 三层递进机制。下游 N4 用这些房间画走廊，N5 用这些房间定规矩并初始化 topicEvolution，N9 给每个房间写服务手册。

**【哲学根基】**
N3 的设计受"边界控制优于微流程切碎"直接驱动——拆节点的首要动机是切换执行体（引入 DET 或 HUMAN 进行兜底/校验），不是"切细了显得专业"。执行体分派判据的优先级是：代价 > 能否硬规则化 > 有无客观答案。

态控架构 v4.2 明确指出——"写诗与修改家具参数都是受限空间内的收敛过程，区别在于收敛空间的连续性"。N3 的状态设计需要同时支持 field_based 和 topic_based 两类任务，taskType 在 N1 阶段标注，N3 继承并固化。

topic_based 任务在 N3 固化 taskType 时，N5 调度器将自动初始化 topicEvolution 三层递进机制。异步规则提炼结果经交叉验证和用户确认后存入领域规则库，同类型新主题初始化时按 topicPath 子图匹配精准注入。规则提炼完全异步，不在状态机流转链路内执行。

**【术语区分】**
- **硬规则化（canHardcode）**：指能否用确定性规则覆盖所有输入-输出映射。这是执行体分派的判据之一。
- **硬拒（hard_reject）**：是 N1 边界清单中的一种护栏处理策略，指系统直接拒绝用户请求。
- 两者含义无关。canHardcode 决定"用程序还是用大模型"，hard_reject 决定"怎么拒绝越界请求"。

## 二、输入

| 输入项 | 来源 | 说明 |
|--------|------|------|
| N1 边界清单（已冻结） | N1 交付物 | 含 doList 每项的 taskType/interactionMode/dataHandling，以及 notDoList 每项的 guardrailStrategy/canReturnAnchor/softLandingIntent |
| N1 意图清单（含 taskType） | N1 交付物 | intentList 数组。系统意图（exit、cancel、help 等）由 ANALYZING 直接路由，不需要独立的 IN_SESSION 子类型 |
| N1 alignScenarioFlag | N1 交付物 | 判断是否启用 align 场景（可选扩展） |
| N1 taskTypeAlignmentStatus | N1 交付物 | aligned / pending_backport。N2 通过时保证为 aligned。N3 不接受 pending_backport 输入 |
| N1 softLandingTargets | N1 交付物 | 软着陆目标 intent 映射（含 N2 护栏升级建议被 N1 采纳后的更新） |
| N2 测试报告 | N2 交付物 | 含 taskTypeDistribution、publicRuleCompliance、topicEvolutionCoverageCheck、ambiguousIntentMatches、n1VersionAtTestTime 等 |

**N3 消费 N2 测试报告的关键字段及用途**：

| N2 字段 | 用途 | N3 如何使用 |
|---------|------|------------|
| topicEvolutionCoverageCheck.crossSessionVerdict | 跨会话续写表达覆盖情况 | 如果 verdict 为 partial_class_two 或 all_class_two，需审查 topicEvolutionConfig.enabledIntents 是否遗漏了某个 topic_based intent |
| topicEvolutionCoverageCheck.topicEvolutionEnabledAnomalies | topicEvolution 启用异常 | 如存在异常（如 topic_based intent 未启用 topicEvolution），需在 topicEvolutionConfig 中修正 |
| ambiguousIntentMatches | intent 匹配模糊的语料 ID 列表 | 如果多条语料指向同一个 intent，考虑在 N3 中细化该 intent（如 query → query_single + query_sum），并在 refinementMap 中记录映射 |
| publicRuleCompliance.detVerified | N2 的公共规则 DET 复验结果 | N3 开始前应确认 rule2_honesty 和 rule3_no_cross_task 均为 passed |
| n1VersionAtTestTime | N2 测试时的 N1 版本号 | 按两档规则校验（见 §3.5.1 详细说明） |
| taskTypeDistribution | taskType 分布统计 | 如果某个 intent 的 taskType 分布与 N1 初判不一致，N3 在固化前应先确认 N1 是否已回溯修正 |

### 3.1 产出一：标准状态集

以下为标准状态集全量菜单，各场景按需选用。必选状态为基础框架建议，特定场景可裁剪但必须在 omittedStates 中声明理由。

| 状态 | 类型 | 默认执行体 | 含义 | 是否必选 | 省略条件 |
|------|------|----------|------|---------|---------|
| IDLE | 稳态 | — | 空闲，等待唤醒 | ✅ 必选 | 不可省略 |
| LISTENING | 稳态 | 人 | 等待用户输入 | ✅ 必选 | 不可省略 |
| ANALYZING | 瞬态 | LLM 轻量 | 意图识别 | ✅ 必选 | 不可省略 |
| IN_SESSION | 稳态容器 | LLM 模型实例 | 环节执行中，环节级自治 | ✅ 必选 | 不可省略 |
| CLARIFYING | 守护态 | DET | 校验失败追问转发（监控标记由调度器日志承载） | ✅ 必选（通用） | 特定场景可省略——监控标记由调度器日志承载，但需在 omittedStates 中声明 |
| WAITING_CONFIRM | 守护态 | 人 | 不可逆操作确认 | ✅ 必选（通用） | 场景无任何不可逆操作（无删除、无支付、无修改已有数据——新增数据不属于不可逆操作）且用户确认可通过隐式方式完成时可省略。隐式确认的操作化定义：由 N4/N5 设计者根据具体场景自行定义（如"用户在同一 IN_SESSION 内连续两次确认同一操作"等），N3 仅判断场景是否符合省略条件 |
| EXECUTING | 瞬态 | DET | 业务执行 | ✅ 必选（通用） | 纯对话引导类场景可省略。topic_based intent 存在的场景不可省略（N5 需在 EXECUTING 阶段追加 event 到 topicEvolution） |
| CLOSING | 瞬态 | DET | 会话清理 | ✅ 必选 | 特定场景可省略——exit 直回 IDLE，无需清理态 |
| SLACK_NODE | 稳态 | LLM 通用 | 松弛节点（闲聊态），仅受根宪法约束，不区分 taskType | ⚪ 可选 | N1 slackNodeDeclaration.enabled=false 时省略 |
| PARAM_PARSING | 瞬态 | LLM 通用 | 参数映射 | ⚪ 可选 | 可合并入 ANALYZING |
| VALIDATING | 瞬态 | DET | 值域/约束校验 | ⚪ 可选 | 可合并入 IN_SESSION + DET 复验 |
| PREVIEWING | 瞬态 | DET | 预览 diff | ⚪ 可选 | 需可视化 diff 的场景启用 |
| CONSTITUTION_GEN | 瞬态 | LLM/DET | 局部宪法生成 | ⚪ 可选 | 动态宪法场景必选 |
| CONFIRM_CONSTITUTION | 守护态 | 人 | 局部宪法确认 | ⚪ 可选 | 低置信度时启用 |

**场景省略声明**：通用场景必须保留 CLARIFYING 和 EXECUTING 状态。特定场景可省略，但必须在 L2 文档中声明省略理由。

**不可逆操作定义**（WAITING_CONFIRM 省略判断依据）：
- 删除：数据不可恢复 → 不可逆
- 支付：资金转移不可撤销 → 不可逆
- 修改已有数据：覆盖后不可回滚 → 不可逆
- 新增数据（如 record）：可后续删除或修改 → 不属于不可逆

### 3.2 产出二：执行体分派表

每个状态必须标注执行体，判据优先级：代价 > 能否硬规则化 > 有无客观答案。

**dispatchCriteria 结构化字段**：
- costLevel：high / low / none
- canHardcode：boolean
- hasObjectiveAnswer：boolean
- taskType：field_based / topic_based / N/A（从 N1 doList 继承，N3 固化后运行时不变）
- topicEvolutionEnabled：boolean（topic_based 为 true，field_based 为 false）

**taskType 多 intent 继承处理**：如果某个 IN_SESSION 子类型对应多个 taskType 不同的 intent，dispatchCriteria 的 taskType 默认按 intent 数量取多数派。少数派的 intent 数量或预期调用频率占比超过 30% 时，设计者必须拆分 IN_SESSION 子类型。预期调用频率的基数来源：以 N2 测试报告中的 taskTypeDistribution 为准，无 N2 数据时以设计者估算为准（但需在注释中声明估算依据和灰度期验证计划）。

**mixed 子类型分发后 taskType 执行规则**：mixed 子类型分发后，被分发到的局部宪法自身的 taskType（来自 N1 doList 中该 intent 的 taskType）生效——askingField 语义、DET 复验方式、N9 环节宪法均按该局部宪法的 taskType 执行，不沿用 mixed 父级的 majority taskType。

**align 子类型的 topicEvolutionEnabled**：align 子类型的 topicEvolutionEnabled 为 false（align 是静默校准模式，不需要主题演化记录）。

**执行体分派速查表**（见附录 A）

**示例执行体分派表**：

| 状态 | 类型 | 执行体 | dispatchCriteria | 分派理由 |
|------|------|--------|-----------------|---------|
| IDLE | 稳态 | — | costLevel=none, topicEvolutionEnabled=false | 初始空闲态 |
| ANALYZING | 瞬态 | LLM 轻量 | costLevel=low, canHardcode=false, hasObjectiveAnswer=true, topicEvolutionEnabled=false | 口语表达多样化，无法用正则穷举所有表达 |
| IN_SESSION(record) | 稳态容器 | LLM 模型实例 | costLevel=low, canHardcode=false, hasObjectiveAnswer=true, taskType=field_based, topicEvolutionEnabled=false | 用户表达多样化（"猫粮35""记个猫粮35块"），无法穷举，有客观答案，代价小 |
| IN_SESSION(delete) | 稳态容器 | LLM+人确认 | costLevel=high, canHardcode=false, hasObjectiveAnswer=true, taskType=field_based, topicEvolutionEnabled=false | 删除涉及数据丢失，代价大 |
| IN_SESSION(roleplay) | 稳态容器 | LLM 模型实例 | costLevel=low, canHardcode=false, hasObjectiveAnswer=false, taskType=topic_based, topicEvolutionEnabled=true | 连续语义空间探索，无客观答案，自动启用 topicEvolution |
| IN_SESSION(align) | 稳态容器 | LLM 模型实例（多模型并行） | costLevel=low, canHardcode=false, hasObjectiveAnswer=false, taskType=topic_based, topicEvolutionEnabled=false | 静默校准模式，不需要主题演化记录 |
| EXECUTING | 瞬态 | DET | costLevel=low, canHardcode=true, hasObjectiveAnswer=true, topicEvolutionEnabled=false | 确定性程序 |
| CLARIFYING | 守护态 | DET | costLevel=low, canHardcode=true, hasObjectiveAnswer=true, topicEvolutionEnabled=false | message 复用，确定性转发 |
| SLACK_NODE | 稳态 | LLM 通用 | costLevel=low, canHardcode=false, hasObjectiveAnswer=false, taskType=N/A, topicEvolutionEnabled=false | 闲聊态仅受根宪法约束，N1 slackNodeDeclaration 控制是否启用 |

### 3.3 产出三：环节内模型实例定位说明

IN_SESSION 内的模型实例属于 LLM 执行体，但拥有环节内自治权——调度器不参与其逐轮对话，只在环节入口（注入宪法+上下文，topic_based 模式额外注入三层注入包：已生效领域规则（按 topicPath 子图匹配精准注入）+ 关联主题摘要 + topicEvolution 分层包）和出口（收 turnType + DET 复验）交互。

**范围限定**：本说明仅适用于 IN_SESSION 状态内的模型实例。其他环节的模型实例受各自宪法约束，不适用此自治规则。

**topic_based 任务 topicEvolution event 追加的责任链**：topic_based 任务的模型实例在返回 turnType 时需顺带输出 changeLevel 预判字段（major/minor/invalid）。调度器在 IN_SESSION 出口收到模型的 changeLevel 预判后，在 EXECUTING 阶段将 event 追加到 topicEvolution，并将主题信息推入规则提炼待处理队列。DET 本身不负责语义判断——event 追加由调度器执行，使用模型判定的 changeLevel。此要求的实现细节由 N5 负责，N3 仅声明 topicEvolutionConfig 的默认参数。

### 3.5 小白操作指南

#### 3.5.0 先搞清楚你在干什么

你现在要干的事：把 N1 划好的"这个智能体能做什么、不能做什么"翻译成两个东西——这个智能体需要哪些"房间"（状态），以及每个房间谁管事（执行体）。

**重要区分**：N1 意图清单中可能包含系统意图（如 exit、cancel、help）。这些系统意图由 ANALYZING 直接路由，不需要你为它们创建 IN_SESSION 子类型。

#### 3.5.1 动手前先检查

| 你要用到的东西 | 谁给你的 | 用来干嘛 |
|-------------|---------|---------|
| N1 意图清单（含 taskType） | N1 的人 | 给每个业务意图开一个 IN_SESSION 子类型房间，taskType 直接抄过来 |
| N1 边界清单（含软着陆目标） | N1 的人 | 确认软着陆指向的意图在 N3 的房间里都有对应入口 |
| N1 alignScenarioFlag | N1 的人 | 判断要不要加 align 房间 |
| N2 测试报告 | N2 的人 | 看任务类型分布、公共规则合规结果、topicEvolution 覆盖检查、ambiguousIntentMatches、n1VersionAtTestTime |

**特别提醒——在动手前，先检查以下 N2 测试报告的关键字段**：

1. **publicRuleCompliance.detVerified**：确认 rule2_honesty 和 rule3_no_cross_task 均为 passed。如为 failed，需先回溯 N2 修正

2. **n1VersionAtTestTime**——按两档规则校验：

   > - **N1 doList 发生了 intent 增删或 taskType 变更**（不仅仅是措辞调整）→ **必须等待 N2 重新执行**（N2 v5.10 §八 8.2 硬规则）。N2 重新通过后再进 N3
   > - **仅有措辞调整**（场景定义修改、描述文字修改）→ 人工审查后继续，在 publicRuleCompliance.rule2_truthfulJudgment.note 中记录" N1 版本有措辞变更，已人工审查，不影响状态设计"
   > - **版本一致** → 正常推进

3. **ambiguousIntentMatches**：如果多条语料指向同一个 intent，考虑在 N3 中细化该 intent（如 query → query_single + query_sum）

#### 3.5.2 三步走

**第一步：勾选公共房间**

必选状态全部勾选。topic_based intent 存在的场景，EXECUTING 不可省略（N5 需在 EXECUTING 阶段追加 event 到 topicEvolution）。特定场景如需省略，必须在 omittedStates 中声明理由。

**第二步：给每个业务意图开一个干活房间**

命名规则：IN_SESSION(意图名)。taskType 从 N1 直接抄过来。topic_based 意图自动启用 topicEvolution（topicEvolutionEnabled=true，N5 会根据此字段初始化 topicEvolution 记录和 topicIndex），field_based 不启用。align 子类型的 topicEvolutionEnabled 为 false。

如果多个意图合到一个房间——taskType 按多数派填，少数派在备注里说明处理方式。少数派占比 >30% 时强制拆分（基数来源：以 N2 测试报告中的 taskTypeDistribution 为准，无 N2 数据时以设计者估算为准）。

检查 N2 测试报告的 ambiguousIntentMatches：如果多条语料指向同一个 intent，考虑在 N3 中细化该 intent（如 query → query_single + query_sum），并在 refinementMap 中记录映射。

确认软着陆房间都存在：去 N1 的 softLandingTargets 看（含 N2 护栏升级建议被 N1 采纳后的更新）——所有标注了软着陆→intent: X 的条目，那个 X 对应的房间在 N3 里开了没有？如果没开——要么补开房间，要么回 N1 改软着陆目标。

**第三步：标注每个房间的边界来源**

框架性房间标注"通用状态（无特定边界对应）"或"间接对应"。IN_SESSION 子类型房间标注追溯到 N1 "做什么"列中对应的功能条目。CLARIFYING 标注"间接对应：N1 对话范围——校验失败处理"。WAITING_CONFIRM 标注"间接对应：N1 交互方式——确认机制"。

## 五、关键纪律

1. 必选状态必须全覆盖，特定场景可裁剪但必须声明理由
2. 执行体分派按三步判据结构化推导，rationale 必须说明具体原因，禁止仅复述字段值
3. IN_SESSION 子类型必须覆盖 N1 业务意图（系统意图除外），IN_SESSION 子类型数量 ≥ N1 业务意图数量（细化时在 refinementMap 中记录映射）
4. 每个状态追溯 N1 边界清单（CLARIFYING 追溯至 N1 对话范围维度，WAITING_CONFIRM 追溯至 N1 交互方式维度，框架性状态可标注"通用状态"或"间接对应"）
5. taskType N3 固化后运行时不变（mixed 子类型分发后按各自 N1 taskType 执行，不沿用父级 majority。详见 §3.2）
6. topic_based intent 必须标注 topicEvolutionEnabled=true，N5 据此初始化 topicEvolution
7. 少数派超 30% 必须拆分 IN_SESSION 子类型（基数以 N2 测试报告中的 taskTypeDistribution 为准，无 N2 数据时以设计者估算为准）
8. N2 回退后 N3 必须重新审视状态设计
9. N1 因采纳 N2 建议而修改 softLandingTargets 时，N3 需重新执行软着陆覆盖检查（源：N1 softLandingTargets，含 N2 护栏升级建议被 N1 采纳后的更新）
10. n1VersionAtTestTime 版本不一致时——intent 增删/taskType 变更必须等 N2 重跑通过；仅措辞调整可人工审查后继续

## 七、常见踩坑

| 坑 | 表现 | 后果 |
|----|------|------|
| 给系统意图建 IN_SESSION 子类型 | 给 exit/cancel 建了 IN_SESSION(exit) | 状态转移图异常 |
| CLARIFYING 直接省略不声明 | 通用场景也省略 CLARIFYING | validation_failed 无独立状态承载，监控标记缺失 |
| topic_based intent 未标注 topicEvolutionEnabled=true | topicEvolutionEnabled=false | N5 不初始化 topicEvolution，跨会话续写失效 |
| taskType 多 intent 未处理 | IN_SESSION(mixed) 对应多个 taskType 但只填一个 | N5 无法判断 askingField 语义 |
| 少数派超 30% 未拆分 | 2 field_based + 1 topic_based 合并未拆分 | 内部路由复杂度过高 |
| mixed 分发后仍沿用父级 majority | 子级按父级 taskType 执行 | minority intent 的 askingField 语义和 DET 复验方式错误 |
| rationale 仅复述字段值 | "costLevel=low, canHardcode=false，故用 LLM" | 失去推导价值 |
| intent 细化未记录映射 | query_single/query_sum 对应 N1 query 但 refinementMap 为空 | N1 回溯时无法追溯细化关系 |
| N2 回退后 N3 未重做 | N1 修改了边界清单但 N3 仍用旧状态设计 | IN_SESSION 子类型与 N1 不一致 |
| 忽略 N2 的 ambiguousIntentMatches | N2 发现多条语料指向同一 intent 但 N3 未细化 | intent 粒度不合理，下游实现困难 |
| N1 版本号不一致时直接推进 | intent 已变更但 N3 没等 N2 重跑 | N3 状态枚举基于过期 N2 测试结果 |

## 八、一句话收 N3

N3 把 N1 划好的边界翻译成状态枚举 + 执行体分派 + taskType 固化 + topicEvolution 初始化配置——必选状态全覆盖（特定场景可省略但必须声明理由），执行体按三步判据推导（rationale 含表达模式举例），dispatchCriteria 含 taskType 和 topicEvolutionEnabled（topic_based 自动启用 topicEvolution），mixed 子类型分发后按各自 taskType 执行，少数派超 30% 强制拆分。IN_SESSION 子类型覆盖 N1 业务意图（子类型数量 ≥ 意图数量），每个状态追溯 N1 边界清单。消费 N2 测试报告的关键字段（ambiguousIntentMatches、topicEvolutionCoverageCheck 等）指导状态设计。n1VersionAtTestTime 按两档规则校验（intent 变更等 N2 重跑，仅措辞人工审查）。公共规则合规检查聚焦 N3 自身交付物。这份状态枚举是后面所有设计的骨架，未经审查不得进入下一步。
