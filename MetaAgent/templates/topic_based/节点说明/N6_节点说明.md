## 四、产出物：三套规矩

### 4.1 先搞清楚两种任务类型

| 任务类型 | 典型场景 | 服务员在干什么 | 转场时带什么 | 质检时查什么 |
|---------|---------|--------------|------------|------------|
| field_based（强收敛） | 记账、改柜体参数、表单填写 | 填表——按字段顺序逐一追问 | 键值对 | 值域校验 |
| topic_based（弱收敛/发散） | 角色扮演、诗歌创作、方案设计 | 聊天——在主题边界内自由探索 | 主题标签 + 对话摘要 | 输出格式校验 + topicEvolution event 追加 |

### 4.3 安保规矩（守卫规则）

#### 全局守卫优先级层级

**最高**：退出词 > **高**：唤醒词（仅IDLE）/取消词/align（仅LISTENING，alignScenarioFlag=true） > **普通**：DET关键词扫描

#### IN_SESSION 内守卫（降级链五项）

| 守卫类型 | 触发条件 | 执行体 | 动作（field_based） | 动作（topic_based） |
|---------|---------|--------|-------------------|-------------------|
| 越界检测辅助 | 每轮用户输入 | DET | 关键词扫描，标记 offTaskSuspicion | 同左 |
| L1 结构校验 | turnType="complete" | DET | JSON.parse+Schema校验 | 输出为JSON时同左；纯文本时做格式检查 |
| DET 值域复验 | turnType="complete" | DET | 值域校验（金额上限、时间格式、必填字段…），规则来源N9 @section validation | 输出格式校验（JSON Schema/文本格式/字数/结构…）。无validation时仅执行L1+confidence检查 |
| confidence 检查 | turnType="complete" | DET | 两步：①数值<threshold（默认0.4）②pattern扫"硬撑式确认"词表。冷启动期软拦截/非冷启动期硬拦截 | 同左 |
| 跨任务延伸检测 | turnType="complete" | DET（默认）/LLM（例外） | DET字段白名单 | 主题边界检查（可能needSemanticExtensionCheck=true追加轻量LLM） |
| 硬编码兜底 | 熔断时 | DET | 硬编码回复。不受validation规则存在与否的影响 | 同左 |
| 校验失败转发 | turnType="validation_failed" | DET | 进入CLARIFYING，复用模型message转发 | 同左 |
| 偏离暂存 | turnType="off-task" | DET | 暂存键值对到_pendingCollectedFields（source="off_task"） | 暂存对话摘要+主题标签（source="off_task"） |
| 放弃清理 | turnType="giveup" | DET | 清空activeSession | 同左 |

### 4.4 topicEvolution 存储规矩

| 约束项 | 规范 |
|--------|------|
| topicId 唯一性 | `{userId}_{intent}_{topicSlug}_{date}` |
| changeLevel 合法性 | 小写 major/minor/invalid，格式错误→默认 minor+警告日志 |
| 分层留存 | major永久 / minor累计5条压缩归档 / invalid不入主表 |
| 注入包结构 | majorEvents+compressedMinorSummary+recentMinorEvents+currentStateSnapshot（与stateSnapshotMark联动） |
| turnHistory changeLevel | field_based可省略（null）；topic_based必填（major/minor/invalid），同步携带 changeLevelReason（major/minor时必填1-100字理由，invalid时可省略） |
| topicEvolutionEvent changeLevelReason | topic_based major/minor 时必填 | 防偷懒机制——理由的存在迫使模型为判定负责。空值/超长→DET拦截+默认"未提供"并记录警告 |

### 4.5 领域规则存储规矩（v4.2 新增）

领域规则不再以平铺列表存储，而是以**有向图**结构组织——每条规则是一个节点，节点之间通过关联边建立关系。

#### domainRules 表字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ruleId | string | ✅ | 规则唯一标识 |
| rule | string | ✅ | 规则文本（自然语言描述） |
| topicPath | JSON数组 | ✅ | 规则作用域的主题路径，如 `["全屋定制","设计","厨房设计"]`。用于子图匹配——context-manager 注入时按当前 topicPath 前缀匹配相关规则 |
| edges | JSON对象 | ✅ | 关联边，含 `conflictsWith`（冲突规则 ID 列表）/ `dependsOn`（依赖规则 ID 列表）/ `strengthenedBy`（强化规则 ID 列表） |
| conditions | JSON对象 | 可选 | 规则生效条件，如 `{"cabinetType": "L型"}`。注入时检查是否满足 |
| immutableLevel | string | ✅ | 权限层级：L0（钢印，不可修改）/ L1（基线，需设计者确认）/ L2（用户确认，可随时修改）/ L3（模型推断，需确认升格） |
| source | string | ✅ | 来源：scenario_seed（场景种子）/ user_initiated（用户主动声明）/ model_inferred（模型推断） |
| level | string | ✅ | 优先级：strong（模型应优先遵循）/ medium（可基于需求偏离）/ suggestion（仅参考） |
| confirmedBy | string | 可选 | 确认人标识（user_initiated 和 model_inferred 升格时必填） |
| confirmedAt | string | 可选 | 确认时间戳 |
| version | integer | ✅ | 规则版本号，每次修改递增 |
| status | string | ✅ | active（生效）/ candidate（候选待验证）/ archived（废弃）/ pending_review（需人工审核） |

#### ruleCandidates 候选规则池

| 字段 | 类型 | 说明 |
|------|------|------|
| candidateId | string | 候选规则唯一标识 |
| rule | string | 候选规则文本 |
| evidenceCount | integer | 证据主题数量 |
| passCount | integer | 交叉验证通过次数 |
| validationRate | float | 验证通过率（passCount / evidenceCount） |
| status | string | pending（待验证）/ confirmed（已确认）/ rejected（已否决） |

#### ruleMiningQueue 规则提炼待处理队列

| 字段 | 类型 | 说明 |
|------|------|------|
| queueId | string | 队列项唯一标识 |
| topicId | string | 来源主题 ID |
| intent | string | 来源意图 |
| checkpointAt | string | 推入队列的时间戳 |
| status | string | pending（待处理）/ processing（处理中）/ done（已完成） |

#### 规则网络结构

规则节点之间通过 edges 中的三种关联边建立有向图：

| 边类型 | 含义 | 注入行为 |
|--------|------|---------|
| dependsOn | 依赖关系 | 注入当前规则时，沿 dependsOn 边外扩一层，同时注入被依赖的规则 |
| conflictsWith | 冲突关系 | 匹配到冲突规则时触发用户询问，按 immutableLevel 决定策略（L0 直接覆盖不询问 / L1 推荐询问 / L2 偏好询问） |
| strengthenedBy | 强化关系 | 注入当前规则时，沿 strengthenedBy 边外扩一层，同时注入强化规则 |

#### 注入匹配逻辑

context-manager 第0层注入时，不读取全量 domainRules，而是执行子图匹配：

1. 获取当前 topicPath（如 `["全屋定制","设计","厨房设计"]`）
2. 在 domainRules 中匹配 topicPath 前缀一致的规则节点
3. 沿 dependsOn 和 strengthenedBy 边向外扩一层
4. 检查 conditions 是否满足（不满足的跳过）
5. 仅注入匹配到的子图规则，上下文占用从 O(领域规则总数) 降为 O(子图规模)

## 五、关键纪律

1. 数据传递必须显式声明，禁止隐式依赖
2. turnHistory 的 askingField 和 changeLevel 字段必须落地
3. 全局守卫优先级最高，退出词优先于取消词
4. **CLOSING 双出边的 _pendingCollectedFields 处理必须区分**（N5 v5.6 回写）
5. 降级链五项检查（L1+DET值域复验+confidence+跨任务延伸检测+硬编码兜底）
6. taskType 在 N1/N3 确定，N6 不得自行更改
7. topic_based 任务必须处理 topicEvolution
8. confidence 区分冷启动/非冷启动
9. _closingSource 与 N4 exitType 映射（intent_exit/exit_word→normal→保留半成品；out_of_boundary*→fallback→清空半成品）
10. _pendingCollectedFields 含 source 标记（off_task/normal_pause）

## 七、领域规则体系（v4.2 新增）

领域规则是态控架构第三层宪法（领域规则层）在生产过程中逐步建立的约束体系。与宪法不同——宪法是设计者预设的刚性框架，领域规则从用户的实际使用中"长出来"。

**规则来源三层**：

| 来源 | 含义 | 灌入时机 | 初始权限 | 交叉验证 |
|------|------|---------|---------|---------|
| scenario_seed | 设计者预设的场景固有规则 | ANALYZING 主题确认时加载到 domainRules | L0/L1 | 不需要 |
| user_initiated | 用户主动声明的偏好/约束 | ANALYZING 规则问询步骤写入 ruleCandidates | L2 | 不需要（用户说的直接采纳） |
| model_inferred | 模型从交互中推断的行为模式 | 异步规则提炼引擎从 topicEvolution 提取，写入 ruleCandidates | L3 | 需要（≥ domain_rule_validation_threshold 后提用户确认） |

**权限分层 L0-L3**（与宪法层级正交——每条领域规则同时具有"宪法层级=第3层领域规则"和"权限层级"两个独立维度）：

| 层级 | 名称 | 可修改 | 冲突行为 |
|------|------|--------|---------|
| L0 | 钢印规则 | ❌ 不可修改 | 无条件胜出，不询问 |
| L1 | 基线规则 | ⚠️ 需设计者确认 | 与 L2/L3 冲突时触发询问 |
| L2 | 用户确认规则 | ✅ 用户可随时修改 | 优先于 L3 |
| L3 | 模型推断规则 | ✅ 需用户确认升格 | 仅参考，不主动注入 |

**规则生命周期**：candidate（候选，在 ruleCandidates 中）→ active（生效，用户确认后升格到 domainRules）→ archived（废弃或被替代，迁入 topicEvolutionArchive）。

**与认知局部性的关系**：子图匹配取代了全量注入——模型不再收到"这个领域的所有规则"，而是只收到"当前主题路径上直接相关的那几条规则及其直接关联的规则"。上下文窗口占用从 O(领域规则总数) 降为 O(当前主题路径规则的子图规模)，是认知局部性原理在领域规则层的落地。

**异步规则提炼机制**（独立后台进程执行，N5 调度器在 EXECUTING/giveup 完成时推入 ruleMiningQueue，N10 管理提炼参数）：

| 维度 | 规范 |
|------|------|
| 触发条件（三选一） | ① 时间窗口触发：cron 表达式由 rule_mining_schedule tunable 配置（默认 `0 2 * * *`）② 数量阈值触发：ruleMiningQueue 累计 ≥ rule_mining_batch_threshold（默认 5）③ 手动触发：用户通过系统界面点击"立即提炼" |
| 提炼流程 | 1. LLM 从 topicEvolution event 中提取候选规则 → 写入 ruleCandidates（status=pending）2. 交叉验证：与同 intent/topicPath 前缀的独立主题 event 比对，validationRate ≥ domain_rule_validation_threshold（默认 0.7）→ 标记为 confirmed 3. 用户确认 → 升格为 L2 或 L3，写入 domainRules（status=active）4. 同类型新主题初始化时，按 topicPath 子图匹配精准注入 |
| 知识真空期 | 规则提炼异步执行，不保证跨相邻任务即时传递。真空期内第0层注入跳过 |
