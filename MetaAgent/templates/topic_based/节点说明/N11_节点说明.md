## 一、位置与定位

```
N10 tunable 槽位声明（已定稿）
  ↓
N11 契约对齐  ← 你在这里
  ↓
N12 L2→L3 拆包
  ↓
N13 骨架生成
  ↓
N14 审骨架 + 跑 case + 查机制
```

**【人话版定位】**
前面 N1-N10 产出了一堆相互关联的交付物——N4 画了走廊图（转移边），N5 定了接待规矩（调度逻辑），N6 定了转场规矩（数据协议），N9 写了各房间服务手册（环节宪法），N10 装了可调旋钮（tunable 参数）。N11 的任务是拿着这些交付物逐项对账——N4 说的"这条边走完要做值域校验"和 N5 第 3 层实际实现的校验逻辑是不是一回事？N9 写的 output-schema 和 N5 第 2 层的 turnType 定义是不是一字不差？N6 说的"IN_SESSION→EXECUTING 携带 changeLevel"和 N9 环节宪法里声明的 changeLevel 判定规则是不是吻合？

**N11 不创造新东西，只检查已有东西之间是否一致**。发现不一致就标注出来，给出修正建议，由设计者决定改哪边。

### 4.1 产出一：契约对齐报告

契约对齐报告是 N11 的核心产出，逐项检查 N4/N5/N6/N7/N9/N10 之间的一致性。报告包含：检查项、上游来源 A、上游来源 B、是否一致、不一致时的差异描述和修正建议。

#### 4.1.1 转移边属性与调度逻辑的对齐

逐条检查 N4 转移边的属性是否与 N5 调度器的对应处理逻辑一致。

| # | 检查项 | N4 来源 | N5 来源 | 对齐标准 |
|---|--------|---------|---------|---------|

| 2 | exitType 与 CLOSING 双出边 | transitions 数组中 fromState="CLOSING" 的 exitType 字段 | N5 第3层 CLOSING 双出边处理逻辑（_closingSource 映射） | N4 的 exitType="normal" 对应 N5 _closingSource="intent_exit"或"exit_word"；exitType="fallback" 对应 N5 _closingSource="out_of_boundary*" |

| 4 | expectsChangeLevel 的读取来源 | N4 路由表条目中的 expectsChangeLevel 字段（非转移边） | N5 第2层 topicEvolution 处理逻辑中 changeLevel 预判的消费位置 | **N5 从 N4 路由表条目读取 expectsChangeLevel，非转移边。如果 N4 转移边 JSON 中的 expectsChangeLevel 与路由表条目不一致，以路由表条目为准。** N5 从转移边读取则视为不一致 |

## 五、关键纪律

1. **N11 不修改上游交付物**——只标注不一致和修正建议。由设计者决定改 N4 还是改 N5
2. **P0 阻断项必须修正后才能进 N12**——P0 项未清零时 N11 质量门禁不予通过
3. **turnType schema 的对齐是最高优先级**——N5 和所有 N9 环节宪法的 turnType 定义必须一字不差
4. **expectsChangeLevel 以 N4 路由表条目为准**——N5 从路由表条目读取，非转移边。如果 N4 转移边 JSON 中的值与路由表不一致，以路由表为准
5. **changeLevelReason 字段规范必须三处一致（N5/N6/N9）**——长度限制（1-100 字）和兜底行为（默认"未提供"）必须完全一致
6. **CLOSING 双出边的 _pendingCollectedFields 处理必须一致**——N4 exitType、N5 _closingSource、N6 协议表三处描述必须对齐
7. **context-manager 接口的调用关系必须明确**——context-manager 内部按 currentTopicId 从 contractStore 查询数据，调度器不预组装
8. **对齐报告须经设计者签字确认**——模型可执行逐字段比对，但最终的对齐策略由设计者决定

## 十、小白操作指南

### 10.0 先搞清楚你在干什么

你现在要干的事：当质检员。前面 N1-N10 产出了一堆相互关联的文档，N11 的任务是拿着这些文档逐项对账，确保它们之间没有矛盾。

**打个比方**：
- N4 画了走廊图，标注"这条走廊走完要做值域校验"
- N5 写了接待规矩，实现了一个"金额不能超过 999999"的检查
- N11 的工作就是拿着 N4 的走廊图，去 N5 的接待规矩里找到对应的检查逻辑，确认两者说的是同一件事

**N11 不创造新东西**。它只检查已有东西之间是否一致。发现不一致就标注出来，给出修正建议，由设计者决定改哪边。

### 10.1 动手前先检查

|-------------|---------|---------|
| N4 转移图 + 路由表 | N4 的人 | 所有转移边的属性。expectsChangeLevel 以路由表条目为准 |

| N6 数据传递协议 | N6 的人 | 16 条转移边的携带/清空数据、turnHistory 约束（含 changeLevelReason） |

| N10 tunable 清单 | N10 的人 | 各参数的消费方 |

### 10.2 六步走

**第一步：检查转移边属性与调度逻辑的对齐（4 项）**

拿 N4 的转移边，去 N5 找对应的处理逻辑：
- N4 说"这条边走完要做格式校验"→ 看 N5 第 3 层有没有 topic_based 输出格式校验
- N4 说"这条边 exitType=fallback"→ 看 N5 的 _closingSource 映射对不对
- N4 说"这条边 topicEvolutionEventAppended=true"→ 看 N5 第 2 层有没有对应的 event 追加逻辑
- **expectsChangeLevel**：N5 从 N4 路由表条目读取，不是转移边。如果两处值不一致，以路由表为准

**第二步：检查 turnType schema 的对齐（6 项）**

拿 N5 的 turnType schema，去和所有 N9 环节宪法的 @section output-schema 逐字段比对：
- turnType 六值的名字一个字都不能差
- askingField 的语义描述（field_based=字段名，topic_based=主题标签）必须一致
- changeLevel 字段：topic_based 的有，field_based 的没有
- changeLevelReason 字段规范：**三处（N5/N6/N9）**必须一致——1-100 字，空值/超长→DET 拦截+默认"未提供"
- topicId 字段：topic_based 的有，描述一致
- 所有环节宪法都落地了规则 1（输出纯 JSON）

**第三步：检查 validation 规则与 DET 复验的对齐（4 项）**

拿 N9 的 @section validation 结构化 JSON，去和 N5 的 DET 复验实现对照：
- N9 用的规则类型（range/format/array_length 等）在 N5 里都有对应的实现
- error_tag 名称两处一致
- needSemanticExtensionCheck 的声明一致（field_based 通常 false，topic_based 按需 true）
- topic_based 没 validation 的时候——N9 保留标签但 rules 为空数组，N5 跳过 DET 值域复验和跨任务延伸检测

**第四步：检查数据传递协议的对齐（4 项）**

拿 N6 的协议表，去和 N4 转移边、N5 调度逻辑对照：
- N6 的 16 条边都来自 N4 的转移边，没有遗漏也没有多余
- topic_based 任务的 IN_SESSION→EXECUTING 携带 changeLevel
- CLOSING 双出边的 _pendingCollectedFields 处理三处一致（N4 exitType / N5 _closingSource / N6 协议表）
- turnHistory 的结构三处一致（N5 schedulerState / N6 约束 / N9 output-schema）

**第五步：检查 taskType 传递链路和规则落地（7 项）**

- 顺着 N1→N3→N5→N9 检查每个 intent 的 taskType 一路传递没变
- 搜索所有环节宪法的 @section role，确认都写了"坚持真实判断"
- 搜索所有环节宪法的 @section off-task-detection，确认都写了"不得主动引入延伸话题"
- topic_based 宪法的 @section topic-boundary，确认都写了"不得引入与主题无关的话题"
- 所有环节宪法的 @section ask-rules，确认都有类型/主题/格式三个维度的追问示例
- N7 第 2 条元指令集词表（唤醒/退出/取消）与 N5 第 0 层和 N6 全局守卫一致

**第六步：生成契约文件**

全部 29 项检查通过后，三份契约文件由模型从各上游交付物中自动汇总生成，设计者只需审查确认：
- **contract-in/out schema**：每个状态各一份，从 N5 的 contract-in/out 定义自动提取
- **turnType-schema.json**：从 N5 第 2 层 turnType 定义自动汇总，与 N9 所有环节宪法的 output-schema 交叉验证
- **interfaces.json**：从 N5 的 context-manager 三接口和 schedulerState 数据结构自动汇总

设计者无需手动编写这三份文件——审查确认一致性和完整性即可。
