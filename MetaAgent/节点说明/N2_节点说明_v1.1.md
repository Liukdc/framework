# N2 节点说明（元智能体 v1.1）
**版本**：v1.1 | **日期**：2026-07-11 | **基于母本**：N2 节点说明 v5.9 / MetaAgent L2 v2.0

---

## 一、位置与定位

```
N1 边界清单
  ↓
N2 边界紧张度测试  ← 你在这里
  ↓
N3 状态枚举
```

N2 验证 N1 的边界清单是否经得起考验——用模拟用户输入测试边界覆盖度。如果 N1 的边界有缺口，N2 会暴露出来。

**双角色串行模型**：角色一（语料猎人）发散生成各种可能的用户输入；角色二（边界测试员）严格判断每条输入能否被 N1 边界清单正确处理。两个角色用不同的系统提示——一个发散、一个严格。

|--------|------|--------|------|

| 角色二系统提示 | 系统 | ✅ 必需 | 严格归类语料 |

| 输入项 | 来源 | 说明 |

| N1 边界清单 | N1 | doList/notDoList 全部条目 |

## 四、产出物

边界紧张度测试报告：

| 字段 | 说明 |
|------|------|
| totalCorpus | 生成的测试语料总数 |
| class1Count | 类一（明确可处理）数量 |
| class2Count | 类二（明确应拒绝）数量 |
| class3Count | 类三（边界模糊）数量 |
| coverage | (类一+类二) / totalCorpus |
| gaps | 类三条目列表，每条含 issue + suggestion |
| passed | coverage ≥ 0.85 |

---

## 五、双角色串行流程

### 5.1 角色一：语料生成

**目标**：基于 N1 边界清单，从正反两个方向生成测试语料。

**三类语料**：
1. **正面语料**：基于 doList 每个 intent，用不同说法生成 3-5 条正常用户输入
2. **负面语料**：基于 notDoList 每个条目，生成 2-3 条触发该边界的输入
3. **模糊语料**：生成意图不明确、或介于两个 intent 之间的输入

**数量要求**：总语料数 ≥ (doList条目×3 + notDoList条目×2)。最少 15 条。

### 5.2 角色二：语料归类

**目标**：用严格标准逐条判断每条语料能否被 N1 边界清单正确处理。

**三类归类**：
- **类一**：直觉可处理——落到某个 doList intent 范围内，有明确的处理路径
- **类二**：直觉应拒绝——落到某个 notDoList 条目范围内，有明确的处理策略
- **类三**：边界模糊——不知道该归哪类。说明边界有缺口

**归类原则**：宁严勿松。稍有犹豫就归为类三——宁可多标注缺口，不放过潜在漏洞。

### 5.3 缺口分析

对类三每条：
- 描述问题：为什么这条语料无法归类
- 定位缺口：是 doList 缺少某个 intent，还是 notDoList 缺少某个边界条目，还是边界描述不够清晰
- 给出建议：应该怎么修改 N1 边界清单

---

## 六、覆盖率判定

- coverage ≥ 0.85 → 通过，进入 N3
- coverage < 0.85 → 不通过，类三条目作为修正建议返回 N1

---

## 七、元智能体特殊处理

元智能体的 N1 边界清单是为"被设计智能体"写的（不是元智能体自己的边界），因此 N2 的测试语料是针对被设计智能体的场景生成。

如果用户设计的智能体场景非常简单（如只有 1 个 intent + 3 条 notDo），降低最小语料数要求——但不能低于 10 条。

---

## 八、关键纪律

1. **角色一和角色二用不同系统提示**——不能用一个提示同时做生成和归类
2. **覆盖率阈值不可低于 0.85**——这是硬指标
3. **类三不能直接跳过**——每条必须有明确的缺口描述和修正建议
4. **宁严勿松**——有疑虑就归为类三

---

## 九、N2 自检清单

- [ ] 正面语料覆盖所有 doList intent（每个至少 3 条）
- [ ] 负面语料覆盖所有 notDoList 条目（每个至少 2 条）
- [ ] 模糊语料至少 3 条
- [ ] 总语料数 ≥ 最小值
- [ ] 所有类三条目有 issue + suggestion
- [ ] 覆盖率 ≥ 0.85 或已标注 gap 建议回 N1 修正
