# N4 节点说明（元智能体 v2.0）

**版本**：v2.0 | **日期**：2026-07-12 | **状态**：已定稿 | **基于母本**：N4 节点说明 v5.6

---

## 一、位置与定位

```
N3 状态枚举 + 执行体分派（已定稿）
  ↓
N4 状态转移图 + 路由表  ← 你在这里
  ↓
N5 调度器核心逻辑
```

**定位**：基于 N3 的 24 个状态，画出所有状态之间的转移边，并配置路由表。全部 field_based，无 topicEvolution，无软着陆。

| 输入项 | 来源 | 说明 |
|--------|------|------|

## 三、产出物

N4 产出五样东西：状态转移图、路由表、降级链路由、Mermaid 图、N4→下游传递契约。

### 3.1 产出一：状态转移图

元智能体 16 步设计流程为严格顺序（P0→N1→[N2跳过]→N3→...→N15），但状态转移图需覆盖所有路径类型。

#### 最优路径

```
IDLE → LISTENING → ANALYZING → IN_SESSION(步骤名) → WAITING_CONFIRM → EXECUTING → LISTENING → ... → CLOSING → IDLE
```

用户唤醒"小智"进入 LISTENING，每轮输入经 ANALYZING 匹配意图后进入对应 IN_SESSION 子类型。步骤完成产出后→WAITING_CONFIRM 弹确认→用户确认→EXECUTING 写入→LISTENING 等待下一步。

**确认机制说明**：元智能体每一步产出均为设计文档/代码文件，保存即写盘。所有产出保存前均需用户确认。重跑已完成步骤时，WAITING_CONFIRM 额外提示"已有产出将被覆盖"。

#### 异常路径

| 路径类型 | 转移边 | 说明 |
|---------|--------|------|
| validation_failed | IN_SESSION → CLARIFYING → IN_SESSION | DET 校验失败，转发模型 message 追问 |
| off_task | IN_SESSION → ANALYZING | 用户偏离当前步骤，重新意图识别 |
| giveup | IN_SESSION → LISTENING | 用户放弃当前步骤 |

#### 特殊路径

| 路径类型 | 转移边 | 说明 |
|---------|--------|------|
| confirm | IN_SESSION → WAITING_CONFIRM → EXECUTING | 每步产出保存前弹确认。重跑已完成步骤时额外提示"已有产出将被覆盖"。每次确认后记录修改日志，日志作为异步规则提炼的素材 |
| exit | CLOSING → IDLE / CLOSING → LISTENING | 退出会话 / 退出当前步骤 |

#### 不适用路径

| 路径类型 | 理由 |
|---------|------|
| chitchat_path | N1 slackNodeDeclaration.enabled=false，无松弛节点 |
| soft_landing_path | N1 softLandingTargets 为空 |
| silent_input | alignScenarioFlag=false |

#### 前置依赖矩阵（供 N5 前置依赖守卫消费）

元智能体 16 步有严格顺序依赖。每个 intent 的 ANALYZING→IN_SESSION 转移边需校验前置依赖是否全部完成。未完成时 N5 前置依赖守卫拦截并引导至正确步骤。

| intent | 描述 | 前置依赖 |
|--------|------|---------|
| p0 | 认知加载 | 无（入口） |
| n1 | 场景定义+边界清单 | p0 |
| n3 | 状态枚举+taskType固化 | n1 |
| n4 | 状态转移图+路由表 | n3 |
| n5 | 调度器核心逻辑 | n1, n3, n4 |
| n6 | 数据传递协议 | n5 |
| n7 | 根宪法编写 | n5, n6 |
| n8 | 公共规则编写 | n7 |
| n9 | 各环节宪法编写 | n7, n8 |
| n10 | tunable槽位声明 | n9 |
| n11 | 契约对齐 | n10 |
| n12 | L2→L3拆包 | n11 |
| n13 | 骨架生成 | n12 |
| n14 | 审骨架+跑case+查机制 | n13 |
| n15 | 调参+交付 | n14 |

**注**：N2 跳过（纯 field_based 场景），不在依赖链中。依赖链为 P0→N1→N3→N4→N5→N6→N7→N8→N9→N10→N11→N12→N13→N14→N15。

### 3.2 产出二：路由表

#### 第一层：状态级默认模型

| 状态 | 默认模型 |
|------|---------|
| ANALYZING | 通用轻量模型 |
| IN_SESSION(p0~n15) | 通用模型 |

#### 第二层：intent 级 modelHint

16 个子类型全部 field_based，modelHint 统一为"通用模型即可"。

### 3.3 产出三：降级链路由

N5 调度器在 IN_SESSION 出口执行五步 DET 防线。N4 为每条降级路径设计转移边：

| 步骤 | 检查内容 | 失败处理 | N4 转移边 |
|------|---------|---------|----------|
| ① L1 结构校验 | JSON.parse + Schema 校验 | 重试 1 次→熔断→硬编码回复 | IN_SESSION → IN_SESSION（重试）/ IN_SESSION → EXECUTING（熔断后落盘兜底回复） |
| ② DET 值域复验 | field_based 值域校验（字段类型/范围/必填性） | 不通过→转发模型 message | IN_SESSION → CLARIFYING → IN_SESSION |
| ③ confidence 检查 | 置信度 < confidence_threshold | 冷启动→软拦截（标记 warning）；非冷启动→硬拦截 | 冷启动：IN_SESSION → EXECUTING（带 warning 继续）。非冷启动：IN_SESSION → CLARIFYING |
| ④ 跨任务延伸检测 | 模型是否越界讨论其他节点内容 | 越界→退回 IN_SESSION | IN_SESSION → IN_SESSION（调度器裁剪越界内容后重入） |
| ⑤ 硬编码兜底 | 以上均不适用时的最终保护 | → 落盘兜底回复 | IN_SESSION → EXECUTING |

**field_based 场景特点**：元智能体全部 field_based，不需要区分 topic_based 的格式校验和 needSemanticExtensionCheck。DET 值域复验直接按各环节宪法的 @section validation 规则执行。

## 四、关键纪律

1. 转移边覆盖全部 24 个状态（不含 N3 省略的 6 个状态）
2. 不适用路径在 inapplicablePathTypes 中声明理由
3. 全部 field_based，topicEvolutionEventAppended 统一 false
4. ANALYZING 节点使用轻量模型
5. 所有路由规则均为 hard_match
6. 每步产出保存前过 WAITING_CONFIRM——用户确认后方可写入。修改已有产出时记录修改日志，日志作为规则提炼素材
7. 每个 intent 的 ANALYZING→IN_SESSION 转移边必填 prerequisiteDependencies——N5 前置依赖守卫消费此字段

## 六、常见踩坑

| 坑 | 表现 | 后果 |
|----|------|------|
| 给省略状态设计转移边 | N3 省略了 SLACK_NODE 但 N4 照样画了边 | 下游引用不存在的状态 |
| IN_SESSION→EXECUTING 跳过了 WAITING_CONFIRM | 产出直接落盘不确认 | 用户不知道保存了什么，无修改日志可追溯 |
| 忘记标注 validationType | validation_failed_path 没标注 value_domain | N5 无法判断校验失败的处理分支 |
| CLOSING 出边只画了一条 | 只画了 CLOSING→IDLE | 越界硬拒无法回 LISTENING |

## 七、小白操作指南

### 7.0 先搞清楚你在干什么

N3 定了 24 个房间。你的任务是画走廊——哪些房间之间有门（转移边），走哪扇门由什么条件决定（路由表）。

### 7.1 动手前先检查

| 你要用到的东西 | 谁给你的 | 用来干嘛 |
|-------------|---------|---------|
| N3 状态集（24 个） | N3 的人 | 画转移边的起终点 |
| N3 省略状态（6 个） | N3 的人 | 确认不为这些状态画边 |
| N3 IN_SESSION 子类型（16 个） | N3 的人 | 写路由表的 intent 级条目 |

### 7.2 三步走

**第一步：画最优路径**

IDLE→LISTENING→ANALYZING→IN_SESSION→WAITING_CONFIRM→EXECUTING→LISTENING（循环）→CLOSING→IDLE。每步产出保存前必经 WAITING_CONFIRM。

**第二步：补异常和特殊路径**

validation_failed（IN_SESSION→CLARIFYING→IN_SESSION）、off_task（IN_SESSION→ANALYZING）、giveup（IN_SESSION→LISTENING）、confirm（IN_SESSION→WAITING_CONFIRM→EXECUTING，每步保存都需确认）、exit（CLOSING→IDLE / CLOSING→LISTENING）

**第三步：填路由表**

第一层：ANALYZING=轻量模型，IN_SESSION=通用模型。第二层：16 个子类型全部"通用模型即可"。全部 hard_match。

### 7.3 自查清单

- [ ] 24 个状态的转移边全覆盖
- [ ] 最优路径含 WAITING_CONFIRM→EXECUTING
- [ ] 不适用路径已声明理由
- [ ] validation_failed_path 已标注 validationType=value_domain
- [ ] CLOSING 出边已标注 exitType（normal + fallback）
- [ ] 每个 intent 的 ANALYZING→IN_SESSION 转移边已填写 prerequisiteDependencies
- [ ] 前置依赖矩阵与 N1 的 16 步顺序链完全一致

## 八、一句话收 N4

元智能体 N4 基于 N3 的 24 个状态画出完整转移图——最优路径 + 异常路径（validation_failed/off_task/giveup）+ 特殊路径（confirm/exit）+ 降级链五步路由 + 前置依赖矩阵 16 行，3 类不适��路径已声明理由。路由表两层结构：ANALYZING=轻量/IN_SESSION=通用，16 个子类型全部 hard_match。全部 field_based，无 topicEvolution 路由。
