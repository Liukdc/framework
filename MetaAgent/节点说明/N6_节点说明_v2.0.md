# N6 节点说明（元智能体 v2.0）

**版本**：v2.0 | **日期**：2026-07-12 | **状态**：已定稿 | **基于母本**：N6 节点说明 v2.7

## 一、位置与定位

```
N5 调度器核心逻辑 + context-manager 三接口（已定稿）
  ↓
N6 数据传递协议 + 守卫规则 + 领域规则存储  ← 你在这里
  ↓
N7 根宪法编写
```

**定位**：基于 N4 转移图和 N5 五层接待规则，定义每个状态转移时携带什么数据、清空什么数据（数据传递协议）、每条路径上有什么守卫规则，以及领域规则（异步规则提炼产出）如何存储。全部 field_based，无 topicEvolution，无 N2 双角色。

## 四、产出物

N6 产出三样东西：数据传递协议表、守卫规则、领域规则存储约束。

### 4.1 数据传递协议表

元智能体 16 步均为 field_based——每步产出为预定义字段集。以下 15 条协议覆盖全部转移路径（删除 2 条：ANALYZING→WAITING_CONFIRM 为 N2 专属、ANALYZING→SLACK_NODE 松弛节点不启用）。

| # | 转移路径 | 携带数据 | 清空数据 |
|---|---------|---------|---------|
| 1 | LISTENING → ANALYZING | userInput | 无 |
| 2 | ANALYZING → IN_SESSION | intent+taskType=field_based+confidence | 无 |
| 3 | IN_SESSION → EXECUTING | result+validationResult+changeLevel+changeLevelReason | 无 |
| 4 | IN_SESSION → ANALYZING | offTaskInput+暂存collectedFields（source="off_task"） | 清空_activeSession |
| 5 | IN_SESSION → LISTENING | 无 | 清空_activeSession |
| 6 | EXECUTING → LISTENING | _lastRecord+_lastRecordId | 清空_activeSession |
| 7 | ANALYZING → LISTENING | 无 | 无 |
| 8 | ANALYZING → CLOSING | 无 | 清空_activeSession |
| 9 | WAITING_CONFIRM → EXECUTING | decision="confirm" | 无 |
| 10 | WAITING_CONFIRM → LISTENING | 无 | 清空当前确认上下文 |
| 11 | CLOSING → IDLE | 无 | 清空_activeSession；**保留_pendingCollectedFields**（按ttl过期） |
| 12 | CLOSING → LISTENING | 无 | 清空_activeSession+**清空_pendingCollectedFields** |
| 13 | IN_SESSION → CLARIFYING | validationResult+message | 无 |
| 14 | CLARIFYING → IN_SESSION | 用户修正输入+validationResult | 无 |
| 15 | 任意状态 → ANALYZING（逃生舱） | contractStore.analyzingContractIn | 清空_activeSession |

**_pendingCollectedFields source 标记**：

| source 值 | 含义 | CLOSING→IDLE 后行为 |
|-----------|------|-------------------|
| `off_task` | 用户跑题导致中断 | 不主动询问；用户可显式恢复 |
| `normal_pause` | 用户明确说"先存着等会儿继续" | 主动询问是否继续 |

### 4.2 守卫规则

#### 全局守卫优先级层级

**最高**：退出词 > **高**：唤醒词（仅IDLE）/取消词 > **普通**：DET关键词扫描

#### IN_SESSION 内守卫（降级链五项）

| 守卫类型 | 触发条件 | 执行体 | field_based 动作 |
|---------|---------|--------|-----------------|
| 越界检测辅助 | 每轮用户输入 | DET | 关键词扫描，标记 offTaskSuspicion |
| L1 结构校验 | turnType="complete" | DET | JSON.parse+Schema校验 |
| DET 值域复验 | turnType="complete" | DET | 值域校验——规则来源 N9 @section validation |
| confidence 检查 | turnType="complete" | DET | 两步：①数值<threshold ②pattern扫"硬撑式确认"词表。冷启动软拦截/非冷启动硬拦截 |
| 跨任务延伸检测 | turnType="complete" | DET | 字段白名单——16步内字段不触发越界 |
| 硬编码兜底 | 熔断时 | DET | 硬编码回复 |
| 校验失败转发 | turnType="validation_failed" | DET | 进入CLARIFYING，复用模型message转发 |
| 偏离暂存 | turnType="off-task" | DET | 暂存键值对到_pendingCollectedFields（source="off_task"） |
| 放弃清理 | turnType="giveup" | DET | 清空activeSession |

### 4.4 领域规则存储约束

元智能体启用异步规则提炼——每条规则直接映射到 stepName，经交叉验证和用户确认后存入领域规则库。

#### domainRules 表字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ruleId | string | ✅ | 规则唯一标识 |
| rule | string | ✅ | 规则文本（自然语言描述） |
| stepName | string | ✅ | 规则作用的环节名——如 n5/n9/n15。context-manager 按 stepName 查询注入 |
| edges | JSON对象 | ✅ | 关联边：conflictsWith/dependsOn/strengthenedBy |
| conditions | JSON对象 | 可选 | 规则生效条件 |
| immutableLevel | string | ✅ | L0钢印/L1基线/L2用户确认/L3模型推断 |
| source | string | ✅ | scenario_seed/user_initiated/model_inferred |
| level | string | ✅ | strong/medium/suggestion |
| confirmedBy | string | 可选 | 确认人标识 |
| confirmedAt | string | 可选 | 确认时间戳 |

### 4.5 contractEvolution 分层留存规则

全部 field_based，topicEvolution 不启用。但 contractEvolution 是 topicEvolution 的 field_based 等价物——changeLevel 事件的分层留存：

| 事件级别 | 留存策略 | 说明 |
|---------|---------|------|
| major | **永久保留**不可压缩 | 核心设计决策的根本转向——如用户将 intent 从 field_based 改为 topic_based |
| minor | **30 条压缩为 1 条摘要** | 渐进调整——如修改某字段默认值、补充边界描述 |
| invalid | **归档后定期删除**（topic_evolution_invalid_retention_days） | 完全撤销的变更——如用户改了又改回原值 |

**三层注入包**（N5 第2层构建）：
- 第0层：已生效领域规则（按 stepName 精确匹配 + immutableLevel 过滤）
- 第1层：候选规则预览卡片（validationRate ≥ 阈值）
- 第2层：变更历史包（majorEvents近期3条 + compressedMinorSummary 30条压缩 + currentStateSnapshot）

**异步规则提炼触发**：ruleMiningQueue 积压 ≥ rule_mining_batch_threshold → N10 定时任务提炼。

#### 4.5.5 sessionCheckpoint：field_based 跨会话进度记忆

全部 field_based，topicEvolution 不启用——但 MetaAgent 的 16 步设计流程需要"用户做到 N7 了，下次继续"的进度记忆。sessionCheckpoint 是 field_based 模式的原生跨会话记忆机制：

| 配置项 | 值 |
|--------|-----|
| 存储表 | sessionCheckpoints（contractStore 新增） |
| 记录内容 | userId / lastCompletedStep / completedSteps[] / stepSnapshots{} |
| 过期策略 | ttl 默认 604800 秒（7 天），过期自动清除 |
| 写入时机 | 每个 IN_SESSION complete → updateGraph 同步更新 checkpoint |
| 消费方 | N5 ANALYZING 阶段——检测到未过期 checkpoint → 发起 resume_chat |

与 contractEvolution 互补：contractEvolution 管"同一环节的规则演变"，sessionCheckpoint 管"16 步流程走到哪儿了"。

#### 4.5.6 roomConversationLog：房间对话日志

sessionCheckpoint 管"流程进度"，roomConversationLog 管"每个房间聊了什么"——切换房间时不丢弃任何对话。每个 turn 实时落盘，按 userId+roomId 索引，支持搜索。

```sql
CREATE TABLE IF NOT EXISTS roomConversationLog (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    userId TEXT NOT NULL, roomId TEXT NOT NULL,
    sessionId TEXT NOT NULL, turnIndex INTEGER NOT NULL,
    userInput TEXT NOT NULL, modelResponse TEXT NOT NULL,
    turnType TEXT NOT NULL, askingField TEXT, changeLevel TEXT,
    timestamp TEXT NOT NULL
);
CREATE INDEX idx_room_user_roomId ON roomConversationLog(userId, roomId);
```

**模型注入**：返回房间时加载该房间最近 20 轮对话流。**搜索**：用户引用历史讨论→DET 关键词扫描 roomId → 全文搜索。

| 配置项 | 值 |
|--------|-----|
| 写入 | 每 IN_SESSION turn 实时追加 |
| 注入 | 最近 20 轮对话（时序流） |
| 保留期 | room_log_retention_days（默认 90 天） |

## 五、关键纪律

1. 每条协议行必须包含 fromState/toState/carryData/clearData 四字段
2. field_based 数据传递无 topicId/changeLevel（除 IN_SESSION→EXECUTING 携带 changeLevel/changeLevelReason 用于修改日志）
3. 降级链五项守卫完整，全部 DET 执行
4. 全局守卫优先级：退出词 > 唤醒/取消 > 关键词扫描
5. domainRules 使用 stepName 映射（非 topicPath）
6. _pendingCollectedFields 区分 off_task/normal_pause 两种 source
7. contractStore 仅含 3 张表（domainRules/ruleCandidates/ruleMiningQueue）

## 六、常见踩坑

| 坑 | 表现 | 后果 |
|----|------|------|
| 协议表保留了 ANALYZING→WAITING_CONFIRM | N2 专属路径 N6 照样定义 | N13 骨架生成时多出一段用不上的代码 |
| changeLevel 标记为可选 | field_based 场景设为 null | 修改日志缺少变更级别，规则提炼无法判断改动幅度 |
| domainRules 用了 topicPath | 照搬基准版 topicPath 字段 | context-manager 按 stepName 查询匹配失败 |
| contractStore 建了 topicEvolution 表 | 照搬基准版 7 张表 | 4 张空表浪费存储 |
