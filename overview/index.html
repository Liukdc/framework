态控架构 v5.5（概要总纲）

定位：智能体设计总纲——回答"如何系统化地构建一个计算输出可控和成本最优的智能体"。

态控架构的目的：在状态机的调度约束下让所有场景大模型（智能体）和用户适配适合的输入和输出。

文档版本：v5.5 日期：2026-07-18

本文档是概要总纲——只含章节索引 + 核心定义 + 速查表。详细内容见对应的细则文件（02-12）。 当前细则版本：§三/§十/附录已更新至 v5.8（对话分段保留 + conversationArchive + 领域规则讨论房间），其余章节 v5.5。 变更记录不在本文件中——所有变更统一记录在"态控架构变更日志.md"。

二、核心概念速查​

2.1 哲学根​

认知局部性原理（输入侧）：任何信息处理主体在 t 时刻处理节点 N 时，最优输入集 = N 在当前目标下的最小充分条件集，不是全集 Γ。

  • 人：工作记忆 7±2 组块
  • LLM：Lost in the Middle + 近因偏置 + 长上下文幻觉率单调递增
  • DET：函数纯度 f(x) 只依赖 x

2.2 taskType 双模式​

维度field_based（离散参数空间）topic_based（连续语义空间）
收敛目标定位唯一正确值采样满足约束的样本
验证方式DET 值域硬校验格式校验 + 语义判断
跨节点载体sessionCheckpointtopicEvolution
典型场景记账、改参数、表单写诗、角色扮演、方案设计

2.3 三类执行体​

执行体性质职责
DET（确定性程序）确定性EXACT_MATCH + 值域复验 + logprobs裁决 + SQL查询 + 文件操作
LLM（概率性）概率性意图识别 + 字段采集 + 偏离判断 + 工具调用决策 + 结果格式化
HUMAN（人类选择）责任性不可逆决策确认 + 规则确认 + 偏好仲裁

2.4 房间比喻（四层结构）​

房间（环节，N1-N16）—— 环节宪法（房间级）
  └── 柜子（项目，如富贵小安/杂碎本）—— 产出物内容（柜子级，隔离）
       └── 格子（文件，精简总纲+细则）—— 正文内容（格子级，独立文件）
            └── 变更日志（独立文件）—— 所有变更追溯（全局统一）
  • 房间隔离：S3 物理隔离（KV Cache 释放），wake/switch/exit/cancel 四类口令
  • 自由切换四支柱：roomConversationLog（全量记录）+ stepSnapshot（对话快照）+ 规则提炼按环节隔离 + outputRegistry（产出物总索引）

三、根宪法4条（约束所有环节模型）​

#条款内容
1环节隔离只能做本环节任务，非本环节任务返回 off-task
2输出格式符合 turnType schema 的结构化 JSON
3收敛义务+坚持真实判断不伪造、不假装理解、不迎合用户、不无中生有
4不编造功能不得生成用户未交代、系统不具备的功能或信息

ANALYZING 特化：只产出意图识别信号（intent/choice/logprobs/inputNature/extracted/topicId），不产出 turnType。

四、架构机制M1-M4（约束调度器/DET）​

编号机制核心内容
M1元指令集四类口令（wake/switch/exit/cancel）+ S3物理KV Cache释放 + roomStateIndex注入
M2降级链四项检查L1结构校验 + DET值域复验 + logprobs检查 + 硬编码兜底（全确定性，不可缩减）
M3仲裁权DET优先DET校验结果优先于模型内部校验
M4节点转换守卫环节间转换的触发条件和守卫规则

五、编程规则宪法7条（约束代码模型）​

#条款核心内容
1执行体分派铁律ANALYZING=LLM, IN_SESSION=LLM, DET复验=DET, EXECUTING=DET（仅写入/删除）。IN_SESSION可调用DET工具
2DET职责边界确定性计算：EXACT_MATCH+值域复验+关键词扫描+logprobs裁决+路由矩阵+规则匹配+文件操作+SQL查询
3LLM职责边界意图识别+字段采集+偏离判断+语义边界+工具调用决策+结果格式化
4降级规则无apiKey时拒绝服务，不降级为正则
5契约对齐严格对齐N11契约schema
6tunable/接口隔离阈值从getTunable()读，调度器只调context-manager三接口（匹配/决策/拼接）
7工具分层发现必用工具+工具清单+选用工具三层 + search_tools meta-tool + LLM不直接写SQL

六、架构级决策清单（v5.1-v5.5）​

#决策版本核心内容
①off-task保障2→ANALYZING重新识别v5.1DET做不了语义验证，交给ANALYZING重新识别
②M1 switch口令"切断房间"v5.1确定性切换，0次LLM，100%可靠
③confidence→logprobsv5.1强制选择+API logprobs，消除幻觉式自信
④S3物理隔离v5.2KV Cache物理释放，零泄漏切换
⑤roomStateIndexv5.3全窗口房间状态摘要，跨房间全局视野
⑥降级链四项检查v5.3移除跨任务延伸检测，全确定性
⑦ANALYZING第二分类维度(S/T/U)v5.1S→SLACK_NODE, T→TEMP_TOPIC, U→追问
⑧outputRegistryv5.2跨房间产出物总索引，DET确定性查询
⑨IN_SESSION tool callingv5.4function calling，查询/对比在IN_SESSION内完成
⑩工具分层发现v5.5必用/清单/选用 + Two-Phase Selection

七、turnType 六值​

turnType含义askingField
ask模型追问用户当前追问字段名/主题标签
reply模型回复用户（非追问）null
complete所有字段齐全/输出满足主题边界null
off-task检测到偏离null
giveup用户放弃null
validation_failed值域/格式校验失败校验失败的字段名/主题维度

八、标准状态集​

状态类型执行体说明
IDLE稳态—空闲，等待唤醒
LISTENING稳态人待命，等待用户输入
ANALYZING瞬态LLM轻量意图识别环节（强制选择+logprobs+S/T/U）
IN_SESSION稳态容器LLM环节执行中（可调用DET工具）
CLARIFYING守护态DET校验失败追问转发（0次LLM）
WAITING_CONFIRM守护态人不可逆操作前确认
EXECUTING瞬态DET纯写入/删除操作
CLOSING瞬态DET会话清理
SLACK_NODE稳态LLM通用闲聊房间（仅根宪法约束）
TEMP_TOPIC稳态LLM通用临时主题房间（仅根宪法约束）

十二、文件管理机制（v5.5-v5.6）​

房间比喻四层结构​

房间（环节，N1-N16）—— 环节宪法（房间级，共享）
  └── 项目（智能体实例，如富贵小安/杂碎本）—— 产出物内容（项目级，隔离）
       └── 格子（细则文件，精简总纲+细则）—— 正文内容（文件级，独立）
            └── 变更日志（独立文件）—— 所有变更追溯（全局统一）

三个机制​

机制解决什么落地方式状态
机制1 项目隔离同环节多项目产出物隔离projectId + S3物理隔离 + projectRegistry表✅ v5.6已落地
机制2 分格大文件拆分，降低算力成本按章节拆成独立细则文件✅ v5.5已落地
机制3 变更日志独立变更可审计，防止误改所有变更统一记录在变更日志.md✅ v5.5已落地

机制1 项目隔离核心​

  • currentProjectId：全局字段，一处变更所有房间联动
  • 默认项目：userLastProject 表记录用户最后操作的项目
  • 项目切换：ANALYZING 关键词匹配 → S3 物理隔离四步 → 加载新项目最后状态
  • 项目切换 vs 房间切换：两者都需要 S3；房间切换变 roomId，项目切换变 projectId
  • 跨项目引用：走 outputRegistry 跨项目查询 + 文件复制（需用户确认）

文件管理纪律​

  1. 修改基线时另存新版本——不直接覆盖原文件
  2. 细则文件只含正文——不含变更记录（变更记录在变更日志.md）
  3. 总纲文件是概要索引——每次读取只读总纲，需要详情时读对应细则
  4. 大改动分批进行——每次只改一个细则文件
